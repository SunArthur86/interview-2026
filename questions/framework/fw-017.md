# 什么是Spring的IoC容器？

### 什么是Spring的IoC容器？

IoC（Inversion of Control，控制反转）容器是 Spring 框架的核心，它担任了 IoC Provider 的职责，管理对象的依赖关系、生命周期和配置。在此基础上，还增加了对 Bean 生命周期的完整管理、依赖注入（DI）以及 AOP 支持。

从整体来看，Spring IoC 容器的作用主要分为两个阶段：

1.  **容器启动阶段**：
    以某种方式（XML、注解、Java Config）将配置的 Bean 信息加载到整个 Spring 应用中。容器会进行 BeanDefinition 的资源定位、解析和注册。

2.  **Bean实例化阶段**：
    当应用请求某个 Bean 时（或者容器启动预实例化单例时），容器将加载的 Bean 配置信息组装成应用需要的实际业务对象。这包括实例化、属性填充、初始化回调等。

容器在这两个阶段都预留了扩展点，如 `BeanFactoryPostProcessor`（启动阶段修改元数据）和 `BeanPostProcessor`（实例化阶段前后置处理）。

**核心接口与类对比**：

| 特性 | BeanFactory | ApplicationContext |
| :--- | :--- | :--- |
| **定位** | Spring 基础设施，IoC 的核心 | 面向开发者，BeanFactory 的子接口 |
| **初始化时机** | 延迟加载（用到时才实例化） | 预实例化（启动时创建所有单例） |
| **功能支持** | 仅基本的 DI 功能 | 国际化、事件传播、资源加载、AOP 等 |
| **适用场景** | 轻量级应用、移动设备 | 企业级应用（绝大多数情况） |

**实战案例**：
在集成第三方库时，我们常需要根据配置文件动态注册 Bean。例如，集成 Redis 时，如果配置文件中有 redis 地址，我们才注册 `RedisTemplate`。这可以通过实现 `ImportBeanDefinitionRegistrar` 在容器启动阶段动态实现。**踩坑经验**：使用 `BeanFactory` 接口获取 Bean 时，如果忘记手动注册 `BeanPostProcessor`，AOP 代理将不会生效，因为 `BeanFactory` 基础实现不自动注册这些后置处理器。

**代码示例（编程式注册 Bean）**：
```java
public class MyBeanDefinitionRegistrar implements ImportBeanDefinitionRegistrar {
    @Override
    public void registerBeanDefinitions(AnnotationMetadata importingClassMetadata, BeanDefinitionRegistry registry) {
        BeanDefinitionBuilder builder = BeanDefinitionBuilder.genericBeanDefinition(MyService.class);
        // 动态设置属性
        builder.addPropertyValue("name", "Dynamic Service"); 
        registry.registerBeanDefinition("myService", builder.getBeanDefinition());
    }
}
```

**一些核心的接口、类介绍**：
1.  **Resource**: 用于解决 IoC 容器中的内容从哪里来的问题，也就是配置文件从哪里读取、配置文件如何读取的问题（如 `ClassPathResource`, `FileSystemResource`）。
2.  **BeanDefinition**: 用于解决 Bean 的具体定义问题，包括 Bean 的名字、类型（Class）、Scope、属性值、依赖关系等。它是 Spring 创建 Bean 的“蓝图”，对应于 XML 中的 `<bean>` 标签或 `@Component` 注解。
3.  **BeanFactory**: Spring IoC 容器的基础接口，用于解决在已获取 Bean 定义的情况下，如何装配、获取 Bean 实例的问题。它提供了基础的 getBean 功能，默认采用延迟初始化策略。
4.  **ApplicationContext**: `BeanFactory` 的子接口，对基础功能进行了高级封装，提供了国际化、事件传播、资源加载和 AOP 等企业级功能。它默认在容器启动时预实例化所有单例 Bean。

从 Bean 的角度来看，其整个生命周期如下（以 Singleton 为例）：

```text
实例化 -> 属性赋值 -> 初始化 -> 使用 -> 销毁
   ↓         ↓           ↓        ↓       ↓
new Object setXxx() init-method业务逻辑 destroy
```
