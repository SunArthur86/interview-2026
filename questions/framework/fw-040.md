# 什么是Bean的作用域？

### Bean 的作用域

在 Spring 中，那些组成应用程序的主体及由 Spring IOC 容器所管理的对象，被称之为 Bean。Bean 的定义以及相互间的依赖关系通过配置元数据描述。Spring 容器在创建 Bean 时，不仅可以控制依赖注入，还可以控制该对象的作用域。

**Spring 支持的 5 种作用域**：

1.  **singleton (单例模式)**
    *   **描述**：Spring IoC 容器中只会存在一个共享的 Bean 实例。所有对 Bean 的请求，只要 ID 匹配，都返回同一个实例。
    *   **生命周期**：容器初始化时创建（默认情况），容器销毁时销毁。
    *   **适用场景**：无状态的 Bean（如 Service、DAO）。

2.  **prototype (原型模式)**
    *   **描述**：每次对该 Bean 的请求（注入或 getBean）都会创建一个新的 Bean 实例。
    *   **生命周期**：容器负责创建，但不负责销毁（需要客户端自己管理销毁）。
    *   **适用场景**：有状态的 Bean（如保存用户信息的 View Object）。

3.  **request**
    *   **描述**：每一次 HTTP 请求都会产生一个新的 Bean，该 Bean 仅在当前 HTTP request 内有效。
    *   **适用环境**：仅适用于 WebApplicationContext 环境。

4.  **session**
    *   **描述**：同一个 HTTP Session 共享一个 Bean，不同的 Session 使用不同的 Bean。
    *   **适用环境**：仅适用于 WebApplicationContext 环境。

5.  **global session**
    *   **描述**：在一个全局的 HTTP Session 中共享一个 Bean，一般用于 Portlet 应用环境。
    *   **适用环境**：仅适用于 WebApplicationContext 环境。

**线程安全性**：
Spring 中的单例 Bean 不是线程安全的。如果单例 Bean 中有可变的状态变量，在多线程环境下可能会出现线程安全问题。因此，对于单例 Bean，应尽量保持无状态（即不存储实例变量），或者使用 ThreadLocal/锁机制来保证线程安全。

**3. 实战案例与对比**

*   **实战案例**：开发了一个导出报表的工具类，内部维护了一个 `List` 存储导出数据。最初将其配置为默认的 `singleton`，导致多用户同时导出时，数据相互混淆（用户A看到了用户B的数据）。将 Scope 修改为 `prototype` 后，每次请求获得独立的工具实例，解决了数据隔离问题。

*   **代码示例 (Java - 配置 Prototype)**：
```java
@Bean
@Scope(value = ConfigurableBeanFactory.SCOPE_PROTOTYPE)
public ReportExporter reportExporter() {
    return new ReportExporter(); // 每次注入都会 new 一个新的
}
```

*   **对比表格：Singleton vs Prototype**

| 特性 | Singleton (单例) | Prototype (原型) |
| :--- | :--- | :--- |
| **实例数量** | 容器中仅有一个 | 每次请求创建一个新的 |
| **创建时机** | 容器启动时预加载 (默认) | 获取 Bean 时即时创建 |
| **生命周期管理** | 容器全权管理 (初始化->销毁) | 容器只负责初始化，不负责销毁 |
| **线程安全** | 不安全，需开发者自行处理 | 相对安全 (每个线程独立实例) |
| **性能开销** | 低 (无重复创建开销) | 高 (频繁创建销毁，GC压力大) |
| **适用场景** | 无状态服务、工具类、组件 | 有状态对象、需要隔离数据的对象 |

## 记忆要点

- 五大作用域：singleton（单例）、prototype（多例），另含request/session/global session（Web环境）。
- singleton全局唯一且容器负责完整生命周期，而prototype每次请求新建且容器不管销毁。
- 核心应用判断：无状态用singleton（注意线程不安全），有状态需隔离用prototype。
- 避坑：单例Bean中定义可变成员变量会导致多线程数据污染，需保持无状态或用ThreadLocal。
