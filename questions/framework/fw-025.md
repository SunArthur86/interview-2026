# Mybatis中一级缓存和二级缓存的区别？

### MyBatis 一级缓存和二级缓存的区别

**1. 作用域不同**
*   **一级缓存**：作用域是 `SqlSession` 级别。在同一个 SqlSession 中共享，不同的 SqlSession 之间缓存不互通。
*   **二级缓存**：作用域是 `Mapper` (Namespace) 级别。多个 SqlSession 操作同一个 Mapper 的 SQL 时，可以共享二级缓存。

**2. 缓存存储与共享**
*   一级缓存是 SqlSession 内部的一个 HashMap，生命周期随 SqlSession 的创建和销毁（或提交）。
*   二级缓存存储在全局配置中，跨 SqlSession 存活。

**3. 实际应用场景（与 Spring 整合）**
*   在 MyBatis 与 Spring 整合后，事务通常由 Spring 管理。在同一个事务方法中，SqlSession 是被共享的，一级缓存有效。
*   但如果没有事务或每次查询都新建 SqlSession，一级缓存就会失效。因此，在分布式或多数据源场景下，二级缓存显得更有用，但需注意脏读风险。

**实战案例**：
在微服务架构下，服务通常是无状态的，一级缓存仅在单次请求的事务范围内有效，利用率较低。为了提升高并发下的查询性能，通常会集成 Redis 替代 MyBatis 原生的二级缓存，因为原生二级缓存是进程内的，无法在集群间共享且会导致内存溢出风险。

**对比表格**：

| 特性 | 一级缓存 (First Level Cache) | 二级缓存 (Second Level Cache) |
| :--- | :--- | :--- |
| **作用域** | `SqlSession` (会话级) | `Mapper Namespace` (命名空间级) |
| **默认状态** | 默认开启，无法关闭 | 默认关闭，需配置开启 |
| **存储结构** | `BaseExecutor` 内的 `HashMap` | `PerpetualCache` (可序列化至磁盘/Redis) |
| **共享性** | 仅当前会话共享 | 跨会话、跨线程共享（同一Mapper） |
| **失效触发** | `commit`, `rollback`, `close`, `clearCache` | `commit`, `rollback`, 或配置的 `flushInterval` |
| **Spring整合影响** | 仅在 `@Transactional` 内有效 | 依然有效，但需解决分布式一致性问题 |

**代码示例**：
```java
// 模拟不同 Session 下的缓存行为
SqlSession session1 = sqlSessionFactory.openSession();
List<User> users1 = session1.selectList("com.example.mapper.UserMapper.selectAll");
// 触发一级缓存，第二次查询不走DB
List<User> users1_retry = session1.selectList("com.example.mapper.UserMapper.selectAll");
session1.commit(); // 释放 session1，一级缓存销毁，但写入二级缓存（如果开启）

SqlSession session2 = sqlSessionFactory.openSession();
// 如果开启了二级缓存，此处直接从 L2 读取，不查询 DB
List<User> users2 = session2.selectList("com.example.mapper.UserMapper.selectAll");
session2.close();
```

## 技术原理

MyBatis 的两级缓存体现了**作用域递进 + 装饰器模式**的设计哲学：一级缓存解决"同一会话内重复查询"，二级缓存解决"跨会话的查询复用"，二级通过装饰一级的 Executor 实现。

- **一级缓存的实现——BaseExecutor 的本地 HashMap**：每个 `SqlSession` 持有一个 `Executor`（默认 `SimpleExecutor` 的父类 `BaseExecutor`），`BaseExecutor` 内部维护一个 `PerpetualCache`（本质是 HashMap）。查询时先以 `MappedStatement + 参数 + 分页` 为 key 查 HashMap，命中则直接返回，未命中才查 DB 并写入缓存。`update/insert/delete/commit/rollback` 都会调用 `clearLocalCache()` 清空一级缓存，避免脏读。一级缓存无法关闭（除非把 `localCacheScope=STATEMENT`，每次查询后立即清空）。
- **二级缓存的实现——CachingExecutor 装饰器**：二级缓存通过 `CachingExecutor` 装饰 `BaseExecutor` 实现（装饰器模式）。查询时先查二级缓存（按 Namespace 共享），未命中再委托给内部的 `BaseExecutor`（走一级缓存→DB）。**关键：数据只有 `commit` 后才会从一级缓存晋升到二级缓存**，所以未提交事务的查询不会污染二级缓存。二级缓存默认关闭，需在 Mapper 上加 `<cache/>` 开启。
- **Spring 整合下的一级缓存坑**：Spring 每次事务方法内复用同一个 `SqlSession`（通过 `SqlSessionTemplate` 的代理），所以 `@Transactional` 内的多次查询能命中一级缓存。但**没有事务时每次查询都新建 SqlSession**，一级缓存命中率为零。这就是为什么 Spring 环境下"一级缓存形同虚设"的根因。
- **跨 Namespace 脏读的本质**：二级缓存按 Namespace 隔离，如果一个 Mapper 的查询 join 了另一个 Mapper 的表（多表联查），更新第二个 Mapper 时不会清掉第一个 Mapper 的二级缓存，导致脏读。这是二级缓存最大设计缺陷，也是生产环境慎用的主因。

## 代码示例

```java
// 1. 一级缓存演示：同 SqlSession 内重复查询只打一次 DB
SqlSession session = sqlSessionFactory.openSession();
UserMapper mapper = session.getMapper(UserMapper.class);
User u1 = mapper.selectById(1L);   // 查 DB，写入一级缓存
User u2 = mapper.selectById(1L);   // 命中一级缓存，不打 DB
assert u1 == u2;                    // 同一对象引用（一级缓存返回的是同一实例）

// update 会清空一级缓存
mapper.updateName(1L, "new");
User u3 = mapper.selectById(1L);    // 缓存已清，再次查 DB
session.commit();
```

```xml
<!-- 2. 开启二级缓存：在 Mapper XML 加 <cache/> -->
<mapper namespace="com.example.mapper.UserMapper">
    <!-- 开启二级缓存（默认 LRU，1024 个对象，刷新间隔无） -->
    <cache eviction="LRU" flushInterval="60000" size="512" readOnly="false"/>

    <select id="selectById" resultType="User" useCache="true">
        SELECT * FROM user WHERE id = #{id}
    </select>

    <!-- 某些频繁变动查询可禁用二级缓存 -->
    <select id="countAll" resultType="int" useCache="false">
        SELECT COUNT(*) FROM user
    </select>
</mapper>
```

```java
// 3. 跨 SqlSession 验证二级缓存
SqlSession s1 = sqlSessionFactory.openSession();
s1.getMapper(UserMapper.class).selectById(1L);   // 查 DB
s1.commit();   // 关键：commit 后才晋升到二级缓存
s1.close();

SqlSession s2 = sqlSessionFactory.openSession();
s2.getMapper(UserMapper.class).selectById(1L);   // 命中二级缓存，不查 DB
s2.close();
```

```java
// 4. 生产推荐：用 Redis 替代 MyBatis 原生二级缓存（解决集群 + 脏读）
// 引入 mybatis-redis 依赖后，配置自定义 Cache 实现
public class RedisCache implements org.apache.ibatis.cache.Cache {
    private final String id;   // Namespace
    private final RedisTemplate<String, Object> redis;

    @Override
    public Object getObject(Object key) {
        return redis.opsForValue().get(key.toString());
    }

    @Override
    public void putObject(Object key, Object value) {
        redis.opsForValue().set(key.toString(), value,
            Duration.ofMinutes(30));   // TTL 防止内存溢出
    }

    @Override
    public void clear() {
        // 按 Namespace 删除所有 key（用前缀扫描）
        redis.delete(redis.keys(id + ":*"));
    }
}
```

## 对比选型

| 维度 | MyBatis 一级缓存 | MyBatis 二级缓存 | Spring Cache + Redis |
| :--- | :--- | :--- | :--- |
| **作用域** | SqlSession（会话级） | Namespace（跨会话） | 全局（跨实例） |
| **集群共享** | 否（单实例） | 否（进程内） | 是（Redis 集群） |
| **默认状态** | 开启（无法关） | 关闭（需 `<cache/>`） | 按注解配置 |
| **脏读风险** | 低（事务内一致） | 高（跨 Namespace 多表） | 中（依赖 TTL/主动失效） |
| **内存控制** | SqlSession 销毁即释放 | 易内存溢出（无 TTL） | Redis 自主管理 |
| **适用场景** | 单事务内去重 | 单实例读多写少 | 微服务集群生产环境 |

## 常见坑

- **Spring 无事务时一级缓存失效**：没有 `@Transactional` 时每次查询新建 SqlSession，一级缓存命中率为零。需要重复查询去重的场景必须包在事务里。
- **二级缓存跨 Namespace 脏读**：多表联查（A Mapper join B 表），更新 B Mapper 不会清 A 的二级缓存。对策：用 `<cache-ref>` 让多个 Mapper 共用一个缓存空间（粒度变粗），或干脆禁用二级缓存改用 Redis。
- **二级缓存对象需可序列化**：`readOnly=false`（默认）要求缓存对象实现 `Serializable`，否则反序列化失败。实体类漏标 `Serializable` 是常见启动报错。
- **分布式环境绝对不要用原生二级缓存**：多实例间缓存不共享，且每个实例都可能持有过期数据。生产环境用 Redis/Caffeine 集群方案替代。
- **`flushInterval` 不能保证精准过期**：二级缓存的 `flushInterval` 是被动检查（查询时判断是否超时），不是定时器主动清理。强时效场景必须靠业务主动 `clearCache`。
- **`readOnly=true` 性能高但不安全**：返回缓存对象同一实例，调用方修改会污染缓存。只在"只读且不变"的字典数据上用。
