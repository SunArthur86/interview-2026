# Spring循环依赖及解决方式是什么？

### Spring 循环依赖及解决方式

**1. 什么是循环依赖**
循环依赖就是循环引用，即两个或两个以上的 Bean 互相持有对方，最终形成闭环（例如 A 依赖 B，B 依赖 A）。注意：这里指的是对象属性的相互依赖，而不是构造函数的循环调用。

**2. Spring 解决循环依赖的机制**
Spring 能够解决**单例 Bean** 的**Setter 注入**循环依赖，但无法解决**构造器注入**的循环依赖。
其解决方式的核心在于**三级缓存**：

*   **一级缓存 (singletonObjects)**：存放已经完全初始化好的 Bean。
*   **二级缓存 (earlySingletonObjects)**：存放提前曝光的（仅实例化但未填充属性）的 Bean，主要用于解决循环依赖。
*   **三级缓存 (singletonFactories)**：存放 Bean 工厂对象（ObjectFactory），用于提前暴露 Bean 的引用（通常用于处理 AOP 代理）。

**解决流程**：
1.  A 创建过程中，先将工厂对象放入三级缓存。
2.  A 注入属性时发现需要 B，开始创建 B。
3.  B 创建过程中，注入属性时发现需要 A。
4.  B 去缓存中找 A：先查一级缓存（无），再查二级缓存（无），最后查三级缓存。
5.  在三级缓存中找到 A 的工厂，调用 getObject() 方法拿到 A 的早期对象（如果是 AOP 代理，此时会返回代理对象），并将其放入二级缓存，同时删除三级缓存中的 A。
6.  B 拿到 A 的引用，完成初始化，放入一级缓存。
7.  A 继续执行，拿到已初始化好的 B，完成初始化，放入一级缓存。

#### 实战案例
开发“订单服务”依赖“用户服务”，而“用户服务”为了发通知又反向依赖“订单服务”的场景。由于 Spring 默认支持单例 Setter 注入的循环依赖，程序能正常启动。但后来引入 `@Async` 注解到用户服务上，导致 Bean 被提前代理，虽然三级缓存能处理 AOP 代理，但如果是构造器注入则会直接报错 `BeanCurrentlyInCreationException`。解决方法是将构造器注入改为 `@Lazy` 注解注入或 Setter 注入。

#### 代码示例（模拟循环依赖检测）
```java
// 模拟 Spring 的循环依赖检测逻辑
public class CircularDependencyChecker {
    private final ThreadLocal<Set<String>> beansBeingCreated = ThreadLocal.withInitial(HashSet::new);

    public void beforeCreation(String beanName) {
        if (beansBeingCreated.get().contains(beanName)) {
            throw new BeanCurrentlyInCreationException(beanName);
        }
        beansBeingCreated.get().add(beanName);
    }

    public void afterCreation(String beanName) {
        beansBeingCreated.get().remove(beanName);
    }
}
```

#### 循环依赖场景与解决对比
| 依赖场景 | Spring 能否解决 | 解决方案/原理 |
| :--- | :--- | :--- |
| **单例 Setter 注入** | ✅ 能 | 三级缓存机制（提前暴露引用） |
| **单例 构造器注入** | ❌ 否 | 无法在实例化前获取引用，需重构代码或使用 `@Lazy` |
| **原型 Prototype** | ❌ 否 | Spring 不缓存 Prototype Bean，无法处理引用 |

## 记忆要点

- 前提：仅支持单例Bean的Setter注入循环依赖，构造器注入必然报错。
- 三级缓存巧解循环：一级存成品，二级存半成品，三级存工厂提前暴露引用。
- 核心解决流程：B注入A时，发现三级缓存有A的工厂，拿到早期A放二级缓存，顺利完成B。
- 注意：引入AOP代理时，三级缓存的工厂会提前生成并返回代理对象。
