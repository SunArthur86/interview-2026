# BeanFactory和ApplicationContext有什么区别？

【基本概念】
- **BeanFactory**：Spring IoC 容器的**顶层接口**，提供基础的 IoC 容器功能（如 `getBean`, `containsBean`），采用**懒加载**机制——即只有在第一次调用 `getBean()` 获取某个 Bean 时，才真正实例化和初始化该 Bean。适用于资源受限的环境。
- **ApplicationContext**：继承自 BeanFactory，是**高级容器**，不仅包含 BeanFactory 的所有功能，还扩展了大量企业级特性，通常在应用启动时**预加载**并初始化所有单例 Bean（可通过 `@Lazy` 改变行为）。

【ApplicationContext 的核心扩展功能】
1.  **国际化支持**：继承 `MessageSource`，支持消息解析和国际化（i18n）。
2.  **事件发布机制**：继承 `ApplicationEventPublisher`，通过简单的发布/订阅模式处理容器事件（如 `ContextRefreshedEvent`）。
3.  **资源加载**：继承 `ResourceLoader`，支持统一的资源访问方式（如 `classpath:`, `file:`, `url:`）。
4.  **AOP 与生命周期管理**：更方便的 AOP 代理集成和 Bean 生命周期回调。
5.  **环境抽象**：集成 `Environment`，支持 Profile（如 dev, prod）和属性源配置。

【BeanFactory 与 ApplicationContext 初始化流程对比】

```text
BeanFactory (Lazy Initialization)
┌─────────────────┐
│ 容器启动         │ → 读取配置 → 解析 BeanDefinition → 不实例化 Bean
└────────┬────────┘
         │
         ▼ 第一次 getBean()
┌─────────────────┐
│ 实例化 Bean      │ → 属性注入 → 初始化 → 返回 Bean
└─────────────────┘

---------------------------------------

ApplicationContext (Eager Initialization)
┌─────────────────┐
│ 容器启动         │ → 读取配置 → 解析 BeanDefinition
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 实例化所有单例   │ → 属性注入 → 初始化 → 完成启动
└────────┬────────┘
         │
         ▼ getBean()
┌─────────────────┐
│ 直接从缓存获取   │
└─────────────────┘
```

**#### 实战案例**
在实现系统启动后预热缓存的需求时，我们监听了 ApplicationContext 发布的 `ContextRefreshedEvent` 事件。利用 Eager Initialization 特性，确保所有 Singleton Bean（包括缓存加载器）已就绪，避免了懒加载可能导致的首个请求超时。

**#### 代码示例**
```java
// 监听容器刷新完成事件，执行初始化逻辑
@Component
public class ApplicationStartupRunner implements ApplicationListener<ContextRefreshedEvent> {
    @Override
    public void onApplicationEvent(ContextRefreshedEvent event) {
        // 防止重复执行（Root ApplicationContext 刷新和子容器刷新都会触发）
        if (event.getApplicationContext().getParent() == null) {
            System.out.println("System fully started, loading cache...");
            // 执行预热逻辑
        }
    }
}
```

【常用实现类】
- **ClassPathXmlApplicationContext**：从类路径加载 XML 配置文件。
- **FileSystemXmlApplicationContext**：从文件系统加载 XML 配置文件。
- **AnnotationConfigApplicationContext**：基于注解的配置，常用于纯 Java Config 启动。
- **WebApplicationContext**：Web 专用容器，如 `XmlWebApplicationContext`，能从 `ServletContext` 获取配置。
- **Spring Boot 默认**：`AnnotationConfigServletWebServerApplicationContext`，支持 Servlet Web 环境。

【## 常见考点】
1. **为什么说 ApplicationContext 是 BeanFactory 的扩展？** ApplicationContext 是一种复合模式的 BeanFactory，它内部持有一个 `DefaultListableBeanFactory` 实例（真正的底层实现），并在此之上包装了资源加载、事件发布等高级功能。源码中 `getBean()` 方法通常直接委托给内部的 BeanFactory 执行。
2. **懒加载 vs 预加载的优缺点？** 预加载（ApplicationContext）可以在启动时就发现配置错误，避免运行时崩溃，但启动时间较长；懒加载启动快，但首次请求响应

## 记忆要点

- 定位不同：BeanFactory是基础顶层容器，而ApplicationContext是扩展的高级容器。
- 加载机制：BeanFactory采用懒加载，而ApplicationContext在启动时预加载单例Bean。
- 高级扩展：ApplicationContext额外集成了国际化、资源加载和事件发布机制。
- 底层组合：ApplicationContext内部其实持有一个DefaultListableBeanFactory实现功能。
