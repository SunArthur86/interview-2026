# 什么是5种不同方式的自动装配？

### 什么是5种不同方式的自动装配？

Spring 装配包括手动装配和自动装配，手动装配是基于 XML 装配、构造方法、setter 方法等。自动装配有五种方式，用于指导 Spring 容器自动进行依赖注入（注意：在现代 Spring 开发中，XML 的 `autowire` 属性已较少使用，更多使用 `@Autowired` 注解，但理解底层机制对面试很重要）。

1.  **no** (默认)
    *   **机制**：默认的方式是不进行自动装配，必须通过显式设置 `ref` 属性（XML）或使用 Java Config/注解来进行装配。

2.  **byName**
    *   **机制**：通过参数名自动装配。Spring 容器查看 Bean 的属性名，在容器中查找是否存在与该属性名 `id` 一致的 Bean，如果找到则注入。
    *   **边界条件**：如果找不到匹配的 Bean，则该属性保持为 null，不会报错。

3.  **byType**
    *   **机制**：通过参数类型自动装配。Spring 容器查找类型与该属性类型一致的 Bean。
    *   **边界条件**：
        *   如果容器中只有一个该类型的 Bean，则注入成功。
        *   **如果有多个 Bean 符合该类型**，则会抛出 `NoUniqueBeanDefinitionException` 异常。
        *   如果没有找到，则该属性为 null。

4.  **constructor**
    *   **机制**：这个方式类似于 byType，但是应用于构造函数参数。Spring 容器会寻找与构造函数参数类型匹配的 Bean 来进行注入。
    *   **边界条件**：如果没有确定的带参数的构造器，或者参数类型无法匹配，将会抛出异常。这在隐式注入中使用较少，通常配合构造器注入显式使用。

5.  **autodetect** (已过时)
    *   **机制**：首先尝试使用 constructor 来自动装配，如果无法工作（例如没有合适的构造器），则使用 byType 方式。
    *   **注意**：在 Spring 3.x 之后已被废弃，建议明确指定具体的注入方式。

**流程对比（byName vs byType）**：

```text
┌─────────────────────────────────────────────────────────────────┐
│ byName 模式                                                     │
├─────────────────────────────────────────────────────────────────┤
│  Bean A 属性名: userService                                     │
│      │                                                          │
│      ▼                                                          │
│  容器查找 id="userService" 的 Bean ──▶ 找到则注入                │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ byType 模式                                                     │
├─────────────────────────────────────────────────────────────────┤
│  Bean A 属性类型: UserService                                   │
│      │                                                          │
│      ▼                                                          │
│  容器查找类型为 UserService 的 Bean ──▶ 唯一则注入               │
│                                多个则报错 (NoUniqueBean...)     │
└─────────────────────────────────────────────────────────────────┘
```

**实战案例**：
在升级第三方 SDK 版本时，若新包中存在多个实现类（如 `NewDataSource` 和 `OldDataSource` 都实现了 `DataSource` 接口），使用 `byType` 自动装配会导致启动报 `NoUniqueBeanDefinitionException`。解决方法是使用 `@Primary` 注解指定优先级，或使用 `@Qualifier("beanName")` 明确指定名称（即退化为 byName 逻辑）。

**代码示例（@Qualifier 解决 byType 冲突）**：
```java
@Service
public class OrderService {
    private final PaymentService paymentService;

    // 实战：当容器中有多个 PaymentService 实现时（如 AlipayService, WechatPayService）
    // 通过 @Qualifier 指定注入名为 "alipayService" 的 Bean
    public OrderService(@Qualifier("alipayService") PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}
```


## 记忆要点

- byName与byType对比：byName按属性名找Bean，byType按属性类型找Bean，遇到多实现会报错。
- 注入冲突解决：byType遇到多个同类型Bean时，用@Primary设优先级或@Qualifier指定具体名称。
- 默认与过时：no代表默认不自动装配；autodetect（先构造后类型）在Spring3.x后已被废弃。
