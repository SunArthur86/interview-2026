# Spring Cloud的核心组件有哪些？

【Spring Cloud Netflix (第一代，大部分已停止维护/进入维护模式)】
- **Eureka**：服务注册与发现中心。遵循 **AP** 原则（可用性+分区容错性），保证高可用，但可能出现短暂的数据不一致。
- **Hystrix**：熔断器。提供熔断、降级、隔离功能，防止雪崩效应。**状态**：已停止维护，不再推荐用于新项目。
- **Zuul 1.x**：API 网关。基于 Servlet 阻塞 IO，性能有限。**状态**：已进入维护模式，被 Spring Cloud Gateway 取代。
- **Ribbon**：客户端负载均衡。**状态**：已被维护模式，Spring Cloud LoadBalancer 是其替代品。
- **Config**：分布式配置中心，支持 Git 仓库。

【Spring Cloud Alibaba (国内主流推荐)】
- **Nacos**：集**服务注册发现**与**分布式配置**于一体的平台。支持 AP/CP 模式切换，控制台友好。
- **Sentinel**：流量卫兵。替代 Hystrix，提供实时监控、熔断降级、系统负载保护。
- **Seata**：分布式事务解决方案。支持 AT、TCC、Saga、XA 等模式。
- **Dubbo**：高性能 RPC 框架。Spring Cloud Alibaba 生态完美整合了 Dubbo（替代 OpenFeign 做高性能内部调用）。

【Spring Cloud 通用核心组件 (第二代/官方维护)】
- **Spring Cloud Gateway**：基于 WebFlux (Reactor 模型) 的响应式网关，性能优于 Zuul，支持动态路由。
- **OpenFeign**：声明式 HTTP 客户端（整合了 Ribbon），简化服务间调用。
- **Spring Cloud LoadBalancer**：官方推出的客户端负载均衡器，用于替代 Ribbon。
- **Spring Cloud Sleuth + Zipkin/Prometheus**：链路追踪与监控。

【实战案例】
在电商大促场景中，我们曾遭遇「配置热更新」导致的雪崩。旧版 Spring Cloud Config 需要配合 `@RefreshScope`，由于是破坏性重建 Bean，高并发下重建瞬间会拒绝请求。后来迁移至 **Nacos Config**，利用其配置监听的 `Listener` 机制，实现了字段级别的热更新（无需重启 Bean），极大提升了系统的平滑度。

【核心组件选型对比表】

| 功能组件 | Netflix (老一代) | Alibaba (国内主流) | Spring Cloud (官方) | 选型建议 |
| :--- | :--- | :--- | :--- | :--- |
| **注册中心** | Eureka (AP) | **Nacos** (AP/CP切换) | Consul (CP) | 首选 Nacos（控制台强、维护好）
| **配置中心** | Spring Cloud Config | **Nacos Config** | Spring Cloud Config | 首选 Nacos（支持动态刷新、同源管理）
| **服务调用** | Feign + Ribbon | Feign + LoadBalancer / **Dubbo** | OpenFeign + LoadBalancer | 内部交互选 Dubbo（高性能），外部选 Feign
| **熔断限流** | Hystrix (停止维护) | **Sentinel** | Resilience4j | 首选 Sentinel（控制台可视化、规则丰富）
| **网关** | Zuul 1.x (阻塞) | - | **Spring Cloud Gateway** (响应式) | 首选 Gateway（性能高、非阻塞） |

## 记忆要点

- 注册配置：Nacos替代Eureka与Config，支持AP/CP切换，实现注册与配置一体化。
- 流量防卫：Sentinel替代Hystrix，提供强大多维度限流、熔断降级与系统保护。
- 网关路由：Spring Cloud Gateway替代Zuul，基于WebFlux提供高性能响应式网关。
- 通信与事务：OpenFeign负责声明式RPC调用，而Seata负责解决跨服务分布式事务。
