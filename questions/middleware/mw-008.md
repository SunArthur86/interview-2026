# Kafka中Partition是如何在Broker之间负载均衡的？

Kafka 中 Partition 的负载均衡涉及三个层面：**分区分配（创建/扩容时）**、**副本迁移**与**消息路由（生产/消费时）**。

### 1. 分区在 Broker 间的分配（集群层负载均衡）
- **创建时的自动分配**：
  - 创建 Topic 时，Kafka 控制器负责执行分配算法。
  - **副本分布原则**：同一个 Partition 的不同副本（Leader 和 Follower）必须分布在不同的 Broker 上（机架感知开启时需跨机架），以保证容错性。
  - **Leader 均衡**：算法会尽量确保所有 Broker 承载的 Leader 分区数量大致相等，以均衡读写 IO 压力（因为 Leader 处理读写，Follower 仅处理读）。
- **副本迁移**：
  - 当集群扩容或 Broker 宕机时，可通过 `kafka-reassign-partitions.sh` 工具手动触发负载均衡，将分区副本迁移到新 Broker 上。

### 2. Producer 发送消息时的负载均衡（应用层路由）
- **分区器 逻辑**：Producer 通过 `Partitioner` 接口决定消息发往哪个分区。
- **路由策略**：
  - **指定 Partition**：代码直接指定，不经过计算。
  - **Key Hash**：`Utils.murmur2(key) % partitionCount`。相同 Key 的消息进入同一分区，保证顺序性。
  - **Sticky Partitioner（粘性分区，默认）**：无 Key 时，随机选择一个分区并尽可能填满该分区的 Batch（默认 16KB 或 linger.ms 时间到），Batch 满后再随机切换。这比早期的 Round-Robin（每条消息换分区）减少了请求次数，极大提升了吞吐量。
- **增强机制**：支持自定义分区器实现，可根据业务字段（如用户 ID、地区）进行定制化路由。

### 3. Consumer 的负载均衡（消费层）
- **Rebalance 机制**：Consumer Group 内的消费者通过组协调器和心跳机制，自动订阅 Topic 的所有 Partition。
- **分配策略**：
  - **Range（范围）**：按分区序号范围分配（默认），可能导致不均。
  - **RoundRobin（轮询）**：轮询分配，尽量均匀。
  - **Sticky（粘性）**：Rebalance 时尽量保留原有分配关系，减少不必要的分区移动。
  - **CooperativeSticky（渐进式）**：平滑 Rebalance，不会停止所有消费。
- **原则**：一个分区在同一时刻只能被组内的一个消费者消费。

### 实战案例
某电商大促期间，新扩容的 Broker 节点没有流量，导致旧节点磁盘写满。这是因为新分区不会自动迁移到新节点，且**分区 Leader 倾向性**未重新计算，必须手动执行 `kafka-preferred-replica-election.sh` 或开启自动 Leader 平衡才能利用新节点 IO 能力。

### 代码示例
Java Producer 自定义分区器实现按区域路由：

```java
public class RegionPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes, Object value, byte[] valueBytes, Cluster cluster) {
        String region = (String) value; // 假设value包含区域信息
        List<PartitionInfo> partitions = cluster.partitionsForTopic(topic);
        // 简单逻辑：北京去偶数分区，上海去奇数分区
        return "Beijing".equals(region) ? 0 : 1;
    }
}
```

### 对比表格：Producer 路由策略对比

| 策略 | 适用场景 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **Key Hash** | 需要保序（如同一用户订单） | 顺序性强 | 负载可能不均（热门Key倾斜） |
| **Round Robin** | 追求最大吞吐量 | 负载最均衡 | 无法保证顺序 |
| **Sticky (默认)** | 高吞吐通用场景 | 批次发送效率高，吞吐接近RR | 短时间内可能出现负载不均 |
