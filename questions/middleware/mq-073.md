# RocketMQ 和 Kafka 的应用

RocketMQ 和 Kafka 在底层 I/O 上都利用了操作系统的零拷贝技术和顺序写特性，但具体实现路径和优化策略有所不同。

### RocketMQ 的存储与 I/O

**1. 存储架构：CommitLog + ConsumeQueue**
RocketMQ 采用混合存储模式。所有 Topic 的消息顺序写入同一个物理文件 `CommitLog`，从而保证极致的写入性能（顺序写）。同时，构建逻辑队列 `ConsumeQueue` 作为索引，记录消息在 CommitLog 中的物理偏移量。

```text
┌─────────────────────────────────────────────┐
│              Broker 存储层                   │
├─────────────────────────────────────────────┤
│  ┌─────────────┐      ┌──────────────────┐  │
│  │ ConsumeQueue│ ───▶ │   CommitLog      │  │
│  │  (逻辑索引)  │ 偏移量 │  (物理数据日志)    │  │
│  │   (mmap)    │      │    (mmap)        │  │
│  └─────────────┘      └──────────────────┘  │
└─────────────────────────────────────────────┘
```

**2. 内存映射与读写**
- **mmap 的应用**：CommitLog 和 ConsumeQueue 均使用了 `mmap`（内存映射），将文件映射到虚拟内存，减少数据在内核态与用户态之间的拷贝。
- **读取特性**：虽然对于特定 Topic 的消息，在 CommitLog 中是离散存储的（随机读），但从整体时间维度看，消费者总是消费旧消息到新消息，磁盘读取表现为“顺序追赶”模式。且利用 OS 的 Page Cache，热数据通常在内存中，无需物理读盘。

**3. 传输模型（非完全零拷贝）**
- **发送消息**：默认使用 `transferMsgByHeap = true`，数据先拷贝到堆内内存，再写入 Socket。这意味着不是真正的零拷贝。
- **拉取消息**：默认也是通过 mappedBuffer 读取。虽然可以通过配置优化，但通常仍需经过一次 Page Cache 到 Socket Buffer 的拷贝，未使用 `sendfile`。

**4. 关键优化：文件预热**
由于 `mmap` 在访问时才触发缺页中断加载数据（惰性加载），这会造成瞬间延迟毛刺。RocketMQ 通过 `warmMappedFile` 进行预热：
1. 写入 0 字节强制触发缺页中断，将数据锁定在物理内存。
2. 调用 `mlock` 系统调用，防止内存被 Swap 交换出去。
3. 调用 `madvise(MADV_WILLNEED)`，建议操作系统预读后续数据。

### Kafka 的存储与 I/O

**1. 存储架构：Partition 分片**
Kafka 每个分区对应一个物理日志文件。这种方式在数据迁移和扩容上更灵活，但如果分区数过多，会在多个文件间频繁切换写入，从全局看可能退化为准随机写。

**2. 零拷贝应用**
- **日志文件**：使用 `FileChannel.transferTo`，底层调用 `sendfile` 系统调用。数据直接从文件系统缓存拷贝到网卡缓冲区，全程 2 次拷贝（且无需 CPU 参与，DMA 拷贝），不经过用户态内存。
- **索引文件**：使用 `mmap`。

**3. Swap 策略**
Kafka 没有像 RocketMQ 那样显式调用 `mlock`，而是依赖 Linux 参数 `vm.swappiness`。建议设置为 1（而非 0），避免内存耗尽时 OOM（Out Of Memory）杀死进程，保留一点 Swap 空间作为应急缓冲，给运维监控留出反应时间。

### 实战案例与代码

#### 实战案例
**RocketMQ 踩坑**：在处理超大消息（接近 `CommitLog` 默认 1G 限制）时，若未开启内存预热，消费者拉取该消息会触发严重的缺页中断（Page Fault），导致瞬时的 RT（响应时间）飙升甚至超时。
**Kafka 优化**：在进行大规模数据迁移（如 MirrorMaker）时，Kafka 的 `sendfile` 优势明显，CPU 负载几乎维持在低位，而若采用传统的用户态转发方式，CPU 早已打满。

#### 代码示例 (Java Mmap 限制)
```java
// RocketMQ 中 MappedFile 创建映射的关键逻辑
// 注意：mmap 映射的大小受限，通常需分段映射（如 RocketMQ 默认 1G 一个文件）
FileChannel fileChannel = new RandomAccessFile(file, "rw").getChannel();
MappedByteBuffer mappedBuffer = fileChannel.map(FileChannel.MapMode.READ_WRITE, 0, fileSize);

// 预热示例：写入 0 触发缺页中断
mappedBuffer.put(0, (byte) 0);
```

### 常见考点
1.  **mmap vs sendfile**：RocketMQ 为何多用 mmap 而 Kafka 多用 sendfile？（RocketMQ 需要在内存中构建索引，频繁读写，mmap 更灵活；Kafka 主要是纯数据管道传输，sendfile 效率最高）。
2.  **Page Cache 的影响**：什么是“缺页中断”？为什么 RocketMQ 需要文件预热而 Kafka 不太需要？（RocketMQ 对低延迟毛刺敏感）。
3.  **对比表格**：

| 特性 | RocketMQ | Kafka |
| :--- | :--- | :--- |
| **核心 I/O 技术** | Mmap (读写) | Sendfile (消费) + Mmap (索引) |
| **数据是否进用户态** | 是 (进堆外内存) | 否 (直接内核态 -> 网卡) |
| **写入模式** | 全局顺序写 | 单 Partition 顺序写 |
| **内存锁定** | 强依赖 mlock | 依赖 vm.swappiness |
| **适用业务** | 低延迟、事务、复杂过滤 | 高吞吐、日志、流计算 |
