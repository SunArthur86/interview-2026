# Broker 的存储

Broker 的存储架构是其高性能的核心，主要涉及本地文件系统。RocketMQ 采用混合型存储策略，将所有 Topic 的消息体存储在同一个文件（CommitLog）中，以保证写入的极致性能。

主要包含三种核心文件：

### 1. CommitLog（消息物理存储文件）
- **作用**：存储所有 Topic 的消息原始数据。
- **存储方式**：顺序写入。无论消息属于哪个 Topic，都追加写入到同一个文件。
- **文件结构**：
  - 单个文件默认大小 **1G**。
  - 文件名以文件起始偏移量命名，固定 20 位数字，不足前面补 0。例如 `00000000000000000000`，下一个文件为 `00000000001073741824`（即 1G 的位置）。
  - 这种命名方式使得可以通过消息的物理偏移量直接计算所在的文件，定位速度 O(1)。

### 2. ConsumeQueue（逻辑消费队列）
- **作用**：作为 CommitLog 的索引文件，类似数据库的索引表，便于消息消费。
- **结构**：
  - 每个 ConsumeQueue 对应一个 Topic 下的一个 MessageQueue。
  - 条目大小固定 **20 字节**，包含三部分：
    1.  **8 字节**：CommitLog 物理偏移量（指向该消息在 CommitLog 中的位置）。
    2.  **4 字节**：消息大小（消息体长度）。
    3.  **8 字节**：Tags HashCode（用于过滤消息）。
- **文件大小**：约 5.72M（30 万条记录 * 20 字节）。

### 3. IndexFile（索引文件）
- **作用**：提供基于 Key 或时间区间的快速查询能力（如通过 MessageID 查询，或业务 Key 查询）。
- **结构**：
  - 文件名以创建时间戳命名。
  - 单个文件固定大小约 **400M**，包含约 **2000 万** 个索引。
  - 采用类似 Hash 索引的结构。

### 实战案例
某日志系统某天因 Bug 写入大量空消息，导致 CommitLog 增长极快但 ConsumeQueue 增长正常，且 CPU 负载升高。排查发现是 ReputMessageService 异步分发线程被阻塞，导致堆积在 CommitLog 中无法被消费索引。

### 代码示例 (Java)
```java
// ConsumeQueue 存储单元结构
public static final int CQ_STORE_UNIT_SIZE = 20;

public void putMessagePositionInfo(DispatchRequest dispatchRequest) {
    // 8字节偏移量 + 4字节大小 + 8字节tagsCode
    this.byteBufferIndex.putLong(dispatchRequest.getCommitLogOffset());
    this.byteBufferIndex.putInt(dispatchRequest.getMsgSize());
    this.byteBufferIndex.putLong(dispatchRequest.getTagsCode());
}
```

### 对比表格：三种核心文件对比

| 文件类型 | 存储内容 | 写入特性 | 读取特性 | 文件大小 |
| :--- | :--- | :--- | :--- | :--- |
| **CommitLog** | 全部消息实体（Topic 混合） | 顺序写（极高吞吐） | 随机读（较少直接读） | 1G (默认) |
| **ConsumeQueue** | 索引信息（偏移量、大小、Tag） | 顺序写（异步分发） | 顺序读（消费时使用） | ~5.72M (30万条) |
| **IndexFile** | 业务 Key 与物理偏移映射 | 随机写（构建索引） | Hash 查询 | ~400M |

### 数据生成流程
消息到达 Broker 后，处理流程如下：

1.  **写入 CommitLog**：消息首先被顺序写入 CommitLog 文件（此时依赖 PageCache）。
2.  **异步分发**：有一个后台线程 `ReputMessageService`，它会实时地扫描 CommitLog，将新写入的消息解析并分发到对应的 ConsumeQueue 和 IndexFile 中。

```text
              Producer
                 │
                 ▼
┌───────────────────────────────────────┐
│              CommitLog                 │
│  [Msg1(TopicA)][Msg2(TopicB)]...      │ ◄── 顺序写入，高性能
│  (混合存储所有 Topic 的消息体)          │
└───────────────┬───────────────────────┘
                │
                │ ReputMessageService (异步线程)
                ▼
     ┌──────────┴──────────┐
     ▼                     ▼
┌───────────┐        ┌───────────┐
│ConsumeQueue│        │ IndexFile │
│ (索引信息) │        │ (业务索引) │
└─────┬─────┘        └─────┬─────┘
      │                    │
      ▼                    ▼
   Consumer           业务查询
```

## 常见考点
1. **为什么 RocketMQ 读写性能这么快？**
   答：1) CommitLog 顺序写，利用磁盘顺序写速度远高于随机写的特性；2) 利用 OS 的 PageCache 进行读写；3) ConsumeQueue 读取的是固定大小的索引，利用 OS 预读机制。
2. **RocketMQ 的消息清理机制？**
   答：默认凌晨 4 点执行，删除超过 72 小时的 CommitLog 文件，且如果磁盘空间使用率达到阈值（如 85%），会强制删除，不关注时间。
3. **ReputMessageService 是如何保证不丢消息的？**
   答：CommitLog 写入成功后才进行分发，即使分发线程挂了，Broker 重启时会通过 CommitLog 的物理偏移量和 ConsumeQueue 的最大偏移量对比，回溯并重新构建未分发的索引。
