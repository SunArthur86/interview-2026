# RocketMQ 事务消息

### RocketMQ 事务消息实现原理

RocketMQ 的事务消息通过**两阶段提交（2PC）**的思想和**事务反查机制**，确保了本地事务执行与消息发送的最终一致性。

#### 核心流程
RocketMQ 的事务消息流程分为三个主要阶段：

1.  **发送半消息**
    *   Producer 发送一条消息给 Broker。
    *   Broker 收到后，将消息存储在 `RMQ_SYS_TRANS_HALF_TOPIC` 这个系统 Topic 中。
    *   **关键点**：此时消息对 Consumer 不可见，且不能被消费。这相当于“锁定”了消息资源。

2.  **执行本地事务**
    *   Producer 收到半消息发送成功的响应后（或者由发送线程异步），开始执行本地业务逻辑（如数据库操作）。
    *   根据本地事务的执行结果，向 Broker 发送提交或回滚的请求。

3.  **提交/回滚事务**
    *   **Commit**：Broker 收到 Commit 指令，将半消息从 `RMQ_SYS_TRANS_HALF_TOPIC` 移动到真正的业务 Topic 队列中，此时 Consumer 可见并消费。
    *   **Rollback**：Broker 收到 Rollback 指令，丢弃半消息，不对 Consumer 暴露。
    *   **未知状态处理**：如果 Broker 未收到 Producer 的确认（如 Producer 挂了、网络断了），Broker 会启动反查机制。

#### 事务反查机制
*   **定时任务**：Broker 端的 `TransactionalMessageCheckService` 定期扫描半消息队列。
*   **反查请求**：Broker 发起 RPC 请求，调用 Producer 实现的 `checkLocalTransaction` 接口，查询本地事务状态（如查数据库）。
*   **结果处理**：
    *   返回 COMMIT：恢复消息。
    *   返回 ROLLBACK：删除消息。
    *   返回 UNKNOWN：继续等待，下次反查（有最大次数限制）。

#### 流程架构图
```text
+------------+       1. 发送半消息        +-----------+
|  Producer  | ----------------------> |   Broker   |
| (本地事务) | <---------------------- | (Half Msg) |
+------------+       2. 返回写入成功      +-----------+
      |
      | 3. 执行本地事务
      |
      +-----------------------+
      |                       |
  成功 |                       | 失败
      v                       v
+------------+       4. Commit/     +------------+
|  Producer  | -----------------> |   Broker   |
|            |       Rollback     | (Move/Drop) |
+------------+                   +------------+
      |
      | [异常中断: 未收到确认]
      v
+------------+       5. 反查请求      +-----------+
|   Broker   | <------------------ |  Producer |
| (CheckSvc) | ------------------> | (回查DB)  |
+------------+   6. 返回事务状态     +-----------+
```

#### 代码使用简述
使用时需实现 `TransactionListener` 接口：
*   `executeLocalTransaction`：执行本地事务并返回状态。
*   `checkLocalTransaction`：处理 Broker 的反查请求。

### 实战深化

**实战案例**：
**大事务导致的反查风暴**。某业务在 `executeLocalTransaction` 中执行了一个耗时极长的 SQL 操作（超过 30秒），导致 Broker 端的反查线程误判为超时，开始频繁发起反查 RPC（每分钟一次），最终压垮了 Producer 端的服务接口。
**解决方案**：本地事务执行应尽可能快；若确实涉及长耗时操作，建议在本地事务中仅记录“待处理”状态，利用异步线程处理实际业务，并更新状态，反查时返回中间状态即可避免大量回查。

**代码示例**：
RocketMQ 事务消息生产者关键实现
```javanpublic class TransactionListenerImpl implements TransactionListener {
    // 本地事务执行
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            // 执行业务 DB 操作
            businessService.doOrder(msg.getBody());
            // 成功返回 COMMIT_MESSAGE
            return LocalTransactionState.COMMIT_MESSAGE;
        } catch (Exception e) {
            // 失败返回 ROLLBACK_MESSAGE
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }

    // Broker 反查接口
    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 关键：必须查询数据库确认状态，避免返回 UNKNOWN 导致死循环
        boolean status = businessService.checkOrderStatus(new String(msg.getBody()));
        if (status) return LocalTransactionState.COMMIT_MESSAGE;
        // 如果未查到，可以返回 UNKNOW 让下次继续查，或者直接 ROLLBACK
        return LocalTransactionState.UNKNOW;
    }
}
```

**对比表格**：

| 状态阶段 | 说明 | 处理动作 | 常见误区 |
| :--- | :--- | :--- | :--- |
| **发送半消息** | 消息存入 `RMQ_SYS_TRANS_HALF_TOPIC` | 暂存，对消费者不可见 | 认为此时消息已发送成功，其实仅是“预存” |
| **执行本地事务** | Producer 执行 DB 业务 | 同步执行，决定后续 Commit/Rollback | 事务内逻辑过于复杂，导致网络超时 |
| **提交/回滚** | 结束事务 | 半消息移动到真实 Topic 或 删除 | 未捕获异常导致异常时未显式回滚 |
| **反查** | 解决消息丢失/状态未知 | Broker 定时回调 `checkLocalTransaction` | 反查逻辑直接返回固定状态，未真实查库 |

## 常见考点
1.  **半消息存储在哪个 Topic？**
    `RMQ_SYS_TRANS_HALF_TOPIC`。这对用户是透明的，主要用于 Broker 内部处理事务状态。
2.  **如果本地事务执行成功了，但是向 Broker 发送 Commit 请求失败了怎么办？**
    依赖 Broker 的反查机制。Broker 会定期扫描半消息，发现未确认的消息后，回调 Producer 的 `checkLocalTransaction` 接口，Producer 查询本地库发现事务成功，回复 COMMIT，从而保证消息最终被投递。
3.  **事务消息可以延时吗？**
    RocketMQ 4.x 版本中，事务消息不支持延时消息等级。因为半消息机制和延时消息的处理逻辑冲突。

## 记忆要点

- 核心思想是两阶段提交(2PC)：一发半消息(不可见)，二提交或回滚
- 半消息发送成功后执行本地事务，根据结果通知 Broker 处理
- 未知状态兜底：Broker 定时扫描未确认半消息，RPC 反查 Producer 状态
- 避坑：executeLocalTransaction 必须快，长事务会引发反查风暴压垮接口
