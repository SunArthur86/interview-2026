# JVM选项

RocketMQ 对 JVM 参数有特定的要求，主要用于应对大量的内存映射文件操作和网络 IO。

**关键参数建议**

1. **堆内存设置**
   - **建议值**：`-Xms4g -Xmx4g`（或 `-Xms8g -Xmx8g`）。
   - **原因**：RocketMQ 利用操作系统的 PageCache 读写 CommitLog，堆内存不需要设置得过大。过大的堆会导致 Full GC 时间过长，影响系统响应。

2. **垃圾回收器**
   - **建议**：使用 G1 收集器 (`-XX:+UseG1GC`)。
   - **原因**：G1 在低延迟和大内存场景下表现稳定，能有效控制 STW（Stop The World）时间。

3. **打印 GC 日志**
   - **建议**：开启详细的 GC 日志以便排查内存泄漏或频繁 GC问题。

**深化实战**

- **实战案例**：某线上环境因堆内存设置过大（16G），在积压消息数千万时，Old Gen 频繁触发 Full GC 导致发送阻塞，将堆内存降至 6G 并开启 G1 后，TPS 恢复正常。
- **代码示例**：
```bash
# runbroker.sh 关键配置
JAVA_OPT="${JAVA_OPT} -Xms6g -Xmx6g"
JAVA_OPT="${JAVA_OPT} -XX:+UseG1GC"
JAVA_OPT="${JAVA_OPT} -XX:MaxGCPauseMillis=200"
JAVA_OPT="${JAVA_OPT} -XX:+HeapDumpOnOutOfMemoryError"
```

## 技术原理

RocketMQ 的 JVM 调优反直觉地「不要大堆」，根本原因在于其存储架构的特殊设计：

- **零拷贝 + PageCache 的存储模型**：RocketMQ 的 CommitLog 是顺序写的内存映射文件（MAPPEDByteBuffer）。Producer 写消息时，数据直接写入 PageCache（内核态），由 OS 异步刷盘；Consumer 消费时通过 sendfile 从 PageCache 直接到网卡。整个热数据路径几乎不经过 JVM 堆——堆里只存运行时对象（请求上下文、Pull 消息的引用、Filter 链等），所以堆不需要承载消息数据本身。
- **为什么大堆有害**：(1) Full GC 会扫描整个 Old Gen，堆越大 STW 越长。16G 堆的 Full GC 可能停顿数秒到数十秒，Broker 心跳超时被踢出集群、Producer 发送超时；(2) 大堆下 G1 的 Region 数量激增，标记阶段开销变大；(3) 堆挤压了 PageCache 的可用内存——OS 把内存让给 JVM 堆，PageCache 变小，消息读写命中率下降，反而退化回磁盘 IO。
- **G1 为何适合**：G1 把堆分成 2048 个左右 Region，每次 GC 只回收部分 Region（Garbage First），可控制 MaxGCPauseMillis。对低延迟要求高的 Broker（发送 RT < 1ms），G1 能把 STW 压在 200ms 内，相比 CMS 的碎片问题和 ParOld 的全堆扫描更可控。
- **堆与 PageCache 的内存分配**：物理内存是定值，堆大了 PageCache 就小。RocketMQ 经验法则是「堆 + 直接内存 + PageCache」大致按 1:1:8 分配。例如 64G 内存机器：堆 6G + DirectMemory 4G + 剩余给 PageCache。

## 注意事项

- **DirectMemory 也要监控**：RocketMQ 的 Netty 使用堆外内存做网络缓冲，`-XX:MaxDirectMemorySize` 默认与堆大小相同。如果消息体大、并发高，堆外内存可能先于堆 OOM。建议显式设置并监控。
- **G1 在 JDK 11+ 才稳定**：JDK 8 的 G1 性能不如 JDK 11/17。生产建议用 LTS 版本（JDK 17 的 G1 + ZGC 特性更成熟），且开启 `-XX:+UseStringDeduplication` 减少重复字符串占用。
- **GC 日志的实战价值**：开启 `-Xlog:gc*:/path/to/gc.log:time,uptime`（JDK 9+ 语法），配合 GCEasy/GCViewer 分析。重点看 Full GC 频率、Young GC 耗时、Humongous Object 分配——RocketMQ 大消息（>Region 的一半）会触发 Humongous 分配，是 G1 调优的重点。
- **不要盲目用 ZGC/Shenandoah**：超低延迟回收器（<10ms STW）在 Broker 场景收益有限（G1 的 200ms 已够），反而吞吐量下降。除非有极致 SLA 要求，否则 G1 是稳妥选择。
- **TransientStorePool 的权衡**：RocketMQ 提供 `transientStorePoolEnable=true` 让 CommitLog 写堆外内存再 flush 到 PageCache，绕过双层缓冲提升写吞吐。但代价是消费时无法享受 PageCache 命中（数据还没到 PageCache），读延迟上升，适合「写多读少」场景，写读均衡场景应关闭。
- **G1 的 Region 大小调优**：默认 Region 大小由堆自动计算（1-32MB）。RocketMQ 大消息（如 4MB 事务消息）容易触发 Humongous Object，导致频繁 Mixed GC。显式设置 `-XX:G1HeapRegionSize=16m` 让大对象分散，减少 Humongous 分配。
- **监控的硬指标**：除了 GC 日志，还要盯两个指标——`broker_fast_offset` 落后量（CommitLog 刷盘延迟）和 `MsgGetTotalRatio`（PageCache 命中率）。命中率跌破 90% 说明 PageCache 被挤压，通常是堆太大或并发消费拉低了缓存效率，要优先降堆而不是加内存。
- **容器化部署的 cgroup 陷阱**：K8s 容器内 JVM 感知的是宿主机内存而非 cgroup limit，`-Xmx6g` 可能超过容器 limit 被 OOMKilled。JDK 10+ 支持 `+UseContainerSupport`（默认开启），用 `-XX:MaxRAMPercentage=50` 按容器内存比例分配更安全。
- **G1 Mixed GC 的触发频率**：RocketMQ 的长生命周期对象（如 PullRequest 队列）容易进入 Old Gen，Mixed GC 频繁触发会拉长 STW。监控 `-XX:+G1UseAdaptiveIHOP` 自适应触发阈值，必要时固定 `-XX:G1HeapWastePercent=5` 控制回收激进程度。
- **堆外内存与 Netty 的配合**：RocketMQ 的 Netty 使用池化的 DirectByteBuffer 做网络 IO，`-XX:MaxDirectMemorySize` 应略大于 `-XX:NettyDirectMemSize`（默认 16MB）+ 业务峰值缓冲。生产建议堆外内存设为 4-8G，并通过 `-Dio.netty.allocator.numDirectArenas` 控制 arena 数量避免分配竞争。
- **JFR（Java Flight Recorder）的实战价值**：JDK 11+ 自带的 JFR 几乎零开销，能记录每次 GC 的耗时、分配热点、线程阻塞事件。RocketMQ 出现「偶发发送卡顿」时，开启 `jcmd <pid> JFR.start duration=60s filename=broker.jfr` 录制 1 分钟，用 JMC 分析能快速定位是 GC、锁竞争还是 IO 等待，比单纯看 GC 日志信息量大得多。

## 记忆要点

- 堆内存建议4g或8g，不宜过大，因为RocketMQ核心读写依赖PageCache而非堆内存
- 垃圾回收器首选G1，因为G1在低延迟场景下能有效控制STW时间
- 堆内存过大会导致Full GC时间过长，引发发送阻塞，调整至6G并开启G1可恢复TPS
