# Redis 中 Pipeline（流水线）技术与 Lua 脚本在执行原子性操作上有何区别？如何根据业务场景选择？

【基本区别】
Pipeline 的主要目的是减少网络 RTT（往返时间），它将多条命令打包一次性发送给服务端，服务端依次执行并返回结果。Pipeline 中的命令是非原子的，执行过程中可能被其他客户端的命令插队。Lua 脚本在 Redis 中是原子执行的，Redis 会在执行脚本期间阻塞其他命令，直到脚本结束，且脚本中可以包含复杂的逻辑判断。选择上，如果只是需要批量执行独立的读写操作以提升吞吐量，首选 Pipeline；如果需要保证一组命令按顺序执行且中间不被打断（如检查并设置、库存扣减），或者需要复杂的业务逻辑判断，则应选择 Lua 脚本。

【深入原理：网络 IO 模型】
- **Pipeline**：基于 Redis 的 RESP 协议，客户端缓冲多条命令，一次性写入 Socket TCP 缓冲区，服务端一次性读取并处理。本质是**批处理**。不保证原子性是因为 Redis 单线程处理命令，Pipeline 内部的命令虽然连续执行，但如果有其他 Pipeline 交错或慢查询，执行流会被打乱。
- **Lua 脚本**：Redis 使用 `EVAL` 命令执行 Lua 脚本。Redis 内嵌了 Lua 解释器，执行脚本时会创建一个伪客户端，按顺序执行脚本中的命令。整个脚本执行期间，Redis 服务器**阻塞**其他命令请求，保证原子性。

【选择策略】
1. **性能敏感但无强一致性要求**：使用 Pipeline。例如批量写入 10,000 条日志。
2. **强一致性/事务性要求**：使用 Lua。例如 `get` 后判断再 `set`（CAS 逻辑），或者分布式锁的解锁逻辑（判断锁归属再删除）。
3. **复杂逻辑**：Lua 支持条件判断、循环，可以在服务端处理数据，减少网络传输数据量。

【注意事项】
- **Pipeline 限制**：一次 Pipeline 包含的命令不宜过多，会导致内存占用暴涨（客户端和服务端缓冲区），建议分批发送。
- **Lua 限制**：脚本执行时间越长，阻塞 Redis 越久，尽量编写 O(1) 或 O(N) 但 N 很小的脚本；不要在 Lua 中写死循环。

【常见考点】
1. **Redis 为什么单线程还快**：基于内存操作、IO 多路复用、避免锁竞争。Pipeline 和 Lua 都是为了减少网络交互开销。
2. **Lua 脚本超时怎么办**：Redis 5+ 支持 `SCRIPT KILL`，但如果脚本已经执行了写操作，则只能 `SHUTDOWN NOSAVE` 来强制停止以保证数据一致性。
3. **原子性实现原理**：Redis 是单线程模型，Lua 脚本执行期间不需要加锁即可保证原子性，因为没有其他线程能抢占 CPU。

**实战案例**：在秒杀场景中，曾错误使用Pipeline批量扣减库存，导致高并发下出现超卖；后改用Lua脚本实现`get-and-compare-decr`原子操作彻底解决。另一次批量导入数据时使用Pipeline单次发送5万条命令，直接导致Redis OOM，改为分批（每批500条）后恢复。

**代码示例**：
```java
// Pipeline 批量操作示例
redisTemplate.executePipelined((RedisCallback<Object>) connection -> {
    for (String key : keys) {
        connection.set(key.getBytes(), value.getBytes());
    }
    return null;
});

// Lua 脚本原子扣减库存示例
String script = "if redis.call('get', KEYS[1]) >= ARGV[1] then return redis.call('decrby', KEYS[1], ARGV[1]) else return -1 end";
DefaultRedisScript<Long> redisScript = new DefaultRedisScript<>(script, Long.class);
Long result = redisTemplate.execute(redisScript, Collections.singletonList("stock:1001"), "1");
```

## 技术原理

**Pipeline：减少网络 RTT，非原子，适合批量无依赖操作**
Pipeline 的本质是客户端缓冲多条命令，一次性写入 TCP 缓冲区发送给 Redis，服务端依次执行后批量返回结果。它只解决网络往返开销（RTT），并不保证原子性。在 Redis 单线程模型下，Pipeline 内部的命令虽然连续执行，但若期间有其他客户端的命令交错或慢查询插队，执行流会被打乱，因此不适合需要原子性的场景。

**Lua 脚本：原子执行，阻塞其他命令，适合强一致性和复杂逻辑**
Redis 内嵌 Lua 解释器，执行 `EVAL` 时会创建伪客户端按顺序执行脚本中的命令。整个脚本执行期间，Redis 主线程独占，阻塞所有其他命令请求，从而保证绝对的原子性。这使得 Lua 天然适合 CAS（Compare-And-Swap）、库存扣减、分布式锁解锁等需要"判断+操作"不可分割的场景。

**性能瓶颈差异：带宽 vs CPU 阻塞时间**
Pipeline 的瓶颈在网络带宽和缓冲区内存：一次发送过多命令会导致客户端和服务端内存暴涨，甚至 OOM。Lua 的瓶颈在 CPU 阻塞时间：脚本执行期间整个 Redis 被独占，若脚本过慢（如 O(N) 大 N 或死循环），会拖垮所有其他请求。

## 代码示例

```java
// Pipeline 批量写入（适合日志、埋点等无依赖场景）
redisTemplate.executePipelined((RedisCallback<Object>) connection -> {
    for (String key : keys) {
        connection.stringCommands().set(
            key.getBytes(), value.getBytes());
    }
    return null; // 一次性发送，减少 N 次 RTT 为 1 次
});
```

```java
// Lua 原子扣减库存（CAS：判断够再扣，中间防插队）
String script =
    "if redis.call('get', KEYS[1]) >= ARGV[1] then " +
    "  return redis.call('decrby', KEYS[1], ARGV[1]) " +
    "else return -1 end";
DefaultRedisScript<Long> redisScript =
    new DefaultRedisScript<>(script, Long.class);
Long result = redisTemplate.execute(
    redisScript, Collections.singletonList("stock:1001"), "1");
```

## 注意事项

- 核心对比：Pipeline 非原子（仅减网络 RTT），而 Lua 绝对原子（防插队）。
- 场景选择：大批量独立读写用 Pipeline，CAS 判断或复杂逻辑用 Lua。
- 底层原理：Lua 执行期间 Redis 单线程阻塞，因为无其他线程抢占保证原子。
- 避坑指南：Pipeline 批量过大易 OOM，Lua 执行过慢会阻塞全局禁死循环。
- Lua 脚本超时只能 `SCRIPT KILL`，若已执行写操作则必须 `SHUTDOWN NOSAVE`。
