# RocketMQ 整体架构设计

整体的架构设计主要分为四大部分，分别是：Producer、Consumer、Broker、NameServer。
为了更贴合实际，我画的都是集群部署，像 Broker 我还画了主从。

**Producer（生产者）**：
- 消息生产者，支持集群部署。
- 启动时会与 NameServer 集群中的某一台建立长连接，定期（默认 30s）从 NameServer 拉取 Topic 路由信息并更新本地缓存。
- 根据路由信息找到对应的 Broker Master 建立长连接，支持多种负载均衡策略（如随机、轮询等）发送消息。

**Consumer（消费者）**：
- 消息消费者，支持集群部署。
- 同样与 NameServer 建立长连接获取路由信息。
- 根据路由信息与对应的 Broker Master/Slave 建立连接。支持集群消费（消息分摊）和广播消费（消息复制）两种模式。

**Broker（代理服务器）**：
- 负责消息的存储、查询、消费转发，支持主从部署（Master/Slave）。
- **Master**：支持读写操作，负责写入消息和响应读请求。
- **Slave**：只支持读操作，通过同步或异步方式复制 Master 的数据，用于提供读服务的分担和 HA 保证。
- Broker 启动后会向集群内**所有** NameServer 注册路由信息（Topic、队列等），并定期发送心跳（默认 30s）。

**NameServer（注册中心）**：
- 轻量级的 Topic 路由注册中心，充当服务发现的角色。
- 支持 Broker 的动态注册与发现，维护 Topic 与 Broker 的映射关系。
- NameServer 节点之间互不通信，无状态设计，每个节点都保存全量的路由信息。

**交互流程概括**：
1. **NameServer** 先启动，各节点独立运行，等待 Broker 注册。
2. **Broker** 启动，向所有 NameServer 发送注册包，随后每隔 30s 发送心跳包（包含 IP、Port、Topic 配置）。NameServer 每隔 10s 扫描一次 Broker 存活状态，超过 120s 未收到心跳则判定下线并移除路由。
3. **Producer** 启动，从 NameServer 获取目标 Topic 的路由信息，与 Master 建立连接发送消息。
4. **Consumer** 启动，从 NameServer 获取路由信息，与 Master/Slave 建立连接拉取或推送消息。

### 实战案例
在某金融支付系统中，曾因 NameServer 网络分区导致部分 Producer 无法感知到 Broker 下线，短暂将消息发送到了不可用的 Broker，从而造成发送超时。为了增强容错，生产环境部署时通常要求客户端配置多个 NameServer 地址，并在代码层实现快速重试机制，以应对 NameServer 节点不可用的情况。

### Broker 向 NameServer 注册核心代码

```java
// BrokerController.start() 中启动定时任务
this.scheduledExecutorService.scheduleAtFixedRate(new Runnable() {
    @Override
    public void run() {
        try {
            BrokerController.this.registerBrokerAll(true, false, brokerConfig.isForceRegister());
        } catch (Throwable e) {
            log.error("registerBrokerAll Exception", e);
        }
    }
}, 1000 * 10, Math.max(10000, Math.min(brokerConfig.getRegisterNameServerPeriod(), 60000)), TimeUnit.MILLISECONDS);
```

### 组件角色对比表

| 组件 | 核心职责 | 是否有状态 | 通信机制 |
| :--- | :--- | :--- | :--- |
| **NameServer** | 路由注册与发现 | 无状态 (内存存路由) | Broker 主动上报心跳 (单向) |
| **Broker** | 消息存储与转发 | 有状态 (磁盘存储) | 与 NameServer (心跳), 与 Client (长连接) |
| **Producer** | 消息发送 | 无状态 (本地缓存路由) | 从 NameServer 拉取, 向 Broker 发送 |
| **Consumer** | 消息消费 | 有状态 (消费进度 Offset) | 从 NameServer 拉取, 从 Broker 拉取/推送 |

## 记忆要点

- 四大组件：NameServer作轻量注册中心，Broker主从存数，生产/消费客户端
- 注册中心：NameServer无状态且节点互不通信，各自保存全量路由极高可用
- 心跳机制：Broker每30s向所有NameServer上报心跳，超120s未收到则剔除

## 结构化回答

**30 秒电梯演讲：** 通过 NameServer 解耦组件， Broker 存储消息， 生产消费分离。打个比方，像餐厅系统，前台（NameServer）导座，后厨（Broker）做菜，食客（Consumer）点餐。

**展开框架：**
1. **四大组件** — NameServer作轻量注册中心，Broker主从存数，生产/消费客户端
2. **注册中心** — NameServer无状态且节点互不通信，各自保存全量路由极高可用
3. **心跳机制** — Broker每30s向所有NameServer上报心跳，超120s未收到则剔除

**收尾：** 我在项目里踩过坑——在某金融支付系统中，曾因 NameServer 网络分区导致部分 Producer 无法感知到 Broker 下线，短暂将消息发送到了不可用的 Broker，从而造成发送超时。您想深入聊哪一段：原理、避坑还是对比选型？
