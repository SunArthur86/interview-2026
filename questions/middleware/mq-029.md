# Kafka的存储结构

Kafka 的存储结构设计体现了算法在工程中的灵活运用，通过分层组织实现了高效的消息存储与检索。

### 1. 逻辑结构与物理映射
- **逻辑结构**：Topic -> Partition（分区）。分区是 Kafka 最小的存储并行度和读写单元。
- **物理结构**：
  - 每个 Partition 对应磁盘上的一个子目录。
  - 目录命名格式：`<topic_name>-<partition_id>`（例如 `order_topic-0`）。
  - 目录下包含多个 **Log Segment**（日志段）。为了方便清理和查找，日志被切分为多个段。

### 2. Log Segment 细分
每个 Log Segment 是一组文件的集合，包含以下核心文件：
- **.log (数据文件)**：实际存储消息内容，文件名命名规则为该段第一条消息的 Base Offset（如 `00000000000000000000.log`）。
- **.index (位移索引)**：存储偏移量到物理位置的映射。
- **.timeindex (时间戳索引)**：存储时间戳到偏移量的映射。
- **.txnindex (事务索引)**：记录 Aborted 的事务（Kafka 0.11+ 支持）。

- **实战案例**：在进行历史数据迁移时，直接操作物理目录发现，仅复制活跃 Segment（正在写的那个）会导致数据丢失或不一致。必须连同 `.index` 和 `.timeindex` 一同复制，否则新 Broker 虽有数据文件但无法通过索引定位消息，导致查询极慢甚至报错。

### 3. 文件命名与查找
Kafka 利用文件名（即 Base Offset）来快速定位 Segment。所有 Segment 的 Base Offset 是递增的。当需要读取某个 Offset 的消息时，通过二分查找文件名即可快速定位到对应的 Segment 目录，无需加载元数据到数据库。

- **代码示例**：
  ```java
  // 查找对应 LogSegment 的逻辑伪代码
  LogSegment findSegment(long targetOffset) {
      List<LogSegment> segments = getSortedSegments(); // 按文件名(BaseOffset)排序
      for (LogSegment segment : segments) {
          if (segment.baseOffset <= targetOffset && 
              (segment.nextOffset > targetOffset || segment.isLast())) {
              return segment;
          }
      }
      return null;
  }
  ```
