# Kafka 的 ISR (In-Sync Replicas) 机制是如何保证数据高可用的？

ISR 是 Leader 维护的一组与 Leader 保持同步的副本集合（包含 Leader 自身）。只有 ISR 中的副本才有资格被选为新的 Leader。在 Kafka 中，`acks=all` (或 -1) 意味着消息必须被 ISR 中所有副本同步成功后，Leader 才会向生产者返回成功。为了防止某个副本故障导致整个写入性能下降，Kafka 引入了 `min.insync.replicas` 参数，只要 ISR 中达到该数量的副本同步即可。如果某个 Follower 长时间未发送 FetchRequest（超过 `replica.lag.time.max.ms`），会被踢出 ISR。当 ISR 副本恢复数据同步后，会重新加入。这种机制在一致性和可用性之间取得了平衡：既保证了数据不丢失（只要 ISR 至少有一个存活），又容忍了临时的网络延迟或节点故障。

**实战案例**：在某高并发日志系统中，曾因某次磁盘 I/O 抖动导致 Follower 频繁被踢出 ISR，使得 `min.insync.replicas` 条件不满足，生产者收到 `NotEnoughReplicas` 异常从而导致写入阻塞。后续通过调大 `replica.lag.time.max.ms` 并优化磁盘性能，有效解决了此类假死故障。

**参数对比**:

| 参数 | 作用 | 推荐设置 | 风险点 |
| :--- | :--- | :--- | :--- |
| `acks=all` | 确认机制，要求 ISR 所有副本确认 | 高可靠性业务必选 | 吞吐量降低 |
| `min.insync.replicas` | 最小同步副本数，配合 acks=all 使用 | 通常设置为 >1 (如 2) | 若 ISR 剩余副本数 < 此值，拒绝写入 |
| `replica.lag.time.max.ms` | 副本滞后时间阈值，超过即被踢出 ISR | 默认 30000ms (视网络状况调整) | 设置过小导致 ISR 频繁变动，设置过大导致故障发现不及时 |

## 技术原理

ISR 是 Kafka 在 CAP 之间动态权衡的核心机制，其设计精妙地结合了「强一致子集」与「最终一致全集」：

- **HW 与 LEO 的协同**：每个副本维护 LEO（Log End Offset，自己已写入的最新 offset）和 HW（High Watermark，所有 ISR 副本中最小的 LEO）。生产者写消息到 Leader 后，Follower 通过 FetchRequest 拉取并追赶。Leader 根据 ISR 中所有副本的 LEO 更新 HW。**只有 HW 以下的消息对消费者可见**——这保证消费者读到的数据必然已在多数副本持久化，避免读到未提交数据后 Leader 切换导致的数据回滚。
- **ISR 动态收缩与扩张**：Leader 周期性检查每个 Follower 的「最后追赶时间」。若某 Follower 在 `replica.lag.time.max.ms`（默认 30 秒）内未完成最新消息的 Fetch，说明它跟不上，Leader 把它移出 ISR（ISR 收缩），此时 `acks=all` 只需剩余 ISR 确认。Follower 恢复后追上 HW，重新被加入 ISR（ISR 扩张）。这种「动态降级」让系统在单副本故障时仍可写入。
- **`min.insync.replicas` 的安全底线**：单纯 `acks=all` 在 ISR 缩减到 1（只剩 Leader）时仍会写入——此时 Leader 故障数据就丢了。`min.insync.replicas=2` 强制要求 ISR 至少 2 个副本才允许写入，否则抛 `NotEnoughReplicasException`，宁可写不进去也不丢数据。这是「可用性换一致性」的显式开关。
- **Leader 选举的 ISR 约束**：旧 Leader 宕机时，Controller 从 ISR 中选第一个存活的副本作为新 Leader。这保证新 Leader 拥有全部已提交消息（HW 以下）。若 ISR 全部宕机且开启 `unclean.leader.election.enable`，可能选非 ISR 副本（数据不全）做 Leader，会丢数据——金融场景必须关闭此选项。

## 代码示例

```bash
# 创建 Topic 时指定副本因子和最小同步副本
bin/kafka-topics.sh --create \
  --topic orders \
  --partitions 12 \
  --replication-factor 3 \          # 3 副本，容忍 1 故障
  --config min.insync.replicas=2 \  # ISR 至少 2 个才允许写入
  --config unclean.leader.election.enable=false  # 禁止脏选举，防丢数据

# 查看某分区的 ISR 状态
bin/kafka-topics.sh --describe --topic orders
# 输出示例：
# Topic: orders  Partition: 0  Leader: 1  Replicas: 1,2,3  Isr: 1,2,3
#                                                    ↑ ISR 列表，等于 Replicas 表示全部健康

# 实时监控 ISR 缩减（运维告警）
bin/kafka-topics.sh --describe --topic orders | grep -v "Isr: .*1,2,3"
# 若输出非空，说明有副本掉队，需排查
```

```properties
# Producer 端配置（高可靠场景）
acks=all                              # 等所有 ISR 确认
retries=2147483647                    # 无限重试
max.in.flight.requests.per.connection=5  # 幂等性开启时可到 5，否则必须 1
enable.idempotence=true               # 配合重试防重复
```

## 注意事项

- **副本数 vs min.insync.replicas 的关系**：建议 `replication.factor=3, min.insync.replicas=2`，容忍 1 副本故障仍可写入，2 副本故障才停写。若设 `min.insync.replicas=3`，任何单副本抖动都会阻塞写入，过于脆弱。
- **`replica.lag.time.max.ms` 调优**：默认 30 秒适合稳定网络。跨机房或磁盘 IO 抖动频繁的场景，可调大到 60~120 秒，避免 Follower 频繁进出 ISR 导致写入毛刺。但调太大会延迟故障发现。
- **unclean.leader.election 的取舍**：金融/订单系统必须 `false`（数据安全优先）；日志/监控类可 `true`（可用性优先，容忍少量丢失）。这是业务决定的，不能一刀切。
- **跨机房 ISR 的延迟问题**：跨机房副本同步受网络延迟影响，Follower 容易因延迟被踢出 ISR。建议同机房部署全部 ISR，或使用 Rack Awareness 把副本分散到机架而非机房。
- **ISR 抖动告警**：生产环境应监控 ISR 缩减事件（`kafka.server:type=ReplicaManager` 的 UnderReplicatedPartitions 指标），及时排查磁盘/网络问题。

## 记忆要点

- ISR是Leader维护的同步副本集合，只有ISR内副本才有资格被选为新Leader。
- acks=all要求所有ISR副本确认，配合min.insync.replicas防止极端丢数。
- Follower若超时未同步会被踢出，恢复后重新加入，在一致性与可用性间平衡。
- IO抖动致Follower频繁剔除引发写入阻塞，可适当调大滞后时间阈值。
