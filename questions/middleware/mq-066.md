# NameServer 的寻址

RocketMQ 客户端（Producer/Consumer）需要知道 NameServer 的地址才能拉取路由信息。

**寻址方式**
推荐使用 **HTTP 静态服务器寻址**。客户端配置一个 URL，通过 HTTP GET 请求定期获取 NameServer 的地址列表（默认 2 分钟刷新一次）。

**实现原理**
客户端启动时或定时任务触发，请求指定的 HTTP 地址（如 `http://app:8080/rocketmq/namesrv`），该地址返回一个纯文本的 NameServer IP:Port 列表（换行分隔）。

```text
+-----------------+     HTTP GET      +------------------+
|   MQ Client     | -----------------> |  Static HTTP SVR |
| (Producer/Cons) | <----------------- | (nginx/file)     |
+--------+--------+   IP List (Text)   +------------------+
         |
         | 192.168.1.1:9876
         | 192.168.1.2:9876
         v
  Connect to NameServer Cluster
```

**优势**
- **动态更新**：当 NameServer 集群扩缩容时，只需更新 HTTP 服务器上的静态文件，客户端无需重启即可感知新的节点列表。
- **统一管理**：避免了在每台机器上手动配置环境变量或启动参数 `namesrvAddr` 的繁琐和易错问题。

**对比表格：寻址方式**

| 方式 | 配置位置 | 动态更新能力 | 优先级 | 推荐场景 |
| :--- | :--- | :--- | :--- | :--- |
| **HTTP 静态服务** | `rocketmq.name-server.domain` | 高（定时拉取） | 低 | 生产环境（推荐） |
| **环境变量** | `NAMESRV_ADDR` | 否（需重启） | 中 | Docker/K8s 容器化部署 |
| **启动参数** | `-Drocketmq.namesrv.addr` | 否（需重启） | 高 | 本地测试、临时覆盖 |

**实战案例**
在云原生环境中，曾因使用硬编码 IP 寻址，导致 NameServer 宕机迁移后 IP 变化，所有 Producer 宕机无法发送。后改为将 NameServer 的 Kubernetes Service 域名配置在 HTTP 后端，客户端通过 HTTP 拉取域名列表（虽然 IP 变，但通过 DNS 解析自动迁移），实现了高可用。

**代码示例 (Java - 优先级配置)**
```java
// 代码中设置优先级最高，通常用于测试覆盖默认配置
// 生产推荐通过-Drocketmq.namesrv.domain=xxx HTTP 方式
DefaultMQProducer producer = new DefaultMQProducer("ProducerGroup");

// 方式1：显式设置（启动参数级优先级）
producer.setNamesrvAddr("127.0.0.1:9876");

// 方式2：通过 HTTP 动态获取（实际工程中不写在代码里，通过启动参数指定 Domain）
// producer.setNamesrvDomain("your.namesrv.domain.com");
```

## 常见考点
1. 除了 HTTP 寻址，还有其他方式吗？（环境变量、启动参数 `-DNAMESRV_ADDR`，优先级高于 HTTP）
2. 客户端多久刷新一次 NameServer 地址列表？（默认 2 分钟，通过 `pollNameServerInterval` 配置）
3. 如果 HTTP 服务器不可用，客户端还能工作吗？（能，客户端会缓存最后一次成功的地址列表）

## 记忆要点

- 优先级口诀：代码配置 > 启动参数 > 环境变量 > HTTP静态服务
- 生产推荐HTTP寻址，因为支持动态更新，集群扩缩容无需重启客户端
- HTTP寻址默认2分钟刷新一次，若服务器不可用，客户端仍可使用缓存地址工作
- NameServer列表通过HTTP请求获取，返回值为纯文本的IP:Port列表
