# Netty的核心组件有哪些？

### Netty 核心组件详解

1.  **Channel**：网络连接的抽象（如 NioSocketChannel），支持异步 I/O 操作。
2.  **EventLoop**：
    -   本质是一个**单线程执行器**（同时维护了一个 Selector 和 TaskQueue）。
    -   负责处理 I/O 事件（Read/Write）和内部任务（register、定时任务）。
    -   **规则**：一个 Channel 在其生命周期内只绑定到一个 EventLoop（线程安全）；一个 EventLoop 可以处理多个 Channel。
3.  **ChannelPipeline**：
    -   责任链模式，由一系列 ChannelHandler 组成。
    -   是**双向链表**结构：Head -> InboundHandler -> OutboundHandler -> Tail。
4.  **ChannelHandler**：
    -   处理 I/O 事件或拦截 I/O 操作。
    -   **ChannelInboundHandler**：处理入站数据（如读、连接激活）。
    -   **ChannelOutboundHandler**：处理出站数据（如写、连接、刷新）。
5.  **ChannelHandlerContext**：
    -   Handler 与 Pipeline 之间的桥梁，保存了 Handler 在链表中的上下文信息。
    -   用于传播事件（`fireChannelRead`）或动态修改 Pipeline。
6.  **ByteBuf**：
    -   Netty 的字节容器，比 NIO `ByteBuffer` 优势在于：**读写指针分离**（ByteBuffer 需 flip），支持**池化**（减少 GC），支持**零拷贝**（CompositeByteBuf）。
7.  **Bootstrap/ServerBootstrap**：启动引导类，配置线程模型、IO 模型、Handler 等。

### 线程模型与事件流转架构图
```text
       Boss EventLoop Group              Worker EventLoop Group
+-------------------------+       +-----------------------------+
| Selector (Accept)       |       | Selector (Read/Write)       |
| 1. 接收新连接            | ----> | 3. 注册 Channel 到 Selector  |
| 2. 分配给 Worker         |       | 4. 轮询处理 IO 事件          |
+-------------------------+       +-----------------------------+
                                         |
                            +------------v------------+
                            | Channel (NioSocketChannel)|
                            |   (绑定到某一线程)        |
                            +------------+------------+
                                         |
                            +------------v------------+
                            |  ChannelPipeline       |
                            |  +------------------+  |
                            |  | HeadContext      |  |
                            |  | v (Inbound)      |  |
  IO Read Event ------------>  | Handler 1         |  |
                            |  | Handle
```

### 8. 实战案例
**场景**：在高性能 API 网关中，利用 `EventLoop` 的无锁化特性，将业务逻辑中的异步回调（如调用下游服务）直接提交回当前的 `EventLoop` 线程执行。**效果**：避免了线程切换带来的上下文切换开销和锁竞争，显著提升了吞吐量。

### 9. 代码示例
**Netty Server 启动与 Pipeline 配置**：
```java
EventLoopGroup bossGroup = new NioEventLoopGroup(1);
EventLoopGroup workerGroup = new NioEventLoopGroup();
try {
    ServerBootstrap b = new ServerBootstrap();
    b.group(bossGroup, workerGroup)
     .channel(NioServerSocketChannel.class)
     .childHandler(new ChannelInitializer<SocketChannel>() {
         @Override
         protected void initChannel(SocketChannel ch) {
             ChannelPipeline p = ch.pipeline();
             p.addLast(new IdleStateHandler(30, 0, 0)); // 心跳检测
             p.addLast(new HttpServerCodec());          // HTTP 编解码
             p.addLast(new MyBusinessHandler());        // 业务逻辑
         }
     });
    ChannelFuture f = b.bind(8080).sync();
} finally {
    bossGroup.shutdownGracefully();
    workerGroup.shutdownGracefully();
}
```

## 记忆要点

- EventLoop是单线程执行器：因为一个Channel生命周期内只绑定一个EventLoop，所以避免了多线程并发锁竞争。
- ChannelPipeline是双向链表：入站用Inbound自Head向Tail传递，出站用Outbound反之。
- ByteBuf优势：因为读写指针分离(无需NIO的flip)，所以支持池化和零拷贝操作。
- BossGroup负责接收新连接，WorkerGroup负责处理具体的读写I/O事件。
