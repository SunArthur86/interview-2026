# Micrometer Tracing与Spring Boot 3.x集成后，如何解决线程池异步任务中的Trace上下文丢失问题？

在Spring Boot 3.x中，Micrometer Tracing基于ThreadLocal管理上下文。默认情况下，`@Async`自动传播上下文，但手动创建的线程池会导致上下文丢失。解决方案：手动线程池需使用`ContextPropagatingExecutorService`包装原有`ExecutorService`，在任务提交时捕获并在执行时恢复上下文。

## 技术原理

- **基于 ThreadLocal 的上下文默认不自动跨线程**：Micrometer Tracing 把 TraceId、SpanId 存在 `ThreadLocal`（实际是 `ScopedValue`/`ThreadLocal`）。主线程进入异步任务后，工作线程是新线程，其 `ThreadLocal` 是空的，所以 TraceId 丢失——日志里看到的是空 TraceId，链路图断开。这是 ThreadLocal "以线程为 key" 存储的固有缺陷。
- **手动线程池需用 `ContextPropagatingExecutorService` 包装**：Spring Boot 3.x 的 `@Async` 默认会被 Micrometer Tracing 的 `ContextPropagation` 拦截自动透传，但**手动 `new ThreadPoolExecutor()`** 创建的线程池不经过这个拦截，所以上下文必然丢。解决：用 `ContextPropagatingExecutorService`（或 `TaskDecorator`）包装原生 Executor，让任务提交时捕获主线程上下文、执行时恢复。
- **包装器负责在任务提交和执行时捕获恢复上下文**：装饰器实现 `submit(Runnable)` 时把当前 ThreadLocal 快照绑定到 Runnable 闭包里；工作线程执行该 Runnable 前 `run` 先把快照恢复到自己的 ThreadLocal，执行后清理。本质是"提交时捕获 + 执行前注入 + 执行后还原"三步走。

## 代码示例

手动线程池的上下文丢失与修复：

```java
// 问题：手动线程池丢 TraceId
ExecutorService pool = Executors.newFixedThreadPool(8);
@GetMapping("/test")
public String demo() {
    String traceId = MDC.get("traceId");      // 主线程有值
    pool.submit(() -> {
        log.info("traceId={}", MDC.get("traceId"));  // null！上下文丢了
    });
}

// 修复方案1：用 ContextPropagatingExecutorService 包装
@Autowired ContextPropagation propagation;
ExecutorService wrapped = new ContextPropagatingExecutorService(pool);

// 修复方案2：用 TaskDecorator 装饰每个 Runnable
class TraceTaskDecorator implements TaskDecorator {
    public Runnable decorate(Runnable r) {
        Map<String, String> snapshot = MDC.getCopyOfContextMap();  // 提交时捕获
        return () -> {
            Map<String, String> prev = MDC.getCopyOfContextMap();
            MDC.setContextMap(snapshot);                          // 执行前注入
            try { r.run(); }
            finally { MDC.setContextMap(prev); }                  // 执行后还原
        };
    }
}
ThreadPoolExecutor pool = new ThreadPoolExecutor(...);
pool.setThreadFactory(r -> {...});
// 配合 Spring 的 @Async 配置
@Bean
public TaskExecutor taskExecutor() {
    ThreadPoolTaskExecutor exec = new ThreadPoolTaskExecutor();
    exec.setTaskDecorators(new TraceTaskDecorator());
    return exec;
}
```

## 对比/选型

| 场景 | 是否自动透传 | 处理方式 |
|------|--------------|----------|
| 同步调用 | 是 | ThreadLocal 自动可见 |
| `@Async` 默认 | 是 | Micrometer 自动拦截 |
| 手动线程池 | 否 | 需 ContextPropagatingExecutorService 包装 |
| CompletableFuture | 否 | 需 `wrap` 装饰 lambda |
| 响应式 WebFlux | 是（需配置） | 通过 Reactor Context + Hooks |

## 常见坑/注意事项

- **CompletableFuture 链也会丢**：`supplyAsync` 用默认 ForkJoinPool 不透传，要用自定义装饰后的 Executor 或手动 wrap 每个 lambda。
- **`InheritableThreadLocal` 在线程池失效**：它只在 `new Thread()` 时继承一次，线程池线程复用时继承的是首次创建时的旧值，必须用 TTL 或装饰器。
- **MDC 清理别忘了**：执行后必须 `finally` 还原/清理，否则线程复用时下一个任务会看到上个任务的 TraceId，污染日志。
- **Reactive 场景要配 Hooks**：WebFlux 没有 ThreadLocal 概念，要用 `Hooks.enableAutomaticContextPropagation()` + Reactor Context。
- **跨进程透传要单独做**：本机制只解决跨线程，跨进程（HTTP/RPC）仍要把 TraceId 塞进 Header 透传给下游。

## 记忆要点

- 上下文存储于ThreadLocal，导致跨线程池时极易发生Trace丢失问题。
- @Async注解默认已支持自动传播，但手动创建的线程池会导致丢失。
- 解决方案：使用ContextPropagatingExecutorService包装线程池。
- 机制是在任务提交时先快照捕获上下文，执行任务时再恢复上下文。
