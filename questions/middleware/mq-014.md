# RocketMQ 中的长轮询

RocketMQ 和 Kafka 选择 **拉模式** 是为了保证 Broker 的轻量级和应对海量的消息堆积。为了解决拉模式存在的**实时性差**和**空轮询**问题，它们都引入了 **长轮询** 机制。

### 1. RocketMQ 的长轮询机制
RocketMQ 的 `PushConsumer` 实际上是在客户端封装了拉取逻辑。

**工作流程**：
1.  Consumer 发起拉取请求到 Broker。
2.  **Broker 端判断**：
    *   **有消息**：立即返回。
    *   **无消息**：不立即返回空，而是将请求挂起，并扫描该 MessageQueue 是否有新消息到来。
3.  **挂起与唤醒**：
    *   Broker 启动一个后台线程 `PullRequestHoldService`，每隔 5 秒检查一次挂起的请求。
    *   同时，当有新消息写入该 Queue 时，会通过 `notifyMessageArriving` 立即唤醒挂起的请求。
    *   如果在默认 **30 秒** 内仍无新消息，则返回空响应，避免客户端一直 Hang 住。

```text
Consumer                  Broker (PullMessageProcessor)
   │                              │
   │ ─── Pull Request (无消息) ──> │
   │                              │ (1) 检查是否有消息
   │                              │ (无) Suspend Request (挂起请求)
   │                              │     ↓
   │                              │  ┌─────────────────────┐
   │                              │  │ PullRequestHoldSvc │ (每5s扫描 或 新消息触发)
   │                              │  └─────────┬───────────┘
   │                              │             │
   │(等待中...)                   │             │ (发现新消息 或 超时30s)
   │                              │             ↓
   │ <──────── Response ────────── │ (2) Wakeup & Process
   │   (消息或空)                  │
```

**优点**：在无消息时，连接不立即断开，而是保持等待，直到有消息或超时，从而实现了准实时的拉取，同时避免了无效的频繁请求。

### 2. Kafka 的长轮询机制
Kafka 的消费者在拉取时，如果设置 `fetch.wait.max.ms`（默认 0，但通常设置为几百毫秒），也会进入长轮询。

**工作流程**：
1.  Consumer 发送 FetchRequest。
2.  Kafka Broker 的 `FetchRequest` 处理线程收到请求。
3.  如果有足够的数据（达到 `fetch.min.bytes`），立即返回。
4.  如果数据不足，线程会 **阻塞** 等待，直到：
    *   数据累积达到 `fetch.min.bytes`。
    *   等待时间达到 `fetch.max.wait.ms`。
    *   有新数据写入。

**区别**：RocketMQ 的挂起检测有独立的线程和唤醒机制，而 Kafka 更依赖于底层的 Selector 和 Accumulator 的数据满足条件来唤醒等待的 I/O 线程，或者基于超时自动返回。

### 3. 实战案例与对比

**实战案例**：
在高并发秒杀场景下，如果 RocketMQ 长轮询配置不当，可能出现“消费平推”现象。例如，默认长轮询挂起时间较长，但当瞬时流量洪峰到达时，如果消费者处理能力不足，Broker 侧会积压大量挂起请求，导致客户端连接数飙升，甚至触发 Broker 的连接池满载。优化手段是将 Consumer 端的 `pullBatchSize` 调小，并合理设置 Broker 的 `maxPullBatchSize`，防止一次拉取过多导致处理阻塞。

**关键代码**：
RocketMQ Consumer 默认配置（Java）
```java
// 设置长轮询超时时间，默认 30s，过大可能导致连接hang住不释放
consumer.setPullTimeOutMills(30000); 
// 设置长轮询核心开关，默认开启
consumer.setPullInterval(0); 
// 实际上 Broker 端 PullRequestHoldService 依赖此配置决定挂起策略
```

**对比表格**：

| 特性 | RocketMQ 长轮询 | Kafka 长轮询 |
| :--- | :--- | :--- |
| **实现机制** | Broker 独立线程 `PullRequestHoldService` 轮询 + 事件唤醒 | `DelayedOperationPurgatory` 延迟操作 + Selector 网络层等待 |
| **唤醒条件** | 新消息写入 OR 超时 OR 定时扫描(5s) | 数据量达到 `min.bytes` OR 超时 OR 连接断开 |
| **默认超时** | 默认挂起 30s（可通过 client/broker 配置调整） | 默认 0 (立即返回)，通常设置 500ms-1000ms |
| **资源消耗** | 线程上下文切换相对较多（Reactor模型+后台线程） | 基于 NIO Selector，空闲态消耗极低 |
| **适用场景** | 适用于业务低延迟推送、订单状态变更等强实时场景 | 适用于日志收集、流计算的大吞吐量场景 |

## 常见考点
1.  **长轮询与短轮询的区别？**
    *   短轮询：无论是否有数据，立即返回，无数据则 Client 立刻再次发起请求（高延迟、高消耗）。
    *   长轮询：无数据时 Hold 住请求，等待一段时间或直到有数据再返回（低延迟、低消耗）。
2.  **RocketMQ 长轮询超时时间是多少？**
    *   默认由 Broker 端 `shortPollingTimeMills`（1秒）和 Client 配置决定，最长挂起通常为 30 秒（由 `DefaultMessageStore` 的 `messageStoreConfig` 控制）。

## 记忆要点

- 本质是披着Push外衣的Pull，解决纯拉模式的空轮询与高延迟
- 核心参数：后台线程每 5 秒扫描，请求默认挂起 30 秒超时
- 双唤醒机制：新消息写入立即唤醒 OR 定时扫描超时唤醒
- RocketMQ 靠独立线程扫描，而 Kafka 依赖底层 Selector 和数据量阻塞
