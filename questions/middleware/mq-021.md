# 比 RocketMQ 更好的事务消息实现是什么

这个问题探讨的是分布式事务中“数据库操作”与“消息发送”的原子性顺序问题。

**1. 核心痛点：MQ 与 DB 的写入顺序**
在分布式事务场景下（如下单后发通知），核心难点在于如何保证 MySQL 事务提交与 MQ 消息发送的一致性。

**2. RocketMQ 方案：半消息机制（先 MQ 后 DB）**
RocketMQ 采用“半消息”机制来解决，其核心流程如下：
1.  **发送半消息**：生产者向 Broker 发送一条“半消息”（对消费者不可见）。
2.  **执行本地事务**：半消息发送成功后，生产者执行本地数据库事务（如 MySQL 插入）。
3.  **提交/回滚**：根据本地事务结果，向 Broker 发送 Commit 或 Rollback 请求。
4.  **消息投递**：Broker 收到 Commit 后，将半消息转为正常消息，供消费者消费。

**局限性与风险**：
这种模式强依赖于 Broker 的可用性。**第一步是发送半消息**，如果此时 MQ 集群故障或网络抖动，导致半消息发送失败，生产者就无法继续执行后续的本地数据库事务。这会导致“数据库明明是活着的，但业务无法进行”的尴尬局面，即 MQ 的故障反向阻塞了主业务流程。

**3. “更好”的实现思路：本地消息表（先 DB 后 MQ）**
为了解决 MQ 故障阻塞业务的问题，更健壮的方案（如 QMQ 或基于本地消息表的实现）采用“先写数据库，再发 MQ”的策略。

*   **原理**：利用本地数据库的事务能力，将“业务操作”和“消息记录”写入同一个本地数据库事务中。
*   **优势**：业务执行不依赖 MQ。只要本地事务提交成功，业务就算完成，即使 MQ 挂了，消息也已持久化在本地数据库中，等待后台任务补偿发送。

#### 实战案例：电商核心链路解耦
在“双11”大促场景下，订单服务是核心。如果采用 RocketMQ 半消息方案，一旦 MQ 集群抖动，下单接口将超时失败，导致交易额暴跌。而采用本地消息表方案，即使 MQ 挂了，用户依然能成功下单，订单落库成功，后续由补偿进程慢慢把消息发出去，保证了核心交易的高可用，最终达到数据一致性。

#### 代码示例（Java + JDBC）
```java
@Transactional // 开启本地数据库事务
public void orderService(Order order) {
    // 1. 执行业务操作：插入订单
    orderMapper.insert(order);
    
    // 2. 插入消息记录到本地消息表（同一事务）
    // 包含：topic, 消息体(JSON), 状态(0-待发送), 重试次数
    localMessageMapper.insert("order_topic", JSON.toJSONString(order));
    
    // 3. 事务提交后，Spring AOP 会触发提交，DB 持久化成功
    // 此时 DB 中有了订单记录和待发送消息记录
}
```

#### 对比表格

| 特性 | RocketMQ (半消息) | 本地消息表 (先DB后MQ) |
| :--- | :--- | :--- |
| **执行顺序** | 先发半消息 -> 后执行事务 | 先执行事务 -> 后发消息 |
| **对 MQ 依赖** | 强依赖（MQ 挂则业务挂） | 弱依赖（MQ 挂业务不受影响） |
| **实现复杂度** | 低（无需建表，依赖 Broker） | 中高（需建表、定时任务、幂等） |
| **一致性延迟** | 低（实时发送） | 中（依赖定时轮询，通常秒级） |
| **适用场景** | 通用、非核心强依赖链路 | 核心高可用业务、金融级场景 |

```text
│  Producer   │ ────────────────────>  │   Database   │
└──────┬──────┘                        └──────┬───────┘
       │                                      │

       ^                          │ 扫描失败  │
       │                          │ 记录重试  │
       └──────────────────────────┘           │
```

## 常见考点
1.  **RocketMQ 反查机制**：如果生产者发送 Commit 请求失败怎么办？（Broker 会主动回调生产者查询本地事务状态）。
2.  **消息丢失风险**：本地消息表方案中，事务提交后发送消息前应用挂了怎么办？（依赖定时任务补偿，保证最终一致性）。
3.  **并发问题**：定时任务扫描消息表时，如何避免重复发送？（需设计状态机和幂等处理）。

## 结构化回答

**30 秒电梯演讲：** MQ 写入顺序决定了系统对 MQ 故障的敏感度。打个比方，RocketMQ 先约再办事，MQ 挂了事就办不成；理想的应该先办事再约。

**展开框架：**
1. **顺序决胜负** — RocketMQ 先 MQ 后 DB；更优解是先 DB 后 MQ(本地消息表)
2. **RocketMQ 痛点** — MQ 故障会反向阻塞核心主业务链路，导致不可用
3. **本地消息表优势** — 业务与消息记录同处一个 DB 事务，MQ 挂了业务照常
