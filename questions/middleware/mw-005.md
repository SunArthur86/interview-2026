# RabbitMQ中Fanout类型的Exchange是什么？

Fanout 是 RabbitMQ 中 Exchange（交换机）的一种类型，特点是把收到的消息**广播（broadcast）**给所有绑定到它的 Queue，完全忽略 routing key。

**工作流程：**
1. Producer 把消息发给 Fanout Exchange（不需要指定 routing key，指定了也被忽略）。
2. Exchange 把消息的副本投递给**每一个**绑定（binding）到它的 Queue。
3. 每个 Queue 的消费者各自处理一份。

**与其它 Exchange 类型对比：**
| 类型 | 路由方式 | 场景 |
|------|---------|------|
| **direct** | routing key 完全匹配 | 点对点定向 |
| **fanout** | 广播给所有绑定 Queue | 群发、通知、日志广播 |
| **topic** | routing key 模式匹配（星号匹配单词、井号匹配多词） | 灵活订阅 |
| **headers** | 匹配消息 header（非 routing key） | 复杂条件 |

**典型场景：**
- 系统通知：一条消息同时发给日志、监控、业务多个服务。
- 发布订阅：多个消费者各自处理同一份事件（如订单创建后通知库存、积分、通知服务）。

**4. 实战深化**

*   **实战案例**：在构建分布式链路追踪系统时，使用 Fanout Exchange 将埋点日志数据同时分发到 Elasticsearch（用于实时搜索分析）、HDFS（用于离数仓归档）和告警服务（用于实时异常检测）。相比于每个服务都订阅同一个 Topic，Fanout 解耦了生产者和下游依赖。

*   **代码示例**：
```java
// Java (Spring AMQP): 声明 Fanout Exchange 并绑定
@Bean
public FanoutExchange fanoutExchange() {
    return new FanoutExchange("logs.fanout");
}

@Bean
public Binding bindingLogA(Queue queueLogA, FanoutExchange fanoutExchange) {
    return BindingBuilder.bind(queueLogA).to(fanoutExchange);
}

@Bean
public Binding bindingLogB(Queue queueLogB, FanoutExchange fanoutExchange) {
    return BindingBuilder.bind(queueLogB).to(fanoutExchange);
}
```

*   **Exchange 选型决策表**：

| 需求维度 | Direct Exchange | Topic Exchange | Fanout Exchange |
| :--- | :--- | :--- | :--- |
| **路由复杂性** | 低 (1:1 或 N:1) | 高 (支持通配符) | 无 (广播 N:N) |
| **性能损耗** | 最低 | 中等 (需匹配模式) | 低 (无匹配计算) |
| **灵活性** | 僵化 (需预知 Key) | 极高 (动态订阅) | 弱 (全收或全不收) |
| **Routing Key** | 必须且精确 | 必须且符合模式 | **忽略** |
| **典型业务** | 明确的任务分发 | 多级分类/地域消息 | 全局广播/日志解耦 |

## 记忆要点

- 核心定义：Fanout忽略routing key，将消息原样广播给所有绑定的Queue
- 模式对比：Direct需精确匹配Key，Topic支持通配符匹配，Fanout是纯无脑全量分发
- 典型场景：因为全量复制且解耦生产者，所以最适合用于全局广播、日志分发或事件的多下游订阅
