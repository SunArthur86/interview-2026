# Kafka消费者的设计原理和消费模型是什么？

Kafka 消费者基于 **Consumer Group**（消费者组）模型，采用 **Pull（拉取）** 模式消费数据。

**核心原理：**
1.  **消费组逻辑**：组内每个消费者负责消费 Topic 下不同 Partition 的数据，一个 Partition 只能被组内一个消费者消费（互斥），实现负载均衡。
2.  **Pull 模式**：消费者主动从 Broker 拉取消息，由消费者控制拉取速率和批处理量。
3.  **Offset 管理**：消费者通过提交 offset 标记消费进度，支持自动提交和手动提交。
4.  **Rebalance 机制**：当组内消费者数量变化时，触发重平衡，重新分配 Partition 与消费者的对应关系。

**消费模型架构图：**
```text
Topic (3 Partitions)
├── P1 ────────────────────┐
├── P2 ────────────┐       ├──> Consumer Group A
└── P3 ────┐       │       │   (C1 消费 P1)
           │       │       │   (C2 消费 P2)
           ├──> Consumer Group B  (C3 消费 P3)
           │       │       │   (注意：不同组互不影响)
           └──> ────┘       └──> 独立 Offset 提交
```

### 实战案例

*   **Rebalance 风暴**：在微服务架构中，由于 Spring Boot 应用启用了健康检查，GC 频繁导致 `max.poll.interval.ms` 超时，引发消费者组频繁 Rebalance。这导致所有消费者停止工作，全链路消费阻塞数分钟。解决方案是将心跳线程与消费处理线程分离，并适当调大超时时间或优化业务处理逻辑。
*   **数据丢失与重复消费**：在处理“订单支付”消息时，如果业务逻辑执行成功但 Offset 提交前机器宕机，重启后会重复消费导致多次扣款（未做幂等）。反之，如果先提交 Offset 再处理业务，处理失败时数据即丢失。实战中必须依赖数据库唯一索引或 Redis 实现幂等性来处理“至少一次”语义。

### 关键代码示例（Java：手动提交与幂等处理）

```java
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        // 实战关键：利用 Redis/setnx 实现幂等处理
        String key = "order_pay:" + record.key();
        boolean isProcessed = redis.setnx(key, "1");
        if (isProcessed) {
            processOrder(record.value()); // 处理业务逻辑
            redis.expire(key, 86400);
        }
    }
    // 实战关键：处理完毕后再手动提交 Sync，防止丢失
    consumer.commitSync(); 
}
```

**补充关键细节：**
*   **心跳机制与线程分离**：新版 Kafka 客户端将心跳和消息处理分离。心跳由后台独立线程负责发送，避免了消息处理耗时过长导致消费者被 Group Coordinator 误判为“死掉”而触发 Rebalance。
*   **消息交付语义**：
    *   **At-Least-Once（至少一次）**：开启自动提交 Offset 或手动同步提交。可能重复消费，但不会丢失。
    *   **At-Most-Once（至多一次）**：在消费前提交 Offset。可能丢失，但不会重复。
    *   **Exactly-Once（精确一次）**：需要配合 Kafka 事务机制或幂等性消费逻辑实现。
*   **分区分配策略**：Rebalance 时通过 `PartitionAssignor` 策略分配，主要包括 `Range`（范围，默认，可能导致不均衡）、`RoundRobin`（轮询）和 `Sticky`（粘性，尽量保留原有分配关系以减少重平衡开销）。
*   **Backpressure（反压）**：Pull 模式的天然优势。如果消费者处理能力下降，它可以降低拉取频率或减少拉取大小，从而保护自身不被压垮。
