# 消息刷盘机制

消息刷盘机制是指消息从内存写入磁盘的过程，是消息可靠性的最后一道防线。

### 刷盘策略
RocketMQ 提供两种刷盘策略，通过配置 `flushDiskType` 指定：

1.  **ASYNC_FLUSH（异步刷盘，默认）**
    -   **流程**：消息写入 CommitLog 对应的内存映射文件后，立即返回成功给 Producer。后台线程通过 `FlushCommitLogService` 按照配置的时间间隔（如 500ms）或页缓存积累量异步执行刷盘。
    -   **优点**：吞吐量极高，延迟低。
    -   **缺点**：如果 Broker 宕机、断电或 kill -9，内存中未刷盘的数据会丢失。

2.  **SYNC_FLUSH（同步刷盘）**
    -   **流程**：消息写入内存后，必须等待真正刷入磁盘成功，才能向 Producer 返回成功。
    -   **优点**：可靠性极高，即使 Broker 宕机，只要刷盘成功消息就在磁盘上。
    -   **缺点**：吞吐量较低，延迟较高（受限于磁盘 IO）。

### 实战案例
某支付系统在旧版本中使用 ASYNC_FLUSH，遭遇机房突然断电，导致约 500ms 的交易数据丢失，造成账目不平。后续升级为 SYNC_FLUSH + 主从同步双写，彻底解决了单点数据丢失问题，但 TPS 从 5 万下降至 2 万。

### 代码示例 (Java)
```java
// GroupCommitRequest 用于同步刷盘的等待机制
public class GroupCommitRequest {
    private final long nextOffset;
    private final CountDownLatch countDownLatch = new CountDownLatch(1);

    public void wakeupCustomer(final long returnCode) {
        this.flushResult = returnCode;
        this.countDownLatch.countDown(); // 唤醒等待的发送线程
    }
}
```

### 可靠性边界分析
针对不同的故障场景，刷盘策略的表现如下：

| 故障场景 | SYNC_FLUSH (同步刷盘) | ASYNC_FLUSH (异步刷盘) |
| :--- | :--- | :--- |
| 1. Broker 进程正常 kill | 不丢失 | 可能丢失少量内存数据 |
| 2. Broker 突然崩溃/OOM | 不丢失 | 可能丢失 |
| 3. 机器掉电/断电 | 不丢失 | 可能丢失 |
| 4. 磁盘损坏 | **无法解决** (依赖副本) | **无法解决** (依赖副本) |

**结论**：
- 同步刷盘保证单机层面的数据不丢失（场景 1-3）。
- 对于场景 4（磁盘损坏）或机器损毁，必须依赖 **主从复制（同步双写）** 来保证数据不丢失。
- 一般金融业务开启 `SYNC_FLUSH` + `SYNC_MASTER`；电商类业务通常开启 `ASYNC_FLUSH` + `ASYNC_MASTER` 追求性能。

### 页缓存与内存映射
RocketMQ 高性能的核心在于对 OS 层机制的利用：

1.  **Page Cache（页缓存）**
    -   OS 对文件的缓存。写入文件实际上是写入 Page Cache，读文件也是先读 Page Cache。
    -   利用 OS 的预读和延迟写入机制，RocketMQ 将复杂的 IO 优化交给 OS 处理。

2.  **Mmap（内存映射）**
    -   将磁盘上的文件映射到虚拟内存地址空间中。
    -   **优势**：减少了数据从“内核态”到“用户态”的拷贝次数。传统 read/write 需要两次拷贝（磁盘->内核->用户，用户->内核->磁盘），而 Mmap 直接在用户态操作映射的内存区域，写入即落入 Page Cache，极大提升了读写效率。

```text
传统 Write 流程:
App Buffer ──(copy)──> Kernel Buffer ──(IO)──> Disk

Mmap Write 流程:
Mapped Memory (User/Kernel Shared) ──(IO)──> Disk
```

## 常见考点
1. **同步刷盘和异步刷盘的性能差距有多大？**
   答：同步刷盘受限于磁盘 IOPS 和寻道时间（虽然也是顺序写），性能通常是异步刷盘的几十分之一甚至更低。
2. **什么是 TransientStorePool？**
   答：为了减少内存拷贝开销，RocketMQ 在开启异步刷盘时，可启用堆外内存缓冲区。消息先写入堆外内存，再写入 Mmap。这样利用了 CPU 的批量指令优化，进一步减少对 JVM GC 的压力。
