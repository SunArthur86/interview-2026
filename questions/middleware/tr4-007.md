# Kafka 事务消息是如何实现 Exactly-Once 语义的？

Kafka 通过幂等性、事务协调器及控制消息实现生产者 Exactly-Once，通过将消费位点与业务结果绑定实现端到端 Exactly-Once。

## 技术原理

- **底层依赖 PID 序列号实现幂等性**：生产者开启 `enable.idempotence=true` 后，Kafka 给每个 Producer 分配一个全局唯一的 PID（Producer ID），每条消息带一个递增的 SequenceNumber。Broker 端按 `<PID, partition, seq>` 去重——重试时即使网络重传，Broker 看到 seq 已存在就丢弃，从而保证单分区内不重复。
- **依赖 Transaction Coordinator 和控制消息，实现跨分区的事务原子性**：跨分区/多 producer 的原子写入由 Transaction Coordinator（TC）管理。Producer 开启事务后发消息前向 TC 注册（`InitProducerId`），TC 维护事务状态；消息正常写入目标分区，同时 TC 写事务日志（`transaction_log` topic）。提交时 Producer 发 `EndTxn`，TC 写入 COMMIT 或 ABORT 的**控制消息**（ControlBatch/Marker）到所有相关分区，消费者通过控制消息判断事务是否完成。
- **端到端语义：必须将下游消费位点提交与业务结果绑定在同一个事务中**：消费端的 Exactly-Once 要求消费者用 `read_committed` 隔离级别（只读已提交事务消息），且把"业务处理 + 提交 offset"放在**同一个 Kafka 事务**里——即把 offset 当作普通的 Kafka 消息发到 `__consumer_offsets`，与业务消息一起 commit/abort。这样要么业务+offset 都生效，要么都回滚，避免"处理成功但 offset 没提交导致重复消费"。

## 代码示例

生产者事务（跨分区原子写入）：

```java
props.put("transactional.id", "my-tx-id-1");   // 必须设，开启事务
props.put("enable.idempotence", "true");        // 幂等性自动开启
Producer<String, String> producer = new KafkaProducer<>(props);
producer.initTransactions();                    // 向 TC 注册，回收僵尸实例

try {
    producer.beginTransaction();
    producer.send(new ProducerRecord<>("topic-A", "k1", "v1"));
    producer.send(new ProducerRecord<>("topic-B", "k2", "v2"));
    // 把消费 offset 也纳入事务
    producer.sendOffsetsToTransaction(
        Collections.singletonMap(
            new TopicPartition("source-topic", 0),
            new OffsetAndMetadata(consumedOffset)),
        "source-group");
    producer.commitTransaction();                // 原子提交：消息 + offset
} catch (Exception e) {
    producer.abortTransaction();
}
```

消费者读已提交：

```java
props.put("isolation.level", "read_committed");  // 只读已 commit 的消息
KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
// 消费者不会看到未提交事务的消息，也不会看到 abort 的消息
```

## 两阶段提交流程

```
Producer ──InitProducerId──> Transaction Coordinator (TC)
                              ↓ 记录 txn 状态
Producer ── Produce ────────> Partition A, B  (with PID+seq)
Producer ── EndTxn(commit) ─> TC
                              ↓ 写 ControlMarker 到 A、B
                              ↓ 标记事务 COMPLETED
Consumer(read_committed) ──> 读到 COMMIT marker → 消息可见
                            读到 ABORT marker → 消息跳过
```

## 常见坑/注意事项

- **transactional.id 必须稳定且唯一**：它是 TC 识别"同一个 producer 跨重启"的依据，固定字符串可让 TC 用 epoch 驱逐僵尸实例；但同一 id 不能多实例共用，否则互相踢。
- **幂等性要求 `acks=all` 和 `retries>0`**：开启 idempotence 会强制这两个，否则启动报错。
- **`read_committed` 性能下降**：消费者要缓冲未提交事务的消息，长事务会拖慢消费。事务要短，避免几分钟级别的大事务。
- **端到端 EOS 要求"读 Kafka-处理-写 Kafka"全闭环**：如果下游是 DB 或外部系统，Kafka 事务管不到 DB 写入，端到端就降级为 At-Least-Once，需业务侧幂等。
- **PID 回收与 epoch**：Producer 重启后 PID 相同但 epoch+1，老 epoch 的实例发送会被拒绝（防僵尸），生产者代码异常退出要确保下次启动用相同 transactional.id。
