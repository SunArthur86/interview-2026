# Kafka为什么要抛弃Zookeeper

### Kafka为什么要抛弃Zookeeper

Kafka 早期架构强依赖 ZooKeeper，ZK 充当了分布式协调者、元数据存储和集群管理者的角色。

## 1. ZooKeeper 的核心机制

ZooKeeper 是一个提供强一致性的分布式协调服务，核心特性包括：

- **ZNode（数据节点）**：类似文件系统的树状结构。
  - **持久节点**：节点创建后一直存在，直到显式删除。
  - **临时节点**：客户端会话失效后，节点自动删除。这是实现**故障检测**的关键。
- **Watcher 机制**：客户端可以监听 ZNode 的变化。一旦节点数据变更或被删除，ZK 会主动通知客户端，实现**发布-订阅**模式。

## 2. ZooKeeper 在 Kafka 中的角色

Kafka 利用 ZK 的上述特性实现了以下功能：

### A. Controller 选举
- **机制**：所有 Broker 启动时，都会尝试在 ZK 上创建临时节点 `/controller`。
- **胜出者**：第一个成功创建节点的 Broker 成为集群 Controller。
- **故障转移**：Controller 宕机会导致 Session 超时，临时节点消失。其他 Broker 通过 Watcher 收到通知，立即发起新一轮抢注，选出新的 Controller。

### B. 元数据存储
- **Broker 注册**：每个 Broker 启动时在 `/brokers/ids` 下创建临时节点，注册自己的 IP、端口、Rack 信息。
- **Topic 注册**：Topic 的分区数、副本分配方案、ISR 列表等配置信息存储在 `/brokers/topics` 下。
- **Consumer 旧版位移**：早期版本中，Consumer 的消费 Offset 存储在 ZK 中（如 `/consumers/[group_id]/offsets`）。

### C. 权限管理 (ACL)
- 用户的创建、删除以及 Topic 的读写权限控制列表存储在 ZK 中。

#### 3. 实战深化

##### 实战案例：ZK Session 过期导致的集群分裂
某次 ZK 集群进行 Full GC，导致 Broker 与 ZK 的心跳超时。Broker 的临时节点被删除，触发 Controller 重新选举。然而，旧 Controller GC 恢复后并未意识到自己已下线，形成了“脑裂”局面：两个 Controller 同时向不同 Broker 发送互斥的 Leader 选举指令，导致数据不一致。**根本原因**：过度依赖 ZK 的 Session 超时作为存活判定。

##### 关键代码
```java
// KafkaController.scala (Controller 选举逻辑简化)
activeControllerId = zkClient.getControllerId()
if (activeControllerId.isDefined) {
  // 已存在 Controller，不作为
} else {
  try {
    // 尝试创建临时节点 /controller
    val (epoch, zkVersion) = zkClient.registerControllerAndIncrementControllerEpoch(config.brokerId)
    // 成功则成为 Controller
    onControllerFailover()
  } catch {
    case e: ZkNodeExistsException => // 节点已存在，选举失败
  }
}
```

##### ZK 存储元数据 VS Raft 内部存储对比

| 特性 | ZooKeeper 模式 (KRaft 之前) | KRaft 模式 (ZK 之外) |
| :--- | :--- | :--- |
| **存储位置** | 外部 ZK 集群 | 内部 `__cluster_metadata` Topic (Partition 0) |
| **Controller 状态** | ZK 临时节点 (/controller) | Quorum Controller (内部状态机) |
| **元数据查询路径** | Broker -> ZK (Read) | Broker -> Quorum Controller (RPC) |
| **扩容复杂度** | 需同时扩 ZK 和 Kafka | 仅需扩 Kafka Broker/Controller |
| **故障检测** | ZK Session Timeout (TCP) | 基于 Raft 的心跳与投票机制 |
| **元数据一致性** | ZK 强一致性 | 基于 Raft Log 的强一致性 |

## 记忆要点

- 核心痛点：ZK不适合元数据频繁写入，且海量分区下ZK推送易造成网络与Controller雪崩
- 扩展瓶颈：单Controller处理元数据，在万级Partition规模下成为性能与可用性瓶颈
- KRaft替代：内部基于Raft协议将元数据存入内部Topic(__cluster_metadata)
- 架构收益：去ZK不仅简化运维，更打破Controller单点瓶颈，支持单集群百万级分区
