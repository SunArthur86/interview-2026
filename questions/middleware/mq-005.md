# 如何保证消息不丢失

保证消息不丢失是消息队列（MQ）最核心的可靠性问题，需要从**生产者**、**Broker（服务端）**、**消费者**三个阶段全链路保障。以下是详细的解决方案及架构流程：

### 一、架构流程图

```text
      生产者                     Broker (服务端)                      消费者
   ┌─────────┐              ┌──────────────────────┐             ┌──────────┐
   │ Producer│              │     Topic / Queue    │             │ Consumer │
   └────┬────┘              └──────────┬───────────┘             └────┬─────┘
        │                              │                               │
        │ 1. 发送消息                  │                               │
        ├─────────────────────────────>│                               │
        │                              │ 2. 写入 PageCache             │
        │                              │                               │
        │          [可选: ACK确认]     │ 3. 刷盘/同步副本              │
        │<─────────────────────────────┤                               │
        │                              │                               │
        │                              │ 4. 推送/拉取消息              │
        │                              ├──────────────────────────────>│
        │                              │                               │
        │                              │                      5. 业务处理 & 提交 Offset
        │                              │                               │
        │                              │                   6. 手动 ACK
        │                              │<──────────────────────────────┤
```

### 二、详细解决方案

#### 1. 生产者阶段：确保发送成功
生产者发送消息面临网络抖动或Broker宕机的风险，需配置以下机制：

*   **发送机制**：
    *   **同步发送**：发送后阻塞等待Broker响应。虽然性能较低，但可靠性最高，能确切知道发送结果。
    *   **异步发送+回调**：发送后不阻塞，通过回调接口处理Broker的响应。必须捕获回调中的异常。
*   **重试机制**：当收到网络错误或Broker返回写入失败时，必须进行重试。
    *   *注意*：重试可能导致消息重复，需要结合消费端的幂等性设计。
*   **异常处理**：对于多次重试失败的消息，需要进行记录日志和报警，甚至落地到本地数据库或文件，通过定时任务兜底恢复。

#### 2. Broker 存储阶段：确保落盘与多副本
Broker收到消息后，必须在数据持久化完成前告知生产者失败，否则断电会导致内存数据丢失。

*   **刷盘策略**：
    *   **ASYNC_FLUSH（异步刷盘）**：高性能，但可能丢数据（消息写入PageCache即返回成功，断电丢失）。
    *   **SYNC_FLUSH（同步刷盘）**：低性能，高可靠（消息写入物理磁盘后才返回成功）。RocketMQ 推荐配置。
*   **集群同步/复制**：
    *   **同步复制**：Master 收到消息后，等待 Slave 也写入成功再回复生产者。可靠性高，但吞吐低。
    *   **异步复制**：Master 收到消息立即回复，后台异步同步给 Slave。高性能，但 Master 宕机可能导致少量数据丢失。

#### 3. 消费者阶段：确保处理成功
*   **手动提交 Offset**：关闭自动提交，在业务逻辑执行成功后再手动提交 Offset。防止业务报错但 Offset 已提交导致的“假消费成功”。
*   **消费重试**：捕获异常，返回 `RECONSUME_LATER` (RocketMQ) 或不发送 ACK (Kafka/RabbitMQ)，让消息重新入队再次消费。

### 深化补充

**【实战案例】**
*   **踩坑**：某金融系统在运行过程中，Broker 磁盘写入虽配置为“异步刷盘”以追求高TPS，但在一次机房瞬间断电事故中，内存中 PageCache 尚未落盘的近千条交易流水丢失，导致账目严重不平。后续整改为“同步刷盘+同步复制”的 Broker 集群模式，虽然 TPS 下降 30%，但满足了金融级数据零丢失要求。

**【配置对比表格（以 RocketMQ 为例）】**

| 配置维度 | 可靠性优先方案 | 性能优先方案 | 选择建议 |
| :--- | :--- | :--- | :--- |
| **刷盘方式** | SYNC_FLUSH (同步刷盘) | ASYNC_FLUSH (异步刷盘) | 核心业务选同步，日志/非核心选异步 |
| **主从复制** | SYNC_MASTER (同步复制) | ASYNC_MASTER (异步复制) | 防止单点故障丢失，选同步 |
| **发送方式** | SYNC (同步发送) | ASYNC (异步发送) | 不可丢数据选同步，流量洪峰选异步 |
| **Offset提交** | 手动提交 | 自动提交 | 必须选手动提交 |

**【关键代码示例 (Kafka Producer 配置)]**
```java
Props props = new Properties();
// 设置 acks=all (或 -1)，表示 leader 和所有 isr follower 都确认收到才算成功
props.put("acks", "all");
// 开启重试，次数设为 3
props.put("retries", 3);
// 当启用 retries 时，如果客户端想要禁止乱序，需配置 max.in.flight.requests.per.connection=1
// 或者开启幂等性 enable.idempotence=true (推荐)
props.put("enable.idempotence", "true");
```
