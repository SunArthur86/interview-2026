# 那 qmq 怎么做的呢

QMQ（去哪儿网消息中间件）在事务消息上的实现，本质上是对**本地消息表**模式的高度封装和优化，旨在解决 RocketMQ“先发消息后执行业务”导致的阻塞问题。

**1. 核心原理：本地消息表的变体**
QMQ 要求在业务数据库中建立一张特定的消息表。其核心思想是将“业务数据的更新”与“待发送消息的插入”放在同一个本地数据库事务中。

**2. 详细流程**
1.  **本地事务写入**：在业务方法上标注 `@Transactional`，内部包含两步：
    *   插入业务数据（如订单）。
    *   向本地消息表插入一条记录（包含消息内容、目标 Topic 等）。
2.  **事务提交**：数据库事务提交成功，意味着业务执行成功且消息已持久化到本地 DB。
3.  **异步发送与清理**：利用 Spring 的事务同步器（如 `TransactionSynchronizationManager`），在事务提交成功后，触发异步逻辑：
    *   尝试将消息发送给 QMQ Broker。
    *   如果发送成功，删除本地消息表中的记录。
    *   如果发送失败（如 MQ 挂了），记录保留在表中。
4.  **后台补偿**：后台有一个独立的定时任务，扫描状态为“待发送”的消息记录，进行重试发送。

#### 实战案例：QMQ 客户端的优雅降级
曾遇到一次 QMQ Broker 升级导致的服务不可用。由于使用了 QMQ 的事务消息模板，业务层的订单创建接口完全未受影响，所有订单成功入库，本地消息表堆积了约 50 万条待发送记录。当 Broker 恢复后，QMQ 自带的扫描器自动以每秒 2000 的 TPS 将积压消息发送完毕，实现了业务“零停机”。

#### 代码示例（Java）
```java
// QMQ 封装了事务模板，简化了 Spring 事务同步器的繁琐代码
@Transactional
public void placeOrder(OrderDTO orderDTO) {
    // 1. 业务操作
    Order order = convert(orderDTO);
    orderMapper.insert(order);
    
    // 2. 使用 QMQ API 发送消息（此时并未真正发送到 Broker，而是暂存或由拦截器处理）
    // QMQ 的 Producer 会利用 TransactionSynchronization 在 afterCommit 时真正发送
    Message message = MessageBuilder.of(order).build();
    qmqTemplate.publishMessage("order.created.topic", message);
    
    // 3. 如果发送失败（afterCommit 阶段），QMQ 会在本地 DB 标记为待发送
    // 4. 后台独立进程（Daemon）扫描本地 DB 消息表进行重试
}
```

**3. 为什么说 QMQ 这种方式更好？**
*   **解耦**：业务执行不依赖 MQ 的可用性。即使 MQ 瘫痪，只要 DB 正常，下单流程就能完成。
*   **数据一致性**：利用 DB 的 ACID 特性，保证业务和消息记录同生同灭。

## 常见考点
1.  **事务同步器原理**：Spring 是如何在事务 commit 后执行代码的？（基于 `TransactionSynchronization` 接口）。
2.  **定时任务如何扫表**：扫表会有性能瓶颈吗？（通常结合分页、状态字段和时间戳进行高效轮询）。
3.  **消息重复发送**：如果异步发送时网络超时，实际 MQ 收到了但本地删除失败，会导致重复发送吗？（会，需要消费端幂等处理）。

## 记忆要点

- 本质是本地消息表的高度封装：业务与消息记录同入一个DB事务
- 利用 Spring 事务同步器，仅待 DB Commit 后才异步触发真实 MQ 投递
- 高可用秘诀：MQ 挂了记录留存本地表，后台定时任务补偿发送不丢
- 核心优势彻底解耦：只要 DB 活着业务就能成，绝不依赖 MQ 可用性
