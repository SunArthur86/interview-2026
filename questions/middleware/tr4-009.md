# 如何利用布隆过滤器解决缓存穿透问题？

缓存穿透是指查询一个根本不存在的数据，导致请求直接穿过缓存打到数据库。布隆过滤器是一种空间效率极高的概率型数据结构，用于判断一个元素是否在一个集合中。其特点是存在误判（可能判断存在但实际不存在），但不会漏判（判断不存在则绝对不存在）。解决方案是在访问缓存前，先查询布隆过滤器。如果布隆过滤器判断 Key 不存在，则直接返回空，无需查询缓存或数据库。布隆过滤器可以通过 RedisBloom 模块或 Guava 库本地加载实现。为了保证准确性，通常需要预加载数据库所有 Key 到布隆过滤器，并设置合适的误判率参数，定期重建过滤器以应对新增 Key。

**实战案例**：在电商“秒杀”场景中，恶意攻击者会构造大量不存在的商品ID发起请求。直接引入布隆过滤器后，DB CPU从90%瞬间降至5%，有效拦截了99%的恶意流量，但需注意必须配合“空值缓存”策略，防止误判击穿DB。

**代码示例 (Java Guava)**：
```java
// 预计插入100万，误判率0.01%
BloomFilter<String> filter = BloomFilter.create(Funnels.stringFunnel(Charsets.UTF_8), 1000000, 0.0001);
// 初始化时加载所有Key
for (String key : getAllKeysFromDB()) { filter.put(key); }

// 请求拦截
public String get(String key) {
    if (!filter.mightContain(key)) { return null; } 
    return redis.get(key);
}
```

**对比表格**：
| 特性 | 布隆过滤器 | 空值缓存 (Cache Null) |
| :--- | :--- | :--- |
| **空间占用** | 极低 (与数据量无关) | 较高 (取决于攻击Key数量) |
| **适用场景** | Key空间巨大，攻击明显 | Key相对固定，少量穿透 |
| **维护成本** | 需定期重建或增量更新 | 自动过期，维护简单 |
| **一致性** | 可能误判 (认为存在) | 强一致性 (绝对存在) |

## 技术原理

布隆过滤器之所以能用极小空间实现海量 Key 的存在性判断，源于其精巧的概率结构：

- **位数组 + 多哈希函数**：布隆过滤器底层是一个长度为 $m$ 的位数组（初始全 0）和 $k$ 个相互独立的哈希函数。写入元素 $x$ 时，计算 $h_1(x), h_2(x), \ldots, h_k(x)$ 这 $k$ 个位置，把位数组对应位置设为 1。查询时同样计算 $k$ 个哈希，若所有位置都是 1 则「可能存在」，若有任一位置为 0 则「绝对不存在」。后者必然正确，因为如果 $x$ 被写入过，那些位必然已被置 1。
- **误判率的数学本质**：当多个元素的哈希位置重叠时，未写入的元素 $y$ 的 $k$ 个哈希位置可能恰好都被其他元素置 1，导致「假阳性」。误判率 $p \approx (1 - e^{-kn/m})^k$，其中 $n$ 是元素数。给定 $n$ 和期望 $p$，可反推最优 $m$ 和 $k$：$m = -n\ln p / (\ln 2)^2$，$k = (m/n)\ln 2$。100 万元素、0.01% 误判率约需 1.8MB 内存——远小于存全量 Key。
- **不支持删除的局限**：标准布隆过滤器的位可能被多个元素共享，直接置 0 会误删其他元素。应对：(1) Counting Bloom Filter（每个位是计数器，删除时减一）；(2) 定期全量重建（适合 Key 集合可枚举的场景，如商品 ID）；(3) Cuckoo Filter（支持删除且空间更优）。
- **与缓存穿透的配合链路**：请求 → 布隆过滤器判断：(1) 不存在 → 直接返回空（拦截 99% 恶意流量）；(2) 可能存在 → 查 Redis 缓存，未命中再查 DB，DB 返回空则写「空值缓存」（TTL 5 分钟）兜底误判。三层防御确保即使布隆误判也不会击穿 DB。

## 代码示例

```java
// Redisson 分布式布隆过滤器（跨实例共享，推荐生产使用）
Config config = new Config();
config.useClusterServers().addNodeAddress("redis://localhost:6379");
RedissonClient redisson = Redisson.create(config);

RBloomFilter<String> productFilter = redisson.getBloomFilter("productFilter");
// 初始化：预计 100 万元素，误判率 0.01%
productFilter.tryInit(1_000_000L, 0.0001);
// 预热：加载所有商品 ID
productDB.findAllIds().forEach(productFilter::add);

// 请求拦截链路
public Product getProduct(String id) {
    // 第一层：布隆过滤器
    if (!productFilter.contains(id)) {
        return null;  // 绝对不存在，直接拦截
    }
    // 第二层：Redis 缓存（含空值缓存）
    String cached = redis.get("product:" + id);
    if ("NULL".equals(cached)) return null;  // 空值缓存命中
    if (cached != null) return JSON.parse(cached);
    // 第三层：查 DB
    Product p = productDB.findById(id);
    if (p == null) {
        redis.setex("product:" + id, 300, "NULL");  // 空值缓存 5 分钟
    } else {
        redis.setex("product:" + id, 3600, JSON.toJSONString(p));
    }
    return p;
}
```

```bash
# RedisBloom 模块原生命令
BF.RESERVE productFilter 0.0001 1000000   # 创建：误判率 0.01%，容量 100 万
BF.ADD productFilter "SKU:1001"            # 添加元素
BF.EXISTS productFilter "SKU:9999"         # 查询：返回 0 绝对不存在，返回 1 可能存在
BF.INFO productFilter                       # 查看容量、已用、误判率
```

## 注意事项

- **新增 Key 的同步问题**：DB 新增数据后必须同步 `BF.ADD`，否则新数据会被布隆过滤器误判为不存在而拦截。建议在写入 DB 的事务中或通过 Binlog 订阅异步同步。
- **容量预估要留余量**：`tryInit(1_000_000)` 设定的容量是上限，超过后误判率会急剧上升。预估实际容量并预留 2~3 倍缓冲，或使用可动态扩容的 Scalable Bloom Filter。
- **分布式一致性**：Redisson 布隆过滤器基于 Redis，多实例共享但受 Redis 延迟影响。若用 Guava 本地布隆，各实例独立，新增 Key 需广播到所有实例，一致性维护复杂。
- **重建窗口的穿透风险**：全量重建期间若下线旧过滤器，会有短暂穿透窗口。平滑方案：双布隆（新旧并行运行一段时间，新过滤器预热完成后切换）。
- **不适用于 Key 频繁删除的场景**：商品下架等删除操作无法反映到标准布隆过滤器。Counting Bloom Filter 或定期重建是应对方案。
