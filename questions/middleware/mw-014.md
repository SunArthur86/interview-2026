# 如何保证MQ中消息的顺序性？

MQ 中消息乱序通常发生在生产者发送、多分区存储、或消费者多线程并行处理阶段。要保证顺序性，需从设计上确保“先进先出”。

### 1. 导致顺序错乱的原因
- **多分区并发**：Topic 有多个 Partition，生产者发送消息 A、B 可能被分发到不同分区，消费者读取时顺序无法保证。
- **消费者并发处理**：即便只有一个 Partition，消费者内部使用多线程处理，由于线程处理速度不同，后处理完的消息可能先写入数据库，导致乱序。

### 2. 保证顺序性的解决方案

**方案一：单一分区 + 单线程消费（最彻底）**
- **Topic 级别**：将 Topic 的 Partition 数设为 1，从物理存储上保证全局有序。
- **消费端**：消费者内部使用单线程顺序处理。
- **缺点**：吞吐量最低，成为性能瓶颈。

**方案二：业务键分区 + 消费者内部排队（推荐）**
- **发送端**：在发送消息时，指定业务 Key（如订单 ID）。Kafka/RabbitMQ 会根据 Hash 将同一订单的所有消息发送到**同一个 Partition** 中，保证存储有序。
- **消费端**：
  1. 多个消费者线程从 Partition 拉取消息。
  2. 拉取后不直接处理，而是根据 Key（如订单 ID） 将消息分发到内存中不同的**本地队列**。
  3. 每个内存队列绑定唯一的 Worker 线程进行处理。
- **效果**：同一业务 Key 的消息由同一个线程顺序处理，不同 Key 的消息并行处理，兼顾了顺序性和性能。

### 常见考点
1. **消息重试导致的乱序**：如果业务处理失败，消息重试时，如何保证重试消息不插队？（通常结合死信队列或业务幂等处理）。
2. **Kafka 发送端的顺序性**：如果生产者使用异步发送且配置了 `retries`，网络抖动可能导致消息乱序，需配置 `max.in.flight.requests.per.connection=1` 或开启幂等生产者（enable.idempotence=true）。
3. **Broker 端的队列数量**：RocketMQ 中如何保证顺序？（通过 MessageQueue 选择算法将相同 orderId 发送到同一个 Queue）。
4. **消费者异常宕机**：消费者消费到一半挂了，Rebalance 后如何保证从断点继续消费而不是重新消费导致乱序？（依赖 Offset 提交机制）。
