# Kafka的消费者组和Rebalance机制是什么？

### 消费者组
一组消费者共同消费一个 Topic 的所有 Partition，实现**负载均衡**。规则：每个 Partition 在同一个消费者组内，只能被**一个**消费者消费（组内单消费，组间多播）。

### Rebalance（重平衡）机制
Rebalance 是消费者组内 Partition 所有权重新分配的过程，期间会导致**Stop-The-World (STW)**，即消费者暂停消费。

#### 1. 触发条件
1.  **消费者数量变化**：新增、宕机、主动退出。
2.  **Topic 变化**：订阅的 Topic Partition 数增加或减少。
3.  **超时**：`session.timeout.ms` 内未收到心跳。

#### 2. 核心组件：Coordinator（协调器）
*   每个 ConsumerGroup 都会找到一个 Broker 作为它的 Coordinator。
*   寻找算法：`Math.abs(groupId.hashCode) % __consumer_offsets.partitionCount`。

#### 3. Rebalance 流程架构图
```text
Consumer 1         Consumer 2          Coordinator         Group Leader
    |                   |                  |                     |
    |---JoinGroup------>|                  |                     |
    |                   |---JoinGroup----->|                     |
    |                   |                  |--- Elect Leader ---->| (选投票数高的)
    |<--SyncGroup-------|                  |<-- Assignments ------| Leader制定分配方案
    | (分配方案)        |<--SyncGroup------|                     |
    |                   | (分配方案)       |                     |
    |--> Start Consume  |--> Start Consume |                     |
```

#### 4. Generation ID（代际机制）
*   Rebalance 后，Generation ID 会增加。消费者提交 Offset 时必须携带正确的 Generation ID，防止“僵尸消费者”提交过期的 Offset。

#### 5. 避免频繁 Rebalance 的参数调优
频繁 Rebalance 会导致吞吐量暴跌和重复消费。

| 参数 | 默认值 | 作用 | 调优建议 |
| :--- | :--- | :--- | :--- |
| `session.timeout.ms` | 45s | Coordinator 多久没收到心跳认为消费者挂了 | 增大 (如 30s，需配合 GC 时间) |
| `heartbeat.interval.ms` | 3s | 消费者发送心跳频率 | 减小 (如 3-5s) |
| `max.poll.interval.ms` | 5min | 两次 poll 的最大间隔时间 | 处理慢时需增大 |

### 6. 常见考点
1.  **Rebalance 期间消息去哪了？**：并没有丢失，而是堆积在 Broker 端，Rebalance 结束后消费者继续拉取，但会增加消费延迟。
2.  **Rebalance 效率低下的原因**：通常是 `max.poll.interval.ms` 设置太小，导致 FullGC 或单次消息处理时间过长，触发 Rebalance，陷入恶性循环（Storm Rebalance）。
3.  **Sticky Assignor（粘性分区分配）**：与 Range/RoundRobin 相比，它在 Rebalance 时会尽可能保留之前的分配关系，减少不必要的 Partition 移动，降低开销。

### 7. 实战案例
**场景**：大促期间由于 FullGC 频繁导致消费者暂停超过 `max.poll.interval.ms`，引发集群级 Rebalance 风暴。**解决**：1. 优化 GC 减少 STW 时间；2. 将 `max.poll.interval.ms` 调大至 `message.process.time.max + buffer`；3. 升级到 Kafka 2.4+ 使用 `CooperativeStickyAssignor`（渐进式重平衡），实现不停止消费的情况下平滑调整分区分配。

### 8. 代码示例
**消费者参数配置**：
```java
properties.put("session.timeout.ms", "30000"); // 30s超时
properties.put("heartbeat.interval.ms", "3000"); // 3s心跳
properties.put("max.poll.interval.ms", "300000"); // 允许5分钟处理时间
// 关键：设置分区分配策略为粘性，减少Rebalance影响
properties.put("partition.assignment.strategy", "org.apache.kafka.clients.consumer.StickyAssignor");
```
