# Controller的请求发送

Controller 在处理完 ZooKeeper 的事件并更新完内存元数据后，需要将决策下发给集群中的 Broker。Controller 只会发送三种特定的请求来实现这一目标。

### 1. LeaderAndIsrRequest
这是最核心的控制请求，用于确定分区的“主从关系”。
- **触发场景**：
  - Controller 刚当选，初始化分区 Leader。
  - Broker 宕机，触发分区 Leader 重新选举。
  - ISR 列表发生变化，需要通知 Leader 调整同步副本。
- **请求参数细节**：
  - `leader`: 分区 Leader 的 Broker ID。
  - `leaderEpoch`: Leader 的纪元（版本号），防止数据滞后的旧 Leader 假死复活后继续提供读写。
  - `ISR`: 当前的同步副本集合。
  - `ZK version`: ZooKeeper 中该分区状态信息的版本号，用于乐观锁校验。

### 2. StopReplicaRequest
用于告诉 Broker 停止某些分区的副本服务。
- **触发场景**：
  - 删除 Topic 时。
  - 分区副本迁移，某个 Broker 不再负责该分区时。
  - Broker 优雅关闭时，Controller 让它停止领导工作并迁移副本。
- **参数细节**：
  - `partitionStates`: 包含要停止的分区列表。
  - `deletePartitions`: 布尔值，指示是否仅仅停止服务还是彻底删除磁盘上的日志文件（通常仅停止，删除由异步任务处理）。

### 3. UpdateMetadataRequest
用于全集群元数据的广播同步。
- **触发场景**：
  - 新建 Topic。
  - 分区 Leader 发生变更（任何 Broker 都需要知道新的 Leader 在哪，以便处理客户端请求）。
  - Broker 上下线。
- **广播机制**：
  - 通常会发送给集群中**所有存活**的 Broker，而不仅仅是相关 Broker。
  - 更新 Broker 内存中的 `MetadataCache`。

### 请求发送流程细节
Controller 的请求发送是由 `ControllerEventThread` 触发的，但实际的网络 I/O 通常是通过底层网络客户端异步完成的，不过 Controller 会确保流程的顺序性。对于关键请求（如 Leader 选举），必须确认收到响应才认为该事件处理完毕。

### 请求流向示意图

```text

     Controller Event Thread
    ┌─────────────────────────────┐
    │ 1. Event: Broker 2 crashed  │
    │ 2. Recalculate State        │
    │    - P0 Leader moves to B1   │
    │    - Remove P1 from B2       │
    └──────────┬──────────────────┘
               │
               │ Build Requests
      ┌────────┴────────────┐
      ▼                     ▼
┌─────────────────┐  ┌──────────────────┐
│ LeaderAndIsrReq │  │StopReplicaRequest│
│ (To Broker 1)   │  │ (To Broker 2)    │◄────┐ (Broker 2 likely
└────────┬────────┘  └──────────────────┘      │  unreachable, may timeout)
         │                  │                 │
         ▼                  ▼                 │
┌─────────────────┐  ┌──────────────────┐      │
│ Broker 1        │  │ Broker 2 (Dead)  │      │
│ Becomes Leader  │  │ (No response)    │      │
└─────────────────┘  └──────────────────┘  
```

#### 💡 实战案例
在删除大 Topic 时，若 Broker 磁盘 I/O 较高，`StopReplicaRequest` 的删除操作可能导致长时间阻塞，进而导致 Controller 认为该 Broker 超时并触发新一轮的不必要 Leader 选举（雪崩效应）。通常建议在业务低峰期执行删除，或调整 `delete.topic.enable` 的策略。

#### 💻 代码片段 (Scala)
```scala
// KafkaController.scala 中发送请求的逻辑简化
val stopReplicaRequest = new StopReplicaRequest.Builder(
  controllerId, controllerEpoch, false, // deletePartitions = false 先停止服务
  stopReplicaPartitionSeq.asJava
).build()

controllerChannelManager.sendRequest(brokerId, stopReplicaRequest, 
  new AbstractControllerRequestSendListener {
    override def onResponse(response: AbstractResponse): Unit = {
      // 只有收到成功响应才标记任务完成
    }
  })
```

#### 📊 请求对比表格

| 特性 | LeaderAndIsrRequest | StopReplicaRequest | UpdateMetadataRequest |
| :--- | :--- | :--- | :--- |
| **核心作用** | 指定主从关系及 ISR | 停止/删除副本数据 | 广播集群元数据快照 |
| **发送目标** | ISR 列表中的 Broker | 特定 Broker | **所有存活 Broker** (广播) |
| **处理优先级** | 最高 (影响读写可用性) | 高 (影响数据一致性) | 中 (影响客户端路由) |
| **失败重试** | 立即重试或触发重选 | 记录失败，稍后异步清理 | 下次事件触发时覆盖更新 |
