# Linux内核参数

为了确保 RocketMQ Broker 的高性能和稳定性，Linux 内核参数和 JVM 参数的调优至关重要。

### 一、JVM 调优参数

1.  **堆内存预分配**：如果不关心 Broker 的启动时间，建议开启 `-XX:+AlwaysPreTouch`。此参数强制 JVM 在启动时将所有堆内存页“触摸”并实际分配给物理内存。虽然增加了启动时间，但避免了运行时因缺页中断导致的延迟抖动。

2.  **锁优化**：RocketMQ 高并发场景下，锁竞争激烈。禁用偏向锁 `-XX:-UseBiasedLocking` 通常能减少撤销偏向锁带来的 JVM 暂停，因为在高并发竞争下，偏向锁反而增加开销。

3.  **垃圾回收器**：JDK 8 推荐使用 G1 收集器。
    *   `-XX:+UseG1GC`：启用 G1。
    *   `-XX:G1HeapRegionSize=16m`：Region 大小设为 16MB，通常匹配 CommitLog 的文件大小，利于大对象分配和扫描。
    *   `-XX:G1ReservePercent=25`：预留 25% 的内存作为存活空间，防止晋升失败。
    *   `-XX:InitiatingHeapOccupancyPercent=30`：当堆占用达到 30% 时触发并发标记。RocketMQ 内存写入量大，调低此阈值可以让 GC 线程更早介入，避免 Full GC。
    *   **注意**：不要将 `-XX:MaxGCPauseMillis` 设置得过小（如 10ms），否则 G1 会过度缩减年轻代，导致极频繁的 Minor GC，适得其反。

4.  **GC 日志**：建议开启滚动日志以便排查问题。
    *   `-XX:+UseGCLogFileRotation`
    *   `-XX:NumberOfGCLogFiles=5`
    *   `-XX:GCLogFileSize=30m`

### 二、Linux 内核参数调优

以下参数是 RocketMQ 官方推荐的内核调优，主要解决文件句柄限制、内存交换延迟和 I/O 瓶颈。

1.  **`vm.min_free_kbytes`**
    *   **作用**：强制系统保留一定数量的可用内存（KB），用于紧急分配。
    *   **建议**：不要低于 1024KB。如果设置过小，系统在高负载下容易触发内存回收死锁或直接 OOM，导致服务不可用。

2.  **`vm.extra_free_kbytes`**
    *   **作用**：在 `min_free_kbytes` 的基础上，增加额外的空闲内存缓冲区。它拉大了内核后台回收启动阈值和直接回收（分配进程）触发阈值之间的差距。
    *   **原理**：RocketMQ 对延迟敏感，增加此参数可避免分配进程被迫进入昂贵的“直接内存回收”路径，从而减少长延迟。

3.  **`vm.max_map_count`**
    *   **作用**：限制一个进程可能拥有的最大内存映射区域数。
    *   **背景**：RocketMQ 使用 `mmap` 加载 CommitLog 和 ConsumeQueue 文件。默认值（通常为 65530）可能不足，导致无法打开新文件。
    *   **建议**：建议调大至 `262144` 或更高（如 655360），以支持海量消息文件的映射。

4.  **`vm.swappiness`**
    *   **作用**：控制内核交换内存页面的积极程度（范围 0-100）。
    *   **建议**：设置为 **10**。RocketMQ 严重依赖 Page Cache 做读写加速，如果发生 Swap（内存交换到磁盘），I/O 性能将呈指数级下降。低数值可最大限度避免 Java 堆内存被换出。

5.  **文件描述符限制**
    *   **作用**：限制进程能打开的文件句柄数。
    *   **建议**：RocketMQ 需要同时打开大量的数据文件和维持大量的网络连接。建议通过 `ulimit -n` 将值设置为 **655350** 或更高。

6.  **磁盘 I/O 调度器**
    *   **建议**：使用 **Deadline** 或 **noop** 调度器。
    *   **原因**：默认的 CFQ 调度器虽然公平，但对高吞吐的消息队列延迟不友好。Deadline 调度器试图为每个 I/O 请求提供有保证的延迟，更适合 RocketMQ 这种顺序写场景。

### 三、实战补充

- **实战案例**：某线上 Broker 在高峰期偶尔出现卡顿，排查发现 `vm.swappiness` 默认为 60，导致 PageCache 被频繁换出，调整为 10 后卡顿消失。另外，未调大 `vm.max_map_count` 会导致 Broker 启动时报 "OutOfMemoryError: Map failed"。
- **代码示例**（配置文件 `/etc/sysctl.conf`）：
```bash
vm.min_free_kbytes = 2097152
vm.extra_free_kbytes = 2097152
vm.max_map_count = 655360
vm.swappiness = 10
```
