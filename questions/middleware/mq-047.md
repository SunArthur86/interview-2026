# 进阶必看的 RocketMQ，这次一网打尽

这个协议是基于 Raft 的，协议具体就不展开了，就理解为它能解决 Controller  Leader 的选举，并且
让所有节点达成共识。
在之前基于 Zookeeper 实现的单个 Controller 在分区数太大的时候还有个问题，故障转移太慢了。
当 Controller 变更的时候，需要重新加载所有的元数据到新的 Controller 身上，并且需要把这些元数
据同步给集群内的所有 Broker。
而 Controller Quorum 中的 Leader 选举切换则很快，因为元数据都已经在 quorum 中同步了，也就
是 quorum 的 Broker 都已经有全部了元数据，所以不需要重新加载元数据！
并且其它 Broker 已经基于 Log 存储了一些元数据，所以只需要增量更新即可，不需要全量了。
这波改造下来就解决了之前元数据过多的问题，可以支持更多的分区！

## 技术原理

KRaft（Kafka Raft）是 Kafka 2.8+ 引入的自研元数据管理协议，目的是彻底移除对外部 ZooKeeper 的依赖：

- **ZooKeeper 模式的瓶颈**：旧架构中 Controller 是单点，它从 ZK 拉取全量元数据到内存。当分区数超过数十万时，Controller 故障切换需要重新从 ZK 加载全部元数据，耗时可达数十秒甚至分钟级，期间集群无法正常处理分区 Leader 选举，造成业务感知的「故障窗口」。
- **KRaft 的 Quorum 复制**：KRaft 用 3 或 5 个 Controller 节点组成 Raft Quorum，元数据以 Raft Log 形式在 Quorum 中强一致复制。Active Controller（Leader）写元数据 Log，Follower 实时同步。Leader 切换时，新 Leader 已持有最新 Log（Quorum 保证），无需重新加载，故障恢复从分钟级降到秒级。
- **Broker 侧的元数据日志**：普通 Broker 也维护一份元数据 Log（作为 Observer），通过拉取 Leader 的 Log 增量更新本地元数据。这取代了原来「Controller 推送全量元数据」的推送模型，变为「Broker 按需拉取增量」的拉取模型，网络开销大幅降低。
- **支持更多分区**：元数据管理从「单点内存」变成「分布式 Log」，可支撑百万级分区，为 Kafka 在超大规模集群（如 LinkedIn、Confluent Cloud）的部署扫清障碍。

## 代码示例

```bash
# KRaft 模式启动 Kafka（无 ZooKeeper）
# 1. 格式化存储（生成 ClusterId）
bin/kafka-storage.sh random-uuid
# 输出: xKc...vQ (假设为 cluster_id)

bin/kafka-storage.sh format \
  -t <cluster_id> \
  -c config/kraft/server.properties

# 2. 直接启动 Kafka（内置 KRaft Controller）
bin/kafka-server-start.sh config/kraft/server.properties

# 3. 查看 Quorum 状态（替代旧的 zkCli.sh）
bin/kafka-metadata-quorum.sh --bootstrap-server localhost:9092 describe --status
# 输出 LeaderId / LeaderEpoch / 高水位等 Raft 状态
```

```properties
# config/kraft/server.properties 关键配置
process.roles=broker,controller      # 单节点同时承担 Broker 和 Controller（开发环境）
node.id=1
controller.quorum.voters=1@localhost:9093   # Quorum 成员列表
listeners=PLAINTEXT://:9092,CONTROLLER://:9093
```

## 注意事项

- **迁移成本**：从 ZK 模式迁移到 KRaft 需要滚动升级 + 元数据格式转换，生产环境务必先在测试集群演练，且备份 ZK 数据以便回滚。
- **Quorum 规模权衡**：3 节点 Quorum 容忍 1 个故障，5 节点容忍 2 个。小集群用 3 节点够用，跨可用区部署建议 5 节点提高容灾。
- **Controller 与 Broker 混合部署**：小集群可让 Broker 同时充当 Controller（combined mode），大集群建议分离部署（dedicated Controller），避免 Broker 负载影响元数据共识延迟。
- **不要回退到 ZK**：Kafka 4.0 已完全移除 ZK 支持，新集群必须用 KRaft。存量集群应尽快规划迁移。

## 记忆要点

- 极速切换：基于Raft的Quorum已同步全量元数据，Leader切换无需重载
- 高效更新：Broker基于日志存储，只需增量更新元数据，告别全量拉取
- 架构演进：早期为求简单复用ZK，规模触及瓶颈后转自研KRaft突破限制
