# RabbitMQ中Exchange有哪几种类型？

RabbitMQ 中 Exchange（交换机）有 4 种类型，决定了消息从交换机路由到队列的规则：

**1. Direct（直连交换机）：**
- **匹配规则**：根据 **routing key 完全匹配**。
- **行为**：Binding Key 与 Routing Key 相等时，消息进入队列。
- **场景**：点对点定向投递（如订单服务只收 `order.create`，日志级别区分 `error`/`info`）。

**2. Fanout（扇出交换机）：**
- **匹配规则**：**广播**给所有绑定的 Queue，**完全忽略 routing key**。
- **行为**：消息会被投递到与该交换机绑定的每一个队列，速度最快。
- **场景**：群发通知、日志广播、发布订阅模式。

**3. Topic（主题交换机）：**
- **匹配规则**：根据 routing key **模式匹配**（通配符匹配）。
- **通配符**：
  - `*` (星号)：匹配**正好一个**单词。
  - `#` (井号)：匹配**零个或多个**单词。
- **例**：Routing Key `order.create.paid` 匹配 `order.*.*`（不匹配 `order.#` 因为 # 只能在末尾通配多级，但 `#.paid` 匹配）。
- **场景**：灵活的多条件订阅过滤（最常用）。

**4. Headers（头交换机）：**
- **匹配规则**：不依赖 routing key，根据消息头中的键值对进行匹配。
- **参数**：绑定队列时指定 `x-match` 参数：`all`（所有键值对都匹配）或 `any`（任意一个匹配）。
- **性能**：性能较差（需解析完整 Header），较少用。
- **场景**：复杂的多条件路由、基于属性的定向发送。

**默认 Exchange：**
- 名为空字符串 `""` 的 Direct Exchange。
- **特殊行为**：每个队列都会自动绑定到默认交换机，Binding Key = 队列名称。
- **用途**：允许直接向队列名发送消息，简化简单场景（`channel.basicPublish("", "queueName", ...)`）。

### 实战案例
在订单系统中，若仅使用 Fanout 交换机广播所有订单事件，库存服务（只需关注“已支付”事件）也会收到“创建”事件，导致无效处理。应改用 **Topic 交换机**，库存服务绑定 `order.paid`，物流服务绑定 `order.shipped`，实现精准过滤。

### 代码示例
Spring Boot 声明 Topic 交换机与绑定关系：

```java
@Bean
public TopicExchange orderExchange() {
    return new TopicExchange("orderTopic");
}

@Bean
public Binding bindingPaid(Queue queue, TopicExchange exchange) {
    // 绑定 routing key 为 "order.paid"
    return BindingBuilder.bind(queue).to(exchange).with("order.paid");
}
```

### 对比表格：Exchange 类型选型

| 类型 | 路由规则 | 复杂度 | 吞吐量 | 典型场景 |
| :--- | :--- | :--- | :--- | :--- |
| **Direct** | 完全匹配 Key | 低 | 高 | 简单的任务分发、日志分级 |
| **Fanout** | 广播 (忽略 Key) | 最低 | 最高 | 广播通知、多系统同步 |
| **Topic** | 模式匹配 (*, #) | 高 | 中高 | 复杂业务事件分发、多维度过滤 |
| **Headers** | 键值对匹配 | 最高 | 低 | 极少使用，高级规则路由 |
