# 如何缓解 Inference 重复问题

## 重复问题的根源
1.  **解码策略缺陷**：
    *   **贪心搜索**：每一步都选概率最大的词，容易陷入“高概率循环”。例如生成“他非常...非常...非常...”。
    *   **束搜索**：虽然保留了多个候选，但如果缺乏多样性惩罚，Beam 之间容易收敛到相似路径，导致重复片段。
2.  **概率分布尖锐**：模型对某些高频词赋予过高的概率，导致缺乏探索性，陷入局部最优。
3.  **训练数据因素**：训练语料中本身包含大量重复模式（如电商评论、日志），模型过拟合了这些重复模式。
4.  **注意力缺陷**：在生成较长文本时，模型可能“遗忘”了已经生成过的上下文，导致重复出现之前的短语。

## 缓解策略详解

### 1. 频率惩罚与重复惩罚
这是最直接有效的后处理手段。
*   **Frequency Penalty**：惩罚 tokens 的出现频率。如果一个 token 已经出现过了，再次生成它的 logits 会减去一个常数。
    *   `logits[token_id] -= alpha * count(token_id)`
*   **Presence Penalty**：只要 token 出现过就惩罚，不管出现几次。这鼓励模型引入新话题。
*   **Repetition Penalty**：对已经生成过的 token 的原始概率进行缩放（通常除以一个大于 1 的系数），而不是简单的减法。

**实战案例**：在某角色扮演项目中，模型在对话中无限循环“好的，没问题”。通过引入 Repetition Penalty (系数 1.1) 并结合 Presence Penalty，成功打破循环，但需注意系数过高（>1.5）会导致模型为了不重复而强行切换话题，破坏连贯性。

**代码示例 (Python)**：
```python
def apply_repetition_penalty(logits, input_ids, penalty=1.15):
    # logits: [vocab_size]
    # input_ids: [seq_len]
    score = torch.gather(logits, 0, input_ids)
    # 如果惩罚系数 > 1，已出现的词概率被降低
    score = torch.where(score < 0, score * penalty, score / penalty)
    logits.scatter_(0, input_ids, score)
    return logits
```

### 2. 采样策略优化
*   **Temperature (温度)**：调高温度（>1.0）使概率分布平滑，增加低概率词被选中的机会。
*   **Top-k / Top-p (Nucleus Sampling)**：
    *   **Top-k**：只从概率最高的 k 个词中采样。
    *   **Top-p**：从累积概率达到 p 的最小词集中采样。
    *   这两种方法能切断长尾的荒谬预测，同时保留一定的随机性，打破死循环。

**对比表格**：
| 策略 | 原理 | 适用场景 | 副作用 |
| :--- | :--- | :--- | :--- |
| **Greedy** | 每次选概率最大的词 | 事实性问答、数学计算 | 极易陷入循环，文风呆板 |
| **Beam Search** | 保留Top N路径，全局寻优 | 翻译、摘要 | Beam间趋同导致重复，显存占用大 |
| **Sampling (Top-p)** | 概率采样，引入随机性 | 创意写作、对话 | 内容可能发散，不够聚焦 |
| **Repetition Penalty** | 降权已生成的Token | 所有生成任务通用 | 调参困难，过高可能导致逻辑断裂 |

### 3. N-gram 阻断
在生成过程中显式地禁止重复的 N-gram。
*   **原理**：记录已生成的 N-gram 集合（如 3-gram 或 4-gram）。如果候选词会导致生成一个已存在的 N-gram，则将其概率强制设为 -inf。

**代码示例 (Python)**：
```python
# 生成时动态计算禁止集合
banned_tokens = set()
for ngram in current_ngrams: # e.g. ('我', '很', '喜欢')
    # 如果下一个词是 '我'，导致 ('很', '喜欢', '我') 重复
    next_token_candidate = ngram[1:] + (target_token_id,)
    if next_token_candidate in generated_history:
        banned_tokens.add(target_token_id)
# 将 banned_tokens 的 logits 设为 -inf
logits[list(banned_tokens)] = -float('inf')
```

### 4. 停止词与截断
*   设置合理的 `max_length` 或 `eos_token`，防止无意义的长循环。

## 常见考点
1.  **Repetition Penalty 实现细节**：是直接减 logits 还是修改概率？直接修改 logits 更常见且稳定。
2.  **Beam Search 重复**：为什么 Beam Search 更容易重复？因为它倾向于寻找局部最优的安全路径。如何解决？引入 Diversity Penalty（Hamming Diversity）。
3.  **参数调优经验**：一般 Temperature 设多少合适？（创意类 0.7-1.0，事实类 <0.7）。Repetition Penalty 一般设为 1.0-1.5 之间。

## 记忆要点

- 源头与惩罚：因贪心搜索和概率分布尖锐易导致循环，所以对已生成Token施加重复惩罚。
- 采样破局：调高温度或用Top-k/Top-p截断长尾，因引入了随机性，所以能打破死循环。
- 硬性阻断：在解码时动态记录历史N-gram，若候选词构成重复N-gram则掩码其概率。

## 延伸：如何缓解大模型inference的时候的重复问题

> 合并自 `llm-064`（相似度 73%）

在大型语言模型(如GPT-4)的推理过程中,生成的文本有时会出现重复.这种重复不仅影响文本质量,还会降低用户体验.回答应该从根源上解释导致重复的原因,并提供缓解重复问题的具体策略.

**一、重复问题的根源**

**1. 语言模型的概率分布倾向**
语言模型通过对训练数据中模式的学习,生成每个可能词的概率分布.然而,以下原因可能导致模型倾向于重复：
*   **高频词汇权重过高：** 在语言模型中,高频词汇的概率往往被模型学到较高的权重 (如“的”、“是”),这使得它们容易在生成过程中反复出现.
*   **局部依赖问题：** 自回归模型基于上文预测下文。如果上文生成了某些词，这些词在Self-Attention中作为Key和Value反复出现，可能导致模型在“已生成的词汇”和“预测词汇”之间形成死循环。
*   **缺乏长程规划：** 生成过程中,模型并没有整体规划文本的结构,可能在局部生成时不断循环相似的内容.

**2. 解码策略的影响**
模型的解码策略直接影响生成文本的质量.以下是常见解码策略导致重复的原因:
*   **贪心搜索：** 每一步选择概率最高的词会让模型倾向于选择安全且重复的词汇，导致“退化循环”。
*   **束搜索：** 束搜索会同时跟踪多个候选序列,选择概率最高的路径.然而,如果没有多样性惩罚机制,多条路径可能趋同,导致相似句子反复生成.
*   **缺乏熵正则化：** 如果未引入熵正则化,模型的输出分布可能变得过于尖锐(集中在少数高概率词上),从而导致重复.

**3. 训练数据的问题**
*   **重复样本的存在：** 如果训练数据中包含大量重复句子或片段（如爬虫抓取的重复页脚、广告语）,模型可能会认为这些重复是合适的语言结构.
*   **语料不平衡：** 某些主题或语言结构占比过高,导致模型偏向这些模式.

**4. 模型的局部最优**
在生成过程中,模型可能进入局部最优解状态，即“吸引子”状态，难以跳出。

**二、缓解重复问题的策略**

**1. 调整温度参数**
温度参数 $T$ 控制模型生成概率分布的平滑程度：$P'(w) = \frac{P(w)^{1/T}}{\sum P(w_i)^{1/T}}$
*   **高温度 ($T > 1$)：** 平滑概率分布,增加低概率词的选择机会,从而提升多样性.
*   **低温度 ($T < 1$)：** 集中概率分布,提升文本确定性,但容易导致重复.
*   **建议：** 对于创意生成，通常设置 $T \in [0.7, 1.0]$，过高会导致不知所云。

**2. 采用采样策略**
*   **Top-k 采样：** 仅从概率排名前 $k$ 的词中采样。$k$ 值过大容易导致随机采样到不连贯的词，过小则容易重复。
*   **Top-p (Nucleus) 采样：** 从累计概率达到阈值 $p$ 的最小词集合中采样。常用范围 $p \in [0.9, 0.95]$。它能动态调整词汇表大小，比 Top-k 更鲁棒。

**3. 重复惩罚**
这是最直接有效的工程手段。对已经生成的词或token降低其logits。
*   **实现原理：** 修改当前步的logits：$l'_i = l_i - \alpha \cdot I(i \in \text{history})$，其中 $\alpha$ 是惩罚系数，$I$ 是指示函数。
*   **Presence Penalty vs Frequency Penalty：**
    *   **Frequency Penalty：** 惩罚力度与词出现的**次数**成正比（重复越狠，罚得越重）。
    *   **Presence Penalty：** 只要词出现过**一次**，就给予固定惩罚（鼓励引入新话题）。

**4. n-gram 阻止**
*   **原理：** 强制禁止重复出现长度为 $n$ 的子序列。
*   **实施细节：** 记录已生成的 $n$-gram (如 $n=3$)。在采样时，如果候选词会导致生成的三元组在历史中已存在，则将其概率置为 -inf。

**5. 引入多样性**
在束搜索中引入多样性惩罚机制,鼓励生成不同的序列，避免所有 Beam 最终汇聚到同一条重复路径。

### 三、实战案例与代码对比

**实战案例**：在一次客服机器人的对话生成中，模型在遇到无法回答的查询时，会陷入重复“非常抱歉，我不太理解您的意思”的死循环。**踩坑经验**：仅增加 `frequency_penalty` 会导致模型为了避讳而造出生僻词，最终结合 `n-gram blocking=3` 和 `temperature=0.8` 才解决了问题，既打破了循环又保持了语句通顺。

**代码示例 (Python - HuggingFace Transformers 采样配置)**：
```python
from transformers import GenerationConfig

# 关键配置：结合频率惩罚与采样策略防止重复
config = GenerationConfig(
    max_new_tokens=256,
    temperature=0.7,          # 提升随机性，打破高频词垄断
    top_p=0.9,                # Nucleus采样，过滤低质量尾部
    repetition_penalty=1.1,   # 频率惩罚系数，大于1.0即惩罚已生成词
    no_repeat_ngram_size=3,   # 强制禁止3-gram重复，硬性阻断死循环
    do_sample=True
)
output = model.generate(input_ids, generation_config=config)
```

**策略对比表**
| 策略 | 适用场景 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **Temperature** | 创意写作、聊天 | 实施简单，平滑分布 | 过高导致逻辑不通，过低无效 |
| **Repetition Penalty** | 通用文本生成 | 灵活微调，软性约束 | 需反复调试参数，可能误伤必要强调 |
| **n-gram Blocking** | 代码生成、翻译 | 硬性避免循环，效果立竿见影 | 过于暴力，可能切断正常重复句式 |

## 常见考点
1.  **高频词重复与死循环的区别：** 如何区分是统计特性导致的“高频词重复”还是注意力机制导致的“逻辑死循环”？（前者通常涉及虚词，后者通常是实词或短语循环）。
2.  **Presence vs Frequency Penalty：** 在什么场景下应该调大 Presence Penalty（如避免话题重复），什么场景下调大 Frequency Penalty（如解决“卡壳”式复读机）。
