# PPL(Perplexity)及其数学公式

### 1. 定义与解释

**PPL (Perplexity, 困惑度)** 是衡量语言模型预测下一个 token 难易程度的经典指标。对于自回归语言模型，给定一个句子 $X = (x_1, x_2, ..., x_N)$，模型输出的联合概率分布为：

$$p(x_1, x_2, ..., x_N) = p(x_1) \cdot p(x_2 \mid x_1) \cdot p(x_3 \mid x_1, x_2) \cdots p(x_N \mid x_1, x_2, ..., x_{N-1})$$

句子的**平均负对数似然 (NLL)** 定义为：

$$NLL = -\frac{1}{N} \sum_{i=1}^N \ln p(x_i \mid x_{<i})$$

其中 $x_{<i}$ 表示 $x_i$ 之前的所有 token。

**PPL** 通常定义为 NLL 的指数形式：

$$PPL(X) = \exp\left(-\frac{1}{N} \sum_{i=1}^N \ln p(x_i \mid x_{<i})\right)$$

如果模型对真实 token 的预测概率很高，则 NLL 值小，PPL 值也更小。PPL 可以理解为“模型每次猜下一个 token 的不确定度”或“分支因子的倒数”，越接近 1 表示模型越确定。

### 2. 如何用 PPL 来评估模型？

**在预训练阶段：** 我们通常会准备一些“开发集”，其中包含百科文本、新闻、代码片段等。每天或每隔一定步数，让模型在这些测试集上做一次前向推理，计算 PPL，观察其是否持续下降。当 PPL 下降到某个稳定水平后，说明模型对通用语料的学习已接近饱和。

**注意事项：** PPL 只适合同一模型或同一家 tokenizer 上做纵向对比。因为不同的 tokenizer 压缩率不同，同一句话拆成的 token 数量会影响 PPL 的绝对数值。

**经验参考：**
*   大模型在常见中/英文混合语料上，如果预训练充分，PPL 通常能下探到 2 左右（甚至更低），否则可能意味着训练不足。
*   不同的场景（如代码 vs 纯文本）可能有不同的典型 PPL 水平。

### 3. 实战案例与代码

**实战案例**：在评估 SFT（有监督微调）后的模型时，发现模型在通用语料（Wiki）上的 PPL 反而比微调前上升了。这**不是**坏事，因为 SFT 模型往往改变了分布（更倾向于生成对话格式），此时更应关注特定领域（如数学题或对话数据集）上的 PPL，或者直接使用人工评估，因为 PPL 并不完全等同于生成质量（可能会出现 PPL 低但生成内容空洞的“Mode Collapse”现象）。

**代码示例**：
```python
import torch
import torch.nn.functional as F

def compute_ppl(model, dataloader, device):
    total_loss = 0
    total_tokens = 0
    model.eval()
    with torch.no_grad():
        for batch in dataloader:
            input_ids = batch['input_ids'].to(device)
            # 只取 logits 和 labels，忽略 prompt 部分（如果有）
            outputs = model(input_ids, labels=input_ids)
            
            # neg_log_likelihood: batch_size * seq_len
            nll = outputs.loss * input_ids.size(1) 
            total_loss += nll.item()
            total_tokens += input_ids.size(1)
    
    avg_nll = total_loss / total_tokens
    ppl = torch.exp(torch.tensor(avg_nll))
    return ppl.item()
```

## 延伸：PPL(Perplexity,困惑度)是什么?如何用于评估大模型

> 合并自 `llm-109`（相似度 68%）

- **PPL (Perplexity):** 衡量概率模型对样本的「不确定性」，即模型「困惑」的程度。

- **数学定义:**
```
PPL(W) = exp(-1/N * Σ log P(wi | context))

等价于:
PPL(W) = 2^CrossEntropy(W) (以2为底时)

PPL越低 → 模型越确信(预测概率高) → 质量通常越好
```

- **实战案例:** 在训练垂直领域大模型时，若通用基座模型的领域PPL显著高于同规模开源模型，直接SFT会导致灾难性遗忘，需先进行领域增量预训练。在评估代码生成模型时，单纯追求低PPL会导致模型倾向于生成常见的API调用模式而忽略了复杂的业务逻辑，需结合Pass@1指标综合考量。

- **代码示例:**
```python
import torch
import torch.nn.functional as F

def compute_ppl(model, input_ids, attention_mask):
    outputs = model(input_ids, attention_mask=attention_mask, labels=input_ids)
    loss = outputs.loss  # CrossEntropy reduction=mean
    ppl = torch.exp(loss)
    return ppl.item()
```

- **直觉理解:**
- PPL=1: 模型完全确定下一个词 (概率为1)
- PPL=10: 模型在 10 个等可能的候选词中犹豫不决
- PPL=100: 模型非常不确定，几乎是在随机生成

- **如何用于评估:**
1. **模型对比** - 同一测试集上 PPL 越低越好（基准线，如 GPT-4 在该集上的 PPL）。
2. **训练监控** - 训练/验证集 PPL 应持续下降；若 Train PPL 降但 Val PPL 升，则过拟合。
3. **语言质量** - 衡量生成文本的流畅度和语法正确性，但不代表事实准确性。

- **PPL 计算边界:**
- **滑动窗口 vs 全文:** 对于超长文本，受限于显存，通常使用 Sliding Window (如 2048 长度) 计算 PPL 并取平均，这会低估长距离依赖带来的真实困惑度。
- **Striding:** 计算 PPL 时，窗口间通常有 stride < window_size 以捕捉上下文依赖。

- **局限性:**
- PPL 低不代表回答准确（只衡量语言建模能力，不代表逻辑或对齐）。
- 不同 Tokenizer 的 PPL 不可直接比较（词表大小、切分粒度影响熵值）。
- 对长文本不敏感（平均效应掩盖了局部的逻辑崩坏）。

- **实践:** 预训练用 PPL 监控模型收敛情况；SFT/RLHF 阶段 PPL 可能会上升（因为模型分布偏离语言模型分布向人类偏好靠拢），此时更应关注人工评分或任务指标。
