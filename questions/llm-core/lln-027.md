# SFT阶段怎么做数据packing

在 SFT 阶段，为了充分利用 GPU 算力并避免 Padding 占用大量显存，通常采用 **Data Packing（数据打包）** 策略。其本质是将多个变长样本拼接成一个固定的 Context Window 长度。

**具体实现（贪心算法）**：
1.  **排序**：将所有样本按长度从长到短排序。
2.  **拼接**：维护一个或多个 `packed_batch`，遍历样本，尝试将当前样本放入当前 `batch` 中剩余空间。
3.  **截断与分隔**：
    - 若能放下，直接拼接，中间插入 `EOS` Token 或特定的分隔符（Document Separator）。
    - 若放不下，从当前样本截断填满 `batch`，剩余部分放入下一个 `batch`。

**关键细节**：
- **Attention Mask**：拼接后的全局 Attention Mask 需确保不同样本之间不互相 Attention（通常将分隔符处的 Mask 设为 0 或 Causal Mask 自然截断）。
- **Position IDs**：拼接后的序列 Position IDs 需要重置，不能跨样本累加（或者在 Sample 之间重置为 0），否则模型会学到错误的位置信息。

```text
Original Samples:
[Sample A: len=10] [EOS]
[Sample B: len=50] [EOS]
[Sample C: len=30] [EOS]

Packed Batch (Max Len=64):
+-----------------------------+---------------------+--+
|      Sample B (50)          | Sample A (10) + EOS |Pa|
+-----------------------------+---------------------+--+
|         Seq 1               |        Seq 2        |Pa|
+-----------------------------+---------------------+--+
```

## 技术原理

- **Padding 浪费的根源**：SFT 数据变长（10~2000 token 不等），若每条单独组 batch，必须 padding 到 batch 内最长样本的长度。短样本占满整个 batch 时，padding 比例可能高达 80%+，GPU 算力大量浪费在 padding 的无效计算上。Packing 通过把多条短样本拼成一个固定长度（如 2048/4096），把 padding 压到最低（仅最后不足一条样本时），GPU 利用率从 30% 提到 90%+。
- **贪心装箱（Greedy Bin Packing）的算法**：①把所有样本按长度降序排序；②维护若干"bin"（每个 bin 容量 = max_seq_len，如 4096）；③对每个样本，尝试放入"还能装下它"的 bin，若没有就开新 bin；④放不下时截断填满当前 bin，剩余部分继续匹配。这是经典的 First Fit Decreasing (FFD) 装箱问题，近似最优解。实践中 batch 内多个 bin（如 8 个 4096 的 bin 组成一个 batch）。
- **跨样本 attention 的隔离（关键陷阱）**：packing 后多条样本拼在同一个序列里，若不做隔离，token 会"看到"相邻样本的内容——Sample A 的 token attend Sample B 的 token，导致训练信号污染。两种隔离方式：①**修改 attention mask**——把矩阵 block 化，只在每个样本内部做 causal attention（`DocumentLevelAttentionMask` 或 `flash_attn_varlen`）；②**插入特殊分隔符 + 让模型自己学**（不推荐，效果差且不可靠）。现代实现都用第一种。
- **Position IDs 的重置**：每条样本独立计算 position id（从 0 开始），而非跨样本累加。否则 Sample A 长 50，Sample B 从 position 51 开始，模型会误以为 B 是 A 的延续，学错位置编码。HuggingFace 的 `DataCollatorForSeq2Seq` 不自动处理，需在 packing 逻辑里显式 reset。

## 代码示例

```python
import torch
from transformers import AutoModelForCausalLM

def pack_samples(samples, max_len=2048, eos_id=2):
    """
    samples: List[List[int]]  每条是 token id 序列
    返回: packed_input_ids, packed_attention_mask, packed_position_ids, cu_seqlens
    """
    # 1. 按长度降序
    samples = sorted(samples, key=len, reverse=True)

    packed_ids, packed_attn, packed_pos, cu_seqlens = [], [], [], [0]
    current_len = 0

    for sample in samples:
        if current_len + len(sample) + 1 > max_len and current_len > 0:
            # 当前 bin 装满，开启新 bin
            cu_seqlens.append(len(packed_ids))
            current_len = 0

        # 拼接样本 + EOS 分隔
        packed_ids.extend(sample + [eos_id])
        # position id 重置（每样本从 0 开始）
        packed_pos.extend(range(len(sample) + 1))
        current_len += len(sample) + 1

    cu_seqlens.append(len(packed_ids))  # 最后一个边界

    # 构造 block-diagonal attention mask（样本内 causal，跨样本隔离）
    seq_len = len(packed_ids)
    attn_mask = torch.zeros(seq_len, seq_len, dtype=torch.bool)
    for i in range(len(cu_seqlens) - 1):
        start, end = cu_seqlens[i], cu_seqlens[i + 1]
        attn_mask[start:end, start:end] = torch.tril(torch.ones(end-start, end-start, dtype=torch.bool))

    return packed_ids, attn_mask, packed_pos, cu_seqlens

# 使用 Flash Attention 的 varlen 接口（高效，无需显式构造 mask）
# from flash_attn import flash_attn_varlen_func
# flash_attn_varlen_func(q, k, v, cu_seqlens, cu_seqlens, max_seqlen, ...)
```

## 对比/选型

| 策略 | Padding 浪费 | GPU 利用率 | 实现复杂度 | 训练稳定性 |
|------|-------------|-----------|-----------|-----------|
| 直接 padding | 高（30~80%） | 低 | 极简 | 稳定 |
| Sorting + padding（按长度分桶） | 中（10~30%） | 中 | 简单 | 稳定 |
| **Packing（贪心装箱）** | 极低（<5%） | 高（>90%） | 复杂（attention mask 隔离） | 需注意跨样本泄漏 |
| Collated packing（HF mainline） | 极低 | 高 | 中（库已支持） | 稳定 |

## 常见坑/注意事项

- **跨样本 attention 泄漏**：最常见的 bug——packing 后忘了改 attention mask，样本间互相 attend，模型训练 loss 假性下降但效果差。一定要用 block-diagonal mask 或 `flash_attn_varlen`。
- **label mask 的同步**：packing 后 labels 也要对应拼接，prompt/eos/padding 部分的 label 仍要设 -100。若漏掉某条样本的 prompt mask，会污染训练。
- **变长 batch 的显存波动**：packing 让每 batch 的 token 数固定，但样本数变化大（一个 batch 可能 1 条长样本或 50 条短样本）。梯度统计、loss 归一化要用 token-mean 而非 sample-mean。
- **推理时不能用 packing**：packing 是训练优化，推理时每个请求独立处理（batch padding）。不要把 packing 逻辑混进推理代码。
- **排序带来的数据分布偏差**：贪心排序让长样本集中在前几个 bin，短样本集中在后几个。若 batch 内样本数差异大，可能导致梯度噪声。可加随机 shuffle + 在线 packing 平衡。
