# 滑动窗口（Sliding Window）注意力机制（如 Longformer, Mistral）是如何将计算复杂度从 O(N^2) 降低到 O(N) 的？它对模型的上游预训练有什么特殊要求？

滑动窗口注意力机制将 Self-Attention 的计算限制在每个 Token 周围的一个固定大小窗口（如 4096）内，而非计算所有 Token 两两之间的注意力。这样每个层只计算局部注意力，复杂度随序列长度线性增长。为了保留全局信息，这些模型通常每隔一层增加一个全局注意力 Token 或 Dilated Sliding Window（空洞卷积式的滑动窗口）来扩大感受野。对上游预训练的特殊要求是：模型在预训练时必须已经使用了这种稀疏注意力模式，或者通过适配训练适应这种模式。因为直接将原本全量注意力训练的模型权重迁移到滑动窗口结构上，会导致模型无法捕捉长距离依赖，性能剧烈下降。

## 技术原理

滑动窗口注意力的本质是引入「空间局部性先验」，用可控的精度损失换取线性复杂度：

- **复杂度推导**：标准 Self-Attention 中，每个 token 要与所有 $N$ 个 token 计算注意力，总计算量 $O(N^2 d)$。滑动窗口限制每个 token 只与前后各 $w/2$ 个 token 交互（窗口大小 $w$），每层计算量降为 $O(N \cdot w \cdot d)$。当 $w$ 固定（如 4096）时，复杂度关于序列长度 $N$ 是线性的。
- **感受野的层级扩展**：单层窗口只能看 $w$ 个 token，但堆叠 $L$ 层后，信息可以通过层间传递跨越 $L \cdot w$ 个 token（类似 CNN 的感受野增长）。这就是为什么 Mistral 用窗口注意力仍能处理 32K 上下文——靠的是深层堆叠，而非单层全连接。
- **全局信息补偿机制**：(1) **全局 Token**（Longformer）：对 [CLS]、标题等关键位置放开全注意力，作为信息枢纽；(2) **Dilated Window**：窗口内做空洞采样（如每隔 2 个 token 取一个），等效窗口扩大但计算量不变；(3) **交替层**：奇数层窗口、偶数层全注意力（BigBird 的 random + window + global 混合）。
- **为何必须预训练适配**：注意力模式是模型学到的信息通路。全注意力模型学到的是「任何两个 token 都可能相关」的权重分布；直接换成窗口模式，相当于强行切断已学好的长程连接，导致已训练的注意力权重失效，表现为 Loss 暴涨、能力崩盘。

## 代码示例

```python
# 滑动窗口注意力的 mask 构造（PyTorch 伪代码）
import torch

def sliding_window_mask(seq_len, window_size=4096, device="cuda"):
    """
    生成 (seq_len, seq_len) 的 bool mask，
    mask[i][j] = True 表示位置 i 可以看到位置 j
    """
    # 1. 基础因果 mask（下三角）
    causal = torch.tril(torch.ones(seq_len, seq_len, device=device, dtype=torch.bool))
    # 2. 限制窗口：|i - j| <= window_size
    pos = torch.arange(seq_len, device=device).unsqueeze(0)
    within_window = (pos.t() - pos).abs() <= window_size
    return causal & within_window

# 应用：把 mask 传给 attention
# attn = scaled_dot_product_attention(q, k, v, attn_mask=sliding_window_mask)
```

## 注意事项

- **窗口大小的选择**：窗口太小（如 512）会丢失段落级上下文；太大（如 16K）失去线性优势。Mistral 选 4096 是平衡点，需结合任务平均文档长度调整。
- **KV Cache 的优化**：窗口注意力下，超出窗口的 KV 可以丢弃，推理时 KV Cache 只需保留最近 $w$ 个 token 的状态，显存占用恒定——这是 Mistral 能做超长推理的关键。
- **与 Flash Attention 的配合**：Flash Attention 原生支持 sliding window（v2 起），无需构造稠密 mask，进一步降低显存。
- **不适合的任务**：需要精确全局对齐的任务（如长文档问答中「第 1 段提到的人名在第 50 段怎么称呼」）窗口注意力容易遗漏，这类场景应用全注意力或全局 token 方案。
- **与全局注意力混合的工程实践**：现代长文本模型（如 Mistral、Mixtral）并非纯窗口注意力，而是「窗口注意力层 + 少数全注意力层」交替，或配合 Sliding Window Attention（SWA）+ chunked attention 的混合方案，在保持线性的同时兼顾全局信息。
- **推理时的 KV Cache 收益**：SWA 模式下推理 KV Cache 大小恒定（只保留最近 $w$ 个 token），可支撑超长推理（如 100K+ token）而不爆显存，这是 Mistral-7B 能处理 32K 上下文的核心原因。但要注意滑动窗口会「遗忘」早期上下文，对需要全文档记忆的任务有局限。
- **窗口大小与任务平均句长对齐**：英文 token 平均 4 字符、中文更密，$w=4096$ 在英文约等于 4-5 段，在中文可能只覆盖 2-3 段。多语言模型要按语言分布重新评估窗口大小，否则中文场景下上下文连贯性会比英文差。
- **与位置编码的耦合**：Sliding Window 通常配合 RoPE/ALiBi 使用，但 ALiBi 本身有「距离衰减」特性，与窗口的硬截断叠加可能导致边界处注意力突变（窗口内最后一个 token 权重断崖式下降）。需在预训练时让模型适应这种边界，或用软掩码（soft mask）平滑过渡。
- **训练效率的隐藏收益**：窗口注意力不仅降显存，还能让训练时 attention 的访存连续（局部 token 在内存中相邻），GPU Tensor Core 利用率比全局注意力高 20-30%，这是 Mistral 训练成本低于 Llama 2 的隐性原因之一。
- **SWA 与 RAG 的协同**：滑动窗口的「遗忘」特性恰好适合 RAG 场景——检索到的片段本就是局部的，窗口注意力专注片段内推理，全局信息由外部检索补充。这是 Mistral + RAG 架构在生产中表现优异的原因之一。
- **Layer-wise 窗口大小衰减**：部分新模型（如 Mistral-8x7B）采用「浅层大窗口、深层小窗口」的策略——浅层捕捉宽泛上下文建立语义，深层聚焦局部做精细推理。这种层级窗口设计比固定窗口的复杂度更优，但实现复杂度高。
- **Chunked Attention 的折中方案**：介于全注意力和滑动窗口之间的方案——把序列分成固定 chunk，chunk 内全注意力、chunk 间用少量全局 token 通信。如 Mistral-7B v0.2 实际用 chunked + sliding 的混合，在长文任务上比纯窗口好且复杂度仍接近线性。
- **多模态场景的特殊性**：图文混合序列中，图像 token 通常密度低（一张图 256 个 token）但信息量大，窗口注意力容易在文本密集区「淹没」图像信号。多模态模型常用「图像 token 设为全局可见」的混合策略，避免窗口截断丢失关键视觉信息。

## 记忆要点

- 降复杂度：因限制只算Token周围固定大小窗口，非算全局两两，所以复杂度降为O(N)
- 保留全局：靠层间堆叠或增加全局/空洞窗口扩大有效感受野来传递信息
- 预训练要求：必须从预训练就用稀疏窗口，直接将全量权重迁移会导致性能崩盘
