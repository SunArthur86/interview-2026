# PagedAttention v2 相比 v1 做了哪些关键改进？它主要解决了什么性能瓶颈？

PagedAttention v1 核心解决了KV Cache显存碎片化问题，提出了将KV块分页存储的机制。而 v2 主要针对计算开销进行了优化，特别是在处理请求并发时上下文切换的开销。v1 中每个批次处理可能需要重算某些KV块或频繁的数据搬运。v2 引入了更高效的内核实现，支持更灵活的块共享和并行解码优化，减少了CPU和GPU之间的通信开销，并显著提高了Cache命中时的计算吞吐。它主要解决了在极高并发场景下，虽然显存利用率高了（得益于v1），但计算速度反而受限于内存读写带宽和调度开销的问题。

**实战案例**：在支撑类似ChatGPT的并发对话场景时，v1版本会因为频繁检查KV Block有效性导致Kernel Launch latency过高，而v2通过重计算部分内核索引，将首字延迟（TTFT）在P99场景下降低了30%以上。

**对比表格**：

| 维度 | PagedAttention v1 | PagedAttention v2 |
| :--- | :--- | :--- |
| **核心目标** | 解决显存碎片化，支持连续批处理 | 优化计算吞吐，降低CPU-GPU交互开销 |
| **计算内核** | 串行处理较多，访存模式较简单 | 引入Fused Kernel，支持Cooperative Fetching |
| **缓存策略** | 侧重于虚拟内存管理 | 优化了KV Cache的物理访问局部性 |
| **适用场景** | 中低并发，长上下文单任务 | 极高并发，大规模实时交互 |

**代码示例**：
```python
# 伪代码对比 v1 与 v2 的内核调用差异
# v1: 频繁的小块Kernel调用，增加Launch开销
for block in blocks:
    flash_attn_v1_kernel(block.kv_cache) # 每个Block独立调度

# v2: 合并迭代，减少Host-Device通信
def flash_attn_v2_kernel(num_blocks, block_table):
    # Kernel内部通过Warp聚合处理多个Block数据
    cooperative_fetch(block_table)
    compute_attention()
```

## 技术原理

PagedAttention 的演进揭示了 LLM 推理优化的两个层面：**显存管理（v1）** 和 **计算访存比（v2）**。v1 解决"存得下"，v2 解决"算得快"：

- **v1 的核心创新——OS 虚拟内存思想搬到 GPU**：传统 KV Cache 按 request 连续分配显存，导致严重的内部碎片（预留 max_seq_len 但用不完）和外部碎片（请求结束释放后难复用），显存利用率常只有 20%-60%。v1 把每个 request 的 KV Cache 切成固定大小的 block（如 16 token 一块），用 block table 记录逻辑块到物理块的映射——和 OS 的页表机制几乎一模一样。这彻底消除了碎片，显存利用率逼近 100%，支持 Continuous Batching。
- **v1 暴露的新瓶颈——Kernel Launch 开销**：碎片解决后，并发请求数从几十涨到几百，每个请求的 KV 分布在不连续的物理块上。v1 的 attention kernel 处理每个 block 都要发一次 GPU kernel launch，CPU 把控制信息通过 PCIe 传给 GPU 的延迟（每次 5-10 微秒）成为瓶颈。数百请求 × 数十 block = 上万次 launch，CPU-GPU 通信被打满，TTFT（首字延迟）居高不下。
- **v2 的两个关键优化**：
  1. **Fused Kernel（融合内核）**：把多个 block 的 attention 计算融合到一次 kernel launch 里——kernel 内部通过 Warp 级别的协作（Cooperative Fetching）让多个 thread block 并行抓取不同物理块的数据，一次 launch 处理一个 request 的全部 block。launch 次数从 O(blocks) 降到 O(requests)。
  2. **改进的 KV Cache 物理布局**：v1 只优化逻辑映射，v2 进一步优化了物理存储——把 K 和 V 分开存储、按 head 维度做 memory coalescing，让 GPU 的 global memory 访问模式从随机变成连续，L2 Cache 命中率提升 2-3 倍。
- **PagedAttention v2 实际上等价于 vLLM 0.3+ 的实现**：开源 vLLM 在 2023 年中引入了这些优化，论文里叫"PagedAttention"，社区里把早期版本叫 v1，优化后的叫 v2。SGLang、TensorRT-LLM 等推理框架也采用了类似的分页 + 融合内核思路。

## 代码示例

```python
# vLLM 中 PagedAttention 的使用（简化展示）
from vllm import LLM, SamplingParams

llm = LLM(
    model="meta-llama/Llama-3-8B",
    # v2 的核心配置：块大小决定分页粒度
    block_size=16,           # 每个物理块容纳 16 个 token 的 KV
    enable_prefix_caching=True,   # v2 新增：跨请求复用相同前缀的 KV 块
    swap_space=4,            # KV Cache 可换出到 CPU 内存（类似 swap）
    tensor_parallel_size=1,
)

# Continuous Batching + PagedAttention：可同时处理数百个变长请求
prompts = ["写一首诗"] * 200 + ["讲个笑话"] * 50
outputs = llm.generate(prompts, SamplingParams(max_tokens=512))
```

```c
// v1 vs v2 的 CUDA kernel 伪代码对比

// ===== v1：每个 block 独立 launch，CPU-GPU 通信频繁 =====
__global__ void paged_attn_v1(
    KVCache* blocks, int* block_table, int num_blocks) {
    for (int i = 0; i < num_blocks; i++) {
        int physical_block = block_table[i];
        // 访问不连续的物理块，global memory 随机访问
        compute_attention(blocks[physical_block]);
    }
}
// CPU 端：每个 request 循环 launch 一次
// for req in requests:
//     for blk in req.blocks:
//         paged_attn_v1<<<grid, block>>>(...)   # 万次 launch

// ===== v2：融合内核 + Cooperative Fetching =====
__global__ void paged_attn_v2(
    KVCache* blocks, int* block_table,
    int num_blocks_per_req, int num_requests) {
    // 一个 kernel 处理一个 request 的所有 block
    int req_id = blockIdx.y;
    int warp_id = threadIdx.x / 32;

    // Cooperative Fetching：多个 warp 并行抓取不同物理块
    for (int i = warp_id; i < num_blocks_per_req; i += num_warps) {
        int physical_block = block_table[req_id * num_blocks_per_req + i];
        // 数据按 head 维度连续存储，memory coalescing 命中
        kv_data = __ldg(&blocks[physical_block]);  // 通过只读缓存
        __syncwarp();
        compute_attention_local(kv_data);
    }
}
// CPU 端：每个 request 一次 launch
// for req in requests:
//     paged_attn_v2<<<num_heads, 256>>>(...)   # 百次 launch
```

## 注意事项

- **块大小（block_size）需要权衡**：v2 默认 16，太大（如 64）导致内部碎片（最后一块填不满），太小（如 4）增加 block table 查询次数和 kernel 内循环开销。实测 8-32 是甜区，不同模型/序列长度最优值不同。
- **Prefix Caching 是 v2 的隐藏杀手锏**：很多请求有相同 system prompt（如 RAG 里的固定引导语），v2 能检测到前缀相同并复用已计算的 KV block，省掉重复的 prefill 计算。某些场景下能把 TTFT 降低 50%+，但要注意缓存失效（system prompt 改动会让缓存全失效）。
- **CPU-GPU 通信在高端卡上仍是瓶颈**：即使 v2 减少了 launch 次数，H100/B100 的算力增长比 PCIe 带宽增长更快，CPU 调度仍可能成为瓶颈。GPU Native Scheduler（如 CUDA Graph）能进一步减少 CPU 介入，把整个推理图固化到 GPU 执行。
- **不要混淆 PagedAttention 和 FlashAttention**：FlashAttention 解决的是 attention 计算本身的 IO 复杂度（用 tiling 减少 HBM 读写）；PagedAttention 解决的是 KV Cache 的显存碎片和分页管理。两者正交，vLLM 实际上把 PagedAttention 和 FlashAttention 结合使用（PagedAttention 管理 KV 块布局，FlashAttention 执行实际计算）。
- **v2 在长上下文场景收益最大**：短上下文（<2K token）时 block 数少，v1 的 launch 开销不明显；长上下文（32K/128K）时每请求上百块，v2 的融合内核收益指数级放大。这正是为什么长文本推理框架（vLLM、SGLang）都在持续优化 PagedAttention v2。
