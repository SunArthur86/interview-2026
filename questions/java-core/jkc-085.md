# 作业完成。

### Hadoop MapReduce 作业的生命周期

#### 1. 作业提交与初始化
用户提交作业后，首先由 **JobClient** 实例将作业资源上传至分布式文件系统（HDFS）。这些资源包括：
- 程序 Jar 包
- 作业配置文件
- 输入分片元信息文件

分片元信息文件记录了每个输入分片的逻辑位置（如 Block 偏移量和 Host 列表）。资源上传完毕后，JobClient 通过 **RPC** 调用通知 **JobTracker**。

JobTracker 收到请求后，由作业调度模块执行初始化：
1. 创建 **JobInProgress** 对象，用于维护整个作业的运行状态和上下文。
2. JobInProgress 为每个 Map/Map Task 和 Reduce Task 创建对应的 **TaskInProgress** 对象。
3. 考虑到任务可能失败或执行缓慢，TaskInProgress 管理多个 **TaskAttempt**（任务尝试），每个尝试拥有唯一的 TaskAttemptID。

#### 2. 任务调度与监控
**JobTracker** 是集群的核心调度与监控节点。
- **心跳机制**：TaskTracker 周期性向 JobTracker 发送心跳。心跳中不仅包含存活信号，还携带了本节点的资源状况（如空闲 Slot 数量）。
- **任务分配**：当 JobTracker 发现空闲 Slot 时，会通过可插拔的任务调度器（如 FIFO、Capacity Scheduler、Fair Scheduler）分配任务。
  - **双层调度策略**：调度器通常先选择一个作业，再从该作业中选择一个任务。
  - **数据本地性**：调度优先级通常遵循：Node-local（数据在本节点） > Rack-local（数据在同一机架） > Off-switch（跨机架）。
- **容错与推测执行**：
  - **失败重试**：若 TaskTracker 或 Task 失败，JobTracker 会将其状态标记为失败，并重新调度该 Task 到其他节点。
  - **推测执行**：若某个 Task 进度显著落后于同作业的其他 Task（如属于长尾任务），JobTracker 会启动一个相同的备用 Task，优先选取执行完成的 Task 结果作为最终输出。

#### 3. 任务运行环境准备
任务运行环境的准备工作由 **TaskTracker** 独立完成：
- **JVM 隔离**：为每个 Task 启动一个独立的 JVM 进程。这样即使某个 Task 崩溃或导致 JVM 退出，也不会影响 TaskTracker 自身或其他 Task。
- **资源隔离**：利用操作系统进程级别的资源限制（如利用线程池控制 CPU，或通过脚本限制内存）来防止 Task 恶意占用过多资源。
- **资源本地化**：TaskTracker 在运行 Task 前，会将所需的 Jar 包、配置文件等从 HDFS 下载到本地磁盘。

#### 4. 任务执行
环境就绪后，TaskTracker 启动 Task 子进程执行任务。
- **进度汇报**：Task 在运行过程中，会通过 RPC 将进度信息（如 Map 输入记录数、Reduce 复制/排序进度）汇报给 TaskTracker。
- **状态聚合**：TaskTracker 汇总本节点所有 Task 的状态，再通过心跳汇报给 JobTracker。JobTracker 更新全局作业进度视图。

#### 5. 作业完成
当所有 Task（包括 Map 和 Reduce）执行成功后，JobTracker 将作业状态标记为 **成功**，并通知 JobClient。

### 实战案例
在一次 ETL 作业中，作业并未报错但一直卡在 99% 的进度。经查看 Web UI，发现是最后几个 Reduce 任务因为数据倾斜处理极慢。通过设置 `mapreduce.job.maxtaskfailures.per.tracker` 增加重试次数，并结合自定义 Partitioner 解决了倾斜问题，最终作业顺利完成。

### 生命周期阶段对比
| 阶段 | 核心组件 | 关键动作 | 潜在瓶颈 |
| :--- | :--- | :--- | :--- |
| **提交/初始化** | Client, JobTracker | 上传Jar, 计算Split | 小文件过多导致Split数爆炸，NameNode压力大 |
| **调度** | JobTracker, Scheduler | 资源分配, 本地性计算 | 集群资源耗尽, 等待Slot时间过长 |
| **环境准备** | TaskTracker | 下载资源, 启动JVM | 磁盘IO繁忙导致JVM启动慢, Container启动超时 |
| **执行** | Task (Map/Reduce) | 读写数据, Shuffle | 数据倾斜, GC过长, 磁盘满 |
| **完成** | JobTracker | 清理临时文件, 更新状态 | _commit_ 操作延迟, 输出目录权限问题 |

### 关键代码 (Java - 作业完成回调)
```java
// 实战：利用作业完成回调发送通知
job.waitForCompletion(true);

if (job.isSuccessful()) {
    Counters counters = job.getCounters();
    // 获取自定义计数器，例如处理的总记录数
    long totalRecs = counters.findCounter("MyGroup", "TotalRecords").getValue();
    System.out.println("Job Finished Successfully. Total Records: " + totalRecs);
    // 发送邮件或告警通知
    sendAlert("Job Success", "Total records processed: " + totalRecs);
} else {
    System.err.println("Job Failed");
}
```
