# 解释一下页面置换算法，例如LRU（最近最少使用）、FIFO（先进先出）等？

页面置换算法是操作系统中在内存不足时，选择将哪些内存页面（数据）换出到磁盘，为新页面腾出空间的策略。常见算法包括：

**1. LRU（最近最少使用）**
*   **原理**：选择最长时间未被访问的页面进行淘汰。基于“局部性原理”，认为最近使用过的页面在未来被访问的概率较高。
*   **实现**：需要维护访问时间或顺序。常见实现包括双向链表+哈希表（O(1) 时间复杂度），或利用硬件的访问位进行近似模拟（如 Clock 算法）。

**2. FIFO（先进先出）**
*   **原理**：选择最早进入内存的页面进行淘汰。类似于队列，先来后到。
*   **特点**：实现简单，但性能较差。它可能会淘汰经常使用的页面（Belady 异常现象：分配给进程的物理页数增加，缺页率反而上升）。

**## 架构示意图（LRU 链表结构）**
LRU 通常使用哈希表配合双向链表实现快速查找和更新：
```text
    哈希表 (快速查找)          双向链表 (维护访问顺序)
┌───────────────┐       ┌─────┐ <────> ┌─────┐ <────> ┌─────┐
│ Key: Page1    │ ────> │ P3  │       │ P1  │       │ P4  │
│ Val: Node(P3) │       └──┬──┘       └──┬──┘       └──┬──┘
└───────────────┘          │ Head      │ Mid       │ Tail
                          (最近)      (一般)      (最久未用 -> 淘汰)
```

**## 常见考点**
1.  什么是 Belady 异常？哪些算法会出现？（FIFO 会出现，LRU 和最优置换算法 OPT 不会）
2.  如何在系统层面高效实现 LRU？（硬件支持位、Clock 算法/时钟置换算法）
3.  LRU 和 LFU（最不经常使用）的区别？（LRU 关注时间间隔，LFU 关注访问频率；LFU 可能导致旧数据因频率高而不被淘汰，LRU 反应更灵敏）

**## 实战案例**
在使用 **Redis** 作为缓存时，如果选择 `maxmemory-policy=allkeys-lru`，当热点数据突然发生变更或被大量冷数据访问（如爬虫抓取）时，LRU 可能会误淘汰真正的热点数据。实战中常采用 **LFU** 策略来保留高频访问的稳定数据，或者使用 **LRU-K**（记录前 K 次访问时间）来避免“缓存污染”问题。

**## 代码示例 (Java: LRU Node)**
```java
// LinkedHashMap 可以轻松实现 LRU，只需重写 removeEldestEntry
// 以下为双向链表节点的核心逻辑示意
class Node<K, V> {
    K key;
    V value;
    Node prev, next; // 双向指针，O(1) 移动节点

    public Node(K key, V value) {
        this.key = key;
        this.value = value;
    }
}
```

## 记忆要点

- LRU淘汰最久未访问，基于局部性原理；FIFO淘汰最先进入，只看先后顺序。
- LRU用哈希表+双向链表实现O(1)复杂度；FIFO实现简单但性能较差。
- 因为分配页数增加缺页率反而上升，所以叫Belady异常，仅FIFO会出现。
- LRU关注时间，LFU关注频率；Redis等缓存实战中常用LFU避免缓存污染。
