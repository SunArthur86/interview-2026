# 任务运行环境准备

任务运行环境准备是指 TaskTracker (MRv1) 或 NodeManager (YARN) 在接收到任务后，为 Task 的实际执行提供隔离环境和必要资源的过程。这一步至关重要，确保不同任务之间互不干扰。

### 实战案例
在生产环境中，曾遇到因 JVM 重用配置不当导致的“任务串行”问题。开启了 JVM 重用后，前一个任务的静态变量未被清理，导致后续复用该 JVM 的任务读取到了脏数据。最终通过关闭重用或在代码的 `cleanup()` 方法中显式重置状态解决了该问题。

### 关键步骤

#### 1. JVM 启动与复用
*   **独立 JVM**：TaskTracker 为每个 Task 启动一个独立的 JVM 进程。
*   **原因**：
    *   **隔离性**：防止用户代码导致 TaskTracker 崩溃（如无限循环、内存溢出）。
    *   **安全**：防止不同作业的用户代码相互干扰或窥探数据。
*   **JVM 重用**：为了减少启动 JVM 的开销，Hadoop 支持 JVM 重用（通过 `mapreduce.job.jvm.numtasks` 配置）。在一个 TaskTracker 上，顺序执行的多个小任务可以串行使用同一个 JVM，但不同任务间不共享变量状态。

#### 2. 资源隔离
*   **基于进程的隔离**：利用操作系统进程级别的内存和 CPU 隔离。
*   **资源限制**：
    *   TaskTracker 会监控 Task 的资源使用情况。如果 Task 超过了分配的内存限制（例如 Map Task 占用内存超过上限），TaskTracker 会主动杀掉该 Task 以防止整个节点宕机。
    *   通常会结合 Linux 的 Shell 脚本（如 `task-controller`）来设置 ulimit 或 cgroup 限制。

#### 3. 任务本地化
*   **文件分发**：TaskTracker 从 HDFS 将作业的 Jar 包、配置文件、Distributed Cache 文件下载到本地工作目录。
*   **工作目录结构**：通常为 `${mapred.local.dir}/taskTracker/${user}/jobcache/${jobId}/${attemptId}/work/`。

### 隔离技术对比
| 特性 | 进程级隔离 (MRv1/Basic YARN) | Cgroup (YARN NodeManager) |
| :--- | :--- | :--- |
| **实现方式** | 单独JVM进程，内存仅基于Heap限制 | Linux Kernel Cgroups，严格限制CPU/内存/IO |
| **内存控制** | 较弱，仅监控Heap，物理内存超限可能OOM | 强，可限制物理内存，直接Kill进程 |
| **CPU控制** | 依赖操作系统抢占式调度 | 可设置CPU份额或核心数绑定 |
| **适用场景** | 简单部署，非多租户 | 多租户混合部署，防止资源争抢 |
