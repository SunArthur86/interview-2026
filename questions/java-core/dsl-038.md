# 计算机系统&Linux

### 代理模式和适配器模式区别

代理模式和适配器模式都属于结构型设计模式，核心区别如下：

| 维度 | 代理模式 | 适配器模式 |
| :--- | :--- | :--- |
| **核心目的** | 控制对对象的访问，添加额外逻辑（如限流、懒加载） | 解决接口不兼容，让两个不兼容的接口能协同工作 |
| **封装对象** | 真实对象 | 被适配者 |
| **结构组成** | 抽象主题、真实主题、代理 | 目标接口、适配器、被适配者 |
| **接口一致性** | 代理接口与真实主题接口通常一致 | 适配器接口与被适配者接口通常不同 |
| **典型场景** | 远程代理、虚拟代理（图片占位）、保护代理（权限控制） | 旧系统接口改造、第三方 SDK 封装、数据格式转换 |

**实战案例**：
- **代理**：在前端项目中，通过 Proxy 对象拦截 Vue 的 data 属性，实现数据劫持和双向绑定。
- **适配器**：后端接入支付宝和微信支付时，两者 API 接口不同，封装一个统一的 PayAdapter 接口，内部分别调用不同的 SDK。

**代码示例（适配器模式）**：
```javascript
// 目标接口
class Target { request() { console.log('普通请求'); } }

// 被适配者
class Adaptee { specificRequest() { console.log('特殊请求'); } }

// 适配器
class Adapter extends Target {
  constructor(adaptee) { super(); this.adaptee = adaptee; }
  request() { this.adaptee.specificRequest(); }
}
```

### 字符编码 (基础知识)

我们知道一个字节有8位，8位在不考虑正负的情况下，能够表示256个数字。同理2个字节就能表示出65536个数字。那么对于字符该如何表示？

通过计算机领域的至理名言：计算机科学领域的任何问题都可以通过增加一个间接的中间层来解决。我们通过在字符和二进制编码之间嵌套一个数字编号，通过数字编号，就能实现字符、二进制编码之间的关系绑定。

## 技术原理

**代理模式：控制对象访问，增加额外操作**
代理模式（Proxy）为真实对象提供一个替身，外部不直接访问真实对象而是通过代理。代理与真实对象实现相同接口，在调用前后可以织入额外逻辑：远程代理（隐藏网络调用细节）、虚拟代理（延迟加载，如大图片占位）、保护代理（权限校验）、智能引用（计数、缓存）。典型应用是 Spring AOP，通过动态代理在方法前后织入事务、日志、限流。

**适配器模式：转换接口，解决不兼容问题**
适配器模式（Adapter）将一个类的接口转换成客户端期望的另一个接口，让原本接口不兼容的类能协同工作。它连接的是两个完全不同的接口，核心作用是"接口转换"。典型场景：旧系统接口改造（保留老接口内部转调新接口）、第三方 SDK 统一封装（支付宝和微信支付 API 不同，统一成 PayService 接口）。

**结构型模式，关注类与对象的组合**
两者都属于结构型设计模式，关注的是类与对象的组合关系。代理强调"控制访问"（接口不变，加逻辑），适配器强调"转换接口"（接口改变，做兼容）。判断依据：接口是否一致——一致是代理，不一致是适配器。

## 代码示例

```java
// 1. 代理模式（JDK 动态代理：接口一致，增加日志/权限逻辑）
public interface UserService {
    User findById(Long id);
}
public class LogProxy implements InvocationHandler {
    private final Object target;
    public LogProxy(Object target) { this.target = target; }
    public Object invoke(Object proxy, Method m, Object[] args) throws Throwable {
        System.out.println("[LOG] 调用 " + m.getName());  // 前置逻辑
        Object result = m.invoke(target, args);
        System.out.println("[LOG] 返回 " + result);        // 后置逻辑
        return result;
    }
}
UserService proxy = (UserService) Proxy.newProxyInstance(
    loader, new Class[]{UserService.class}, new LogProxy(realService));
```

```java
// 2. 适配器模式（统一不同支付 SDK 的接口）
public interface PayService { void pay(String orderId, BigDecimal amount); }

// 支付宝 SDK（接口不兼容，需适配）
public class AlipayAdapter implements PayService {
    private final AlipayClient client = new AlipayClient();
    public void pay(String orderId, BigDecimal amount) {
        client.tradePay(orderId, amount.doubleValue()); // 转调第三方 SDK
    }
}
// 微信支付 SDK
public class WechatAdapter implements PayService {
    private final WxPaySdk sdk = new WxPaySdk();
    public void pay(String orderId, BigDecimal amount) {
        sdk.unifiedOrder(orderId, amount.intValue());   // 接口不同，转换
    }
}
```

## 注意事项

- 目的对比：代理重在控制访问加逻辑，而适配器重在解决接口不兼容问题。
- 接口差异：代理与原对象接口一致，而适配器连接完全不同的两个接口。
- 应用区分：Proxy 实现 Vue 双向绑定是代理，统一支付 SDK 接口是适配器。
- 字符编码：因为字符与二进制需映射，所以引入 Unicode 中间层，MySQL 需警惕乱码。
- 装饰器模式与代理结构相同但意图不同：装饰器强调"增强功能"（可叠加多层），代理强调"控制访问"。
