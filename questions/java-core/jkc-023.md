# 序列化用户远程对象传输

Java 序列化是实现 Java RMI（远程方法调用）和分布式系统的基础。当需要在网络上传输对象时，由于网络只能传输字节流，必须将对象序列化为字节序列，接收端再进行反序列化以恢复对象。这使得不同 JVM 之间的对象通信变得简单，Java 序列化 API 提供了标准机制来处理这一过程。

**增强细节与原理**：
*   **RMI 机制**：在 RMI 调用中，客户端调用的参数和返回值如果是复杂对象，必须实现 Serializable 接口。RMI 框架底层使用 JRMP（Java Remote Method Protocol），其传输层依赖于序列化。
*   **分布式对象传输**：不仅仅是 RMI，在微服务调用（如 Dubbo 默认支持 Hessian 及 Java 序列化）、消息队列（JMS）传输 ObjectMessage 时，都需要序列化。
*   **Stub 与 Skeleton**：在早期的 RMI 模型中，客户端拥有 Stub（桩），服务端拥有 Skeleton（骨架）。Stub 负责将调用请求和参数序列化后发送给 Skeleton，Skeleton 反序列化后调用真实对象，并将结果反向序列化传回。

**数据流示意图**：
```text
Client JVM                       Network                        Server JVM
┌───────────────┐                                              ┌───────────────┐
│  Object (A)   │                                              │  Object (A')  │
│  (Original)   │                                              │ (Reconstructed)│
└───────┬───────┘                                              └───────▲───────┘
        │ Serialization                                                 │
        ▼                                                               │
┌───────────────┐  Serialization Stream (Bytes)  ┌───────────────────────┐│
``` 

### 实战案例
在某金融支付网关升级中，服务端新增了一个敏感字段 `securityToken`。由于未升级的客户端仍然使用旧版本类定义进行反序列化，导致解析异常。生产环境中，通常建议避免直接使用 Java 原生序列化进行跨服务传输，因其对类结构变异极其敏感，且存在安全隐患。

### 序列化方案对比 (实战选型)
| 特性 | Java Native Serialization | Protobuf / Hessian | JSON (Jackson/Gson) |
| :--- | :--- | :--- | :--- |
| **传输效率** | 低 (字节码冗余多) | 高 (二进制紧凑) | 中 (文本体积大) |
| **序列化速度** | 慢 (反射机制重) | 快 | 较慢 (需构建AST) |
| **跨语言支持** | 仅限 Java | 优秀 (C++, Python, Go等) | 优秀 (通用标准) |
| **可读性** | 不可读 (二进制乱码) | 不可读 | 优秀 (便于人眼排查) |
| **破坏性变更** | 敏感 (新增字段可能报错) | 兼容性好 (需配置) | 容错性强 (忽略未知字段) |

## 常见考点
1.  **序列化版本不一致**：在分布式系统中，服务提供方和消费方的类定义可能不同步（如新增字段）。如何通过 `serialVersionUID` 和兼容性策略处理这种差异。
2.  **替代协议**：Java 原生序列化在网络传输中效率低且跨语言支持差（很难被 Python、C++ 解析）。面试中常问及使用 JSON、Protobuf、Thrift 或 Avro 替代 Java 序列化的原因和场景。
3.  **安全性**：反序列化不可信网络数据极易遭受远程代码执行（RCE）攻击。在生产环境直接反序列化外部输入是非常危险的。

## 技术原理

序列化的本质是**把内存中分散的对象图（对象图通过引用连接）拍平成线性的字节序列**，反序列化时再重建对象图。这涉及引用处理、版本兼容、安全三大核心问题。

- **对象图的遍历与引用处理**：一个对象可能引用其他对象（成员变量），序列化时必须递归遍历整个对象图（深度优先）。为处理循环引用（A 引用 B，B 又引用 A），`ObjectOutputStream` 维护一个 `IdentityHashMap` 记录已序列化的对象，遇到重复的写引用编号而非重复序列化。`transient` 关键字标记的字段跳过，`static` 字段属于类不属于实例所以也不序列化。
- **serialVersionUID 的版本兼容机制**：反序列化时 JVM 会比对字节流里的 `serialVersionUID` 和本地类的 `serialVersionUID`，不一致直接抛 `InvalidClassException`。如果不显式声明，编译器会根据类结构（字段、方法签名）自动生成一个 hash——这意味着"加一个字段"都会让自动生成的 UID 变化，导致老版本客户端反序列化失败。所以**强烈建议显式声明 `serialVersionUID`**，让类结构变化可控。
- **RMI 的 Stub/Skeleton 演进**：早期 Java RMI 需要用 `rmic` 编译生成 Stub（客户端代理）和 Skeleton（服务端骨架），手工处理序列化。Java 5 后用动态代理替代 Stub，Skeleton 直接合并进服务端，开发简化。RMI 底层用 JRMP 协议，只支持 Java-Java 通信，这是它无法跨语言的根因。
- **反序列化漏洞的根因**：Java 序列化字节流里包含类名，反序列化时 `ObjectInputStream.readObject()` 会根据类名加载类并实例化。如果攻击者构造特殊字节流，触发某些"反序列化时自动执行代码"的类（如 `AnnotationInvocationHandler`、`TemplatesImpl`），就能实现 RCE（远程代码执行）。Apache Commons Collections 的反序列化漏洞就是这么来的——生产环境绝对不要反序列化外部不可信数据。

## 代码示例

```java
// 1. 基础序列化：实现 Serializable + 显式 serialVersionUID
import java.io.*;

public class User implements Serializable {
    // 强烈建议显式声明，避免类变化导致反序列化失败
    private static final long serialVersionUID = 1L;

    private String name;
    private int age;
    private transient String password;   // transient 不序列化（敏感字段）
    private static String GLOBAL = "x";  // static 不序列化

    public User(String name, int age, String password) {
        this.name = name;
        this.age = age;
        this.password = password;
    }
}

// 序列化到文件
User user = new User("Alice", 30, "secret");
try (ObjectOutputStream oos = new ObjectOutputStream(
        new FileOutputStream("user.dat"))) {
    oos.writeObject(user);
}

// 反序列化
try (ObjectInputStream ois = new ObjectInputStream(
        new FileInputStream("user.dat"))) {
    User restored = (User) ois.readObject();
    // restored.password == null（transient 不还原）
}
```

```java
// 2. RMI 远程调用示例（底层依赖序列化）
import java.rmi.*;

// 远程接口必须继承 Remote，方法必须 throws RemoteException
public interface UserService extends Remote {
    User getUser(Long id) throws RemoteException;   // User 必须 Serializable
}

// 服务端实现
public class UserServiceImpl extends UnicastRemoteObject implements UserService {
    public UserServiceImpl() throws RemoteException {}

    @Override
    public User getUser(Long id) throws RemoteException {
        return userDao.findById(id);   // 返回值会序列化传给客户端
    }
}

// 注册服务
LocateRegistry.createRegistry(1099);
Naming.rebind("rmi://localhost/UserService", new UserServiceImpl());

// 客户端调用（参数和返回值都走序列化）
UserService stub = (UserService) Naming.lookup("rmi://localhost/UserService");
User u = stub.getUser(1L);
```

```java
// 3. 自定义序列化：控制序列化逻辑（敏感数据加密、版本兼容）
public class SecureUser implements Serializable {
    private static final long serialVersionUID = 2L;
    private String name;
    private String creditCard;

    // 自定义序列化逻辑
    private void writeObject(ObjectOutputStream oos) throws IOException {
        oos.defaultWriteObject();              // 先走默认序列化
        oos.writeUTF(encrypt(creditCard));     // 信用卡加密后再写
    }

    private void readObject(ObjectInputStream ois)
            throws IOException, ClassNotFoundException {
        ois.defaultReadObject();
        this.creditCard = decrypt(ois.readUTF());   // 读出后解密
    }

    private String encrypt(String s) { return Base64.getEncoder().encodeToString(s.getBytes()); }
    private String decrypt(String s) { return new String(Base64.getDecoder().decode(s)); }
}
```

```java
// 4. 生产推荐：用 Protobuf 替代 Java 原生序列化
// 定义 proto 文件后生成的类
UserProto.User protoUser = UserProto.User.newBuilder()
    .setName("Alice")
    .setAge(30)
    .build();
byte[] bytes = protoUser.toByteArray();   // 序列化（体积小 5-10 倍）
UserProto.User restored = UserProto.User.parseFrom(bytes);  // 反序列化
```

## 常见坑

- **不显式声明 serialVersionUID 是定时炸弹**：类任何结构变化（加字段、改方法签名）都会让自动生成的 UID 变化，导致线上灰度发布时老版本反序列化失败。所有 `Serializable` 类必须显式 `private static final long serialVersionUID = 1L;`。
- **反序列化外部输入是高危操作**：`readObject` 会触发类加载和初始化代码，构造恶意字节流可 RCE。生产环境绝不要反序列化 HTTP 请求体、MQ 消息等不可信数据，用 JSON/Protobuf 替代。
- **transient 不等于不传输**：`transient` 字段不参与序列化，但通过自定义 `writeObject` 仍可手动写入。反过来，以为加了 `transient` 就安全了而忘记自定义加密，敏感数据仍可能泄露。
- **序列化静态字段需手动**：`static` 字段不序列化（属于类），如果业务需要传输静态配置，要单独处理或在 `writeObject` 里手动写。
- **序列化会触发父类构造器**：如果父类不实现 `Serializable`，反序列化时会调用父类的无参构造器——所以父类必须有无参构造器，否则反序列化失败。
- **Lambda/匿名类不可序列化**：默认不支持，需实现 `Serializable` 的函数接口（如 `Function<T,R> & Serializable`），但会产生额外的脱合成类，性能差且调试困难。
- **序列化单例会被破坏**：`readObject` 会创建新实例绕过单例控制。单例类需实现 `readResolve()` 返回单例实例，或用枚举单例（枚举的序列化由 JVM 保证唯一性）。
