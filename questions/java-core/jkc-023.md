# 序列化用户远程对象传输

Java 序列化是实现 Java RMI（远程方法调用）和分布式系统的基础。当需要在网络上传输对象时，由于网络只能传输字节流，必须将对象序列化为字节序列，接收端再进行反序列化以恢复对象。这使得不同 JVM 之间的对象通信变得简单，Java 序列化 API 提供了标准机制来处理这一过程。

**增强细节与原理**：
*   **RMI 机制**：在 RMI 调用中，客户端调用的参数和返回值如果是复杂对象，必须实现 Serializable 接口。RMI 框架底层使用 JRMP（Java Remote Method Protocol），其传输层依赖于序列化。
*   **分布式对象传输**：不仅仅是 RMI，在微服务调用（如 Dubbo 默认支持 Hessian 及 Java 序列化）、消息队列（JMS）传输 ObjectMessage 时，都需要序列化。
*   **Stub 与 Skeleton**：在早期的 RMI 模型中，客户端拥有 Stub（桩），服务端拥有 Skeleton（骨架）。Stub 负责将调用请求和参数序列化后发送给 Skeleton，Skeleton 反序列化后调用真实对象，并将结果反向序列化传回。

**数据流示意图**：
```text
Client JVM                       Network                        Server JVM
┌───────────────┐                                              ┌───────────────┐
│  Object (A)   │                                              │  Object (A')  │
│  (Original)   │                                              │ (Reconstructed)│
└───────┬───────┘                                              └───────▲───────┘
        │ Serialization                                                 │
        ▼                                                               │
┌───────────────┐  Serialization Stream (Bytes)  ┌───────────────────────┐│
``` 

### 实战案例
在某金融支付网关升级中，服务端新增了一个敏感字段 `securityToken`。由于未升级的客户端仍然使用旧版本类定义进行反序列化，导致解析异常。生产环境中，通常建议避免直接使用 Java 原生序列化进行跨服务传输，因其对类结构变异极其敏感，且存在安全隐患。

### 序列化方案对比 (实战选型)
| 特性 | Java Native Serialization | Protobuf / Hessian | JSON (Jackson/Gson) |
| :--- | :--- | :--- | :--- |
| **传输效率** | 低 (字节码冗余多) | 高 (二进制紧凑) | 中 (文本体积大) |
| **序列化速度** | 慢 (反射机制重) | 快 | 较慢 (需构建AST) |
| **跨语言支持** | 仅限 Java | 优秀 (C++, Python, Go等) | 优秀 (通用标准) |
| **可读性** | 不可读 (二进制乱码) | 不可读 | 优秀 (便于人眼排查) |
| **破坏性变更** | 敏感 (新增字段可能报错) | 兼容性好 (需配置) | 容错性强 (忽略未知字段) |

## 常见考点
1.  **序列化版本不一致**：在分布式系统中，服务提供方和消费方的类定义可能不同步（如新增字段）。如何通过 `serialVersionUID` 和兼容性策略处理这种差异。
2.  **替代协议**：Java 原生序列化在网络传输中效率低且跨语言支持差（很难被 Python、C++ 解析）。面试中常问及使用 JSON、Protobuf、Thrift 或 Avro 替代 Java 序列化的原因和场景。
3.  **安全性**：反序列化不可信网络数据极易遭受远程代码执行（RCE）攻击。在生产环境直接反序列化外部输入是非常危险的。

## 记忆要点

- 核心目的：因为网络只能传字节流，所以必须序列化对象，是RMI和分布式传输的基础
- 性能对比：Java原生序列化低效且仅限Java，生产多用JSON或Protobuf替代
- 安全隐患：反序列化外部输入极易触发RCE攻击，生产环境极力避免原生序列化
