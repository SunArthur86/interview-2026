# 场景题 & 系统设计

设计一个高性能的秒杀系统需要针对“瞬时高并发、读多写少、库存有限”的特点进行架构优化。核心设计理念包括限流、削峰、异步处理、内存缓存和可扩展性。

**1. 架构设计思路**
- **限流**：在系统入口处（如网关层）拦截大部分无效或超额请求，只允许少量请求进入后端逻辑。常用手段有 Nginx 限制连接数、应用层限流算法（令牌桶、漏桶）。
- **削峰填谷**：利用消息队列（MQ）将瞬间的海量请求缓存起来，后端按照自己的处理能力平滑消费，避免压垮数据库。
- **异步处理**：将业务流程拆解，非核心流程（如发短信、更新积分）通过 MQ 异步执行，核心流程（扣减库存）同步或异步快速处理，以此提高响应速度和吞吐量。
- **内存缓存**：秒杀读多写少，利用 Redis 预减库存。所有请求先在 Redis 层进行库存扣减，扣减成功的才进入 DB 层下单，极大减少数据库压力。
- **水平扩展**：设计无状态服务，支持动态扩容。流量激增时通过 Kubernetes 等容器编排快速增加 Pod 数量。

**3. 关键细节与边界条件**
- **超卖问题**：利用 Redis 的 `decr` 原子性操作预减库存，或者使用 Lua 脚本保证“检查库存”和“扣减库存”的原子性。数据库层面使用乐观锁（`UPDATE stock SET num = num - 1 WHERE num > 0`）。
- **允许少量等待**：前端点击秒杀后，按钮置灰，客户端轮询查询秒杀结果，而不是长时间阻塞等待服务端响应。
- **热点隔离**：将秒杀单独部署独立的域名或服务，避免流量洪峰冲垮主站业务。

**实战案例**：在某大促活动中，因 Redis 缓存预热未完成且没有降级开关，流量涌入穿透 Redis 直接打到 MySQL，导致主库 CPU 飙升 100%。后续引入了“Redis 宕机自动降级返回默认页”的熔断机制。

**代码示例（Redis Lua 脚本扣库存）**：
```lua
-- redis_lua_script.lua
local stock = redis.call('get', KEYS[1])
if not stock then return -1 end
if tonumber(stock) > 0 then
    return redis.call('decr', KEYS[1])
else
    return 0
end
```

**对比：秒杀三种扣库存方案**

| 方案 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **直接扣数据库** | 实现简单，数据绝对一致 | 并发性能极差，DB 容易死锁 | 低并发内部系统 |
| **Redis 扣减 + DB 异步** | 性能极高，支持海量并发 | 可能少卖（Redis 与 DB 数据不一致），需补偿 | 高并发秒杀，允许少量少卖 |
| **Redis 预减 + 扣减 DB** | 性能较好，数据相对准确 | 实现较复杂，需处理 MQ 消费堆积 | 大多数电商秒杀场景 |

## 常见考点
1. 如何解决秒杀系统中的“超卖”问题？（Redis 原子性、数据库乐观锁）
2. 消息队列在秒杀中起到了什么作用？（削峰填谷、异步解耦）
3. 如果 Redis 挂了怎么办？（降级方案、本地库存缓存兜底或直接拒绝流量）
