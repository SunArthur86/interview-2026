# 深拷贝和浅拷贝有什么区别？

**深拷贝**和**浅拷贝**的区别：

**1. 核心区别**
*   **浅拷贝**：只复制对象本身的基本数据类型字段，对于引用类型字段，仅复制引用地址（指针），不复制引用指向的对象。因此，新对象和原对象共享引用类型字段指向的内存空间，修改其中一个会影响另一个。
*   **深拷贝**：不仅复制对象本身，还会递归复制所有引用类型字段指向的对象。修改新对象不会影响原对象，两者完全独立。

**2. 实现方式与边界条件**
*   **浅拷贝实现**：在 Java 中，通常通过实现 `Cloneable` 接口并调用 `super.clone()` 实现。
*   **深拷贝实现**：
    *   **重写 clone() 方法**：在 `clone()` 方法中对所有引用类型字段手动调用 `clone()`。注意多层嵌套对象的递归处理。
    *   **序列化**：将对象序列化为字节流（二进制），再反序列化回对象。这种方法能自动处理多层嵌套和循环引用（前提是对象图可序列化），且无需修改每个引用类型的代码，但性能开销较大。

**3. 实战深化与对比**

#### 1. 实战案例
*   **原型模式**：在创建大对象（如复杂的配置对象或游戏地图场景）时，为了性能优化，往往使用原型模式。如果浅拷贝导致修改了原型中的引用对象（如修改了地图中的怪物列表），会导致所有基于该原型生成的实例都被污染。实战中必须深拷贝（或对引用字段做防御性拷贝）。
*   **不可变对象**：在实战中，为了保证线程安全，通常会将对象设计为不可变（如 `String`、所有 `final` 字段）。当返回内部引用类型时（如 `Date[]`），如果不进行深拷贝，外部调用者可以直接修改数组内容，破坏封装性。推荐在 getter 中返回 `Collections.unmodifiableList` 或拷贝后的副本。

#### 2. 代码示例 (Java - 序列化深拷贝)
```java
import java.io.*;

@SuppressWarnings("unchecked")
public static  T deepClone(T obj) throws IOException, ClassNotFoundException {
    // 1. 序列化到内存流（字节数组输出流），避免磁盘 IO
    ByteArrayOutputStream bos = new ByteArrayOutputStream();
    ObjectOutputStream oos = new ObjectOutputStream(bos);
    oos.writeObject(obj);
    oos.flush();

    // 2. 从内存流反序列化，生成全新对象
    ByteArrayInputStream bis = new ByteArrayInputStream(bos.toByteArray());
    ObjectInputStream ois = new ObjectInputStream(bis);
    return (T) ois.readObject();
}
```

#### 3. 选型对比表

| 维度 | 浅拷贝 | 深拷贝 | 推荐方案 |
| :--- | :--- | :--- | :--- |
| **实现复杂度** | 低 (一行代码) | 高 (需递归或序列化) | 简单对象用 `super.clone()` |
| **性能开销** | 低 (仅复制引用) | 高 (需创建所有嵌套对象) | 性能敏感场景慎用序列化 |
| **数据独立性** | 差 (共享引用) | 好 (完全独立) | 需要隔离修改时必用 |
| **循环引用** | 安全 | 序列化方式安全，手动 Clone 易死循环 | 优先选序列化处理复杂图 |

**## 常见考点**
1.  如何修改 `clone()` 方法来实现深拷贝？（需逐层调用引用类型的 clone）
2.  为什么推荐使用序列化实现深拷贝，它的缺点是什么？（缺点是性能慢，且所有对象必须实现 `Serializable`）
3.  如果引用对象中包含不可变对象（如 String），是否需要深拷贝？（不需要，不可变对象天生线程安全且不可变，共享引用无风险）
