# 什么是HTTP/1.1？

HTTP/1.1 是目前使用最广泛的 HTTP 协议版本，它在 HTTP/1.0 的基础上进行了显著的性能和功能增强，旨在解决性能瓶颈并扩展协议能力。它是 HTTP/2.0 出现前长期的标准。

**主要特性详解**：

1. **长连接**：
   HTTP/1.0 默认为短连接，每次请求都需要建立和断开 TCP 连接（三次握手/四次挥手开销大）。HTTP/1.1 默认开启 `Connection: keep-alive`，允许在同一个 TCP 连接上连续发送多个请求和响应，减少了 TCP 建立连接的开销和延迟。

2. **管道化**：
   基于长连接，HTTP/1.1 引入了管道机制。客户端可以不用等待前一个请求的响应回来，就直接发送下一个请求。
   
   **局限性**：
   - 管道化要求响应必须按照请求的顺序返回（FIFO）。
   - 如果前一个请求处理耗时过长，后续的请求即使处理完了也必须等待（队列前部阻塞），这被称为**队头阻塞**。
   - 实际上，由于代理服务器和服务器实现的复杂性，大多数浏览器默认并没有开启 Pipelining。

3. **缓存处理**：
   引入了更多缓存控制字段，如 `Cache-Control`, `Etag`, `If-Match`, `If-None-Match` 等，提供了比 1.0 更精细的缓存策略控制。

4. **Host 头域**：
   在 HTTP/1.0 中，一个 IP 只能对应一个域名。HTTP/1.1 增加了 `Host` 请求头，使得一台服务器（同一个 IP）可以托管多个虚拟网站，服务器根据 `Host` 头区分访问的是哪个站点。

5. **带宽优化与断点续传**：
   引入了 `Range` 头域（请求头 `Range: bytes=0-1023`），允许客户端只请求资源的某一部分。服务器返回状态码 `206 Partial Content`。这支持了断点续传和视频的分段加载。

6. **错误通知**：
   新增了 24 个状态码（如 409 Conflict, 410 Gone 等），更精准地描述服务器状态。

**HTTP/1.1 请求流程（长连接+管道化示意）**：
```
客户端                                        服务端
  │                                            │
  │  ┌──── TCP 连接建立 ────┐                  │
  │<────────────────────────>                  │
  │  └──────────────────────┘                  │
  │                                            │
  ├─ Request A ──────────────────────────────>│
  ├─ Request B (不等A响应，直接发送) ────────>│ (队列处理)
  ├─ Request C ──────────────────────────────>│
  │                                            │
  │<──────────────── Response A ──────────────┤ (必须先返回A)
  │<──────────────── Response B ──────────────┤
  │<──────────────── Response C ──────────────┤
  │                                            │
  │  ┌──── TCP 连接保持/断开 ────┐              │
  │<────────────────────────>                  │
  │  └───────────────────────────┘              │
```

**实战案例**：在视频点播场景中，若不支持 `Range` 请求，用户拖动进度条时需重新下载整个视频文件，流量消耗巨大且体验极差；使用 HTTP/1.1 的 Range 头可实现精准分段加载。

**代码示例**：
```go
// Go: 服务器处理 Range 请求示例
func handler(w http.ResponseWriter, r *http.Request) {
    file, _ := os.Open("video.mp4")
    defer file.Close()
    // 解析 Range 头: bytes=0-1023
    rangeHeader := r.Header.Get("Range")
    if rangeHeader != "" {
        // ...解析 range 参数，计算 offset
        w.Header().Set("Content-Range", fmt.Sprintf("bytes %d-%d/%d", start, end, size))
        w.WriteHeader(http.StatusPartialContent) // 206
        io.CopyN(w, file, end-start+1)
    }
}
```

## 常见考点
1. **HTTP/1.1 的队头阻塞问题**：虽然管道化允许并发发送请求，但为什么还说它有队头阻塞？（因为响应必须串行返回）。
2. **Keep-Alive 的优缺点**：长连接一直占用资源，如何管理？（通过 timeout 和 max 参数控制）。
3. **为什么 HTTP/1.1 实际上并不能实现真正的多路复用**？（这是 HTTP/2 的特性）。


## 记忆要点

- 长连接：默认开启keep-alive，复用TCP连接省去了频繁握手开销。

- 缓存控制：引入Etag和Cache-Control等灵活字段，替代1.0简陋的Expires。
- 断点续传：新增Range头支持分段请求资源（返回206状态码）。
- 致命痛点：管道化要求响应FIFO串行返回，依然存在HTTP层严重的队头阻塞。
