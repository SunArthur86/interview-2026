# 一致性（Consistency）

数据库事务（ACID）

事务是作为单个逻辑工作单元执行的一系列操作，这些操作作为一个整体向系统提交，要么都执行、要么都不执行。事务必须具备以下四个属性（ACID）：

1.  **原子性**：事务是一个完整的操作，事务的各步操作是不可分的（原子的）；要么全做，要么全不做。
2.  **一致性**：当事务完成时，数据必须处于一致状态。即事务执行前后，数据库的完整性约束没有被破坏。
3.  **隔离性**：对数据进行修改的所有并发事务是彼此隔离的，表明事务必须是独立的，不应以任何方式依赖于或影响其他事务。
4.  **永久性**：事务完成后，它对数据库的修改被永久保持，事务日志能够保持事务的永久性。

### ACID 实现与 CAP 理论的关系
在分布式数据库领域，一致性（Consistency）这个术语容易产生歧义：
- **ACID 中的 Consistency**：指数据库状态符合业务规则和约束（如余额不能为负），更多指应用层面的状态正确性。
- **CAP 定理中的 Consistency**：指强一致性，即所有节点在同一时间看到的数据是一致的。

在面试中回答一致性问题时，建议区分这两种场景。对于传统关系型数据库（单机事务），通常指 ACID 中的约束一致性；对于分布式系统（如 Cassandra, DynamoDB），通常指 CAP 中的数据一致性。

### 分布式一致性模型（补充）
针对分布式环境，一致性通常分为不同级别：
- **强一致性**：更新后，任何后续读取都能获取到最新值。
- **最终一致性**：更新后，如果不进行新的写入，系统最终（经过一段同步时间后）所有副本的数据将达到一致。

### 实战案例
在电商库存扣减场景中，若使用最终一致性（如 DynamoDB 默认设置），用户下单瞬间可能读到旧库存导致**超卖**。此类核心业务需配置为强一致性（Cassandra 中使用 `QUORUM` 或 `LOCAL_QUORUM` 读写），即使牺牲部分延迟也要保证数据准确。

### 一致性模型对比
| 模型 | 读取延迟 | 写入延迟 | 数据准确性 | 典型应用 |
| :--- | :--- | :--- | :--- | :--- |
| 强一致性 | 高 | 高 | 实时准确 | 银行转账、库存扣减 |
| 最终一致性 | 低 | 低 | 短暂不一致 | 社交动态、评论点赞 |
| 弱一致性 | 极低 | 极低 | 不保证 | DNS 缓存、实时统计 |

### 常见考点
1.  **一致性级别的权衡**：在 Cassandra 中如何调节一致性级别（ONE, QUORUM, ALL）？
2.  **ACID 中的 C 与 CAP 中的 C 的区别**：这是面试中极易混淆的概念。
3.  **BASE 理论**：Basically Available（基本可用）、Soft state（软状态）、Eventually consistent（最终一致性），如何补充 ACID 在高可用场景下的不足？

## 技术原理

- **ACID 的 C 是"业务约束级"一致性**：事务执行前后，数据库必须满足所有预定义约束——主键唯一、外键引用有效、CHECK 约束（如 `balance >= 0`）、唯一索引、触发器。它**不是数据库引擎强制的**，而是"如果事务正确编写 + 数据库约束到位，那么事务结束时数据状态必然合法"。C 是 A（原子性）+ I（隔离性）+ D（持久性）共同保障的**结果**——A 保证全做或全不做，I 保证并发不互相破坏，D 保证提交不丢，三者合力让 C 成立。
- **CAP 的 C 是"多副本数据级"一致性**：分布式系统中同一数据有多个副本，CAP 的 C 要求"读任意副本都返回最新写入的值"。这需要**写时同步所有副本**（强一致）或**读时聚合多数副本**（Quorum）。与 ACID 的 C 完全不同——ACID C 关注单机业务规则，CAP C 关注多副本同步。面试混淆二者是减分项，必须区分。
- **强一致 vs 最终一致的本质权衡**：强一致（如线性一致性）——任何读都看到最新写，但写要等所有副本 ACK，延迟高；最终一致——写完立即返回，副本异步同步，读可能看到旧值，但延迟低、可用性高。**这是 CAP 在 P（分区）必然下的 C vs A 权衡**——网络分区时，要么拒绝写（保 C 牺牲可用性），要么允许写但数据可能不一致（保 A 牺牲 C）。
- **Quorum 机制的数学基础**：N 个副本，写需 W 个 ACK，读需 R 个副本响应。只要 `W + R > N`，读和写必然有交集，读到最新数据的概率高。常见配置 `W=R=N/2+1`（多数派）。Cassandra 的 `QUORUM` 即此机制，让用户在延迟和一致性间动态调节。

## 对比/选型

| 模型 | 读延迟 | 写延迟 | 数据准确性 | 可用性 | 典型场景 |
|------|--------|--------|-----------|--------|---------|
| 强一致（线性） | 高 | 高 | 实时准确 | 低 | 银行转账、库存扣减 |
| 顺序一致 | 中 | 中 | 全局有序 | 中 | ZooKeeper |
| 因果一致 | 中 | 中 | 相关事件有序 | 中 | 社交时间线 |
| 读你所写 | 低 | 低 | 自读准确 | 高 | 用户资料 |
| 最终一致 | 低 | 低 | 短暂不一致 | 高 | 点赞、评论 |
| 弱一致 | 极低 | 极低 | 不保证 | 极高 | DNS、监控 |

## 代码示例

```java
// ===== 1. ACID 的 C：数据库约束保证业务一致性 =====
@Transactional
public void transfer(Long from, Long to, BigDecimal amount) {
    Account a = accountDao.findById(from);
    Account b = accountDao.findById(to);
    if (a.getBalance().compareTo(amount) < 0) {
        throw new InsufficientBalanceException();
    }
    a.setBalance(a.getBalance().subtract(amount));
    b.setBalance(b.getBalance().add(amount));
    accountDao.save(a);
    accountDao.save(b);
    // CHECK 约束 balance >= 0 由数据库保证（DDL: CHECK (balance >= 0)）
    // 若扣减后余额 < 0，DB 抛异常，整个事务回滚，C 成立
}

// ===== 2. 强一致（Quorum）—— Cassandra =====
// 写需多数副本 ACK，读也查多数副本
Statement write = SimpleStatement.builder("INSERT INTO orders (id, amount) VALUES (?, ?)")
    .setConsistencyLevel(ConsistencyLevel.QUORUM)   // W = N/2+1
    .build();
session.execute(write);

Statement read = SimpleStatement.builder("SELECT * FROM orders WHERE id = ?")
    .setConsistencyLevel(ConsistencyLevel.QUORUM)   // R = N/2+1
    .build();
// W + R = N + 2 > N，读必能看到最新写

// ===== 3. 最终一致—— DynamoDB 默认 =====
// 写一份立即返回，其他副本异步同步
AmazonDynamoDB client = AmazonDynamoDBClientBuilder.standard().build();
client.putItem(new PutItemRequest().withTableName("likes")
    .addItemEntry("user_id", new AttributeValue("u123"))
    .addItemEntry("count", new AttributeValue().withN("1")));
// 读可能暂时看不到（副本未同步），几秒后同步完成

// ===== 4. 分布式事务跨服务保 C（Seata）=====
@GlobalTransactional
public void placeOrder(OrderDTO order) {
    inventoryService.deduct(order.getProductId());   // 跨服务扣库存
    orderDao.insert(order);                          // 本地订单
    pointService.add(order.getUserId());             // 跨服务加积分
    // 任一失败，全局回滚，跨服务 C 成立（最终一致）
}
```

## 常见坑/注意事项

- **ACID 的 C 不是免费的**：约束检查（CHECK、外键、唯一索引）每次写都要校验，性能损耗可能 10-30%。高并发场景常牺牲 C 换性能（如 MySQL 关闭外键检查、应用层保证）。
- **强一致的延迟陷阱**：跨机房强一致（如 Spanner）要等所有机房 ACK，RTT 可能 50-100ms。电商大促时强一致扣库存会让接口 RT 飙升。折中：核心交易强一致，非核心（如浏览数）最终一致。
- **Quorum 不是绝对强一致**：`W+R>N` 只保证"大概率读到最新"，极端情况下（如副本修复期间）仍可能读到旧值。严格的线性一致需要 Paxos/Raft 协议。
- **最终一致的"最终"不可预测**：副本同步可能几秒（正常）也可能几分钟（网络抖动）。对用户可见的数据（如订单状态），"最终一致"期间要显示"处理中"避免用户困惑。
- **跨服务事务不能用 DB 的 C**：订单服务 + 库存服务用不同 DB，单机 ACID 无效。必须用 Seata、TCC、Saga 等分布式事务方案，且通常是最终一致而非强一致。
