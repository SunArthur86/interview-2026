# 输入URL回车后发生了什么

### 输入 URL 回车后发生了什么

#### 简易流程
1.  **URL 解析**：检查输入合法性，处理编码。
2.  **查找缓存**：检查浏览器缓存（强缓存/协商缓存）。
3.  **DNS 解析**：将域名解析为 IP 地址（浏览器缓存 -> 系统缓存 -> Hosts -> DNS 服务器）。
4.  **TCP 连接**：三次握手建立连接（HTTP/1.1）。如果是 HTTPS，还需 TLS/SSL 握手。
5.  **发送 HTTP 请求**：构建请求报文发送。
6.  **服务器处理**：接收请求，处理逻辑，返回响应。
7.  **浏览器解析渲染**：解析 HTML，构建 DOM/CSSOM，渲染页面。
8.  **TCP 断开**：四次挥手关闭连接。

#### 详细导航阶段

**1. URL 请求过程**
*   **输入处理**：浏览器判断输入是搜索关键字还是 URL。如果是关键字，使用默认搜索引擎合成 URL；如果是地址，补全协议（如 https://）。
*   **进程间通信**：浏览器进程通过 IPC 将 URL 请求发送给网络进程。

**2. 网络进程处理**
*   **缓存检查**：
    *   **强缓存**：检查 `Cache-Control`/`Expires`。命中则直接使用资源，不发送请求。
    *   **协商缓存**：强缓存未命中，发送请求。检查 `Last-Modified`/`ETag`。命中则返回 304，使用本地缓存。
*   **DNS 解析**：
    *   顺序：浏览器缓存 -> 本机缓存 -> Hosts 文件 -> 路由器缓存 -> ISP DNS 缓存 -> 根域 -> 顶级域 -> 权威域。
*   **TCP 连接**：
    *   浏览器为每个域名维护最多 6 个 TCP 连接（HTTP/1.1）。若排队，可采用域名分片或升级 HTTP/2。
    *   **三次握手**：
        1.  客户端发送 SYN=1, seq=x，进入 SYN_SENT。
        2.  服务端回复 SYN=1, ACK=1, seq=y, ack=x+1，进入 SYN_RCVD。
        3.  客户端回复 ACK=1, seq=x+1, ack=y+1，双方进入 ESTABLISHED。
    *   *握手原因*：防止历史连接的报文突然传到服务端导致服务端错误开启连接，浪费资源。
*   **发送请求**：构建 HTTP 请求行、请求头、请求体发送。

**3. 响应处理**
*   **状态码处理**：
    *   **301/302**：重定向，从 Location 字段获取新地址，重新发起请求。
    *   **200**：检查 `Content-Type`。若为 `text/html` 则准备渲染，若为 `application/octet-stream` 则触发下载。
*   **提交文档**：网络进程将数据提交给渲染进程。

#### 渲染阶段

1.  **构建 DOM 树**：浏览器将 HTML 解析为节点树。
2.  **构建 CSSOM 树**：解析 CSS 生成样式规则树。
3.  **生成渲染树**：合并 DOM 和 CSSOM，忽略 `display:none` 等不可见元素（但保留 `visibility:hidden`）。
4.  **布局**：计算每个节点的位置和几何信息。
5.  **绘制**：将渲染树绘制到图层上。
6.  **合成**：将图层按正确顺序组合显示到屏幕上。

#### TCP 断开（四次挥手）
1.  主动方发送 FIN=1，seq=u，停止发送数据。
2.  被动方回复 ACK=1， seq=v, ack=u+1（此时被动方可能还有数据要发）。
3.  被动方发送 FIN=1, ACK=1, seq=w, ack=u+1。
4.  主动方回复 ACK=1, seq=u+1, ack=w+1，连接关闭。
*   *挥手原因*：全双工通信中，双方都可以发送数据。一方停止发送后，仍需接收对方数据，因此 FIN 和 ACK 通常分两次发送。
