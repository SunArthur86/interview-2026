# 输入URL回车后发生了什么

### 输入 URL 回车后发生了什么

#### 简易流程
1.  **URL 解析**：检查输入合法性，处理编码。
2.  **查找缓存**：检查浏览器缓存（强缓存/协商缓存）。
3.  **DNS 解析**：将域名解析为 IP 地址（浏览器缓存 -> 系统缓存 -> Hosts -> DNS 服务器）。
4.  **TCP 连接**：三次握手建立连接（HTTP/1.1）。如果是 HTTPS，还需 TLS/SSL 握手。
5.  **发送 HTTP 请求**：构建请求报文发送。
6.  **服务器处理**：接收请求，处理逻辑，返回响应。
7.  **浏览器解析渲染**：解析 HTML，构建 DOM/CSSOM，渲染页面。
8.  **TCP 断开**：四次挥手关闭连接。

#### 详细导航阶段

**1. URL 请求过程**
*   **输入处理**：浏览器判断输入是搜索关键字还是 URL。如果是关键字，使用默认搜索引擎合成 URL；如果是地址，补全协议（如 https://）。
*   **进程间通信**：浏览器进程通过 IPC 将 URL 请求发送给网络进程。

**2. 网络进程处理**
*   **缓存检查**：
    *   **强缓存**：检查 `Cache-Control`/`Expires`。命中则直接使用资源，不发送请求。
    *   **协商缓存**：强缓存未命中，发送请求。检查 `Last-Modified`/`ETag`。命中则返回 304，使用本地缓存。
*   **DNS 解析**：
    *   顺序：浏览器缓存 -> 本机缓存 -> Hosts 文件 -> 路由器缓存 -> ISP DNS 缓存 -> 根域 -> 顶级域 -> 权威域。
*   **TCP 连接**：
    *   浏览器为每个域名维护最多 6 个 TCP 连接（HTTP/1.1）。若排队，可采用域名分片或升级 HTTP/2。
    *   **三次握手**：
        1.  客户端发送 SYN=1, seq=x，进入 SYN_SENT。
        2.  服务端回复 SYN=1, ACK=1, seq=y, ack=x+1，进入 SYN_RCVD。
        3.  客户端回复 ACK=1, seq=x+1, ack=y+1，双方进入 ESTABLISHED。
    *   *握手原因*：防止历史连接的报文突然传到服务端导致服务端错误开启连接，浪费资源。
*   **发送请求**：构建 HTTP 请求行、请求头、请求体发送。

**3. 响应处理**
*   **状态码处理**：
    *   **301/302**：重定向，从 Location 字段获取新地址，重新发起请求。
    *   **200**：检查 `Content-Type`。若为 `text/html` 则准备渲染，若为 `application/octet-stream` 则触发下载。
*   **提交文档**：网络进程将数据提交给渲染进程。

#### 渲染阶段

1.  **构建 DOM 树**：浏览器将 HTML 解析为节点树。
2.  **构建 CSSOM 树**：解析 CSS 生成样式规则树。
3.  **生成渲染树**：合并 DOM 和 CSSOM，忽略 `display:none` 等不可见元素（但保留 `visibility:hidden`）。
4.  **布局**：计算每个节点的位置和几何信息。
5.  **绘制**：将渲染树绘制到图层上。
6.  **合成**：将图层按正确顺序组合显示到屏幕上。

#### TCP 断开（四次挥手）
1.  主动方发送 FIN=1，seq=u，停止发送数据。
2.  被动方回复 ACK=1， seq=v, ack=u+1（此时被动方可能还有数据要发）。
3.  被动方发送 FIN=1, ACK=1, seq=w, ack=u+1。
4.  主动方回复 ACK=1, seq=u+1, ack=w+1，连接关闭。
*   *挥手原因*：全双工通信中，双方都可以发送数据。一方停止发送后，仍需接收对方数据，因此 FIN 和 ACK 通常分两次发送。

### 1. 认证机制背景：SSO 单点登录

**普通的登录认证机制**：
- 登录成功后，服务器在 Session 中存储用户信息，并生成对应的 Cookie 返回给浏览器。
- 再次访问时，浏览器自动携带 Cookie，服务器根据 Cookie 查找 Session 验证身份。
- **缺点**：在多系统环境下，每个系统都有独立的 Session 和 Cookie，用户需要多次登录。

**单点登录**：
- 允许用户在一次登录后访问多个关联的应用程序或服务，而无需再次输入凭据。
- **SSO 的优点**：降低密码重用风险、用户体验流畅（一次登录全网通行）、简化应用开发（解耦认证逻辑）。

## 延伸：在浏览器中输入URL并按下回车之后会发生什么是什么？

> 合并自 `core-280`（相似度 65%）

该过程是一个经典的网络通信流程，涉及应用层、传输层、网络层、链路层等多个层面的协作。

### 1. URL 解析
浏览器首先检查输入的内容是否是合法的 URL，或者是否是搜索关键字。如果是 URL，则进行解析：
*   **协议**：如 `http`、`https`。决定通信的端口（默认 80/443）和加密方式。
*   **域名**：如 `www.google.com`，用于寻找服务器 IP。
*   **端口**：如果 URL 显式指定了端口（如 `:8080`），则使用指定端口，否则使用默认端口。
*   **路径**：请求的服务器文件路径，若省略则默认访问根目录（如 `/`），服务器通常配置为返回 `index.html` 或 `default.html`。

### 2. DNS 域名解析
浏览器需要将域名转换为 IP 地址才能进行通信。

#### 查询流程
```text
[浏览器缓存] -> [操作系统缓存(hosts)] -> [路由器缓存] -> [ISP DNS服务器] -> [根域名服务器] 
                                                                   |
                                                                   v
                                                            [顶级域名服务器] 
                                                                   |
                                                                   v
                                                            [权威域名服务器] -> 返回 IP
```
*   **递归查询**：主机向本地域名服务器发出的查询。
*   **迭代查询**：本地域名服务器向根域等发出的查询。

### 3. 建立 TCP 连接 (三次握手)
获得 IP 后，浏览器通过操作系统内核发起 TCP 连接（SYN -> SYN/ACK -> ACK）。

### 4. 发送 HTTP 请求
连接建立后，浏览器构建 HTTP 请求报文发送给服务器。
*   **请求行**：`GET /index.html HTTP/1.1`
*   **请求头**：包含浏览器信息、Cookie、缓存策略等。

### 5. 服务器处理请求
服务器接收请求，解析参数，处理业务逻辑（可能涉及查询数据库等），生成响应数据。

### 6. 返回 HTTP 响应
服务器返回 HTTP 响应报文。
*   **状态码**：如 200 (成功), 301 (重定向), 404 (未找到), 500 (服务器错误)。
*   **响应头**：包含内容类型、内容长度、缓存控制等。
*   **响应体**：实际的 HTML、JSON 或图片数据。

### 7. 浏览器渲染页面
浏览器解析 HTML，构建 DOM 树和 CSSOM 树，合并生成渲染树，最终绘制页面。

### 8. 断开 TCP 连接 (四次挥手)
数据传输完毕后，通过四次挥手断开连接（FIN -> ACK -> FIN -> ACK）。

## 常见考点
1.  **输入 URL 后，如果还没等到 DNS 结果，浏览器在做什么？**
    现代浏览器（如 Chrome）会进行预解析，在 UI 线程解析 URL 的同时，网络线程可能已经开始尝试建立连接或查找缓存。
2.  **为什么要进行三次握手，两次不行吗？**
    为了防止失效的连接请求报文段被服务端接收，从而产生错误。主要目的是确认双方的接收与发送能力是正常的。
3.  **HTTP 1.1 和 2.0 的主要区别？**
    HTTP 1.1 默认持久连接；HTTP 2.0 支持多路复用（一个连接并发处理多个请求）、头部压缩、服务器推送等。
4.  **什么是 HTTPS，它与 HTTP 的区别？**
    HTTPS 在 HTTP 下加入了 SSL/TLS 协议，通过加密传输数据、验证服务器身份、保证数据完整性。端口默认为 443。

- 渲染流程：构建DOM树和CSSOM树，合并生成渲染树最终绘制
