# 什么是Virtual Threads（虚拟线程）？它与传统线程有什么区别？

🎯 **本质**：虚拟线程是JDK 21引入的轻量级线程，由JVM管理而非操作系统，打破了Java线程与OS线程的1:1映射关系。

🧒 **类比**：传统线程就像出租车（昂贵、数量有限），虚拟线程就像共享单车（便宜、可以停满整条街）。

📊 **核心区别**：
| 特性 | 平台线程 | 虚拟线程 |
|------|---------|---------|
| 映射 | 1:1 OS线程 | N:M 调度到载体线程 |
| 内存 | ~1-2MB/线程 | ~几KB/线程 |
| 创建成本 | 高(系统调用) | 极低(纯JVM) |
| 数量上限 | 几百~几千 | 可达数百万 |
| 适用场景 | CPU密集型 | IO密集型 |

🔧 **工作原理（架构图）**：
```text
  应用代码              调度器              操作系统
┌─────────────┐      ┌───────────────────────┐     ┌───────────────┐
│ 虚拟线程 VT1 │───→  │ 载体线程 PT1 (Carrier) │────→ │ OS Thread 1   │
│ (I/O 操作)   │      │ (执行 VT1 的任务)      │     │               │
└─────────────┘      └───────────────────────┘     └───────────────┘
       │
       │ (遇到阻塞 I/O)
       │ JVM 将 VT1 从 PT1 上卸载
       ↓
┌─────────────┐      ┌───────────────────────┐
│ 虚拟线程 VT1 │ (Heap)│ 载体线程 PT1 闲置      │────→ │ OS Thread 1   │
│ (挂起状态)   │      │ 可立即调度 VT2        │     │ (运行 VT2)    │
└─────────────┘      └───────────────────────┘     └───────────────┘
```

💻 **使用方式**：
```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 100_000).forEach(i → executor.submit(() → {
        Thread.sleep(Duration.ofSeconds(1)); // 模拟IO阻塞
        return i;
    }));
}
```

**实战案例**：在重构一个高并发网关服务时，原先使用传统线程池处理请求，由于等待下游服务响应（阻塞IO），线程池很快耗尽导致大量拒绝请求。迁移到虚拟线程后，我们创建了10万个虚拟线程处理并发请求，物理CPU核心数未变，但吞吐量提升了3倍，且不再需要维护复杂的ThreadPool参数调优。

⚠️ **注意事项**：
1. 不适合CPU密集型任务
2. 不要池化虚拟线程——用完即弃
3. synchronized块内IO会pin载体线程，建议用ReentrantLock
4. ThreadLocal在百万虚拟线程中会导致内存问题，考虑Scoped Values

## 常见考点
1. **虚拟线程是绿色线程吗？有什么区别？**
   - 概念相似，都是用户态线程。区别在于：早期的绿色线程（JDK 1.0）是 M:1 映射（所有绿色线程映射到一个 OS 线程），无法利用多核；Java 虚拟线程是 M:N 映射，可以利用所有可用的 CPU 核心。
2. **载体线程是固定的吗？**
   - 不固定。虚拟线程在生命周期中会被 JVM 调度到不同的载体线程上运行。因此，在虚拟线程中调用 `Thread.currentThread()` 返回的永远是该虚拟线程本身，而非底层的载体线程。


## 记忆要点

- 对比：传统线程内存~MB级且数量受限，虚拟线程~KB级可达百万并发

## 延伸：什么是Java 21的虚拟线程（Virtual Threads）？它与平台线程（Platform Threads）有什么本质区别？

> 合并自 `jnf-001`（相似度 72%）

虚拟线程是Java 21（JEP 444）正式发布的革命性特性，由Project Loom团队开发。它是一种由JVM管理的轻量级线程，不与操作系统线程一一绑定。传统的平台线程是操作系统线程的薄包装，每个线程需要约1MB的栈空间，创建和上下文切换都涉及昂贵的系统调用，实际并发量通常限制在数千个以内。

虚拟线程由JVM在少量载体线程上调度多路复用。当虚拟线程在I/O操作上阻塞时（如数据库查询、HTTP请求、文件读写），JVM会将其从载体线程上卸载，释放载体线程去运行其他虚拟线程。当I/O完成后，JVM会在可用的载体线程上重新挂载虚拟线程，从暂停的位置恢复执行。这种机制允许数百万个虚拟线程同时存在，以极低的内存开销实现高并发。

```java
// 方式1：直接创建虚拟线程
Thread vThread = Thread.ofVirtual().start(() → {
    System.out.println("Running on virtual thread: " + Thread.currentThread());
});

// 方式2：使用Executors
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    // 轻松启动10,000个虚拟线程处理请求
    for (int i = 0; i < 10_000; i++) {
        executor.submit(() → {
            // 模拟I/O密集型操作
            return fetchDataFromRemote();
        });
    }
}

// 方式3：Spring Boot 3.2+ 开启虚拟线程
// application.properties: spring.threads.virtual.enabled=true
```

**虚拟线程 vs 平台线程 对比**：

| 维度 | 平台线程 | 虚拟线程
| :--- | :--- | :---
| **实现主体** | 操作系统内核 | JVM (用户模式)
| **创建成本** | 高 (系统调用) | 极低 (对象分配)
| **内存占用** | 高 (~1MB 栈空间) | 低 (几KB 栈帧, 按需扩容)
| **上下文切换** | 昂贵 (内核态切换) | 便宜 (JVM 仅保存栈帧)
| **阻塞行为** | 阻塞 OS 线程 | JVM 卸载，释放 Carrier
| **适用场景** | CPU 密集型/计算任务 | I/O 密集型/高并发请求

**实战案例与避坑**：
1.  **Synchronized 锁定**：在生产环境迁移时，曾遇到旧版 SDK 使用 `synchronized` 导致虚拟线程无法卸载，吞吐量不升反降。**解决方案**：将 `synchronized` 替换为 `ReentrantLock`，确保在阻塞时能释放载体线程。
2.  **ThreadLocal 污染**：由于虚拟线程数量巨大，传统的 ThreadLocal（如 SimpleDateFormat）会堆积数百万个对象导致 OOM。**解决方案**：不再依赖 ThreadLocal 缓存非线程安全对象，直接在方法内创建（由于对象创建成本极低），或使用 Java 21 的 Scoped Values。

## 记忆要点

- 因为由JVM管理且按需挂载载体线程，所以虚拟线程能以极低内存支撑百万级高并发。
- 平台线程绑OS且占1MB，而虚拟线程属用户态仅几KB，且仅在IO阻塞时才卸载释放载体。
- 适用场景对比：平台线程用于CPU密集计算，而虚拟线程专治I/O密集型高并发请求。
- 两大避坑核心：因为synchronized会固定线程，所以必须换ReentrantLock；因为百万线程会导致OOM，所以禁用ThreadLocal。
