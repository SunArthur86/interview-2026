# 什么是Java 21的结构化并发（Structured Concurrency）？它解决了什么问题？

结构化并发是Java 21引入的预览特性（JEP 453），它将多个作为单个操作组成部分执行的并发任务视为一个工作单元，其生命周期限定在生成它们的范围内。它解决了并发编程中最危险且最难调试的一类问题：任务超出其逻辑父作用域存活、某些任务失败时其他任务默默成功、以及部分任务完成后其他任务仍在运行导致的资源泄漏。

在结构化并发之前，展开多个并发任务并收集结果需要仔细管理ExecutorService、Future对象、CompletableFuture链和显式错误处理——很容易出错。如果一个任务失败，必须显式取消其他任务。如果线程被中断，其他线程中运行的任务可能会作为孤儿后台线程继续运行，消耗资源。

```java
// 传统方式 - 容易出错，错误处理复杂
public Response handleRequest() throws ExecutionException, InterruptedException {
    ExecutorService executor = Executors.newFixedThreadPool(3);
    try {
        Future<User> userFuture = executor.submit(() -> findUser());
        Future<Order> orderFuture = executor.submit(() -> fetchOrder());
        Future<Inventory> inventoryFuture = executor.submit(() -> getInventory());
        
        // 如果fetchOrder()抛异常，其他两个Future不会被自动取消！
        User user = userFuture.get();
        Order order = orderFuture.get();      // 如果这里抛异常
        Inventory inv = inventoryFuture.get(); // 这行不会执行，但线程仍在跑
        return new Response(user, order, inv);
    } finally {
        executor.shutdown();
    }
}

// Java 21 结构化并发 - 优雅且安全
public Response handleRequest() throws Exception {
    try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
        // 并行启动子任务
        var userTask = scope.fork(() -> findUser());
        var orderTask = scope.fork(() -> fetchOrder());
        var inventoryTask = scope.fork(() -> getInventory());
        
        // 等待所有任务完成（或任一失败）
        scope.join();          // 等待所有fork的任务完成
        scope.throwIfFailed(); // 如果任何任务失败，抛出异常
        
        // 所有任务都成功，获取结果
        return new Response(
            userTask.get(),
            orderTask.get(),
            inventoryTask.get()
        );
    } // try-with-resources自动取消任何未完成的任务
}
```

StructuredTaskScope的工作原理：scope的join()方法等待所有子任务完成；scope的close()方法（由try-with-resources自动调用）取消任何尚未完成的任务，保证没有任务超出scope存活。ShutdownOnFailure策略在任一任务失败时立即取消所有剩余任务，然后在join()后重新抛出异常。ShutdownOnSuccess策略在任一任务成功时立即取消所有剩余任务，返回第一个成功的结果。

### 核心原理：结构化树

结构化并发将并发任务建模为一棵树。父任务可以派生子任务，子任务必须在父任务结束前完成。如果父任务被取消或失败，所有子任务也会被自动取消。这种模型确保了并发代码的可预测性和资源管理的安全性，类似于结构化编程中的代码块作用域。

#### 实战案例
在高并发网关场景中，需要并行调用用户服务、积分服务和风控服务。传统CompletableFuture组合在超时或异常处理上极易出现“任务取消但连接未释放”的问题。使用结构化并发后，配合`deadline`设置，超时能自动中断所有子任务的底层网络连接，有效防止了连接池耗尽。

#### 自定义策略实现（针对竞速场景）

```java
// 实战代码：针对多级缓存查询，任一成功即返回
public Data queryCache() {
    try (var scope = new StructuredTaskScope.ShutdownOnSuccess<Data>()) {
        scope.fork(() -> redisClient.get(key));
        scope.fork(() -> memcachedClient.get(key));
        scope.fork(() -> localCache.get(key));
        
        scope.join();
        return scope.result(); // 返回第一个成功的值，其他自动取消
    } catch (Exception e) {
        throw new RuntimeException("All caches failed", e);
    }
}
```

#### 对比表格：结构化并发 vs CompletableFuture

| 特性 | 结构化并发 | CompletableFuture |
| :--- | :--- | :--- |
| **错误传播** | 自动聚合子任务异常，直接抛出到主线程 | 需手动调用 exceptionally 或 handle，容易遗漏 |
| **取消机制** | 父任务取消/失败自动级联取消子任务 | 需显式调用 cancel()，且逻辑代码冗长 |
| **线程可见性** | 树状结构清晰，调试栈追踪可读性高 | 异步回调链割裂，线程Dump难以追踪血缘 |
| **资源管理** | try-with-resources 自动保证清理 | 依赖 finally 块或 Cleaner，易泄漏 |
| **适用场景** | 并行子任务聚合（如微服务聚合） | 复杂流式处理、管道转换 |

## 记忆要点

- 一句话定义：将多个并发任务视为单一逻辑工作单元，子任务生命周期绝不超出父作用域。
- 痛点对比：传统Future任一失败时其他任务默默成孤儿线程，而结构化并发自动级联取消。
- 核心API：用ShutdownOnFailure策略，配合join()等待及throwIfFailed()统一抛错。
- 竞速场景实战：使用ShutdownOnSuccess策略，任一成功即取消其他，适合多级缓存查询。
