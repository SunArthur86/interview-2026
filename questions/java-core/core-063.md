# 什么是HTTP特性与简述？

HTTP特性与简述

**特性**：简单、灵活、易于扩展、应用广泛和跨平台、无状态、明文传输。

**简述**：
Web 上的通信都是建立在 HTTP 协议上的
1. 客户端发起 HTTP 请求；
2. 服务器做出响应处理后，返回 HTTP 响应报文
最初设想的基本理念是：借助多文档之间相互关联形成的超文本，连成可相互参阅的 WWW（World Wide Web，万维网）。

**WWW 构建技术（3 项）**：
1. HTML (HyperText Markup Language)：作为页面的文本标记语言
2. HTTP (HyperText Transfer Protocol)：文档传递协议；
3. URL (Uniform Resource Locator)：指定文档所在地址

**HTTP 版本演进**：
1. **HTTP/0.9** (1991)：仅支持 GET 请求，仅用于传输 HTML，无 Header。
2. **HTTP/1.0** (1996, RFC1945)：增加了 HEAD、POST 等方法，引入了状态码、Header 和 MIME 类型，每次连接只能发送一个请求（短连接）。
3. **HTTP/1.1** (1997)：**目前主流版本**。引入了持久连接、管道机制、分块传输编码、更多的缓存控制头。
4. **HTTP/2.0** (2015)：引入了**多路复用**、二进制分帧层、头部压缩、服务端推送，解决了 HTTP/1.1 的队头阻塞问题。
5. **HTTP/3.0** (2022)：基于 UDP 和 QUIC 协议，解决了 TCP 层面的队头阻塞，建立了更快的连接。

**核心特性详解**：

1. **简单**
基本报文格式为 header+body，头部信息也是 key-value 简单文本的形式，易于理解。HTTP 协议的语法简单，HTTP/1.1 之前的报文甚至是纯文本的，人类可读。

2. **灵活和易于扩展**
- HTTP 协议里的各种请求方法、URI/URL、状态码、头字段等每个组成要求都没有被固定死，允许开发人员自定义和扩充（例如自定义 `X-Token` 头）；
- HTTP 工作在应用层（OSI 第七层），下层可以随意变化（从 IPv4 到 IPv6）；
- 扩展协议：HTTPS 就是在 HTTP 与 TCP 之间增加了 SSL/TLS 安全传输层；HTTP/2 和 HTTP/3 在传输机制上做了重大更新，但应用层语义基本不变。

3. **无状态**
- **定义**：服务器不会去记忆 HTTP 的状态，所以不需要额外的资源来记录状态信息，这能减轻服务器的负担。但它在完成有关联性的操作时会非常麻烦（如电商购物车）。
- **解决方案**：引入 Cookie 和 Session 技术，或使用 JWT (JSON Web Token) 等无状态认证方案。

4. **明文传输**
- 传输过程中的信息，是可方便阅读的，通过浏览器的 F12 控制台或 Wireshark 抓包都可以直接肉眼查看。
- **优点**：调试方便，开发者易于排查问题。
- **缺点**：信息透明，容易被窃听（如中间人攻击窃取密码）。

5. **不安全**
1. 通信使用明文（不加密），内容可能被窃听。
2. 不验证通信方的身份，因此有可能遭遇伪装（DNS 劫持等）。
3. 无法证明报文的完整性，所以有可能已遭篡改。
- **解决**：使用 HTTPS。通过引入 SSL/TLS 层，提供数据加密、身份认证（CA 证书）和数据完整性校验（MAC/HASH）。

**HTTPS 协议结构图**：
```
┌───────────────────────────────────────────────────────┐
│                   应用层 (HTTP)                        │
├───────────────────────────────────────────────────────┤
│              安全层 (SSL/TLS) / 握手加密               │
├───────────────────────────────────────────────────────┤
│               传输层 (TCP) / (QUIC)                    │
├───────────────────────────────────────────────────────┤
│               网络层 (IP)                              │
└───────────────────────────────────────────────────────┘
```

**实战案例**：曾开发过一个公共 API 服务，初期使用 HTTP 明文传输，被运营商 HTTP 劫持注入广告脚本导致客户端解析 JSON 失败。迁移至 HTTPS 后彻底解决了劫持和中间人窃听问题。

**代码示例**：
```python
# Python (Flask): 自定义 Header 体现灵活性，使用 HTTPS 强制跳转
from flask import Flask, request, redirect

app = Flask(__name__)

@app.before_request
def enforce_https():
    # 解决不安全问题：强制使用 HTTPS
    if not request.is_secure:
        return redirect(request.url.replace('http://', 'https://'), code=301)

@app.route('/api/data')
def get_data():
    # 体现易扩展性：自定义业务头
    return {'data': 'value'}, 200, {'X-Custom-Request-Id': 'req-12345'}
```

**HTTP 特性对比**：

| 特性 | 描述 | 优点 | 缺点 | 解决方案/改进 |
| :--- | :--- | :--- | :--- | :--- |
| **无状态** | 服务器不保存上下文 | 服务器扩展性强，内存占用低 | 无法处理关联业务（购物车等） | Cookie, Session, JWT Token |
| **明文传输** | 报文为纯文本 | 调试方便，利于学习 | 易被窃听、篡改 | 升级为 HTTPS (TLS 加密) |
| **请求/响应模型** | 一问一答 | 逻辑简单 | 无法由服务器主动推送 | WebSocket, HTTP/2 Server Push, SSE |
