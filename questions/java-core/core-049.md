# TCP与UDP的区别？

### TCP 与 UDP 的区别

#### 1. 连接性
- **TCP (面向连接)**：传输数据前必须通过**三次握手**建立连接，传输结束后通过**四次挥手**断开连接。这确保了通信双方的准备就绪。
- **UDP (无连接)**：不需要建立连接，想发就发。减少了建立连接时的延迟（DNS 查询使用 UDP 的原因之一）。

#### 2. 服务形式
- **TCP**：仅支持**一对一**（点对点）的全双工通信。
- **UDP**：支持**一对一**、**一对多**（广播）、**多对多**（多播）。适用于广播消息或群聊场景。

#### 3. 可靠性
- **TCP**：**可靠传输**。通过序列号、确认应答（ACK）、超时重传、校验和等机制，保证数据**无差错、不丢失、不重复、且按序到达**。
- **UDP**：**不可靠传输**。尽最大努力交付。不确认、不重传、不排序，丢包后数据直接丢失，由应用层负责处理（如视频卡顿）。

#### 4. 流量控制与拥塞控制
- **TCP**：拥有完善的控制机制。
    - **流量控制**：使用**滑动窗口**协议，防止发送方发太快导致接收方缓冲区溢出。
    - **拥塞控制**：包含慢启动、拥塞避免、快重传、快恢复，防止过多的数据注入到网络中造成网络拥塞。
- **UDP**：无流量控制和拥塞控制。网络拥堵时也不会降低发送速率（这对于实时应用很重要，但也可能加剧网络拥塞）。

#### 5. 首部开销
- **TCP**：首部最小 **20 字节**（包含源端口、目的端口、序列号、确认号、窗口大小等），若有选项最大可达 60 字节。
- **UDP**：首部固定 **8 字节**（源端口、目的端口、长度、校验和），开销极小。

#### 6. 传输方式
- **TCP**：**面向字节流**。TCP 将数据看成一连串无结构的字节流，不保留应用层消息的边界（可能产生粘包/拆包问题）。
- **UDP**：**面向报文**。UDP 对应用层交下来的报文，既不合并也不拆分，保留报文的边界（发多少收多少，若报文过大则可能在 IP 层分片）。

#### 7. 分片方式
- **TCP**：数据大小超过 **MSS (最大报文段长度)** 时，在 **TCP 层**进行分片。接收方在 TCP 层重组，丢失时只需重传对应的 TCP 分片。
- **UDP**：数据大小超过 **MTU (最大传输单元)** 时，在 **IP 层**进行分片。接收方在 IP 层重组。一旦发生某个 IP 分片丢失，整个 UDP 报文都被丢弃（IP 层无重传机制），效率极差。

**实战案例**：在一次 Kubernetes 容器网络排查中，发现服务间调用偶发超时。经抓包分析，是因为 UDP 包大小超过了 MTU（1500），在物理网卡进行 IP 分片后，中间某设备的防火墙丢弃了部分分片，导致应用层收不到完整数据。最终通过调小应用层 UDP 包大小解决。

## 常见考点
1.  **基于 TCP 的 UDP 实现**：
    - 问：如何在 UDP 上实现可靠传输（如 QUIC 协议）？
    - 答：需要应用层实现 TCP 的机制，包括序列号、ACK 确认、超时重传和拥塞控制。QUIC 协议就是一个典型的例子，它基于 UDP 实现了可靠传输，解决了 TCP 的队头阻塞问题。
