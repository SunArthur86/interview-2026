# 三次握手的过程，以及为什么是三次，而不是四次，两次？

### 三次握手的过程

1.  **第一次握手 (SYN)**：客户端向服务器发送 SYN 报文（SYN=1），并随机初始化序列号 `seq=x`。客户端进入 `SYN_SENT` 状态。
2.  **第二次握手 (SYN+ACK)**：服务器收到 SYN 报文后，发送 SYN+ACK 报文给客户端。
    *   ACK=1，确认号 `ack=x+1`（表示收到客户端的 x）。
    *   SYN=1，初始化自己的序列号 `seq=y`。
    *   服务器进入 `SYN_RCVD` 状态。
3.  **第三次握手 (ACK)**：客户端收到 SYN+ACK 报文后，向服务器发送 ACK 报文（ACK=1），确认号 `ack=y+1`，序列号 `seq=x+1`。客户端进入 `ESTABLISHED` 状态。服务器收到 ACK 后也进入 `ESTABLISHED` 状态。

### 为什么是三次，而不是两次或四次？

1.  **防止已失效的连接请求突然又传到了服务端**：
    *   如果是两次握手，假设客户端发送的第一个连接请求在网络中滞留，延误到连接释放后才到达服务器。服务器误以为是新的连接请求，于是向客户端发送确认，建立连接，等待客户端发数据。客户端此时并未发起连接，会忽略服务器的确认，导致服务器一直空等，浪费资源。
    *   三次握手时，如果服务端回复第二次握手，客户端没发起连接就不会回复第三次握手（ACK），服务端便知道连接未建立，关闭连接。

2.  **同步双方的初始序列号 (ISN)**：
    *   TCP 是可靠传输，需要对每个字节进行编号。
    *   第一次握手：客户端告诉服务器自己的 ISN。
    *   第二次握手：服务器告诉客户端自己的 ISN，并确认收到客户端的 ISN。
    *   第三次握手：客户端确认收到服务器的 ISN。
    *   如果是两次握手，服务端无法确认客户端是否收到了自己的序列号，连接不可靠。

3.  **为什么不是四次**：
    *   三次握手已经足以上述两个条件（同步双方序列号 + 确认双方接收发送能力）。四次握手虽然也可以，但会浪费一次报文传输，降低效率。

### 状态流转图

```text
Client                                    Server
  │                                         │
 │ ─────── SYN (seq=x) ────────────────────> │  (SYN_SENT)
  │                                         │  (LISTEN -> SYN_RCVD)
 │ <────── SYN+ACK (seq=y, ack=x+1) ────── │
  │                                         │
 │ ─────── ACK (seq=x+1, ack=y+1) ────────> │  (ESTABLISHED)
  │ (ESTABLISHED)                           │
```

### 实战案例

*   **连接超时排查**：在微服务架构中，如果服务端负载过高导致全连接队列 溢出，客户端会发出 SYN 后收不到 ACK，或者收到 ACK 但连接被丢弃。日志中常伴随 `TCP: time wait bucket table overflow` 或 `TCP: request_sock_TCP: Possible SYN flooding`。此时需调整 `net.core.somaxconn` 和 `net.ipv4.tcp_max_syn_backlog` 参数，并开启 `tcp_syncookies` 防御 SYN Flood 攻击。

### 代码示例 (Linux内核参数调优)

```bash
# /etc/sysctl.conf
# 开启 SYN Cookies，防御 SYN 攻击
net.ipv4.tcp_syncookies = 1
# 增加全连接队列长度，默认 128，高并发建议调大至 1024 或 4096
net.core.somaxconn = 4096
# 增加半连接队列长度
net.ipv4.tcp_max_syn_backlog = 8192
# 减少重试次数，快速失败，避免客户端长时间阻塞
net.ipv4.tcp_syn_retries = 2
```

## 常见考点
1.  **SYN Flood 攻击**：攻击者大量发送 SYN 报文但不回第三次握手，导致服务端维护大量 `SYN_RCVD` 队列，耗尽资源。如何防御？（Syn Cookies，增加半连接队列上限等）。
2.  **第三次握手丢包**：客户端发了 ACK 但丢了，客户端认为已连接，服务端认为未连接。此时会发生什么？（服务端重传 SYN+ACK，客户端再次响应 ACK）。
3.  **全连接队列与半连接队列**：握手过程中涉及的两个队列长度限制及溢出行为（`net.core.somaxconn`, `tcp_max_syn_backlog`）。
