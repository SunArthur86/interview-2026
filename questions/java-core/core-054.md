# 三次握手的过程，以及为什么是三次，而不是四次，两次？

### 三次握手的过程

1.  **第一次握手 (SYN)**：客户端向服务器发送 SYN 报文（SYN=1），并随机初始化序列号 `seq=x`。客户端进入 `SYN_SENT` 状态。
2.  **第二次握手 (SYN+ACK)**：服务器收到 SYN 报文后，发送 SYN+ACK 报文给客户端。
    *   ACK=1，确认号 `ack=x+1`（表示收到客户端的 x）。
    *   SYN=1，初始化自己的序列号 `seq=y`。
    *   服务器进入 `SYN_RCVD` 状态。
3.  **第三次握手 (ACK)**：客户端收到 SYN+ACK 报文后，向服务器发送 ACK 报文（ACK=1），确认号 `ack=y+1`，序列号 `seq=x+1`。客户端进入 `ESTABLISHED` 状态。服务器收到 ACK 后也进入 `ESTABLISHED` 状态。

### 为什么是三次，而不是两次或四次？

1.  **防止已失效的连接请求突然又传到了服务端**：
    *   如果是两次握手，假设客户端发送的第一个连接请求在网络中滞留，延误到连接释放后才到达服务器。服务器误以为是新的连接请求，于是向客户端发送确认，建立连接，等待客户端发数据。客户端此时并未发起连接，会忽略服务器的确认，导致服务器一直空等，浪费资源。
    *   三次握手时，如果服务端回复第二次握手，客户端没发起连接就不会回复第三次握手（ACK），服务端便知道连接未建立，关闭连接。

2.  **同步双方的初始序列号 (ISN)**：
    *   TCP 是可靠传输，需要对每个字节进行编号。
    *   第一次握手：客户端告诉服务器自己的 ISN。
    *   第二次握手：服务器告诉客户端自己的 ISN，并确认收到客户端的 ISN。
    *   第三次握手：客户端确认收到服务器的 ISN。
    *   如果是两次握手，服务端无法确认客户端是否收到了自己的序列号，连接不可靠。

3.  **为什么不是四次**：
    *   三次握手已经足以上述两个条件（同步双方序列号 + 确认双方接收发送能力）。四次握手虽然也可以，但会浪费一次报文传输，降低效率。

### 实战案例

*   **连接超时排查**：在微服务架构中，如果服务端负载过高导致全连接队列 溢出，客户端会发出 SYN 后收不到 ACK，或者收到 ACK 但连接被丢弃。日志中常伴随 `TCP: time wait bucket table overflow` 或 `TCP: request_sock_TCP: Possible SYN flooding`。此时需调整 `net.core.somaxconn` 和 `net.ipv4.tcp_max_syn_backlog` 参数，并开启 `tcp_syncookies` 防御 SYN Flood 攻击。

### 代码示例 (Linux内核参数调优)

```bash
# /etc/sysctl.conf
# 开启 SYN Cookies，防御 SYN 攻击
net.ipv4.tcp_syncookies = 1
# 增加全连接队列长度，默认 128，高并发建议调大至 1024 或 4096
net.core.somaxconn = 4096
# 增加半连接队列长度
net.ipv4.tcp_max_syn_backlog = 8192
# 减少重试次数，快速失败，避免客户端长时间阻塞
net.ipv4.tcp_syn_retries = 2
```

## 常见考点
1.  **SYN Flood 攻击**：攻击者大量发送 SYN 报文但不回第三次握手，导致服务端维护大量 `SYN_RCVD` 队列，耗尽资源。如何防御？（Syn Cookies，增加半连接队列上限等）。
2.  **第三次握手丢包**：客户端发了 ACK 但丢了，客户端认为已连接，服务端认为未连接。此时会发生什么？（服务端重传 SYN+ACK，客户端再次响应 ACK）。
3.  **全连接队列与半连接队列**：握手过程中涉及的两个队列长度限制及溢出行为（`net.core.somaxconn`, `tcp_max_syn_backlog`）。

## 延伸：为什么需要三次握手？

> 合并自 `core-056`（相似度 78%）

### 三次握手详细过程

1、第一次握手：SYN报文：  
客户端随机初始化序列号`client_isn`，放进TCP首部序列号段，然后把SYN置1。把SYN报文发送给服务端，表示发起连接，之后客户端处于`SYN-SENT`状态。此时不携带应用数据。

2、第⼆次握手：SYN+ACK报文：  
服务端收到客户端的SYN报文之后，会把自己随机初始化的序号`server_isn`放进TCP首部序列号段，「确认应答号」填⼊`client_isn + 1`，把SYN和ACK标志位置为1。把SYN+ACK报文发送给客户端，然后进⼊`SYN-RCVD`状态。此时不携带应用数据（部分实现如 TCP Fast Open 可能携带，但标准流程通常不携带）。

3、第三次握手：ACK报文：  
客户端收到服务端报文后，还要向服务端回应最后一个应答报文。首先该应答报文 TCP 首部 ACK 标志位置为1，其次「确认应答号」字段填⼊ `server_isn + 1`，最后把报文发送给服务端，**这次报文可以携带客户端到服务器的数据**，之后客户端处于`ESTABLISHED`状态。服务器收到客户端的应答报文后，也进⼊`ESTABLISHED`状态。

此时，TCP 连接已经建立起来，通信双方可以开始进行数据传输。

### 为什么需要三次握手？

三次握手是**为了确认双方的接收与发送能力同时具备**，保证可靠传输。

**总结**：
1.  **阻止重复历史连接的初始化 (主因)**：
    *   如果是两次握手，客户端发送的滞留报文（旧连接的 SYN）突然到达服务端，服务端误以为是新连接并立即建立，导致客户端收到重复连接请求而拒绝，服务端资源被浪费。
    *   三次握手时，如果服务端响应了滞留的 SYN，客户端发现这不是当前连接的请求（序列号不对或状态不对），就不会回复 ACK，服务端无法建立连接。

2.  **同步双方的初始序列号 (ISN)**：
    *   TCP 连接是双向的，需要分别初始化双方的序列号。
    *   第一次：Client 告诉 Server 自己的 ISN。
    *   第二次：Server 告诉 Client 自己的 ISN，并确认收到 Client 的 ISN。
    *   第三次：Client 确认收到 Server 的 ISN。
    *   如果只有两次，Server 无法确认 Client 是否收到了自己的 ISN，导致序列号不同步，无法保证可靠传输。

3.  **避免资源浪费**：
    *   避免了因重复报文导致的无效连接占用服务器资源。

## 常见考点
1.  **半连接队列与全连接队列**：SYN_RCVD 状态的连接存放在半连接队列，ESTABLISHED 状态存放在全连接队列。队列满了会发生什么？（丢包或发送 RST）。
2.  **SYN 攻击原理**：恶意发送大量 SYN 包，填满半连接队列，导致正常连接被拒绝。
3.  **ISN 的生成**：初始序列号不是随机的 0，而是基于时钟加密生成的，防止被猜测从而伪造 TCP 报文进行攻击。

## 记忆要点

- 核心目的：确认双方的接收与发送能力正常，保障可靠传输。
- 主因：防止失效的历史连接请求被服务端误当新连接，导致资源浪费。
- 次因：同步双方的初始序列号(ISN)，两报文仅能单向同步。
- 全双工同步：三次是建立全双工可靠通信，并同步双向状态的理论最小次数。

## 延伸：什么是三次握手？

> 合并自 `core-085`（相似度 76%）

### 三次握手详细过程

1. **第一次握手**：
   - 客户端发送 `SYN=1` 的报文，并随机生成一个序列号 `seq=x`。
   - 客户端进入 `SYN_SENT` 状态。
   - **作用**：告诉服务端，客户端请求建立连接。

2. **第二次握手**：
   - 服务端收到 SYN 报文，需要确认客户端的序列号，即 `ack=x+1`。
   - 同时自己也发送一个 SYN 请求，即 `SYN=1`，并随机生成自己的序列号 `seq=y`。
   - 服务端进入 `SYN_RCVD` 状态。
   - **作用**：服务端告诉客户端，“我收到你的请求了（ACK），我也想和你建立连接（SYN）”。

3. **第三次握手**：
   - 客户端收到服务端的 SYN+ACK 报文。
   - 检查 `ack` 是否为 `x+1`，以及 `ACK` 标志位是否为 1。若正确，则发送确认包 `ACK=1`，`ack=y+1`，`seq=x+1`。
   - 客户端进入 `ESTABLISHED` 状态。
   - 服务端收到该包后也进入 `ESTABLISHED` 状态。
   - **作用**：客户端告诉服务端，“我收到你的确认了，连接建立成功”。

#### TCP 报文选项补充
- **MSS (Maximum Segment Size)**：在 SYN 报文中协商最大报文段长度，避免分片。
- **Window Scale**：协商窗口扩大因子，突破 TCP 头部 16 位窗口大小限制。
- **SACK (Selective Acknowledgment)**：选择性确认，允许只重传丢失的数据包，提高效率。

## 记忆要点

- 一问(SYN)：客户端发连接请求，进入SYN_SENT状态
- 二答(SYN+ACK)：服务端确认并附带自身请求，进入SYN_RCVD状态
- 三确认(ACK)：客户端确认服务端请求，双端进入ESTABLISHED状态
- 附带选项：首报文协商MSS防分片、Window Scale扩容窗口、SACK支持选择性重传
