# 什么是接口隔离？

### 接口隔离原则

**定义**
不应该强迫客户依赖于它们不需要的接口。建立单一接口，不要建立庞大臃肿的接口，尽量细化接口，接口中的方法尽量少。

**理解**
- 客户端不应该依赖它不需要的接口。
- 类间的依赖关系应该建立在最小的接口上。

**优点**
- 将臃肿的大接口拆分为多个特定的小接口，避免了实现类实现不需要的方法（即「污染」类）。
- 提高了系统的灵活性和可维护性，符合“高内聚低耦合”的设计思想。
- 接口职责单一，变更影响范围小，降低了重构风险。

**对比**
- **单一职责原则 (SRP)**：注重的是**类**的职责单一，一个类只负责一项业务。针对的是类中的方法集合。
- **接口隔离原则 (ISP)**：注重的是**接口**的设计，要求接口尽量细化，方法尽量少。针对的是接口对外暴露的契约。

**示例场景**
假设有一个接口 `Door` 包含方法 `open()`, `close()`, `lock()`, `unlock()`。
- `AlarmDoor`（报警门）需要全部四个方法。
- `NormalDoor`（普通门）不需要报警方法 `lock/unlock`。
- 根据 ISP，应该拆分为 `Door` (open, close) 和 `Lockable` (lock, unlock) 两个接口，普通门只需实现 `Door`。

### 对比分析：SRP vs ISP

| 维度 | 单一职责原则 (SRP) | 接口隔离原则 (ISP) |
| :--- | :--- | :--- |
| **侧重点** | **类** 的内部职责划分 | **接口** 的设计粒度 |
| **关注点** | 逻辑内聚，一个类只做一件事 | 依赖解耦，客户端不依赖不需要的方法 |
| **约束对象** | 类实现 | 接口定义 |
| **解决痛点** | 类变得庞大复杂，由于职责过多导致频繁修改 | 实现类被迫实现空方法或抛出异常（接口污染） |

## 常见考点
1. **接口隔离原则与单一职责原则的区别？**
   - SRP 侧重于类内部的职责划分，要求类只有一个引起它变化的原因；ISP 侧重于接口的设计，要求客户端不应该看到它不需要的方法。从视图角度看，SRP 是类的设计约束，ISP 是接口暴露的约束。
2. **在代码重构中如何应用接口隔离？**
   - 当发现某个类实现了一个接口，但却包含了空方法或抛出 `UnsupportedOperationException` 时，通常意味着接口过于臃肿，违反了 ISP，需要将接口拆分。

### 实战补充

**实战案例**：在重构一个老旧的支付网关时，发现 `PaymentService` 接口包含了 `pay()`, `refund()`, `query()`, `uploadAttachment()` 等十几个方法。对接第三方渠道（如微信）时，由于不支持上传附件，不得不在实现类中写空方法。重构后将其拆分为 `Payable`, `Refundable`, `Queryable` 等小接口，代码清爽且易于测试。

**代码示例（违反 ISP vs 符合 ISP）**：
```java
// 违反 ISP：臃肿接口
interface OldPaymentService {
    void pay();
    void refund();
    void uploadLog(); // 依赖方可能不需要
}

// 符合 ISP：职责拆分
interface PaymentAction { void pay(); }
interface RefundAction { void refund(); }

class WeChatPayService implements PaymentAction, RefundAction {
    public void pay() { /* ... */ }
    public void refund() { /* ... */ }
    // 不需要实现 uploadLog
}
```
