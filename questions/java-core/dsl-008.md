# 消息中间件

### 缓存击穿
- **定义**：热点 Key 过期瞬间，大量请求直接击穿到数据库。
- **解决方案**：
    1.  **互斥锁**：使用 `SETNX` (Redis) 或分布式锁（如 Redisson），获取锁成功者去查库回写缓存，失败者短暂休眠重试。保证只有一个请求查库。
    2.  **逻辑过期**：不设置 TTL，而是在 Value 中包含过期时间。后台通过异步任务定期更新缓存，保证数据一致性，热点数据永不物理过期。

**实战案例**：
在秒杀场景下，商品详情页 Key 失效导致万级 QoS 直接打挂 MySQL。改用逻辑过期方案后，通过后台线程独立刷新，彻底消除了数据库突发流量。

**代码示例（互斥锁伪代码）**：
```java
public String getWithLock(String key) {
    String value = redis.get(key);
    if (value == null) {
        try {
            if (redis.setnx(key + ":lock", "1", 10)) { // 获取锁
                value = db.query(); // 查库
                redis.set(key, value, 30);
            } else {
                Thread.sleep(100); value = getWithLock(key); // 失败重试
            }
        } finally { redis.del(key + ":lock"); }
    }
    return value;
}
```

### 缓存穿透
- **定义**：查询一个不存在的数据，缓存没命中，数据库也没有，导致每次请求都直接打到数据库。
- **解决方案**：
    1.  **缓存空对象**：当数据库查询为空时，也将该 Key 缓存（Value 设为 Null），并设置较短的过期时间。
    2.  **布隆过滤器**：在访问缓存前，先通过布隆过滤器判断 Key 是否可能存在。如果判断不存在，直接返回，避免查库。

**实战案例**：
遭受恶意攻击，用户使用大量随机 ID 请求接口。引入 Redisson 布隆过滤器预热所有合法 ID 后，非法请求在网关层直接拦截，DB 压力归零。

**代码示例（Guava 布隆过滤器初始化）**：
```java
BloomFilter<String> filter = BloomFilter.create(Funnels.stringFunnel(Charset.defaultCharset()), 10000, 0.01);
// 预热：将所有存在的 Key 放入过滤器
validKeys.forEach(filter::put);

if (!filter.mightContain(inputKey)) {
    return null; // 一定不存在，直接返回
}
```

### MQ 消息不丢失
```text
Producer ──(发送)──> MQ Server ──(投递)──> Consumer
    │               │               │
  ①Confirm      ②持久化        ③手动ACK
  (事务/回调)    (Queue+Msg)     (业务完成后)
```
1.  **生产者**：
    - **Confirm 机制**（推荐）：消息发送到 MQ 后，MQ 返回 ACK。生产者通过异步回调确认发送成功，失败时可重试。
    - **事务机制**：发送消息前开启本地事务，保证消息发送与本地事务原子性（性能较低）。
2.  **MQ 服务端（以 RabbitMQ 为例）**：
    - 开启持久化：Queue 持久化 (`durable=true`) + Message 持久化 (`deliveryMode=2`)。
    - 集群部署：开启镜像队列，防止单节点宕机丢失。
3.  **消费者**：
    - 关闭自动 ACK (`autoAck=false`)。
    - 业务逻辑执行成功后，手动发送 ACK 给 MQ；如果失败，NACK 或拒绝并重新入队。

**实战案例**：
生产者因网络波动未收到 Broker ACK 导致消息重复投递，造成下游资金流水重复。生产者端引入本地消息表做“对账”重试，确保发送不丢且不重。

### MQ 消息顺序性
- **原因**：多线程消费、多消费者分片消费、网络重试等原因导致乱序。
- **解决方案**：
    1.  **单一分区/队列 + 单消费者**：一个 Topic 分多个 Queue，同一业务 ID（如订单 ID）的消息通过 Hash 算法发到同一个 Queue，且该 Queue 只能由一个消费者消费。
    2.  **内存队列重排**：消费者内部多线程接收后，放入内存队列，由单线程进行排序处理（复杂度高，性能损耗大）。通常推荐方案 1。

**代码示例（RocketMQ 顺序消息发送）**：
```java
SendResult sendResult = producer.send(msg, new MessageQueueSelector() {
    @Override
    public MessageQueue select(List<MessageQueue> mqs, Message msg, Object arg) {
        Integer id = (Integer) arg;  // 订单 ID
        int index = id % mqs.size(); // 根据 ID Hash 选择队列
        return mqs.get(index);
    }
}, orderId);
```

### 消息积压处理
- **场景**：消费者故障或处理速度过慢。
- **方案**：
    1.  修复消费者逻辑。
    2.  临时扩容：新建 Topic，分区数扩容 N 倍（如 10 倍）。
    3.  撰写临时转发程序：只负责从旧 Topic 快速消费消息并转发到新 Topic（不做复杂业务处理）。
    4.  部署 N 倍数量的消费者去消费新 Topic。

**实战案例**：
下游日志消费服务挂机 2 小时，导致 Kafka 积压 5 亿条数据。采用了“扩容 20 倍临时消费者 + 转发至新 Topic”的方案，仅耗时 30 分钟即消化完积压。

## 常见考点
1.  **消息幂等性**：如何保证消息不重复消费？解决方案：唯一 ID + 数据库唯一索引 / Redis 原子性 SetNX。
2.  **高可用架构**：RabbitMQ 的镜像队列或 Kafka 的 ISR 机制。
3.  **事务消息**（如 RocketMQ）：通过 Half Message 状态解决本地事务与消息发送的一致性问题。

### 方案对比：缓存击穿 vs 缓存穿透 vs 缓存雪崩

| 特性 | 缓存击穿 | 缓存穿透 | 缓存雪崩 |
| :--- | :--- | :--- | :--- |
| **根本原因** | 热点 Key 失效 | 查询不存在的数据 | 大量 Key 集中失效/宕机 |
| **影响对象** | 单个热点 Key | 随机无效 Key | 大量 Key/全盘 |
| **DB 压力** | 瞬时激增（高并发） | 持续高压（无缓存） | 瞬时激增（海量并发） |
| **核心方案** | 互斥锁 / 逻辑过期 | 布隆过滤器 / 缓存空值 | 随机 TTL / 多级缓存 / 高可用集群 |
| **适用场景** | 秒杀商品、热搜词条 | 恶意攻击、查询空值 | 缓存服务器重启、大促前夕 |

## 记忆要点

- 击穿与穿透：单点过期用互斥锁或逻辑过期，查不存在用布隆过滤器或缓存空值
- MQ不丢消息口诀：生产端用Confirm确认，服务端持久化存储，消费端关闭自动ACK改手动
- MQ顺序消费：因为多队列并发会乱序，所以按业务ID Hash至单一队列，且仅由单消费者消费
