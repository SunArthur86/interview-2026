# 任务调度与监控。

任务调度与监控是 MapReduce 框架的核心组件，负责将任务分配给合适的节点，并确保任务的容错和顺利执行。在 MRv1 中主要由 JobTracker 完成，在 YARN 中由 ResourceManager 和 ApplicationMaster 共同完成（此处以经典 MRv1 模式为主描述原理，YARN 逻辑类似但更复杂）。

### 实战案例
在一个运行在异构机房的集群中，部分节点磁盘性能较差，导致 Map 任务长时间无法完成。通过开启推测执行并设置 `mapreduce.map.speculative=true`，系统能够在慢节点上启动备份任务，使得整体作业耗时减少了约 40%。

### 核心机制

#### 1. 心跳与资源汇报
*   **机制**：TaskTracker 周期性地向 JobTracker 发送心跳。
*   **信息内容**：
    *   节点是否存活。
    *   当前节点的剩余资源（内存、CPU slot 数量）。
    *   已运行任务的进度和状态。

#### 2. 任务调度
*   **触发**：JobTracker 在收到心跳并发现有空闲资源时，会进行任务分配。
*   **调度策略**：调度器是可插拔的（如 FIFO、Capacity Scheduler、Fair Scheduler）。
*   **双层调度**：
    1.  **选择作业**：根据优先级、资源池等策略从队列中选出一个作业。
    2.  **选择任务**：从选中的作业中选择一个任务。
*   **数据本地性**：这是 MR 调度的关键。
    *   **Node-local**：数据和计算在同一个节点（最优）。
    *   **Rack-local**：数据在同一机架，通过交换机传输（次优）。
    *   **Off-switch**：数据在不同机架（最差）。

#### 3. 任务监控与容错
JobTracker 跟踪整个作业运行过程，提供全方位保障：
*   **任务失败重试**：如果 Task 汇报失败，JobTracker 会标记该任务失败，并重新调度（最多重试次数默认为 4）。
*   **speculation (推测执行)**：
    *   **场景**：当某个 Task 执行进度明显落后于同作业的其他 Task（拖后腿任务/Straggler）。
    *   **策略**：JobTracker 会启动一个相同的备份 Task，优先选取计算快的那个结果作为最终结果，同时杀掉慢的那个。
*   **TaskTracker 失效**：如果 JobTracker 在一段时间内未收到某 TaskTracker 的心跳，则判定其宕机。该节点上的所有 Task 将被重新调度到其他节点。

### 调度器对比
| 调度器 | 适用场景 | 特点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **FIFO** | 实验室/测试 | 先进先出，实现简单 | 大作业会阻塞小作业，吞吐量低，不适合多用户共享 |
| **Capacity** | 多部门共享集群 | 每个队列分配保证容量，空闲资源可借用 | 配置复杂，资源回收可能导致队列饥饿 |
| **Fair** | 交互式查询/小作业 | 动态资源分配，所有作业平均分配资源 | 短作业开启延迟，对大作业吞吐量有影响 |

## 记忆要点

- 调度核心：TaskTracker靠心跳汇报资源并领取任务。
- 数据本地性原则：因为减少网络传输，所以优先同节点，次选同机架。
- 推测执行：因为慢节点拖后腿，所以为长尾任务并行启动备份Task。
