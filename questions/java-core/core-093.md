# 什么是半连接队列？

### 半连接队列与全连接队列

在 TCP 三次握手过程中，服务器维护着两个重要的队列来管理连接状态。

#### 1. 半连接队列 (SYN Queue)
- **定义**：用于存放已经收到客户端 SYN 报文，但尚未完成三次握手的连接请求。
- **状态**：服务器收到 SYN 后，连接状态变为 `SYN_RCVD`，相关内核结构体会被放入半连接队列。
- **作用**：防止 SYN Flood 攻击（通过 `syncookies` 机制或队列长度限制）。

#### 2. 全连接队列 (Accept Queue)
- **定义**：用于存放已经完成三次握手（TCP 连接已建立），等待应用层调用 `accept()` 取走的连接。
- **状态**：第三次握手完成后，连接状态变为 `ESTABLISHED`，内核会将连接从半连接队列移到全连接队列（如果全连接队列未满）。
- **作用**：缓存已建立但未被应用程序处理的连接。

#### 内核处理流程
```text
    Client                              Server (Kernel)
      │                                     │
      │ ───── SYN (seq=x) ─────────────────> │
      │                                     │ (1) 检查半连接队列
      │                                     │ ┌───────────────────┐
      │                                     │ │ Half-Open Queue   │
      │                                     │ │ (SYN_RCVD)        │ ◄── 新增条目
      │                                     │ └───────────────────┘
      │                                     │
      │ <──── SYN+ACK (seq=y, ack=x+1) ──── │
      │                                     │
      │ ───── ACK (seq=x+1, ack=y+1) ─────> │ (2) 检查全连接队列
      │                                     │ ┌───────────────────┐
      │                                     │ │ Accept Queue      │
      │                                     │ │ (ESTABLISHED)     │ ◄── 从半连接移至此
      │                                     │ └───────────────────┘
      │                                     │
      │                                     │ (3) 等待 App 调用 accept()
      │                                     │
```

#### 实战案例
在高并发抢购场景中，若应用层处理逻辑阻塞导致 `accept()` 调用不及时，全连接队列堆积。一旦队列满，服务端会丢弃客户端的 ACK，客户端触发超时重传，导致服务端 RTT 飙升，甚至出现“服务端正常但客户端连接失败”的假象。

#### 代码示例 (Shell/Linux)
```bash
# 临时调整全连接队列长度（对应 net.core.somaxconn）
echo 65535 > /proc/sys/net/core/somaxconn
# 开启 SYN Cookies 防御 SYN Flood 攻击
echo 1 > /proc/sys/net/ipv4/tcp_syncookies
```

#### 对比表格：半连接队列 vs 全连接队列
| 特性 | 半连接队列 (SYN Queue) | 全连接队列 (Accept Queue) |
| :--- | :--- | :--- |
| **存储状态** | SYN_RCVD (收到 SYN，未收到 ACK) | ESTABLISHED (完成三次握手) |
| **触发事件** | 收到客户端 SYN 报文 | 收到客户端第三次握手 ACK 报文 |
| **取出方式** | 第三次握手成功后自动移出 | 应用层调用 `accept()` 取出 |
| **溢出后果** | 丢弃 SYN 或启用 SYN Cookies | 丢弃 ACK，客户端重传或连接超时 |
| **主要内核参数** | `net.ipv4.tcp_max_syn_backlog` | `net.core.somaxconn`, `listen()` backlog 参数 |

#### ## 常见考点
1. **队列溢出后果**：全连接队列满会导致服务端忽略客户端的 ACK，客户端超时重传；半连接队列满通常直接丢弃 SYN。
2. **相关 Linux 参数**：
   - `net.core.somaxconn`：指定全连接队列的最大长度。
   - `net.ipv4.tcp_max_syn_backlog`：指定半连接队列的最大长度。
   - `net.ipv4.tcp_abort_on_overflow`：全连接满时，是直接丢弃 (0) 还是发送 RST (1)。
