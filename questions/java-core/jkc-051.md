# 虚拟节点（down机多节点托管）

虚拟节点是针对标准一致性哈希中数据分布不均和负载过热问题的优化方案。

### 问题背景
在标准一致性哈希中，如果节点数较少，每个物理节点在环上只负责一段连续的区间。这会导致两个问题：
1.  **数据倾斜**：由于 Hash 分布的随机性，某些节点可能负责更多的数据，导致存储和负载不均。
2.  **热点雪崩（多米诺效应）**：如果一个负责热点数据的节点宕机，其负载会全部转移给顺时针的下一个节点。如果下一个节点本身负载已高，可能因此压垮，进而引发连锁反应。

### 虚拟节点解决方案
Cassandra 1.2 引入了虚拟节点（vnodes）：
- 每个物理节点不再分配单一的 Token，而是负责环上的多个 Token（默认为 256 个），即多个小片段。
- 这些片段在环上是交错分布的，不再连续。

### 架构示意图
```text
       [ 0 ]
       /   
   TokenRing (Hash Ring)
    /     \\
  [10]   [100] ...
   |       |
  Node A  Node B
(物理节点 A)(物理节点 B)
   |       |
  [A1]    [B1]  <-- 虚拟节点 Token
  [A2]    [B2] 
  [A3]    [B3] 
```

### 优势
1.  **负载均衡**：一个物理节点分散负责环上的多个小段，统计上数据分布更均匀，无需手动分配 Token。
2.  **故障转移更平滑**：当一个物理节点宕机时，它原本负责的多个小段数据会分散托管给集群中的多个其他节点（每个邻居只分担约 1/256 的负载），而不是全部压给一个邻居，避免了单点过载。
3.  **简化扩容**：新增节点时，无需手动调整现有节点的 Token，新节点会自动接管部分数据区间。

## 常见考点
- **Token 数量设置**：默认是 256 个，为什么不是更多或更少？（太少导致负载不均，太多增加元数据管理开销）。
- **数据修复**：虚拟节点如何影响 Repair 操作？（每个 vnode 独立进行，可能会产生大量流式传输，需使用 Sub-range Repair）。
- **快照**：由于每个物理节点对应多个 Token，冷备或快照时需要针对所有 Keypace/CF 统一处理。

### 💡 深化实战
**实战案例**：在 Cassandra 集群维护中，利用虚拟节点特性进行滚动重启。由于每个节点负责的数据碎片分散在整个环上，重启一台节点时，集群的整体吞吐量仅下降了约 5%（1/20），而非传统 Token 设计下的 20%-30%，极大降低了运维对业务的影响。

**代码示例（Java 虚拟节点计算）**：
```java
// 为每个物理节点生成多个虚拟节点 Hash 值
for (Node node : physicalNodes) {
    for (int i = 0; i < VIRTUAL_NODE_COUNT; i++) {
        // 虚拟节点 Key 格式: "NodeIP#索引"
        String virtualNodeKey = node.getIp() + "#" + i;
        int hash = hashFunction.hash(virtualNodeKey);
        sortedMap.put(hash, node); // 存入有序 TreeMap
    }
}
```

**对比表格：标准 Hash vs 虚拟节点**
| 维度 | 标准一致性 Hash | 虚拟节点 | 
| :--- | :--- | :--- |
| **数据分布** | 依赖 Hash 算法随机性，易倾斜 | 统计均匀，近似正态分布 |
| **节点宕机** | 流量全部迁移至下一个邻居 (风险高) | 流量均摊至环上所有节点 (风险低) |
| **扩容难度** | 需手动计算 Token 以平衡负载 | 自动计算 Token，即插即用 |
| **元数据开销** | 小 (仅物理节点信息) | 较大 (需维护 N * V 个 Token 映射) |
| **适用场景** | 节点规模固定且稳定 | 大规模动态扩缩容集群 |
