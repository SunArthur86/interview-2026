# select/poll/epoll的区别和联系？

### select/poll/epoll 的区别和联系

**联系**：三者在 Linux 中都是 I/O 多路复用机制，都允许一个单独的线程或进程同时监视多个文件描述符，当某个文件描述符就绪（如可读或可写）时，通知应用程序进行读写操作。

#### 1. select
*   **原理**：使用位图来存储文件描述符集合。调用时需要将 fd_set 从用户态拷贝到内核态，内核遍历所有 fd 查看状态。
*   **关键细节与参数**：
    *   `FD_SETSIZE`：默认为 1024，修改需重新编译内核。
    *   `nfds`参数：值为监听的最大 fd 值 + 1，内核需轮询 0 到 nfds。
*   **缺点**：
    *   **数量限制**：默认最大监听 1024 个 FD。
    *   **性能开销**：每次调用都需要把 fd 集合从用户态拷贝到内核态；内核轮询所有 FD 效率低（O(n)）。
    *   **线性扫描**：返回后也需要应用程序遍历整个 fd_set 找到就绪的 FD。

#### 2. poll
*   **原理**：使用结构体数组来存储 FD 及其关注的事件。
*   **关键细节**：
    *   结构体包含 `events`（输入事件）和 `revents`（返回事件），解决了 select 参数传入传出混用的问题。
*   **改进**：解决了 select 的数量限制问题（理论上无上限，受系统内存限制）。
*   **缺点**：
    *   **性能开销**：每次调用仍需将数组从用户态拷贝到内核态。
    *   **线性扫描**：内核和应用层仍需轮询遍历，随着 FD 数量增加，性能线性下降（O(n)）。

#### 3. epoll
*   **原理**：基于事件驱动。在内核中维护一颗红黑树和一张就绪链表。
*   **核心机制**：
    1.  **epoll_ctl**：添加 FD 时，通过回调机制在该 FD 就绪时将其加入就绪链表。
    2.  **epoll_wait**：直接检查就绪链表，无需遍历所有 FD。
*   **关键细节与模式**：
    *   **ET (Edge Triggered)**：边缘触发。只有状态发生变化时才通知，必须一次性读完数据，支持非阻塞 IO。
    *   **LT (Level Triggered)**：水平触发（默认）。只要条件满足（如缓冲区有数据）就一直通知，支持阻塞和非阻塞 IO。
*   **优点**：
    *   **高效**：时间复杂度为 O(1)（取决于活跃连接数，而非总连接数）。
    *   **内存拷贝**：使用 mmap 或者高效拷贝，避免频繁拷贝 FD 集合。
    *   **支持触发模式**：支持水平触发（LT）和边缘触发（ET）。
*   **限制**：仅适用于 Linux。

### 对比表格

| 特性 | select | poll | epoll |
| :--- | :--- | :--- | :--- |
| **底层数据结构** | bitmap (位图) | 数组 | 红黑树 + 就绪链表 |
| **最大连接数** | 有限制 (默认1024) | 无限制 (受内存限制) | 无限制 (受内存限制，1G内存约10万连接) |
| **查找效率** | O(n) 线性遍历 | O(n) 线性遍历 | O(1) 事件驱动 |
| **消息传递方式** | 每次调用进行完整拷贝 | 每次调用进行完整拷贝 | 仅在注册时拷贝，使用 mmap 共享内存 |
| **适用场景** | 连接数少且活跃 | 连接数多但活跃度高 | 高并发，连接数多但活跃度低 |

### 实战案例

*   **C10K 问题**：在早期的 Apache 服务器中使用 select 模型，当并发连接数超过 1024 时性能急剧下降。Nginx 和 Redis 采用 epoll 机制，能够轻松处理数万甚至数十万并发连接，是现代高并发服务器的标配。
*   **epoll 惊群效应**：在多线程环境下，如果多个线程都在 `epoll_wait` 同一个 epoll 对象，当一个事件到来时，所有线程都可能被唤醒，但实际上只有一个线程需要处理它，造成资源浪费。解决方案是使用 `EPOLLEXCLUSIVE` 标志（Linux 4.5+）或使用全局锁（如 Nginx 的 `accept_mutex`）确保同一时刻只有一个线程在 accept。

### 代码示例 (C)

```c
// epoll 使用关键片段
int epfd = epoll_create1(0);
struct epoll_event ev, events[MAX_EVENTS];

// 添加监听 fd
ev.events = EPOLLIN; // 监听可读事件
ev.data.fd = listen_sock;
epoll_ctl(epfd, EPOLL_CTL_ADD, listen_sock, &ev);

// 等待事件
int nfds = epoll_wait(epfd, events, MAX_EVENTS, -1);
for (int i = 0; i < nfds; ++i) {
    if (events[i].data.fd == listen_sock) {
        // 处理新连接
    } else {
        // 处理已连接 fd 的数据
    }
}
```
