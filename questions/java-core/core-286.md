# 工厂方法模式的设计思想和应用场景是什么？

### 工厂方法模式

#### 设计思想
工厂方法模式是对简单工厂模式的进一步解耦。在简单工厂模式中，工厂类负责创建所有产品，随着产品种类增加，工厂类会变得臃肿且违反开闭原则（OCP）。

工厂方法模式定义了一个用于创建对象的接口，但让实现这个接口的类来决定实例化哪一个类。工厂方法让类的实例化推迟到子类。

*   **核心角色**：
    1.  **抽象工厂**：工厂创建对象的接口。
    2.  **具体工厂**：实现抽象工厂接口，负责创建具体产品。
    3.  **抽象产品**：产品的接口。
    4.  **具体产品**：具体工厂创建的对象。

#### 架构图示

```text
      Client (客户端)
         |
         v
+---------------------+
|   <<interface>>     |
|     Factory         | <--- 抽象工厂
+---------------------+
| +createProduct()    |
+---------+-----------+
          ^
          | implements
+---------+-----------+           +---------------------+
|  ConcreteFactoryA  |---------> |  ConcreteProductA  |
+---------------------+  creates  +---------------------+
                                   | <<interface>>      |
                          ^        |      Product       | <--- 抽象产品
                          |        +---------------------+
          | implements           |          ^
+---------+-----------+           |          |
|  ConcreteFactoryB  |--------+  |          | implements
+---------------------+  creates |  |    +---------------------+
                                 |  +----|  ConcreteProductB  |
                                 |       +---------------------+
                                 |
                         (无需修改 Client)
```

#### 优点与缺点
*   **优点**：
    *   符合**开闭原则**：新增产品时只需增加具体工厂和具体产品，无需修改原有代码（如 Client 代码）。
    *   符合**单一职责原则**：每个具体工厂只负责创建一种产品。
    *   解耦：高层模块（Client）只需要知道抽象工厂接口，不依赖具体产品类。
*   **缺点**：
    *   类的个数成对增加，增加了系统的复杂度。
    *   增加了系统的抽象性和理解难度。

#### 实际应用场景
1.  **JDK Collection**：`Collection` 接口的 `iterator()` 方法相当于抽象工厂，`ArrayList` 和 `LinkedList` 等具体集合类是实现工厂，它们分别返回适合自己的 `Iterator`（具体产品）。
2.  **JDK DateFormat**：`DateFormat` 类中的 `getDateInstance()` 等方法。
3.  **日志框架**：如 Log4j、Logback 中获取 Logger 的过程。
4.  **数据库驱动**：JDBC 中 `DriverManager.getConnection()` 实际上是利用了不同厂商实现的 `Driver` 类（虽然具体实现细节略有不同，但体现了工厂思想）。

### 实战案例
在设计**支付网关**系统时，早期通过 `if-else` 判断是“支付宝”还是“微信支付”。随着接入渠道增多（银联、PayPal等），代码变得难以维护。重构为工厂方法模式后，新增渠道只需新增一个 `PayFactory` 实现类和 `PayStrategy` 实现类，核心支付流程无需改动，极大提升了系统的扩展性。

### 代码片段（扩展性强）
```java
// 抽象工厂
public interface LoggerFactory {
    Logger createLogger();
}

// 具体工厂 - 文件日志
public class FileLoggerFactory implements LoggerFactory {
    @Override
    public Logger createLogger() {
        return new FileLogger();
    }
}

// 具体工厂 - 数据库日志
public class DatabaseLoggerFactory implements LoggerFactory {
    @Override
    public Logger createLogger() {
        return new DatabaseLogger();
    }
}
```
