# 变量可见性

### 变量可见性与 volatile 关键字

`volatile` 是 Java 虚拟机提供的**轻量级同步机制**，主要用于保证多线程环境下的**可见性**和**有序性**，但不能保证原子性。

#### 1. 变量可见性

**JMM (Java 内存模型) 规定**：
- 每个线程拥有自己的**工作内存**（Working Memory，对应 CPU Cache/寄存器），保存了主内存中变量的副本。
- 普通变量：线程修改副本后，何时写回主内存是不确定的。
- **volatile 变量**：
  - **写操作**：强制将修改后的值**立即刷新**回主内存。
  - **读操作**：强制从主内存**重新读取**最新值。

这就保证了当某线程修改了 volatile 变量，其他线程能立刻感知到变化。

```text
┌───────────┐      Write      ┌─────────────┐
│ Thread A  │ ──────────────> │  Main Memory│
│ (Local)   │  (Volatile)     │  (Volatile V)│
└───────────┘                  └─────────────┘
       ^                               │
       │       Read (Volatile)         │
       └───────────────────────────────┘
                    ↓
             ┌───────────┐
             │ Thread B  │
             │ (Local)   │
             └───────────┘
```

**实战案例**：在实现基于轮询的“停止线程”逻辑时，若 `boolean running` 未加 volatile，主线程修改了标志位，工作线程可能因读取本地缓存导致死循环无法停止。

**代码示例**：
```java
// 错误示范：可能导致死循环
// private boolean running = true;

// 正确示范：使用 volatile 保证可见性
private volatile boolean running = true;

public void stop() { running = false; }

public void run() {
    while (running) { 
        // do work 
    }
}
```

#### 2. 禁止指令重排序

为了优化性能，编译器和处理器会对指令序列进行**重排序**。在单线程下没问题，但在多线程下可能导致逻辑错误（如单例模式的双重检查锁问题）。

**volatile 通过插入「内存屏障」** 来禁止特定类型的重排序：
- 在 volatile 变量写操作**之前**的普通操作，不会被重排序到写操作**之后**。
- 在 volatile 变量读操作**之后**的普通操作，不会被重排序到读操作**之前**。

#### 3. 无法保证原子性

`volatile` 仅能保证单次读/写操作的原子性。对于复合操作（如 `i++`），它包含「取值-加1-写回」三步，即便 `i` 是 volatile，这三步也不具备原子性，仍需加锁或使用原子类（如 `AtomicInteger`）。

**对比表格**：volatile vs synchronized vs Atomic

| 特性 | volatile | synchronized | AtomicInteger (CAS) |
| :--- | :--- | :--- | :--- |
| **可见性** | 保证 | 保证 | 保证 |
| **有序性** | 保证 (禁止重排序) | 保证 | 不保证 |
| **原子性** | 不保证 (仅单次读写) | 保证 | 保证 (通过循环 CAS) |
| **阻塞** | 非阻塞 | 阻塞 (重量级锁) | 非阻塞 (自旋) |
| **适用场景** | 状态标志位、单例 DCL | 代码块/方法同步 | 计数器、序列号生成 |

## 常见考点
1. **volatile 和 synchronized 的区别？**
   - **volatile**：轻量，保证可见性和有序性，不阻塞线程，无法保证原子性。
   - **synchronized**：重量级（但 JVM 做了大量优化），保证可见性、有序性和原子性，会阻塞线程。
2. **单例模式 DCL 为什么需要 volatile？**
   - 防止「实例初始化」和「引用指向对象地址」发生指令重排序。若重排序，另一个线程可能获取到一个「半初始化」的对象，导致报错。
3. **什么场景适合用 volatile？**
   - 状态标志位（如 `volatile boolean shutdownRequested`）；
   - 单次写/读操作（如 `volatile long lastTimestamp`）。


## 记忆要点

- 强制可见性：强制线程写后立即刷主存、读前从主存重拉，保障多线程数据可见
