# CSS性能优化

### CSS 性能优化

CSS 性能优化是提升前端页面加载速度和渲染流畅度（FPS）的关键环节。以下是核心优化策略及原理补充。

#### 1. 内联首屏关键 CSS

-   **原理**：浏览器渲染流程是构建 DOM -> 构建 CSSOM -> 合并 RenderTree -> 布局 -> 绘制。如果 CSS 文件很大或加载慢，浏览器会阻塞渲染直到 CSS 下载解析完成（白屏）。
-   **策略**：提取首屏渲染所需的 CSS（通常小于 20KB），直接内联在 HTML 的 `<style>` 标签中，让页面快速渲染出首屏内容。其余 CSS 异步加载。

#### 2. 异步加载 CSS

-   对于非首屏 CSS，避免阻塞渲染。可以使用以下方式：
    ```html
    <link rel="preload" href="style.css" as="style" onload="this.onload=null;this.rel='stylesheet'">
    <noscript><link rel="stylesheet" href="style.css"></noscript>
    ```
    -   使用 `media` 属性欺骗（如 `media="print"`），加载后通过 JS 改回 `all`。

#### 3. 资源压缩

-   **Gzip / Brotli**：在服务器端开启压缩，可减少 60%-80% 的传输体积。
-   **代码压缩**：使用 `cssnano`, `clean-css` 等工具去除空格、注释、无用代码。

#### 4. 合理使用选择器

-   **避免通配符**：`*` 匹配所有元素，性能开销大。
-   **避免深层嵌套**：CSS 选择器是从右向左匹配的。例如 `.nav .item a span`，浏览器需先找到所有 `span`，再向上匹配。尽量保持选择器简短（如使用 BEM 命名法，类名直接控制）。
-   **避免标签选择器**：如 `div.header`，直接 `.header` 更快。

#### 5. 减少使用昂贵的属性

-   某些属性会触发大量重绘或复杂的计算，慎用：
    -   `box-shadow` (层级越多开销越大)
    -   `border-radius`
    -   `:nth-child` 等伪类选择器（特别是复杂的表达式）
    -   `filter` (高耗能，需硬件加速)
    -   `position: fixed` (会导致滚动时的重绘)

#### 6. 不要使用 @import

-   **原因**：`@import` 是同步加载的，且必须等待父 CSS 解析完才开始请求子 CSS，导致下载链路串行化，延长页面加载时间。推荐使用多个 `<link>` 并行加载。

#### 7. CSS 雪碧图

-   **原理**：将多个小图标合并成一张大图，通过 `background-position` 显示不同部分。
    -   **优点**：减少 HTTP 请求数（在 HTTP/1.1 环境下有效）。
    -   **缺点**：维护困难，内存占用稍高。在 HTTP/2 及现代环境下，多图请求开销已降低，SVG Sprites 或 Icon Fonts 可能是更好的选择。

#### 8. 小图片转为 Base64 编码

-   **原理**：将图片数据编码为 Base64 字符串直接嵌入 CSS。
-   **场景**：适用于极小的图标（如几 KB），避免额外的 HTTP 请求。
-   **缺点**：Base64 编码后体积会增大约 33%，且无法利用浏览器缓存。大图片严禁使用。

#### 9. 继承与公共类

-   利用 CSS 的继承机制（如 `color`, `font-size`, `line-height`）减少代码冗余。
-   抽取公共类（如 `.text-center`, `.clearfix`），虽然可能增加 DOM 类名，但能减少 CSS 体积并提高复用性。

#### 实战案例：减少重排提升滚动流畅度
在开发长列表页面时，如果在 `scroll` 事件中直接修改元素 `top` 属性，会导致页面频繁重排，造成严重卡顿。实战中应使用 `transform: translate3d(...)` 代替，这只会触发合成，避免了布局计算和重绘，能维持 60FPS 的流畅度。

#### 关键代码示例：使用 contain 属性优化
```css
/* 告诉浏览器该元素内部变化不会影响外部布局，减少渲染范围 */
.card {
  /* strict 声明元素在布局、样式、绘制上与外部完全隔离 */
  contain: strict; 
  /* 或者仅针对布局优化 */
  /* contain: layout; */
}
```

#### 渲染路径对比

| 操作类型 | 说明 | 触发环节 | 性能影响 |
| :--- | :--- | :--- | :--- |
| **Layout (重排)** | 几何属性变化，计算位置和大小 | DOM/CSSOM -> RenderTree | 高（最耗性能） |
| **Paint (重绘)** | 样式变化（如颜色），未改变几何属性 | RenderTree -> Layer | 中 |
| **Composite (合成)** | 图层合并，利用 GPU 硬件加速 | Layer -> Screen | 低（最理想） |

## 常见考点

1.  **浏览器解析 CSS 选择器的方向**：为什么说“从右向左”匹配？如何写出高性能选择器？
2.  **重排与重绘**：哪些 CSS 属性会触发重排，哪些只触发重绘？如何减少重排？（使用 `transform` 代替 `top/left` 动画）。
3.  **Critical CSS 的提取原理**：如何计算哪些 CSS 是首屏必需的？（通常依赖工具模拟视口运行页面，收集用到的样式）。
4.  **CSS 阻塞渲染**：`<link>` 在 `<head>` 中为何重要？JS 操作 CSSOM 时会发生什么？

## 记忆要点

- 选择器匹配：因为浏览器从右向左解析，所以避免深层嵌套和通配符
- 渲染消耗对比：重排(几何) > 重绘(样式) > 合成(GPU)，慎用 box-shadow/filter 等昂贵属性
- 加载优化：内联首屏关键CSS，异步加载非首屏CSS，坚决不用 @import 串行加载
- 阻塞优化：因为操作 DOM 的 top 会引发频繁重排，所以滚动动画用 transform 替代
