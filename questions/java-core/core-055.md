# 为什么 TIME_WAIT 等待的时间是 2MSL？

### 为什么 TIME_WAIT 等待的时间是 2MSL？

**MSL (Maximum Segment Lifetime)**：报文最大生存时间，指任何报文在网络上存在的最长时间。

**2MSL 的原因**：

1.  **保证最后一个 ACK 能到达被动关闭方，确保其正常关闭**：
    *   如果第四次挥手的 ACK 丢失，被动关闭方（服务端）会重传 FIN 报文。
    *   主动关闭方（客户端）需要维持在 TIME_WAIT 状态来接收这个重传的 FIN，并再次发送 ACK。
    *   1 个 MSL 用于等待被动方的重传 FIN 报文到达，1 个 MSL 用于等待客户端重发的 ACK 报文到达，一来一回正好 2MSL。

2.  **让旧连接的所有报文在网络中消失**：
    *   等待 2MSL 后，本次连接产生的所有报文（包括迟到的 FIN 或重传的 FIN）都会从网络中丢弃。
    *   这样可以确保下一个新的连接中，不会收到旧连接的残留报文，避免数据混乱。

### 为什么挥手需要四次？

*   **原因**：TCP 是全双工通信，双方都可以发送和接收数据。
    *   当 A 发送 FIN 时，仅代表 A 没有数据要发送了，但 A 仍可以接收数据。
    *   B 收到 FIN 后，可能还有数据需要处理和发送给 A，所以 B 先发送 ACK（表示我知道你要关闭了），等 B 发完剩余数据后，再发送 FIN 给 A。
    *   因此，ACK 和 FIN 通常是分开发送的，导致比三次握手多了一次。

### 挥手状态流转图

```text
Active Side (e.g. Client)       Passive Side (e.g. Server)
       │                                 │
       │ ────── FIN (seq=x) ────────────> │  (ESTABLISHED -> CLOSE_WAIT)
       │  (FIN_WAIT_1)                   │
       │ <───── ACK (ack=x+1) ─────────── │
       │  (FIN_WAIT_2)                   │  (App can still send data)
       │                                 │
       │ <───── FIN (seq=y) ──────────── │  (LAST_ACK)
       │  (TIME_WAIT)                    │
       │ ────── ACK (ack=y+1) ──────────> │  (CLOSED)
       │                                 │
 (Wait 2MSL)                            │
       │  (CLOSED)                       │
```

## 常见考点
1.  **TIME_WAIT 过多危害**：占用连接数（端口资源），占用内存。
2.  **如何优化 TIME_WAIT**：开启 `net.ipv4.tcp_tw_reuse`（允许将 TIME_WAIT sockets 重新用于新的 TCP 连接），`tcp_tw_recycle`（已被废弃，因 NAT 环境下时间戳问题）。
3.  **CLOSE_WAIT 状态**：如果服务端出现大量 CLOSE_WAIT，通常意味着代码层面未正确关闭流（如未调用 `close()` 或 Socket 未读到底）。
4.  **MSL 典型值**：Linux 默认通常为 30秒 或 60秒，因此 TIME_WAIT 默认为 60秒 或 120秒。
