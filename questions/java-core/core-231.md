# 什么是协商缓存？

### 什么是协商缓存？

**定义**：
协商缓存，又称对比缓存。是指当浏览器（客户端）再次请求一个资源时，服务器会根据请求头中的特定验证字段来验证该资源在服务器端是否发生了修改。**如果没有修改，服务器返回 304 Not Modified 状态码（不返回资源实体），通知浏览器使用本地缓存；如果已修改，则返回 200 OK 和最新的资源内容。**

**前提条件**：
协商缓存通常发生在**强制缓存**失效之后（即 Cache-Control 的 max-age 过期或设置了 no-cache）。

**协商缓存的主要机制**：

#### 1. 基于 Last-Modified 和 If-Modified-Since
这是 HTTP/1.0 中引入的机制。

*   **流程**：
    1.  客户端第一次请求资源，服务器读取文件最后修改时间，将其放入响应头的 `Last-Modified` 字段返回。
    2.  客户端再次请求时，将上次的 `Last-Modified` 值写入请求头的 `If-Modified-Since` 字段发送给服务器。
    3.  服务器收到请求，对比服务器资源的实际修改时间与 `If-Modified-Since`。
        *   **未修改**：返回 304 Not Modified，响应体为空。
        *   **已修改**：返回 200 OK，返回新资源，并更新 `Last-Modified`。

*   **缺点（ASCII 流程图示意）**：
    ```text
    Client                      Server
      │                            │
      │───── Request (No Cache) ──>│
      │<──── Response 200 + Last-Modified ──│ (e.g., 2023-01-01 10:00:00)
      │                            │
      │ (缓存过期，再次请求)          │
      │───── Request + If-Modified-Since (2023-01-01 10:00:00) ──>│
      │                            │ [比较实际修改时间]
      │<──── Response 304 (未改) ──│ (或 200 + 数据，若已改)
    ```

#### 2. 基于 ETag 和 If-None-Match
这是 HTTP/1.1 中引入的机制，优先级高于 `Last-Modified`。

*   **原理**：
    *   **ETag**：服务器根据文件内容生成的唯一标识符（通常是文件内容的 hash 值或版本号）。
    *   **If-None-Match**：客户端请求时携带上次收到的 ETag 值。
*   **流程**：
    1.  客户端第一次请求，服务器计算 ETag，放入响应头返回。
    2.  客户端再次请求，携带 `If-None-Match: "etag-value"`。
    3.  服务器计算当前资源的 ETag，并与 `If-None-Match` 对比。
        *   **一致**：返回 304 Not Modified。
        *   **不一致**：返回 200 OK，返回新资源和新 ETag。

*   **ETag 的两种类型**：
    *   **强验证**：基于字节精确生成，哪怕改一个标点符号，ETag 也会变（准确但消耗计算）。
    *   **弱验证**：基于文件属性部分生成，以 `W/` 开头。允许内容等价但字节不同的情况（如 HTML 压缩前后），速度快但精确度稍低。

*   **Last-Modified vs ETag 对比**：
    | 特性 | Last-Modified | ETag |
    | :--- | :--- | :--- |
    | **精确度** | 低 (秒级) | 高 (内容级) |
    | **性能** | 读取文件属性，开销小 | 需计算 Hash，开销相对大 |
    | **优先级** | 低 | 高 |
    | **缺陷** | 1. 秒内修改无法感知<br>2. 文件改回来时间没变 | 计算消耗服务器资源 |

**实战案例**：
在某电商后台管理系统中，静态 JS 资源更新频繁但文件名未做 Hash 化，导致用户在 CDN 没刷新时总是加载到旧脚本引发报错。后改为 **ETag** 协商缓存，确保文件内容一变必回源，同时结合强缓存策略解决了缓存一致性问题。

**代码示例**（Node.js 设置 ETag）：
```javascript
// 实战：Node.js Express 启用强 ETag 验证
const express = require('express');
const app = express();

// 'strong' 模式会对内容进行严格校验，适合高一致性要求的 API
app.set('etag', 'strong');

app.get('/api/data', (req, res) => {
  const data = { id: 1, val: 'test' };
  // Express 自动根据 body 计算 ETag 并匹配 req.headers['if-none-match']
  res.json(data);
});
```

**对比表格**：

| 缓存类型 | 强制缓存 | 协商缓存 |
| :--- | :--- | :--- |
| **核心标识** | Cache-Control (max-age), Expires | ETag, Last-Modified |
| **状态码** | 200 (from disk/memory/cache) | 304 (Not Modified) |
| **数据来源** | 浏览器本地缓存（未过期不请求） | 服务器验证（请求头发送，但无 body） |
| **优先级** | 高（若命中，不走协商） | 低（强缓存失效后才触发） |
