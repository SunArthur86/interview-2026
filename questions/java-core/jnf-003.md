# 什么是密封类（Sealed Classes）？它如何与模式匹配（Pattern Matching）配合使用？

密封类在Java 15预览、Java 17正式发布（JEP 409）。它允许类或接口使用sealed关键字和permits子句显式限制哪些类型可以继承或实现它。每个被许可的子类必须声明为final（不可再继承）、sealed（继续限制其子类型）或non-sealed（重新开放继承）。

```java
// 定义密封接口
public sealed interface Shape 
    permits Circle, Rectangle, Triangle {}

// 使用Record实现密封接口
public record Circle(double radius) implements Shape {}
public record Rectangle(double width, double height) implements Shape {}
public record Triangle(double a, double b, double c) implements Shape {}
```

密封类的最大价值在于启用编译器可验证的穷举模式匹配。当对密封类型的引用使用switch表达式时，编译器知道所有可能的子类型，可以验证每个具体类型是否被处理。

```java
// Java 21 模式匹配 for switch（JEP 441）
public double calculateArea(Shape shape) {
    return switch (shape) {
        case Circle c → Math.PI * c.radius() * c.radius();
        case Rectangle r → r.width() * r.height();
        case Triangle t → { /* ... yield ... */ }
        // 不需要default！编译器知道这是穷举的
    };
}
```

**实战案例：支付系统状态机**
在支付系统中，订单状态通常有限且封闭。使用 Sealed Class 可以彻底消除 `default` 分支，防止新增状态（如 `REFUNDING`）时漏处理。
*   **代码示例**：
    ```java
    public sealed interface PaymentStatus permits Paid, Failed, Pending {}
    public record Paid(String transactionId) implements PaymentStatus {}
    public record Failed(String errorCode) implements PaymentStatus {}
    public record Pending() implements PaymentStatus {}

    // 处理逻辑，如果新增 RefundStatus，编译器会报错强制处理
    public void handlePayment(PaymentStatus status) {
        switch (status) {
            case Paid p -> saveAudit(p.transactionId());
            case Failed f -> alertSupport(f.errorCode());
            case Pending p -> scheduleRetry();
        }
    }
    ```

**Sealed Class 模式设计对比**：

| 设计模式 | 传统接口/抽象类 | Sealed Classes
| :--- | :--- | :---
| **扩展性** | 开放扩展（任何人可继承） | 封闭扩展（显式许可）
| **安全性** | 依赖运行时类型检查 | 编译时穷举检查
| **维护性** | 修改父类可能破坏子类 | 新增子类时编译器提示所有调用点
| **Switch 覆盖** | 必须 default 分支 | 不需要 default (穷尽时)
