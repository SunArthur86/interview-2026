# 常见的负载均衡算法有哪些？

### 常见负载均衡算法

负载均衡算法决定了如何将传入的网络流量分发到后端的服务器集群。

**1. 轮询：**
- **原理**：请求按顺序轮流分配给每台服务器（A, B, C, A, B, C...）。
- **特点**：简单公平，适用于服务器性能相近的场景。
- **缺点**：不考虑服务器当前的负载和性能差异，可能导致慢服务器积压请求。

**2. 加权轮询：**
- **原理**：给性能强的服务器分配更高权重，按权重比例分配请求。
- **示例**：服务器 A(权重3)、B(权重1)，则分配序列为 A, A, A, B, A, A, A, B...
- **应用**：适合服务器硬件配置差异明显的场景。

**3. 随机：**
- **原理**：随机选取一台服务器。在请求数量级很大（大数定律）时，分布趋近于轮询。
- **变种**：加权随机，按权重设置概率区间。

**4. 最少连接：**
- **原理**：把请求发给当前**活动连接数**最少的服务器。
- **应用**：适合长连接（如数据库连接、WebSocket）或处理时间差异大的场景。能动态平衡负载。

**5. 源地址哈希（IP Hash）：**
- **原理**：`Hash(client_ip) % N`，根据计算结果固定映射到同一台服务器。
- **应用**：实现「会话保持」，无需 Session 共享即可保证同一客户端访问同一服务器。
- **缺点**：服务器增删或宕机会导致取模结果变化，造成**大量请求重新路由**，导致 Session 失效。

**6. 一致性哈希：**
- **原理**：将服务器节点和数据请求 Key 都映射到一个 0 ~ 2^32-1 的哈希环上，顺时针查找最近的节点。
- **优点**：当节点增删时，只影响相邻节点的数据，不会导致全量 Rehash，具有良好的容错性和扩展性。
- **优化**：引入**虚拟节点**解决数据倾斜问题，使数据更均匀地分布。

**7. 最短响应时间：**
- **原理**：优先选择平均响应时间最短的服务器。
- **优点**：能动态感知后端实例的健康状况和性能波动。

**8. 四层 vs 七层负载均衡**
- **四层（L4, 如 LVS）**：基于 IP + 端口进行转发，性能极高，但无法识别内容。
- **七层（L7, 如 Nginx）**：基于 HTTP 协议内容（URL、Cookie、Header）转发，功能灵活（如动静分离），但性能略低于四层。

## 实战案例
在**双十一大促**场景下，电商推荐服务通常使用**最少连接数**算法而非轮询，因为不同商品的推荐计算耗时差异极大（秒杀商品计算复杂，普通商品简单），轮询会导致慢请求积压压垮特定节点，而最少连接能避免队列溢出。对于 WebSocket 推送服务，必须使用 IP Hash 或一致性哈希以保证连接不断开。

## 算法选型对比

| 算法 | 核心逻辑 | 适用场景 | 缺点 | 稳定性 |
| :--- | :--- | :--- | :--- | :--- |
| 轮询 | 按顺序分发 | 服务器性能均等 | 无法感知负载差异 | 高
| 加权轮询 | 按权重比例分发 | 服务器性能不均 | 权重设置不当易倾斜 | 高
| 最少连接 | 优先分发连接数最少者 | 长连接、请求耗时差异大 | 需维护连接状态 | 中
| 一致性哈希 | 哈希环顺时针查找 | 缓存服务、有状态服务 | 实现较复杂，偶发数据倾斜 | 高 |
