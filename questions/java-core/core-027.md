# 什么是递归查询和迭代查询？

### 递归查询和迭代查询
递归查询和迭代查询是在 DNS 解析过程中用于获取域名解析信息的两种不同交互模式。

#### 1. 递归查询
*   **定义**：DNS 客户端（通常是本地 DNS 解析器 Stub Resolver 或配置了递归的 DNS Server）向 DNS 服务器发起查询，并要求对方**必须**返回最终的 IP 地址，或者是报错。
*   **过程**：
    1.  客户端询问 Local DNS："www.google.com 的 IP 是多少？"
    2.  Local DNS 负责向根、顶级域、权威域等服务器依次查询，直到拿到结果。
    3.  Local DNS 将最终 IP 返回给客户端。
*   **特点**：客户端负载轻，只需发一次请求；被查询的服务器负担重。

#### 2. 迭代查询
*   **定义**：DNS 服务器（通常是根服务器或顶级域服务器）收到查询请求后，**不直接去查下一级**，而是告诉客户端："我不知道答案，但是你去问 X 服务器（下级 NS 记录地址）可能知道"。
*   **过程**：
    1.  Local DNS 询问 Root Server："www.google.com 的 IP 是多少？"
    2.  Root Server 回复："我不知道，你去问 .com TLD 服务器（IP: x.x.x.x）"。
    3.  Local DNS 询问 .com TLD Server："www.google.com 的 IP 是多少？"
    4.  .com TLD Server 回复："我不知道，你去问 google.com 的权威服务器（IP: y.y.y.y）"。
    5.  Local DNS 询问权威服务器，获得 IP。
*   **特点**：服务器负载轻，只返回指针；客户端（或发起迭代的 DNS）需要发起多次请求，进行多次网络往返。

#### 交互模式对比表
| 特性 | 递归查询 | 迭代查询 |
| :--- | :--- | :--- |
| **发起方** | 客户端 -> Local DNS | DNS 服务器 -> DNS 服务器 |
| **服务器压力** | 高（需负责解析到底） | 低（只返回下一跳地址） |
| **网络开销** | 低（客户端只需一次请求） | 高（可能需要多次往返） |
| **解析复杂度** | 集中在 Local DNS | 分散在各级服务器 |

#### 区分场景
*   **递归查询**：适合**客户端**和**本地 DNS 服务器**之间（客户端发出递归，本地 DNS 服务于客户端）。
*   **迭代查询**：适合**DNS 服务器之间**（如 Local DNS 查询 Root、TLD、权威服务器时，通常是迭代模式，减轻根服务器压力）。

### 实战案例
在企业内网配置 DNS 时，如果开启了 **"转发器" (Forwarder)**，内网 DNS 会采用递归查询将请求转发给外部 DNS（如 8.8.8.8）。但如果外部 DNS 禁用了递归（大多数公共 DNS 如 8.8.8.8 为了防 DDoS 攻击会禁用非客户端递归），则解析会失败。解决方法是在内网建立自己的权威解析区域或配置支持递归的上游 DNS。

### 代码示例（使用 Go 实现 DNS 迭代查询客户端逻辑）
```go
// 模拟简化版的 DNS 迭代查询逻辑
func resolveIteratively(domain string) string {
    servers := []string{"根服务器IP"}
    
    for len(servers) > 0 {
        currentServer := servers[0]
        // 向当前服务器发起查询
        ip, nextServers := queryDNS(currentServer, domain)
        
        if ip != "" {
            return ip // 找到最终 IP
        }
        // 没找到，将下一级服务器加入列表继续迭代
        servers = append(servers[1:], nextServers...)
    }
    return "未找到"
}
```
