# React性能优化

React 性能优化通常遵循三个核心方向：减少渲染次数、降低渲染计算量、利用缓存。虚拟 DOM (VDOM) 虽然提高了性能下限，但并非总是比原生 DOM 操作快。

### 1. 虚拟 DOM 与 Diff 算法

*   **原理**：VDOM 是 JS 对象，Diff 算法比较新旧 VDOM 树的差异，只更新变化的部分到真实 DOM。
*   **Diff 策略 (O(n))**：
    1.  **树层级**：只对同层级节点比较。
    2.  **组件类型**：如果组件类型变了，销毁旧组件，创建新组件（子树不复用）。
    3.  **Key 作用**：同级列表通过 Key 判断节点是移动、新增还是删除。

### 2. 组件渲染优化

*   **React.PureComponent**：
    *   类组件中使用。
    *   自动实现了 `shouldComponentUpdate`，对 `props` 和 `state` 进行**浅比较** (Shallow Comparison)。
    *   注意：如果数据结构是深层嵌套对象， PureComponent 可能失效（引用没变但内容变了）。
*   **React.memo**：
    *   函数组件中使用的高阶组件。
    *   默认对 `props` 进行浅比较。
    *   可以接受第二个参数 `(prevProps, nextProps) => boolean` 进行自定义比较逻辑（类似 `shouldComponentUpdate`）。

### 3. 减少计算量

*   **useMemo**：
    *   缓存“函数的返回值”。
    *   适用于：复杂的数学计算、大数组操作。
*   **useCallback**：
    *   缓存“函数本身”。
    *   适用于：传递给经过 `React.memo` 包裹的子组件，防止父组件每次渲染都生成新的函数引用，导致子组件不必要的重渲染。

### 4. 列表渲染优化

*   **使用唯一的 Key**：避免使用 `index` 作为 Key（除非列表是静态且不会重新排序的）。使用稳定的 ID（如 `item.id`）帮助 React 高效复用 DOM 节点。

### 5. 避免不必要的操作

*   **避免 bind / inline function / inline object**：在 JSX 中直接写 `onClick={() => handleClick()}` 或 `style={{color: 'red'}}` 会在每次渲染时创建新的函数或对象，导致子组件即使使用了 `PureComponent/memo` 也会因为 `props` 浅比较不通过而更新。
    *   解决：使用 `useCallback` 缓存函数，或在类组件中将方法绑定在构造函数中。

## 常见考点

1.  **React.PureComponent 和 Component 的区别？**
    *   **考点**：`PureComponent` 内置了浅比较的 `shouldComponentUpdate`。`Component` 默认每次 `setState` 或父组件更新都会重新渲染。
2.  **Key 的作用是什么？为什么不能用 Index？**
    *   **考点**：Key 是 React 追踪列表元素状态的唯一标识。用 Index 会导致：在列表插入/删除/排序时，React 可能会错误地复用 DOM（导致输入框内容错位、状态混乱），或者降低复用效率。
3.  **useMemo 和 useCallback 的区别？**
    *   **考点**：`useMemo` 缓存值，`useCallback` 缓存函数。`useCallback(fn, deps)` 实际上相当于 `useMemo(() => fn, deps)`。前者用于优化计算，后者用于保持引用稳定以避免子组件重渲染。

### 深化内容

#### 实战案例
在一个复杂的**表格表头筛选**项目中，表头组件父组件更新频率很高（如排序变化）。若直接在列定义中写 `render: (text) => <ExpensiveCell text={text} />`，会导致整个表格行重渲染。实战中通过 `React.memo` 包裹行组件，并配合 `useCallback` 传递 filter 函数，成功将 1000 条数据的渲染时间从 800ms 降低到 100ms 以内，解决了筛选时的页面冻结问题。

#### 代码示例：React.memo 配合自定义比较
```javascript
// 仅在 props.id 变化时才重新渲染，忽略其他 props 变化
const MyComponent = React.memo(function MyComponent({ name, id, data }) {
  return <div>{name}: {data}</div>
}, (prevProps, nextProps) => {
  // 返回 true 表示 props 相等，不渲染；返回 false 表示不等，渲染
  return prevProps.id === nextProps.id;
});
```

#### 对比表格：useMemo vs useCallback vs React.memo

| 特性 | 作用目标 | 适用场景 | 性能代价 |
| :--- | :--- | :--- | :--- |
| **React.memo** | 组件本身 | 避免父组件更新导致子组件无脑重渲染 | 增加props浅比较的开销 (通常可忽略) |
| **useMemo** | 计算结果 (值) | 昂贵计算、大对象引用、依赖项不变时缓存 | 每次渲染需检查依赖数组 + 内存占用 |
| **useCallback** | 函数引用 | 传递给子组件且子组件被 memo 包裹时 | 每次渲染需检查依赖数组 + 内存占用 |
