# 既然提到了拥塞控制，那你能说说说拥塞控制是怎么实现的嘛

### TCP 拥塞控制
**目的**：防止过多的数据注入到网络中，导致网络路由器或链路过载。

**关键变量**：
- `cwnd` (Congestion Window)：拥塞窗口，发送方当前认为网络能承受的数据量。
- `ssthresh` (Slow Start Threshold)：慢启动阈值。

**四种核心算法**：

#### 1. 慢启动
- **机制**：连接刚建立时，`cwnd` 初始为 1（或 2-10 个 MSS），每收到一个 ACK，`cwnd` 加 1（每 RTT 翻倍，指数增长）。
- **目的**：快速试探网络带宽。
- **边界**：当 `cwnd >= ssthresh` 时，进入拥塞避免。

#### 2. 拥塞避免
- **机制**：每经过一个 RTT，`cwnd` 增加 1（线性增长，加法增）。
- **目的**：避免增长过快导致拥塞。

#### 3. 快重传
- **触发**：接收方收到乱序报文，发送重复 ACK。发送方**连续收到 3 个重复 ACK (dupack)**。
- **动作**：立即重传丢失的报文段，而不必等待 RTO 超时。

#### 4. 快恢复
- **触发**：发生快重传时（认为网络拥塞不严重，丢包是个别原因）。
- **动作**：
  1. 将 `ssthresh` 设置为当前 `cwnd` 的一半。
  2. 将 `cwnd` 设置为新的 `ssthresh`（减半）。
  3. 直接进入拥塞避免阶段（**跳过慢启动**）。
- **注意**：如果是超时导致的拥塞（未收到 3 个 dupack），则 `ssthresh` 减半，`cwnd` 重置为 1，重新进入慢启动。

### Keep-Alive 区别
- **HTTP Keep-Alive**：应用层长连接，复用 TCP 连接发送多个请求，减少握手开销。
- **TCP Keepalive**：内核层保活机制，定期发送探测报文检测连接是否断开（通常空闲 2 小时+，默认关闭，用于检测死连接而非保活）。

#### 实战深化

**实战案例**：
在做大文件传输服务时，发现跨洋传输速度极慢。原因是传统 Reno 算法在发生丢包时会大幅降低 `cwnd`。我们在服务器内核开启了 BBR（Bottleneck Bandwidth and Round-trip propagation time）拥塞控制算法，充分利用带宽缓冲区，传输速度提升了 3 倍。注意在高并发短连接场景，BBR 可能会导致网络拥塞加剧，需按需开启。

**代码示例 (Linux 调整 TCP 参数)**：
```bash
# 开启 BBR 拥塞控制算法 (Linux Kernel 4.9+)
echo "net.core.default_qdisc=fq" >> /etc/sysctl.conf
echo "net.ipv4.tcp_congestion_control=bbr" >> /etc/sysctl.conf
sysctl -p

# 验证
sysctl net.ipv4.tcp_congestion_control
# 输出: net.ipv4.tcp_congestion_control = bbr
```

**TCP 拥塞控制算法对比**：

| 算法 | 核心逻辑 | 丢包响应 | 适用场景 | 特点 |
| :--- | :--- | :--- | :--- | :--- |
| **Reno** | AIMD (加法增乘法减) | cwnd 减半，超时置 1 | 传统网络，通用性高 | 安全但保守，带宽利用率低 |
| **Cubic** | 基于 RTT 的立方体函数增长 | cwnd 减半 | 高带宽低延迟网络 | Linux 默认，比 Reno 更激进 |
| **BBR** | 测量带宽和 RTT，不依赖丢包 | 不降低发送速率，重新计算带宽 | 高丢包率、长肥管道 | 不把丢包当拥塞，极速但可能抢占资源 |
| **VEGAS** | 基于 RTT 延迟增加进行拥塞检测 | 提前降低 cwnd | 对延迟敏感的局域网 | 容易被 Reno 挤占带宽，公网较少用 |

## 常见考点
1. **流量控制 vs 拥塞控制**：流量控制（滑动窗口）是为了保护接收方，拥塞控制是为了保护网络。
2. **超时与丢包**：发生超时和发生 3 个重复 ACK 时，拥塞窗口 `cwnd` 的变化有何不同？（前者重置为 1，后者减半）。
3. **延迟 ACK**：接收方通常不立即发送 ACK，而是累积发送，这对拥塞控制有何影响？
4. **BBR 算法**：除了传统的 Reno/Cubic 算法，了解过 BBR（基于带宽和 RTT 的拥塞控制）吗？

## 记忆要点

- 四大核心算法：慢启动探带宽，拥塞避免防过载，快重传连收三个ACK不等超时，快恢复跳过慢启动直接减半
- 阈值与窗口判断：遇3个重复ACK仅cwnd减半，若发生RTO超时则cwnd直接降为1
- 流量vs拥塞：滑动窗口做流量控制是为了保护接收方，拥塞控制是为了防止网络瘫痪

## 延伸：TCP拥塞控制是怎么实现的？

### TCP拥塞控制

TCP拥塞控制可以在网络出现拥塞时动态地调整数据传输的速率，以防止网络过载。TCP拥塞控制的主要机制包括以下几个方面：

1.  **慢启动**：初始阶段，TCP发送方会以较小的发送窗口开始传输数据（通常初始 cwnd 为 1 或 10 MSS）。随着每次成功收到的确认数据，发送方逐渐增加发送窗口的大小（每收到一个 ACK，cwnd 加 1），实现指数级的增长。这有助于在网络刚开始传输时谨慎地逐步增加速率。

2.  **拥塞避免**：一旦达到慢启动阈值，TCP发送方就会进入拥塞避免阶段。在这个阶段，发送方以线性增加的方式增加发送窗口的大小（每收到一个 ACK，cwnd 增加 1/cwnd），有助于控制发送速率，避免过快导致拥塞。

3.  **快速重传**：如果发送方连续收到3个重复的ACK，它会认为发生了数据包的丢失，并快速重传未确认的数据包，而不必等待超时。这通常意味着中间路由器有缓存能力，网络未完全瘫痪。

4.  **快速恢复**：在发生快速重传后，TCP进入快速恢复阶段。发送方将慢启动阈值设置为当前窗口的一半，并将拥塞窗口大小设置为该值（ssthresh + 3），然后进入拥塞避免阶段。这样避免了将拥塞窗口降为 1 导致的剧烈性能抖动，有助于更快地从拥塞中恢复。

**状态流转示意图：**

```
      慢启动
    /-> [指数增长] 
   /     (cwnd < ssthresh) 
  /         |
 |          v
 |      拥塞避免
 |    [线性增长 (cwnd += 1/cwnd)]
 |         |
 |    +----+----+
 |    | 丢包检测 |
 |    +----+----+
 |    |         |
 |  [3个ACK] [超时 RTO]
 |    |         |
 |    v         v
 | 快速恢复   重置 cwnd=1
 | [调整 ssthresh]  |
 |    |         |
 |    +----+----+
 |         |
 v         v
    拥塞避免
```

**## 常见考点**
1.  **慢启动的“慢”是指什么？**：是指初始窗口小，但增长速度是指数级的（翻倍增长），实际上并不慢。
2.  **为什么需要快速重传和快速恢复？**：为了避免超时重传带来的长时间等待（RTO 通常较大），通过 3 个重复 ACK 快速探测丢包并恢复，减少网络空闲时间。
3.  **cwnd 和 rwnd 的区别？**：cwnd 是拥塞窗口（发送方根据网络状况调整），rwnd 是接收窗口（接收方根据处理能力告知发送方），实际发送窗口 `swnd = min(cwnd, rwnd)`。

#### 4. 实战深化
*   **实战案例**：在弱网环境（如跨国专线）下传输大文件，TCP 拥塞控制导致带宽利用率上不去。通过开启 `BBR` (Bottleneck Bandwidth and RTT) 拥塞控制算法（Linux 内核 4.9+），吞吐量相比传统的 `Cubic` 算法提升了数倍，因为它不仅关注丢包，还测量真实带宽和最小 RTT。

*   **代码示例 (Linux 调整 BBR)**：
    ```bash
    # 开启 TCP BBR 拥塞控制算法 (sysctl.conf)
    net.core.default_qdisc=fq
    net.ipv4.tcp_congestion_control=bbr
    
    # 验证是否生效
    sysctl net.ipv4.tcp_congestion_control
    # 输出: net.ipv4.tcp_congestion_control = bbr
    ```

## 记忆要点

- 慢启动：cwnd初始为1，每收一个ACK翻倍增长，直至达到阈值ssthresh。
- 拥塞避免：达阈值后由指数转为线性增长，每次ACK仅加1/cwnd。
- 拥塞判断：超时重传代表严重拥塞，直接重置cwnd为1；3个重复ACK代表轻微。
- 快速恢复：遇3个重复ACK时阈值减半，cwnd设为减半后的阈值并线性恢复。
