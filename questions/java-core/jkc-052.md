# Gossip节点的通信方式及收敛性

Gossip 协议是 Cassandra 中用于节点间状态同步和发现的去中心化协议，灵感来源于“八卦传播”。

### 通信方式
两个节点（A、B）之间主要有三种数据交换方式：
1.  **Push（推）**：A 将自己的数据推送给 B，B 更新本地旧数据。
2.  **Pull（拉）**：A 仅将数据摘要（Key/Version）发给 B，B 将本地比 A 新的数据回传给 A。
3.  **Push-Pull（推拉）**：A 发送摘要给 B，B 回传新数据给 A，随后 A 再将本地新数据推送给 B。这是 Cassandra 2.0+ 的默认模式。

| 模式 | 通信次数 | 收敛速度 | 适用场景 |
| :--- | :--- | :--- | :--- |
| Push | 1 | 慢 | 节点加入、状态快速广播 |
| Pull | 2 | 中 | 节点启动时同步最新状态 |
| Push-Pull | 3 | 最快 | 默认周期性心跳（Anti-Entropy） |

### 收敛性与 Seed List
- **最终一致性**：Gossip 协议不要求强一致性，节点随机与其他节点通信（每秒 1 次，每次随机选 1-3 个活节点），经过一定时间后，集群状态会达成一致（收敛）。
- **数学原理**：收敛速度遵循指数级增长（$O(\log N)$），通常在几秒内即可传播到全网。
- **Seed List（种子节点）**：为了防止集群分裂（如网络分区导致形成两个孤岛），集群配置中需要指定相同的 Seed List。新节点启动时，首先联系 Seed List 中的节点以获取初始集群状态。种子节点本身不是单点故障，仅起引导作用。

### 故障检测
Gossip 协议结合 **Phi Accrual Failure Detector** 算法来判断节点是否宕机，而不是简单的超时机制。它基于历史交互记录计算一个“怀疑分数”，适应网络抖动更平滑。

### 实战案例
在一次扩容操作中，由于新节点的 `cassandra.yaml` 中 Seed List 配置错误（指向了旧集群的种子），导致新节点未能正确加入集群，反而形成了“脑裂”。修复方法是将 Seed List 统一为跨机架的 3 个核心节点，并重启节点。

## 常见考点
- **Seed 节点数量**：建议每个 DC 配置多少个？（通常 3 个，太少导致引导脆弱，太多增加 Gossip 开销）。
- **Split Brain**：为什么配置不同的 Seed List 会导致集群分裂？（Gossip 传播域隔离）。
- **消息内容**：Gossip 主要交换哪些数据？（节点状态、Schema 版本、负载信息等）。
