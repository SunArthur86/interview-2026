# HashMap JDK 8 的底层原理？为什么引入红黑树？扩容机制是怎样的？

【HashMap JDK8 结构】数组 + 链表 + 红黑树。

**## 增强底层原理架构图**

【put 流程详解】
1. **扰动函数**：`hash(key) = (h = key.hashCode()) ^ (h >>> 16)`。让高16位参与异或，使得低位特征更加随机，减少碰撞。
2. **定位桶**：`index = hash & (n-1)`。要求 n 必须是 2 的幂，此时 `(n-1)` 的二进制全为 1，位运算效果等同于取模，但效率极高。
3. **插入逻辑**：
   - 桶为空（无冲突）：直接插入。
   - 桶非空（冲突）：
     - 如果是 Node（链表结构）：遍历，Key 相同覆盖，不同则**尾插法**（JDK 7 是头插）追加。
     - 如果是 TreeNode（红黑树结构）：调用树插入方法。
4. **树化与退化**：
   - 树化条件：链表长度 >= 8 **且** 数组长度 >= 64。若数组长度 < 64，则优先扩容而非树化（因为扩容也能分散冲突）。
   - 退化条件：红黑树节点数 <= 6 时，退化为链表。
5. **扩容触发**：`size > threshold`（threshold = capacity * loadFactor，默认 16 * 0.75 = 12）。

【为什么引入红黑树？】
- **痛点**：在 hash 碰撞极其严重时，链表查询的时间复杂度为 O(n)，性能急剧下降。
- **解决**：红黑树是一种自平衡二叉查找树，查询时间复杂度稳定在 O(logn)。
- **阈值 8 的依据**：根据泊松分布统计，链表长度达到 8 的概率极低（0.00000006）。此时说明 hash 碰撞非常严重，必须优化。

【扩容机制：JDK 8 的性能优化】
- **容量翻倍**：`newCap = oldCap << 1`。
- **元素迁移优化（关键点）**：
  由于 n 始终为 2 的幂，扩容后元素的位置要么在原索引，要么在 `原索引 + oldCap`。
  - 判断方式：`(e.hash & oldCap) == 0` → 原索引（高位为 0）
  - 否则 → 原索引 + oldCap（高位为 1）
  - **意义**：避免了重新计算 hash，且避免了像 JDK 7 那样在扩容时链表逆序导致的死循环问题（尽管 JDK 8 改尾插法已从根源解决死循环，但该优化依然提升了迁移效率）。

【loadFactor=0.75 的原因】
- **时间与空间权衡**：
  - 太大（如 1.0）：空间利用率高，但冲突频繁，链表长，查询慢。
  - 太小（如 0.5）：冲突少，查询快，但数组扩容频繁，空间浪费大。
- **泊松分布**：0.75 是在统计学上达到平均查找次数与空间占用平衡的最佳经验值。

## 延伸：HashMap（数组+链表+红黑树）是什么？

> 合并自 `core-154`（相似度 75%）

HashMap 是基于哈希表的 Map 接口实现，采用 **数组 + 链表 + 红黑树** 的组合结构。

### 数据结构演变
1. **数组**：HashMap 的主体，用于存储数据。根据 Key 的 Hash 值计算数组下标。
2. **链表**（JDK 1.7 及 JDK 1.8 的默认）：当不同的 Key 计算出相同的数组下标时（哈希冲突），通过链表将这些节点串联起来。
3. **红黑树**（JDK 1.8 引入）：为了解决链表过长导致查询效率降低的问题，当链表长度超过 **8** 且数组长度大于等于 **64** 时，链表会转换为红黑树。红黑树的时间复杂度为 O(logN)，查询效率远高于链表的 O(N)。

### 核心参数
- **initialCapacity**：初始容量，默认 16。必须为 2 的幂次方。
- **loadFactor**：负载因子，默认 0.75。用于计算扩容阈值（`capacity * loadFactor`）。这是时间与空间成本的折中。
- **threshold**：扩容阈值，当元素个数超过此值时触发扩容。
- **treeify_threshold**：树化阈值，默认 8。
- **untreeify_threshold**：退化阈值，默认 6。

**HashMap 内部结构图：**
```text
      ┌─────────────────────────────────────────┐
      │         HashMap (Node[] table)          │
      ├─────────────────────────────────────────┤
      │  Index 0  │  Index 1  │  Index 2  │ ... │
      ├───────────┼───────────┼───────────┼─────┤
      │   Node    │   Node    │   TreeNode│ ... │
      │    │      │    │      │    │      │     │
      │    ▼      │    ▼      │    ▼      │     │
      │  [A] -> B │  [C]      │  [D] - E  │     │
      │  (链表)   │  (无冲突) │  (红黑树) │     │
      └───────────┴───────────┴───────────┴─────┘
```

### 基本操作流程
1. **计算 Hash**：
   - `h = key.hashCode()`
   - `h = (h == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16)` 
   - **原理**：高 16 位异或低 16 位（扰动函数）。为了在数组长度较小时，让高位的特征参与到下标计算中，减少哈希冲突。
2. **定位索引**：`index = h & (n - 1)`（n 为数组长度）。
   - **原理**：位运算替代取模运算，要求 n 必须是 2 的幂次方，这样 `n-1` 的二进制形式全为 1（如 15=1111），能保证散列均匀。
3. **处理冲突**：
   - 如果无冲突，直接插入。
   - 如果有冲突，遍历链表或红黑树：
     - Key 相同（hash 相同且 equals 返回 true），则覆盖 Value。
     - Key 不同，则追加到尾部（JDK 1.8 采用尾插法）。

### 代码示例 (Java 扰动函数模拟)
```java
static final int hash(Object key) {
    int h;
    // key.hashCode() 扰动：高16位异或低16位
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

### 实战案例
在排查一次慢查询时，发现由于业务对象重写的 `hashCode()` 方法返回值固定为 1，导致所有数据落入 HashMap 同一槽位形成超长链表，查询退化为 O(N)。修复 hashCode 逻辑后性能恢复正常。

## 常见考点
1. **为什么负载因子默认是 0.75？**
   - 太小（如 0.5）：数组利用率低，扩容频繁，浪费内存。
   - 太大（如 1.0）：数组利用率高，但哈希冲突加剧，链表过长，导致查询效率下降（O(n)）。
   - 0.75 是根据泊松分布统计得出的经验值，此时空间和时间成本最为均衡。
2. **为什么链表转红黑树的阈值是 8，而退回来是 6？**
   - 如果 7 和 8 互换，频繁在链表和树之间转换会消耗性能。
   - 根据 Poisson 分布，哈希冲突概率为 0.00000006 的极端情况下才会达到 8，大部分情况链表长度都很短。6 和 7 之间设置了缓冲带，避免频繁转换。
3. **为什么 HashMap 容量要是 2 的幂次方？**
   - 为了使 `index = hash & (n - 1)` 位运算等效于 `hash % n`，同时保证索引分布均匀。如果 n 不是 2 的幂，`n-1` 的二进制低位不全为 1，会导致哈希冲突增加。

## 记忆要点

- 结构骨架：主体数组+链表解决哈希冲突，JDK1.8引入红黑树解决长链表O(N)查询慢问题
- 核心参数：初始容量16，负载因子0.75，扩容阈值=两者乘积
- 树化门槛：链表长度>8 且 数组长度>=64 才转红黑树，长度降至6则退化为链表
- 高效寻址：因为容量必为2的幂次方，所以能将hash%n替换为高效的hash&(n-1)位运算
- Hash扰动：高16位异或低16位，让高位特征参与运算减少哈希冲突
