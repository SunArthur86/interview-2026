# HashMap JDK 8 的底层原理？为什么引入红黑树？扩容机制是怎样的？

【HashMap JDK8 结构】数组 + 链表 + 红黑树。

【put 流程详解】
1. **扰动函数**：`hash(key) = (h = key.hashCode()) ^ (h >>> 16)`。让高16位参与异或，使得低位特征更加随机，减少碰撞。
2. **定位桶**：`index = hash & (n-1)`。要求 n 必须是 2 的幂，此时 `(n-1)` 的二进制全为 1，位运算效果等同于取模，但效率极高。
3. **插入逻辑**：
   - 桶为空（无冲突）：直接插入。
   - 桶非空（冲突）：
     - 如果是 Node（链表结构）：遍历，Key 相同覆盖，不同则**尾插法**（JDK 7 是头插）追加。
     - 如果是 TreeNode（红黑树结构）：调用树插入方法。
4. **树化与退化**：
   - 树化条件：链表长度 >= 8 **且** 数组长度 >= 64。若数组长度 < 64，则优先扩容而非树化（因为扩容也能分散冲突）。
   - 退化条件：红黑树节点数 <= 6 时，退化为链表。
5. **扩容触发**：`size > threshold`（threshold = capacity * loadFactor，默认 16 * 0.75 = 12）。

【为什么引入红黑树？】
- **痛点**：在 hash 碰撞极其严重时，链表查询的时间复杂度为 O(n)，性能急剧下降。
- **解决**：红黑树是一种自平衡二叉查找树，查询时间复杂度稳定在 O(logn)。
- **阈值 8 的依据**：根据泊松分布统计，链表长度达到 8 的概率极低（0.00000006）。此时说明 hash 碰撞非常严重，必须优化。

【扩容机制：JDK 8 的性能优化】
- **容量翻倍**：`newCap = oldCap << 1`。
- **元素迁移优化（关键点）**：
  由于 n 始终为 2 的幂，扩容后元素的位置要么在原索引，要么在 `原索引 + oldCap`。
  - 判断方式：`(e.hash & oldCap) == 0` → 原索引（高位为 0）
  - 否则 → 原索引 + oldCap（高位为 1）
  - **意义**：避免了重新计算 hash，且避免了像 JDK 7 那样在扩容时链表逆序导致的死循环问题（尽管 JDK 8 改尾插法已从根源解决死循环，但该优化依然提升了迁移效率）。

【loadFactor=0.75 的原因】
- **时间与空间权衡**：
  - 太大（如 1.0）：空间利用率高，但冲突频繁，链表长，查询慢。
  - 太小（如 0.5）：冲突少，查询快，但数组扩容频繁，空间浪费大。
- **泊松分布**：0.75 是在统计学上达到平均查找次数与空间占用平衡的最佳经验值。

**## 常见考点**
1. **为什么 JDK 8 改用尾插法？**
   JDK 7 头插法扩容时，链表会逆序，多线程并发扩容容易导致链表成环。

#### 深化实战补充

1. **实战案例（Map 并发扩容死锁）**：
   在旧系统维护中，遇到由于 JDK 7 HashMap 多线程 put 导致 CPU 100% 的问题。原因是扩容时链表形成环形引用（A.next=B, B.next=A）。虽然 JDK 8 修复了此问题，但在高并发场景下仍推荐使用 `ConcurrentHashMap`，因为 HashMap 依然存在数据覆盖丢失的风险。

2. **代码示例（扩容下标计算核心逻辑）**：
   ```java
   // JDK 8 HashMap 扩容迁移核心源码简化
   if ((e.hash & oldCap) == 0) {
       // 低位索引不变，放入 lo 链表
       if (loTail == null) loHead = e;
       else loTail.next = e;
       loTail = e;
   } else {
       // 高位索引 = 原索引 + oldCap，放入 hi 链表
       if (hiTail == null) hiHead = e;
       else hiTail.next = e;
       hiTail = e;
   }
   ```

3. **对比表格（JDK 7 vs JDK 8 HashMap）**
   | 特性 | JDK 7 HashMap | JDK 8 HashMap |
   | :--- | :--- | :--- |
   | **底层结构** | 数组 + 链表 | 数组 + 链表 + 红黑树 |
   | **插入方式** | 头插法 | 尾插法 |
   | **扩容后顺序** | 逆序（易死循环） | 保持原序（解决死循环） |
   | **哈希扰动** | 4次位运算 + 5次异或（复杂） | 1次位运算 + 1次异或（高效） |
   | **节点类型** | Entry 类 | Node 类（TreeNode 继承 Node） |
