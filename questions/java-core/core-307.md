# 观察者模式的原理是什么？Java中如何实现？

**观察者模式**

### 原理
属于行为型模式的一种，它定义了一种一对多的依赖关系，让多个观察者对象同时监听某一个主题对象。这个主题对象在状态变化时，会通知所有的观察者对象，使他们能够自动更新自己。

### 架构图
```text
      ┌──────────────┐         ┌──────────────┐
      │   Subject    │         │   Observer   │
      │  (主题/被观察)│         │   (观察者)    │
      └──────┬───────┘         └──────┬───────┘
             │    1. register()        │
             │ ◄──────────────────────┤
             │                         │
             │    2. setState()        │
             │ (状态改变)              │
             │                         │
             │    3. notifyObservers() │
             ├───────────────────────>│
             │                         │
             │    4. update(data)      │
             ├───────────────────────>│
             │                         │
             ▼                         ▼
┌─────────────────────┐   ┌─────────────────────┐
│  ConcreteSubject    │   │  ConcreteObserver   │
│  - state            │   │  - update()         │
│  - observers[]      │   │  - 业务逻辑处理     │
└─────────────────────┘   └─────────────────────┘
```

### 主要构成
- **Subject（主题）**：主题是被观察的对象，它包含了一组观察者对象，并提供了添加、删除和通知观察者的方法。主题通常有一个状态，当状态改变时，通知所有观察者。
- **Observer（观察者）**：观察者是依赖于主题的对象，当主题的状态发生改变时，观察者得到通知并进行相应的更新。观察者的具体实现类需要实现更新的方法。
- **ConcreteSubject（具体主题）**：具体主题是主题的具体实现类，它维护了一个状态，并在状态改变时通知观察者。通常内部使用一个 List 集合来存储观察者。
- **ConcreteObserver（具体观察者）**：具体观察者是观察者的具体实现类，实现了更新的方法，以便在接收到通知时进行相应的处理。

### 案例
Carl哥的微信公众号是被观察者，微信用户是观察者。有多个微信用户关注了Carl哥的公众号，当Carl哥的公众号更新订阅时就会通知这些订阅了的微信用户。

### 优点
解除耦合，让耦合的双方都依赖于抽象，从而使得各自的变换都不会影响另一边的变换。

### Java 实现细节
1. **java.util.Observable 类**：作为主题的基类（已过时，不推荐使用，因为它是类继承，无法多继承，且线程安全控制不够灵活）。
2. **java.util.Observer 接口**：仅包含 `update(Observable o, Object arg)` 方法。
3. **现代实现**：通常使用 `java.util.Observable` 的替代方案，如自定义接口+事件总线，或者 JDK 9+ 推荐的 `Flow` API（响应式流）。

### 实战案例
- **跨系统通知**：在订单支付成功后，需同时触发积分服务增加积分、物流服务生成运单、短信服务发送通知。引入观察者模式（或消息队列如 MQ）后，支付服务只需发布"支付成功"事件，无需感知下游依赖，极大地降低了系统耦合度。
- **内存泄漏踩坑**：在 Android 开发中，Activity 注册了某个 EventBus 作为观察者，但在 `onDestroy` 中忘记反注册。导致 Activity 实例被静态持有的 EventBus 引用，无法回收，造成严重的内存泄漏（OOM）。**教训**：务必在生命周期结束处成对调用 `unregister`。

### 代码示例 (自定义接口实现)
```java
// 定义观察者接口
interface Observer {
    void update(String message);
}

// 定义主题接口
interface Subject {
    void registerObserver(Observer o);
    void removeObserver(Observer o);
    void notifyObservers();
}

// 具体主题
class MessageServer implements Subject {
    private List<Observer> observers = new ArrayList<>();
    private String message;

    @Override
    public void registerObserver(Observer o) {
        observers.add(o);
    }

    @Override
    public void notifyObservers() {
        for (Observer o : observers) {
            o.update(message);
        }
    }
}
```

#### 对比表格：推模型 vs 拉模型
| 特性 | 推模型 | 拉模型 |
| :--- | :--- | :--- |
| **通知方式** | 主题主动传递参数 `update(data)` | 主题仅传递自身引用 `update(this)` |
| **观察者动作** | 直接处理数据 | 需调用主题的 getter 获取数据 |
| **耦合度** | 较高 (观察者需知道数据结构) | 较低 (观察者只需持有主题引用) |
| **适用场景** | 数据明确，无需频繁变更 | 数据量较大，观察者按需获取 |

## 常见考点
1. **内存泄漏问题**：如果观察者没有及时取消注册，可能会导致对象无法被 GC 回收。
2. **线程安全**：当多个线程同时注册和通知时，需要使用 `CopyOnWriteArrayList` 或 `synchronized` 块来保证 `observers` 集合的线程安全。
3. **执行顺序**：如果观察者的执行顺序有依赖（如 A 必须在 B 之前执行），简单的 List 顺序遍历是可控的，但在异步场景（如 Reactor/MQ）中顺序无法保证，设计时需避免观察者间的逻辑依赖。
