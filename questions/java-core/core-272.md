# 什么是逃逸分析？

**逃逸分析**

分析对象动态作用域，当一个对象在方法里面被定义后，它可能被外部方法所引用，例如作为调用参数传递到其他方法中，这种称为**方法逃逸**；甚至还有可能被外部线程访问到，譬如赋值给可以在其他线程中访问的实例变量，这种称为**线程逃逸**。从不逃逸、方法逃逸到线程逃逸，称为对象由低到高的不同逃逸程度。

逃逸分析与类型继承关系分析一样，并不是直接优化代码的手段，而是为其他优化措施提供依据的分析技术。如果一个对象不会逃到方法和线程之外，即其他方法和线程无法通过任何手段访问到这个对象，或者逃逸程度低（只是方法逃逸）则可以为这个对象实现不同程度的优化：

*   **栈上分配**
如果确定对象不会逃逸出线程，即对象只在本线程内使用，那么可以将对象分配在栈上而非堆上。栈上分配的优点是随着方法栈帧的出栈，内存自动回收，无需垃圾回收器（GC）介入，减轻GC压力。

*   **标量替换**
若对象不会逃逸，且可以被拆解，JVM可能不会创建该对象，而是将其成员变量分解为独立的局部变量（标量）。这样原本需要在堆上分配连续内存的操作，变成了在栈或寄存器中分配几个基本类型变量。

*   **同步消除**
如果确定对象只能在一个线程被访问，那么该对象上的同步锁（如 synchronized）是多余的，编译器会在编译阶段去除这些同步操作。

### 实战案例
在早期的 JDK 6 中，逃逸分析默认是关闭的，且实现尚不成熟，有时开启后反而导致性能下降（因为分析成本高于分配收益）。但在现代 JDK（如 JDK 8u 之后）中，逃逸分析通常默认开启。**踩坑经验**：曾遇到一个高频调用的日志工具类，其中定义了 `private static final Date formatterDate = new Date();`，本以为非逃逸会被优化，结果因在多线程环境下被引用逃逸，导致严重的伪共享问题，后来改用 `ThreadLocal` 或 `java.time` 的不可变类才解决。另一个场景是，在 `for` 循环中 `new` 一个仅用于内部计算的辅助对象，若参数开启合理，JIT 会直接将其拆解为局部变量，完全消除对象分配开销。

### 代码示例
```java
// Java 伪代码：展示标量替换与同步消除的场景
public void test() {
    // 对象 alloc 没有逃逸出 test 方法
    // JIT 编译后，可能不会创建 Point 对象，而是直接使用 int x, y
    // 且 synchronized 块会被直接移除
    synchronized(new Object()) { 
        int x = 10;
        int y = 20;
        System.out.println(x + y);
    }
}
// 编译优化后等价于：
// public void test() {
//     int x = 10;
//     int y = 20;
//     System.out.println(x + y); 
// }
```

### 对比分析
| 优化手段 | 前置条件 (逃逸状态) | 优化效果 | 局限性/注意事项 |
| :--- | :--- | :--- | :--- |
| **栈上分配** | 不逃逸 (线程内) | 消除堆内存分配，无 GC 扫描压力 | HotSpot 中并未真正实现“栈上分配”，通常通过“标量替换”实现类似效果（对象不创建） |
| **标量替换** | 不逃逸 | 彻底消除对象创建，变量直接入栈/寄存器 | 对象结构复杂或字段过多时，替换收益可能低于分析成本 |
| **同步消除** | 不逃逸 (线程内) | 移除锁指令 (monitorenter/exit)，消除阻塞开销 | 必须确保锁对象本身不会逃逸，否则多线程下会导致数据错误 |
| **标量替换 (部分)** | 方法逃逸 (但未线程逃逸) | 可能优化部分字段分配 | 无法完全消除对象，仍需在堆上分配 |

## 记忆要点

- 本质定义：分析对象动态作用域，按由低到高分不逃逸、方法与线程逃逸。
- 标量替换：对象不逃逸且可拆解，打散为基本类型直接在栈寄存器分配。
- 同步消除：确认对象仅单线程访问，编译器自动擦除冗余的同步锁。
- 栈上分配：对象不逃逸仅线程内用，随栈帧弹出自动回收，免GC扫描。
