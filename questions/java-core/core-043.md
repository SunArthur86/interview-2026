# Keep-Alive是什么？

### Keep-Alive 是什么？
**Keep-Alive** 是一种 HTTP 机制，也称为**HTTP 长连接**。它允许在一次 TCP 连接上进行多次 HTTP 请求和响应，而不是每次请求都建立一个新的 TCP 连接。

### 1. HTTP/1.0 中的 Keep-Alive
- HTTP/1.0 默认使用**短连接**（每次请求都需要建立和断开 TCP 连接）。
- 如果想保持长连接，需要在请求头中显式添加 `Connection: Keep-Alive`。

### 2. HTTP/1.1 中的 Keep-Alive
- HTTP/1.1 **默认开启**长连接（除非显式设置为 `Connection: close`）。
- 这意味着一个 TCP 连接可以处理多个 HTTP 请求，减少了建立和断开连接的握手开销（RTT）。

### 3. 工作原理与配置
- 客户端在请求头中包含 `Connection: Keep-Alive`，表示希望保持连接。
- 服务端如果同意，响应头中也会包含 `Connection: Keep-Alive`，并可能附带 `Keep-Alive: timeout=5, max=100`（表示空闲 5 秒后断开，或最多处理 100 个请求后断开）。

### 4. 数据流程对比
```text
短连接:
┌─────────┐         SYN/ACK         ┌─────────┐
│ Client  │ <---------------------> │ Server  │
└─────────┘                           └─────────┘
      |  Request 1  -----------------> |
      | <----------------- Response 1 |
      | FIN/ACK  --------------------> |
      |                               |
      |         SYN/ACK               |
      | <--------------------->       |
      |  Request 2  -----------------> |
      | <----------------- Response 2 |
      | FIN/ACK  --------------------> |

长连接:
┌─────────┐         SYN/ACK         ┌─────────┐
│ Client  │ <---------------------> │ Server  │
└─────────┘   (建立一次连接)        └─────────┘
      |  Request 1  -----------------> |
      | <----------------- Response 1 |
      |  Request 2  -----------------> | (复用连接)
      | <----------------- Response 2 |
      |  Request 3  -----------------> |
      | <----------------- Response 3 |
      | FIN/ACK  (超时/结束) --------> |
```

### 5. 优缺点
- **优点**：减少 TCP 握手/挥手延迟，降低服务器负载，提升页面加载速度，允许 HTTP/1.1 管道化（非必须但可配合）。
- **缺点**：长时间保持连接会占用服务器资源（文件描述符、内存），需要合理配置超时时间，可能发生“队头阻塞”（HOL）问题。

### 实战案例
在高并发 API 网关上线初期，因 `keepalive_timeout` 设置过长（60s），导致服务端积压了大量处于 `TIME_WAIT` 或 `ESTABLISHED` 状态的空闲连接，最终耗尽了服务器的文件描述符，导致新连接被拒绝。调整超时为 5s 并开启 Nginx 的 `keepalive_requests` 限制后，系统吞吐量恢复了 30%。

### 代码示例 (Go 客户端配置)

```go
// Go HTTP 客户端开启长连接并配置 Transport
transport := &http.Transport{
    // MaxIdleConns 控制最大空闲连接数
    MaxIdleConns:        100,
    MaxIdleConnsPerHost: 10, // 每个主机的最大空闲连接数
    // 空闲连接超时时间
    IdleConnTimeout:     90 * time.Second,
}
client := &http.Client{
    Transport: transport,
    Timeout:   30 * time.Second,
}
// 此 client 会自动复用 TCP 连接
```

### 对比表格

| 特性 | 短连接 | 长连接 |
| :--- | :--- | :--- |
| **连接建立** | 每次请求都新建 TCP | 一次 TCP，多次请求 |
| **RTT 延迟** | 高（每次都有握手开销） | 低（仅首部有握手开销） |
| **CPU/负载** | 高（频繁建连断连） | 低（复用连接） |
| **资源占用** | 低（无持续占用） | 较高（占用文件描述符） |
| **适用场景** | 并发量低或无后续请求 | 高并发 API、Web 服务 |

## 常见考点
1. **HTTP 管道化**：Keep-Alive 是否支持管道化？为什么 HTTP/1.1 管道化没普及？（答案：支持，但因服务器处理顺序和队头阻塞问题导致实现复杂，Chrome/Firefox 早已默认禁用）。
2. **服务端配置**：Nginx/Apache 中如何配置 Keep-Alive 超时？过短或过长有什么影响？（答案：`keepalive_timeout`；太短导致频繁握手，太长占用连接）。
3. **与 TCP Keep-Alive 的区别**：HTTP Keep-Alive 和 TCP 层的 SO_KEEPALIVE 有什么不同？（答案：HTTP Keep-Alive 是应用层复用连接，TCP Keep-Alive 是传输层心跳检测，用于死连接排查）。
