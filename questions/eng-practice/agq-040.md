# 流式输出是怎么实现的

**Situation：** 非流式模式下,用户需要等待 LLM 完全生成后才能看到回答,感知延迟长.

**Task：** 实现全链路流式输出,优化用户感知延迟.

**Action：** 
1. **TTFT(Time To First Token)优化:**
   - **目标：** TTFT < 500ms.
   - **原理：** 意图识别 + 检索在 LLM 开始生成前完成，且使用 Prompt Prefetch 技术提前将上下文传输至 GPU 显存。
   - **数据：** LLM 的第一个 token 通常在 200-500ms 内返回.

2. **TBT(Time Between Tokens)优化:**
   - **模型选择：** 选择输出速度快的模型(GPT-4o-mini 的 TBT 约 10ms/token).
   - **架构调整：** 避免后端缓冲累积 token 再发送，采用“即产即发”模式，减少后端到前端的网络延迟。

3. **流式中的工具调用处理:**
   - **策略：** 检测到 Action 开始时,发送状态消息"正在查询...".
   - **执行：** 工具执行完成后,继续流式输出 Final Answer.
   - **前端：** 展示 loading 动画和工具调用状态.

**流式处理数据流图：**
```text
[LLM Provider]
    │
    │ SSE (Server-Sent Events)
    │ stream: "token1", "token2", ...
    ▼
[Backend Gateway]
    │
    │ 1. Bufferize (Optional for buffering small chunks)
    │ 2. Forward to Client (Passthrough)
    ▼
[Client Frontend]
    │
    ▼
EventSource.onmessage = (e) => {
    if (e.data === "[TOOL_CALL_START]") {
        showLoading("正在查询资料...");
    } else {
        appendToChatWindow(e.data); // 逐字渲染
    }
}
```

**Result：** 
- TTFT P50 = 450ms, P99 = 800ms.
- 用户感知的响应速度提升 5x.
- 用户满意度提升 22%.

### 深化内容

**实战案例：**
在使用流式输出进行长文本总结时，曾出现 Markdown 格式错乱问题。原因是 LLM 输出表格内容时被切片发送。解决方案是引入 **前端 Markdown 增量解析器**，支持不闭合标签的临时渲染，并配合后端在行尾添加零宽空格确保字符边界正确。

**代码示例：**
```python
from fastapi.responses import StreamingResponse

async def generate_stream(query: str):
    # 模拟从 LLM 获取流式数据
    async for chunk in llm_client.stream(query):
        # 关键：不进行 accumulate，直接 yield
        yield f"data: {chunk.json()}\n\n"

return StreamingResponse(
    generate_stream(user_query),
    media_type="text/event-stream",
    headers={"Cache-Control": "no-cache", "Connection": "keep-alive"}
)
```

**对比表格：**

| 维度 | SSE (Server-Sent Events) | WebSocket |
| :--- | :--- | :--- |
| **通信方向** | 单向 | 双向 |
| **协议** | 基于 HTTP/1.1 | 独立协议 (ws://) |
| **自动重连** | 浏览器原生支持 | 需手动实现 (心跳机制) |
| **防火墙穿透** | 好 (标准 HTTP 端口) | 一般 (部分代理会拦截 Upgrade 头) |
| **适用场景** | AI 生成、推送通知 | 实时聊天、协同编辑 |

## 常见考点
1. **流式输出下的“思维链”隐藏处理？** 
   *追问点*：如果 LLM 输出了 ` 
  
 
  
` 标签或内部的推理步骤，后端如何识别并拦截，确保前端用户只看到最终答案（需解析 SSE 流并进行正则匹配过滤）。
2. **网络中断导致流式中断怎么办？** 
   *追问点*：是否有断点续传机制？通常答案是幂等的，前端需自动重试或提示用户重新提问，后端需保证请求的幂等性。
3. **SSE vs WebSocket 的选择？** 
   *追问点*：为何选择 SSE（Server-Sent Events）而不是 WebSocket？SSE 是单向传输（Server -> Client），更适合这种纯生成场景，且基于 HTTP 协议，便于穿透防火墙和负载均衡。
