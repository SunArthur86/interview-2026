# Young GC什么时候触发？Full GC什么时候触发？

**Young GC 触发条件**：Eden区空间不足时自动触发，非常频繁。

**Full GC 触发条件**：
①调用System.gc()（只是建议，不保证立即执行）
②老年代空间不足
③空间分配担保失败：Minor GC前检查老年代可用空间 < 新生代历次晋升平均值
④JDK 1.7前永久代空间不足
⑤CMS的Concurrent Mode Failure：并发回收时老年代满
⑥Evacuation Failure：G1没有足够Region存放存活对象

**触发机制深度解析**：

**Young GC (Minor GC) 触发流程：**
```text
新对象分配请求
      │
      ├─> Eden 空间足够? ──Yes──> 分配成功
      │
      No (Eden 满)
      │
      ├─> 触发 Minor GC
      │       │
      │       ├─> 清理 Eden + Survivor
      │       └─> 晋升存活对象到 Old Gen
      │
      └─> GC 后 Eden 仍放不下? 
              │
              Yes ──> 触发担保机制 (可能导致 Full GC)
```

**Full GC 触发细节补充**：
1. **显式调用**：`System.gc()`，可通过 `-XX:+DisableExplicitGC` 禁用（例如 RMI 使用时）。
2. **老年代空间不足**：
   - 大对象直接进入老年代，但老年代放不下。
   - 长期存活对象晋升，老年代放不下。
3. **Metaspace (元空间) 溢出**：JDK 8+ 中，如果加载的类太多，元空间达到 `MaxMetaspaceSize`，也会触发 Full GC。
4. **CMS/G1 特有失败**：
   - **CMS Concurrent Mode Failure**：CMS 回收慢，对象分配快，老年代在回收过程中填满，触发 Full GC (Serial Old)。
   - **G1 Allocation Failure / Evacuation Failure**：复制存活对象时，没有空闲 Region 可用，触发 Full GC (Serial Old)。
5. **Heap Dump / 性能监控工具**：如 jmap -histo:live 可能触发 Full GC 来统计存活对象。

## 常见考点
1. **System.gc() 真的会立即触发 Full GC 吗？** 
   不一定。`System.gc()` 只是给 JVM 一个建议。如果开启了 `-XX:+DisableExplicitGC`，该调用会被忽略。如果未开启，大多数 JVM 会尝试触发 Full GC（或 Concurrent GC cycle），但不保证立即发生。
2. **为什么 JDK 8 的 Metaspace 溢出也会导致 Full GC？** 
   Full GC 除了清理堆，也会清理方法区（元空间）。JVM 尝试通过 Full GC 卸载无用的类来释放元空间。如果 Full GC 后依然不够，才抛出 `OutOfMemoryError: Metaspace`。
3. **CMS 的 Concurrent Mode Failure 和 Promotion Failed 有什么区别？** 
   两者常伴随发生。**Promotion Failed** 指的是 Minor GC 时，Survivor 区放不下，需要晋升到老年代，但老年代也没有足够的**连续空间**（由于 CMS 使用的是标记-清理算法，产生碎片）。**Concurrent Mode Failure** 指的是 CMS 在并发回收的过程中，老年代分配速度 > 回收速度，导致老年代填满。两者最终都会导致 CMS 退化并触发 Full GC。

## 记忆要点
- Young GC触发：因为Eden区空间不足，所以自动触发，非常频繁
- Full GC触发：System.gc()、老年代空间不足、空间分配担保失败、元空间溢出
- CMS的Concurrent Mode Failure或G1的Evacuation Failure均会退化为Full GC
- 空间分配担保失败：Minor GC前老年代连续空间小于历次晋升平均大小即触发
