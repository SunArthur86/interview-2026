# 深入解析 G1 GC 的 Mixed GC 触发条件与 Region 回收机制。

G1 (Garbage First) 垃圾回收器将堆划分为多个大小相等的 Region。Mixed GC 旨在回收整个堆中收益（垃圾占比）最高的 Region。其触发条件通常有两个：一是堆整体占用率达到阈值（默认 45%），即 IHOP（Initiating Heap Occupancy Percent）；二是显式调用 System.gc()。在回收过程中，G1 会根据 Pause Prediction Model 预测回收若干个 Region 所需的时间，控制在用户设定的最大停顿时间（MaxGCPauseMillis）内。它首先并发标记出存活对象，然后混合回收 Eden、Survivor 以及部分 Old Region。如果 Mixed GC 回收速度跟不上分配速度，会退化为 Full GC（单线程），此时需优化堆大小或调整 IHOP 阈值。

**实战案例**：
在高并发大促场景下，曾出现服务因对象分配速率过快，导致 Mixed GC 回收速度跟不上分配速度（Evacuation Failure），频繁触发 Full GC (Serial Old) 导致接口超时。通过调大堆内存并提高 `InitiatingHeapOccupancyPercent` (如 45% -> 60%) 让并发标记更早介入，成功规避了 Full GC。

**关键配置代码**：
```bash
# JVM 参数示例：设置目标停顿时间，调整 IHOP 阈值，避免过早或过晚触发
-XX:+UseG1GC \
-XX:MaxGCPauseMillis=200 \
-XX:InitiatingHeapOccupancyPercent=60 \
-XX:G1HeapRegionSize=16m
```

**GC 模式对比**：

| 特性 | Young GC | Mixed GC | Full GC (G1) |
| :--- | :--- | :--- | :--- |
| **回收对象** | Eden 区、Survivor 区 | Eden + Survivor + **部分 Old** (高收益) | **整个堆** (Young + Old) |
| **触发时机** | Eden 区满了 | 堆占用达到 IHOP 阈值 | Mixed GC 效率不足/分配失败/显式调用 |
| **STW 范围** | 所有应用线程 | 所有应用线程 | 所有应用线程 (单线程，极慢) |
| **优化目标** | 优化新生代分配效率 | 在允许停顿时间内回收最多垃圾 | 最后手段，应尽量避免 |

## 技术原理

- **Region 化堆布局**：G1 把堆切成 2048 个左右的等大 Region（默认 1~32MB，`-XX:G1HeapRegionSize` 指定），每个 Region **动态归属** Eden / Survivor / Old / Humongous，不再物理分代。这让 G1 可以选择性回收"垃圾最多的 Region"（Garbage First 由此得名），而非整代扫描。
- **Mixed GC 的两阶段**：①**并发标记**（Concurrent Marking）——从 GC Roots 出发，标记 Old Region 的存活对象，标记完成后每个 Region 有"存活率"统计；②**混合回收**——在设定的 `MaxGCPauseMillis` 内，用 Pause Prediction Model 挑选**回收价值最高**（垃圾占比高、回收收益大）的若干 Old Region，连同整个 Young 区一起回收。回收过程 STW 但**区域可控**。
- **Pause Prediction Model（停顿预测模型）**：G1 维护每个 Region 的历史回收成本（复制速度、RSet 扫描成本等），用**线性回归**预测"回收 N 个 Region 需要多久"。给定目标停顿时间，反向求解"能回收多少个 Region"。这是 G1 与 CMS 的本质差异——CMS 停顿不可控，G1 把停顿作为约束做"最大化收益"。
- **IHOP（Initiating Heap Occupancy Percent）的自适应**：默认 45%，即老年代占堆 45% 时启动并发标记。JDK 9+ 引入**自适应 IHOP**（`-XX:+G1UseAdaptiveIHOP`，默认开启）——根据历史 Mixed GC 表现动态调整 IHOP，避免过早触发（浪费 CPU）或过晚触发（来不及回收）。

## 命令演示

```bash
# G1 关键参数调优
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200              # 目标停顿（不是硬保证，G1 尽力而为）
-XX:InitiatingHeapOccupancyPercent=45  # 老年代占比超 45% 启动并发标记
-XX:G1HeapRegionSize=16m              # Region 大小（< 32MB 对象不算 Humongous）
-XX:G1NewSizePercent=20               # 新生代下限（避免 Mixed GC 时 Young 过大）
-XX:G1MaxNewSizePercent=60            # 新生代上限
-XX:G1MixedGCCountTarget=8            # 期望 8 次 Mixed GC 回收完所有 Old Region
-XX:G1MixedGCLiveThresholdPercent=85  # Old Region 存活率超 85% 不参与 Mixed GC（成本太高）

# 开启 GC 日志（JDK 9+ 统一日志格式）
-Xlog:gc*=info,gc+heap=debug,gc+task=debug:file=/var/log/gc.log:time,uptime,level,tags:filecount=10,filesize=100M

# 关键日志识别
# "GC pause (G1 Evacuation Pause) (mixed)" → Mixed GC 触发
# "to-space exhausted" → Evacuation Failure，会退化为 Full GC
# "Concurrent Mark Cycle" → 并发标记周期开始
```

## 常见坑/注意事项

- **MaxGCPauseMillis 不是硬保证**：G1 尽力但不保证每次都达标。设定过小（如 10ms）会让 G1 每次只回收极少的 Region，导致回收速度跟不上分配速度，最终触发 Full GC。一般设为 100~300ms。
- **Evacuation Failure 导致 Full GC**：Mixed GC 时若存活对象复制不到 Survivor/Old Region（空间不足），触发 `to-space exhausted`，G1 退化为 Serial Old Full GC（单线程 STW 秒级停顿）。预防：调大堆、提高 IHOP 让并发标记更早介入、调大 `G1ReservePercent`（默认 10%）留出预留空间。
- **Humongous 对象的坑**：单个对象 > Region 大小的一半就标记为 Humongous，直接进 Old Region，且不参与 Young GC 回收。大量 Humongous 对象（如大数组、大 String）会让老年代膨胀。对策：调大 `G1HeapRegionSize`（让对象不超过半 Region），或从代码层面避免大对象分配。
- **System.gc() 的强制 Full GC**：默认 `System.gc()` 触发 Full GC（非 Mixed），破坏 G1 的回收节奏。生产环境建议加 `-XX:+DisableExplicitGC` 禁用，或 `-XX:+ExplicitGCInvokesConcurrent` 让它走并发标记而非 Full GC。
- **Mixed GC 的回收顺序**：G1 按"垃圾占比"排序回收 Old Region，但首批 Mixed GC 不一定回收最多垃圾的 Region——它会先回收"复制成本/收益"最优的。若发现 Mixed GC 后 Old 仍涨，检查 `G1MixedGCCountTarget` 是否过小。
