# 深入解析 G1 GC 的 Mixed GC 触发条件与 Region 回收机制。

G1 (Garbage First) 垃圾回收器将堆划分为多个大小相等的 Region。Mixed GC 旨在回收整个堆中收益（垃圾占比）最高的 Region。其触发条件通常有两个：一是堆整体占用率达到阈值（默认 45%），即 IHOP（Initiating Heap Occupancy Percent）；二是显式调用 System.gc()。在回收过程中，G1 会根据 Pause Prediction Model 预测回收若干个 Region 所需的时间，控制在用户设定的最大停顿时间（MaxGCPauseMillis）内。它首先并发标记出存活对象，然后混合回收 Eden、Survivor 以及部分 Old Region。如果 Mixed GC 回收速度跟不上分配速度，会退化为 Full GC（单线程），此时需优化堆大小或调整 IHOP 阈值。

**实战案例**：
在高并发大促场景下，曾出现服务因对象分配速率过快，导致 Mixed GC 回收速度跟不上分配速度（Evacuation Failure），频繁触发 Full GC (Serial Old) 导致接口超时。通过调大堆内存并提高 `InitiatingHeapOccupancyPercent` (如 45% -> 60%) 让并发标记更早介入，成功规避了 Full GC。

**关键配置代码**：
```bash
# JVM 参数示例：设置目标停顿时间，调整 IHOP 阈值，避免过早或过晚触发
-XX:+UseG1GC \
-XX:MaxGCPauseMillis=200 \
-XX:InitiatingHeapOccupancyPercent=60 \
-XX:G1HeapRegionSize=16m
```

**GC 模式对比**：

| 特性 | Young GC | Mixed GC | Full GC (G1) |
| :--- | :--- | :--- | :--- |
| **回收对象** | Eden 区、Survivor 区 | Eden + Survivor + **部分 Old** (高收益) | **整个堆** (Young + Old) |
| **触发时机** | Eden 区满了 | 堆占用达到 IHOP 阈值 | Mixed GC 效率不足/分配失败/显式调用 |
| **STW 范围** | 所有应用线程 | 所有应用线程 | 所有应用线程 (单线程，极慢) |
| **优化目标** | 优化新生代分配效率 | 在允许停顿时间内回收最多垃圾 | 最后手段，应尽量避免 |

## 记忆要点

- Mixed GC本质：混合回收Eden、Survivor与垃圾最多的Old Region，而非整堆回收
- 触发条件：堆内存占用达IHOP阈值（默认45%）或显式调用System.gc()
- 停顿控制：G1依赖Pause Prediction Model，在设定的MaxGCPauseMillis内挑收益最高的Region回收
