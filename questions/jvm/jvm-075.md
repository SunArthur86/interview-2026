# 频繁Full GC如何排查和解决？

### 排查步骤
1. **监控 GC 频率**：使用 `jstat -gcutil <PID> 1000` 每秒监控一次 GC 情况。重点关注 YGC 到 FGC 的频率以及 FGCT (Full GC Time) 的耗时。如果 Old 区 usage 不断接近 100%，极易触发 Full GC。
2. **分析对象分布**：使用 `jmap -histo:live <PID> | head -20` 查看存活对象排行，初步判断是否有大量可疑对象（如自定义集合、代理对象等）。
3. **转储堆内存**：使用 `jmap -dump:format=b,file=heap.hprof <PID>` 导出堆快照。如果堆过大，需注意 dump 过程中应用可能会暂停（STW），建议在低峰期进行。
4. **分析 Dump 文件**：使用 MAT (Memory Analyzer Tool) 或 VisualVM 打开 dump 文件，查看 Dominator Tree（支配树）或 Histogram（直方图），分析大对象的引用链（GC Roots），定位内存泄漏点。

### 常见原因及解决方案
1. **内存泄漏**：对象无法被回收，Old 区逐渐填满。
   - **解决**：通过 MAT 分析泄漏对象的 GC Chain，修正代码（如未关闭 IO 流、静态集合未清理等）。
2. **大对象直接进入老年代**：超过 `-XX:PretenureSizeThreshold` 设置的大对象直接在 Old 区分配。
   - **解决**：调整阈值，或者优化代码避免大对象频繁创建。
3. **老年代空间不足 / 配置不合理**：新生代调优导致晋升过快，或者老年代本身太小。
   - **解决**：调整 `-Xmx`（最大堆内存）或 `-XX:NewRatio`（新生代与老年代比例），降低晋升年龄 `-XX:MaxTenuringThreshold`。
4. **显式调用 System.gc()**：代码中手动建议进行 GC。
   - **解决**：添加 JVM 参数 `-XX:+DisableExplicitGC` 忽略该调用（但需注意 RMI/NIO 可能依赖它，如使用 Netty 堆外内存需谨慎）。
5. **元空间使用溢出**：加载类过多，导致元空间满了触发 Full GC。
   - **解决**：增大 `-XX:MaxMetaspaceSize`，并排查是否存在动态生成类（如 CGLIB、反射过多）的情况。
6. **CMS/G1 的碎片问题**（CMS 尤甚）：碎片化严重导致无法分配大对象而触发 Full GC。
   - **解决**：CMS 可开启 `-XX:+UseCMSCompactAtFullCollection` (Full GC 时整理) 或切换为 G1 收集器。

### 实战案例
某报表服务在上线后不久开始频繁 Full GC。通过 `jstat` 发现老年代使用率在每次 GC 后几乎不降（无回收）。Dump 分析发现，一个静态的 `HashMap` 中缓存了数百万个报表 key，且没有过期策略。修改代码改为 `Guava Cache` 并设置基于时间的淘汰策略后，老年代水位稳定在 30% 左右。
