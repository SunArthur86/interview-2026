# CMS收集器（多线程标记清除算法）是什么？

CMS（Concurrent Mark Sweep）收集器是一种以获取**最短回收停顿时间**为目标的年老代收集器。它是基于**标记-清除**算法实现的，整个过程分为4个阶段：

1. **初始标记**：
   - 标记 GC Roots 能直接关联到的对象。
   - **细节**：速度很快，但仍需“Stop The World”（STW）。

2. **并发标记**：
   - 进行 GC Roots Tracing，找出存活对象。
   - **细节**：与用户线程**并发**执行，不需要 STW。

3. **重新标记**：
   - 修正并发标记期间因用户线程继续运行而导致标记产生变动的那部分对象的记录。
   - **细节**：通常比初始标记稍长，但仍需 STW。主要使用 **CMS Remark（通常使用 SATB 原始快照）** 来减少停顿时间。

4. **并发清除**：
   - 清理掉死亡对象。
   - **细节**：与用户线程**并发**执行。

由于垃圾收集器线程与用户线程并发执行，CMS 默认回收线程数为 `(CPU核数 + 3) / 4`。

```text
        CMS 工作流程图

    ┌──────────────┐
    │   应用运行   │
    └──────┬───────┘
           │ (GC 触发)
           ▼
    ┌──────────────┐  (STW)  ┌──────────────┐
    │  初始标记     │ ──────> │   并发标记    │ ──────> (用户线程并发运行)
    │ (Roots直接关联)│        │ (Tracing)   │
    └──────┬───────┘         └──────┬───────┘
           │                         │
           ▼                         ▼
    ┌──────────────┐         ┌──────────────┐
    │  重新标记     │ <────── │  (修正变动)   │
    │ (Remark/SATB)│ (STW)   └──────┬───────┘
    └──────┬───────┘                │
           │                        ▼
           │                ┌──────────────┐
           │                │   并发清除    │ ──────> (用户线程并发运行)
           │                │ (清除死亡对象)│
           │                └──────┬───────┘
           │                       │
           └───────────────────────┘
                   ▼
             ┌──────────────┐
             │   应用恢复    │
             └──────────────┘
```

**缺点**：
1. 对 CPU 资源敏感（并发阶段占用 CPU）。
2. **无法处理浮动垃圾**：并发清除时用户线程产生的垃圾只能在下一次 GC 回收。
3. **内存碎片**：基于“标记-清除”，会产生大量不连续空间。当碎片过多无法分配大对象时，不得不触发 Full GC（默认使用 Serial Old 单线程），导致长时间停顿。

---

#### 实战案例
在电商大促期间，老年代 CMS 因碎片化严重导致 Promotion Failed（晋升失败），回退到 Serial Old 进行单线程 Full GC，系统卡顿超 30 秒。解决策略是开启 `-XX:+UseCMSCompactAtFullCollection` 并在业务低峰期手动触发 System.gc() 整理内存。

#### 代码示例 (JVM 参数配置)
```bash
# 启用 CMS，并设置在 Full GC 时进行内存整理（避免碎片过多）
-XX:+UseConcMarkSweepGC 
-XX:CMSInitiatingOccupancyFraction=75  # 老年代使用 75% 时触发 CMS
-XX:+UseCMSCompactAtFullCollection     # Full GC 时开启碎片整理
-XX:CMSFullGCsBeforeCompaction=0       # 每次都强制整理
```

#### 对比表格：CMS vs Parallel Scavenge

| 特性 | CMS 收集器 | Parallel Scavenge (Parallel Old) |
| :--- | :--- | :--- |
| **核心目标** | 最小停顿时间 (低延迟) | 最高吞吐量 (高效利用 CPU) |
| **算法** | 标记-清除 | 标记-整理 (复制算法-新生代) |
| **CPU 占用** | 高 (并发阶段与用户线程抢占) | 低 (STW 期间独占 CPU) |
| **内存碎片** | 有 (可能导致 Promotion Failed) | 无 (整理后内存连续) |
| **适用场景** | Web 服务、响应敏感系统 | 批处理、计算任务、后台运算 |

## 常见考点
1. **CMS 为什么会产生内存碎片？**
   - 因为它是标记-清除算法，不进行整理，导致内存不连续。
2. **CMS 在什么情况下会退化成 Serial Old？**
   - 当并发失败或内存碎片过多无法分配大对象时，触发 Full GC，此时回退到 Serial Old。
3. **CMS 的并发标记阶段为什么可以不 STW？**
   - 因为它只跟踪引用关系，不调整对象位置，且通过“原始快照（SATB）”或其他机制保证准确性。
