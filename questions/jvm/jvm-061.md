# 什么是空间分配担保？

**空间分配担保**：Minor GC前，JVM检查老年代最大可用的连续空间是否大于新生代所有对象总空间，或大于历次晋升到老年代对象的平均大小。

如果大于，认为这次Minor GC是安全的。
如果小于且不允许担保失败（HandlePromotionFailure=false），直接触发Full GC。
如果小于但允许担保失败，尝试Minor GC；但如果Survivor放不下晋升对象，仍需要Full GC。

这个机制确保Minor GC时Survivor区放不下的对象能安全进入老年代。

**逻辑流程图**：
```text
要进行 Minor GC
      │
      ▼
老年代最大连续空间 > 新生代所有对象大小?
      │
      ├─ Yes ──────────────> 进行 Minor GC (最安全)
      │
      No
      │
      ▼
老年代最大连续空间 > 历次晋升平均大小?
      │
      ├─ Yes ──────────────> 尝试 Minor GC (有风险)
      │                               │
      │                               ▼
      │                     晋升时 Survivor/老年代空间足够?
      │                               │
      │                     ├─ Yes ─> 成功
      │                     │
      │                     └─ No ──> [担保失败] 触发 Full GC
      │
      └─ No ───────────────> [风险太高] 直接触发 Full GC
```

**关键细节与参数**：
1. **目的**：为了在 Minor GC 之前，确保老年代有足够空间容纳“最坏情况”下的所有存活对象（即所有对象都晋升到老年代），避免 Minor GC 进行到一半发现老年代放不下，导致 JVM 崩溃或不得不进行昂贵的 Full GC。
2. **参数控制**：
   - JDK 6 Update 24 之前：`HandlePromotionFailure` 参数决定是否允许冒险。
   - JDK 6 Update 24 之后：规则变为：只要老年代最大连续空间大于历次晋升大小，就允许 Minor GC（相当于`HandlePromotionFailure`永远为true）。
3. **最坏情况 vs 平均情况**：
   - 检查“所有对象总大小”是最坏情况担保，非常保守。
   - 检查“历次晋升平均值”是概率担保，假设这次晋升规模和以往差不多。

## 常见考点
1. **HandlePromotionFailure 参数在 JDK 8 中还存在吗？** 
   不再起作用。从 JDK 6 Update 24 开始，该参数的逻辑被废弃，JVM 只要检测到老年代可用连续空间大于历次晋升平均值，就会尝试 Minor GC；否则直接 Full GC。因此，面试时应强调这是老版本的概念。
2. **为什么“历次晋升平均值”比“所有对象总大小”更实用？** 
   因为新生代绝大多数对象都是朝生夕死的，真正晋升到老年代的对象通常很少（如 Eden 区 90%+ 都是垃圾）。如果每次都按“所有对象存活”预留空间，会导致老年代大量闲置浪费。使用平均值是空间利用率和安全性的折中。
3. **如果发生了“担保失败”，会怎样？** 
   此时意味着 Minor GC 尝试失败了。JVM 需要重新发起一次 Full GC (通常是 Serial Old)，Stop-The-World 时间更长。因此生产环境中尽量减少这种情况发生，通常通过调低 `-XX:MaxTenuringThreshold` 或调大新生代来优化。
