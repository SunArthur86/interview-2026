# CMS垃圾收集器的四个阶段是什么？

CMS（Concurrent Mark Sweep）收集器以获取最短回收停顿时间为目标，采用“标记-清除”算法，整个过程分为四个阶段：

1. **初始标记**
   - 状态：STW (Stop The World)
   - 动作：仅标记 GC Roots 能直接关联到的对象。
   - 耗时：非常短，因为只是直接引用查找，不遍历引用链。

2. **并发标记**
   - 状态：并发运行
   - 动作：从 GC Roots 直接关联的对象开始遍历整个对象图。
   - 耗时：较长，但与用户线程同时执行，不阻塞应用。

3. **重新标记**
   - 状态：STW
   - 动作：修正并发标记期间因用户线程运行而导致标记产生变动的那部分对象。
   - 细节：通常比初始标记稍长，但远短于并发标记。使用**增量更新**（Incremental Update）算法，将并发期间新产生引用记录下来，在此阶段重新扫描。

4. **并发清除**
   - 状态：并发运行
   - 动作：清除判定为死亡的对象，释放内存。
   - 耗时：与用户线程并发执行。

```text
CMS 工作流程时间轴：

用户线程:  ████████████        ████████████████████        ████████████████
GC线程  :     └─[初始标记]─┘       └────[并发标记]────┘ └─[重新标记]─┘ └─[并发清除]─┘
停顿(STW):       ░░░                               ░░░
说明：仅初始标记和重新标记需要短暂停顿（STW）。
```

**优缺点详解**：
- **优点**：低停顿，适合对延迟敏感的互联网应用（B/S架构服务端）。
- **缺点**：
  1. **内存碎片**：因为是“标记-清除”算法，不进行压缩，会导致大量离散内存空间，大对象分配困难。
  2. **CPU 敏感**：并发阶段占用 CPU 资源，虽然默认开启（`(CMS+1)/CPU数量`），但在核数少时会降低吞吐量。
  3. **浮动垃圾**：并发清除时用户线程产生的垃圾只能留到下一次回收。
  4. **Concurrent Mode Failure**：如果并发清除期间预留的内存无法满足新对象分配请求，会触发此错误，此时回退到 Serial Old 收集器进行 Full GC，导致长时间的 STW。

## 常见考点
1. CMS 为什么会退化成 Serial Old？
   触发 Concurrent Mode Failure 或 Promotion Failed（晋升失败）。通常是因为碎片太多找不到连续空间，或者并发回收速度赶不上分配速度。
2. CMS 的并发标记是基于什么算法？
   基于三色标记法，配合增量更新来解决漏标问题。
3. 如何开启 CMS？
   `-XX:+UseConcMarkSweepGC`。JDK 9 标记废弃，JDK 14 中移除。
4. CMS 对老年代和新生代分别使用什么收集器？
   老年代是 CMS 本身；新生代通常配合 ParNew 收集器使用。

## 记忆要点
- 四阶段口诀：初始标记、并发标记、重新标记、并发清除
- 并发特性：只有初始标记和重新标记需要短暂STW，其余与用户线程并发
- 解决漏标：重新标记阶段使用增量更新算法修正并发标记期间的变动
- 三大缺点：标记-清除算法产生内存碎片、CPU敏感、产生浮动垃圾
- 退化警告：担保失败或空间不足会触发Concurrent Mode Failure，退化为Serial Old
