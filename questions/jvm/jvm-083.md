# ZGC 为什么能做到亚毫秒级停顿？染色指针和读屏障的原理是什么？

【ZGC 核心目标】无论堆大小（4GB 还是 16TB），停顿时间 < 10ms（JDK 17），JDK 21 达到 < 1ms。它是一个 Region-based 的并发现代垃圾回收器。

【ZGC 的 3 大核心技术】
1. **染色指针**
   利用 64 位地址中较高的未使用位（目前使用高 4 位）来存储 GC 状态信息（Marked0, Marked1, Remapped, Finalizable）。
   - **优势**：指针自描述，无需单独存储位图；对象移动时只需修改指针中的位，无需遍历对象修改其头。
   - **多重映射**：Linux/Unix 系统支持将同一块物理内存映射到多个不同的虚拟地址空间。ZGC 利用这一点，让不同的染色位对应不同的虚拟内存视图，但在物理上指向同一数据，实现了零成本的元数据切换。

2. **读屏障**
   - **原理**：JVM 在从堆中读取引用（Load 指令）时，动态插入一段微小的代码。它检查引用的染色位，如果对象处于「 relocated 」状态，则将其修复为新的地址。
   - **作用**：将对象移动后的引用修复工作分摊到每一次读取操作中，使得应用线程与 GC 线程并发执行成为可能。

3. **并发整理**
   标记、转移、重定位都是并发执行的。只有两个极短 STW 阶段（初始标记：Roots 扫描 + 再标记），其余全部并发。

【ZGC 运作流程图】
```text
┌──────────────┐
│  1. 初始标记  │ (STW, 极短, 标记 GC Roots 直接关联对象)
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  2. 并发标记  │ (遍历对象图, 使用读屏障处理转发)
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  3. 预处理重定位 │ (准备 Region 集合)
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  4. 并发转移  │ (移动活对象到新 Region, 更新染色指针)
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  5. 初始重定位 │ (STW, 极短, 更新 Roots 引用)
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  6. 并发重定位 │ (应用线程通过读屏障, 慢慢修复剩余引用)
└──────────────┘
```

【Generational ZGC（JDK 21）】
非分代 ZGC 每次都要扫描全堆。分代 ZGC 把堆分为年轻代和老年代，新生对象频繁在年轻代回收，老年代存活期长。这使得 Young GC 只扫描年轻代，大幅降低单次 GC 的开销和总 CPU 消耗。

【传统 GC 为什么停顿长？】
传统 GC（如 CMS, G1）的整理阶段：必须暂停所有应用线程（STW），遍历所有引用并更新为新地址（对象移动图）。ZGC 通过读屏障将引用更新延迟到读的时候发生，从而去除了这种大规模的 STW 需求。

## 常见考点
1. **染色指针的限制**：ZGC 依赖 64 位系统，且需要操作系统支持多重映射，因此在 32 位系统或某些特定受限环境下无法使用。
2. **读屏障的性能开销**：虽然读屏障很快（主要是一两条指令），但由于每次读引用都要执行，对于读密集型应用可能带来轻微的 CPU 开销。
3. **不支持指针压缩**：因为高 4 位被借用于染色，ZGC 默认关闭指针压缩（-XX:-UseCompressedOops），这意味着堆内存占用可能比 G1 稍大，但在 64 位大内存下通常不是瓶颈。
4. **Load Barriers 的位置**：ZGC 只在 Load（读引用）时加屏障，Store（写引用）时不加，这与 G1 等使用 SATB 的收集器有所不同。

## 记忆要点

- 极低停顿原因：除初始标记外，对象的转移和重定位阶段全部并发执行
- 染色指针：借用 64 位指针高位记录对象状态，无需修改对象头即可实现标记转移
- 读屏障：读取引用时插入微小代码，发现对象被移动则自我修正指向新地址
- 代价对比：ZGC 在读引用时加屏障，而 G1/CMS 主要依赖写屏障，ZGC 实现了并发整理
