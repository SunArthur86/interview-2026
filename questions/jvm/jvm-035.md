# 什么是类加载器？

### 什么是类加载器？

**定义**
类加载器是Java虚拟机（JVM）的重要组成部分，负责实现类的加载动作。即在类加载阶段的“加载”步骤中，通过一个类的全限定名来获取描述该类的二进制字节流，并将其转化为方法区中的运行时数据结构。

**核心特性**
1. **唯一性**：Java中的任意一个类，都必须由加载它的类加载器和这个类本身一起共同确立其在JVM中的唯一性。每一个类加载器都拥有一个独立的类名称空间。
2. **比较逻辑**：判断两个类是否“相等”（包括`equals()`、`isAssignableFrom()`等），只有在这两个类是由同一个类加载器加载的前提下才有意义。否则，即使两个类来源于同一个Class文件，被同一个JVM加载，但加载器不同，它们也是两个截然不同的类。

**JVM提供的类加载器**
JVM设计团队将加载动作放到JVM外部实现，以便让应用程序决定如何获取所需的类。JDK 8及以前主要提供以下三种类加载器：

1. **启动类加载器**
   - 负责加载JVM的核心类库，如`<JAVA_HOME>\lib`目录下，或被`-Xbootclasspath`参数指定的路径中的类库。
   - 是JVM自身的一部分，通常无法被Java程序直接引用（返回 null）。

2. **扩展类加载器**
   - 负责加载`<JAVA_HOME>\lib\ext`目录下，或被`java.ext.dirs`系统变量所指定的路径中的所有类库。
   - 开发者可以直接使用该加载器。

3. **应用程序类加载器**
   - 也称为系统类加载器。它负责加载用户类路径上所有的类库。
   - 如果应用程序中没有自定义过类加载器，程序中默认的类加载器通常就是这一种。

**补充细节（JDK 9+ 变化）**：
在 JDK 9 及之后，引入了模块化系统。扩展类加载器被**平台类加载器**取代，用于加载 Java SE 平台但非核心模块的类。双亲委派模型在模块化下依然适用，但类查找范围基于模块定义。

**双亲委派模型**
虽然JVM提供了这三种加载器，但大多数情况下，类加载是通过**双亲委派模型**来工作的：如果一个类加载器收到了类加载的请求，它首先不会自己去尝试加载这个类，而是把这个请求委派给父类加载器去完成，每一个层次的类加载器都是如此。因此所有的加载请求最终都应该传送到最顶层的启动类加载器中，只有当父加载器反馈自己无法完成这个加载请求（找不到所需的类）时，子加载器才会尝试自己去加载。

**双亲委派工作流程图：**
```text
1. AppClassLoader 收到请求
       │
       v
2. 委派给 ExtClassLoader
       │
       v
3. 委派给 BootstrapClassLoader
       │
       ├──> 能加载？ -> 成功返回 Class
       │
       └──> 不能加载 -> 回退到 ExtClassLoader 尝试加载
                     │
                     ├──> 能加载？ -> 成功返回 Class
                     │
                     └──> 不能加载 -> 回退到 AppClassLoader 尝试加载
                                   │
                                   ├──> 能加载？ -> 成功返回 Class
                                   │
                                   └──> 不能加载 -> 抛出 ClassNotFoundException
```
**为什么要双亲委派？**
1. **安全性**：防止核心类库被篡改。例如用户自定义一个 `java.lang.String`，如果没有双亲委派，JVM 可能会加载用户代码，导致核心 API 不安全。双亲委派确保 `java.lang.*` 始终由启动类加载器加载。
2. **避免重复加载**：如果父加载器已经加载过该类，子加载器无需再次加载，保证类的唯一性。

### 4. 实战深化

**实战案例**：在一个容器化环境（如 Tomcat 或 Spring Boot FatJar）中，应用依赖 `spring-context` 4.3.x，而容器底层依赖 `spring-context` 5.x。若严格遵循双亲委派，会加载底层的版本导致应用冲突。解决方案是破坏双亲委派，实现自定义类加载器，优先加载应用 WEB-INF/lib 下的类（即 Child-First 策略）。

**代码示例 (破坏双亲委派 - 简化版 Tomcat 类加载)**：
```java
public class WebAppClassLoader extends URLClassLoader {
    @Override
    public Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
        // 1. 检查类是否已加载
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                // 2. 违背双亲委派：先尝试自己加载 (WebApp 路径)
                c = findClass(name);
            } catch (ClassNotFoundException e) {
                // 3. 失败才委托给父加载器
                c = super.loadClass(name, resolve);
            }
        }
        if (resolve) resolveClass(c);
        return c;
    }
}
```

**对比表格**：

| 特性 | 双亲委派模型 | 破坏双亲委派 (如 Tomcat, OSGi, SPI) |
| :--- | :--- | :--- |
| **加载顺序** | 父 -> 子 (自上而下) | 子 -> 父 (自下而上) 或 特定顺序 |
| **核心优势** | 安全、避免重复加载 | 隔离性、支持版本多并存 |
| **典型场景** | 标准Java应用 | Web容器、模块化热部署、JDBC SPI |
| **类唯一性** | JVM 全局唯一 | ClassLoader 命名空间内唯一 |

## 常见考点
1. **如何破坏双亲委派模型？**
   - 追问 SPI（Service Provider Interface）机制（如 JDBC, JNDI），由于接口是 Java 核心库（由启动类加载器加载），而实现类是第三方库（由应用类加载器加载），启动类加载器无法“向下”加载实现类。
   - 解决方案：利用线程上下文类加载器（Thread ContextClassLoader）来突破双亲委派。
