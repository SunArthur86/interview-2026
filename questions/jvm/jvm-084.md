# 生产环境 JVM 如何调优？堆大小、GC 收集器、关键参数怎么选择？

【调优目标】
- **吞吐量优先**（后台批处理、科学计算）：-XX:GCTimeRatio=99（GC 时间占比 1%）。选择 PS (Parallel Scavenge) + PO (Parallel Old)。
- **延迟优先**（在线服务、Web API）：-XX:MaxGCPauseMillis=200（期望停顿时间）。选择 G1 或 ZGC。
- **最小内存占用**（嵌入式、客户端）：Serial GC。

【堆大小设置】
- **-Xms 和 -Xmx 设相同值**：避免运行时动态扩容和缩容带来的性能抖动及内存碎片整理开销。
- **经验值**：
  - 4GB - 8GB 堆内存是现代微服务的常见起步值。
  - 新生代:老年代比例：在 G1 中无需手动设（由 Pause Goal 动态调整），在 Parallel GC 中通常默认 1:2。
  - **限制**：堆内存不应超过物理内存的 50%-70%，需给操作系统缓存和 JVM 自身内存（元空间、栈、CodeCache）留空间，避免因 Swap 导致严重的 GC 停顿。

【GC 收集器选择路线图】
- **JDK 8**：推荐 G1（-XX:+UseG1GC），大堆优于 Parallel GC。
- **JDK 11**：默认 G1。若追求极低延迟（<10ms）且堆较大，尝试 ZGC（需解锁 -XX:+UnlockExperimentalVMOptions -XX:+UseZGC）。
- **JDK 15+**：ZGC 正式发布。
- **JDK 21+**：默认 Generational ZGC（分代 ZGC），成为低延迟首选。
- **选择策略**：
  - 堆 < 4GB：Serial 或 Parallel。
  - 4GB < 堆 < 32GB：G1。
  - 堆 > 32GB 且对延迟敏感：ZGC。

【G1 关键参数调优】
1. **-XX:MaxGCPauseMillis=200**：最核心参数。G1 会根据历史数据自动调整 Region 回收数量以尽量满足该目标（但并非绝对保证）。
2. **-XX:InitiatingHeapOccupancyPercent (IHOP)**：默认 45%。当堆占用率达到 45% 时启动并发标记周期。调大此值可减少并发标记频率，但可能导致混合回收（Mixed GC）来不及清理而退化为 Full GC。
3. **-XX:G1HeapRegionSize**：Region 大小（2MB-32MB，需为 2 的幂）。大对象（> Region 50%）跨 Region 存储（Humongous Region）。Region 太大导致小块内存浪费，太小导致维护开销增加。
4. **-XX:G1ReservePercent**：默认 10%。为 To-Space（存活对象预留空间）预留的堆百分比。如果频繁发生 To-Space Exhausted（转移失败），可适当调大此值。

【通用必开参数】
1. **-XX:+HeapDumpOnOutOfMemoryError**：内存溢出时自动 Dump 堆快照（hprof 文件）。
2. **-XX:HeapDumpPath=/tmp/heapdump.hprof**：指定 Dump 路径。
3. **-XX:ErrorFile=/tmp/hs_err_pid<pid>.log**：JVM 崩溃时记录错误日志。
4. **GC 日志（JDK 9+）**：
   -Xlog:gc*:file=gc.log:time,tags:filecount=5,filesize=10m
   包含 GC 详情、停顿时间、引用处理等。

## 常见考点
1. **Young GC 频繁 vs Mixed GC/Full GC 频繁**：YGC 频繁通常是因为 Eden 区太小或对象过早晋升；Full GC 频繁通常是因为并发标记未能在堆满前完成，需调整 IHOP。
2. **G1 的 Evacuation Failure（转移失败）**：当 Survivor 或 Old 区没有足够空间容纳存活对象时发生，会导致 Full GC。调优方向：调大堆、增加 G1ReservePercent 或加快并发标记。
3. **元空间溢出**：-XX:MaxMetaspaceSize 需合理设置，防止动态类加载过多导致元空间撑爆内存。
4. **CPU 开销过高**：ZGC 或 G1 在并发阶段会占用较多 CPU，若 CPU 资源紧张，可能需要平衡 GC 线程数（-XX:ConcGCThreads）或考虑降级为 Parallel GC。
