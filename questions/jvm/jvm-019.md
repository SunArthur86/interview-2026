# JVM方法区（永久代）的作用是什么？

方法区是JVM运行时数据区的一块，它是各个线程共享的内存区域。它用于存储已被虚拟机加载的类信息、常量、静态变量、即时编译器编译后的代码等数据。虽然它被称为“永久代”，但这仅仅是HotSpot VM的实现方式（JDK 1.8之前），在JVM规范中并不存在永久代的概念。

**主要组成部分**：
1. **类型信息**：类的完整名称、父类完整名称、类型修饰符等。
2. **常量池**：存放编译期生成的各种字面量和符号引用。
3. **字段信息**：字段名称、修饰符、类型描述符。
4. **方法信息**：方法名称、修饰符、返回类型、参数类型、字节码、异常表等。
5. **静态变量**：被所有类实例共享的变量。
6. **类加载器引用**：指向加载该类的类加载器。

**内存回收**：
方法区的内存回收目标主要针对两部分：
1. **常量池的回收**：如字面量、符号引用如果没有被引用，可以被回收。
2. **类型的卸载**：该类所有的实例都已被回收，且加载该类的类加载器已被回收（难度较大，条件苛刻），则该类可以被卸载。由于回收条件苛刻且收益一般较小，方法区的垃圾回收效率通常较低。

**版本差异**：
- **JDK 1.7**：已经逐步“去永久代”，字符串常量池、静态变量移至堆中。
- **JDK 1.8**：完全取消了永久代，使用元空间替代。元空间使用本地内存，受到`MaxMetaspaceSize`参数限制。

## 常见考点
1. **JDK 1.8为何移除永久代？**
   - 永久代大小很难确定，容易导致`PermGen OOM`或内存浪费。
   - GC复杂，需要对永久代进行特殊处理。
   - 融合HotSpot与JRockit，JRockit本来就没有永久代。
2. **元空间与永久代的区别？**
   - 存储位置：元空间使用本地内存，永久代使用JVM堆内存。
   - 大小限制：元空间默认只受本地内存限制，可通过`MaxMetaspaceSize`控制；永久代受`-XX:MaxPermSize`控制。
3. **运行时常量池的作用？**
   它是Class文件中常量池表的运行时表示，包含编译期生成的字面量和符号引用，部分也会在运行时放入（如`String.intern()`）。

### 实战拓展

**实战案例**：
在频繁使用 Spring AOP 或 CGLIB 动态生成代理类的系统中，若未设置 `MaxMetaspaceSize`，可能会因为类元数据膨胀吃光服务器本地内存，导致容器被 Kill（而非抛出 OOM）。生产环境必须设置 `-XX:MaxMetaspaceSize=256m` 等上限。

**代码示例（验证字符串常量位置）**：
```java
// JDK 1.8+ 示例
public class MetaSpaceDemo {
    public static void main(String[] args) {
        // JDK 6/7 字符串常量在 PermGen，易 OOM
        // JDK 8+ 字符串常量在堆中，受 -Xmx 限制
        // 若要模拟元空间溢出，需使用 CGLib/ByteBuddy 动态生成大量类
        try {
            List<String> list = new ArrayList<>();
            int i = 0;
            while (true) {
                list.add(String.valueOf(i++).intern());
            }
        } catch (Error e) {
            e.printStackTrace();
        }
    }
}
```

**永久代 vs 元空间 对比**：

| 特性 | 永久代 (PermGen, JDK < 1.8) | 元空间 | 
| :--- | :--- | :--- |
| **存储位置** | JVM 堆内存 | 本地内存 |
| **大小限制** | 固定大小 (`-XX:MaxPermSize`) | 默认无限 (受限于物理内存，可设 `MaxMetaspaceSize`) |
| **GC 影响** | Full GC 会扫描，效率低 | GC 效率较高，调整更灵活 |
| **字符串存储** | 包含字符串常量池 | 移至堆中 |
| **常见错误** | `java.lang.OutOfMemoryError: PermGen space` | `java.lang.OutOfMemoryError: Metaspace` / Native Memory OOM |


## 记忆要点
- 存储核心数据：因为存放类信息、常量、静态变量和JIT代码，所以是各线程共享的逻辑内存区域。
- 版本演变对比：JDK8前叫永久代占用堆内存，JDK8后叫元空间改用本地物理内存。
- 移除永久代原因：因为永久代大小极难调优且容易OOM，所以融合JRockit改用元空间使用本地内存。
- 垃圾回收效率低：因为类的卸载条件极为苛刻（实例全清且加载器全清），所以方法区回收收益通常很小。
