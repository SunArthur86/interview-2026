# Tomcat的类加载机制是怎样的？

### Tomcat 类加载器层次结构
Tomcat 为了实现隔离和灵活性，自定义了一套类加载器结构，默认如下：

1. **Bootstrap ClassLoader** (JVM 启动类加载器)：加载 JVM 核心类。
2. **System ClassLoader** (JVM 默认 AppClassLoader)：加载 Tomcat 启动类 (`bootstrap.jar` 等)。
3. **Common ClassLoader**：加载 Tomcat 内部和 WebApp **可见**的通用类库（位于 `common/` 目录）。
4. **Catalina ClassLoader**：加载 Tomcat 内部**私有**类库（位于 `server/` 目录），对 WebApp 不可见。
5. **Shared ClassLoader**：加载所有 WebApp **共享**的类库（位于 `shared/` 目录），对 Tomcat 内部不可见。
6. **WebApp ClassLoader**：每个 Web 应用独立一个，加载 `/WEB-INF/classes` 和 `/WEB-INF/lib` 中的类。

### WebAppClassLoader 的破坏双亲委派机制
为了实现优先加载应用自身的类，WebAppClassLoader 重写了 `loadClass` 方法，加载顺序如下：
1. **Cache 缓存**：先从本地缓存查找是否已加载。
2. **JVM 基础类委托**：如果没有，且类名以 `java.` 或 `javax.` 开头，**必须**委派给 Bootstrap ClassLoader（防止篡改核心类）。
3. **Web 应用自身加载**：调用 `findClass` 在 `/WEB-INF/classes` 和 `/WEB-INF/lib` 中查找并加载。
4. **父加载器委托**：如果还没找到，才委派给父加载器（即 CommonClassLoader 等）。

### 设计原因
1. **多应用依赖隔离**：不同 WebApp 可以依赖同一个框架的不同版本（如 Spring 4 vs Spring 5），互不干扰。
2. **自身安全与稳定**：Tomcat 自身的类库不会被应用的类覆盖（Catalina ClassLoader 隔离），防止应用破坏容器运行。
3. **资源复用**：通过 Common 或 Shared ClassLoader，可以实现一些通用库的共享，节省内存。

## 常见考点
1. **如果 Tomcat 加载了 Spring，但应用自己也带了 Spring，哪个会生效？**（WebAppClassLoader 优先加载本地 `/WEB-INF/lib`，所以应用带的 Spring 生效）。
2. **如何在 Tomcat 中实现多个 WebApp 共享同一个 jar 以减少内存？**（放入 `common/lib` 或 `lib` 目录，由 CommonClassLoader 加载）。
3. **为什么不能直接使用 Java 默认的双亲委派？**（会导致版本冲突，无法实现多应用隔离）。


## 记忆要点
- 多应用隔离：每个 WebApp 独立一个 WebAppClassLoader，优先加载本地依赖
- 核心类保护：遇到 java.* 等核心类，必须强制委派给父加载器防止篡改
- 加载优先级：本地缓存 > JVM 核心类委托 > WEB-INF 本地加载 > 父加载器委托
- 资源共享：放入 Common 或 Shared 目录的类库可被多个 WebApp 共享复用
