# 类加载器及加载机制是什么？

### 类加载器及加载机制

**1. 类加载机制**
类加载机制指的是JVM将类的数据从Class文件加载到内存，并对数据进行校验、转换解析和初始化，最终形成可以被JVM直接使用的Java类型的过程。

类的生命周期包括7个阶段：
1. **加载**：获取二进制字节流，生成Class对象。
2. **验证**：确保Class文件的字节流包含的信息符合当前虚拟机的要求。
3. **准备**：为类的静态变量分配内存，并设置默认初始值。
4. **解析**：将常量池内的符号引用替换为直接引用。
5. **初始化**：执行类构造器`<clinit>()`方法，为静态变量赋值。
6. **使用**
7. **卸载**

**2. 类加载器**
类加载器是实现加载阶段的代码模块。
- **唯一性**：比较两个类是否相等，只有在这两个类由同一个类加载器加载的前提下才有意义。
- **双亲委派模型**：
  如果一个类加载器收到了类加载的请求，它首先不会自己去尝试加载这个类，而是把这个请求委派给父类加载器去完成。只有当父加载器反馈自己无法完成这个加载请求时，子加载器才会尝试自己去加载。

**双亲委派模型架构图**
```
                    Bootstrap ClassLoader
                    (启动类加载器 / C++实现)
                             ↑
                             │ 委派
                             │
              Extension ClassLoader (Platform ClassLoader)
              (扩展类加载器 / Java实现)   ←--- JDK9后名为Platform
                             ↑
                             │ 委派
                             │
         Application ClassLoader (System ClassLoader)
         (应用程序类加载器)              ←--- 加载用户ClassPath
                             ↑
                             │ 委派
                             │
              Custom ClassLoaders (用户自定义)
```

  - **启动类加载器**：加载核心库（如JAVA_HOME/lib/rt.jar, resources.jar）。
  - **扩展类加载器**：加载`<JAVA_HOME>/lib/ext`目录下的库（JDK8及以前）。
  - **应用程序类加载器**：加载用户ClassPath上的类。
  - **自定义类加载器**：用户自定义的加载逻辑。

**3. 双亲委派的好处**
- **安全性**：Java核心类库（如`java.lang.String`）随着启动类加载器加载，无法被篡改。例如用户自己写了一个`java.lang.String`，由于双亲委派，JVM永远只会加载标准库中的String，防止核心API被恶意替换。
- **避免重复加载**：如果父加载器已经加载过该类，子加载器就没必要再加载一次。

**## 常见考点**
1. **如何打破双亲委派模型？**
   继承ClassLoader类，重写loadClass方法（不建议，违背双亲委派）或使用线程上下文类加载器。典型应用场景包括：Tomcat类加载机制（实现应用隔离）、JDBC（Driver接口定义在rt.jar由Bootstrap加载，实现类由厂商提供需由AppClassLoader加载，使用ContextClassLoader破坏双亲委派以获取实现类）。
2. **全盘负责委托机制**：当一个类加载器负责加载某个Class时，该Class所依赖和引用的其它Class通常也由该类加载器负责加载（除非显式指定其它加载器）。
3. **JDK9模块化对类加载的影响**：引入了平台类加载器和模块化层，扩展类加载器被Platform ClassLoader取代，双亲委派结构依然保持但层级和职责有所调整。

**实战案例**：
在中间件开发中，为了实现插件热部署（不重启服务更新代码），我们自定义了类加载器，打破双亲委派机制，优先从插件目录加载类。如果直接使用默认加载机制，修改后的插件类会被已加载的旧类覆盖，导致更新不生效。

**代码示例**：
```java
// 自定义类加载器打破双亲委派（优先本地加载）
public class HotSwapClassLoader extends ClassLoader {
    @Override
    protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
        // 1. 先检查类是否已加载
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                // 2. 尝试自己加载（打破双亲委派，不先调父类）
                c = findClass(name); 
            } catch (ClassNotFoundException e) {
                // 3. 自己找不到才委托给父加载器
                c = super.loadClass(name, resolve);
            }
        }
        return c;
    }
}
```

**对比表格**：

| 特性 | 双亲委派模型 | 破坏双亲委派模型 |
| :--- | :--- | :--- |
| **加载顺序** | 委派给父加载器，无法加载时才自己加载 | 优先尝试自己加载，失败后再委派（或直接不委派） |
| **核心优势** | 安全（防止核心类被篡改）、避免重复加载 | 灵活、支持类隔离、支持热替换 |
| **典型场景** | 标准Java应用 | Web容器(Tomcat)、OSGi、JDBC SPI、热部署工具 |
| **实现方式** | 遵循 `java.lang.ClassLoader.loadClass` 逻辑 | 重写 `loadClass` 方法或使用 `Thread.getContextClassLoader()` |
