# NIO的实现原理是什么？

Java NIO（New IO）的核心实现原理基于操作系统底层的**IO多路复用**模型（主要依赖 select/poll 或 epoll 机制），主要包含以下三个核心组件：

**1. Channel（通道）**
- 数据的源头或目的地，类似于流，但它是双向的（可读可写）。
- 常见实现：FileChannel（文件读写，不能设为非阻塞模式）、SocketChannel（TCP读写）、ServerSocketChannel（TCP监听）、DatagramChannel（UDP读写）。

**2. Buffer（缓冲区）**
- 容器，Channel读写的数据都必须经过Buffer。
- 本质是一块可读写内存，通过内部指针（position, limit, capacity）控制数据读写。
- **关键细节**：支持 DirectByteBuffer（堆外内存），减少一次 JVM 堆内存到本地内存的拷贝，提高 IO 效率，但分配成本高。

**3. Selector（选择器）**
- **核心组件**，对应 IO 多路复用模型中的“复用器”。
- 单个线程可以监听多个 Channel 的 IO 事件（连接、读、写）。
- **原理细节**：应用程序向 Selector 注册 Channel 及感兴趣的事件（SelectionKey）。Selector 底层调用系统调用（如 Linux 下的 epoll_wait），由内核监控这些文件描述符。只有当 Channel 真正有就绪事件时，才进行读写操作，避免线程阻塞。

### 实战案例
在开发百万连接的即时通讯（IM）推送服务时，使用传统的 BIO 会耗尽线程池资源。采用 NIO 的 Selector 模型后，单核即可轻松处理数万连接，但需小心处理“半包读写”问题（TCP粘包/拆包），这通常通过在 ByteBuffer 前加长度头来解决。

### 代码示例
```java
// NIO 服务端核心循环
Selector selector = Selector.open();
ServerSocketChannel server = ServerSocketChannel.open();
server.configureBlocking(false);
server.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    // 阻塞直到有事件发生，避免CPU空转
    selector.select(); 
    Set<SelectionKey> keys = selector.selectedKeys();
    for (SelectionKey key : keys) {
        if (key.isAcceptable()) handleAccept(key);
        if (key.isReadable()) handleRead(key);
    }
    keys.clear(); // 必须手动移除已处理的key
}
```

### IO模型对比

| 模型 | 特点 | 适用场景 |
| :--- | :--- | :--- |
| **BIO** | 同步阻塞，一个连接一个线程 | 连接数少，架构简单 |
| **NIO** | 同步非阻塞，多路复用，一个线程处理多连接 | 高并发、连接数多、轻量级操作 |
| **AIO** | 异步非阻塞，操作系统回调 | 连接数非常多，且读写操作耗时很长 |

### ## 常见考点
1. **Selector 是如何实现线程不阻塞的？**
   - 底层依赖操作系统的多路复用系统调用（如 `select`、`poll`、`epoll`）。在没有事件时，线程在 `select()` 处阻塞；有事件时唤醒。
2. **什么是多路复用的“空轮询”Bug？**
   - 在早期 Linux 的 `select`/`poll` 实现中，可能发生 `Selector.select()` 返回 0 但没有唤醒原因，导致 CPU 100%。Netty 等框架对此有专门的解决逻辑（如重建 Selector）。
3. **ByteBuffer 的 allocate 与 allocateDirect 的区别？**
   - `allocate` 分配堆内存，受 GC 管理，读写可能发生内存拷贝；`allocateDirect` 分配堆外内存，避免拷贝，但分配和回收成本高，适合长生命周期的大块 IO 缓冲。
