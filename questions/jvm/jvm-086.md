# JVM 创建对象的完整过程是什么？为什么 new 不是原子的？指针压缩是什么？

【对象创建 5 步详解】
1. **类加载检查**：虚拟机遇到 `new` 指令时，首先去检查这个指令的参数能否在常量池中定位到一个类的符号引用，并检查该类是否已被加载、解析和初始化过。
2. **分配内存**：
   - **指针碰撞**：堆内存规整（Serial/Parallel GC），指针向空闲区移动所需大小。
   - **空闲列表**：堆内存不规整（CMS/G1），维护列表记录空闲块。
   - **TLAB (Thread Local Allocation Buffer)**：为了避免多线程分配内存时的 CAS 竞争，每个线程在 Eden 区预先分配一块私有内存。新对象优先在 TLAB 分配，只有 TLAB 不足时才加锁在堆中分配。
3. **初始化零值**：将分配到的内存空间（不包括对象头）都初始化为零值，保证了对象的实例字段在 Java 代码中可以不赋值就直接使用（如 int 为 0，boolean 为 false）。
4. **设置对象头**：
   - **Mark Word**：存储哈希码、GC 分代年龄、锁状态标志等。
   - **类型指针**：指向对象所属类的元数据（Class 对象）。如果是数组，还需记录数组长度。
5. **执行构造函数**：按照初始化顺序（实例变量初始化 -> 构造代码块 -> 构造函数）进行初始化。

【为什么 new 不是原子的？】
`instance = new Singleton()` 实际上对应 3 条字节码指令：
1. `new`：分配内存（此时对象尚未初始化，内存全是零值）。
2. `invokespecial`：调用构造函数初始化对象。
3. `astore`：将引用赋值给变量。

由于指令重排序（Instruction Reordering），2 和 3 可能颠倒。
**并发场景**：线程 A 执行了 1->3（引用非 null），线程 B 此时判断 `instance != null` 并拿去使用，此时线程 B 访问的是一个「半初始化」的对象（字段为 0 或 null），导致错误。解决：`volatile` 禁止重排 + DCL 双重检查。

【指针压缩】
- **原理**：64 位 JVM 默认使用 64 位指针（8 字节）。开启指针压缩（默认开启 `-XX:+UseCompressedOops`）后，堆中的引用存储为 32 位偏移量（4 字节），真实地址 = `基址 + 偏移量 * 8`。
- **好处**：节省 50% 的引用内存占用，显著提升 CPU 缓存命中率（Cache Line 能装下更多数据）。
- **限制**：寻址范围受限于 32 位偏移量。当堆内存超过 32GB 时，32 位偏移量不够用，压缩自动失效，指针变回 8 字节，对象可能变大使堆内存进一步吃紧。

【对象内存布局】
```text
┌──────────────────────────────────────┐
│          Object Header               │
│  ┌─────────┬─────────┬───────────────┐ │
│  │ Mark Word (8B) │ Class Ptr (4B)  │ │ (开启压缩时 12B)
│  └─────────┴─────────┴───────────────┘ │
├──────────────────────────────────────┤
│         Instance Data                │
│  (Fields, 可能因重排序顺序不同)        │
├──────────────────────────────────────┤
│           Padding                    │
│ (补齐至 8 字节的整数倍，无实际意义)      │
└──────────────────────────────────────┘
```

## 常见考点
1. **对象头的 Mark Word 结构**：需能画出在不同锁状态（无锁、偏向锁、轻量级锁、重量级锁）下 Mark Word 的内容分布（如 last 2 bits 存锁标志）。
2. **TLAB 的生命周期**：TLAB 是线程私有的，在 Eden 区分配。线程创建时分配，GC 时回收。若 TLAB 空间不足，会重新申请一个新的 TAB 或直接在 Eden 区分配（慢路径）。
3. **数组对象头**：数组对象头比普通对象多 4 字节用于存储数组长度（Mark Word + Class Ptr + Array Length）。
4. **逃逸分析**：如果对象的作用域仅限于方法内且未发生逃逸，JIT 编译器可能会优化，将其直接分配在栈上（栈上分配）甚至进行标量替换，根本不创建堆对象，从而降低 GC 压力。
