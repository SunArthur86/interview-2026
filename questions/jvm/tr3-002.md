# ZGC Generational（分代）模式相比非分代模式，在GC停顿时间和吞吐量上做了哪些底层优化？

ZGC原本是单一区域，高分配率下需频繁扫描整个堆。Generational ZGC（JDK 21+）将堆分代，核心优化：1. 针对年轻代频繁消亡特点，只扫描年轻代，大幅减少扫描引用和重定位集大小；2. 保留基于读屏障和染色指针的并发整理，避免碎片整理的Stop-The-World。该设计在保持亚毫秒级停顿同时，显著降低CPU负载提升吞吐量，适合高并发、高分配率场景。

## 技术原理

- **堆物理分代，独立回收年轻代，避免全堆扫描**：非分代 ZGC 把整个堆当一块区域管，每次 GC 都要扫描全部对象找存活集，高分配率下 GC 频繁且扫描成本随堆线性增长。Generational ZGC（JDK 21 默认，JEP 439）把堆物理切分为年轻代（Young）和老年代（Old），利用"新生代朝生夕死"的弱分代假说——绝大多数新对象很快消亡，所以高频 Minor GC 只扫描年轻代（容量小），老年代用低频 Major GC 处理。扫描引用和重定位集大幅缩小，CPU 开销显著下降。
- **保留读屏障与染色指针，实现并发整理**：ZGC 的核心科技是**染色指针**（在 64 位指针的高位 bit 编码 Marked0/Marked1/Remapped/Finalizable 等状态）和**读屏障**（每次读对象引用时检查指针染色并按需转发）。这让对象移动（整理碎片）可以和应用线程**并发执行**——移动时只需更新指针染色，读屏障保证应用总能拿到正确地址。分代模式保留这套机制，所以 STW 仍维持亚毫秒级（< 1ms），不受堆大小影响。
- **降低 CPU 开销，提升吞吐量，停顿维持亚毫秒级**：非分代 ZGC 虽然停顿低但 CPU 开销高（频繁全堆扫描+并发整理占算力），分代把高频回收限制在年轻代，整体 GC 工作量减少，吞吐量提升。这是分代相对非分代的核心收益——同样的低停顿下吞吐更好，而非停顿更低。

## 对比/选型

| 维度 | 非分代 ZGC | Generational ZGC（JDK 21+） |
|------|------------|-------------------------------|
| 堆分区 | 单区域 | 年轻代 + 老年代 |
| GC 扫描范围 | 全堆 | 仅年轻代（高频）|
| CPU 开销 | 高 | 低 |
| 吞吐量 | 中 | 高（提升约 10-20%） |
| 停顿 | <1ms | <1ms |
| 写屏障 | 无（仅读屏障） | 增加写屏障维护记忆集 |

## 命令演示

JDK 21+ 启用分代 ZGC（默认已开启）：

```bash
# JDK 21+ 默认就是分代 ZGC
java -XX:+UseZGC -Xmx32g -Xlog:gc*:file=gc.log -jar app.jar

# JDK 21 显式指定（兼容老参数）
java -XX:+UseZGC -XX:+ZGenerational -Xmx32g -jar app.jar

# JDK 17 可以试用（非默认）
java -XX:+UseZGC -XX:+ZGenerational -Xmx32g -jar app.jar
# JDK 21 之后 -XX:+ZGenerational 成为默认，JDK 23+ 移除非分代模式
```

观察 GC 日志：

```
[gc] GC(42) Pause Young Start ...
[gc] GC(42) Pause Young Mark Start 0.023ms
[gc] GC(42) Pause Young Relocate Start 0.041ms
[gc] GC(42) Pause Young 1M->0M(512M) 0.823ms    ← 停顿亚毫秒
```

## 常见坑/注意事项

- **JDK 版本决定是否默认分代**：JDK 21 默认开启分代，JDK 17 需手动加 `-XX:+ZGenerational`（实验性），JDK 23+ 完全移除非分代模式。升级前确认。
- **写屏障的额外开销**：分代模式需要写屏障维护"老年代指向年轻代"的记忆集，写密集场景可能比非分代稍慢。但绝大多数场景年轻代收益远超写屏障开销。
- **大对象直接进老年代**：ZGC 分代对大对象有特殊处理（避免在年轻代频繁复制），但极端大对象流（如缓存）可能让老年代压力大，需监控。
- **亚毫秒停顿不等于零停顿**：仍有 <1ms 的 STW（初始标记、再标记等几个阶段），对要求绝对零延迟的场景（如金融交易）仍需评估。JDK 21+ 的 ZGC 已把 STW 压到几乎不可感知。
- **堆太小不建议用 ZGC**：ZGC 优势在于大堆（>4GB）的低停顿，小堆（<2GB）用 G1 更划算，ZGC 的元数据开销占比反而高。
- **预热很重要**：ZGC 启动初期会学习应用的对象分配模式，前几分钟吞吐可能略低，稳态后性能最佳。压测要充分预热。
