# 分代收集算法的核心思想是什么？

### 分代收集算法的核心思想

分代收集算法是目前大部分JVM垃圾收集器采用的策略。其核心思想是：**根据对象存活周期的不同，将内存划分为不同的区域（分代），针对不同生命周期的对象采用不同的垃圾收集算法，从而提高垃圾收集的效率。**

通常将Java堆划分为**新生代**和**老年代**。

**1. 新生代**
- **特点**：每次垃圾收集时都有大量对象死去，只有少量存活。
- **策略**：采用**复制算法**。
- **实现**：将内存分为一块较大的Eden区和两块较小的Survivor区。每次使用Eden区和其中一块Survivor区。回收时，将Eden和Survivor中存活的对象一次性复制到另一块Survivor区，然后清理掉Eden和刚才用过的Survivor区。
- **优点**：只需复制少量存活对象，效率高，且不会产生内存碎片。

**2. 老年代**
- **特点**：对象存活率高，没有额外的空间进行分配担保。
- **策略**：采用**标记-整理算法**或**标记-清除算法**。
- **原因**：老年代中存活对象多，如果使用复制算法，需要有一半的内存空间做预留，代价太高；使用标记-整理可以有效利用内存并减少碎片。

**对象晋升机制**
对象一般在新生代分配。如果对象在Survivor区中熬过了一定次数的GC（默认15次），就会被移动到老年代中。此外，如果Survivor区空间不足，大对象也可能直接通过分配担保机制进入老年代。

**架构示意图**
```
┌─────────────────────────────────────────────────────────────┐
│                        Java Heap                            │
├──────────────────────────────┬──────────────────────────────┤
│          Young Gen           │          Old Gen             │
│   (复制算法 Copying)         │   (标记-整理/清除)           │
├──────────────┬───────────────┤                              │
│    Eden      │  Survivor 0/1 │                              │
│  (80%)       │    (10%)      │      (存活率高)              │
│              │               │                              │
│ 对象分配/回收│ 晋升中转站    │   长期存活对象 / 大对象      │
└──────────────┴───────────────┴──────────────────────────────┘
```

### 4. 实战深化

**实战案例**：在促销活动场景下，系统瞬间产生大量短期订单对象（新生代），同时也存在缓存的本地用户信息（老年代）。若未根据内存比例调整新生代大小（默认 1:2），会导致频繁 Full GC。优化方案是调大新生代比例（如 `-XX:NewRatio=1`），让短期对象在 Young GC 中被回收，减少进入老年代的流量，降低 CMS/G1 的 Old GC 频率。

**对比表格**：

| 维度 | 新生代 | 老年代 |
| :--- | :--- | :--- |
| **存活率** | 低 (朝生夕死) | 高 (长期存活/缓存) |
| **GC 算法** | 复制算法 | 标记-清除 / 标记-整理 |
| **GC 类型** | Minor GC / Young GC | Major GC / Full GC |
| **GC 速度** | 快 (空间小，存活少) | 慢 (空间大，需整理/扫描) |
| **内存碎片** | 无 (复制算法特性) | 可能有 (标记-清除) 或 无 (标记-整理) |

**代码示例 (JVM 参数调优示例)**：
```bash
# 设置新生代与老年代比例为 1:2 (默认新生代占1/3)
-XX:NewRatio=2

# 设置 Eden 与 Survivor 区比例为 8:1
-XX:SurvivorRatio=8

# 设置大对象直接进入老年代的阈值 (避免在Survivor间复制)
# 例如：超过 1MB 的对象直接进老年代
-XX:PretenureSizeThreshold=1048576
```

**## 常见考点**
1. **动态年龄判定**：除了达到年龄阈值，如果Survivor区中相同年龄所有对象大小的总和大于Survivor空间的一半，年龄大于或等于该年龄的对象就可以直接进入老年代，无需等到MaxTenuringThreshold。
2. **空间分配担保**：在发生Minor GC前，JVM会检查老年代最大可用的连续空间是否大于新生代所有对象的总空间。如果大于，Minor GC确保安全；如果不大于，则检查HandlePromotionFailure是否允许担保失败，如果不允许或虽然允许但小于历史晋升平均值，则改为Full GC。
3. **大对象直接进入老年代**：参数-XX:PretenureSizeThreshold的作用及对性能的影响（避免大对象在Eden和Survivor间发生大量内存复制）。
