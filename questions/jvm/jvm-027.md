# 什么是JAVA IO/NIO？

Java IO/NIO 涵盖了不同的 I/O 模型和实现方式。

### 1. IO 模型分类

*   **阻塞 IO (BIO)**：
    *   **原理**：线程发起读写请求后，一直阻塞等待直到操作完成。
    *   **特点**：简单，但并发能力受限（一连接一线程）。

*   **非阻塞 IO**：
    *   **原理**：线程发起请求后立即返回，若未就绪则不断轮询，直到数据就绪。
    *   **特点**：线程不阻塞，但忙轮询耗费 CPU 资源。

*   **多路复用 IO (NIO)**：
    *   **原理**：利用 Selector（选择器）注册多个通道，由内核轮询监控。仅当通道有就绪事件（如读就绪）时，才通知线程进行处理。
    *   **特点**：一个线程管理多个连接，适合高并发。

*   **信号驱动 IO**：
    *   **原理**：线程发起请求并注册信号函数，继续执行；数据就绪时内核发送信号，线程在信号处理函数中执行 IO。

*   **异步 IO (AIO)**：
    *   **原理**：线程发起请求后立即返回，不阻塞。内核负责等待数据准备和拷贝，全部完成后通知线程（回调）。
    *   **特点**：真正的异步，两个阶段都不阻塞。

### 2. Java NIO 核心组件

1.  **Buffer（缓冲区）**：
    *   NIO 数据读写的中转站。本质上是一个数组（如 ByteBuffer），提供指针（position, limit, capacity）来管理读写状态。
2.  **Channel（通道）**：
    *   数据的源头或目标，全双工（双向读写）。数据必须通过 Buffer 与 Channel 交互。
3.  **Selector（选择器）**：
    *   多路复用的核心。用于检测多个 Channel 上是否有事件发生（如连接、读取），从而实现单线程管理多连接。

### 实战案例
早期使用 Tomcat 的 BIO 模式处理高并发请求时，经常因线程数达到上限（默认200）导致“Connection Refused”。切换至 NIO 模式（修改 `protocol="org.apache.coyote.http11.Http11NioProtocol"`）后，线程数降低，吞吐量提升3倍以上。

### 代码示例
```java
// BIO 典型代码：一连接一线程
ServerSocket serverSocket = new ServerSocket(8080);
while (true) {
    // 阻塞等待连接
    Socket socket = serverSocket.accept();
    // 为每个连接新建线程，资源消耗大
    new Thread(() -> {
        try (InputStream input = socket.getInputStream()) {
            // 阻塞读取数据
            byte[] buf = new byte[1024];
            input.read(buf);
        } catch (IOException e) {}
    }).start();
}
```

### 详细对比表
| 维度 | BIO (Blocking IO) | NIO (Non-blocking / New IO) | AIO (Asynchronous IO) |
| :--- | :--- | :--- | :--- |
| **Java 类** | `InputStream`, `OutputStream` | `Channel`, `Buffer`, `Selector` | `AsynchronousSocketChannel` |
| **核心机制** | 流 | 缓冲区 + 通道 + 选择器 | 回调 / Future |
| **阻塞特性** | 阻塞 | 非阻塞 (可设为阻塞) | 异步 (完全不阻塞) |
| **线程模型** | 1:1 (连接:线程) | M:N (多路复用，少量线程) | 操作系统内核回调 |
| **适用场景** | 连接数小，简单架构 | 高并发，轻量级操作 | 连接数极多，IO耗时极长 |
| **性能瓶颈** | 线程上下文切换 | Selector 处理复杂性 / 编码复杂 | 操作系统支持 (Linux下AIO成熟度不如NIO) |


## 记忆要点
- 核心定义：JIT将运行频繁的热点代码动态编译缓存为本地机器码，大幅提升执行效率。
- 混合模式：启动时靠解释器零延迟响应，稳定后靠JIT编译优化峰值性能。
- 热点探测：基于方法调用或回边计数器，触发OSR栈上替换直接转换执行中的循环。
- C1与C2分层：C1追求快速简单编译，C2负责耗时但深度的逃逸分析与极致优化。
