# Java 内存模型（JMM）的 happens-before 原则是什么？它如何保证多线程可见性？

【JMM 核心问题】多线程环境下，一个线程的写操作对另一个线程何时可见？JMM 定义了 happens-before 规则来回答这个问题。它是判断数据是否存在竞争、线程是否安全的主要依据，也是 Java 内存模型封装底层硬件指令的体现。

【happens-before 8 大规则】
1. **程序顺序规则**：单线程内，代码书写顺序在前面的操作 happens-before 写在后面的操作。（注：重排序不改变单线程执行结果）
2. **volatile 变量规则**：对一个 volatile 变量的写操作 happens-before 后续对这个 volatile 变量的读操作。
3. **锁规则**：一个 unlock 操作 happens-before 后续对同一把锁的 lock 操作。
4. **线程启动规则**：Thread 对象的 start() 方法 happens-before 该线程的每一个动作。
5. **线程终止规则**：线程中的所有操作都 happens-before 其他线程从该线程的 join() 方法成功返回。
6. **线程中断规则**：对线程 interrupt() 方法的调用 happens-before 被中断线程的代码检测到中断事件的发生。
7. **对象初始化规则**：构造函数执行结束 happens-before finalize() 方法的开始。
8. **传递性**：如果 A happens-before B，且 B happens-before C，那么 A happens-before C。

【happens-before 的本质与内存屏障】
happens-before 不是说「时间上先执行」，而是说「前一个操作的结果对后一个操作可见，且前一个操作的执行顺序排在后一个操作之前」。
JMM 通过**内存屏障**来禁止特定类型的处理器重排序来实现这些规则。主要有四类屏障：
- LoadLoad：禁止读操作重排序（Load1 < Load2）。
- StoreStore：禁止写操作重排序（Store1 < Store2）。
- LoadStore：禁止读写重排序（Load1 < Store2）。
- StoreLoad：禁止写读重排序（Store1 < Load2），开销最大。

【实例】双重检查单例（DCL）必须用 volatile：
`instance = new Singleton();` 在字节码层面分为 3 步：
1. 分配对象内存空间
2. 初始化对象
3. 将引用指向分配的内存地址

不加 volatile 时，步骤 2 和 3 可能发生重排序（2 和 3 对单线程无影响）。若线程 A 执行了 1->3（引用非空但未初始化），线程 B 此时判断 `instance != null` 直接拿去使用，就会报错或访问到错误数据。volatile 通过 StoreStore 屏障禁止了这种重排。

## 常见考点
1. **as-if-serial 语义**：单线程内不管怎么重排序，执行结果不能改变，这也是 happens-before 规则保持单线程正确性的基础。
2. **volatile 的可见性实现**：追问 volatile 的实现原理，需回答「MESI 缓存一致性协议 + Lock 前缀指令（总线嗅探）」。
3. **双重检查锁定（DCL）细节**：为什么需要 volatile？不仅为了可见性，更是为了禁止指令重排序。
4. **final 域的 happens-before**：对象构造函数内对 final 域的写入， happens-before 于对象引用被其他线程读到的时刻（final 的安全发布）。

## 记忆要点

- 本质定义：不是指时间上先执行，而是前一个操作的结果对后一个操作保证可见
- 传递性：如果 A 发生于 B 前，且 B 发生于 C 前，那么 A 发生于 C 前
- 规则口诀：程序次序、volatile 变量、锁、线程启动/终止、传递性
- DCL 细节：双重检查单例必须加 volatile，不仅为可见性，更为了禁止指令重排序
