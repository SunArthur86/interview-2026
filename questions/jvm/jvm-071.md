# CMS和G1有什么区别？如何选择？

CMS 和 G1 是 JVM 发展史上两个非常重要的里程碑收集器，它们的设计目标有显著差异。

| 特性 | CMS (Concurrent Mark Sweep) | G1 (Garbage-First) |
| :--- | :--- | :--- |
| **核心算法** | 标记-清除 | 标记-复制 (逻辑上是标记-整理) |
| **内存结构** | 物理隔离的新生代和老年代 | 化整为零的 Region 块（逻辑分代） |
| **内存碎片** | **有碎片**（因为是清除算法） | **无碎片**（复制整理天然规整） |
| **停顿预测** | **不可预测**（尽力而为） | **可预测**（-XX:MaxGCPauseMillis） |
| **适用堆大小** | 较小堆（通常 < 6GB） | 大堆（推荐 6GB - 32GB+） |
| **并发标记** | 三色标记 + 增量更新 | 三色标记 + SATB |
| **最终状态** | JDK 9 弃用，JDK 14 移除 | JDK 9 起成为默认 GC |
| **浮动垃圾处理** | 增量更新（重新扫描脏卡） | SATB（快照，吞吐量优） |

**架构对比图**：
```text
CMS 布局 (物理分代):        G1 布局 (Region 离散):
┌───────────────────┐        ┌─────┬─────┬─────┐
│    Old Gen (整块) │        │ E   │ E   │ O   │
│   (产生碎片)      │        ├─────┼─────┼─────┤
├───────────────────┤        │ E   │ O   │ O   │
│    Young Gen      │        ├─────┼─────┼─────┤
│ (Eden/Survivor)   │        │ S   │ O   │ H   │
└───────────────────┘        └─────┴─────┴─────┘
                             E:Eden S:Survivor O:Old H:Humongous
```

**选择建议**：
1. **小堆（< 4GB）且追求极致低延迟**：
   如果仍在使用 JDK 8，可以选择 CMS（需警惕碎片化），或者直接使用 ZGC（JDK 11/15+）。如果是追求吞吐量，Parallel GC 更好。
   
2. **中大堆（4GB - 32GB）**：
   **首选 G1**。它在吞吐量和延迟之间取得了很好的平衡，且 JDK 9+ 默认支持，稳定可靠。

3. **超大堆（> 32GB）或 亚毫秒级延迟**：
   **选择 ZGC**（JDK 17+ 推荐）。ZGC 打破了堆大小对停顿时间的影响。

4. **吞吐量优先（后台计算、批处理）**：
   **选择 Parallel Scavenge + Parallel Old**。

### 实战案例
在从 JDK 8 升级到 JDK 11 的过程中，某电商订单服务因保留 CMS 参数导致启动报错。迁移到 G1 后，虽然平均停顿时间降低了，但在秒杀场景下因对象分配速率过快，导致 G1 的并发标记周期未结束就触发了 Full GC（Evacuation Failure）。通过调大 `-XX:InitiatingHeapOccupancyPercent` 提前触发回收解决了该问题。

### 关键代码配置
```bash
# CMS 切换 G1 核心变更
# 旧版 (CMS)
-XX:+UseConcMarkSweepGC -XX:CMSInitiatingOccupancyFraction=70 

# 新版 (G1)
-XX:+UseG1GC -XX:MaxGCPauseMillis=200 -XX:InitiatingHeapOccupancyPercent=45
```

## 常见考点
1. CMS 在什么情况下比 G1 好？
   在非常小的堆（如 2GB 以下）且对象生命周期极短的场景下，CMS 的简单结构可能比 G1 的复杂 Region 管理略快，但 CMS 的碎片化风险始终存在。总体而言 G1 是 CMS 的官方替代者。
2. 为什么说 G1 是“可预测停顿”的？
   G1 有一个停顿时间模型，它会记录每个 Region 的回收历史耗时。在 Mixed GC 时，它会计算在指定时间内（如 200ms）回收哪些 Region 的收益最高（垃圾多且耗时少），从而满足用户设定的停顿目标。
3. G1 的 Mixed GC 和 Full GC 有什么区别？
   Mixed GC 只回收部分高价值的老年代 Region；Full GC 是单线程的、回收整个堆（包括新生代和老年代），是 G1 发生退化时的最后手段，性能极差，应极力避免。
4. 如何从 CMS 迁移到 G1？
   主要改动 JVM 参数：去掉 `-XX:+UseConcMarkSweepGC`，加入 `-XX:+UseG1GC`。移除 CMS 特有的参数（如 `-XX:CMSInitiatingOccupancyFraction`），调整为 G1 的 `-XX:MaxGCPauseMillis` 和 `-XX:InitiatingHeapOccupancyPercent`。
