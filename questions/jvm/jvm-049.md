# JVM对象的内存布局是怎样的？

### JVM对象的内存布局

在HotSpot虚拟机中，对象在堆内存中的存储布局可以分为三个部分：**对象头**、**实例数据**和**对齐填充**。

**1. 对象头**
对象头包括两类信息：
- **Mark Word**：用于存储对象自身的运行时数据，如哈希码、GC分代年龄、锁状态标志、线程持有的锁、偏向线程ID、偏向时间戳等。Mark Word根据锁状态不同，其存储的内容会发生变化（如无锁、轻量级锁、重量级锁）。
- **类型指针**：即对象指向它的类元数据的指针，虚拟机通过这个指针来确定这个对象是哪个类的实例。
- **数组长度**：如果对象是一个数组，那么对象头中还必须有一块用于记录数组长度的数据。

**2. 实例数据**
- 实例数据部分是对象真正存储的有效信息，即我们在程序代码里面所定义的各种类型的字段内容。
- 无论是从父类继承下来的，还是在子类中定义的字段，都需要记录下来。
- **存储顺序**：HotSpot虚拟机默认的分配策略为：longs/doubles、ints、shorts/chars、bytes/booleans、oop（Ordinary Object Pointers）。相同宽度的字段总是被分配到在一起。在满足这个前提条件的情况下，在父类中定义的变量会出现在子类之前。

**3. 对齐填充**
- 这部分并不是必然存在的，也没有特别的含义，它仅仅起着占位符的作用。
- **原因**：HotSpot虚拟机的自动内存管理系统要求对象起始地址必须是8字节的整数倍。换句话说，就是对象的大小必须是8字节的整数倍。
- 而对象头部分已经是8字节的倍数了（在32位系统中是4字节，但通常会被扩充），因此当对象实例数据部分没有对齐时，就需要通过对齐填充来补全。

### 关键细节补充
- **Mark Word 的动态变化**（64位JVM为例）：
  - **无锁状态**：25位unused + 31位HashCode + 1位unused(分代年龄) + 4位分代年龄 + 2位锁状态(01)。
  - **偏向锁**：54位线程ID + 2位Epoch + 1位分代年龄 + 4位锁状态标志(101)。
  - **轻量级锁**：62位指向栈中Lock Record的指针 + 2位锁状态(00)。
  - **重量级锁**：62位指向堆中Monitor对象的指针 + 2位锁状态(10)。
  - **GC标记**：62位空 + 2位锁状态(11)。
- **指针压缩**：在64位JVM中，启用指针压缩（`-XX:+UseCompressedOops`，默认开启）后，类型指针大小从8字节变为4字节，Mark Word保持8字节。这使得对象头通常为12字节，加上对齐填充变为16字节。
- **字段重排序**：HotSpot VM可能会对字段进行重排序以优化内存占用（如将相同类型的字段放在一起），但`@Contended`注解或`volatile`字段会影响布局。

### 对象内存布局结构图 (64位 JVM, 开启指针压缩)
```text
+------------------------+
|      Mark Word         |  (8 bytes)
|  (Hash, Age, Lock, ...) |
+------------------------+
|   Class Pointer *      |  (4 bytes, compressed) 若未压缩则8bytes
+------------------------+
|  Array Length (optional)| (4 bytes, 仅数组对象)
+------------------------+
|                        |
|    Instance Data       |  (字段内容)
|   (Fields, References...) |
|                        |
+------------------------+
|      Padding           |  (0~7 bytes, 保证总大小为8的倍数)
+------------------------+
```

### 实战进阶

**1. 实战案例**
在进行 **内存占用分析**（如排查内存溢出）时，如果一个只有少量字段的简单对象（如只包含一个 int 的 DTO）在堆 Dump 中占用显示为 16 字节而非预期的 12 字节（8头+4数据），这通常是因为**对齐填充**导致的。在千万级对象的缓存场景中，这种无用的 Padding 会造成巨大的内存浪费（每个对象浪费4字节）。可以通过开启 `-XX:+UseCompressedOops` 并合理调整字段顺序（如将所有 boolean 合并处理）来减少 Padding。

**2. 代码示例**
```java
public class LayoutDemo {
    // 开启指针压缩(-XX:+UseCompressedOops)时：
    // Header: 12 bytes (8 Mark + 4 Class)
    // int: 4 bytes
    // 总计: 16 bytes (无需Padding，刚好是8的倍数)
    private int id; 

    // 如果在此增加一个 boolean flag (1 byte)
    // Header(12) + int(4) + boolean(1) = 17 bytes
    // 对齐填充需要补齐到 24 bytes，浪费 7 bytes！
    private boolean flag; 
}
```
