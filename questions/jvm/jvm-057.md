# Minor GC、Major GC、Mixed GC和Full GC有什么区别？

①**Minor GC（Young GC）**：回收新生代（Eden + Survivor），触发频繁但速度快。Eden区满时触发。
②**Major GC（Old GC）**：回收老年代，速度比Minor GC慢10倍以上。通常是CMS收集器的行为。
③**Mixed GC**：G1收集器特有，回收整个新生代和部分老年代Region。
④**Full GC**：回收整个堆和方法区，最耗时。触发条件：老年代空间不足、空间分配担保失败、System.gc()调用、元空间不足。

**优化目标**：尽量减少Full GC频率。

**GC 范围与架构图示**：
```text
       JVM Heap Memory Layout
  ┌─────────────────────────────────┐
│  方法区/元空间 (Method Area)     │ ◄── Full GC 范围
  ├─────────────────────────────────┤
│      老年代 (Old Gen)            │ ◄── Major GC / Full GC 范围
  │  ┌───────┐ ┌───────┐ ┌───────┐ │
  │  │ Reg 1 │ │ Reg 2 │ │ Reg 3 │ │ ◄── Mixed GC (G1: 部分+Young)
  │  └───────┘ └───────┘ └───────┘ │
  ├─────────────────────────────────┤
│      新生代 (Young Gen)          │ ◄── Minor GC 范围
  │  ┌─────┐ ┌─────┐ ┌─────┐      │
  │  │Eden │ │ S0  │ │ S1  │      │
  │  └─────┘ └─────┘ └─────┘      │
  └─────────────────────────────────┘
```

**关键细节与说明**：
- **Major GC vs Full GC**：很多语境下 Major GC 指代 Full GC，但严格定义上 Major GC 仅清理老年代。目前只有 CMS 针对 Old Gen 进行单独回收叫 Major GC；其他收集器（如 Serial, Parallel）清理老年代时通常伴随新生代回收（即 Full GC）。
- **Mixed GC (G1 特有)**：G1 不物理隔离老年代，而是划分为多个 Region。Mixed GC 是根据 Global Concurrent Marking 阶段标记出的回收价值最高的 Region 进行回收。G1 的目的是避免 Full GC，但如果 Mixed GC 回收速度跟不上分配速度，会退化回 Full GC（Serial Old）。
- **GC 速度对比**：Minor GC (毫秒级) << Major GC/Mixed GC (百毫秒~秒级) < Full GC (秒级~分钟级)。

## 常见考点
1. **Major GC 和 Full GC 到底有什么区别？** 
   核心区别在于是否清理新生代。Major GC 只清理老年代（通常伴随 CMS，但也可能退化）；Full GC 清理整个堆（新生代+老年代+方法区）。在实际生产日志中，很多收集器老年代回收时都会日志显示为 Full GC。
2. **CMS 在什么情况下会发生 Concurrent Mode Failure 从而退化成 Full GC？** 
   CMS 是并发回收的。如果 CMS 回收期间，老年代预留的空间无法满足用户线程的新对象分配请求（即 "Promotion Failed" 或 "Concurrent Mode Failure"），就会触发 Full GC（Stop-The-World，使用 Serial Old 单线程）。
3. **G1 的 Mixed GC 一定会回收整个新生代吗？** 
   是的。Mixed GC 一定会回收整个新生代，同时回收一部分老年代 Region。这是 G1 的设计机制，为了配合新生代的 eden 区快速释放。

## 技术原理

- **Minor GC（Young GC）的复制算法**：新生代用复制算法——Eden 区满时触发，把 Eden + From Survivor 的存活对象复制到 To Survivor，清空原区域。默认 `Eden:Survivor = 8:1`，新生代利用率 90%。因为新生代对象"朝生夕死"，存活对象少，复制成本极低，Minor GC 通常几毫秒到几十毫秒。`-XX:MaxTenuringThreshold=15` 控制对象在 Survivor 中熬过多少次 Minor GC 后晋升老年代。
- **Major GC 的严格定义 vs 俗称**：严格定义上，**只有 CMS 收集器针对 Old Gen 的单独回收**才叫 Major GC。其他收集器（Serial/Parallel/G1/ZGC）清理老年代时往往伴随新生代回收，即 Full GC。但生产环境中很多工程师把"老年代 GC"都叫 Major GC，造成术语混淆。面试时强调"严格定义只有 CMS"能展示深度。
- **Mixed GC 是 G1 的"折中方案"**：G1 把堆切成 Region，不做物理分代隔离。Mixed GC 回收**整个 Young + 部分 Old**（垃圾占比高、回收收益大的 Region）。这是 G1 在"避免 Full GC"和"控制停顿"间的核心设计——通过选择性回收高价值 Region，把回收节奏切细，而非等到 Full GC 一锅端。
- **Full GC 是兜底但最痛**：触发条件包括①老年代空间不足；②空间分配担保失败（Minor GC 前检查老年代剩余 < 历次晋升平均大小）；③Metaspace 空间不足；④`System.gc()` 调用；⑤CMS Concurrent Mode Failure。Full GC 回收整堆+方法区，单线程（Serial Old）或极慢，停顿可达秒级甚至分钟级。**优化核心就是减少 Full GC**。

## 对比/选型

| GC 类型 | 回收区域 | 频率 | 单次耗时 | 触发条件 | 收集器 |
|---------|---------|------|---------|---------|--------|
| Minor GC | Young（Eden+S0+S1） | 高（毫秒~秒级间隔） | 毫秒级 | Eden 满 | 所有 |
| Major GC | Old（严格定义） | 低 | 百毫秒~秒级 | Old 占用高 | 仅 CMS |
| Mixed GC | Young + 部分 Old | 中（G1 特有） | 受 MaxGCPauseMillis 约束 | IHOP 阈值（默认 45%） | 仅 G1 |
| Full GC | 整堆 + Metaspace | 极低（应避免） | 秒~分钟级 | Old 不足/担保失败/System.gc | 所有 |

## 命令演示

```bash
# 关键参数
-XX:NewRatio=2                       # Old:Young = 2:1（让 Young 适中）
-XX:SurvivorRatio=8                  # Eden:S = 8:1（每个 Survivor 占 Young 1/10）
-XX:MaxTenuringThreshold=15          # 对象熬过 15 次 Minor GC 晋升 Old
-XX:PretenureSizeThreshold=1m        # 大于 1MB 的对象直接进 Old（跳过 Young）
-XX:+DisableExplicitGC               # 禁用 System.gc()（防强制 Full GC）
-XX:+UseG1GC                         # G1 收集器
-XX:MaxGCPauseMillis=200             # G1 目标停顿

# JDK 9+ 统一日志
-Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=10,filesize=50M

# 日志识别（关键字）
# "GC pause (G1 Evacuation Pause) (young)"    → Minor GC
# "GC pause (G1 Evacuation Pause) (mixed)"    → Mixed GC
# "Pause Full (G1 Compaction Pause)"          → Full GC（G1 退化）
# "Concurrent Mark Cycle"                     → 并发标记周期（Mixed GC 前置）
```

## 常见坑/注意事项

- **生产环境的 Full GC 必须告警**：Full GC 是"红色警报"，通常意味着系统要崩。监控 GC 日志，Full GC 频率 > 1次/分钟 或停顿 > 1秒需立即介入。Prometheus + Grafana 配置 `jvm_gc_pause_seconds_max` 告警。
- **大对象直接进 Old 是隐患**：`-XX:PretenureSizeThreshold` 让大对象跳过 Young 直接进 Old，频繁分配大对象（如 byte[1MB]）会让 Old 迅速填满触发 Full GC。要么调小阈值让大对象先进 Young，要么从代码减少大对象分配。
- **担保失败（Promotion Failed）**：Minor GC 前检查"老年代剩余空间 ≥ 历次晋升平均大小"，不满足则触发 Full GC。频繁担保失败说明 Young 设得太大或 Old 太小，调整 `-XX:NewRatio` 或直接堆扩容。
- **CMS 的 Concurrent Mode Failure**：CMS 并发回收期间老年代预留空间不足，退化成 Serial Old Full GC（单线程秒级停顿）。调 `-XX:CMSInitiatingOccupancyFraction=70` 让 CMS 更早启动，`-XX:+UseCMSInitiatingOccupancyOnly` 禁用自动触发阈值。
- **System.gc() 的坑**：NIO/反射/某些框架会调用 System.gc() 期望清理，但触发 Full GC。生产必加 `-XX:+DisableExplicitGC`，或 `-XX:+ExplicitGCInvokesConcurrent` 让它走并发回收而非 Full GC。
