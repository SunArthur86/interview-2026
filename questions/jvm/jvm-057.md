# Minor GC、Major GC、Mixed GC和Full GC有什么区别？

①**Minor GC（Young GC）**：回收新生代（Eden + Survivor），触发频繁但速度快。Eden区满时触发。
②**Major GC（Old GC）**：回收老年代，速度比Minor GC慢10倍以上。通常是CMS收集器的行为。
③**Mixed GC**：G1收集器特有，回收整个新生代和部分老年代Region。
④**Full GC**：回收整个堆和方法区，最耗时。触发条件：老年代空间不足、空间分配担保失败、System.gc()调用、元空间不足。

**优化目标**：尽量减少Full GC频率。

**GC 范围与架构图示**：
```text
       JVM Heap Memory Layout
  ┌─────────────────────────────────┐
│  方法区/元空间 (Method Area)     │ ◄── Full GC 范围
  ├─────────────────────────────────┤
│      老年代 (Old Gen)            │ ◄── Major GC / Full GC 范围
  │  ┌───────┐ ┌───────┐ ┌───────┐ │
  │  │ Reg 1 │ │ Reg 2 │ │ Reg 3 │ │ ◄── Mixed GC (G1: 部分+Young)
  │  └───────┘ └───────┘ └───────┘ │
  ├─────────────────────────────────┤
│      新生代 (Young Gen)          │ ◄── Minor GC 范围
  │  ┌─────┐ ┌─────┐ ┌─────┐      │
  │  │Eden │ │ S0  │ │ S1  │      │
  │  └─────┘ └─────┘ └─────┘      │
  └─────────────────────────────────┘
```

**关键细节与说明**：
- **Major GC vs Full GC**：很多语境下 Major GC 指代 Full GC，但严格定义上 Major GC 仅清理老年代。目前只有 CMS 针对 Old Gen 进行单独回收叫 Major GC；其他收集器（如 Serial, Parallel）清理老年代时通常伴随新生代回收（即 Full GC）。
- **Mixed GC (G1 特有)**：G1 不物理隔离老年代，而是划分为多个 Region。Mixed GC 是根据 Global Concurrent Marking 阶段标记出的回收价值最高的 Region 进行回收。G1 的目的是避免 Full GC，但如果 Mixed GC 回收速度跟不上分配速度，会退化回 Full GC（Serial Old）。
- **GC 速度对比**：Minor GC (毫秒级) << Major GC/Mixed GC (百毫秒~秒级) < Full GC (秒级~分钟级)。

## 常见考点
1. **Major GC 和 Full GC 到底有什么区别？** 
   核心区别在于是否清理新生代。Major GC 只清理老年代（通常伴随 CMS，但也可能退化）；Full GC 清理整个堆（新生代+老年代+方法区）。在实际生产日志中，很多收集器老年代回收时都会日志显示为 Full GC。
2. **CMS 在什么情况下会发生 Concurrent Mode Failure 从而退化成 Full GC？** 
   CMS 是并发回收的。如果 CMS 回收期间，老年代预留的空间无法满足用户线程的新对象分配请求（即 "Promotion Failed" 或 "Concurrent Mode Failure"），就会触发 Full GC（Stop-The-World，使用 Serial Old 单线程）。
3. **G1 的 Mixed GC 一定会回收整个新生代吗？** 
   是的。Mixed GC 一定会回收整个新生代，同时回收一部分老年代 Region。这是 G1 的设计机制，为了配合新生代的 eden 区快速释放。
