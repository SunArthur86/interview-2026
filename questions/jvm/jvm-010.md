# Java 基本类型和包装类型（Wrapper Class）有什么区别？

### 核心区别
**基本类型**：byte, short, int, long, float, double, char, boolean。它们是 Java 语言内置的、非面向对象的数据类型，直接存储数据值。
**包装类型**：Byte, Short, Integer, Long, Float, Double, Character, Boolean。它们是 Java 对应基本类型的类封装，位于 `java.lang` 包下，是引用类型。

| 维度 | 基本类型 | 包装类型 |
| :--- | :--- | :--- |
| **默认值** | 0 (int), 0.0 (double), false (boolean) | **null** |
| **存储位置** | **栈**（局部变量常量池）/ 方法区（静态final） | **堆**（对象实例） |
| **内存占用** | 极小（例如 int 占 4 字节） | 较大（包含对象头 Mark Word + 类型指针 + 实际数据） |
| **比较方式** | `==` 比较数值 | `==` 比较内存地址；`equals` 比较数值 |
| **泛型支持** | 不支持（不能用 `List<int>`） | 支持（`List<Integer>`） |
| **序列化** | 默认不支持 | 支持（实现 Serializable 接口） |

### 自动装箱与拆箱
Java 5 引入了自动装箱和拆箱，这是编译器在编译期间的语法糖，实际上还是调用了对应的方法。

```java
Integer a = 10;     // 装箱：编译器自动替换为 Integer.valueOf(10)
int b = a;          // 拆箱：编译器自动替换为 a.intValue()
List<Integer> list = new ArrayList<>();
list.add(1);        // 装箱
```

### 关键细节与陷阱

1. **Integer 缓存机制**
   `Integer` 类内部维护了一个静态缓存数组 `IntegerCache`，默认缓存了 -128 到 127 之间的所有 Integer 对象。
   - 在这个范围内，`valueOf()` 方法直接返回缓存中的引用，不会创建新对象。
   - 超出这个范围，`valueOf()` 会 `new Integer(...)`。
   ```java
   Integer i1 = 100; // 命中缓存
   Integer i2 = 100;
   System.out.println(i1 == i2); // true

   Integer i3 = 200; // 超出缓存
   Integer i4 = 200;
   System.out.println(i3 == i4); // false
   ```
   *注意：除了 Integer，Byte, Short, Long 也有缓存，范围是 -128 到 127；Character 缓存范围是 0 到 127；Boolean 缓存了 TRUE 和 FALSE。Float 和 Double 没有缓存。*

2. **空指针异常（NPE）**
   拆箱操作是调用包装对象的 `xxxValue()` 方法（如 `intValue()`）。如果包装类对象为 `null`，调用该方法会直接抛出 `NullPointerException`。
   ```java
   Integer x = null;
   int y = x; // 运行时抛出 NPE
   ```

3. **性能开销**
   包装类型是对象，在堆上分配内存。频繁的装箱和拆箱会产生大量临时对象，增加 GC 压力。在进行大量数学运算（如循环累加）时，应显式使用基本类型，避免在循环体内自动拆箱。

### 实战案例：数据库与 RPC 接口的坑
在对接数据库或上游 RPC 接口时，如果返回值定义为了 `Integer`，而业务代码中使用 `int` 接收，当数据库为 `NULL` 或接口返回 `null` 时，会直接触发 NPE 导致服务崩溃。在高并发场景下（如电商促销），这种 NPE 可能导致级联失败。

### 实战代码示例：安全比较工具类
为了避免装箱和 NPE 问题，工程中常封装工具类来处理包装类比较，强制使用 `equals` 或判空处理。

```java
public static boolean compareInteger(Integer a, Integer b) {
    if (a == null || b == null) {
        return a == b; // 两个都为 null 返回 true，否则 false
    }
    return a.equals(b); // 避免直接使用 ==，防止超出缓存范围比较出错
}

// 反面教材：坑点代码
public int sumScore(Integer a, Integer b) {
    return a + b; // 如果 a 或 b 为 null，这里会在自动拆箱时抛 NPE
}
```

- 只有包装类型支持泛型（如List<Integer>）和实现Serializable接口，基本类型无法直接使用。

## 延伸：解释器与编译器？

> 合并自 `jvm-011`（相似度 82%）

### 核心概念
即时编译器（JIT）：
当虚拟机发现某个方法或代码块的运行特别频繁，就会把这些代码认定为“热点代码”。为了提高热点代码的执行效率，在运行时，虚拟机将会把这些代码编译成本地机器码，并以各种手段尽可能地进行代码优化，运行时完成这个任务的后端编译器被称为即时编译器。

解释器与编译器：
主流的虚拟机内部都同时包含解释器与编译器，⼆者各有优势：当程序需要迅速启动和执行的时候，解释器可以首先发挥作用，省去编译的时间，立即运行。当程序启动后，随着时间的推移，编译器逐渐发挥作用，把越来越多的代码编译成本地代码，这样可以减少解释器的中间损耗，获得更高的执行效率。

编译器是将代码转换为本机机器码保存下来，需要占用内存。所以内存资源不多的时候用解释器执行，反之可以用编译器来提高程序执行速度和效率。同时，编译器的优化不能保证完全成功，一些激进优化措施是有失败的风险，此时就不能再用编译器，要退回到解释器状态继续执行。

### 编译模式对比
| 模式 | 描述 | 适用场景 |
| :--- | :--- | :--- |
| **混合模式** | 解释器与编译器搭配使用，平衡启动速度和执行效率 | **默认模式**，适合绝大多数通用应用 |
| **解释模式** | 仅解释器工作，编译器完全不工作 | 程序启动极快，但执行效率低，用于调试或快速验证 |
| **编译模式** | 优先采用编译器，但编译器无法进行时采用解释器 | 追求极致性能，且能够忍受较长启动时间的场景 |

### 分层编译
HotSpot 内置了“客户端编译器（C1）”和“服务端编译器（C2）”。为了在启动时间和峰值性能之间取得最佳平衡，JDK 7+ 默认开启了分层编译：
- **第0层**：程序纯解释执行，不开启性能监控。
- **第1层**（C1）：将字节码编译为本地代码，进行简单可靠的优化，不开启性能监控。
- **第2层**（C1）：执行编译后的代码，开启**有限**的性能监控（如方法调用、回边次数）。
- **第3层**（C1）：执行编译后的代码，开启**全部**性能监控（收集分支跳转、类型继承等激进优化所需的统计信息）。
- **第4层**（C2）：使用服务端编译器，根据第3层收集的性能监控信息进行激进优化，生成最高效的机器码。

解释器执行的启动速度是比编译器快的，所以程序先在第0层用解释器执行。而 C1 编译器启动要比 C2 快，但 C2 编译出的代码执行效率更好。

### 实战案例：由 C2 优化陷阱引发的生产事故
在某高性能计算服务中，一段复杂的哈希计算逻辑在运行数小时后突然出现精度错误。排查发现是 C2 编译器在激进优化时错误地消除了某个关键的 null 检查。这种由于激进优化导致的 rare bug，通常通过增加 JVM 参数 `-XX:-EliminateAllocations` 或升级 JDK 版本来修复。这也说明了分层编译中保留解释器作为“安全网”的重要性。

### 实战代码示例：观察 JIT 编译状态
可以通过 JMX 或代码动态查看某个方法是否已经被编译（C1 或 C2），这对于排查性能瓶颈非常有用。

```java
import com.sun.management.HotSpotDiagnosticMXBean;
import java.lang.management.ManagementFactory;

public void checkCompilationStatus() {
    HotSpotDiagnosticMXBean mxBean = ManagementFactory.getPlatformMXBean(HotSpotDiagnosticMXBean.class);
    // 通过 VM Option 获取编译日志，实际生产中常开启 -XX:+PrintCompilation 
    // 以下是模拟获取编译策略的伪代码逻辑
    String vmVersion = mxBean.getVMOption("VMVersion").getValue();
    System.out.println("Current JVM Version: " + vmVersion);
    
    // 实战中常用 jstat -compiler <pid> 查看 JIT 编译统计
}
```

## 记忆要点

- 各取所长：解释器负责启动快无编译开销，编译器（JIT）负责将热点代码编译为本地机器码以追求峰值性能。
- 因为激进优化可能失败，所以当编译优化出错时，可直接退回解释器继续执行作为安全网兜底。
- 分层编译模型：C1编译器负责快速启动并收集信息，C2编译器根据信息进行激进优化追求极致性能。
- 默认混合模式：程序由解释器先执行，随运行时间推移，JIT逐渐把越来越多的热点代码编译优化。
