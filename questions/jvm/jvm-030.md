# 什么是Java 内存模型？

Java 内存模型（JMM）是一种抽象的概念，**并不等同于 JVM 运行时数据区**。它的主要目的是为了屏蔽不同硬件和操作系统的内存访问差异，保证 Java 程序在各平台下达到一致的并发效果。

### 1. JMM 抽象结构
JMM 规定了所有的变量都存储在**主内存**（Main Memory）中。
每个线程还有自己的**工作内存**（Working Memory，对应 CPU 缓存/寄存器），工作内存中保存了该线程使用到的变量的主内存副本拷贝。

线程对变量的所有操作（读取、赋值）都必须在工作内存中进行，而不能直接读写主内存。

**内存交互流程图：**
```text
   ┌──────────────┐      1.lock/unlock      ┌──────────────┐
   │    线程 A    │ ──────────────────────> │              │
   └──────┬───────┘                         │              │
          │ 2.read (从主存传送到工作内存)    │   主内存     │
          v                                 │  (共享变量)  │
   ┌──────────────┐                         │              │
   │ 工作内存 A   │ ──────────────────────> │              │
   │ (副本)       │  3.load/work/use/store  └──────────────┘
   └──────────────┘      4.write (回写主存)       ^
          ▲                                        │
          │ 5.assign (赋值)                         │
   ┌──────┴───────┐                                │
   │    线程 B    │ <─────────────────────────────┘
   └──────────────┘
```
**关键细节**：
- **原子性变量**：针对 `long` 和 `double` 型变量，JMM 要求 load、store、read、write 操作具有原子性（虽然大部分 JVM 实现已保证，但规范允许非原子性，推荐使用 `volatile` 修饰）。
- **指令重排序**：为了优化性能，编译器和处理器会对指令进行重排序，但 JMM 通过**内存屏障**（Memory Barriers）禁止特定类型的重排序（如 StoreLoad 屏障）。

### 2. 三大特性
1.  **原子性**：一个操作是不可中断的，要么全部执行成功要么全部不执行。
    - *补充*：JMM 只保证基本数据类型的读写和 `volatile` 变量的读写是原子性的，类似 `i++` 这种复合操作不保证原子性，需依赖 `synchronized` 或 `Atomic` 类。
2.  **可见性**：当一个线程修改了共享变量的值，其他线程能够立即看到修改的值。
    - *补充*：`synchronized` 和 `final` 也能保证可见性。`final` 字段在构造器内一旦初始化完成，且构造器未将 `this` 引用逃逸，则对其他线程可见。
3.  **有序性**：指令序列在执行过程中，不会因为重排序而导致逻辑错误。
    - *补充*：如果在线程内观察，所有操作都是有序的；如果在多线程中观察，操作则是无序的。`volatile` 关键字通过插入内存屏障禁止特定重排序。

### 3. 先行发生原则
JMM 中定义的一套规则，用于判断数据是否存在竞争、线程是否安全。如果不满足以下规则，就需要通过加锁或 volatile 来保证：
-   程序次序规则
-   管程锁定规则
-   volatile 规则
-   线程启动规则
-   线程终止规则
-   线程中断规则
-   对象终结规则
-   传递性

### 实战案例：DCL 双重检查锁问题
在单例模式的双重检查锁定（DCL）中，若不加 `volatile`，由于指令重排序，实例的初始化（对象半初始化）可能发生在引用赋值之后，导致其他线程获取到未初始化完全的对象。必须通过 `volatile` 禁止对象初始化与引用赋值的重排序。

### 关键代码示例：Volatile 误用修正
```java
public class Counter {
    // 场景：多线程环境下的计数
    // 错误示例：volatile 无法保证 i++ 的原子性
    private volatile int wrongCount = 0; 

    // 正确示例：使用 Atomic 类保证原子性
    private AtomicInteger rightCount = new AtomicInteger(0);

    public void increment() {
        wrongCount++; // 非原子操作：读-改-写，线程不安全
        rightCount.incrementAndGet(); // 原子操作，线程安全
    }
}
```

### JMM 特性对比表
| 特性 | 关键字/机制 | 保证内容 | 典型应用场景 | 注意事项 |
| :--- | :--- | :--- | :--- | :--- |
| **原子性** | synchronized, Atomic, Lock | 操作不可中断 | 计数器、转账、状态变更 | volatile 仅保证单次读/写原子性，不保证复合操作（如i++） |
| **可见性** | volatile, synchronized, final | 变量修改对其他线程立即可见 | 状态标志位（如 running flag）、发布对象 | synchronized 解锁时会刷新内存到主存，加锁时会从主存读取 |
| **有序性** | volatile, synchronized | 禁止特定类型指令重排序 | 单例模式 DCL、观察者模式 | Happens-Before 原则是判断有序性的核心依据 |

## 常见考点
1. **Volatile 的实现原理**：追问如何通过内存屏障保证可见性和禁止指令重排（Lock 前缀指令、汇编层面的分析）。
2. **Synchronized 与 JMM 的关系**：追问 synchronized 的语义（原子性、可见性）在 JMM 中是如何体现的（Monitor Enter/Exit 规则）。

## 记忆要点

- 核心对比：JVM采用基于栈的指令集实现跨平台，而物理机多采用基于寄存器追求高性能。
- 栈架构特点：指令紧凑代码体积小，但需频繁入栈出栈，导致执行与内存访问效率偏低。
- 执行流程：执行引擎按程序计数器取指令，解码后作用于虚拟机栈中的操作数执行。
