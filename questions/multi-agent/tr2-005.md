# 在 Multi-Agent 系统中，CrewAI 的“角色扮演”模式与 LangGraph 的“图状态”模式在构建协作逻辑时有什么本质区别？

CrewAI 基于角色扮演的预设逻辑，核心是将 Agent 定义为具有特定角色、目标和背景故事的“员工”，通过顺序或层级的过程来协作，适合结构化明确、分工清晰的业务流（如营销团队模拟）。其逻辑相对固定，依赖预定义的 Task 链。LangGraph 则基于有向循环图和全局状态机，核心是定义状态如何在节点间流转和转换。它更加灵活，支持复杂的条件跳转、回退和动态分支，适合构建需要根据中间结果动态改变执行路径的应用（如复杂的客服审批流）。简言之，CrewAI 更像排好的剧本，LangGraph 更像一套可编程的逻辑控制器。

### 深化内容

**实战案例**：在构建代码审计系统时，CrewAI 容易因为某一个静态任务报错导致整个流程中断，必须手动重试；而利用 LangGraph 的图结构，我实现了“代码审查失败自动回退到开发节点要求修改”的逻辑，不仅容错率高，还能根据审查结果动态决定是继续测试还是驳回。

**代码示例**：
```python
# LangGraph 条件边实现：根据状态决定下一步路由
def should_review(state: AgentState):
    if state["error_count"] > 3:
        return "escalate_to_human"  # 动态跳转人工
    return "auto_fix"              # 继续自动修复

workflow.add_conditional_edges("review_node", should_review, {
    "auto_fix": "fixer_node",
    "escalate_to_human": "human_node"
})
```

**对比表格**：

| 特性维度 | CrewAI (角色扮演) | LangGraph (图状态机) |
| :--- | :--- | :--- |
| **核心逻辑** | 顺序执行的预设任务链 | 循环图与状态流转机 |
| **动态能力** | 弱，主要依赖静态配置 | 强，支持运行时条件分支和回退 |
| **适用场景** | 职责明确的流水线（如内容生成、翻译） | 复杂决策流（如客服审批、自动化运维） |
| **数据共享** | 通过 Task 输出传递 | 全局共享 State，适合多轮记忆累积 |
| **调试难度** | 较低，流程线性可视 | 较高，需追踪节点间状态变化 |

## 技术原理

两者的本质差异在于「控制流模型」——CrewAI 是命令式编排，LangGraph 是声明式状态机：

- **CrewAI 的角色编排模型**：CrewAI 的核心抽象是 `Agent`（角色）+ `Task`（任务）+ `Process`（流程）。流程决定了 Task 在 Agent 间的流转方式：Sequential 按列表顺序执行（前一个 Task 的 output 作为后一个的 input），Hierarchical 由 Manager Agent 动态分发。控制流在「编写代码时」就已确定，运行时不会改变。每个 Agent 内部用 ReAct 循环（思考→行动→观察）完成任务，但 Agent 之间的协作路径是静态的。这让它简单易用，但遇到「需要根据中间结果改变后续路径」的场景就显得僵化。
- **LangGraph 的状态机模型**：LangGraph 把 Agent 系统建模为有向图——节点（Node）是处理单元（可以是 Agent、工具、判断函数），边（Edge）是状态流转。核心是全局 `State` 对象（TypedDict），每个节点接收 State、修改 State、返回更新。条件边（Conditional Edge）让运行时根据 State 内容动态决定下一个节点，支持 if/else 分支、循环（如「审查失败回到修复节点重试」）、甚至人工介入（human-in-the-loop）。这是图灵完备的控制流，能表达任意复杂逻辑。
- **数据传递的根本差异**：CrewAI 通过 Task 的 `context` 参数显式传递前序输出，每个 Agent 只看到指定上下文，数据流是点对点的。LangGraph 用全局共享 State，所有节点读写同一个 State，天然支持累积式记忆（如多轮对话历史、累计的错误次数）。后者更适合需要「全局视角」的复杂决策。
- **执行引擎**：CrewAI 底层依赖 LangChain 的 AgentExecutor，每个 Agent 独立运行。LangGraph 基于 LangChain 但引入了 Pregel（图计算框架）的思想，支持节点并行执行、状态 checkpoint（断点恢复）、流式输出等高级特性。

## 代码示例

```python
# === LangGraph 状态机：带条件分支与回退的代码审查流 ===
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator

class CodeReviewState(TypedDict):
    code: str
    review_result: str
    error_count: int
    iterations: Annotated[list[str], operator.add]  # 累积式记忆

def reviewer_node(state: CodeReviewState):
    result = reviewer_agent.invoke(state["code"])
    return {"review_result": result, "iterations": ["review"]}

def fixer_node(state: CodeReviewState):
    fixed = fixer_agent.invoke(state["code"], state["review_result"])
    return {"code": fixed, "iterations": ["fix"]}

def should_continue(state: CodeReviewState):
    if "通过" in state["review_result"]:
        return END                    # 审查通过，结束
    if state["error_count"] > 3:
        return "human"                # 超过 3 次失败，转人工
    return "fix"                      # 否则继续自动修复

# 构建图
workflow = StateGraph(CodeReviewState)
workflow.add_node("review", reviewer_node)
workflow.add_node("fix", fixer_node)
workflow.add_node("human", lambda s: {"review_result": "等待人工处理"})
workflow.set_entry_point("review")
workflow.add_conditional_edges("review", should_continue)  # 动态路由
workflow.add_edge("fix", "review")    # 修复后回到审查（循环）
graph = workflow.compile()
```

```python
# === CrewAI 角色流：同样任务的线性实现 ===
from crewai import Agent, Task, Crew, Process

reviewer = Agent(role="代码审查员", goal="审查代码质量", backstory="...", tools=[])
fixer = Agent(role="修复工程师", goal="按审查意见修复代码", backstory="...", tools=[])

# CrewAI 只能线性执行，无法根据审查结果动态决定是否修复
review_task = Task(description="审查代码", expected_output="审查报告", agent=reviewer)
fix_task = Task(description="修复代码", expected_output="修复后代码",
                agent=fixer, context=[review_task])
crew = Crew(agents=[reviewer, fixer], tasks=[review_task, fix_task],
            process=Process.sequential)  # 固定顺序，无条件分支
```

## 注意事项

- **CrewAI 不擅长容错和重试**：线性流程中任一 Task 失败会中断整条链，需手动重试或捕获异常。LangGraph 的循环+条件边天然支持「失败回退重试」，容错性强很多。
- **LangGraph 的学习曲线陡**：状态机思维（定义 State、节点、边）比 CrewAI 的「写角色和任务」复杂得多。简单任务用 LangGraph 是过度设计。
- **状态膨胀问题**：LangGraph 的全局 State 如果累积太多（如完整对话历史），会导致 token 溢出和性能下降。要设计 State 的清理策略（如滑动窗口保留最近 N 轮）。
- **调试与可观测性**：CrewAI 的线性流程容易追踪（按顺序看日志）；LangGraph 的图执行可能并行、循环，调试需要可视化工具（LangSmith 的 graph view）。两者都建议接入 LangSmith 做轨迹追踪。
- **选型速查**：明确分工的流水线（内容生产、数据处理）→ CrewAI；需要条件判断、循环重试、人工介入的复杂流程（客服、审批、代码审查）→ LangGraph；极简单任务直接用 LangChain Agent 即可，不必上多 Agent 框架。
- **混用可能**：两者并非互斥，可以在 LangGraph 的某个节点里调用一个 CrewAI Crew 作为子流程，结合静态编排和动态调度的优势。
