# JDK 8 中 ConcurrentHashMap 在计算 size() 方法时，为什么不再是简单地累加所有 Segment（JDK 7 做法）或 baseCount？它是如何保证高并发下的准确性与性能的？

JDK 8 放弃了 Segment 分段锁，采用 CAS + synchronized 控制节点。如果 size() 只是简单遍历所有 Node 累加，在并发极高时会无限循环或耗时过长。JDK 8 引入了 `baseCount` 作为基础值，以及 `CounterCell` 数组。

**核心原理**
当多个线程同时更新计数发生竞争时，部分线程会随机更新 `CounterCell` 数组中的某个槽位（类似于 LongAdder 的 Striped 机制）。统计 size() 时，先将 `baseCount` 与所有 `CounterCell` 的值累加。如果检测到统计过程中数据发生了变化（通过 `modCount` 识别），则再次尝试统计，最多重试两次（默认）。

**源码细节**
*   **`addCount`**：首选 CAS 更新 `baseCount`。如果失败，说明有竞争，会尝试初始化或更新 `CounterCell` 数组（线程探针算法）。
*   **`sumCount`**：计算逻辑为 `baseCount + sum(CounterCells)`。这是一个弱一致性快照，无法保证统计时刻的原子性。
*   **重试机制**：如果在累加过程中 `modCount`（数据结构变更次数）发生变化，说明有并发修改，会触发重试。若重试 3 次（`RETRIES_BEFORE_LOCK`）后仍不稳定，则会直接加锁（锁住所有 Node 数组桶）再统计一次，这是极端情况下的保底策略。

**权衡**
这种机制牺牲了微小的绝对实时准确性（可能在极高并发下统计值有微小偏差），换取了极高的性能。它将热点分散，避免了在 `size()` 操作上的全局锁竞争。

**实战案例**
在电商大促场景统计“当前在线人数”时，直接遍历 Node 会导致服务暂停响应（Stop-the-world 效应）。采用 LongAdder 机制后，统计耗时从 ms 级降为 us 级，虽然统计数字可能存在 ±10 的误差，但对监控大盘无影响。

**代码示例**
```java
// 模拟 JDK 8 ConcurrentHashMap 的 sumCount 逻辑
long sumCount() {
    long sum = baseCount; // 获取基础值
    CounterCell[] cs = counterCells;
    if (cs != null) {
        for (CounterCell c : cs) {
            if (c != null) {
                sum += c.value; // 分散累加，无锁读取
            }
        }
    }
    return sum;
}
```

**对比表格**

| 特性 | JDK 7 ConcurrentHashMap | JDK 8 ConcurrentHashMap |
| :--- | :--- | :--- |
| **计数结构** | Segment 中 volatile count | baseCount + CounterCell[] |
| **并发策略** | 分段锁 (ReentrantLock) | CAS + synchronized (Node级) |
| **Size() 性能** | 锁所有 Segment，耗时长 | 快照读取，弱一致性 |
| **高并发表现** | 锁竞争随 Segment 数量固定 | 竞争分散，性能随 CPU 核数线性增长 |
| **实现复杂度** | 较低 | 较高 (类似 LongAdder) |

## 常见考点
1.  **为什么 JDK 8 的 size() 方法是弱一致的？**（答：只做快照累加，不锁全表）
2.  **`CounterCell` 数组是如何扩容的？**（答：当数组长度达到 CPU 核心数或发生多次竞争时，会扩容为 2 倍）
3.  **这种设计与 JDK 7 的 Segment 有何区别？**（答：JDK 7 锁粒度是 Segment 级别，JDK 8 锁粒度是 Node 头节点级别，计数逻辑从 Segment 转为类似 LongAdder 的分散计数）

## 记忆要点

- 因为高并发直接累加Node会引发锁竞争，所以引入CounterCell数组分散热点
- 统计逻辑为baseCount加上所有CounterCell的值，本质是LongAdder机制
- 通过modCount检测并发修改，size()是弱一致性快照而非绝对实时准确

## 结构化回答



**30 秒电梯演讲：** 就像商场数客流，平时只在大门口计数器(baseCount)加；人多抢不到时，随机派你去旁边的分机(CounterCell)记；要查总数时，把大门口和所有分机加一下，如果这期间有人进出就重试，实在不行关大门数。

**展开框架：**
1. **Segment** — 放弃Segment，引入baseCount与CounterCell数组
2. **CAS** — 更新时优先CAS baseCount，失败则随机更新CounterCell分散竞争
3. **size()** — size()累加值若检测到并发修改则重试，极限情况加锁全表
