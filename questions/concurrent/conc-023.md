# 什么是协同式调度？

协同式调度（Cooperative Scheduling）是一种线程调度策略，线程的执行时间由**线程自己决定**。

### 核心机制

线程在执行完一个任务单元后，必须**主动**通知操作系统（或运行时）“我已经做完了，请切换到下一个线程”。线程如果不主动让出 CPU，其他线程将永远无法运行（可能导致“饿死”）。

### 调度流程对比

```
+-------------------+          +-------------------+
|   抢占式调度       |          |   协同式调度       |
+-------------------+          +-------------------+
| OS 分配时间片     |          | 线程主动 yield    |
|                   |          |                   |
| [Thread A] Running|          | [Thread A] Running|
|       |           |          |       |           |
|    (中断/超时)    |          |  (主动让出/IO)    |
|       v           |          |       v           |
| [Thread B] Running|          | [Thread B] Running|
+-------------------+          +-------------------+
```

### 特点

1.  **实现简单**：调度器逻辑简单，不需要复杂的时钟中断和抢占逻辑。
2.  **开销可控**：切换发生在程序逻辑明确的点，上下文切换的开销通常是可以预期的。
3.  **可靠性风险**：如果一个线程编写不当（例如死循环或发生阻塞）而不主动让出 CPU，整个系统就会挂起，其他任务无法响应。

### 实战案例
在使用 Go 语言或 Java 虚拟线程开发时，如果编写一个密集的 `for` 循环进行计算且不调用任何会触发阻塞的库函数，在其他语言中可能导致线程“饿死”。但在 Go 1.14+ 引入异步抢占后，这个问题得到缓解；而 Java 虚拟线程依赖 JDK 库的主动挂载，纯计算密集型虚拟线程仍会阻塞 Carrier 线程。

### 代码示例 (Java - 虚拟线程的协同式体现)
```java
// Java 21+ Virtual Thread
Thread.ofVirtual().start(() -> {
    // 1. 执行计算
    processData(); 
    // 2. 调用阻塞 IO（此处 JVM 会触发协同调度，让出底层 Carrier 线程）
    // 传统的 Thread.sleep() 也会挂起虚拟线程，而非占用 OS 线程
    try { Thread.sleep(1000); } catch (InterruptedException e) {}
    // 3. 被唤醒后继续执行
    continueProcess();
});
```

### 对比表格：协同式 vs 抢占式调度

| 维度 | 协同式调度 | 抢占式调度 (主流 OS) |
| :--- | :--- | :--- |
| **控制权** | 线程主动让出 | 操作系统强制剥夺 |
| **上下文切换位置** | 明确的代码边界 | 任意指令周期 (中断) |
| **CPU 密集任务影响** | 极易导致系统“假死” | 通过时间片保证响应性 |
| **实现复杂度** | 低 | 高 (需硬件中断支持) |
| **Java 体现** | 虚拟线程 | 传统线程 (Thread) |

### Java 中的体现

- Java 默认使用**抢占式调度**（基于 JVM 对 OS 线程的映射）。
- `Thread.yield()` 是一种“协同式”的提示，告诉调度器“我愿意让出 CPU”，但调度器可能忽略它。
- **Java 虚拟线程**：在 Project Loom 中，虚拟线程使用的是**协同式调度**（在 JVM 层面）。当虚拟线程执行 I/O 操作阻塞时，它会主动将控制权交还给 Carrier 线程（载体线程），让出 CPU 资源，从而实现高并发。

### 常见考点

1.  **为什么早期的操作系统（如 Windows 3.1, MacOS 9）使用协同式多任务，后来都改成了抢占式？**
    - 因为协同式极度依赖应用程序的稳定性。一个恶意或有 Bug 的程序（死循环）会导致整个系统死机。抢占式保证了 OS 对资源的最高控制权，即使某个进程卡死，OS 也能强制剥夺其 CPU 资源，调度其他进程运行。
2.  **Go 语言 Goroutine 的调度属于哪种？**
    - Go 的调度器（GMP 模型）结合了两者。从 Goroutine 视角看，它是协同式的（遇到函数调用或阻塞时调度器介入），但从 M（系统线程）视角看，如果是死循环计算且不调用任何函数，Go 调度器会在 Sysmon 监控线程中通过栈溢出检查等机制插入抢占点，实现**异步抢占**，防止 Goroutine 饿死。
3.  **Java 虚拟线程如何解决协同式调度的阻塞问题？**
    - Java 虚拟线程在 JDK 库层面进行了大量改造（如 Socket、File I/O）。当虚拟线程调用阻塞 API 时，JVM 并不会阻塞底层的 OS 线程，而是将虚拟线程挂起，并释放 Carrier 线程去执行其他虚拟线程。这种机制使得在高并发 I/O 场景下，少量的 Carrier 线程即可支撑百万级虚拟线程。
