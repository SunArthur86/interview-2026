# 数据库并发控制策略有哪些？

数据库并发控制的核心是「在保证隔离性的同时最大化并发度」，主要策略：

**1. 锁机制（悲观）：**
- **行锁/表锁**：粒度越小并发越高但开销越大。InnoDB 支持行锁。
- **共享锁（S）/排他锁（X）**：读读共享、读写/写写互斥。
- **意向锁（IS/IX）**：表级快速判断是否有行锁，提升加锁效率（如 IS 表示意向加行级共享锁）。

**2. MVCC（多版本并发控制）：** InnoDB 在 RC/RR 隔离级别下，读不加锁（快照读），通过 undo log + Read View 让每个事务看到一致性快照，读写不互相阻塞，大幅提升并发。

**3. 乐观锁：** 假设冲突少，提交时用 version/CAS 检查，冲突则重试。适合读多写少。

**4. 隔离级别权衡：**
- READ UNCOMMITTED：并发最高但脏读。
- READ COMMITTED（RC）：不可重复读，每次 Select 生成新快照。
- REPEATABLE READ（RR，MySQL 默认）：通过 Read View 解决不可重复读，通过 Next-Key Lock 解决幻读。
- SERIALIZABLE：并发最低但完全隔离，读写全加锁。

### 实战案例
在高并发秒杀场景中，如果直接使用 `select ... for update` 悲观锁扣减库存，数据库连接池会迅速耗尽，导致服务不可用。**实战优化**方案是采用 Redis Lua 脚本预先扣减（原子性）做前置保护，或使用乐观锁 `UPDATE stock SET num = num - 1 WHERE id = 1 AND num > 0`，将并发冲突交由数据库层面的 CAS 机制处理，避免长事务阻塞。

### 代码示例
```sql
-- 乐观锁扣减库存示例：利用 version 字段防止ABA问题
UPDATE goods_inventory 
SET stock = stock - 1, version = version + 1 
WHERE id = 100 AND version = 10;

-- 检查影响行数 (Affected Rows) > 0 则成功，否则说明版本已被修改
```

### 对比表格
| 机制 | 实现方式 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **悲观锁** | Select for update / 读写锁 | 数据强一致性 | 并发度低，死锁风险高 | 写多读少，冲突强（如金融转账） |
| **乐观锁** | CAS / Version 轮询 | 无锁，吞吐量高 | 冲突高时失败重试开销大 | 读多写少（如商品详情更新） |
| **MVCC** | Undo log + Read View | 读写不冲突 | 需维护历史版本，空间开销 | 通用的数据库默认隔离级别实现 |

**## 常见考点**
1. **MVCC 读视图在 RC 和 RR 下的区别？**
   - RC：每次 Select 都生成一个新的 Read View，能看到其他事务提交的修改。
   - RR：只在事务第一次 Select 时生成 Read View，后续复用，确保可重复读。
2. **什么是当前读和快照读？**
   - 快照读：普通的 Select，读取 MVCC 版本历史，不加锁。
   - 当前读：`select ... for update`、`insert`、`update`、`delete`，读取最新数据并加锁。
3. **Next-Key Lock 是如何解决幻读的？**
   - 它是 Record Lock（行锁） + Gap Lock（间隙锁）的组合，锁定记录本身以及记录前的间隙，阻止其他事务在间隙插入新记录。
4. **乐观锁如何实现？**
   - 通常使用 version 字段（`update table set ver=ver+1, ... where id=1 and ver=old_ver`）或 CAS 机制，失败则重试。

## 记忆要点

- 核心对比：悲观锁防冲突先加锁，而乐观锁假设无冲突提交时CAS校验
- MVCC机制：快照读不加锁，靠 undo log + Read View 实现读写不阻塞
- 隔离级别：RC 每次Select生成新视图，而 RR 仅首次生成视图保证可重复读
- 实战秒杀：用 `UPDATE ... WHERE version=?` 乐观锁，避免 for update 长事务阻塞
