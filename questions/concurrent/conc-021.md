# 什么是什么情况下会产生死锁？

死锁是指两个或两个以上的线程在执行过程中，因争夺资源而造成的一种**互相等待**的现象。如果没有外力干涉，它们都将无法推进下去。

### 死锁产生的四个必要条件（必须同时满足）：

1.  **互斥条件**：资源是独占的，一个资源只能被一个线程使用。
2.  **请求与保持条件**：一个线程因请求资源而阻塞时，对已获得的资源保持不放。
3.  **不剥夺条件**：线程已获得的资源，在未使用完之前，不能被强行剥夺，只能主动释放。
4.  **循环等待条件**：若干线程之间形成一种头尾相接的循环等待资源关系（例如 T1 等 T2，T2 等 T1）。

### 死锁模型图

```
    线程 1 (T1)                      线程 2 (T2)
-------------                    -------------
     |                                  |
     |持有 资源 A                        |持有 资源 B
     |                                  |
     |  ---> 请求 资源 B ---------------> |
     |         (阻塞，等待 B 释放)        |
     |                                  |
     | <--- 请求 资源 A ---------------- |
     |    (阻塞，等待 A 释放)            |
     |                                  |
   (死锁僵局)                        (死锁僵局)
```

### 如何避免/预防死锁

- **破坏循环等待**：最常用的方法。强制所有线程按固定的顺序申请锁（如都先获取 A 锁再获取 B 锁）。
- **破坏请求与保持**：一次性申请所有所需的资源，如果不满足则阻塞，且不占用已有资源（并发编程中较难实现，效率低）。
- **破坏不剥夺**：当线程持有部分资源但又申请新资源失败时，主动释放自己持有的资源（适用于资源状态可保存和恢复的场景）。
- **超时机制**：使用 `tryLock`（如 ReentrantLock）设置超时时间，获取不到锁则放弃并释放已持有的资源。

### 实战案例
在转账业务中，线程 A 从账户 1 向账户 2 转账（锁定 1，请求 2），线程 B 从账户 2 向账户 1 转账（锁定 2，请求 1）。如果并发执行，极易产生死锁。解决方案是引入 `System.identityHashCode(a) - System.identityHashCode(b)` 来强制决定锁的获取顺序。

### 代码示例 (Java - 超时机制)
```java
ReentrantLock lock1 = new ReentrantLock();
ReentrantLock lock2 = new ReentrantLock();

// 尝试获取 lock1
if (lock1.tryLock()) {
    try {
        // 尝试获取 lock2，仅等待 100ms
        if (lock2.tryLock(100, TimeUnit.MILLISECONDS)) {
            try {
                // 业务逻辑
            } finally {
                lock2.unlock();
            }
        } else {
            // 获取 lock2 超时，释放 lock1 退避
            lock1.unlock();
        }
    } finally {
        if (lock1.isHeldByCurrentThread()) lock1.unlock();
    }
}
```

### 常见考点

1.  **如何检测死锁？**
    - **JConsole / VisualVM**：JDK 自带工具可以检测并自动展示死锁循环。
    - **JStack**：命令行工具 `jstack <pid>`，输出线程堆栈，会打印 "Found one Java-level deadlock:" 信息，明确指出哪些线程在等待哪些锁。
    - **操作系统层面**：Linux 下使用 `pstack` 或分析 `/proc/<pid>/status`。
2.  **为什么说 "破坏循环等待" 是最实用的方法？**
    - 因为它对代码逻辑的侵入性相对较小，只需要约定锁的全局顺序即可。其他方法（如一次性申请所有资源）可能降低并发度，或者导致资源管理复杂化。
3.  **数据库死锁是如何处理的？**
    - 数据库通常有死锁检测机制。当检测到死锁时，数据库会根据策略（如回滚代价最小）选择其中一个事务作为牺牲品，抛出异常并回滚，以解除死锁。
