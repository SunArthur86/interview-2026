# 什么是线程上下文切换？如何减少上下文切换？

### 什么是线程上下文切换？

**定义**：
CPU 从执行一个线程（或任务）切换到执行另一个线程的过程。它包括保存当前线程的状态（上下文）并加载下一个线程的状态。

**上下文内容**：
主要由 **CPU 寄存器**（通用寄存器、程序计数器 PC、状态寄存器）和 **程序计数器** 组成。这些信息记录了线程当前执行到的位置以及计算产生的中间结果。

**存储位置**：
线程切换时，上下文信息会被保存在该线程的 **TCB（线程控制块）** 或操作系统内核的栈中。

**实战案例**：在压测高并发服务时，发现 CPU 使用率不高但系统吞吐量上不去（Load 很高）。通过 `vmstat` 查看发现 `cs`（上下文切换次数）高达数十万。排查发现是代码中存在大量 `synchronized` 锁竞争，导致线程频繁从 Running 切换到 Blocked，优化为 CAS 无锁编程后，吞吐量提升了 3 倍。

**代码示例（避免锁竞争减少切换）**：
```java
// 优化前：使用 synchronized，竞争激烈会导致线程挂起（上下文切换）
// synchronized(counter) { counter++; }

// 优化后：使用 AtomicBoolean/CAS，在用户态自旋，避免挂起
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet(); // 底层是 CAS 乐观锁
```

**切换开销详情**：
1.  **直接开销**：寄存器保存/恢复、系统调度器的代码执行、TLB（转译后备缓冲器）刷新。
2.  **间接开销**：缓存失效，CPU 缓存中原本热点的数据被新线程的数据替换，导致新线程初期性能下降。

**对比表格**：

| 切换类型 | 切换单位 | 开销大小 | 主要代价 |
| :--- | :--- | :--- | :--- |
| **进程切换** | 进程 (PCB) | 高 | 刷新 TLB (页表切换), 缓存失效大 |
| **线程切换** | 线程 (TCB) | 中 | 不刷新 TLB (共享进程页表), 仅寄存器/栈 |
| **协程切换** | 协程 | 低 | 仅保存少量寄存器 (CPU/用户态), 不进内核 |

### 如何减少上下文切换？
1.  **无锁并发编程**：使用 CAS 等乐观锁机制减少线程因竞争锁而陷入内核态导致的切换。
2.  **使用协程**：在用户态进行线程切换，避免内核态与用户态的切换开销。
3.  **减少线程数量**：根据 CPU 核数合理设置线程池大小，避免大量线程处于 Runnable 状态频繁抢夺 CPU。
4.  **CAS 算法**：利用原子类减少 synchronized 锁竞争带来的阻塞。
5.  **协程/用户态线程**：如 Java 中的虚拟线程或 Go 的 Goroutine，由运行时调度，对 OS 透明，不涉及内核切换。

```text
CPU 执行流
Thread A: | [执行 A] | <------> | [执行 B] | [执行 A] |
          ^          |          ^
          |          |          |
       保存上下文    |       恢复上下文
          |          |          |
          v          |          v
       寄存器 -> PCB |      PCB -> 寄存器
```

### 常见考点
1.  **Linux 进程切换与线程切换的开销差异**：由于线程共享进程的虚拟内存空间（页表），线程切换通常不需要切换 CR3 寄存器（页表基址），开销略小于进程切换，但主要开销仍在寄存器和缓存。
2.  **监控上下文切换**：如何通过 `vmstat` 或 `pidstat` 命令查看系统或特定进程的上下文切换次数（`cs` 列或 `cswch/s`）。
3.  **零拷贝技术**：虽然不是直接减少上下文切换，但减少了数据在内核态和用户态之间的拷贝次数，间接提升了系统性能。
