# 什么是I/O模型？

### 字节流与字符流

Java IO 流主要分为字节流和字符流。

1. **字节流**：
   - 基本单位是字节（byte），适合处理二进制数据（如图片、视频、音频）。
   - 底层设备永远只接受字节数据，字节流直接与文件交互，不涉及缓冲区（由操作系统决定）。
   - 核心类：`InputStream`、`OutputStream`。

2. **字符流**：
   - 基本单位是字符（char），专门用于处理文本数据。
   - 字符流是字节流的包装，内部将字符转换为字节（写入）或将字节转换为字符（读取），解决了编码问题。
   - 使用缓冲区提高性能。
   - 核心类：`Reader`、`Writer`。

**注意**：字符流不能处理非文本文件（如图片），因为转换过程可能导致数据损坏。

### I/O 模型

1. **BIO (Blocking I/O)**：
   - 同步阻塞模型。线程发起 I/O 请求后，会被阻塞直到操作完成。
   - 适用于连接数少且固定的场景，高并发下性能差（线程上下文切换开销大）。

2. **NIO (Non-blocking I/O)**：
   - 同步非阻塞模型。线程发起 I/O 请求后立即返回，通过轮询（polling）检查操作是否完成。
   - 优点是线程不会被阻塞，缺点是轮询消耗 CPU 资源。

3. **I/O 多路复用**：
   - 一个线程可以管理多个 I/O 通道（如 Socket）。
   - 通过 `select`、`poll`、`epoll` 等机制监听多个文件描述符，当某个描述符就绪时通知应用程序进行读写。
   - 是高性能网络编程的基础（如 Redis、Nginx）。

### ## 面试追问
1. **零拷贝**：在 NIO 和多路复用模型中，传统 I/O 需要将数据从内核缓冲区复制到用户缓冲区再发送，而零拷贝（如 `sendfile`、`mmap`）是如何减少甚至消除这部分 CPU 拷贝和上下文切换开销的？
2. **水平触发 vs 边缘触发**：在 epoll 中，LT（Level Triggered）和 ET（Edge Triggered）有什么区别？在编程处理上，为什么 ET 模式必须将 socket 设置为非阻塞，并且必须一次性读写完所有数据？
3. **C10K/C10M 问题**：随着连接数从一万增加到一千万，I/O 多路复用模型会面临哪些瓶颈（如内存、上下文切换、锁竞争），有哪些相应的优化策略？

### ## 易错点
1. **混淆 NIO 与非阻塞 I/O**：Java 中的 NIO（New IO/Non-blocking IO）包虽然支持非阻塞 I/O，但其核心组件是 Channel 和 Buffer。真正的 I/O 多路复用依赖于 `Selector`。单纯的非阻塞 I/O（不使用 Selector）会由于忙轮询导致 CPU 100%，并不是高性能方案。
2. **多路复用是同步还是异步**：I/O 多路复用本质上是**同步 I/O**模型。因为虽然“等待数据准备好”这一步由 epoll 监听被阻塞了，但第二步“将数据从内核复制到用户空间”仍然需要进程在调用 `recvfrom` 时同步阻塞等待（或立即返回后拷贝）。真正的异步 I/O（AIO）是“两个阶段都不需要阻塞”。
