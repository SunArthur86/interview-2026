# 高效的Reactor线程模型是什么？

### 高效的 Reactor 线程模型

Reactor 模式是一种事件驱动模式，用于处理多个客户端的并发请求。常见的模型有三种：

#### 1. Reactor 单线程模型
- **结构**：所有的 I/O 操作（连接、读写）都由同一个 NIO 线程完成。
- **优点**：结构简单，无线程切换问题，无线程竞争。
- **缺点**：无法利用多核性能，处理耗时业务逻辑会导致后续请求阻塞（单点故障风险）。
- **适用**：小规模应用、业务处理极快的场景。

#### 2. Reactor 多线程模型
- **结构**：
  - 一个专门的 NIO 线程负责监听服务端，接收客户端 TCP 连接。
  - 一个 NIO 线程池负责 I/O 读写和业务逻辑处理。
- **优点**：利用多核，分离了连接建立和读写处理。
- **缺点**：如果业务逻辑非常耗时，仍会阻塞 I/O 线程。

#### 3. 主从 Reactor 多线程模型
- **结构**：
  - **Main Reactor**（主线程池）：只负责监听服务端 Socket，建立连接，将新建的 SocketChannel 注册到 Sub Reactor。
  - **Sub Reactor**（子线程池）：负责已建立连接的 SocketChannel 的读写操作。
  - **Worker 线程池**：Sub Reactor 检测到 I/O 事件后，如果业务逻辑复杂，可能扔给业务线程池处理（如 Netty 的 Pipeline 机制），以免阻塞 I/O 线程。
- **优点**：
  - 连接建立与 I/O 读写分离，压力分散。
  - 利用多核，性能极高。
  - **Netty 的默认模型**：`EventLoopGroup bossGroup` (Main Reactor) 和 `workerGroup` (Sub Reactor)。

```text
      Clients
         │
         ▼
┌─────────────────┐
│  Main Reactor   │  (Boss Group: 监听 Accept 事件)
│  (Selector)     │
└────────┬────────┘
         │ 建立连接后注册
         ▼
┌─────────────────┐      ┌──────────────────┐
│  Sub Reactor 1  │      │  Sub Reactor N   │ (Worker Group: 处理 Read/Write)
│  (Selector)     │ ...  │  (Selector)      │
└────────┬────────┘      └────────┬─────────┘
         │                       │
         │ (Dispatch I/O事件)    │
         ▼                       ▼
┌─────────────────┐      ┌──────────────────┐
│   Handler/      │      │   Handler/       │
│   Pipeline      │      │   Pipeline       │
└─────────────────┘      └──────────────────┘
         │
         ▼ (如果任务重，异步提交)
┌─────────────────┐
│  Worker Thread  │ (Business Thread Pool)
│      Pool       │
└─────────────────┘
```

#### 补充：Netty 性能优化
- **零拷贝**：使用堆外直接内存，减少数据在 JVM 堆和 Native 堆之间的拷贝；CompositeByteBuf 实现逻辑组合而非物理拷贝。
- **内存池**：重用 Buffer（基于 PooledByteBufAllocator），减少分配回收开销，降低 GC 压力。
- **串行化设计**：避免多线程竞争，EventLoop 绑定特定线程，保证线程安全且无锁化。
