# Java 中的 CompletableFuture 编排多个异步任务时，`thenApply` 和 `thenApplyAsync` 在线程执行上有何区别？如何利用自定义线程池来隔离不同业务级别的计算资源？

`thenApply` 默认使用上一个任务所在的线程执行（如果上一个任务执行太快还未结束，可能由主线程或调用线程直接执行），导致任务串行化，阻塞链路；`thenApplyAsync` 则会将任务提交到默认的 `ForkJoinPool.commonPool()` 执行，实现异步非阻塞。为了资源隔离，不应依赖默认公共池，而应在编排链的第一个任务或显式指定 `Executor`。例如 `CompletableFuture.supplyAsync(task1, bizPool).thenApplyAsync(task2, ioPool)`，这样可以将 CPU 密集型计算和 I/O 密集型操作分配到不同的线程池中，避免互相抢占资源，实现更精细的线程池隔离和调优。

**核心机制与线程切换**
*   **`thenApply`**：依赖于执行上一个 Task 的线程。如果上一个任务是由 `supplyAsync` 提交到线程池执行的，那么 `thenApply` 也会复用该线程池中的同一个线程继续执行，称为“线程复用”或“栈内执行”。
*   **`thenApplyAsync`**：显式地将后续任务调度到线程池。如果不指定线程池，则使用 `ForkJoinPool.commonPool()`（它是 Daemon 线程，主线程结束时它也会立即结束，可能阻碍非 Daemon 任务）。

**死锁风险与资源隔离**
*   **自定义线程池必要性**：如果所有任务都使用 `ForkJoinPool.commonPool()`，一旦有耗时的 I/O 阻塞任务，会占满公共池的线程，导致其他正常的 CPU 计算任务饥饿。
*   **回调地狱预防**：通过显式传入不同的 `Executor`（如 `ioPool`, `cpuPool`, `jdbcPool`），可以有效隔离资源，避免一个业务模块的异常阻塞影响全局。
*   **注意点**：`thenApplyAsync(fn, executor)` 如果传入的 `executor` 是当前执行线程所在的线程池（且线程池已满），可能会发生拒绝策略或排队，但不会导致死锁；最危险的是在 `synchronized` 块中调用异步任务等待结果。

**流程对比图**
```
场景 A: thenApply (同步/复用)
Main Thread -> submit(T1) -> [Pool Thread 1 executes T1] -> [Pool Thread 1 executes T2]

场景 B: thenApplyAsync (异步/调度)
Main Thread -> submit(T1) -> [Pool Thread 1 executes T1] -> submit(T2) -> [Pool Thread 2 executes T2]
```

**实战案例**
在微服务聚合场景中，若不隔离线程池，某个下游慢查询占用了所有 `ForkJoinPool` 线程，会导致同一进程内健康的接口也因无法获取线程而超时。实战中通常按业务线拆分：`orderQueryPool`、`paymentQueryPool`。

**代码示例**
```java
ExecutorService ioPool = newFixedThreadPool(100);
ExecutorService cpuPool = newFixedThreadPool(Runtime.getRuntime().availableProcessors());

CompletableFuture.supplyAsync(() -> callRemoteService(), ioPool) // IO密集
    .thenApplyAsync(data -> {
        // 复杂计算在CPU池执行，不阻塞IO等待
        return heavyComputation(data); 
    }, cpuPool); // 显式切换线程池
```

## 常见考点
1.  **`thenApply` 和 `thenApplyAsync` 在异常处理上有区别吗？**（答：没有区别，均通过 `exceptionally` 处理，区别仅在于线程执行模型）
2.  **如果自定义线程池满了，`thenApplyAsync` 会怎样？**（答：根据线程池的拒绝策略处理，默认是 AbortPolicy）
3.  **为什么不建议使用 `ForkJoinPool.commonPool()` 做重度业务？**（答：它是全局共享的，JVM 内其他流式计算也可能使用，资源无法控制，容易产生竞争）

## 记忆要点

- 区别：thenApply复用前一个任务的线程（可能阻塞），thenApplyAsync则切换线程
- thenApplyAsync默认用ForkJoinPool公共池，极易被耗时I/O任务占满导致饥饿
- 最佳实践：显式传入自定义线程池进行资源隔离，避免业务间互相影响
