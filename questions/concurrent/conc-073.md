# 什么是Java守护线程（Daemon Thread）？

### Java 守护线程

1. **定义**：
   - 守护线程是指在程序运行时在后台提供一种通用服务的线程，也被称为“后台线程”或“服务线程”。

2. **特性**：
   - **生命周期依赖于用户线程**：当 JVM 中只剩下守护线程时，JVM 会自动退出。
   - **优先级较低**：通常用于执行后台任务，如垃圾回收、定时任务等。
   - **继承性**：在守护线程中创建的新线程默认也是守护线程。

3. **设置方法**：
   - 通过 `thread.setDaemon(true)` 设置。
   - **注意**：必须在 `thread.start()` 之前调用，否则会抛出 `IllegalThreadStateException`。

4. **典型应用**：
   - **垃圾回收器**：这是最经典的守护线程。当程序中没有非守护线程运行时，GC 也就没有存在的意义，JVM 随之退出。

5. **使用场景与注意事项**：
   - 适用于后台辅助任务，不应在守护线程中执行关键业务逻辑或 I/O 操作（因为可能随时被强制终止）。
   - 在 Web 容器（如 Tomcat）中，自身启动的线程通常与 Web 应用生命周期不绑定，需注意资源清理。

### ## 常见考点
1. **守护线程中 finally 块是否一定会执行**：不一定。当 JVM 退出时，守护线程会立即被终止，finally 块可能来不及执行。因此不要在守护线程中执行必须完成的资源释放逻辑。
2. **ThreadLocal 与守护线程的关系**：ThreadLocal 的值是依附于线程的。如果使用线程池，且核心线程是守护线程（极少见），可能导致任务结束后 ThreadLocal 无法及时被 GC 回收（需手动 remove）。
3. **Java 中的 Daemon 线程与 Linux 后台进程的区别**：概念不同，Java 的守护线程仅指依赖非守护线程存在的 JVM 线程，与操作系统的 `nohup` 或后台进程机制无关。

### ## 面试追问
1. **线程池与守护线程**：如果将 Java 线程池的核心线程设置为守护线程，当主线程执行完毕退出时，线程池中的任务会立即停止吗？如果在任务执行过程中 JVM 退出，对数据一致性有何影响？
2. **ShutdownHook**：当 JVM 因剩余守护线程而退出时，我们能否通过注册 `Runtime.getRuntime().addShutdownHook(...)` 来捕获退出信号并执行资源清理？此时守护线程的状态如何？
3. **RMI (Remote Method Invocation)**：Java RMI 的分布式垃圾回收（DGC）机制中涉及到守护线程吗？如果 RMI 客户端异常退出，服务端如何感知并清理相关资源？

### ## 易错点
1. **setDaemon 的时机**：必须在 `start()` 之前设置 `setDaemon(true)`。如果线程已经启动（状态不再是 NEW）再调用，JVM 会直接抛出 `IllegalThreadStateException`，且不会改变该线程的守护属性。
2. **finally 块的执行幻觉**：很多人认为 `finally` 必定执行，但在守护线程中，如果 JVM 在执行 `try` 块时决定退出（如所有用户线程结束），`finally` 块中的代码将根本不会被调度执行。此外，`System.exit(0)` 也会强制跳过 `finally`（除非在 ShutdownHook 中）。


## 记忆要点

- 一句话定义：后台服务线程，生命周期随用户线程消亡而终止（如GC线程）。
- 设置时机铁律：必须在thread.start()前调用setDaemon(true)，否则抛异常。
- 严重误区：守护线程被强制终止时，其finally块可能不执行，严禁做核心资源释放。
- 继承特性：守护线程中派生的新线程，默认依然是守护线程。
