# 虚拟线程中的Pin问题（Carrier线程被钉住）具体是什么原因造成的？如何排查和缓解？

虚拟线程运行在Carrier线程（平台线程）之上。当虚拟线程在执行过程中进入`synchronized`代码块或调用本地方法时，它无法被卸载，导致承载它的Carrier线程被“钉住”，直到操作完成。在高并发下，如果大量Carrier线程被钉住，会导致系统退化为类似传统线程池的性能瓶颈。排查方法：使用JDK参数`-Djdk.virtualThreadScheduler.parallelism=N`调整并行度，或开启`-Djdk.tracePinnedThreads=full`打印被钉住的线程堆栈。缓解措施包括：1. 将`synchronized`替换为`ReentrantLock`（后者支持虚拟线程挂起）；2. 避免在虚拟线程中调用耗时较长的JNI代码；3. 升级JDK版本，新版本JVM正在逐步减少对synchronized的依赖。

**实战案例**
在一次将高并发网关迁移至虚拟线程时，发现 QPS 上不去且 CPU 飙升。排查发现日志库使用 `synchronized` 保护 IO 写入，导致大量虚拟线程在等待日志 I/O 时“钉住”了 Carrier 线程。**解决方案**是将日志库升级到支持 `java.util.concurrent.locks` 的版本，或者使用异步日志框架，解决了 Pinning 问题，吞吐量恢复预期。

**关键代码**
```java
// 错误示例：会导致 Carrier 线程被钉住
public void badBlock() {
    synchronized (this) { // 虚拟线程在此无法卸载
        Thread.sleep(1000); // 模拟耗时操作
    }
}

// 正确示例：支持虚拟线程挂起
private final ReentrantLock lock = new ReentrantLock();

public void goodBlock() {
    lock.lock();
    try {
        Thread.sleep(1000); // 虚拟线程挂起，Carrier 线程可去执行其他任务
    } finally {
        lock.unlock();
    }
}
```

**对比表格：Synchronized vs ReentrantLock (在虚拟线程环境下)**

| 特性 | synchronized (原生锁) | ReentrantLock (显式锁) |
| :--- | :--- | :--- |
| **Pinning 行为** | **会钉住 Carrier 线程**，阻塞期间 Carrier 无法调度其他虚拟线程 | **不会钉住 Carrier 线程**，虚拟线程可正常挂起和卸载 |
| **实现机制** | 依赖 JVM 内部 monitor 对象 | 依赖 `AbstractQueuedSynchronizer (AQS)`，完全基于 Java API |
| **适用场景** | 极短的操作（如简单的计数器递增），无阻塞风险 | 可能发生 I/O 或等待的操作，高并发临界区 |
| **性能影响** | 在 Pinning 场景下极差，严重影响吞吐量 | 性能稳定，能充分发挥虚拟线程高并发优势 |
| **排查建议** | 使用 `-Djdk.tracePinnedThreads=full` 重点排查 | 无需特殊排查，但仍需注意死锁风险 |

## 技术原理

Pin 问题的根源是 JVM 无法在"持有 monitor 锁"或"执行本地代码"时安全地卸载虚拟线程。理解这点要看虚拟线程的调度模型：

- **虚拟线程的挂起-恢复机制**：虚拟线程运行在 Carrier 平台线程上，遇到阻塞 IO（如 Socket.read）时，JVM 会把虚拟线程的栈帧从 Carrier 上"卸载"（unmount），让 Carrier 去跑其他虚拟线程；IO 就绪后再"挂载"（mount）回来继续。这是虚拟线程能用极少 Carrier 承载百万并发的关键。
- **synchronized 为什么破坏卸载**：synchronized 基于 JVM 内部的 ObjectMonitor，monitor 的状态记录在 Carrier 线程（平台线程）上而非虚拟线程上。如果允许虚拟线程在持锁时卸载，monitor 状态会丢失或错乱。所以 JVM 规定：进入 synchronized 块期间，虚拟线程必须钉在 Carrier 上不能卸载。ReentrantLock 基于 AQS（纯 Java 实现），状态存在 AQS 对象里而非线程上，所以虚拟线程持锁时可以安全卸载。
- **JNI/本地方法的钉住**：本地代码（C/C++）执行时 JVM 无法控制其栈帧，无法做 unmount，所以虚拟线程在 JNI 调用期间也被钉住。
- **Carrier 耗尽 = 退化**：ForkJoinPool 默认并行度 = CPU 核心数。如果大量 Carrier 被钉住（比如全部在 synchronized 里等 IO），剩余虚拟线程无法被调度，系统退化成"CPU 核心数个传统线程"的瓶颈。

## 注意事项

1. **synchronized 替换为 ReentrantLock**：这是最核心的缓解措施，ReentrantLock 基于 AQS 支持虚拟线程完美挂起，不钉住 Carrier。
2. **排查用 tracePinnedThreads**：开启 `-Djdk.tracePinnedThreads=full` 打印被钉住的堆栈，定位是哪段 synchronized 或 JNI 导致。
3. **警惕隐式 synchronized**：日志库、第三方库里可能藏着 synchronized（如旧版 log4j 保护 IO），迁移虚拟线程前要审计依赖。
4. **JNI 耗时操作要避免**：虚拟线程里别调用耗时的本地代码，改用平台线程执行后再回调。

## 记忆要点

- Pin 钉住问题：虚拟线程遇到 synchronized 代码块或原生 JNI 时无法被卸载。
- 导致后果：Carrier 平台线程被钉死阻塞，系统彻底退化为传统线程池瓶颈。
- 排查命令：开启 -Djdk.tracePinnedThreads=full 直接打印被钉住的堆栈。
- 核心缓解：将 synchronized 强烈替换为 ReentrantLock，因为后者支持完美挂起。
