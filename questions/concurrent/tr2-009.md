# 在 Java 并发编程中，`Phaser` 和 `CyclicBarrier` 都是多线程同步栅栏。`Phaser` 解决了 `CyclicBarrier` 的哪些痛点？请举例说明一个适用 `Phaser` 的动态并发场景。

`CyclicBarrier` 适用于固定数量的线程在栅栏处等待，且无法动态调整参与方数量。`Phaser` 则是更高级的同步器，支持动态注册和注销参与者。Phaser 解决了两个痛点：一是可以在运行时通过 `register()` 或 `arriveAndDeregister()` 动态增加或减少同步的线程数；二是支持分层的 Phase（阶段）概念，可以像倒计时一样重复使用，且能控制不同阶段的并发逻辑。典型场景是“多阶段并行任务”，例如将一个大数据集拆分成 N 个子任务，每个子任务内部又分为 Map、Reduce 两个阶段。如果 Map 阶段某些任务执行失败需要放弃，可以动态注销参与方，而不会像 CyclicBarrier 那样导致所有线程永久等待。

**实战案例**
在实现一个自适应爬虫系统时，主线程负责分发初始种子 URL，工作线程爬取页面后解析出新的子链接并动态派发新的爬虫任务（即动态注册新 Party 到 Phaser）。当网络波动导致部分连接超时，这些任务会自动注销，系统不会因此卡死，只有当所有活跃任务全部完成后，Phaser 才进入下一阶段（数据持久化）。

**关键代码**
```java
Phaser phaser = new Phaser(1); // 主线程初始注册

void runTask() {
    int phase = phaser.register(); // 动态注册一个参与者
    new Thread(() -> {
        try {
            doWork();
        } finally {
            phaser.arriveAndDeregister(); // 完成并注销，不阻塞后续阶段
        }
    }).start();
}

// 主线程等待当前阶段所有子任务完成
phaser.arriveAndAwaitAdvance();
```

**对比表格：CyclicBarrier vs Phaser**

| 特性 | CyclicBarrier | Phaser |
| :--- | :--- | :--- |
| **参与方数量** | 构造时固定，运行时不可变 | 运行时可动态增减 (`register`/`deregister`) |
| **重用性** | 可重用 (`reset()`)，但需手动触发 | 自动推进到下一 Phase，天生支持多阶段并发 |
| **异常处理** | 一个线程抛出异常，Barrier 破损，需修复 | 支持高级异常处理，可指定部分失败是否影响整体 |
| **并发控制** | 简单的栅栏同步 | 支持分层 (`Tiered`) Phasers，适用于大规模并发树形结构 |
| **适用场景** | 固定小组的并行计算（如矩阵分块计算） | 动态任务集、流水线处理（如多阶段 MapReduce 模拟） |

## 记忆要点

- 痛点对比：CyclicBarrier 固定且不可变，而 Phaser 支持运行时动态增减。
- 核心优势：Phaser 自动推进多阶段，且支持运行中 register/deregister。
- 异常处理强：Phaser 局部任务失败可动态注销，不会像前者导致全员死锁卡死。
- 实战场景：适用于多阶段流水线，或动态派生任务的自适应爬虫系统。
