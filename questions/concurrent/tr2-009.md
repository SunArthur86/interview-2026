# 在 Java 并发编程中，`Phaser` 和 `CyclicBarrier` 都是多线程同步栅栏。`Phaser` 解决了 `CyclicBarrier` 的哪些痛点？请举例说明一个适用 `Phaser` 的动态并发场景。

`CyclicBarrier` 适用于固定数量的线程在栅栏处等待，且无法动态调整参与方数量。`Phaser` 则是更高级的同步器，支持动态注册和注销参与者。Phaser 解决了两个痛点：一是可以在运行时通过 `register()` 或 `arriveAndDeregister()` 动态增加或减少同步的线程数；二是支持分层的 Phase（阶段）概念，可以像倒计时一样重复使用，且能控制不同阶段的并发逻辑。典型场景是“多阶段并行任务”，例如将一个大数据集拆分成 N 个子任务，每个子任务内部又分为 Map、Reduce 两个阶段。如果 Map 阶段某些任务执行失败需要放弃，可以动态注销参与方，而不会像 CyclicBarrier 那样导致所有线程永久等待。

**实战案例**
在实现一个自适应爬虫系统时，主线程负责分发初始种子 URL，工作线程爬取页面后解析出新的子链接并动态派发新的爬虫任务（即动态注册新 Party 到 Phaser）。当网络波动导致部分连接超时，这些任务会自动注销，系统不会因此卡死，只有当所有活跃任务全部完成后，Phaser 才进入下一阶段（数据持久化）。

**关键代码**
```java
Phaser phaser = new Phaser(1); // 主线程初始注册

void runTask() {
    int phase = phaser.register(); // 动态注册一个参与者
    new Thread(() -> {
        try {
            doWork();
        } finally {
            phaser.arriveAndDeregister(); // 完成并注销，不阻塞后续阶段
        }
    }).start();
}

// 主线程等待当前阶段所有子任务完成
phaser.arriveAndAwaitAdvance();
```

**对比表格：CyclicBarrier vs Phaser**

| 特性 | CyclicBarrier | Phaser |
| :--- | :--- | :--- |
| **参与方数量** | 构造时固定，运行时不可变 | 运行时可动态增减 (`register`/`deregister`) |
| **重用性** | 可重用 (`reset()`)，但需手动触发 | 自动推进到下一 Phase，天生支持多阶段并发 |
| **异常处理** | 一个线程抛出异常，Barrier 破损，需修复 | 支持高级异常处理，可指定部分失败是否影响整体 |
| **并发控制** | 简单的栅栏同步 | 支持分层 (`Tiered`) Phasers，适用于大规模并发树形结构 |
| **适用场景** | 固定小组的并行计算（如矩阵分块计算） | 动态任务集、流水线处理（如多阶段 MapReduce 模拟） |

## 技术原理

Phaser 相比 CyclicBarrier 的灵活性，源于它在数据结构和状态机设计上的根本差异：

- **基于状态机的 Phase 推进**：Phaser 内部维护一个 `phase` 计数器（从 0 递增），每个阶段所有注册方都 `arrive()` 后，phase 自动 +1 进入下一阶段，无需像 CyclicBarrier 那样手动 reset。这个自动推进让 Phaser 天生支持"阶段 1→阶段 2→阶段 3"的流水线语义。
- **动态参与方的计数原理**：Phaser 用一个 volatile 的 `parties` 字段记录当前注册方数量，`register()` 原子地 +1，`arriveAndDeregister()` 原子地 -1。栅栏触发条件是"已到达数 == 注册数"，所以运行中增减参与方不会破坏栅栏——这正是 CyclicBarrier 做不到的（它的 parties 在构造时定死）。
- **分层 Phaser 应对大规模并发**：单个 Phaser 在数千线程下会有竞争瓶颈。Phaser 支持 tiering——多个子 Phaser 挂在一个父 Phaser 下，形成树形结构。子 Phaser 的阶段完成事件向上汇总到父，把竞争分散到树的各层，类似 ForkJoinPool 的 work-stealing 思路。
- **故障隔离的 deregister 机制**：CyclicBarrier 一个线程异常会导致 Barrier 破损，所有等待线程永久阻塞；Phaser 允许失败任务 `arriveAndDeregister()` 优雅退出，剩余任务仍能继续推进，不会卡死。

## 注意事项

1. **主线程初始注册别忘 +1**：通常 `new Phaser(1)` 让主线程占一个名额，子任务 register 后主线程 `arriveAndAwaitAdvance()` 等待，否则 Phaser 在 parties=0 时会直接终止。
2. **deregister 要在 finally 里**：任务异常时也要 deregister，否则残留的注册方会让栅栏永远无法满足，导致整体死锁。
3. **大规模并发用分层**：数千线程单 Phaser 竞争激烈，用树形分层 Phaser 分散竞争。
4. **onAdvance 可自定义阶段逻辑**：重写 `onAdvance(phase, parties)` 可以在每阶段切换时插入自定义逻辑（如校验、日志），返回 true 可终止 Phaser。

## 对比表

| 维度 | CyclicBarrier | Phaser | CountDownLatch |
|:---|:---|:---|:---|
| **参与方数量** | 构造时固定 | 运行时动态增减 | 构造时固定 |
| **重用性** | 可 reset 重用 | 自动推进多阶段 | 不可重用（一次性） |
| **阶段概念** | 单阶段 | 多阶段（phase 计数） | 无 |

## 代码示例

```java
import java.util.concurrent.Phaser;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

// Phaser 实现动态多阶段任务（自适应爬虫场景）
public class AdaptiveCrawler {
    private final Phaser phaser = new Phaser(1); // 主线程占 1 个名额
    private final ExecutorService pool = Executors.newCachedThreadPool();

    public void submitUrl(String url) {
        phaser.register(); // 动态注册一个参与者（运行时增减）
        pool.submit(() -> {
            try {
                crawl(url);  // 爬取页面，可能解析出新的子链接再次 submitUrl
            } catch (Exception e) {
                // 故障隔离：异常不破坏栅栏
            } finally {
                phaser.arriveAndDeregister(); // 完成（或失败）后注销，finally 防死锁
            }
        });
    }

    public void awaitAll() {
        phaser.arriveAndAwaitAdvance(); // 主线程等当前阶段所有子任务完成
        pool.shutdown();
        // onAdvance 可重写以在阶段切换时插入自定义逻辑（如数据持久化）
    }
}

// 对比：CyclicBarrier 做不到动态注册
// CyclicBarrier barrier = new CyclicBarrier(5); // 构造时定死 5，运行中无法变
```

## 记忆要点

- 痛点对比：CyclicBarrier 固定且不可变，而 Phaser 支持运行时动态增减。
- 核心优势：Phaser 自动推进多阶段，且支持运行中 register/deregister。
- 异常处理强：Phaser 局部任务失败可动态注销，不会像前者导致全员死锁卡死。
- 实战场景：适用于多阶段流水线，或动态派生任务的自适应爬虫系统。
