# JVM中有哪些锁优化技术？

JDK 1.6 引入了一系列 synchronized 锁优化技术（即「锁升级」），大幅降低锁开销：

**1. 偏向锁：**
- 第一个访问锁的线程，在对象头 Mark Word 记下线程 ID。
- 之后该线程进出同步块无需 CAS（只需比对 ID），几乎零开销。
- 适合「同一个线程反复进入同步块」场景。
- JDK 15 后默认关闭（竞争多时偏向锁撤销成本高）。

**2. 轻量级锁：**
- 当有第二个线程竞争，偏向锁撤销升级为轻量级锁。
- 用 CAS 把 Mark Word 复制到栈帧的 Lock Record，成功则获锁。
- 适合「交替执行、竞争不激烈」场景，靠自旋避免阻塞。

**3. 重量级锁：**
- 自旋失败（或竞争激烈）升级为重量级锁。
- 基于 OS 的 Mutex Lock（ObjectMonitor），未获锁的线程 park 阻塞。
- 用户态↔内核态切换，开销大。

**4. 自旋锁：**
- 轻量级锁竞争时，不立即阻塞，而是空转（自旋）等一会儿。
- 适应式自旋：根据历史成功率动态调整自旋次数。

**5. 锁消除：**
- 逃逸分析证明对象不会逃逸出方法（无线程竞争），JIT 自动去掉同步。
- 如 StringBuffer.append 在单线程内被消除锁。

**6. 锁粗化：**
- 相邻的同步块合并成一个，减少加锁/解锁次数。

**7. 边界情况**
- **锁膨胀不可逆**：锁只能从偏向升级到轻量级，再到重量级，不能降级（HotSpot VM 实现）。
- **偏向锁撤销**：调用 `hashCode()` 会导致偏向锁撤销（因为 Mark Word 存不下 HashCode），直接升级为轻量级或重量级锁。
- **批量重偏向/撤销**：JVM 会针对类级别的锁竞争进行批量重偏向或撤销，以减少单次撤销的 STW 开销。

锁升级是单向的（不可降级，JDK 早期可批量撤销偏向）。

### 锁升级状态流转图

```text
                      Object (New)
                           |
                           v
                [Biased Lock] (Thread ID)
                           |
              (Other Thread Contest / HashCode)
                           |
                           v
            [Lightweight Lock] (CAS Lock Record)
                           |
               (CAS Failure / Spin)
                           |
                           v
             [Heavyweight Lock] (OS Mutex)
```

### 实战与深化

- **实战案例**：高性能 RPC 框架中，调用链路极深，若每个方法都加 synchronized，偏向锁会带来极大性能损耗；但在高并发网关接口中，偏向锁因撤销开销反而导致 STW（Stop-The-World），生产环境中常通过 JVM 参数 `-XX:-UseBiasedLocking` 关闭偏向锁以提升吞吐量。
- **代码示例**：
```java
// 锁粗化示例
// JVM 会自动将这段代码优化为扩大一次锁的范围
for (int i = 0; i < 100; i++) {
    synchronized(list) {
        list.add(i); // JIT 优化后，锁只在循环外加一次
    }
}
```
- **对比表格**：

| 锁状态 | 适用场景 | 实现机制 | 性能开销 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **偏向锁** | 仅有一个线程访问 | Mark Word 存储线程 ID | 极低（几乎为零） | 无 CAS 操作，执行快 | 竞争激烈时撤销成本高（STW） |
| **轻量级锁** | 两个线程交替执行 | CAS + Lock Record | 低（用户态自旋） | 不阻塞线程，响应快 | 自旋消耗 CPU，竞争久会膨胀 |
| **重量级锁** | 多线程激烈竞争 | OS Mutex (内核态) | 高（上下文切换） | 稳定，不浪费 CPU | 线程阻塞，系统调度开销大 |

## 面试追问

3. 锁消除中的“逃逸分析”除了消除锁，还有哪些优化作用（如栈上分配）？

## 易错点
1. **认为自旋锁一定好**：如果持有锁的线程执行时间很长，自旋线程会空转消耗大量 CPU，反而降低性能。
2. **忽略 hashCode 的影响**：偏向锁状态下对象头没有空间存储 hashCode，一旦调用 hashCode，偏向锁被迫撤销，造成不必要的性能损耗。

## 记忆要点

- 升级方向：无锁 -> 偏向 -> 轻量级(自旋) -> 重量级(OS阻塞)，且不可降级
- 偏向锁：仅记录线程ID无CAS，适合单线程重入，但竞争时撤销代价极高
- 轻量级锁：竞争轻度时用 CAS+自旋避免线程阻塞，但自旋消耗 CPU
- 编译器优化：基于逃逸分析做锁消除，以及合并相邻同步块的锁粗化
