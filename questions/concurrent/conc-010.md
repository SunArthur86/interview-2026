# Java线程调度的机制是什么？

Java 线程调度由 JVM 和操作系统协同完成，主要采用**抢占式调度**模型。

**调度方式对比：**
- **协同式调度：** 线程执行完主动通知 OS 切换。缺点是恶意死循环线程会独占 CPU。
- **抢占式调度：** 系统分配时间片，时间片用完后强制切换。Java 采用此模式。

**Java 调度的核心机制：**
1. **优先级**：
   - Java 线程有 1-10 的优先级（`Thread.NORM_PRIORITY`=5）。
   - 映射机制：JVM 只是将 Java 优先级映射到 OS 优先级（如 Linux 只有 1-99 的动态优先级，并非一一对应）。
   - **注意**：不要完全依赖优先级做业务逻辑，可能导致饥饿或跨平台行为不一致。

2. **时间片轮转：**
   - 底层由 OS 调度器（如 Linux 的 CFS - Completely Fair Scheduler）实现。
   - 线程在 `RUNNABLE` 状态下竞争 CPU 时间片。

**线程状态转换图：**

```text
         +----------------------+
         |        NEW           |
         +----------+-----------+
                    | start()
                    v
         +----------------------+
    +--->|      RUNNABLE        |
    |    | (Ready + Running)    |
    |    +----------+-----------+
    |               |   sleep/wait/IO/block
    |               |               |
    |    run done   |               v
    |               |    +-----------------------+
    |               |    | BLOCKED (Monitor lock)|
    |               |    +-----------+-----------+
    |               |                |
    |               |                v
    |               |    +-----------------------+
    |               |    | WAITING (join/wait)   |
    |               |    +-----------+-----------+
    |               |                |
    |               |    +-----------+-----------+
    |               |    | TIMED_WAITING         |
    |               |    +-----------+-----------+
    |               |                |
    |               |  notify/unlock/time over
    |               |                |
    |               +<---------------+
    |
    v
+-----------------------+
|     TERMINATED        |
+-----------------------+
```

**关键方法：**
- `Thread.yield()`：提示调度器“当前线程愿意放弃当前 CPU”，可能从 `Running` 切回 `Ready`，**不保证**一定让出，也不保证切换到哪个线程。

### 实战案例
在 Linux 服务器上运行死循环线程，即使设置了 `Thread.setPriority(Thread.MAX_PRIORITY)`，该线程依然未能独占 CPU，其他低优先级线程依然在执行。这是因为 Linux 内核的 CFS 调度器不完全依赖静态优先级，还根据动态优先级和交互性进行调度，因此 Java 优先级在跨平台时往往“仅供参考”。

### 代码示例 (防止死循环占满CPU)
```java
Thread heavyTask = new Thread(() -> {
    while (true) {
        // 执行繁重计算
        doHeavyWork();
        // 主动让出CPU，避免将其他线程“饿死”
        Thread.yield(); 
    }
});
heavyTask.setPriority(Thread.NORM_PRIORITY - 1);
heavyTask.start();
```

### 调度机制对比
| 特性 | 抢占式调度 | 协同式调度 | 分时调度 | 实时调度 |
| :--- | :--- | :--- | :--- | :--- |
| **控制权** | 操作系统 | 线程自身 | 操作系统 | 操作系统 |
| **时间片** | 有 (OS分配) | 无 | 有 (固定) | 有 (依优先级) |
| **响应性** | 高 (强制剥夺) | 低 (依赖主动释放) | 高 | 极高 |
| **稳定性** | 高 (防止死循环占CPU) | 低 (死循环会导致卡死) | 高 | 高 |
| **Java 采用** | ✅ 是 | ❌ 否 | ✅ (底层机制) | ❌ 否 |

## 常见考点
1. **RUNNABLE 包含 Ready 和 Running**：Java 将“就绪”和“运行中”统一视为 `RUNNABLE` 状态，调用 `yield()` 或时间片用完都会回到就绪队列。
2. **线程优先级问题**：在 Windows 上效果较明显，但在 Linux 上由于 `nice` 值映射机制，不同 Java 优先级可能表现为同一 OS 优先级，导致优先级设置失效。

## 记忆要点

- 核心机制：Java采用抢占式调度，依赖系统分配时间片强制切换
- 优先级坑：因映射依赖OS底层逻辑，所以跨平台不一致不可靠
- 状态流转：阻塞或等待会交出CPU，解除后重回RUNNABLE竞争
- 让步方法：yield仅暗示让出CPU，因不保证生效故不能作同步控制
