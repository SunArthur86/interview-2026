# 什么是Java线程池工作过程？

线程池的工作过程主要遵循以下逻辑，旨在复用线程并控制并发资源：

1. **判断核心线程数**：当调用 `execute()` 提交新任务时，如果当前运行的线程数小于 `corePoolSize`，线程池会创建一个新线程来执行任务，即使有空闲的核心线程也会创建（除非配置了 `allowCoreThreadTimeOut`）。

2. **入队等待**：如果当前运行的线程数大于或等于 `corePoolSize`，线程池会将任务放入工作队列（如 `LinkedBlockingQueue`）中等待执行。

3. **判断最大线程数**：如果工作队列已满，且当前运行的线程数小于 `maximumPoolSize`，线程池会创建非核心线程（临时工）来立即执行此任务。

4. **执行拒绝策略**：如果工作队列已满，且当前运行的线程数已经达到 `maximumPoolSize`，线程池会拒绝该任务，并调用 `RejectedExecutionHandler` 执行拒绝策略（默认抛出异常）。

**回收机制**：
- 当线程数超过 `corePoolSize` 时，多余的空闲线程如果空闲时间超过 `keepAliveTime`，会被回收销毁。
- 最终，线程池在没有任务时会收缩到 `corePoolSize` 大小。

**实战案例**：
在双十一流量高峰期，若使用 `newFixedThreadPool`（相当于无界队列），当下游数据库响应变慢，任务会无限积压在队列中，最终导致 JVM OOM 崩溃，而触发不了拒绝策略保护机制。

**代码示例（自定义线程池以防止OOM）**：
```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2, // corePoolSize
    4, // maximumPoolSize
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100), // 有界队列，防止OOM
    new ThreadPoolExecutor.CallerRunsPolicy() // 拒绝策略：由调用线程执行
);
```

**对比表格（常见队列类型影响）**：

| 队列类型 | 特性 | 拒绝触发时机 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **SynchronousQueue** | 不存储元素，直接传递 | 线程数 >= corePoolSize | 高吞吐量，如 Executors.newCachedThreadPool |
| **LinkedBlockingQueue** | 无界队列（默认Integer.MAX） | 永不触发（可能导致OOM） | 任务平稳，需要缓冲的场景 |
| **ArrayBlockingQueue** | 有界队列 | 队列满且线程数 >= max | 需要严格资源控制的系统 |

**补充细节：**
- **线程创建**：核心线程和非核心线程在本质上没有区别，仅仅是在数量控制和回收策略上不同。所有的线程都是由 `ThreadFactory` 创建的。
- **队列选择**：不同的队列会产生不同的行为。例如 `SynchronousQueue`（无缓冲队列）会导致任务直接进入第3步（判断最大线程数），适用于高吞吐量场景；而 `LinkedBlockingQueue`（无界队列）可能导致第3步永远不会执行，最终可能造成 OOM。

## 记忆要点

- 核心四步曲：核心线程满 -> 入队列 -> 队列满 -> 达最大线程数 -> 拒绝策略
- 核心线程不满创核心，满则排队，队列满创非核心，极限触发拒绝策略
- 注意：无界队列（如默认 LinkedBlockingQueue）永远不满，maxThreads 失效易致 OOM
- 空闲超时回收：非核心线程超 keepAliveTime 会被销毁
