# 什么是互斥锁Mutex？

互斥锁（Mutex，Mutual Exclusion）是一种用于**保护临界区**、防止多个线程同时访问共享资源的同步机制。

### 基本原理

互斥锁通常只有两种状态：**加锁** 和 **解锁**。

1.  **加锁**：线程在进入临界区（访问共享资源）之前，必须尝试获取锁。如果锁已被其他线程持有，当前线程会被阻塞（挂起），直到锁可用。
2.  **解锁**：线程在离开临界区后，必须释放锁，以便其他等待的线程可以获取锁并进入临界区。

### 流程图：互斥锁状态流转

```
     线程 A                       线程 B
-------                      -------
  |                              |
  |  lock()                      |  lock()
  |  (获取成功)                  |  (锁被A持有)
  v                              v
 [ 持有锁 ]  <--(阻塞等待)---> [ 阻塞态 ]
  |                              |
  |  执行临界区代码               |  (Context Switch: 释放CPU)
  |                              |
  |  unlock()                    |
  |  (释放锁)                    |
  |          -->(唤醒B)--->      |
  v                              v
 [ 释放锁 ]                   [ 获取锁 ]
```

### 特点

- **原子性**：加锁和解锁操作通常是原子操作，由底层硬件或操作系统保证。
- **互斥性**：同一时刻，只能有一个线程持有互斥锁。
- **阻塞**：获取锁失败的线程通常会进入睡眠状态，释放 CPU 给其他线程（区别于自旋锁的忙等待）。

### 常见实现

- **Java**：`synchronized` 关键字、`ReentrantLock` 类。
- **Linux/C**：`pthread_mutex_t`（POSIX 线程库）。相关 API 包括：
    - `pthread_mutex_init`：初始化锁。
    - `pthread_mutex_lock`：加锁（阻塞）。
    - `pthread_mutex_trylock`：尝试加锁（非阻塞）。
    - `pthread_mutex_unlock`：解锁。
    - `pthread_mutex_destroy`：销毁锁。

### 实战案例
在 Java 双检锁单例模式中，如果忘记使用 `volatile` 修饰实例变量，由于指令重排序，可能导致其他线程获取到锁后看到“半初始化”的对象（引用不为空但字段未初始化），从而引发 NPE 异常。这是典型的在使用 Mutex 时未能正确处理内存语义导致的坑。

### 代码示例 (Java - ReentrantLock)
```java
ReentrantLock lock = new ReentrantLock();
try {
    // 尝试获取锁，支持响应中断
    if (lock.tryLock(100, TimeUnit.MILLISECONDS)) {
        try {
            // 临界区：操作共享资源
            sharedResource.update();
        } finally {
            lock.unlock(); // 必须在 finally 中释放
        }
    }
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
}
```

### 对比表格：Mutex vs 自旋锁

| 特性 | 互斥锁 | 自旋锁 |
| :--- | :--- | :--- |
| **等待策略** | 线程挂起，释放 CPU | 线程循环检测 |
| **CPU 开销** | 低（不占用 CPU），但有上下文切换成本 | 高（一直占用 CPU）
| **适用场景** | 临界区长、并发度高 | 临界区极短、多核 CPU |
| **实现层次** | 主要依赖 OS 内核 | 依赖硬件 CAS 指令 |

### 常见考点

1.  **Mutex 和自旋锁的区别？**
    - **Mutex**：线程阻塞，释放 CPU，适合临界区长或并发度高的场景，避免 CPU 空转，但有上下文切换开销。
    - **自旋锁**：线程忙等待（死循环），不释放 CPU，适合临界区极短的场景，避免了上下文切换，但会浪费 CPU 时间片。
2.  **什么是可重入锁？**
    - 同一个线程可以多次获取同一把锁而不会死锁。例如 `synchronized` 和 `ReentrantLock` 都是可重入的。实现原理通常是在锁对象中记录“持有者线程”和“重入次数”，每次加锁次数+1，解锁次数-1，直到为0才真正释放锁。
3.  **公平锁与非公平锁的区别？**
    - **公平锁**：严格按照请求锁的顺序来获取锁（FIFO），吞吐量较低但避免饥饿。
    - **非公平锁**：允许插队，线程尝试获取锁时直接竞争，失败再加入队列。通常吞吐量更高（`synchronized` 是非公平的，`ReentrantLock` 默认也是非公平的）。
