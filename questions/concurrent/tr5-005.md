# Disruptor是如何实现无锁高并发队列的？请解释其核心设计思想（如RingBuffer、序号屏障）。

Disruptor通过预分配环形数组、CAS无锁操作、序号屏障协调及消除伪共享，实现高性能无锁并发。

## 技术原理

- **环形数组预分配内存，消除 GC 与分配开销**：Disruptor 用一个固定大小的 `RingBuffer`（大小必须是 2 的幂，便于用位运算取模 `seq & (size-1)`），在启动时一次性预分配所有槽位对象。生产者只是覆盖已有槽位的字段，不创建新对象，避免了 GC 压力和内存分配开销，对延迟敏感场景至关重要。
- **CAS 与序号屏障代替锁，实现线程安全无等待**：生产者通过 CAS 申请下一个可用序号（避免多生产者冲突），写入数据后再发布序号；消费者通过 `SequenceBarrier` 等待序号被发布，只读不锁。序号还用于构建消费者依赖链（如 C1 必须在 C2 之后），用屏障编排复杂依赖，无需锁协调。
- **缓存行填充消除伪共享**：`Sequence` 这种被多线程高频读写的字段，若与相邻变量落在同一 CPU 缓存行（通常 64 字节），一个核写它会失效所有核的整个缓存行（伪共享）。Disruptor 在 `Sequence` 前后填充无用的 `long` 占位（`p1..p7`），让每个 `Sequence` 独占一个缓存行，消除多核间的缓存失效抖动。

## 对比/选型

| 维度 | ArrayBlockingQueue | Disruptor |
|------|--------------------|-----------|
| 同步机制 | 加锁（ReentrantLock） | CAS + 序号屏障 |
| 内存分配 | 入队时可能分配 | 启动时预分配，运行时无 |
| 伪共享 | 有 | 缓存行填充消除 |
| 吞吐 | 数百万 ops/s | 数千万 ops/s |
| 延迟 | 微秒级，有抖动 | 纳秒级，稳定 |
| 复杂度 | 简单 | 学习成本高 |

## 代码示例

Disruptor 最简生产-消费：

```java
// 1. 定义事件
class LongEvent { long value; }

// 2. 事件工厂（预分配用）
EventFactory<LongEvent> factory = LongEvent::new;

// 3. 构建 Disruptor（RingBuffer 大小必须是 2 的幂）
Disruptor<LongEvent> disruptor = new Disruptor<>(
    factory, 1024, DaemonThreadBuilder.INSTANCE,
    ProducerType.MULTI, new BlockingWaitStrategy());

// 4. 注册消费者
disruptor.handleEventsWith((event, seq, end) -> System.out.println(event.value));
disruptor.start();

// 5. 生产者发布事件
RingBuffer<LongEvent> ring = disruptor.getRingBuffer();
long seq = ring.next();                         // CAS 申请序号
try {
    LongEvent e = ring.get(seq);
    e.value = 42L;                              // 覆盖预分配对象，无 GC
} finally {
    ring.publish(seq);                          // 发布序号，消费者可见
}
```

缓存行填充消除伪共享（Disruptor 内部 Sequence 简化版）：

```java
class PaddedSequence {
    private long p1, p2, p3, p4, p5, p6, p7;   // 填充前 7 个 long
    private volatile long value;                 // 独占一个 64B 缓存行
    private long q1, q2, q3, q4, q5, q6, q7;   // 填充后 7 个 long
}
```

## 常见坑/注意事项

- **RingBuffer 大小必须是 2 的幂**：因为用 `seq & (size-1)` 取模，非 2 的幂会位运算错乱，启动时会有校验。
- **WaitStrategy 选择**：`BlockingWaitStrategy`（低 CPU、高延迟）适合低吞吐；`YieldingWaitStrategy`/`BusySpinWaitStrategy`（高 CPU、低延迟）适合极致低延迟；`SleepingWaitStrategy` 是折中。选错要么延迟抖动，要么 CPU 空转。
- **消费者依赖链编排**：多消费者有依赖时用 `after()` 链式声明，不要在消费者内部手工同步，否则破坏无锁模型。
- **伪共享是隐性杀手**：自己实现高性能结构时，凡是多线程高频写的共享变量都要考虑缓存行填充，否则性能可能差一个数量级。


## 记忆要点

- 因为数组预分配且利用CAS，所以避免了频繁的GC和锁竞争开销
- 序号屏障：通过序列号控制消费者，实现无锁并发与依赖链
- 消除伪共享：通过缓存行填充，解决了CPU缓存失效带来的性能损耗
