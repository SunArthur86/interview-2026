# 什么是进程调度？

### 进程调度

**定义**：
操作系统决定哪个就绪进程获得 CPU 使用权，并将 CPU 分配给它的过程。

**调度层次**：
1. **高级调度（作业调度）**：从外存后备队列中选择作业调入内存，创建进程。发生频率最低（分钟/小时级）。
2. **中级调度（内存调度）**：将暂时不运行的进程调至外存（挂起），以腾出内存空间；或者将外存中具备运行条件的进程调入内存（对换）。
3. **低级调度（进程调度）**：决定就绪队列中哪个进程获得 CPU（频率最高，毫秒级）。

**调度算法的评价指标**：
- **CPU 利用率**：CPU 忙碌时间的比例。
- **系统吞吐量**：单位时间内完成的作业数量。
- **周转时间**：作业从提交到完成的总耗时（等待+执行+I/O）。
- **响应时间**：从提交请求到首次产生响应的时间。
- **公平性**：保证每个进程都能获得合理的 CPU 时间。

**三种调度层次协作图**：
```text
     外存 (磁盘)
  ┌─────────────────────┐
  │   后备队列 (作业)    │
  └──────────┬──────────┘
             │ 高级调度 (作业调度)
             ▼
     ┌─────────────────────────────────────┐
     │            内存                      │
     │  ┌─────────┐       ┌───────────────┐ │
     │  │ 就绪队列│◀──────│   PCB 对换区   │ │◀─── 中级调度 (内存/挂起)
     │  └────┬────┘       └───────────────┘ │
     └───────┼──────────────────────────────┘
             │ 低级调度 (进程调度)
             ▼
     ┌─────────────────────┐
     │       CPU           │
     │   (执行进程)        │
     └─────────────────────┘
```

**实战案例**：在运行大规模 MapReduce 任务的服务器上，默认的 Linux 调度器（CFS）偶尔会导致长耗时的计算任务抢占交互式 Shell 的 CPU，导致运维操作卡顿。通过调整进程的 `nice` 值（降低计算进程优先级），保证了管理命令的响应速度，同时未显著降低整体吞吐量。

**代码示例**：
```bash
# Linux 下调整进程优先级的实战命令
# 启动进程时指定 nice 值 (范围 -20 到 19, 默认 0, 越高优先级越低)
nice -n 10 java -jar heavy-computation.jar

# 修改正在运行进程的优先级
renice +5 -p 1234
```

**对比表格（常见调度算法）**：
| 算法 | 模式 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **FCFS (先来先服务)** | 非抢占 | 简单，公平 | 可能导致“护航效应”（短进程等待长进程） | 简单的批处理系统 |
| **SJF (短作业优先)** | 非抢占/抢占 | 平均等待时间最短 | 难以准确预测作业执行时间 | 批处理系统 |
| **RR (时间片轮转)** | 抢占 | 公平，响应时间好 | 时间片过大退化为FCFS，过小切换开销大 | 分时系统、交互式系统 |
| **多级反馈队列** | 抢占 | 兼顾短作业响应和长作业吞吐 | 算法复杂 | 通用操作系统（如 UNIX, Windows） |
