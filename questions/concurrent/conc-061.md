# redis是单线程的吗？那如何支持高并发是什么？

**Redis 是单线程的吗？**
主要指 Redis 处理网络请求和命令执行的模块（主要是命令执行器）在传统模式下是单线程的。

**为什么是单线程？**
- Redis 的性能瓶颈通常在内存和网络带宽，而非 CPU。
- 单线程模型避免了多线程上下文切换和锁竞争的开销，实现简单且效率高。

**处理流程（I/O 多路复用 + 单线程命令执行）：**
```text
Client A ──┐
Client B ──┼──► Socket Queue ──► I/O Multiplexing ──►
Client C ──┘        │                         │
                     ▼                         ▼
             (事件循环/Reactor)     (单线程事件处理器)
                     │                         │
                     └──────► Command Parser ──┴──► Executor (In-Memory)
                                                     │
                                                     ▼
                                             Response to Client
```

**如何支持高并发？**
1. **基于内存操作**：所有数据都在内存中，读写速度极快（纳秒级）。
2. **I/O 多路复用**：使用文件事件处理器（如 epoll），同时监听多个 Socket 连接。一旦有请求到达，就触发事件处理器进行处理，实现“一个线程处理多个连接”。
3. **高效数据结构**：如 SDS（简单动态字符串）、跳跃表、哈希表、压缩列表等，针对特定场景优化，降低了时间和空间复杂度。

**注意**：Redis 4.0 之后引入了多线程用于后台删除大键（如 unlink 命令）等操作，Redis 6.0 引入了多线程处理网络 I/O（数据的读写和解码），但核心**命令执行依然是单线程**的，以保证原子性。

**### 1. 实战案例**
在某大促活动中，Redis 突然出现 CPU 飙升但 QPS 未明显增长的卡顿。排查发现是业务代码执行了 `KEYS *` 操作（时间复杂度 O(N)），阻塞了单线程的事件循环，导致其他正常请求排队。生产环境严禁使用 `KEYS`，应使用 `SCAN` 代替。

**### 2. 代码示例**
```bash
# 验证 Redis 单线程阻塞的命令
# 在一个终端执行 bigkey 操作（如 hset 包含百万字段的 hash），
# 在另一个终端执行简单的 ping，会发现 ping 延迟急剧增加。
redis-cli hset my_hash field1 "value_string_10mb"
# 监控延迟
redis-cli --latency
```

**### 3. 对比表格**

| 特性 | Redis (单线程模型) | 多线程数据库 (如 Memcached)
| :--- | :--- | :--- |
| **CPU 利用率** | 无法利用多核优势（单实例），通常开多实例分片 | 可充分利用多核 CPU
| **并发处理** | 事件循环，非阻塞 I/O，单线程排队处理 | 多线程并行处理，需处理锁竞争
| **开发/维护** | 代码简单，无死锁风险，调试方便 | 逻辑复杂，需关注并发安全、上下文切换
| **适用瓶颈** | 内存/带宽受限场景 | CPU 计算密集型场景

**## 常见考点**
1. **Redis 6.0 多线程变化**：6.0 引入多线程具体做了什么？（仅用于网络数据读写和协议解析，命令执行仍单线程）
2. **为什么单线程依然快**：纯内存操作、单线程避免了上下文切换、非阻塞 I/O、高效数据结构。
3. **纯缓存场景 vs 持久化场景**：如果开启了 RDB/AOF，fork 子进程时会阻塞吗？（会，fork 操作在瞬时是阻塞的，大内存实例需注意）。
4. **Pipeline 是什么**：如何减少 RTT（往返时间）？（Pipeline 可以一次发送多条命令，减少网络往返开销，但这不是并发，是批处理）。

## 记忆要点

- 单线程本质：核心命令执行始终单线程，避免了多线程上下文切换和锁竞争的开销。
- 高并发三板斧：纯内存操作 + IO多路复用(epoll) + 高效数据结构。
- 版本演进：6.0 引入多线程仅处理网络读写，命令执行依然单线程以保原子性。
- 致命避坑：单线程怕阻塞，生产环境严禁使用 KEYS * 等时间复杂度 O(N) 的命令。
