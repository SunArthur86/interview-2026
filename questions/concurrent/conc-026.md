# 什么是水平分表？

### 水平分表

**定义**：
水平分表是将同一个表中的**数据行**拆分到多个结构相同的数据库表中。

**核心原理**：
- **表结构不变**：拆分后的表拥有完全相同的列结构。
- **数据分散**：每个表只包含原表的一部分数据行。
- **全集逻辑**：所有拆分表的数据合集即为原表的全量数据。

**应用场景**：
当单表数据量达到千万级，查询和写入性能明显下降（索引树过大、IO 瓶颈）时使用。

**实战案例**：在电商订单系统中，单表订单突破 5000 万行后，查询 `order by create_time` 变得极慢。采用按 `user_id` 取模分表后，查询某用户的订单直接定位到单张物理表，性能提升 10 倍以上；但遇到按 `time_range` 统计全平台销量时，必须扫描所有分表，性能反而不如单表，因此通常将此类冷数据同步到 ES 或大数据平台处理。

**拆分策略（路由规则）**：
1.  **Range（范围）**：如按时间、ID 区间拆分。优点是扩容简单；缺点是数据分布不均，可能产生热点。
2.  **Hash（哈希）**：如对 ID 取模。优点是数据分布均匀；缺点是扩容时数据迁移量大（通常需倍数扩容）。
3.  **地理位置/列表**：如按省份、业务类别拆分。

**代码示例（分片路由逻辑）**：
```java
public String determineTable(Long userId, int tableCount) {
    // 简单的取模路由算法
    int suffix = (int) (userId % tableCount);
    return "order_" + suffix;
}
```

**对比表格**：

| 维度 | 水平分表 | 垂直分表 |
| :--- | :--- | :--- |
| **拆分维度** | 按“行”拆分 | 按“列”拆分 |
| **表结构** | 拆分后表结构**完全相同** | 拆分后表结构**不同**（字段减少） |
| **解决问题** | 解决单表数据量过大（行多）的问题 | 解决单表数据宽度大（列多）、IO 瓶颈问题 |
| **复杂度** | 主要是数据路由、跨表 Join | 主要是业务逻辑拆分、关联查询 |

**与垂直分表的区别**：
- **垂直分表**：基于“列”拆分，解决“表太宽”的问题（如大字段、冷热字段分离）。
- **水平分表**：基于“行”拆分，解决“表太长”的问题（数据量过大）。

```text
原表: User (1亿行)
+----+------+-------+
| ID | Name | Email |
+----+------+-------+
| 1  | ...  | ...   |
| 2  | ...  | ...   |
| ...| ...  | ...   |
+----+------+-------+
         | 水平拆分 (ID % 2)
         v
+----------------+       +----------------+
| User_0 (偶数)  |       | User_1 (奇数)  |
+----+------+-------+    +----+------+-------+
| 2  | ...  | ...   |    | 1  | ...  | ...   |
| 4  | ...  | ...   |    | 3  | ...  | ...   |
+----+------+-------+    +----+------+-------+
```

### 常见考点
1.  **水平分后的 Join 问题**：跨表 Join 如何处理（通常在应用层组装，或通过全局表/ER 分片解决）。
2.  **分表后的分页查询**：如 `limit 100000, 10` 在分表场景下性能极差，如何优化（如搜索撮合、二次排序）。
3.  **主键 ID 生成策略**：分表后如何保证 ID 全局唯一（如雪花算法 Snowflake、UUID、数据库步长）。
