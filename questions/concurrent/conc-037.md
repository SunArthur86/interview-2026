# Cassandra一致Hash和虚拟节点是什么？

Cassandra 采用一致性 Hash 环进行数据分布，并引入虚拟节点来优化负载均衡。这是分布式系统分片的经典实现。

**1. 一致性 Hash**
- **原理**：将 Hash 空间（通常是 0 ~ 2^32-1）构建成一个环。每个物理节点根据 Hash（通常是 Token）分配一个位置落在环上。
- **数据分布**：数据的 Key 经过 Hash 后也落在环上，顺时针方向找到的第一个节点即为负责存储该数据的节点。
- **优势**：解决传统取模 Hash 在节点增减时导致大量数据迁移的问题，只影响相邻节点的数据（影响范围最小化）。

**2. 虚拟节点**
- **概念**：每个物理节点在环上映射多个 Token（虚拟节点）。例如，一个物理节点可能对应 Token A, B, C...（在 Cassandra 中通常每个节点配置 256 个虚拟节点，即 vnodes）。
- **作用**：
  - **负载均衡**：避免因物理节点 Hash 分布不均导致的数据倾斜。随机分布的 vnode 使得数据更均匀。
  - **故障隔离**：当物理节点宕机时，其原本承担的数据由环上相邻的多个虚拟节点分担，负载能更均匀地散落到整个集群，而不是压垮某一台邻居机器。

### 实战案例
在某分布式缓存扩容实战中，未引入虚拟节点时，新节点上线导致其顺时针的后继节点（邻居）瞬间接收了 30% 的数据流量，直接打满了 CPU 和带宽，引发雪崩。**引入 Vnodes 后**，扩容时的数据请求被均匀打散到集群中的几乎所有节点，扩容过程对业务无感知，QPS 波动控制在 5% 以内。

### 代码示例 (Java 伪代码)
```java
// 模拟一致性 Hash 寻找节点
long hashKey = hash(key); // 计算 Key 的 Hash
SortedMap<Long, String> ring = new TreeMap<>(); // Token -> Node IP

// 引入虚拟节点：一个物理节点对应 100 个虚拟 Token
for (int i = 0; i < 100; i++) {
    ring.put(hash(nodeIp + "#" + i), nodeIp);
}

// 寻找顺时针第一个节点（TailMap 用于找大于等于 hashKey 的部分）
return ring.get(ring.tailMap(hashKey).firstKey());
```

### 对比表格
| 特性 | 传统取模 Hash | 一致性 Hash (无 Vnode) | 一致性 Hash (含 Vnode) |
| :--- | :--- | :--- | :--- |
| **节点变动影响** | 几乎全量数据迁移 | 仅影响相邻节点 | 仅影响相邻节点，但数据分散到全网 |
| **数据倾斜** | 易倾斜，取决于节点数 | 极易倾斜（Token随机性差） | **低倾斜**（统计学均匀） |
| **负载均衡** | 差 | 差（热点邻居压力大） | **极好**（压力均摊） |
| **实现复杂度** | 低 | 中 | 高 (需管理大量 Token) |

**## 常见考点**
1. **节点上线/下线时的数据迁移量是多少？**
   - 在引入 vnode 后，数据的迁移是分散的。节点下线时，它持有的 vnode 数据会被顺时针的后继节点接收。由于 vnode 是随机分布的，新节点会分担整个环的责任，数据从整个集群迁入，而不是仅从某一台迁出，极大减轻了热点压力。
2. **Cassandra 的复制策略如何结合 Consistent Hash？**
   - 通常使用 `SimpleStrategy` 或 `NetworkTopologyStrategy`。数据写入时，找到主节点（Primary）后，会继续在环上顺时针寻找 N-1 个节点作为副本节点。
3. **Hash 环的数据倾斜问题？**
   - 如果没有虚拟节点，节点少时很难均匀分布。引入 vnode 后，每个节点负责环上的多个小段，统计学上趋向于均匀。
4. **什么是 Gossip 协议？**
   - Cassandra 节点间通过 Gossip 协议交换状态（如节点存活、Schema 变化），这是维护 Token Ring 状态一致性的基础。
