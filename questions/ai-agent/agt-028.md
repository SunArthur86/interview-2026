# Tool Use / Tool Calling

### Tool Use / Tool Calling

**定义**：Tool 是 Agent 的“能力单元”，封装了名称、描述、参数 Schema 和执行函数。它是连接大模型（LLM）逻辑世界与物理/数字世界（API、数据库、硬件等）的桥梁。

**核心原理**：
模型本身无法执行代码或网络请求。Tool Calling 机制通过特殊 Prompt 注入工具的定义，让模型输出结构化的 JSON（包含 `name` 和 `arguments`），而非普通文本。宿主程序解析该 JSON 并执行实际逻辑，将结果回传给模型继续推理。

**执行流程图**：
```text
┌─────────┐   1. Query+Tool_Schemas   ┌───────────┐
│   User  │ ──────────────────────> │   LLM     │
└─────────┘                         └─────┬─────┘
                                          │
                                    2. Generate Tool Call
                                    (JSON: name, args)
                                          │
                                          ▼
┌─────────┐   3. Execute Function    ┌───────────┐
│  Host   │ <────────────────────── │ Executor  │
│  (App)  │   ─────────────────────> │ (Function)│
└────┬────┘   4. Return Result       └───────────┘
     │
     │ 5. Inject Result into Context
     ▼
┌─────────┐   6. Final Answer       ┌───────────┐
│   User  │ <────────────────────── │   LLM     │
└─────────┘                         └───────────┘
```

**最佳实践**：
1.  **描述**：是模型选择工具的关键。应包含功能、何时用、何时不用的示例。使用 `Few-Shot` 示例显著提升准确率。
2.  **参数设计**：少而精，优先枚举，避免自由文本，日期统一 ISO 格式。参数名应具有自解释性。
3.  **返回值**：优先 JSON，超大结果需摘要或分页，错误信息结构化。返回给模型的内容应经过压缩，去除无关 HTML/噪音。
4.  **错误处理**：区分可重试（429、5xx）与不可重试（4xx）错误，敏感操作需人工确认。

**实战案例**：
某运维 Agent 需根据日志排查故障。最初 Tool 直接返回原始 50MB 日志文件，导致 Token 瞬间耗尽且模型无法聚焦。优化后，Tool 先执行 `grep - ERROR` 抽取报错行及前后 2 行上下文，将返回文本压缩至 2KB 以内，诊断成功率由 0% 提升至 80%。

**代码示例（带超时与异常处理的 Tool）**：
```python
from langchain_core.tools import tool
import requests
from functools import wraps
import time

def handle_errors(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        try:
            return func(*args, **kwargs)
        except requests.exceptions.Timeout:
            return "Error: API request timed out after 5s. Please try again."
        except Exception as e:
            return f"Error: {str(e)}"
    return wrapper

@handle_errors
@tool
def get_current_stock_price(symbol: str) -> str:
    """获取实时股价。仅当用户明确询问股票价格时使用。参数必须是有效的股票代码（如 AAPL）。"""
    # 限制超时，防止 Agent 阻塞
    resp = requests.get(f"https://api.market.com/price?q={symbol}", timeout=5)
    return resp.json()
```

**面试金句**：
*   **Q：Tool 与普通函数的区别？**
*   **A**：Tool 多了面向模型的元数据（描述、Schema）和统一执行接口（日志、超时、权限）。普通函数直接由代码逻辑调用，Tool 由 LLM “意念” 驱动调用。
*   **Q：工具返回 10MB 日志怎么办？**
*   **A**：不应直接返回。应摘要、写入对象存储返回链接、或提供 grep 工具缩小范围。直接塞入上下文会导致 Token 爆炸且淹没注意力。

## 常见考点
1.  **并行调用**：当模型返回多个 `tool_calls` 时，如何识别无依赖关系并进行并行执行以降低延迟？
2.  **Tool Choice 参数**：`auto`（模型决定）、`none`（强制不调用）、`{"type": "function", "function": {"name": xxx}}`（强制指定）。

**对比表格：Tool Calling vs Direct API Execution**

| 维度 | Tool Calling | Direct API Execution (传统代码) |
| :--- | :--- | :--- |
| **触发方** | LLM (基于意图推理) | 硬编码逻辑 / 用户点击 |
| **灵活性** | 极高，可动态组合工具 | 低，需预定义流程 |
| **上下文感知** | 结合对话历史动态决策 | 通常无状态或状态固化 |
| **容错性** | 需处理模型幻觉参数 | 仅处理系统异常 |
| **调试难度** | 较高 (非确定性输出) | 较低 (确定性逻辑) |

## 结构化回答

**30 秒电梯演讲：** Tool 就是给模型发一个"带说明书的遥控器"——封装了名称、描述、参数 Schema 和执行函数。它和普通函数的区别是多了面向模型的元数据，由 LLM 意念驱动调用。描述写不好模型就选错，返回值太大 Token 就爆，这是 Tool 设计的两大命门。

**展开框架：**
1. **描述是选择命门** — 描述要含功能、何时用、何时不用、Few-shot 示例，直接决定模型会不会选这个工具。
2. **参数少而精** — 优先枚举避免自由文本、日期统一 ISO 格式、参数名自解释，降低幻觉概率。
3. **返回值要压缩** — 优先 JSON，超大结果要摘要/分页/给 grep 工具缩小范围；直接返回 50MB 日志会让 Token 爆炸。
4. **错误分级处理** — 区分可重试（429/5xx）与不可重试（4xx），敏感操作（删数据）需人工确认。

**收尾：** 我做过运维 Agent，最初 Tool 直接返回 50MB 日志 Token 瞬间耗尽，改成 grep 报错行压缩到 2KB 后诊断成功率从 0% 升到 80%。您想深入聊描述写法、参数设计还是大结果处理？
