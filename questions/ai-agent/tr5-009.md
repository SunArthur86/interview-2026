# 在LLM应用架构中，Map-Reduce 模式是如何处理长文档摘要任务的？它有什么潜在问题？

Map-Reduce 模式通过切分与汇总两阶段处理超长文本：Map阶段将长文档分块并各自生成摘要，Reduce阶段将各分块摘要拼接后再次汇总。其优势在于突破单次推理Token限制，但潜在问题是信息丢失（Map阶段过滤细节）和传递偏差（递归累积误差）。优化方案包括重叠切分保持上下文、Prompt侧重关键信息提取、结构化摘要降低遗漏风险。

## 技术原理

- **Map 阶段并行生成子摘要，Reduce 阶段汇总生成最终结果**：Map 阶段把长文档按固定 token 切成多个 chunk，对每个 chunk 并行调用 LLM 生成局部摘要（彼此无依赖，可并发加速）；Reduce 阶段把所有子摘要拼接，再让 LLM 做一次汇总。当子摘要总量仍超窗口时，Reduce 可递归进行（Map-Reduce-Reduce）。
- **解决超长文本超出上下文窗口的限制**：单次推理受模型上下文窗口约束（如 128K），百万字文档无法一次性处理。分块后每个 chunk 远小于窗口上限，以"空间（多次调用）换算力（单次上下文长度）"，让任意长度文档都可处理。
- **潜在风险：信息丢失与递归汇总偏差累积**：Map 阶段每个 chunk 独立摘要会丢失跨 chunk 的全局信息（如人物关系、伏笔）；Reduce 递归层数越多，底层细节在层层压缩中损失越大，偏差逐级放大。
- **补偿手段：重叠切分、优化 Prompt、结构化输出**：相邻 chunk 保留 10%-20% 重叠保持上下文连续；Map 的 Prompt 强调关键实体/数字/结论的提取而非笼统概括；用结构化（JSON）摘要降低关键信息遗漏概率。

## 代码示例

Python 实现 Map-Reduce 摘要（带重叠切分与并发）：

```python
import asyncio
from typing import List

def chunk_text(text: str, size: int = 3000, overlap: int = 500) -> List[str]:
    chunks = []
    start = 0
    while start < len(text):
        chunks.append(text[start:start + size])
        start += size - overlap              # 重叠切分保持上下文连续
    return chunks

async def map_summarize(chunk: str) -> str:
    prompt = "提取关键实体、数字、结论，输出结构化摘要，忽略冗余：" + chunk
    return await llm.complete(prompt)

async def reduce_summarize(summaries: List[str]) -> str:
    return await llm.complete("综合以下子摘要生成总摘要：\n" + "\n".join(summaries))

async def map_reduce_summarize(doc: str) -> str:
    chunks = chunk_text(doc, size=3000, overlap=500)
    partial = await asyncio.gather(*[map_summarize(c) for c in chunks])  # Map 并发
    # 若子摘要总量仍超窗口，递归再 Map-Reduce
    return await reduce_summarize(partial)
```

## 对比/选型

| 方案 | 适用场景 | 信息完整性 | 成本 |
|------|----------|------------|------|
| Stuff（全塞进去） | 文档 < 上下文窗口 | 最好 | 单次调用 |
| Map-Reduce | 超长文档、需并行 | 中（有丢失） | 多次调用 |
| Refine（滚动精炼） | 中等长度、强调细节 | 较好 | 串行，慢 |
| Map-Reduce + 重叠 | 长文档、需平衡 | 较好 | 多次调用 |

## 常见坑/注意事项

- **分块大小与重叠的权衡**：chunk 太大单次摘要抓不住重点；太小则跨 chunk 信息割裂。经验值 chunk≈模型窗口的 1/4，重叠 10%-20%。
- **Map 摘要的 Prompt 决定上限**：笼统的"总结这段"会丢细节，应针对任务定制（如"提取所有金额、日期、责任人"），最好要求结构化输出便于 Reduce 对齐。
- **Reduce 偏差累积**：递归 Reduce 层数越多偏差越大，层数尽量 ≤2-3 层；可在 Reduce 阶段引入原文片段做 grounding 校准。
- **顺序敏感内容不适合纯切分**：时间线、推理链这类强顺序内容，分块会打断因果，建议改用 Refine 滚动精炼。

## 结构化回答

**30 秒电梯演讲：** Map-Reduce 处理长文档摘要就是分治法——Map 阶段把长文档切块各自并行摘要，Reduce 阶段把子摘要拼接再汇总。像把厚书分给多人读梗概，主编最后汇总成总摘要。它突破了单次 Token 限制，但两大隐患是 Map 过滤丢细节、Reduce 递归累积偏差。

**展开框架：**
1. **两阶段流程** — Map 阶段长文档分块并行生成子摘要，Reduce 阶段拼接子摘要再汇总生成最终结果，以空间换算力。
2. **核心价值** — 解决超长文本超出上下文窗口的限制，分块处理突破单次推理 Token 上限。
3. **两大隐患与优化** — Map 易过滤丢细节、Reduce 递归累积传递偏差；用重叠切分保上下文连续、结构化提取防关键信息遗漏。

**收尾：** 我做过百万字文档摘要，Map-Reduce 是唯一能跑通的方案，但得用重叠切分和结构化 Prompt 压细节丢失。您想聊 Map 分块大小怎么定，还是 Reduce 偏差怎么校准？
