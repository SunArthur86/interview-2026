# 短期记忆

短期记忆一般指当前会话或任务周期内模型可直接访问的信息，受 Token 上限与延迟约束。

### 核心形式
1.  **会话上下文**：按时间拼接完整历史。信息保真度高，但随对话变长会导致成本增加、延迟升高及模型注意力分散（"迷失中间"现象，Lost in the Middle）。
2.  **滑动窗口**：仅保留最近 K 轮或 N 个 Token。成本可控，但可能丢失早期关键约束。
3.  **摘要缓存**：将旧对话压缩为摘要，仅保留摘要和近期对话。这是滑动窗口的升级版，能保留更久远的核心意图。

### Token 管理
由于计费和上下文窗口限制，需进行预算分配：System 指令 > 安全策略 > 工具定义 > 高优先级记忆 > 近期对话。超限时常采用截断、摘要或检索增强处理。

### 实现细节与边界
*   **System Prompt 优先级**：必须始终保存在上下文最顶部，即使被截断也不能丢弃。
*   **Token 计算优化**：Message 中的 JSON 键（如 `role`, `content`）也会占用 Token，需计入预算（通常单条 Message 消耗约 5-10 tokens 的额外开销）。
*   **滑动策略**：通常保留 `System` + `Recent N`，或者在保留 `System` 的基础上，保留 `Summary` + `Recent N`。

### 架构图：Token 预算分层管理
```text
┌─────────────────────────────────────────────────────────────┐
│                    Context Window (L)                       │
├─────────────────────────────────────────────────────────────┤
│ Layer 1: System Prompt (Fixed, High Priority)              │
├─────────────────────────────────────────────────────────────┤
│ Layer 2: Tools & Safety (Static or Semi-dynamic)            │

│ Layer 3: Compressed Memory (Summary of Distant History)     │

│ │  User Msg   │ │  Agent Msg  │ │  User Msg   │   ...       │
│ └─────────────┘ └─────────────┘ └─────────────┘             │

```

### 实战案例
*   **踩坑经验**：在实现滑动窗口时，若简单地按“条数”截断，可能会因为中间某条回复包含大量代码而截断了最新的用户指令。实战中必须改为按“Token数”滑动，且强制保留最近一条 User Message，哪怕超出预算。

### 对比表格：策略选择
| 策略 | 适用场景 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **全量上下文** | 短任务、代码分析 | 信息无损，推理准确 | 极易超限，成本高，长时注意力分散 |
| **滑动窗口** | 实时对话、闲聊 | 实现简单，低延迟，成本低 | 容易丢失早期设定的“人设”或“核心约束” |
| **摘要缓存** | 长周期 Agent、客服 | 兼顾上下文长度与历史意图 | 摘要生成有额外成本与延迟，可能丢失细节 |

### 代码逻辑（Python）
```python
import tiktoken

def build_context(system_msg, history, max_tokens, enc):
    context = [system_msg]
    used_tokens = len(enc.encode(str(system_msg)))
    
    # 倒序遍历，保留最近对话，直到填满预算
    for msg in reversed(history):
        msg_tokens = len(enc.encode(msg['content']))
        if used_tokens + msg_tokens > max_tokens:
            break
        context.insert(1, msg) # 插入到 system 之后，保持顺序
        used_tokens += msg_tokens
    return context
```

## 记忆要点

- 核心定义：当前会话内模型直接访问的信息，受 Token 上限与延迟约束。
- 三种形式：全量上下文（保真但贵）、滑动窗口（截断旧信息）、摘要缓存（压缩历史）。
- 预算优先级：System > 安全 > 工具 > 记忆 > 近期对话，System 必须置顶。
- 实战避坑：按 Token 数而非条数截断，强制保留最近一条用户指令。
- 额外开销：Message 中的 JSON 键（role/content）也会消耗约 5-10 tokens。

## 结构化回答

**30 秒电梯演讲：** 短期记忆就是当前会话里模型直接能看的信息，受 Token 上限约束。三种形式：全量上下文保真但贵、滑动窗口截断旧信息、摘要缓存压缩历史。最大坑是别按"条数"截断要按"Token 数"，而且 System Prompt 永远置顶不能丢。

**展开框架：**
1. **三种形式权衡** — 全量上下文信息无损但极易超限；滑动窗口实现简单但丢早期约束；摘要缓存兼顾长度和意图但有压缩成本。
2. **Token 预算分层** — System > 安全策略 > 工具定义 > 高优先级记忆 > 近期对话，System 必须始终置顶，截断也不能丢。
3. **按 Token 数不按条数** — 按条数截断可能因某条回复含大量代码而截断最新用户指令；必须按 Token 滑动并强制保留最近一条 User Message。
4. **隐藏开销** — Message 的 JSON 键（role/content）每条额外消耗 5-10 tokens，预算计算要算进去。

**收尾：** 我踩过坑——按条数截断把含代码的回复留下、最新用户指令丢了，改成按 Token 数滑动 + 强制保留最近 User Message 才稳。您想深入聊窗口策略、摘要压缩还是 Token 预算分配？
