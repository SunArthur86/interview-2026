# 记忆系统概述

### 记忆系统概述

**定义**：Agent 为实现多轮对话、长期任务与个性化，对信息进行存储、组织、检索与更新的机制。记忆是 Agent 拥有“人格”和“历史经验”的基础。

**必要性**：
1.  **多轮一致性**：避免前后矛盾（如上文说“我住北京”，下文问天气不应查上海）。
2.  **长任务**：延续工具调用链和子目标状态。
3.  **个性化**：记住用户偏好与术语。
4.  **成本控制**：不必每次全量输入历史，按需加载，利用 RAG 压缩上下文。

**分类体系**：
1.  **短期记忆**：对话上下文，易失但访问快。通常由 LLM 的 Context Window 承载（Sliding Window 策略）。
2.  **长期记忆**：向量库、数据库、知识图谱，持久化存储。通过 RAG 检索。
3.  **情景记忆**：“发生了什么”（特定事件，如用户上周投诉过什么）。
4.  **语义记忆**：“知道什么”（通用知识，如公司政策）。

**人类记忆类比**：
*   **感觉记忆**：流式输入缓冲。
*   **工作记忆**：当前的 Scratchpad（思维链、中间变量）。
*   **长期记忆**：外挂的知识库。

**面试金句**：
*   **Q：Agent 为什么需要记忆？**
*   **A**：没有记忆就是无状态函数，无法处理多轮任务、保持一致性或提供个性化服务。记忆是 Agent 从“聊天机器人”进化为“助理”的关键。

**代码示例（记忆写入与检索）**：
```python
# 模拟写入记忆
def write_memory(user_id: str, content: str, type: "episodic|semantic"):
    embedding = get_embedding(content)
    vector_db.insert(id=uuid(), values=embedding, metadata={"user": user_id, "type": type})
    
# 模拟读取记忆
def recall_memory(user_id: str, query: str, top_k: int = 3):
    query_vec = get_embedding(query)
    # 简单的向量检索 + 过滤用户ID
    results = vector_db.search(query_vec, filter={"user": user_id}, limit=top_k)
    return [r.text for r in results]
```

**实战案例**：
在客服 Agent 中，我们发现全量保存历史对话导致检索噪音极大，Agent 经常混淆不同订单的细节。**教训**：不能只存对话记录，必须引入**“信息提取”**环节，将非结构化对话清洗为结构化 Facts（如 `Order_ID: 123, Status: Refunded`）再存入 KV 或图数据库。

**记忆存储策略对比**：

| 存储介质 | 适用类型 | 写入速度 | 检索特点 | 典型场景 |
| :--- | :--- | :--- | :--- | :--- |
| **Context Window** | 短期记忆/缓存 | N/A (即时) | 最快，但受限于 Token 长度 | 当前对话流、思维链上下文 |
| **Vector DB** | 语义/情景记忆 | 中等 | 模糊匹配，适合相似度查询 | 查找“上次提到的那个类似问题” |
| **KV Store (SQL/NoSQL)** | 语义/事实 | 快 | 精确匹配，适合 Key-Value | 用户偏好设置、具体的订单号 |
| **Knowledge Graph** | 复杂关系 | 慢 | 跳跃查询，推理能力强 | 企业知识库、人物关系网 |

## 常见考点
1.  **记忆存储策略**：什么时候存？（关键信息提取后存）；存什么？（而非全量存储原始对话，需 Summarize 或提取 Facts）。
2.  **记忆检索策略**：什么时候检索？（每次对话开始前）；检索多少？（根据 Token 预算动态调整 Top-K）。
3.  **窗口管理**：当对话历史超过上下文窗口时，如何处理？（滑动窗口、摘要压缩、选择性遗忘）。

## 记忆要点

- 定义：存储、组织、检索信息的机制，是Agent拥有”人格”和”历史经验”的基础。
- 分类：短期记忆（Context Window）、长期记忆（Vector DB/SQL）、情景/语义记忆。
- 必要性：保持多轮一致性、延续长任务状态、提供个性化服务、控制成本。
- 存储策略：非结构化对话需清洗为结构化Facts存入KV或图库，避免检索噪音。
- 窗口管理：超长历史用滑动窗口、摘要压缩或选择性遗忘处理。

## 结构化回答

**30 秒电梯演讲：** 记忆系统是 Agent 从”聊天机器人”进化成”助理”的关键——没有记忆就是无状态函数。分短期（Context Window）和长期（Vector DB/SQL），还能细分情景记忆（发生了什么）和语义记忆（知道什么）。核心坑是别全量存原始对话，要清洗成结构化 Facts 再入库。

**展开框架：**
1. **四大必要性** — 多轮一致性（别前后矛盾）、长任务延续、个性化（记偏好）、成本控制（按需加载别全量塞）。
2. **分类体系** — 短期（Context Window 滑动窗口）、长期（Vector DB/知识图谱）、情景记忆（事件）、语义记忆（通用知识）。
3. **存储介质选型** — Context Window 最快但贵、Vector DB 适合模糊匹配、KV Store 适合精确事实、知识图谱适合关系推理。
4. **存储策略命门** — 不能只存原始对话，必须引入信息提取环节清洗成结构化 Facts（如 Order_ID:123, Status:Refunded）再入库，否则检索噪音极大。

**收尾：** 我做过客服 Agent，全量存历史对话导致检索混淆不同订单，引入 Facts 提取后清爽多了。您想深入聊存储介质选型、Facts 提取还是窗口管理？
