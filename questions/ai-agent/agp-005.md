# Few-shot 示例顺序会影响结果吗

会。这是 LLM 中的“最近偏差”或“位置偏差”表现。模型可能更关注开头或结尾的示例，或者受示例顺序隐含的某种逻辑关系影响。工程上不应依赖某种“神秘的最佳顺序”，而应通过明确的规则（如按随机顺序、按难度排序）和严格的格式约束来增强鲁棒性；建议通过 A/B 测试对比不同顺序（如正序、逆序、随机）的效果差异。

**实战案例**：在做情感分类时，测试集包含 80% 负面评论。如果 Few-shot 示例按照“正、正、负、负”排列，模型往往会受到 Majority Bias（多数类偏差）和 Recency Bias（近因偏差）的双重影响，倾向于预测为“负面”。将示例调整为“正、负、正、负”交叉排列后，预测平衡性显著改善。

**代码示例**：
```python
# 动态打乱示例顺序以缓解位置偏差
import random

def build_few_shot_prompt(examples, user_query):
    # examples 是包含 (input, output) 的列表
    # 每次请求时动态打乱顺序，避免模型死记硬背顺序特征
    shuffled_examples = random.sample(examples, k=len(examples))
    
    demo_block = ""
    for ex in shuffled_examples:
        demo_block += f"Input: {ex['input']}\nOutput: {ex['output']}\n\n"
        
    return f"{instructions}\n{demo_block}Input: {user_query}\nOutput:"
```

## 边界情况
1. **类别极度不平衡**：当正负样本比例极度悬殊（如 1:99）时，简单的打乱可能无法消除偏差，可能需要采用过采样少数类或欠采样多数类的策略构建示例池。
2. **逻辑依赖顺序**：在某些数学推理或代码解释任务中，示例本身可能隐含教学逻辑（从简单到复杂），此时完全随机打乱可能会破坏“学习路径”，反而降低效果，需权衡随机性与逻辑性。
3. **首尾效应**：模型对首个示例和最后一个示例的关注度通常最高，若关键案例被夹在中间可能被忽略，需确保核心约束出现在首尾位置。

## 面试追问
1. **偏差类型**：除了位置偏差，还有哪些常见偏差？（Majority bias 多数标签偏差、Recency bias 近因偏差、Self-consistency 一致性偏差）。
2. **缓解措施**：如何彻底消除顺序影响？（除了动态打乱，可使用集成投票机制，即用多种不同顺序生成多次答案再投票选出最终结果）。
3. **格式干扰**：示例中的格式不一致会导致什么问题？（可能导致模型输出格式混乱，需严格保持示例模板一致性，特别是 Key 的顺序）。

## 易错点
1. **静态顺序导致的死记硬背**：在服务端写死 Few-shot 示例的顺序，可能导致模型在特定位置输出特定结果，一旦调整顺序或插入新示例，效果骤降。
2. **忽视输入样本的顺序**：只打乱了示例的顺序，但忽略了如果用户一次性输入了多个待处理 item，这些 item 的排列顺序也会影响模型的处理优先级和结果。

## 技术原理

Few-shot 示例顺序影响结果，根源在于 LLM 的 Attention 机制和训练分布偏差：

- **位置偏差（Position Bias）**：Transformer 的 Self-Attention 对序列首尾的 token 关注度更高（首部是"锚定效应"，尾部是"近因效应"）。放在中间的示例容易被"稀释"，关键约束应出现在首尾位置才能最大化影响力。
- **多数类偏差（Majority Bias）**：如果 Few-shot 示例里某一类标签占多数（如 4 个正面 1 个负面），模型会偏向预测多数类。这是训练数据分布不均衡的延续——模型学到了"多数类先验"。交叉排列（正负正负）能平衡这种偏差。
- **隐含逻辑传递**：示例顺序可能隐含某种教学逻辑（从简到繁、从具体到抽象），模型会捕捉这种顺序特征并外推。如果顺序随机但模型误以为有逻辑，反而会误导推理；反之数学推理等任务按从简到繁排序能形成"学习路径"提升效果。

工程上的对策不是寻找"神秘的最佳顺序"，而是通过动态打乱、按难度排序、集成投票等手段让结果对顺序不敏感。

## 注意事项

1. **别静态写死顺序**：服务端写死 Few-shot 顺序会让模型死记位置特征，一旦调整顺序或插入新示例效果骤降。建议每次请求动态打乱或按 Embedding 检索最相关示例。
2. **正负样本交叉排列**：类别不平衡时，简单打乱可能不够，需主动把正负样本交叉排列防 Majority Bias；极端不平衡（1:99）还要配合过采样少数类。
3. **核心约束放首尾**：模型对首尾关注度高，关键案例或核心约束应出现在首尾位置，别夹在中间被忽略。
4. **数学推理按难度排序**：有逻辑依赖的任务（数学、代码解释）按从简到繁排序能保留学习路径，完全随机反而降效，需权衡随机性与逻辑性。

## 对比表

| 偏差类型 | 表现 | 缓解策略 | 适用场景 |
|:---|:---|:---|:---|
| **位置偏差（Position）** | 对首尾 token 关注度高，中间被稀释 | 关键约束放首尾位置 | 所有 Few-shot 任务 |
| **近因偏差（Recency）** | 更关注最后一个示例 | 集成投票：多顺序生成再投票 | 分类、生成 |
| **多数类偏差（Majority）** | 示例里某类标签占多数时偏向预测该类 | 正负样本交叉排列；过采样少数类 | 情感分析、风控 |
| **隐含逻辑传递** | 模型捕捉示例顺序特征并外推 | 数学推理按从简到繁保留学习路径 | 数学、代码推理 |

| 顺序策略 | 实现方式 | 优点 | 缺点 |
|:---|:---|:---|:---|
| **动态打乱** | `random.sample(examples)` 每次请求重排 | 防死记硬背，鲁棒 | 破坏逻辑依赖 |
| **按难度排序** | 按示例长度或 Embedding 距离排序 | 形成学习路径 | 仅适合推理类任务 |
| **集成投票** | 多顺序生成 N 次，取多数答案 | 彻底消除顺序影响 | 成本 N 倍 |
| **Embedding 检索** | 按与 query 相似度动态选示例 | 相关性最高 | 需向量库 |
