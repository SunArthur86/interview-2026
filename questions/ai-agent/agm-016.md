# 如何评测记忆系统好坏

### 离线评估
1.  **检索质量**：基于标注集计算 Recall@K（召回率）和 Precision@K（精确率）。
2.  **记忆一致性**：检查新写入的记忆是否与旧记忆冲突（Contradiction Rate），如“用户喜欢猫”与“用户对猫过敏”共存。
3.  **摘要质量**：使用 LLM-as-a-Judge 评估摘要是否保留了关键信息且无幻觉。

### 在线评估
1.  **任务成功率**：最终任务是否完成（如预约是否成功）。
2.  **用户干预率**：用户手动纠正 Agent 错误记忆的频率。
3.  **Token 效率**：检索到的记忆中有多少 Token 被模型实际引用（利用 Attention 权重分析）。
4.  **延迟**：记忆检索耗时（P99）。

### 对比表格：离线 vs 在线评估
| 维度 | 离线评估 | 在线评估 |
| :--- | :--- | :--- |
| **核心指标** | Recall@K, MRR, NDCG | 任务成功率, 用户纠正率 |
| **数据来源** | 历史标注集 / 黄金测试集 | 真实用户流量日志 |
| **反馈周期** | 短（天/小时级），快速迭代 | 长（周/月级），依赖业务表现 |
| **主要目的** | 模型选型、参数调优 | 验证业务价值、监控线上健康度 |

### Bad Case 归因分析
```text
Bad Case 发生
      │
      ├─ 检索错 (Recall低) ─> 向量模型不匹配 / 切片策略不合理 / 查询扩展失效
      │
      ├─ 排序错 (Precision低) ─> Reranker 模型能力不足 / 缺乏时间衰减权重
      │
      ├─ 写入错 (污染) ─> 提取信息时产生幻觉 / 未去重
      │
      └─ 摘要错 (失真) ─> 压缩率过高 / 丢失了关键实体/意图
```

### 实战案例
在客服场景中发现 Recall@5 很高，但用户仍反复说“你记错了”。排查发现记忆中存在过期的物流单号（旧信息未被覆盖或标记失效），导致 Agent 优先引用了旧数据。引入“时间衰减权重”和“状态校验”机制后，准确率提升 20%。

### 代码示例 (Python - LLM-as-a-Judge 逻辑)
```python
def evaluate_memory_consistency(new_mem: str, existing_mems: list[str]):
    prompt = f"""
    Existing Memories: {existing_mems}
    New Information: {new_mem}
    Question: Does the New Information contradict any Existing Memories?
    Answer strictly 'YES' or 'NO'.
    """
    response = llm.generate(prompt)
    if "YES" in response:
        log_conflict(new_mem, existing_mems) # 记录冲突用于后续分析
        return False
    return True
```

### ## 常见考点
- 如何构建记忆系统的标注数据集？（答案：根据对话日志反推“当时如果有哪条记忆，任务会完成得更好”）
- 如何判断记忆是否过时？（答案：结合时间衰减权重，或显式询问用户确认）
- Recall 和 Precision 在记忆系统中哪个更重要？（答案：通常 Recall 更重要，因为 Precision 可以交给 LLM 在上下文中过滤，但漏掉关键信息往往无法挽回）

### ## 边界情况
1.  **冷启动评测**：新上线阶段无历史数据或用户反馈时，如何建立 Baseline？（答案：使用少量 Golden Set 进行合成测试，或利用 Agent 自我博弈生成评测数据）。
2.  **多模态记忆**：当记忆包含图片、音频时，如何评估检索质量？（答案：需定义模态对齐的评测标准，如“图文匹配度”，通常需依赖多模态 Judge 模型）。
3.  **隐私泄露评测**：在评测检索效果的同时，需增加“隐私合规”指标，评估检索结果是否包含了不应出现的敏感信息（如 PII）。

### ## 面试追问
1.  如何量化“记忆重要性”？即如何判断检索回来的哪条记忆对当前任务的完成贡献最大？
2.  如果发现线上“任务成功率”上升，但“用户干预率”也上升了，可能是什么原因？
3.  你提到的“Token 效率”利用 Attention 权重，具体是如何实现的？如何处理长上下文中 Attention 饱和的问题？

### ## 易错点
1.  **过度依赖离线指标**：认为离线 Recall@K 高就能保证线上效果好，忽略了 Context Window 有限导致的“信息干扰”问题，即检索回的信息虽然相关，但挤占了关键 Prompt 的位置。
2.  **忽视“遗忘”机制的评测**：只评测记住了什么，不评测“该忘的是否忘了”。例如用户搬家后，系统是否能有效降低旧地址的权重或将其标记失效。

## 结构化回答

**30 秒电梯演讲：** 评测要分两层：离线看检索质量（Recall@K、记忆一致性冲突率、摘要质量用 LLM-as-Judge），在线看业务价值（任务成功率、用户干预率、Token 效率）。Bad Case 要归因到具体环节——检索错、排序错、写入错还是摘要错。最大的评测陷阱是只测"记住"不测"遗忘"，得评估旧信息是否有效降权失效。

**展开框架：**
1. **离线评估** — Recall@K 召回率、Precision@K 精确率、Contradiction Rate 冲突率；摘要质量用 LLM-as-Judge 看是否保留关键信息无幻觉。
2. **在线评估** — 任务成功率（预约是否成功）、用户干预率（手动纠正频率）、Token 效率（Attention 权重看多少被引用）、延迟 P99。
3. **归因与陷阱** — Bad Case 归因到检索/排序/写入/摘要四环节；陷阱是只测记住不测遗忘，要评估旧信息降权或失效。

**收尾：** 做客服时踩过坑——Recall@5 很高但用户老说"你记错了"，排查是过期物流单号没被覆盖，加时间衰减和状态校验后准确率提升 20%。您想聊哪块，Attention 引用分析还是 ground truth 怎么构造？
