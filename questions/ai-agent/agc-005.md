# Agent 的记忆一般怎么设计

Agent 的记忆设计核心目标是解决 **LLM 上下文窗口有限** 和 **长期知识持久化** 的问题。通常采用分层架构设计。

**1. 记忆分层架构**

*   **Sensory Memory (感官记忆)**
    *   **作用**：最原始的输入缓冲，如当前 Prompt、最新的几轮对话。
    *   **实现**：直接作为 LLM 的 Input Context。

*   **Short-term Memory (短期记忆 / 工作记忆)**
    *   **作用**：存储当前任务链中的关键信息、中间步骤的推理结果。
    *   **实现**：通常使用滑动窗口或 Token 限制内的摘要机制。

*   **Long-term Memory (长期记忆)**
    *   **作用**：存储跨会话的用户偏好、历史事实、知识库。
    *   **实现**：向量数据库 存储。通过 Embedding 相似度检索 (RAG) 将相关记忆注入当前上下文。

**2. 关键设计细节**
*   **Reflect / 重要性判断**：不是所有信息都值得存入长期记忆。需要有一个机制（让 LLM 评分或筛选）判断哪些是「重要事实」，哪些是「临时寒暄」。
*   **带时间戳与来源**：每条记忆应记录 `timestamp` 和 `source`，支持时间范围检索和溯源，方便更新或撤销过时信息。
*   **结构化与半结构化**：对于关键数据（如用户姓名、喜好），尽量提取为结构化 JSON 存储，而非纯文本，以便精准查询。

**记忆读写流程图：**
```text

  │ User Query   │ ────────────────────────▶ │ Vector Database   │
  └──────┬───────┘                          │ (Long-term Memory)│
         │                                  └─────────┬─────────┘
```

**实战案例：**
在构建企业知识库 Agent 时，我们遇到一个严重问题：当公司政策（如报销额度）更新后，Agent 依然检索到旧的向量文档并给出错误建议。解决方案是引入“记忆衰减与版本控制”：将旧记忆标记为 `is_obsolete` 而非物理删除，并在 Prompt 中强调“优先检索时间戳在最近一个月内的记忆”，确保信息的时效性。

**边界情况：**
- **上下文窗口溢出**：即使使用了 RAG，检索到的相关记忆依然可能撑爆当前 Prompt 的 Token 限制。需引入“重排序”和“记忆压缩”机制，只保留 Top-K 高价值片段。
- **记忆冲突**：长期记忆中可能存在矛盾的信息（如用户在不同时间表达了相反的喜好），检索时需返回元数据供 LLM 判别优先级。

**面试追问：**
1. 当对话历史非常长时，如何保证 Agent 既能记住很久之前的细节，又不超出 Token 限制？（提示：滑动窗口、摘要树、MemGPT 概念）
2. 向量数据库检索（语义搜索）有时候不如关键词搜索精准（如查具体的订单号），如何设计混合检索策略？

**易错点：**
- **无差别存储**：将所有对话历史都存入向量库，导致检索噪音巨大，且写入成本高昂，且无法区分用户的“指令”和“闲聊”。

**代码示例（Memory 检索与写入）：**
```python
import chromadb
from datetime import datetime

class AgentMemory:
    def __init__(self):
        self.client = chromadb.Client()
        self.collection = self.client.create_collection("user_memories")

    def write(self, content, metadata={}):
        # 写入时带时间戳，便于后续清理或排序
        meta_with_ts = {
            **metadata, 
            "timestamp": datetime.now().isoformat()
        }
        # 这里可以加一层过滤，只有重要信息才调用 write
        # if self._is_important(content): 
        #    self.collection.add(...)
        self.collection.add(
            documents=[content],
            metadatas=[meta_with_ts],
            ids=[f"mem_{datetime.now().timestamp()}"]
        )

    def read(self, query, n_results=3):
        # 简单的语义检索
        results = self.collection.query(
            query_texts=[query], 
            n_results=n_results
        )
        return results

    # 易错点修正：清理过时记忆或处理冲突的逻辑
    def update_memory(self, old_id, new_content):
        self.collection.update(
            ids=[old_id],
            documents=[new_content],
            metadatas=[{"timestamp": datetime.now().isoformat(), "updated": True}]
        )
```

## 记忆要点

- 感官记忆：当前Prompt，直接作为Input。
- 短期记忆：任务链中的关键信息，用滑动窗口或摘要。
- 长期记忆：向量数据库存RAG，解决跨会话持久化。
- 写入机制：需带时间戳和来源，且需LLM判断重要性。
- 实战痛点：需解决上下文溢出（压缩）和记忆冲突（时效性）。

## 结构化回答

**30 秒电梯演讲：** Agent 记忆一般分三层，像人脑一样。感官记忆就是当前 Prompt 直接进上下文；短期记忆用滑动窗口或摘要存当前任务链的关键信息；长期记忆存到向量数据库靠 RAG 检索，解决跨会话持久化。设计上每条记忆都要带时间戳和来源，并且让 LLM 判断重要性，不能一股脑全存。

**展开框架：**
1. **三层架构** — 感官（当前输入）、短期（任务链摘要）、长期（向量库 RAG），各司其职。
2. **写入要挑** — 不是所有对话都值得存，让 LLM 评分筛选重要事实，避免噪音和写入成本。
3. **冲突与溢出** — 旧记忆标记 `is_obsolete` 而非删除保证时效；检索结果做重排序和压缩防 Token 溢出。

**收尾：** 我做企业知识库 Agent 踩过坑——政策更新后还在检索旧文档，加了时间戳优先策略才解决。您想深入聊哪块，混合检索还是记忆压缩？

## 延伸：用人类记忆模型设计 Agent 记忆有什么好处

> 合并自 `agm-002`（相似度 67%）

用人类记忆模型（如感觉记忆、短期/工作记忆、长期记忆）来设计 Agent 记忆架构，主要好处在于**模块化分工**与**可解释性**，同时为数据生命周期管理提供了理论指导。

**好处与对应关系**：
1.  **模块化架构**：
    - **感觉记忆**：对应流缓冲或原始输入队列，负责暂存海量原始信息，进行初步过滤。
    - **短期/工作记忆**：对应 Context Window 或 Scratchpad（思维链草稿纸）。用于存放当前推理过程、即时指令、高频访问的数据。特点是读写极快，但容量有限。
    - **长期记忆**：对应向量数据库、图数据库或磁盘存储。存放用户画像、历史事实、领域知识。特点是容量大，但访问有延迟。
2.  **生命周期管理指导**：
    - 人类记忆有“巩固”和“遗忘”机制。类比到 Agent，我们需要设计显式的工程逻辑：将重要信息从 Context 刷入向量库，或者定期清理过时的 Cache。
3.  **可解释性**：
    - 当开发者或用户询问 Agent 为什么“记得”某件事时，可以追踪到是从长期记忆检索的，还是当前对话中暂存的。

**风险与工程差异**：
- **无神经可塑性**：计算机不会像人脑那样通过突触连接自动增强记忆。必须显式实现记忆的写入、检索和更新算法。
- **冲突解决**：人脑能自动调和矛盾记忆，Agent 则需要明确的策略（如时间戳优先、信任度优先）来处理冲突信息。

```text
人类记忆模型 -> Agent 架构映射

+-------------------+
|  感觉记忆        | <---| 原始输入流 / 聊天缓冲区
| (瞬时, 海量)      |     |
+-------------------+     |
         | (注意力筛选)    |
         v                 |
+-------------------+     |
|  工作/短期记忆    | <---| LLM Context Window / Scratchpad
| (处理, 容量小)    |     |
+-------------------+     |
         | (重要性过滤/巩固) |
         v                 |
+-------------------+     |
|  长期记忆        | <---| Vector DB / Knowledge Graph
| (存储, 容量大)    |     |
+-------------------+     |
```

### 实战案例
在构建企业知识库 Agent 时，我们模拟海马体的“睡眠巩固”机制：当工作记忆中的对话轮数超过 5 轮且包含“决策”、“结论”等关键词时，异步触发摘要写入向量数据库。这一机制解决了长期记忆中充斥大量无效寒暄的问题，将检索准确率提升了 20%。

### 代码示例 (模拟记忆分级写入)
```python
class HumanMemoryAgent:
    def __init__(self):
        self.working_memory = [] # 对应 Context Window
        self.long_term_memory = VectorStore() # 对应 外部存储

    def process_input(self, text):
        # 1. 感知寄存：总是先进入工作记忆
        self.working_memory.append(text)
        
        # 2. 巩固机制：如果工作记忆过长，将旧信息压缩存入长期记忆
        if len(self.working_memory) > 10:
            old_info = self.working_memory.pop(0)
            self.long_term_memory.add(text=old_info, metadata={"source": "consolidated"})
            
        # 3. 检索：从长期记忆召回相关信息辅助当前推理
        relevant_mem = self.long_term_memory.similarity_search(text)
        return self._generate_response(self.working_memory, relevant_mem)
```

## 常见考点
1.  **如何实现“遗忘”机制？**：可以基于 LRU（最近最少使用）算法清理向量库，或基于时间窗口让记忆过期，防止噪声污染。
2.  **RAG（检索增强生成）在人类记忆模型中处于什么位置？**：属于长期记忆的“回溯”过程，即在需要时从外部知识库检索相关信息加载到工作记忆中。

## 记忆要点

- 人类记忆映射：感觉记忆对应输入流；短期记忆对应Context/Scratchpad；长期记忆对应向量库。
- 好处：模块化分工明确，指导数据生命周期（如巩固机制），提升架构可解释性。
- 工程差异：计算机需显式实现写入/检索/遗忘，无法像人脑自动突触增强。
- 实战：模拟海马体巩固机制，对话超5轮含关键词时异步写向量库，检索准确率提升20%。

## 结构化回答

**30 秒电梯演讲：** 用人类记忆模型设计 Agent 好处是模块化分工和可解释性。感觉记忆对应输入流缓冲，短期记忆对应 Context Window 或 Scratchpad，长期记忆对应向量库。这种分层给了数据生命周期管理理论指导——像海马体的巩固机制，重要信息从工作记忆刷入长期记忆。最大工程差异是计算机没有神经可塑性，写入、检索、遗忘都得显式实现。

**展开框架：**
1. **三层映射** — 感觉记忆到输入流、短期记忆到 Context、长期记忆到向量库，分工明确。
2. **生命周期指导** — 巩固机制让重要信息刷入长期记忆，遗忘机制定期清过时 Cache。
3. **工程差异** — 无神经可塑性，冲突解决要靠时间戳或信任度优先策略显式处理。

**收尾：** 我做企业知识库时模拟海马体巩固——对话超 5 轮且含"决策""结论"关键词才异步写向量库，避免无效寒暄污染，检索准确率提升 20%。您想深入聊哪块，遗忘机制设计还是冲突解决策略？
