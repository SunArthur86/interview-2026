# GraphRAG 的原理是什么?它与传统向量 RAG 有什么区别

- **GraphRAG** 是微软提出的知识图谱增强 RAG 方案,通过构建结构化知识图谱来增强检索.

- **ASCII 架构图 (GraphRAG 流程):**
```
输入文档集合
  │
  ▼ [阶段 1: 索引构建 (离线)]
┌─────────────────────────────────────┐
│ 1. 文本分块 → 实体/关系抽取         │
│    (LLM 提取 Nodes & Edges)         │
├─────────────────────────────────────┤
│ 2. 构建同质图 → Leiden 算法社区发现 │
│    (将节点聚类为 Community)         │
├─────────────────────────────────────┤
│ 3. 生成社区摘要                     │
│    (LLM 总结每个社区的 "主题")      │
└─────────────────────────────────────┘
  │
  ▼ [阶段 2: 检索生成 (在线)]
用户查询
  │
  ├─→ 检索相关实体 (社区内)
  │
  ├─→ 检索相关社区摘要 (全局)
  │
  ▼ → 组装 Context → LLM 生成答案
```

- **核心原理:**
1. 文档预处理阶段:用 LLM 从文档中抽取实体和关系
2. 构建知识图谱:实体为节点,关系为边
3. 社区发现:将相关实体聚类
4. 生成社区摘要:为每个社区生成高层摘要
5. 查询时:先匹配社区,再在社区内精确检索

- **与传统向量 RAG 的区别:**

| 维度 | 传统向量 RAG | GraphRAG |
| :--- | :--- | :--- |
| **索引结构** | 向量数据库 | 知识图谱 + 社区摘要 |
| **检索逻辑** | 语义相似度匹配 | 实体遍历 + 社区层级检索 |
| **全局理解** | 差 (难以跨文档汇总) | 强 (基于社区摘要) |
| **多跳推理** | 依赖向量偶然碰撞 | 基于显式路径推理 |
| **构建成本** | 低 (仅 Embedding) | 高 (大量 LLM 抽取) |
| **更新维护** | 容易 (增量 Vector) | 困难 (需局部重构图) |

- **适用场景:**
- 需要多跳推理的复杂问答
- 实体关系密集的领域(金融、法律、医学)
- 全局摘要类查询

- **代价:** 构图成本高(需要大量 LLM 调用),更新维护复杂.

- **实战案例:** 在某金融研报分析项目中，使用传统 RAG 无法回答“某行业全年的整体风险趋势”，因为答案散落在数百份报告中；切换到 GraphRAG 后，利用社区摘要直接生成了各风险主题的宏观分析，召回率提升了 40%，但索引构建耗时从 30 分钟增加到了 4 小时。

- **代码示例:**
```python
# 使用 LlamaIndex 模拟 GraphRAG 的核心：基于实体关系的检索
from llama_index.core import KnowledgeGraphIndex
from llama_index.graph_stores import NebulaGraphStore

# 1. 构建图索引 (离线)
kg_index = KnowledgeGraphIndex.from_documents(
    documents,
    storage_context=storage_context,
    max_triplets_per_chunk=10,
    include_embeddings=True
)

# 2. 查询时进行多跳推理 (在线)
# 引用同质图中发现的社区路径
response = kg_index.as_query_engine(
    include_text=True, 
    retrieval_mode="keyword" # 显式基于实体/关键词查找
).query("A公司收购B公司后对供应链有什么影响?")
```

## 延伸：GraphRAG是什么？它和传统RAG有什么区别？

> 合并自 `sar-001`（相似度 72%）

🎯 本质：GraphRAG是微软提出的RAG增强方案，通过构建知识图谱来捕获文档间的实体关系，解决传统RAG只能回答局部问题、无法做全局推理的局限。

🧒 类比：传统RAG像在一堆散落的文件里按关键词搜索（找到相关文件但看不到文件之间的联系）；GraphRAG像先画了一张关系图（谁认识谁、谁在哪工作），然后基于关系图回答问题。

**实战案例**：在法律合同审查系统中，传统RAG无法回答“该集团的最终受益人是谁”这类跨文档问题，因为受益人信息分散在多层股权结构的文档中。引入GraphRAG后，通过提取实体关系构建股权图谱，Agent能够进行多跳推理，直接在图中追溯出最终控制人，准确率从20%提升至95%。

**代码示例（Python/NetworkX概念）**：
```python
import networkx as nx

# 1. 构建图谱（通常由LLM提取后构建）
G = nx.DiGraph()
G.add_edge("Company A", "Company B", relation="owns", stake="51%")
G.add_edge("Company B", "Company C", relation="owns", stake="60%")

# 2. 查询逻辑：找到所有指向C的节点
def find_owners(target_entity):
    return list(G.predecessors(target_entity))

# 实际应用中会将此结构索引入向量数据库或图数据库（如Neo4j）
print(f"Owners of C: {find_owners('Company C')}")
```

📊 GraphRAG vs 传统RAG：
| 特性 | 传统RAG | GraphRAG |
|------|---------------|----------|
| 数据结构 | 向量索引 | 知识图谱+向量 |
| 擅长 | 具体事实查询 | 全局/关系推理 |
| 示例 | "X的定义是什么" | "整个文档集的主要主题" |
| 构建 | 快(切分+向量化) | 慢(LLM提取实体关系) |
| 更新 | 增量容易 | 需重建社区 |

GraphRAG工作流程：
1. Indexing阶段
  - 用LLM从文档中提取实体和关系
  - 构建知识图谱（节点=实体，边=关系）
  - 用Leiden算法进行社区检测（将图分为主题社区）
  - 为每个社区生成摘要（LLM总结）

2. Query阶段
  - 全局查询：从社区摘要中汇总答案
  - 局部查询：从相关实体/关系出发回答
  - 混合查询：结合两者

优势：能回答跨文档的全局性问题、能做多跳推理、答案更有结构
劣势：索引构建慢且贵（大量LLM调用）、增量更新困难、存储成本高

适用场景：法律文档分析、学术论文综述、企业知识库全局分析
