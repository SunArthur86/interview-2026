# 为什么结构化输出对Agent至关重要?如何确保100%的格式正确率

- **结构化输出的重要性:**

Agent的每一步都依赖结构化数据:
- 工具调用参数必须是合法JSON
- 任务状态必须是预定义枚举
- Agent间通信需要结构化消息

- **确保格式正确的方法:**

1. **Function Calling / Tool Use(最可靠)**
- 模型层面保证输出合法JSON
- OpenAI/Anthropic/Gemini都支持

2. **JSON Mode**
- response_format=json_object
- 保证合法JSON但不保证schema

3. **约束解码(Constrained Decoding)**
- 在推理时强制输出符合grammar的token
- llama.cpp的GBNF / Outlines / SGLang
- **100%保证格式正确**

- **约束解码原理图:**
```text
LLM Logits Output
       │
       ▼
┌──────────────┐
│  Masking     │  ──>  允许符合 JSON Schema 的 Token (如 {, ", : )
│  Engine      │  ──>  禁止不符合的 Token (如 random text)
└──────┬───────┘
       │
       ▼
Sampled Token
```

4. **Pydantic验证 + 重试**
```python
try: 
    result = MyModel.parse_raw(llm_output)
except ValidationError:
    # 把错误信息发回给LLM让其修正
    result = retry_with_error(llm_output, str(e))
```

- **实战案例:**：在开发数据库查询Agent时，遇到LLM输出的JSON中字段名带有转义字符（如 `\"table_name\"`），导致后端解析失败。引入 `lm-format-enforcer` 进行严格约束解码后，彻底消除了此类非法转义问题，拦截了约3%的错误请求。

- **代码示例 (Python - 使用Outlines库):**
```python
import outlines
import torch

model = outlines.models.transformers("HuggingFaceTB/SmolLM2-1.7B-Instruct")

# 定义JSON Schema强制约束
schema = json_schema({"type": "object", "properties": {"action": {"type": "string"}}})

generator = outlines.generate.json(model, schema)
structured_output = generator("执行数据库查询")
# structured_output 保证 100% 是 dict 类型，符合 schema
```

- **方法对比:**

| 方法 | 可靠性 | 性能影响 | 适用场景 |
| :--- | :--- | :--- | :--- |
| Function Calling | 高 | 无 | 通用、主流模型API调用 |
| JSON Mode | 中 | 无 | 简单JSON结构，非关键路径 |
| 约束解码 | 极高 (100%) | 轻微 (推理阶段) | 本地部署、强结构要求、高频调用 |
| Pydantic + 重试 | 低 (依赖概率) | 高 (多轮对话) | 低频、容错性高场景 |

- **实际建议:** 优先用Function Calling,其次约束解码,最后才是重试

## 常见考点
1. **如何处理嵌套复杂的 JSON Schema？**
   - 问点：当结构非常深时，模型经常出错怎么办？
   - 答案：降维打击，将复杂Schema拆分为多个简单的Function Call分步获取，再在代码层组装。
2. **什么是 Constrained Decoding 的性能损耗？**
   - 问点：它比普通推理慢吗？
   - 答案：需要构建前缀树或有限状态机(FSM)来过滤Logits，会有轻微的计算开销，但避免了重试带来的巨额Token消耗，总体更快。
3. **BNF Grammar 是什么？**
   - 问点：如何定义输出格式？
   - 答案：使用 GBNF (GBNF Grammar) 定义文法规则，例如 `root ::= "{" string ":" number "}"`。

## 记忆要点

- 结构化输出是工具调用基础，确保100%正确率首选约束解码(Constrained Decoding)。
- 方法对比：Function Calling最通用，JSON Mode保格式不保Schema，约束解码100%正确。
- 约束解码原理：推理时用Masking Engine过滤Logits，只允许符合Schema的Token。
- 复杂Schema处理：降维拆分，分步调用Function Call再组装，避免单次出错。

## 结构化回答

**30 秒电梯演讲：** 结构化输出是 Agent 调工具的基础——JSON 格式错了整个流程就崩。最可靠的是约束解码，在推理时直接用 Masking Engine 过滤 Logits，保证 100% 符合 Schema。通用方案是 Function Calling，JSON Mode 只保语法不保结构。

**展开框架：**
1. **方法对比** — Function Calling 最通用、JSON Mode 保格式不保 Schema、约束解码 100% 正确、Pydantic+重试是兜底。
2. **约束解码原理** — 推理时用 Masking Engine 过滤 Logits，只允许符合 Schema 的 Token，从根上杜绝格式错误。
3. **复杂 Schema 处理** — 降维拆分，分步调用 Function Call 再组装，避免单次出错；优先级是 Function Calling > 约束解码 > 重试。

**收尾：** 约束解码不是免费的——构建 FSM 有开销，但比重试烧 Token 划算，我可以聊聊 GBNF 文法怎么定义。
