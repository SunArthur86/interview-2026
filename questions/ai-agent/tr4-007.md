# Semantic Kernel 的 Planner（规划器）是如何实现复杂任务分解的？它和 LangChain 的 Agent 有何不同？

Semantic Kernel (SK) 的 Planner 利用 LLM 理解意图并将任务分解为可执行的函数链。SK 的核心特点是**代码优先**，Skills/Plugins 由 C# 或 Python 等强类型函数定义，Planner 基于函数语义描述和 Schema 构建执行计划。与 LangChain Agent 相比，SK 的优势在于**类型安全**和**企业级集成**：LangChain 侧重链式调用和 Prompt 编排，灵活性高但调试复杂；SK 则作为编排层，能无缝复用现有业务代码。此外，SK 的 Handlebars 模板支持复杂的变量传递和结构化计划生成，适合需要精确控制流程的企业级场景。

## 记忆要点

- SK Planner 代码优先，基于强类型函数 Schema 构建计划。
- LangChain 侧重 Prompt 编排，灵活但调试难；SK 侧重企业级集成。
- SK 优势：类型安全、复用现有业务代码、Handlebars 模板支持复杂变量。
- SK 适合精确控制流程的企业场景，LangChain 适合快速原型开发。
- 核心差异：SK 是函数编排层，LangChain 是链式调用框架。

## 结构化回答

**30 秒电梯演讲：** Semantic Kernel 的 Planner 是"代码优先"的任务编排器——用 LLM 把意图分解成强类型函数调用链，直接指挥 C#/Python 函数干活。和 LangChain 的区别是：SK 像总工程师强调类型安全和企业级业务代码复用，LangChain 像协调员靠 Prompt 编排，灵活但调试难。

**展开框架：**
1. **代码优先** — Skills/Plugins 由强类型函数定义，Planner 基于函数语义描述和 Schema 构建执行计划，原生接入现有代码库。
2. **对比 LangChain** — LangChain 侧重链式调用和 Prompt 编排，灵活适合快速原型；SK 侧重企业级集成和精确流程控制。
3. **核心优势** — 类型安全、无缝复用现有业务代码、Handlebars 模板支持复杂结构化变量传递和流程控制。

**收尾：** 我在企业项目里用 SK Planner 直接编排已有的 C# 业务函数库，类型安全让调试比 LangChain 的 Prompt 链轻松很多。您想聊 Handlebars 模板怎么写复杂变量传递，还是 SK 和 LangChain 在多 Agent 协作上的差异？
