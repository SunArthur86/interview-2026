# Semantic Kernel 的 Planner（规划器）是如何实现复杂任务分解的？它和 LangChain 的 Agent 有何不同？

Semantic Kernel (SK) 的 Planner 利用 LLM 理解意图并将任务分解为可执行的函数链。SK 的核心特点是**代码优先**，Skills/Plugins 由 C# 或 Python 等强类型函数定义，Planner 基于函数语义描述和 Schema 构建执行计划。与 LangChain Agent 相比，SK 的优势在于**类型安全**和**企业级集成**：LangChain 侧重链式调用和 Prompt 编排，灵活性高但调试复杂；SK 则作为编排层，能无缝复用现有业务代码。此外，SK 的 Handlebars 模板支持复杂的变量传递和结构化计划生成，适合需要精确控制流程的企业级场景。

## 技术原理

- **利用 LLM 将意图分解为函数调用链，强调原生代码集成**：Planner 接收用户意图和一组已注册的 Plugin 函数（含语义描述 + 参数 Schema），让 LLM 在函数集合上做规划，输出一个由原生函数调用组成的执行计划。函数本身是 C#/Python 的真实代码，不是 Prompt 模拟，执行结果直接走强类型返回值。
- **类型安全和企业级业务逻辑复用**：因为函数是强类型定义的，参数和返回值在编译期可校验，IDE 能智能提示和重构；企业里已有的业务代码库（订单/库存/权限等）可以直接包装成 Plugin 被 Planner 调用，而不是用 Prompt 重新描述一遍逻辑。
- **Handlebars 模板实现结构化变量传递与流程控制**：SK 用 Handlebars 描述计划的结构，支持变量插值、条件分支、循环，把 LLM 生成的计划固化为可序列化、可缓存、可人工审查的中间表示，而不是每次都让 LLM 即兴 ReAct。

## 对比/选型

| 维度 | Semantic Kernel Planner | LangChain Agent |
|------|--------------------------|------------------|
| 编排范式 | 代码优先，强类型函数调用链 | Prompt 优先，链式 ReAct/Tool calling |
| 类型安全 | 强（编译期校验参数/返回值） | 弱（Prompt 字符串约定） |
| 业务代码复用 | 直接包装现有 C#/Python 函数 | 需重写为 Tool 描述 |
| 调试体验 | 计划可序列化、可缓存、可审查 | Prompt 链黑盒，调试难 |
| 适合场景 | 企业级精确流程控制 | 快速原型、灵活探索 |
| 本质定位 | 函数编排层 | 链式调用框架 |

## 代码示例

SK（Python）定义强类型 Plugin 并交给 Planner：

```python
from semantic_kernel.functions import kernel_function

class SearchPlugin:
    @kernel_function(description="在知识库中检索相关文档", name="search")
    def search(self, query: str, top_k: int = 5) -> list[str]:
        return vector_db.query(query, k=top_k)        # 复用现有业务代码

class SummarizePlugin:
    @kernel_function(description="对文档列表生成摘要", name="summarize")
    def summarize(self, docs: list[str]) -> str:
        return llm.summarize("\n".join(docs))

# 注册后让 Planner 自动分解"检索并总结"
kernel.add_plugin(SearchPlugin(), "Search")
kernel.add_plugin(SummarizePlugin(), "Summarize")
plan = await planner.create_plan(kernel, "检索 RAG 一致性并总结")
# Planner 输出: Search.search -> Summarize.summarize，参数类型已校验
result = await kernel.invoke(plan)
```

Handlebars 计划模板（结构化变量传递）：

```handlebars
{{!-- 由 Planner 生成，可序列化缓存 --}}
{{Search.search query=kernel.userInput top_k=5}}
{{Summarize.summarize docs=Search.search.result}}
```

## 常见坑/注意事项

- **函数数量过多导致 Planner 幻觉**：注册几十个 Plugin 时 LLM 容易选错函数或编造参数。建议用函数过滤（Function Filtering）按意图动态缩小候选集，或分层规划（先选 Skill 再选函数）。
- **类型不匹配的运行时错误**：虽然强类型能在编译期校验，但 LLM 生成的计划可能传错值。生产环境仍需对参数做防御性校验和异常兜底。
- **与 LangChain 不是非此即彼**：SK 适合确定性高的企业流程，LangChain 适合探索性强的原型；复杂系统常两者混用——SK 做核心编排，LangChain 做灵活工具调用。

## 记忆要点

- SK Planner 代码优先，基于强类型函数 Schema 构建计划。
- LangChain 侧重 Prompt 编排，灵活但调试难；SK 侧重企业级集成。
- SK 优势：类型安全、复用现有业务代码、Handlebars 模板支持复杂变量。
- SK 适合精确控制流程的企业场景，LangChain 适合快速原型开发。
- 核心差异：SK 是函数编排层，LangChain 是链式调用框架。
