# Agent工具调用失败时的错误恢复策略有哪些?如何设计鲁棒的Agent

- **错误类型与恢复策略:**

| 错误类型 | 示例 | 恢复策略 |
|---------|------|---------|
| 网络超时 | API不响应 | 重试+指数退避 |
| 参数错误 | 传错参数 | LLM修正参数后重试 |
| 权限不足 | 403 | 换工具/降级方案 |
| 结果异常 | 返回空/格式错误 | 验证+重新调用 |
| 工具不存在 | 404 | 换替代工具 |
| 逻辑冲突 | 并发写冲突 | 重试/串行化 |

- **实战案例:**
在电商场景中，由于LLM幻觉经常传入错误的SKU ID导致库存查询接口报错400。我们在Prompt中加入"若接口报错400，请尝试提取商品名重新调用SearchProduct工具"的策略后，Agent具备了自愈能力，无需人工干预即可通过模糊搜索完成任务，任务成功率提升了30%。

- **代码示例 (Python):**
```python
import time

def call_tool_with_retry(tool, args, max_retries=3):
    for attempt in range(max_retries):
        try:
            result = tool.execute(**args)
            # 验证结果格式
            if not result.get("data"):
                raise ValueError("Empty result")
            return result
        except (ConnectionError, Timeout) as e:
            if attempt == max_retries - 1:
                return {"error": "Service unavailable after retries"}
            time.sleep(2 ** attempt) # 指数退避
        except ValueError as e:
            # 参数或结果错误，抛出给 LLM 修正
            return {"error": str(e), "fix_instruction": "Check SKU ID and try search"}
```

- **鲁棒Agent设计原则:**

1. **结构化错误处理**
```python
try: 
    result = tool.execute(params)
    if not validate(result):
        raise ToolError("Invalid result")
except RetryableError:
    retry_with_backoff()
except FatalError:
    fallback_strategy()
```

2. **让LLM自己处理错误**
- 将错误信息返回给LLM
- LLM决定:修正参数/换工具/放弃
- 这是Agent的核心优势

3. **断路器模式**
- 连续N次失败后停止调用该工具
- 避免浪费Token

4. **检查点与回滚**
- 每N步保存状态
- 失败后可回滚到检查点

- **错误处理策略对比:**

| 策略 | 触发条件 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **自动重试** | 网络抖动/5xx | 透明，速度快 | 无法修复逻辑错误，可能 avalanche |
| **LLM自愈** | 参数错误/4xx | 灵活，能理解语义 | 消耗Token，耗时较长 |
| **Fallback** | 核心服务挂断 | 保证最终可用性 | 降级体验，结果可能不精确 |
| **人工介入** | 账单/支付风险 | 100%准确 | 破坏自动化体验 |

## 结构化回答

**30 秒电梯演讲：** 鲁棒 Agent 的核心是把错误分类处理：网络错误重试退避，参数错误让 LLM 自愈，服务挂了走 Fallback。设计关键是结构化错误处理——把错误信息转成 LLM 能理解的指令，再加上断路器和检查点兜底。

**展开框架：**
1. **错误分类与策略** — 网络超时重试退避、参数错误让 LLM 自愈、权限不足换工具降级、结果异常验证后重调。
2. **核心是 LLM 自愈** — 将错误信息返回给 LLM，由它决定修正参数、换工具还是放弃，这是 Agent 区别于普通程序的优势。
3. **工程兜底** — 断路器模式连续失败就停防 Token 浪费；关键操作设检查点支持失败回滚。

**收尾：** 错误处理做得好，Agent 才算能上生产——我可以讲个 SKU ID 幻觉如何靠自愈策略提升 30% 成功率的例子。
