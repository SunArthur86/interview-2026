# Seata AT 模式在电商下单场景的使用

以下描述 Seata AT 模式在电商下单（扣减库存 + 生成订单）场景下的工作原理及细节。

### 业务背景
协调者 `shopping-service` 调用参与者 `repo-service`（扣库存）和 `order-service`（生成订单）。

### 全局事务流程
1. **注册全局事务**：`shopping-service` 向 Seata 注册，生成 XID。
2. **分支事务执行（一阶段）**：
   - `repo-service` 和 `order-service` 分别执行查询与写操作，并生成 Undo Log（记录 `t_repo` 和 `t_order` 的前后镜像）。
   - 将业务数据和 Undo Log 在同一个本地事务中提交。
   - 向 TC 注册分支事务并汇报提交状态。
3. **决议（二阶段）**：
   - TC 汇总所有分支状态，决定全局提交或回滚。
   - 通知各参与者执行最终操作。

### 关键机制解析

#### 1. 提交流程
- 由于业务操作已在**一阶段提交**，二阶段的“提交”操作只需**异步删除 Undo Log**，非常高效。

#### 2. 回滚流程与防脏写
- **问题**：一阶段提交后到二阶段回滚前，数据可能被其他事务修改（脏写）。
- **解决方案**：
  - RM 拿到 Undo Log 后，利用 **After Image**（修改后数据）与当前数据库数据进行比对。
  - **若数据一致**：说明无脏写，执行补偿 SQL（基于 Before Image）回滚数据。
  - **若数据不一致**：说明有脏写，需人工介入或采取特定策略（报错重试），防止数据覆盖。

#### 3. 写隔离与全局锁
在电商高并发场景下，写隔离至关重要。Seata AT 模式通过 **Global Lock** 实现：
- **一阶段本地提交前**：RM 必须向 TC 申请该记录的全局锁。
- **竞争场景**：如果两个全局事务同时修改同一行库存：
  - 事务 A 先拿到本地锁，申请全局锁成功，提交，释放本地锁。
  - 事务 B 拿到本地锁，申请全局锁失败（被 A 锁住），事务 B 会尝试重试获取全局锁。
  - 此时事务 A 全局回滚：事务 A 重新获取本地锁，回滚数据，释放全局锁。
  - 事务 B 拿到全局锁，继续提交。

### 实战案例
在双11大促期间，库存扣减接口频繁报“全局锁获取超时”。经排查，是因为下游物流服务响应缓慢，导致全局事务持有全局锁的时间过长，阻塞了后续的库存更新请求。优化方案是将非核心的物流调用异步化，并调大了 `global.lock.retry.interval` 参数以适应网络抖动。

### 核心代码示例
```java
// 模拟 Seata 全局锁获取过程 (伪代码)
public boolean acquireGlobalLock(String xid, String pk) {
    // 向 TC 发送请求
    while (true) {
        boolean locked = tc.lockQuery(xid, pk);
        if (locked) return true;
        // 配置重试间隔，默认 10ms
        Thread.sleep(GlobalLockConfig.getRetryInterval()); 
        // 超时抛出异常，回滚本地事务
        if (System.currentTimeMillis() > startTime + timeout) {
            throw new GlobalLockException("获取全局锁超时");
        }
    }
}
```

### 常见考点
1. **全局锁对性能的影响？**：全局锁机制会导致并发写串行化。在热点数据（如秒杀库存）场景下，全局锁竞争会导致大量重试，甚至死锁。因此 Seata AT 不适合高并发的单行热点更新。
2. **Undo Log 什么时候删除？**：只有在全局事务决议为 Commit 时，才会异步删除。如果决议为 Rollback，则在回滚完成后删除。如果一直不删除（例如二阶段没收到指令），会有定时任务清理。
3. **TC 如何感知 RM 是成功还是失败？**：RM 在本地事务提交成功后，会显式发送消息给 TC；如果本地事务提交失败（如 DB 挂了），RM 不会发送成功消息，也不会注册分支，TC 自然会决议回滚。

## 记忆要点

- 一阶段：各微服务执行SQL生成Undo Log，在提交本地事务前必须获取该行的全局锁。
- 二阶段提交：业务数据已落库，只需异步删除Undo Log，非常高效。
- 二阶段回滚防脏写：比对After Image与当前DB，若一致则用前镜像回滚，若不一致则报错人工介入。
- 性能避坑：全局锁会导致并发写串行化，不适合秒杀等单行热点高并发场景。
