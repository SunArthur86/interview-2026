# 分布式ID有哪些生成方案？

1. **UUID**：
- **优点**：简单、本地生成、无中心化、性能高。
- **缺点**：无序（36字符长），作为MySQL主键会导致B+树频繁页分裂，插入性能差；无业务含义，不可排序。

2. **雪花算法**：
- **结构**：64位long型ID = 1位符号(0) + 41位时间戳(毫秒级) + 10位机器ID(5位数据中心+5位工作机器) + 12位序列号(毫秒内自增)。
- **优点**：趋势递增（依赖时间），性能高（毫秒级生成409万个ID），不依赖第三方组件。
- **问题**：**时钟回拨**。服务器时钟回拨可能导致ID重复或服务拒绝。
- **解决**：美团Leaf-segment引入ZooKeeper记录时间；百度UidGenerator利用RingBuffer预生成ID缓冲。

3. **号段模式（美团Leaf-segment）**：
- **原理**：数据库存储当前最大ID（max_id）和步长。业务服务一次性获取一段ID（如[1000, 2000]），加载到内存。
- **双Buffer优化**：当前号段用了一半时，异步去DB取下一个号段，保证几乎无IO等待。
- **优点**：性能极高（内存生成），ID趋势递增。
- **问题**：服务重启可能导致ID不连续；强依赖DB。

4. **Redis自增**：
- 利用 `INCR` 命令，原子性自增。
- **缺点**：维护成本高，Redis集群数据同步和持久化配置麻烦，强依赖Redis可用性。

5. **数据库自增**：
- 利用 `auto_increment`。
- **缺点**：性能瓶颈明显，分库分表后ID冲突难以处理（需配置不同步长或起始位）。

**实战案例**：
某电商大促期间，因容器化部署机器IP不固定导致Snowflake的WorkerID冲突，引发ID重复风险。后改为通过ZooKeeper临时顺序节点自动注册WorkerID解决。

**代码示例（Java雪花算法核心）**：
```java
public synchronized long nextId() {
    long timestamp = timeGen();
    if (timestamp < lastTimestamp) {
        throw new RuntimeException("时钟回拨");
    }
    if (lastTimestamp == timestamp) {
        sequence = (sequence + 1) & sequenceMask;
        if (sequence == 0) timestamp = tilNextMillis(lastTimestamp);
    } else {
        sequence = 0;
    }
    lastTimestamp = timestamp;
    return ((timestamp - twepoch) << timestampShift) | (workerId << workerIdShift) | sequence;
}
```

**方案对比**：
| 方案 | 性能 | 有序性 | 复杂度 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **UUID** | 极高 | 无序 | 低 | 非DB主键、token生成 |
| **Snowflake** | 高 | 趋势递增 | 中 | 分布式主键、分片键 |
| **号段模式** | 极高 | 严格递增 | 中高 | 高并发业务ID |
| **Redis** | 高 | 严格递增 | 中 | 计数器、简单序列 |

**推荐**：
- 中小规模、对连续性要求不敏感：雪花算法。
- 大规模、高并发、追求性能：号段模式（如美团Leaf-segment）。

## 常见考点
1. **雪花算法时钟回拨处理**：通常策略是直接报错等待；或者利用“上次最大时间”作为基准，回拨较小时间时使用上次时间（序列号增大）。
2. **雪花算法的机器ID分配**：10位机器ID支持1024个节点。可以通过配置文件指定，或者利用ZooKeeper的顺序节点自动注册分配。
3. **为什么Kafka的ID是有序的**？Kafka早期使用ZK生成，性能差；后期使用Snowflake算法，保证了Partition内的有序性和单调性。
