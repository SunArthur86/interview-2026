# 异步确保型事务使用示例

# 异步确保型事务使用示例

## 核心流程
1.  **发送半消息**：事务发起方首先发送半消息到 MQ（Half Message）；
2.  **ACK 确认**：MQ 通知发送方消息发送成功；
3.  **执行本地事务**：在发送半消息成功后执行本地事务；
4.  **提交/回滚**：根据本地事务执行结果返回 Commit 或 Rollback；
5.  **MQ 路由决策**：
    *   如果是 Rollback：MQ 将丢弃该消息不投递；
    *   如果是 Commit：MQ 将消息发送给消息订阅方；
6.  **订阅方消费**：订阅方根据消息执行本地事务；
7.  **消费确认**：订阅方执行本地事务成功后再从 MQ 中将该消息标记为已消费；
8.  **异常处理（反查机制）**：如果执行本地事务过程中，执行端挂掉或超时，MQ 服务器端将不停地询问 Producer 来获取事务状态；
9.  **一致性保证**：Consumer 端的消费成功机制由 MQ 保证（通常结合幂等性设计）。

## 业务实战示例

假设存在业务规则：**某笔订单成功后，为用户加一定的积分**。
在这条规则里，管理订单数据源的服务为事务发起方，管理积分数据源的服务为事务跟随者。

### 架构流程图
```text
    订单服务 (发起方)              RocketMQ              积分服务 (跟随者)
       |                               |                          |
       |------- 1. 发送半消息 ---------->|                          |
       |<------ 2. ACK (发送成功) ------|                          |
       |                               |                          |
       | 3. 执行本地事务 (创建订单)      |                          |
       |                               |                          |
       |------- 4. 提交/回滚 ---------->|                          |
       |       (Commit)                |                          |
       |                               |--- 5. 投递消息 --------->|
       |                               |                          |
       |         [异常情况]             |                          |
       |<------ 8. 反查事务状态 --------|                          |
       |------- 8. 返回最终状态 ------->|                          |
       |                               |                          |
                                       |       6. 执行加积分       |
                                       |       7. ACK 消费成功     |
```

### 方案分析
从这个过程可以看到，基于消息队列实现的事务存在以下操作：
1.  订单服务创建订单，提交本地事务；
2.  订单服务发布一条消息（半消息机制保证原子性）；
3.  积分服务收到消息后加积分。

**优点**：
*   **过程简单，性能消耗小**：相比 TCC 或 2PC，不需要锁资源，吞吐量高。
*   **削峰填谷**：发起方与跟随者之间的流量峰值可以使用队列填平。
*   **开发成本低**：业务开发工作量基本与单机事务没有差别，不需要编写反向的业务回滚逻辑（Cancel 阶段）。

因此，基于消息队列实现的事务是我们除了单机事务外最优先考虑使用的形态。

### 代码示例 (Java Spring Boot + RocketMQ)

```java
// 发送方：构建并发送事务消息
@Service
public class OrderService {
    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    public void createOrder(OrderDTO order) {
        // 构建消息，将订单ID作为业务Key
        Message<OrderDTO> msg = MessageBuilder.withPayload(order).setHeader("orderId", order.getId()).build();
        // 发送事务消息，第一个参数是Topic，第二个参数是消息，第三个参数是自定义参数
        rocketMQTemplate.sendMessageInTransaction("order-topic", msg, null);
    }
}

// 消费方：监听消息并处理，必须保证幂等性
@RocketMQMessageListener(topic = "order-topic", consumerGroup = "points-group")
@Service
public class PointsConsumer implements RocketMQListener<OrderDTO> {
    @Autowired
    private PointsService pointsService;

    @Override
    public void onMessage(OrderDTO message) {
        // 幂等性校验：先查是否已加积分
        if (pointsService.hasAdded(message.getId())) {
            return;
        }
        // 执行加积分逻辑
        pointsService.addPoints(message.getUserId(), message.getPoints());
    }
}
```

### 常见考点与踩坑
1.  **消息丢失怎么办**？
    *   虽然半消息机制保证了发送方和 MQ 的一致性，但如果 Consumer 消费失败且未重新投递，会造成数据不一致。需在 Consumer 端配合 **幂等性设计** 和 **重试机制**。
2.  **实战踩坑**：
    *   **积压问题**：如果积分服务挂掉时间过长，消息积压在 MQ 中，恢复后可能瞬间打崩积分数据库。实战中需要 Consumer 端具备 **限流降级** 能力，或者在 Broker 端配置合适的 `maxReconsumeTimes`。
    *   **乱序问题**：如果业务对顺序敏感（如先扣减后增加），需确保消息进入同一个队列，且 Consumer 单线程消费或有序消费。

### MQ 事务消息方案

MQ 事务消息方案是异步确保型事务的主流实现方式。旨在将同步事务转为异步执行，解决分布式事务的性能瓶颈。

#### 核心机制：半消息
该方案借鉴了 2PC 的思想，通过**半消息**来协调业务事务与消息事务。

**执行流程**：
1.  **发送半消息**：生产者向 MQ 发送一条“半消息”。该消息已被 MQ 存储，但此时对消费者不可见（不能被消费）。
2.  **执行本地事务**：生产者执行本地业务逻辑（如数据库操作）。
3.  **提交/回滚**：
    *   如果本地事务执行成功，向 MQ 发送 Commit 请求，MQ 将消息标记为可投递，消费者进行消费；
    *   如果本地事务执行失败，向 MQ 发送 Rollback 请求，MQ 丢弃该半消息。
4.  **异常处理（回查）**：如果 MQ 在一定时间内未收到生产者的确认（Commit/Rollback），会主动回查生产者的事务状态（通过回调接口），以确保消息的最终一致性。

#### RocketMQ 实现原理细节
RocketMQ 通过 `TransactionalMessageCheckService` 线程定时处理 `Half Topic` 中的消息。

```text
+-------------------+
| RocketMQ Broker   |
+-------------------+
|   Topic: RMQ_SYS_TRANS_HALF_TOPIC  (存放半消息)         |
|   Topic: RMQ_SYS_TRANS_OP_HALF_TOPIC (存放操作结果)    |
+-------------------+
        ^  |
        |  | 1. 写半消息
        |  | 2. 写 Commit/Rollback Op 消息
        |  | 3. 回查时读取 Half Topic
   Producer | 4. 根据 Op Topic 决定是否再次回查
```

*   **Half Topic**：所有半消息实际存储在这里，对消费者不可见。
*   **Op Topic**：记录二阶段的结果（Commit 或 Rollback）。
*   **回查逻辑**：定时任务扫描 Half Topic，如果消息超过一定时间未收到对应的 Op 消息，则判定为状态不确定，发起回查。

#### 常见考点
1.  **消息丢失风险**：在步骤 2（执行本地事务）成功但步骤 3（发送 Commit）网络断开时，依靠回查机制是否一定能保证不丢数据？（是的，只要本地事务成功，回查接口返回 Commit 即可）。
2.  **并发回查压力**：如果大量事务消息都超时，Broker 的回查请求是否会压垮生产者？（通常有最小回查间隔限制）。
3.  **与本地消息表方案的选型**：如果不想依赖特定的 MQ 中间件（如 RabbitMQ），是否还能用事务消息方案？（不能，通常退而求其次选择本地消息表方案）。

#### 实战案例
某支付系统在重构时，发现由于数据库主从延迟导致事务消息回查查询到旧数据（未提交状态），从而错误回滚了资金交易。解决方案是在 `checkLocalTransaction` 中强制读主库或增加短暂的等待重试逻辑。

#### 对比表格：MQ 事务消息 vs 本地消息表

| 特性 | MQ 事务消息 | 本地消息表 |
| :--- | :--- | :--- |
| **实现复杂度** | 低（依赖 MQ 原生支持，如 RocketMQ） | 高（需自己开发定时任务扫表） |
| **侵入性** | 中（需引入特定 MQ 客户端） | 低（仅依赖 DB 和通用 MQ） |
| **一致性** | 最终一致性 | 最终一致性 |
| **存储依赖** | 依赖 MQ 磁盘 | 依赖业务数据库磁盘 |
| **适用场景** | 高并发、对吞吐量要求高的核心链路 | 异步化要求高、技术栈受限（如用 RabbitMQ/Kafka） |
