# 微服务限流方案有哪些？令牌桶、漏桶、滑动窗口如何选择？

🎯 本质：限流是保护系统在高并发下不被打垮的核心手段，通过限制单位时间内的请求量来保证系统稳定性。

**主流算法对比：**

| 算法 | 核心特性 | 适用场景 | 突发流量处理 |
|------|---------|---------|------------|
| **固定窗口** | 实现简单，内存占用极低 | 简单的API防护 | 差（边界突刺） |
| **滑动窗口** | 流量平滑，精度高 | 高并发精确限流 | 中等 |
| **漏桶** | 强制恒定速率输出 | 保护脆弱的数据库/下游 | 拒绝突发 |
| **令牌桶** | 允许一定程度的突发 | 网关入口限流、秒杀预热 | 好（消耗存量） |

1. **固定窗口计数器**
   - **原理**：每个时间窗口内计数，超过阈值拒绝。
   - **缺点**：临界点问题（窗口边界突发流量翻倍），例如限流100/min，00:59来100个，01:01来100个，瞬间系统承受200个请求。

2. **滑动窗口计数器**
   - **原理**：将窗口分为多个小格子，按格子滑动计数。
   - **优点**：解决了临界点问题，流量控制更平滑。
   - **实现细节**：时间轮算法或环形数组，随着时间流逝移除过期格子。

3. **漏桶(Leaky Bucket)**
   - **原理**：请求像水滴一样滴入桶中，桶以固定速率漏水（消费请求）。如果桶满了，溢出的请求被拒绝。
   - **特点**：输出速率恒定，强制平滑流量。
   - **适用**：保护下游系统（如数据库、第三方API），防止突发流量打垮消费者。

4. **令牌桶(Token Bucket)**
   - **原理**：以固定速率往桶里放令牌，请求消耗令牌。如果桶里没令牌，请求被拒绝。
   - **特点**：允许突发流量（只要桶里有存量令牌，可以一次性拿取，直到上限）。
   - **适用**：API网关限流（允许短时突发，同时保证平均速率）。

**实战案例**：在抢购业务中，为了保证公平性，我们采用了"漏桶"算法均匀透传流量到后端，防止恶意用户在1秒内脚本刷完所有库存；而在普通API网关层，使用"令牌桶"允许短暂的高并发访问以应对正常业务波峰。

**Redis + Lua实现令牌桶（增强版）：**
```lua
local key = KEYS[1]
local rate = tonumber(ARGV[1])      -- 生成速率（令牌/秒）
local capacity = tonumber(ARGV[2])  -- 桶容量
local now = tonumber(ARGV[3])       -- 当前时间戳（秒）
local requested = tonumber(ARGV[4] or 1) -- 本次请求令牌数

-- 获取当前令牌数和上次注入时间
local tokens = tonumber(redis.call('get', key..':tokens') or capacity)
local last_time = tonumber(redis.call('get', key..':time') or now)

-- 计算 delta 并补充令牌
local delta = math.max(0, now - last_time)
local filled_tokens = math.min(capacity, tokens + delta * rate)

-- 判断是否足够
if filled_tokens >= requested then
    local new_tokens = filled_tokens - requested
    redis.call('set', key..':tokens', new_tokens)
    redis.call('set', key..':time', now)
    return 1 -- 允许
else
    redis.call('set', key..':tokens', filled_tokens)
    redis.call('set', key..':time', now)
    return 0 -- 拒绝
end
```

**限流层级：**
1. **网关层**：Nginx（漏桶 limit_req），限制总入口流量。
2. **应用层**：Sentinel/Resilience4j（滑动窗口），针对具体接口或服务粒度限流。
3. **中间件层**：Redis Lua（令牌桶），分布式环境下精确限流。

**选择建议：**
- 需要允许突发流量（如秒杀场景预热）→ **令牌桶**
- 需要严格平滑输出（保护脆弱DB）→ **漏桶**
- 需要高性能内存限流（单机）→ **滑动窗口**

## 常见考点
1. **分布式限流如何保证原子性？**
   答：必须使用 Redis + Lua 脚本，将读取、计算、回写打包成一个原子操作，避免并发带来的竞态条件（如超卖）。
2. **预热如何实现？**
   答：令牌桶的初始 tokens 设为 0，或者在启动时采用“冷启动”策略（如 Guava RateLimiter 的 smoothWarmingUp），缓慢增加生成速率直到稳定。
3. **限流阈值如何设定？**
   答：通过压测确定系统最大容量（QPS），然后设置阈值 = 最大容量 * 80%（留余量），并结合线上监控动态调整。
