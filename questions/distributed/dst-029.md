# 柔性事务的分类

### 柔性事务的分类

在电商等高并发互联网场景下，刚性事务（如 2PC/XA）在性能和处理能力上暴露出瓶颈。柔性事务应运而生。

#### 柔性事务的特性
1.  **基本可用**：分布式系统出现故障时，允许损失一部分可用性；
2.  **柔性状态**：允许系统存在中间状态，且该中间状态不影响系统整体可用性（如主从同步延迟）。

柔性事务追求的是**最终一致性**，而非强一致性。

#### 分类
柔性事务主要分为两类：

1.  **补偿型**（同步）：
    *   **TCC**（Try-Confirm-Cancel）：
        *   **Try**：资源预留和业务检查。
        *   **Confirm**：确认执行业务，使用 Try 阶段预留的资源。
        *   **Cancel**：取消执行业务，释放 Try 阶段预留的资源。
        *   *特点*：侵入性强，需为每个操作编写三个接口，性能较好，无长期锁。
    *   **Saga**：
        *   将长事务拆分为多个本地短事务。
        *   每个短事务都有对应的补偿事务。
        *   若某步失败，按相反顺序执行补偿事务。
        *   *特点*：不要求资源预留，适合业务流程长的场景。

2.  **通知型**（异步）：
    *   **异步确保型事务**（基于本地消息表）：利用消息队列确保事务最终一致，上游执行成功后发送消息，下游监听消费。
    *   **最大努力通知事务**：上游尽力通知下游，但不保证下游一定能成功，通常配合下游查询接口。

#### 柔性事务对比图

```text
+----------------+------------------------+-----------------------+-------------------+
|     类型       |          代表方案       |        一致性         |    适用场景       |
+----------------+------------------------+-----------------------+-------------------+
| 补偿型 (同步)  | TCC, Saga              | 最终一致性 (实时性高) | 高并发、核心链路 |
+----------------+------------------------+-----------------------+-------------------+
| 通知型 (异步)  | 本地消息表, 事务消息   | 最终一致性 (有延迟)   | 跨系统、非核心链路|
+----------------+------------------------+-----------------------+-------------------+
```

#### 💡 实战案例：TCC 空回滚防挂
在双十一大促下单场景中，由于网络超时导致 Try 阶段请求超时，事务管理器触发了 Cancel，但随后 Try 请求才到达。若未处理，会导致资源预留后直接被释放，业务后续无法 Confirm，产生“悬挂”数据。解决方案是在 Cancel 阶段先记录事务标识，若后续 Try 到来检测到已 Cancel，则直接拒绝执行。

#### 💻 代码示例：TCC 防悬挂处理（Java）
```java
public void try(BusinessActionContext context, String userId) {
    String txId = context.getXid();
    // 1. 检查是否已取消（防悬挂）
    if (transactionLogDao.isCancelled(txId)) {
        throw new BusinessException("Transaction悬挂，拒绝执行");
    }
    // 2. 记录Try日志，插入幂等控制
    transactionLogDao.insertLog(txId, status="TRYING");
    // 3. 冻结资源（如扣减库存冻结层）
    inventoryDao.freeze(userId, amount);
}

public void cancel(BusinessActionContext context) {
    String txId = context.getXid();
    // 1. 先插入Cancel记录（防止悬挂）
    transactionLogDao.insertCancelLog(txId);
    // 2. 如果Try已执行，则释放资源；否则为空回滚
    if (transactionLogDao.isTried(txId)) {
        inventoryDao.unfreeze(userId, amount);
    }
}
```

#### 📊 选型对比：TCC vs Saga

| 维度 | TCC (Try-Confirm-Cancel) | Saga (长事务拆分) |
| :--- | :--- | :--- |
| **一致性** | 强（两阶段提交保证） | 较弱（中间状态可见，需处理脏写） |
| **并发性能** | 高（资源锁定粒度可控） | 中（需处理子事务间并发冲突） |
| **开发成本** | 高（需写 Try/Confirm/Cancel 三个接口） | 中（正向+反向逻辑） |
| **适用场景** | 一致性要求高、资产类核心业务 | 业务流程长、非核心或允许中间状态可见 |
| **锁机制** | Try 阶段预留资源 | 无预留，补偿时回滚修改 |

## 常见考点
1.  **TCC 的空回滚与悬挂**：如何解决 Try 阶段未执行，Cancel 却先到的“空回滚”问题？以及 Cancel 比 Try 晚到的“悬挂”问题？
2.  **Saga 的并发问题**：在 Saga 模式中，如果多个子事务并发执行且产生了脏写，补偿机制如何处理？（通常引入锁或版本号）。
3.  **本地消息表的定时轮询**：本地消息表是如何通过定时任务轮询发送失败的消息的？轮询频率如何设计？
