# 微服务熔断器(Circuit Breaker)的工作原理是什么？有哪些状态？

🎯 本质：熔断器是一种保护机制，当下游服务故障率超过阈值时自动断路，快速失败而不是等待超时，防止级联故障（雪崩效应）。

**三状态模型及转换：**

```text
      失败率/慢调用比例超过阈值
+---------------------------------------+
|                                       v
+-----------+         失败        +------------+
|  CLOSED   | <---------------- |   OPEN     |
|  (关闭)   | (半开状态探测失败)|  (熔断)    |
+-----------+                   +------------+
    ^    |                           |   
    |    | 探测成功                   | 等待时间结束
    |    +------------------------->+
    |             (半开放行一个请求)
+-----------+
| HALF_OPEN |
|  (半开)   |
+-----------+
```

**详细状态说明：**

1. **CLOSED（关闭）- 正常状态**
   - 初始状态，所有请求正常通过。
   - **滑动窗口**内统计失败率或慢调用比例。
   - 触发条件：失败率 >= `failureRateThreshold` 且 请求数 >= `minimumNumberOfCalls`。

2. **OPEN（打开）- 熔断状态**
   - 熔断器打开，所有请求直接抛出 `CallNotPermittedException`，不发起实际远程调用。
   - 持续时间：`waitDurationInOpenState`（如 30s）。此期间是为了给下游服务恢复的时间。

3. **HALF_OPEN（半开）- 试探状态**
   - OPEN 状态持续时间结束后进入。
   - **放行少量请求**（如 1 个或 N 个），作为“探针”检测下游是否恢复。
   - 判定结果：
     - 成功：重置计数器，切换回 CLOSED，恢复流量。
     - 失败：立即切回 OPEN，重新计时。

**关键参数配置（Resilience4j 为例）：**
- `failureRateThreshold`: 失败率阈值，默认 50%。
- `slowCallRateThreshold`: 慢调用比例阈值（如超过 500ms 视为慢调用）。
- `waitDurationInOpenState`: 熔断器打开后的等待时间。
- `slidingWindowType`: 计数器类型（COUNT_BASED 基于请求数 / TIME_BASED 基于时间）。
- `permittedNumberOfCallsInHalfOpenState`: 半开状态允许的探测请求数。

**降级与熔断的区别：**
- **熔断**：是主动触发，停止调用下游，直接报错或走降级逻辑。
- **降级**：是兜底方案，当服务不可用或熔断触发时，返回默认值、缓存或空响应。

### 💡 实战深化

**1. 实战案例**
在大促压测中，某核心接口调用下游积分服务超时，由于未配置熔断，导致 Tomcat 线程池全部阻塞在等待积分服务响应上，最终拖垮整个网关。接入熔断器后，积分服务故障时网关直接返回“积分处理中”，成功保住了主交易链路的可用性。

**2. 代码示例（Resilience4j）**
```java
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)                    // 失败率50%触发
    .waitDurationInOpenState(Duration.ofSeconds(30)) // 熔断30秒
    .permittedNumberOfCallsInHalfOpenState(3)    // 半开放行3个探测
    .slidingWindowSize(10)                       // 滑动窗口10个请求
    .build();

CircuitBreakerRegistry registry = CircuitBreakerRegistry.of(config);
CircuitBreaker cb = registry.circuitBreaker("inventoryService");

// 装饰远程调用
Supplier<String> supplier = CircuitBreaker.decorateSupplier(cb, () -> inventoryService.deductStock());
String result = supplier.get();
```

**3. 对比表格：熔断 vs 限流**

| 维度 | 熔断 | 限流 |
| :--- | :--- | :--- |
| **核心目标** | 防止故障扩散（保护调用方） | 防止系统过载（保护被调用方） |
| **触发条件** | 下游服务异常/慢调用比例高 | 上游请求并发数/QPS 超过阈值 |
| **处理策略** | 直接拒绝，一段时间内不再调用 | 拒掉多余请求，放过阈值内请求 |
| **典型场景** | 依赖不稳定的第三方API | 秒杀抢购、防止数据库被打挂 |

## 常见考点
1. **熔断和限流的区别是什么？**
   答：限流是为了防止系统过载（流量太大）；熔断是为了防止故障扩散（下游挂了）。限流是拒掉多余流量，熔断是停止对故障服务的调用。
2. ** HALF_OPEN 状态如果允许大量请求通过会怎样？**
   答：如果下游服务处于“半死不活”状态（如DB连接池耗尽），大量请求涌入可能导致下游直接崩溃，熔断器永远不会恢复。所以 HALF_OPEN 通常只放行 1 个请求。
3. **如何避免熔断器的“震荡”？**
   答：合理配置 `waitDurationInOpenState`，避免过快重试；确保 HALF_OPEN 的探测请求具有代表性；记录打点日志分析失败原因。
