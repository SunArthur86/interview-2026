# XA 模式的使用

### Seata XA 模式的使用

#### 1. 编程模型
*   从编程模型上看，XA 模式与 AT 模式保持**完全一致**。
*   开发者只需使用标准的 Spring 数据源配置和 `@GlobalTransactional` 注解。

#### 2. 配置差异（数据源代理）
要在 XA 模式和 AT 模式之间切换，仅需修改数据源 Bean 的配置：

```java
@Bean("dataSource")
public DataSource dataSource(DruidDataSource druidDataSource) {
    // AT 模式使用 DataSourceProxy
    // return new DataSourceProxy(druidDataSource);

    // XA 模式使用 DataSourceProxyXA
    return new DataSourceProxyXA(druidDataSource);
}
```

#### 3. 典型场景
适用于对数据一致性要求极高、业务逻辑相对简单、且使用支持 XA 协议的数据库（如 MySQL、Oracle）的场景。

#### 4. 优化方向（延迟注册）
目前的实现是在 XA start 前注册分支。未来的优化方向是类似 AT 模式：
*   **延迟分支注册**：尽量延后到本地事务提交前再注册，避免分支执行失败时产生无意义的注册开销。
*   这需要配合 BranchId 生成机制的调整。

#### 5. 与 AT 模式的切换细节
*   **数据库要求**：XA 模式要求数据库必须支持 XA 协议（绝大多数关系型数据库都支持，且性能依赖驱动实现）。
*   **隔离级别**：XA 模式依赖数据库自身的隔离级别实现（通常为 Read Committed 或 Repeatable Read），而 AT 模式利用 Seata 的全局锁机制实现了“读未提交”的隔离级别（默认情况，可配置为读已提交）。

#### 6. 实战案例
**坑点**：在 MySQL 5.7 及以下版本使用 XA 时，如果长事务持有数据库锁时间过长，极易导致 `Undo Log` 空间膨胀或连接池耗尽，生产环境务必配合连接池的 `testWhileIdle` 参数使用，防止连接僵死。

#### 7. XA 与 AT 模式深度对比

| 特性 | Seata XA 模式 | Seata AT 模式 |
| :--- | :--- | :--- |
| **一致性** | 强一致性（2PC） | 最终一致性（默认读未提交） |
| **锁机制** | 数据库级别锁，持有时间长 | Seata 全局锁，持有时间短（本地提交释放） |
| **性能** | 低（锁资源严重，阻塞性强） | 高（无锁读，写锁快照） |
| **代码侵入** | 低（仅需替换数据源） | 低（仅需替换数据源） |
| **回滚机制** | 依赖数据库驱动 Rollback | 自动解析 SQL 生成 Undo Log 反向执行 |
| **适用场景** | 短事务、资金类核心业务 | 高并发电商业务、库存扣减 |

## 常见考点
1.  **如何无缝从 AT 切换到 XA？** 只需修改数据源代理类，业务代码无需改动，体现了 Seata 良好的扩展性设计。
2.  **XA 模式是否需要 Undo Log？** 不需要。XA 模式依赖数据库自身的 Redo/Undo 机制进行回滚，不涉及 Seata 的自动回滚解析。
3.  **一阶段失败如何处理？** 如果在 XA End 之前发生业务异常，直接执行 XA Rollback；如果在 XA Prepare 之后全局事务回滚，TC 会通知 RM 执行 XA Rollback。

## 记忆要点

- 开发无缝切换：编程模型与 AT 模式完全一致，仅需把数据源代理换为 DataSourceProxyXA。
- 回滚机制：不需要 Undo Log，直接依赖数据库底层自身的 Redo/Undo 机制进行回滚。
- 隔离级别：XA 依赖 DB 自身隔离级别；而 AT 默认依赖全局锁实现读已提交。
- 模式对比：XA 强一致性但持锁长性能低；AT 最终一致但无锁读性能高。
