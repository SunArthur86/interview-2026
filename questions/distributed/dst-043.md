# TCC的工作流程

TCC（Try-Confirm-Cancel）是一种应用层面的分布式事务解决方案，旨在解决复杂业务中跨服务、跨库的资源协调问题。其核心思想是将业务逻辑拆分为三个阶段：

**1. Try 阶段（预留资源）**
- 完成所有业务检查（如：账户余额是否充足、库存是否足够）。
- 预留必须的业务资源（如：冻结余额、扣减预占库存）。
- **原理**：此阶段主要是“检查”和“预留”，并未真正提交底层数据库事务，因此不能长时间持有数据库锁，需快速执行。

**2. Confirm 阶段（确认执行）**
- 真正执行业务逻辑，使用 Try 阶段预留的资源（如：将冻结余额扣除、扣减真实库存）。
- 不做任何业务检查，只要 Try 成功，Confirm 必须能成功（空回滚 prevention）。
- **约束**：必须满足幂等性，防止因网络重试导致重复扣款。

**3. Cancel 阶段（取消回滚）**
- 释放 Try 阶段预留的业务资源（如：解冻余额、回滚预占库存）。
- **边界情况**：如果 Try 阶段未执行（由于网络抖动导致超时），也需要 Cancel 能正确处理（即“空回滚”处理）。
- **约束**：必须满足幂等性。

### 异常场景处理：空回滚与悬挂

| 异常场景 | 描述 | 解决方案 |
| :--- | :--- | :--- |
| **空回滚** | Try 阶段未执行或失败，但由于网络超时，事务管理器误认为 Try 成功，触发了 Cancel。 | Cancel 接口执行前，先检查事务日志中是否有 Try 记录。若无 Try 记录，直接返回成功，不做资源释放。 |
| **悬挂** | Cancel 执行了（可能是空回滚），之后网络恢复，Try 请求才迟到了。此时 Try 应当被拒绝，否则资源预留后永远没人释放。 | Try 接口执行前，检查是否已有 Cancel 记录。若已 Cancel，拒绝执行 Try。 |
| **幂等性** | Confirm/Cancel 调用超时触发重试。 | 利用事务日志表记录状态，Confirm/Cancel 执行时更新状态，重复调用时判断状态若已完成则直接返回。 |

### 实战案例
在一个积分兑换系统中，曾遇到“Try 成功但调用方未收到响应导致重试 Try”的并发问题，导致用户余额被冻结两次。**修复方案**：Try 阶段不仅要有 `UPDATE user SET freeze = freeze + amount`，还需要在事务日志表插入一条唯一标识（txn_id）的记录。利用数据库唯一键冲突拦截重复 Try 请求，保证 Try 本身的幂等性。

### 代码示例
```java
// Java: Try 阶段防止悬挂和重复 Try 的实现
@Transactional
public boolean tryPrepare(BusinessActionContext context, String accountId, BigDecimal amount) {
    String txnId = context.getXid(); // 全局事务ID
    
    // 1. 防悬挂：检查该事务是否已经取消过
    if (transactionLogDao.existsCancel(txnId)) {
        throw new IllegalStateException("Transaction cancelled, cannot try");
    }

    // 2. 插入事务日志 (防重复 Try，利用DB唯一索引)
    // 状态为 TRYING
    transactionLogDao.insert(txnId, "TRYING"); 

    // 3. 预留资源 (冻结余额)
    int rows = accountMapper.freezeBalance(accountId, amount);
    if (rows == 0) {
        throw new RuntimeException("Insufficient balance or failed to freeze");
    }
    return true;
}

// Confirm 阶段：基于状态判断幂等
public boolean confirm(BusinessActionContext context) {
    String txnId = context.getXid();
    // 利用 DB 的行锁或状态机判断
    int rows = transactionLogDao.updateStatus(txnId, "TRYING", "CONFIRMED");
    if (rows == 0) {
        return true; // 已经是 CONFIRMED 或其他状态，直接返回成功
    }
    // 执行实际扣款：freeze -> balance
    accountMapper.commitDeduct(context.getActionContext("accountId"), context.getActionContext("amount"));
    return true;
}
```

**异常处理**：
- 如果 Try 阶段失败，则执行 Cancel。
- 如果 Confirm 或 Cancel 阶段失败，事务管理器会通过事务日志进行重试（要求接口幂等）；若重试失败，需人工介入处理。

**流程架构图**：
```text
      事务开始      
         │
         ▼
┌─────────────────┐
│   Try 阶段      │─── Fail ──> Cancel (回滚资源)
│ (检查 & 预留)    │
└────────┬────────┘
         │ Success
         ▼
┌─────────────────┐
│  Confirm 阶段   │─── Fail ──> 重试 (需幂等) ──> 人工介入
│ (确认提交业务)   │
└────────┬────────┘
         │
         ▼
      事务结束
```

## 常见考点
1. **空回滚与悬挂**：Try 未执行但先执行了 Cancel（空回滚），或 Cancel 执行了之后 Try 才到达（悬挂），如何通过事务日志或状态表解决？
2. **幂等性实现**：Confirm 和 Cancel 阶段如何通过唯一业务主键或 Token 机制实现幂等？
3. **事务日志**：每个分支事务需记录独立的事务日志，用于故障恢复，这部分数据如何落盘？
