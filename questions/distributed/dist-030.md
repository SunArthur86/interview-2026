# 什么是柔性事务？最大努力通知型事务是什么？

### 三阶段提交协议 (3PC)
3PC 是 2PC 的改进版，旨在解决 2PC 的阻塞问题。引入了超时机制，并将准备阶段拆分为 CanCommit 和 PreCommit。

#### 阶段一：CanCommit
协调者询问参与者是否可以执行事务，参与者只需检查资源是否充足，无需执行操作。

#### 阶段二：PreCommit
如果参与者都回复 Yes，协调者发送 PreCommit 请求。参与者执行事务操作并写入日志，但不提交。

#### 阶段三：DoCommit
协调者发送 DoCommit 请求，参与者正式提交。若协调者宕机，参与者根据超时机制决定提交（PreCommit 阶段后超时默认提交），从而减少阻塞。

### 柔性事务
在互联网高并发场景下，为了平衡一致性与可用性，放弃强一致性（ACID），采用 BASE 理论实现最终一致性。

#### BASE 理论
- **BA (Basically Available)**：基本可用，允许损失部分可用性（如响应变慢、功能降级）。
- **S (Soft State)**：软状态，允许数据存在中间状态（如“支付中”），不代表最终状态。
- **E (Eventual Consistency)**：最终一致性，经过一段时间后，数据最终达到一致。

#### 常见柔性事务类型
1. **两阶段型 (2PC)**：强一致，性能差。
2. **补偿型 (TCC)**：Try-Confirm-Cancel，业务侵入性强，但性能和一致性较好。
3. **异步确保型**：利用消息队列确保最终一致（如本地消息表）。
4. **最大努力通知型**：不保证强一致，通过多次重试通知，适合对实时性要求不高的场景（如支付结果通知）。

### 最大努力通知型详解
**原理**：
这是一种利用消息中间件或轮询机制进行重复通知的方案。业务活动的主动方在完成业务处理后，向被动方发送消息，允许消息丢失，主动方通过重试机制保证“最大努力”将消息传递给被动方。

**流程**：
1. 主动方调用业务接口。
2. 主动方记录本地消息表（状态为“发送中”）。
3. 调用消息通知服务（如 MQ 或 HTTP），通知被动方。
4. 被动方接收消息，处理业务，并返回 ACK。
5. 主动方收到 ACK，更新消息表状态为“已发送”。
6. 如果未收到 ACK，定时任务扫描“发送中”的消息，进行重试（通常有最大重试次数限制）。

### 实战案例：支付回调地址变更导致通知失败
某支付系统通过最大努力通知商户系统。某次变更导致商户回调接口暂时下线。由于支付侧设计了“阶梯式重试策略”（1min, 5min, 10min, 30min... 持续24小时），在商户服务恢复后的第1次重试中成功通知，避免了资金对账差异。若仅做简单的同步调用，这笔订单将永久处于“待支付”状态。

### 代码示例：本地消息表 + 定时任务 (Java伪代码)
```javan@Transactional
public void processPayment(Order order) {
    // 1. 执行业务逻辑
    paymentDao.deduct(order);
    // 2. 写入本地消息表 (同一本地事务，保证原子性)
    LocalMessage msg = new LocalMessage(order.getId(), "PAY_NOTIFY", "SENDING");
    messageDao.insert(msg);
}

// 定时任务扫描
@Scheduled(fixedDelay = 30000)
public void retryNotify() {
    List<LocalMessage> msgs = messageDao.queryByStatus("SENDING", 5); // 限制重试次数
    for (LocalMessage msg : msgs) {
        try {
            boolean success = httpClient.post(msg.getTargetUrl(), msg.getContent());
            if (success) {
                messageDao.updateStatus(msg.getId(), "SENT");
            }
        } catch (Exception e) {
            // 记录异常，下次继续重试
            messageDao.incrRetryCount(msg.getId());
        }
    }
}
```

### 对比表格：常见分布式事务方案
| 方案 | 一致性 | 并发性能 | 复杂度 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **2PC/XA** | 强一致 | 低 (锁资源) | 低 | 传统传统后台，并发不敏感 |
| **TCC** | 最终一致 | 高 | 高 (需写三个接口) | 核心高并发业务，如账户扣款 |
| **本地消息表** | 最终一致 | 中 | 中 | 异步业务，如跨系统数据同步 |
| **最大努力通知** | 最终一致 | 高 | 低 | 通知类业务，如支付结果通知 |
| **Seata AT** | 最终一致 | 中 | 低 | 业务逻辑简单的微服务场景 |

### 常见考点
1. **TCC 事务的空回滚和悬挂是什么？**（提示：空回滚是 Try 未执行但 Cancel 执行了；悬挂是 Cancel 执行完了 Try 才执行）。
2. **本地消息表如何保证一致性？**（提示：利用本地事务保证业务操作和消息写入的原子性，配合定时任务重试）。
3. **3PC 相比 2PC 到底解决了什么问题？**（提示：主要解决了协调者故障后参与者的阻塞问题，但引入了网络分区导致数据不一致的风险）。


## 记忆要点

- 柔性事务基于BASE理论，牺牲强一致性（ACID），换取高可用和最终一致性
- 最大努力通知：主动方配合本地消息表+定时任务阶梯重试，被动方不需返回结果保证
- 对比：本地消息表重在内部数据同步一致性，最大努力通知重在跨平台外部结果通知（如支付回调）
