# 2PC（标准XA模型）

### 2PC (Two-Phase Commit，二阶段提交)

2PC 是分布式事务中最经典的协议，广泛应用于数据库领域。绝大多数关系型数据库都基于 2PC 完成分布式事务处理。旨在保证分布式架构下所有节点在进行事务处理时能够保持原子性和一致性。

#### 1. 模型角色
- **协调者**：事务的发起者和调度者（通常对应XA中的TM）。
- **参与者**：执行具体事务操作的资源节点（通常对应XA中的RM）。

#### 2. 执行流程图

```text
      Coordinator (TM)                Participants (RMs)
     +-----------------+             +---------------------+
     |                 |             |                     |
     |  Phase 1: Prepare             |                     |
     |      (CanCommit?)|------------>|  Execute Transaction|
     |                 |             |  Write Redo/Undo Log|
     |                 |<------------|  (Yes/No)           |
     |                 |             |                     |
     |  Phase 2: Commit|             |                     |
     |      (DoCommit) |------------>|  Commit             |
     |      or Abort   |   (All Yes) |  Release Locks      |
     |                 |             |                     |
     |      (DoAbort)  |------------>|  Rollback           |
     |      (Any No)   |             |  Undo Log           |
     +-----------------+             +---------------------+
```

#### 3. 执行流程详解

2PC 分为两个阶段：**准备阶段** 和 **提交阶段**。

##### 阶段一：提交事务请求
1. **事务询问**：协调者向所有参与者发送事务内容，询问是否可以执行提交操作，并开始等待响应。
2. **执行事务**：各参与者节点执行事务操作，并将 Undo 和 Redo 操作记入本机事务日志（此时不提交，资源被锁定）。
3. **反馈响应**：各参与者向协调者反馈事务询问的响应。成功执行返回 Yes，否则返回 No。

##### 阶段二：执行事务提交
协调者根据阶段一的反馈结果决定提交或中断：
- **情况A（所有参与者都回复Yes）**：
  1. 发送“提交请求”。
  2. 参与者执行正式提交，释放资源。
  3. 反馈“完成”确认。
- **情况B（任意参与者回复No 或 超时）**：
  1. 发送“中断请求”。
  2. 参与者利用 Undo 信息执行事务回滚。
  3. 反馈“回滚”确认。

#### 4. 优缺点
- **优点**：原理简单，实现强一致性。
- **缺点**：
  - **同步阻塞**：所有参与者阻塞等待，导致性能低下。
  - **单点故障**：协调者故障会导致参与者一直阻塞。
  - **数据不一致**：在阶段二出现网络分区时，可能导致部分提交部分回滚。

#### 5. 实战深化：阻塞与优化

**实战案例**：在支付系统中，如果2PC阶段一耗时较长（如因为慢SQL），会导致数据库连接被长时间占用无法释放，进而拖垮整个数据库的吞吐量，生产环境需监控 Prepare 阶段的耗时。

**代码示例：模拟2PC协调者伪代码**
```java
public class TwoPhaseCommitCoordinator {
    public boolean executeTransaction(List<Participant> participants) {
        // Phase 1: Prepare
        boolean allPrepared = true;
        for (Participant p : participants) {
            if (!p.prepare()) { // 执行业务并锁定资源
                allPrepared = false;
                break;
            }
        }

        // Phase 2: Commit or Rollback
        if (allPrepared) {
            participants.forEach(Participant::commit);
            return true;
        } else {
            participants.forEach(Participant::rollback);
            return false;
        }
    }
}
```

**对比表格：2PC vs 3PC**

| 协议 | 阶段数 | 阻塞风险 | 单点故障容忍度 | 一致性保障 |
| :--- | :--- | :--- | :--- | :--- |
| **2PC** | 2 (Prepare, Commit) | 高 (锁住直到结束) | 低 (Coordinator挂机则阻塞) | 强一致性 (可能阻塞) |
| **3PC** | 3 (CanCommit, PreCommit, DoCommit) | 中 (引入超时机制) | 中 (参与者可超时决策) | 在无分区下强一致 |

## 常见考点
1. **同步阻塞的根本原因**：为什么2PC在Prepare阶段必须锁定资源？如果锁定时间过长对数据库吞吐量有什么影响？
2. **协调者单点故障**：如果协调者在Phase 2发出Commit指令前宕机，参与者将如何处理？（通常依赖超时机制，但可能造成数据不一致）
3. **脑裂问题**：在Phase 2，如果只有部分参与者收到了Commit指令，而网络发生分区，会产生什么后果？
4. **2PC与3PC的区别**：3PC引入的PreCommit阶段如何缓解阻塞，但为什么又引入了新的问题？
