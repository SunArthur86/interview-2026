# Java应用在容器中常见的OOM和性能问题有哪些？如何排查？

🎯 本质：容器环境下的Java内存管理受 Cgroup 限制，容易出现容器 OOM Killed（系统杀进程）和 JVM OOM（应用抛异常）两种不同层级的问题。

**1. 常见内存问题分类**

**A. 容器 OOM Killed (Cgroup 层面)**
- **现象**：Pod 状态为 `OOMKilled`，退出码 137 (128 + 9 SIGKILL)，应用日志可能戛然而止。
- **原因**：JVM 总内存（Heap + Metaspace + Direct Memory + Stack + Code Cache + GC开销）超过了 K8s 设置的 `limits.memory`。
- **排查**：`kubectl describe pod <pod-name>` 查看 `Last State: Terminated (OOMKilled)`。
- **解决**：
  - 使用 `-XX:MaxRAMPercentage=75.0` 动态限制堆大小。
  - 留出足够内存给非堆区域（建议预留 20-30%）。

**B. JVM Heap OOM (应用层面)**
- **现象**：日志抛出 `java.lang.OutOfMemoryError: Java heap space`。
- **原因**：内存泄漏或堆内对象过多，GC 无法回收。
- **排查**：
  - 启动参数加 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/logs/heap.hprof`。
  - 使用 MAT (Memory Analyzer Tool) 分析 Dump 文件，找 Retained Heap 最大的对象。

**C. Metaspace OOM**
- **现象**：`OutOfMemoryError: Metaspace`。
- **原因**：动态加载的类过多（如反射生成代理类、Groovy脚本、大量JSP）。
- **解决**：调整 `-XX:MaxMetaspaceSize=256m`，排查是否存在类加载泄漏。

**D. Direct Memory OOM**
- **现象**：`OutOfMemoryError: Direct buffer memory`。
- **原因**：Netty/NIO 堆外内存未释放，或未设置限制。
- **解决**：`-XX:MaxDirectMemorySize=256m`，开启 Native Memory Tracking (`-XX:NativeMemoryTracking=detail`) 排查。

**E. GC overhead limit**
- **现象**：`java.lang.OutOfMemoryError: GC overhead limit exceeded`。
- **原因**：应用运行时间 98% 都在做 GC，回收的内存少于 2%。
- **解决**：检查内存泄漏，或扩容 Heap，或调整 GC 策略（如从 Serial 切换到 G1）。

**2. 容器内存布局与计算**
```
┌──────────────────────────────────────────────────┐
│  Container Limit (K8s Limit, e.g., 4GB)          │
├──────────────────────────────────────────────────┤
│  JVM Heap (-Xmx or MaxRAMPercentage ~75%)        │
│  (Java Objects)                                  │
├──────────────────────────────────────────────────┤
│  Non-Heap Memory (~25%)                          │
│  - Metaspace (Class metadata)                    │
│  - Direct Memory (NIO/Netty buffers)             │
│  - Thread Stacks (Xss * 线程数)                  │
│  - Code Cache (JIT compiled code)                │
│  - GC Structures (StringDedup, etc.)             │
│  - Other (OS overhead, Socket buffers)           │
└──────────────────────────────────────────────────┘
```

**3. 关键启动参数配置**
```bash
# 推荐配置 (Java 8u191+ / Java 11+ 支持容器感知)
java -jar app.jar \
  -XX:+UseContainerSupport \
  -XX:MaxRAMPercentage=75.0 \
  -XX:InitialRAMPercentage=75.0 \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/tmp/heapdump.hprof \
  -XX:+PrintGCDetails -XX:+PrintGCDateStamps
```

**实战案例**：某服务在容器中重启，排查发现是堆外内存溢出导致容器被杀。由于未设置 `-XX:MaxDirectMemorySize`，默认等同于 Max Heap，导致 Netty 分配大量堆外 Buffer 时突破了容器 Limit。解决方案是将 MaxRAMPercentage 调至 60%，并为非堆内存预留足够空间。

## 记忆要点

- OOM层级区分：JVM OOM是堆溢出抛异常(需Dump分析)，而容器OOMKilled是总内存超Cgroup被系统杀进程。
- 内存比例：因为JVM非堆内存需开销，所以堆内存建议配置MaxRAMPercentage=75%而非占满Limit。
- 排查指令：堆溢出用HeapDumpOnOutOfMemoryError生成hprof文件分析，而堆外内存溢出需开启NativeMemoryTracking。
