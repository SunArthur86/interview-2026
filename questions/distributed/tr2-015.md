# 在分布式系统中，如何设计一个“延迟消息”队列？如果不想引入 RabbitMQ 或 RocketMQ 等中间件的延迟投递功能，基于 Redis 或数据库如何实现？

如果不依赖 MQ 中间件，常见的基于 Redis 的实现方案有两种：1. 使用 ZSet（有序集合），将 `score` 设为消息执行的时间戳，`value` 为消息内容。开启一个后台线程轮询检查 ZSet 中 score 最小的元素，如果当前时间 >= score，则弹出并处理；2. 使用 Redis Keyspace Notifications（键空间通知），监听 Key 的过期事件，但该方案在高并发下存在丢失消息风险且不可靠。基于数据库的实现通常是设计一张包含 `execute_time` 和 `status` 的任务表，通过一个定时任务（如 Spring Schedule）每秒轮询查询到期的任务，更新状态并执行。为保证高性能，数据库方案通常配合分表或使用 `skip-locked` 特性。相比 MQ 的专用磁盘结构，自研方案需解决轮询性能、消息可靠性和集群重复消费等问题。

## 技术原理

- **Redis ZSet 利用 Score 存时间戳，通过轮询实现有序延迟投递**：ZSet 的 score 用消息的执行时间戳（毫秒），value 用消息 ID 或 JSON。后台线程（或 Lua 脚本）每秒执行 `ZRANGEBYSCORE key 0 now LIMIT 0 N` 拉取到期消息，再 `ZREM` 删除并投递。ZSet 天然按 score 排序，复杂度 O(logN) 入队、O(logN+M) 出队。
- **数据库 Job 通过 execute_time 索引轮询，需处理锁和分表性能问题**：建一张 `delay_task(execute_time, status, payload)` 表，在 `(status, execute_time)` 上建联合索引。多节点用 `SELECT ... FOR UPDATE SKIP LOCKED`（MySQL 8.0+/PG）抢任务避免重复消费；分表按时间或 hash 分片缓解单表写入和轮询压力。
- **相比 MQ，自研方案需自行解决时钟回退、消息丢失及集群重复消费等隐患**：MQ 的延迟队列（RocketMQ 18 个延迟等级、RabbitMQ TTL+DLX）天然解决持久化和投递可靠性。自研方案要么靠 Redis AOF/RDB（仍有秒级丢失风险），要么靠 DB 事务（重但可靠），还要自己处理集群时钟漂移、节点宕机的任务重平衡。

## 代码示例

Redis ZSet + Lua 实现原子拉取（避免轮询线程间竞争）：

```lua
-- KEYS[1]=zset key  ARGV[1]=now  ARGV[2]=batch
local items = redis.call('ZRANGEBYSCORE', KEYS[1], 0, ARGV[1], 'LIMIT', 0, ARGV[2])
if #items > 0 then
    redis.call('ZREM', KEYS[1], unpack(items))
end
return items
```

数据库 + SKIP LOCKED（MySQL 8.0）：

```sql
CREATE TABLE delay_task (
    id           BIGINT      PRIMARY KEY AUTO_INCREMENT,
    execute_time BIGINT      NOT NULL,
    status       TINYINT     NOT NULL DEFAULT 0 COMMENT '0=待执行 1=已领取 2=完成',
    payload      TEXT        NOT NULL,
    INDEX idx_time_status (status, execute_time)
);

-- 多节点并发抢任务，SKIP LOCKED 跳过已被其他节点锁定的行
START TRANSACTION;
SELECT id, payload FROM delay_task
 WHERE status = 0 AND execute_time <= UNIX_TIMESTAMP()*1000
 ORDER BY execute_time LIMIT 50 FOR UPDATE SKIP LOCKED;

UPDATE delay_task SET status = 1 WHERE id IN (...);
COMMIT;
```

## 对比/选型

| 方案 | 精度 | 可靠性 | 吞吐 | 复杂度 |
|------|------|--------|------|--------|
| Redis ZSet 轮询 | 秒级 | 中（RDB/AOF 丢窗口） | 高 | 低 |
| Redis Keyspace Notification | 秒级 | 极低（事件易丢） | 中 | 极低 |
| DB + SKIP LOCKED | 秒级 | 高（事务保证） | 中 | 中 |
| RocketMQ 延迟等级 | 秒级（固定等级） | 高 | 高 | 低 |
| 时间轮（Netty） | 毫秒 | 低（内存） | 极高 | 高 |

## 常见坑/注意事项

- **键空间通知极不可靠**：Redis 的 `__keyevent@0__:expired` 是"尽力而为"，Keyspace 通知若客户端断连就丢，且 expire 事件本身在 Redis 内存压力下可能延迟触发甚至丢失，切忌作为核心方案。
- **ZSet 大 key 问题**：单个 ZSet 积压百万消息会阻塞 Redis，建议按业务/时间分片（如 `delay:202607:order`）。
- **时钟回退**：NTP 校时或容器时钟漂移会导致 `now < score` 的任务永远不执行，记录绝对时间戳并加补偿扫描。
- **集群重复消费**：DB 方案必须用 `FOR UPDATE SKIP LOCKED` 或乐观锁版本号，否则多节点会重复投递；Redis 方案用 Lua 保证 `ZRANGEBYSCORE + ZREM` 原子。
- **任务投递后的可靠性**：拉到任务后若处理失败要能重试，建议标记 `status=1` 但加超时回滚机制，避免任务卡死。
