# 在分布式系统中，如何设计一个“延迟消息”队列？如果不想引入 RabbitMQ 或 RocketMQ 等中间件的延迟投递功能，基于 Redis 或数据库如何实现？

如果不依赖 MQ 中间件，常见的基于 Redis 的实现方案有两种：1. 使用 ZSet（有序集合），将 `score` 设为消息执行的时间戳，`value` 为消息内容。开启一个后台线程轮询检查 ZSet 中 score 最小的元素，如果当前时间 >= score，则弹出并处理；2. 使用 Redis Keyspace Notifications（键空间通知），监听 Key 的过期事件，但该方案在高并发下存在丢失消息风险且不可靠。基于数据库的实现通常是设计一张包含 `execute_time` 和 `status` 的任务表，通过一个定时任务（如 Spring Schedule）每秒轮询查询到期的任务，更新状态并执行。为保证高性能，数据库方案通常配合分表或使用 `skip-locked` 特性。相比 MQ 的专用磁盘结构，自研方案需解决轮询性能、消息可靠性和集群重复消费等问题。

## 记忆要点

- Redis ZSet方案：score存执行时间戳，value存消息，后台线程轮询当前时间>=score的任务。
- 数据库方案：新建任务表存execute_time，定时轮询到期任务并更新状态。
- DB防并发争抢：利用数据库的 skip-locked 特性避免集群多个节点重复拉取消费。
- 键空间通知陷阱：Redis过期事件极不可靠，高并发下会丢消息，切忌作核心方案。

## 结构化回答

**30 秒电梯演讲：** 利用有序结构或定时扫描，将“时间维度的任务”转化为“当前维度的执行”。打比方——就像把未来的闹钟设按时间排好队(Redis ZSet)，或者安排专人每分钟查看一下记事本(数据库轮询)，时间到了就叫醒你。落到工程上，Redis ZSet 利用 Score 存时间戳，通过轮询实现有序延迟投递。

**展开框架：**
1. **Redis ZSet** — Redis ZSet 利用 Score 存时间戳，通过轮询实现有序延迟投递
2. **数据库 Job** — 数据库 Job 通过 execute_time 索引轮询，需处理锁和分表性能问题
3. **相比 MQ** — 相比 MQ，自研方案需自行解决时钟回退、消息丢失及集群重复消费等隐患
