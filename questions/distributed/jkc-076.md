# N：在分布式存储系统中，有多少份备份数据

在分布式存储系统（如Dynamo风格系统）中，**N** 代表数据对象的总副本数量，即一份数据在集群中被保存的份数。

**NWR 模型核心定义**：
- **N (Replicas)**：副本总数，通常设置为3或5。N越大，数据可靠性越高，但存储成本和网络开销越大。
- **W (Write Quorum)**：写一致性级别，表示一次成功的写操作需要至少 W 个副本确认写入成功。
- **R (Read Quorum)**：读一致性级别，表示一次成功的读操作需要至少读取 R 个副本。

**一致性与可用性的权衡**：
通过调整 W 和 R 的值，可以权衡一致性与性能：
1. **强一致性**：当 **W + R > N** 时，读写集合必有交集，能保证读到最新写入的数据。
2. **弱一致性（最终一致性）**：当 **W + R <= N** 时，读写可能不重叠，系统允许读取到旧数据，但写入延迟更低，可用性更高。

**常见配置示例**：
- N=3, W=2, R=2：保证强一致性（2+2>3），适合读多写少的场景。
- N=3, W=1, R=1：保证最终一致性，吞吐量最高，适合允许短暂不一致的场景。

```text
  节点A   节点B   节点C  (假设 N=3)
    |      |      |
    |--- W --->|      |  
    |      |--- W --->|  
           (写操作 W=2: 只要任意两个节点成功即返回)

    |<-- R ---|      |
    |             (读操作 R=2: 读任意两个节点，取时间戳最新的)
```

**实战案例**：在电商大促库存扣减场景中，为了防止超卖，不能容忍“脏读”，因此我们将 N=3, W=2, R=2 升级为 N=3, W=3, R=1（或 W=2, R=2）。反之，在用户头像更新场景中，为了保证高并发下的写入成功率，我们通常会采用 N=3, W=1, R=1 的配置，允许短时间内部分用户看到旧头像。

**关键代码示例 (Cassandra/Java)**：
```java
// 设置一致性级别为 QUORUM (N/2 + 1)，即保证强一致性
Statement statement = QueryBuilder.insertInto("users")
    .value("id", 123)
    .value("name", "Alice")
    .setConsistencyLevel(ConsistencyLevel.QUORUM); // 对应 W=2 (当N=3时)

session.execute(statement);
```

**常见考点**：
1. **Quorum (法定人数)**：W和R通常设置为 N/2 + 1（向上取整），这就是通常说的Quorum机制。
2. **Hinted Handoff**：当 W 个节点写入失败时，系统可能暂存数据并在节点恢复时传递，这如何影响 W 的定义？
3. **R/W 对性能的影响**：W 较大会导致写入延迟增加；R 较大会导致读取延迟增加。
4. **故障场景**：在 N=3, W=2 时，如果有 1 个节点宕机，系统是否还能写入？（可以，因为还能找到2个节点）。


## 记忆要点

- NWR 定义：N 代表数据总副本数，W 代表写入成功数，R 代表读取成功数。
- 强一致公式：当 W + R > N 时，读写必有交集，必能读到最新数据（如 N=3, W=2, R=2）。
- 最终一致：当 W + R <= N 时，允许部分节点读取旧数据，换取更高的吞吐和可用性。
- Quorum 法定：通常将 W 和 R 设置为 N/2 + 1（多数派），兼顾一致性与性能。
