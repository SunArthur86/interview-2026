# 最大努力通知

# 最大努力通知

## 核心定义
最大努力通知方案的目标，是发起通知方通过一定的机制，最大努力将业务处理结果通知到接收方。

**本质**：通过引入定期校验机制实现最终一致性，对业务的侵入性较低。
**适用性**：
*   **场景**：对最终一致性敏感度较低、业务链路较短的场景。
*   **对象**：主要用于**外部系统**交互。因为外部网络环境更加复杂和不可信，跨防火墙、跨企业系统稳定性难以保证，所以只能尽最大努力去通知。
*   **例子**：充值平台与运营商、支付对接（支付宝/微信回调）、商户通知等。

*(注：异步确保型事务主要适用于内部系统的数据最终一致性保障，因为内部环境相对可控。)*

## 方案难点
普通消息无法解决本地事务执行和消息发送的一致性问题。
*   **原因**：消息发送是网络通信过程，存在发送失败、超时。超时意味着状态未知（可能成功，可能失败），此时无论提交还是回滚事务，都可能出现不一致。
*   **解决**：核心在于保障“投递消息”和“参与者本地事务”的一致性。

## 实现流程
最大努力通知在投递流程上与异步确保型一致，通常包含两个分支：
1.  **基于 MQ 自身的事务消息方案**（如 RocketMQ 事务消息）。
2.  **基于 DB 的本地消息表方案**（适用于不支持事务消息的 MQ）。

**针对“最大努力”的特殊增强机制**：
除了常规的消息投递，最大努力通知通常还会辅以以下策略：
*   **主动查询接口**：接收方提供查询接口，发送方在通知失败或超时后，主动调用接口查询结果。
*   **定期对账**：双方系统定期通过文件或接口进行数据核对，发现不一致时进行人工或自动补偿。

## 流程图
```text
    通知方 (如支付平台)                      网络                    接收方 (如商户系统)
          |                                    |                             |
          |--- 1. 执行本地事务 (扣款)              |                             |
          |                                    |                             |
          |--- 2. 记录通知日志 (状态: 待通知)       |                             |
          |                                    |                             |
          |--- 3. 发送通知 (HTTP/MQ) -----------> | (网络抖动/超时)             |
          |                                    | -------------> [未收到]       |
          |                                    |                             |
          |--- 4. 有限次数重试 (如 5次) ---------> |                             |
          |      (间隔递增: 1m, 5m, 10m...)      |                             |
          |                                    |                             |
          |--- 5. 超过阈值, 停止通知              |                             |
          |                                    |                             |
          |--- 6. 进入被动对账流程                |                             |
          |    (每日凌晨 T+1 生成对账文件)          |                             |
          |                           
```

### 实战案例
对接某银行支付接口时，因银行端防火墙策略变更，导致我方回调请求被全部拦截。**踩坑经验**：除了重试，必须要求合作方提供“支付结果查询接口”，当重试 N 次失败后，自动降级为轮询查询接口，确保业务闭环。

### 代码示例
```java
// Java: 间隔递增的重试策略伪代码
public void notifyWithRetry(String url, Object payload) {
    int retryTimes = 0;
    int[] intervals = {1, 2, 5, 10, 30}; // 分钟
    while (retryTimes < intervals.length) {
        try {
            restTemplate.postForObject(url, payload, Void.class);
            logService.updateStatus(msgId, "SUCCESS");
            return;
        } catch (Exception e) {
            Thread.sleep(intervals[retryTimes] * 60 * 1000);
            retryTimes++;
        }
    }
    // 超过重试次数，触发报警或转入对账队列
    alertService.send(msgId);
}
```

- 本质：通过定期校验和重试兜底实现最终一致性，主要用于外部系统对接。

- 流程：先执行本地事务并记日志，若通知失败则按递增间隔重试。
