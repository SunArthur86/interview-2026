# 什么是TCC 事务模型

TCC（Try-Confirm-Cancel）分布式事务模型是一种应用层的柔性事务解决方案，它将业务逻辑拆分为两个阶段，通过补偿机制来保证数据的一致性。

### 核心概念
TCC 模型来源于 Pat Helland 的论文，其核心思想是将一个完整的业务操作拆分为三个子操作：

1.  **Try（预留阶段）**：完成所有业务检查，预留必须的业务资源（如冻结余额、锁定库存）。
2.  **Confirm（确认阶段）**：真正执行业务，不再做业务检查。直接使用 Try 阶段预留的资源，完成业务操作。Confirm 操作必须满足幂等性。
3.  **Cancel（取消/回滚阶段）**：释放 Try 阶段预留的资源，回滚业务。Cancel 操作必须满足幂等性。

### 角色划分

1.  **主业务服务**：整个业务活动的发起方，负责编排全局事务，发起 Try、Confirm 或 Cancel 指令。
2.  **从业务服务**：业务活动的参与者，提供 TCC 接口。
3.  **业务活动管理器**：管理全局事务状态（如初始、Try 成功、Confirm 成功、Cancel 成功等）以及各个参与者的子事务状态。

### TCC vs Saga vs 2PC/3PC 对比

| 特性 | TCC | Saga | 2PC/3PC (XA) |
| :--- | :--- | :--- | :--- |
| **一致性** | 最终一致（强可控） | 最终一致（长事务友好） | 强一致 (ACID) |
| **并发性能** | 高（锁资源由业务控制） | 中 | 低（资源锁住时间长） |
| **实现复杂度** | 高（需编写三个接口） | 中（需定义正向/反向补偿） | 低（数据库/中间件支持） |
| **适用场景** | 高并发、性能敏感的核心业务 | 长流程、多步骤的业务流转 | 传统单体、内部系统或对强一致性要求极高的场景 |
| **锁定机制** | 应用层锁（如冻结余额） | 无锁或乐观锁 | 数据库层面的行锁/表锁 |

### 实战案例
在金融转账系统中，若直接使用 XA 两阶段提交，数据库锁会从 Try 阶段一直持有到 Commit，导致吞吐量极低。采用 TCC 后，Try 阶段仅冻结“待转账金额”（内存/DB字段标记），不阻塞账户其他支出；Confirm 阶段才实际扣减。**踩坑经验**：曾因 Try 阶段预留资源未实现数据库唯一约束，导致并发请求下“预留余额”被多次累加，Confirm 时余额扣成负数。**教训**：Try 操作本身对资源的操作也必须是幂等的（如 `UPDATE account SET freeze = freeze + 100 WHERE id=?`），而非简单地内存赋值。

### 代码示例
```java
// Java: TCC 资金账户业务接口示例
public interface AccountTccService {
    /**
     * Try: 预留资源
     */
    @TwoPhaseBusinessAction(name = "accountTry", commitMethod = "confirm", rollbackMethod = "cancel")
    public boolean prepareDecreaseAccount(@BusinessActionContextParameter(paramName = "accountId") String accountId, 
                                          double amount);

    /**
     * Confirm: 真正扣减余额 (幂等)
     */
    public boolean confirm(BusinessActionContext context);

    /**
     * Cancel: 释放冻结余额 (幂等)
     */
    public boolean cancel(BusinessActionContext context);
}
```

### 处理流程

```
      主业务服务                 从业务服务 A                 从业务服务 B
          |                          |                          |
   1. 开启全局事务                |                          |
          |                          |                          |
   2. Try 调用  ---------------->|   Try: 检查并预留资源      |                          |
          |                        (如：冻结余额)               |                          |
          |<---------------------   Try Success                |                          |
          |                          |                          |
   3. Try 调用  ---------------------------------------->|   Try: 检查并预留资源      |
          |                                              (如：锁定库存)               |
          |<-----------------------------------------   Try Success                |
          |                          |                          |
      (所有 Try 成功)               |                          |
          |                          |                          |
   4. Confirm 调用 -------------->|  Confirm: 使用余额扣款     |                          |
          |<---------------------   Confirm Success            |                          |
          |                          |                          |
   5. Confirm 调用 ------------------------------------->|  Confirm: 使用库存扣减     |
          |                                              (释放锁)                    |
          |<-----------------------------------------   Confirm Success            |
          |                          |                          |
   6. 全局事务结束                |                          |
```

## 记忆要点

- 一句话：TCC是应用层柔性事务，将业务拆分预留确认与取消三步接口。
- Try阶段：仅完成业务检查并预留资源（如冻结资金），不持有DB长锁。
- Confirm阶段：直接使用Try预留资源执行真业务，要求必须实现幂等。
- Cancel阶段：释放Try预留资源执行回滚，同样要求必须实现幂等。
- 对比2PC：TCC位于应用层，锁粒度由业务灵活控制，并发性能大幅提升。
