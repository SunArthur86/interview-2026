# 什么是Service Mesh？它和Spring Cloud有什么区别？如何选择？

🎯 本质：Service Mesh是一个基础设施层，通过Sidecar代理处理服务间通信，将服务治理能力从应用代码中解耦出来。

🧒 类比：Spring Cloud像每辆车自带导航系统（每个服务都要集成SDK）；Service Mesh像道路上的交通信号灯系统，所有车经过时自动获得导航、限速、改道能力，车本身不需要任何改动。

**架构与数据流**：
```text
                 ┌───────────────────────────────────────────┐
                 │           Service Mesh (控制面)           │
                 │  (Pilot: 配置分发, Citadel: 证书, Galley) │
                 └───────────────┬───────────────────────────┘
                               │ ▼ (配置推送)
┌─────────────┐         ┌──────┴───────────┐         ┌─────────────┐
 │   Service A │◄──────►│  Envoy Sidecar  │◄──────►│  Envoy Sidecar│◄──────►│   Service B │
 └─────────────┘  RPC   └──────────────────┘   RPC   └──────────────────┘  RPC   └─────────────┘
   (应用容器)         (透明代理/数据面)               (透明代理/数据面)      (应用容器)
```

📊 **Service Mesh vs Spring Cloud**：
| 维度 | Spring Cloud | Service Mesh(Istio) |
| :--- | :--- | :--- |
| **治理位置** | 应用层(SDK) | 基础设施层(Sidecar) |
| **语言** | 仅Java | 多语言支持 |
| **升级** | 需重新部署应用 | 只升级Sidecar |
| **侵入性** | 高(需改代码) | 低(零侵入) |
| **功能** | 服务注册+熔断+配置 | 流量管理+安全+可观测 |
| **性能损耗** | 较低(进程内调用) | 较高(多一次网络跳转/IPC) |
| **运维复杂度** | 中等(需维护注册中心) | 极高(控制面+数据面) |

**Service Mesh核心架构**：
数据面：Envoy/Sidecar代理，负责流量转发、负载均衡、熔断。它以iptables/ebpf方式劫持流量，应用无感知。
控制面：Istio Pilot(配置分发)、Citadel(安全证书)、Galley(配置验证)

**关键能力**：
1. 流量管理 - 金丝雀发布（按权重/Header分流）、A/B测试、流量镜像（无损测试）
2. 安全 - mTLS双向认证（自动下发证书）、授权策略（RBAC）
3. 可观测性 - 分布式追踪（自动注入Trace ID）、指标
4. 弹性 - 自动重试、超时、熔断、连接池管理

**选择建议**：
- **Spring Cloud**：纯Java技术栈、中小规模（<50服务）、团队熟悉Spring生态、追求低延迟。
- **Service Mesh**：多语言异构系统（Go/Python/Java混用）、大规模（K8s集群）、需要精细化流量治理、对应用代码零侵入有强要求。

## 常见考点
1. **Sidecar流量劫持原理**：如何通过iptables将流量劫持到Sidecar？（PREROUTING/OUTPUT链）
2. **性能损耗与优化**：多了一跳Sidecar，延迟增加了多少？如何通过链路透传减少上下文切换？
3. **控制面高可用**：控制面挂了，数据面的存量调用还能通吗？（通常可以，因为配置下发到了本地）
4. **配置推送机制**：Pilot是如何高效将配置更新推送到成千上万个Sidecar的？（gRPC流式连接、Delta增量配置）

**实战案例**：某跨国电商系统，从 Spring Cloud 迁移至 Istio。主要动机是新接入的 Node.js 推荐服务无法直接复用 Java 的 Hystrix 熔断机制。引入 Istio 后，利用 VirtualService 实现了基于 Header 的灰度发布，仅让 5% 的特定用户流量流向新版本，且无需修改任何业务代码。但初期遇到“Sidecar 启动慢导致 Pod Ready 延迟”的问题，需调整 `readinessProbe` 的探针策略。

## 记忆要点

- 治理位置对比：Spring Cloud治理在应用层(需引入SDK有侵入)，而Service Mesh在基础设施层(Sidecar零侵入)。
- 语言与性能：Spring Cloud仅限Java但进程内调用快，而Service Mesh支持多语言但多一跳网络损耗。
- 核心架构：数据面负责流量劫持与转发，而控制面负责下发路由和安全策略配置。
