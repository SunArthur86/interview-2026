# 微服务监控需要关注哪些指标？如何构建可观测性体系？

🎯 本质：微服务可观测性由三大支柱组成：Metrics(指标)、Logging(日志)、Tracing(链路追踪)。构建体系的核心目标是从“发现问题”到“定位问题”再到“解决问题”。

**三大支柱详解：**

1. **Metrics(指标) - 聚合数据，看趋势**
   - **黄金信号**：
     - **Latency**：延迟（P50, P95, P99 分位值）。
     - **Errors**：错误率（HTTP 5xx, 异常计数）。
     - **Traffic**：流量（QPS, RPS, 带宽）。
     - **Saturation**：饱和度（CPU 使用率, 内存使用率, 磁盘 I/O wait, 线程池队列长度）。
   - **RED 方法**：Rate (请求率), Errors (错误数), Duration (耗时)。
   - **工具**：Prometheus (采集), Grafana (展示)。

2. **Logging(日志) - 离散事件，查细节**
   - **结构化日志**：推荐 JSON 格式，包含 `timestamp`, `level`, `traceId`, `userId`, `serviceName`, `msg`。
   - **日志聚合**：ELK (Elasticsearch + Logstash/Filebeat + Kibana) 或 EFK (Fluentd)。
   - **最佳实践**：避免在循环中打印日志；错误日志必须包含堆栈；敏感信息脱敏。

3. **Tracing(链路追踪) - 请求全生命周期，定边界**
   - **核心概念**：
     - `TraceID`：全局唯一的请求ID。
     - `SpanID`：每个服务/调用的单元ID。
     - `Annotations`：事件标记（如 cs: Client Sent, sr: Server Received）。
   - **穿透流程**：
```text
User Request -> [ Gateway (TraceID: 100) ] 
                  |  (Header: X-Trace-Id=100)
                  v
             [ Service A (SpanID: 1) ] 
                  |  (Header: X-Trace-Id=100, X-Parent-Span=1)
                  v
             [ Service B (SpanID: 2) ] 
                  |  (Header: X-Trace-Id=100, X-Parent-Span=1)
                  v
             [ Service C (SpanID: 3) ]
```
   - **工具**：SkyWalking (国产，APM 全能), Jaeger (Uber), Zipkin (Twitter)。

**可观测性体系架构：**
```text
[微服务节点]
    | Metrics      | Logs         | Traces
    v              v              v
[ Agent (Filebeat/Prometheus Exporter/Otel SDK) ]
    |              |              |
    +--------------+--------------+
                   | (数据清洗/缓冲)
                   v
            [ Kafka / Pulsar ] (可选，用于削峰)
                   |
     +-------------+-------------+
     |             |             |
     v             v             v
[ Prometheus ] [ Elasticsearch ] [ Jaeger/Elasticsearch ]
     |             |             |
     +-------------+-------------+
                   |
                   v
            [ Grafana ] (统一关联)
```

### 💡 实战深化

**1. 实战案例**
某线上接口 P99 延迟突然飙升至 5s。首先在 Grafana 观察到 Metrics 指出下游服务 B 的 RT 突增，然后通过 TraceID 在 SkyWalking 中查询全链路，发现瓶颈在于服务 B 调用 Redis 的一个慢查询（BigKey），最后结合日志确认了具体的 Key 值，定位问题耗时缩短到分钟级。

**2. 代码示例（Java + MDC 链路透传）**
```java
// 日志配置中引用 TraceId (Logback)
<pattern>%d{yyyy-MM-dd HH:mm:ss} [%X{traceId}] [%thread] %-5level %logger{36} - %msg%n</pattern>

// 代码中手动透传（若无框架集成）
String traceId = request.getHeader("X-Trace-Id");
if (traceId == null) traceId = UUID.randomUUID().toString();
MDC.put("traceId", traceId);
try {
    // 业务逻辑
} finally {
    MDC.clear();
}
```

**3. 对比表格：Metrics vs Logging vs Tracing**

| 维度 | Metrics (指标) | Logging (日志) | Tracing (追踪) |
| :--- | :--- | :--- | :--- |
| **数据特征** | 数值、聚合、时序 | 文本、离散事件 | 结构化、层级调用图 |
| **核心作用** | 监控报警、趋势分析 | 故障定界、审计 | 问题定位、依赖分析 |
| **存储成本** | 低 | 高 | 中 |
| **时效性** | 实时 | 近实时 | 实时/准实时 |
| **典型问题** | 系统负载高吗？ | 报错堆栈是什么？ | 请求慢在哪里？ |
