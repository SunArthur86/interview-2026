# 通知型事务

### 通知型事务

通知型事务通过 MQ（消息队列）通知其他事务参与者自己的执行状态。引入 MQ 有效解耦了参与者，支持异步执行，因此也被称为**异步事务**。

#### 特点
*   **异步执行**：各参与者不需要同步等待。
*   **最终一致性**：允许短暂的数据不一致，经过一段时间后达到一致。
*   **适用场景**：对数据实时性要求较低，需异步更新数据的场景（如支付成功后发积分、发优惠券）。

#### 主要分类
1.  **异步确保型事务**：
    *   主要适用于内部系统（如订单和购物车、支付与结算）。内部环境相对可控，通过“本地消息表”或“事务消息”等机制保障消息必达，从而保证最终一致性。

2.  **最大努力通知**：
    *   主要用于跨系统的外部对接（如支付平台与运营商、银行对接）。外部网络环境复杂且不可信，只能“尽最大努力”重试通知，通常有重试次数限制（如 5 次），若超过次数仍未成功，需依赖人工或定期对账接口来查询状态，以实现数据的最终一致性。

#### 异步确保型架构流

```text
+----------+     1.本地事务 & 写消息表      +-------+     2.发消息      +-------+
|  业务方  | --------------------------> | MQ    | ----------------> | 下游 |
| (上游)   | <-------------------------- |       | <---------------- | (消费)|
+----------+     3.定时任务轮询并重试      +-------+                  +-------+
     |
     | 说明：
     | 1. 上游在同一个本地DB事务中写入业务数据和消息状态（待发送）。
     | 2. 独立线程或定时任务扫描消息表，发送消息到MQ。
     | 3. 确认MQ收到ACK后更新状态为“已发送”。
```

#### 💡 实战案例：支付后的积分异步入账
在用户支付成功后，需同步返回支付结果给用户，而积分发放是异步的。我们曾遇到 MQ 消息积压导致用户 10 分钟后仍未收到积分的问题。优化方案是引入独立的“消息发送服务”，本地消息表仅做缓冲，通过独立的 Job 高频轮询并强制将消息推送到 MQ，同时设置告警阈值，保证积分在 1 分钟内到账。

#### 💻 代码示例：本地消息表实现（伪代码）
```java
@Transactional(rollbackFor = Exception.class)
public void processPayment(Order order) {
    // 1. 执行业务逻辑（更新订单状态）
    orderDao.updateStatus(order.getId(), "PAID");
    
    // 2. 在同一事务中写入本地消息表
    LocalMessage msg = new LocalMessage();
    msg.setTopic("points-grant");
    msg.setContent(order.getId()); // 业务ID
    msg.setStatus("SENDING");
    localMessageDao.insert(msg);
    // 事务提交后，DB 和 消息表 同时成功
}

// 独立定时任务逻辑
public void scanAndSend() {
    List<LocalMessage> msgs = localMessageDao.queryByStatus("SENDING", limit=100);
    msgs.forEach(msg -> {
        try {
            mqProducer.send(msg.getTopic(), msg.getContent());
            localMessageDao.updateStatus(msg.getId(), "SENT");
        } catch (Exception e) {
            // 记录失败次数，超过阈值报警
            localMessageDao.increaseRetryCount(msg.getId());
        }
    });
}
```

#### 📊 方案对比：异步确保 vs 最大努力通知

| 维度 | 异步确保型事务 | 最大努力通知 |
| :--- | :--- | :--- |
| **核心机制** | 本地消息表 + 事务 + 定时轮询 | 主动轮询调用 / 回调 + 有限重试 |
| **一致性保障** | 强（利用本地事务保证业务和消息同生共死） | 弱（仅尽力，依赖最终对账兜底） |
| **适用场景** | 微服务内部调用（如订单->积分->物流） | 跨系统对接（如支付->银行/第三方） |
| **时效性** | 高（秒级/分钟级） | 低（分钟级/小时级，甚至T+1） |
| **依赖组件** | 数据库、消息队列（MQ） | HTTP 接口、短信/邮件通知渠道 |

## 常见考点
1.  **消息表与业务表的耦合**：本地消息表通常与业务表在同一个数据库，这是否会影响业务性能？如何做分库分表？
2.  **消息重复消费**：下游如何保证幂等性？（如使用唯一业务ID去重）。
3.  **最大努力通知的停止机制**：最大努力通知在重试多次失败后如何处理？（转为T+1对账或报警）。
4.  **与 RPC 调用的区别**：通知型事务与传统的同步 RPC 调用在处理网络超时时的策略有何不同？


## 记忆要点

- 核心定义：引入MQ解耦，支持异步执行，仅追求最终一致性
- 异步确保型：适用于内部系统，通过本地消息表或事务消息保障必达
- 最大努力通知：适用于外部跨系统对接，有重试上限且需下游主动查询对账
- 架构关键：上下游均需保证幂等性，以防消息重复消费
