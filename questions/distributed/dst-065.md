# Seata的数据隔离性

Seata AT 模式基于本地 ACID 事务（如 MySQL），其全局隔离级别设计如下：

### 1. 写隔离（全局独占锁）
Seata 通过“全局锁”机制实现写隔离，确保数据不会出现脏写。核心流程如下：

1.  **获取本地锁**：分支事务开启本地事务，进行数据修改（此时未提交），获取数据库本地行锁。
2.  **获取全局锁**：本地事务提交前，必须向 TC（事务协调者）申请该记录的全局锁。
3.  **提交本地事务**：拿到全局锁后，提交本地事务，释放本地锁，并释放数据库连接。
4.  **释放全局锁**：当全局事务提交（二阶段）时，释放全局锁。

**互斥机制**：
*   如果两个全局事务修改同一行数据，先拿到全局锁的事务可提交，后者必须重试等待。
*   若全局事务回滚，由于持有全局锁，它能重新获取本地锁进行反向补偿（回滚），而等待中的全局事务会因为拿不到全局锁而超时回滚。

### 2. 读隔离

*   **默认级别（读未提交）**：Seata AT 模式默认的全局隔离级别是 Read Uncommitted。这是出于性能考虑，因为本地数据库事务已提交，数据立即可见，不需要额外的锁开销。

*   **读已提交实现**：如果业务场景要求“读已提交”，Seata 通过代理 `SELECT FOR UPDATE` 语句实现。
    - 代理执行 `SELECT FOR UPDATE` 时，会申请该记录的**全局锁**。
    - 如果全局锁被其他事务持有，则释放本地锁并重试，查询会被阻塞，直到拿到全局锁（即等到前序事务提交）。
    - 这保证了读取到的是已提交的数据。

### 实战案例
在支付系统的流水查询中，曾出现由于未使用 `@GlobalTransactional` 或 `select for update`，导致用户在支付成功（一阶段提交）但事务未完全结束时，立即查询余额看到了已扣款但流水未生成（状态不一致）的情况。对于这种强一致性查询需求，必须显式使用 Seata 代理的 `select for update` 或将查询放入同一个全局事务中。

### 核心代码示例
```java
// 强制读取已提交数据 (Java)
// 必须在 Seata 代理的 DataSource 下执行，会被自动拦截增强
public AccountDTO getAccountForUpdate(Long userId) {
    // Seata 会拦截此语句，先申请全局锁，确保读到的是其他全局事务已提交的数据
    String sql = "SELECT * FROM account WHERE user_id = ? FOR UPDATE";
    return jdbcTemplate.queryForObject(sql, new Object[]{userId}, AccountDTO.class);
}
```

### 架构流程：写隔离与锁竞争

```text
      全局事务 A (Global Lock Holder)          全局事务 B (Competitor)
      ┌──────────────────────────────┐        ┌──────────────────────────────┐
      │ 1. 开启本地 DB 事务           │        │ 1. 开启本地 DB 事务           │
      │ 2. 执行 SQL (获取本地锁)      │        │ 2. 执行 SQL (获取本地锁)      │
      │ 3. 向 TC 申请全局锁           │        │ 3. 向 TC 申请全局锁           │
      │    ────────────> 成功         │        │    ────────────> 失败/等待   │
      │ 4. 提交本地事务 (释放本地锁)   │        │    [重试中...]               │
      │ 5. 释放全局锁 (二阶段提交时)   │        │                             │
      └──────────────────────────────┘        │    等到 A 释放全局锁          │
                                             │    ────────────> 成功         │
                                             │ 4. 提交本地事务               │
                                             └──────────────────────────────┘
```

### 关键细节补充
*   **锁释放时机**：一阶段提交本地事务后，连接释放回连接池，但全局锁由 Session Holding 机制持有，直到二阶段结束。这防止了本地事务提交后，全局事务未提交前，其他事务修改数据。
*   **死锁风险**：若全局事务持有全局锁但无法快速提交（如网络故障），会导致其他全局事务大量阻塞，需配置全局锁超时时间 (`global.lock.retry.internal` 和 `global.lock.retry.times`) 进行快速失败回滚。

### 常见考点
1.  **为什么默认是 Read Uncommitted？**
    - 为了保证高并发性能。如果要做到 Read Committed，所有查询都需要加锁，会严重降低吞吐量。通常业务上能容忍短暂的脏读（因为一阶段已提交），或者通过 `SELECT FOR UPDATE` 在需要时提升隔离级别。
2.  **如何解决脏读问题？**
    - 如果业务不接受脏读，应在 SQL 中使用 `SELECT FOR UPDATE`，Seata 会自动将其升级为全局锁控制，阻塞直到前序全局事务提交。


## 记忆要点

- 写隔离靠全局锁：一阶段本地提交前必须向TC申请记录的全局锁，二阶段决议后才释放。
- 读隔离默认级别：全局读未提交，因为本地事务一阶段提交后数据即对外可见。
- 强一致读：必须使用 `select for update`，Seata会拦截并强制申请全局锁以阻塞等待。
- 防脏写：全局事务回滚时，依赖全局锁拦截其他修改，保障基于前镜像的补偿安全。
