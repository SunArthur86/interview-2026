# Saga和TCC对比

Saga 和 TCC 都是补偿型事务，但实现机制和适用场景有所不同。

### 1. 缺少预留动作
Saga 相比 TCC 的主要缺点是缺少资源预留阶段。
- **TCC 模式**：Try 阶段预留资源（如保存邮件草稿、冻结余额），Confirm 提交（发送、扣款），Cancel 直接释放资源（删除草稿、解冻余额）。实现清晰，隔离性好，能避免脏写。
- **Saga 模式**：直接提交业务操作。若需撤销，必须执行逆向逻辑（Ci，如发送另一封撤销邮件、执行退款转账）。实现较为复杂，且若下游服务链路长，补偿层级会很深。

### 2. 优势
虽然没有预留动作，但这在简单场景下反而是优势：
- **改造简单**：业务逻辑变动小，只需在原逻辑上增加一个补偿接口，而 TCC 需要将业务拆分为 Try, Confirm, Cancel 三个阶段，改造量大，且 Try 阶段逻辑往往难以设计。
- **通信成本低**：TCC 最少通信次数为 2n（每个服务 Try + Confirm/Cancel），Saga 为 n（正向调用 + 异步补偿），网络开销更小。
- **兼容第三方**：对于无法提供 Try 接口的第三方服务，TCC 难以实现，而 Saga 只需封装正常调用和补偿调用即可。
- **异常处理简单**：无需担心资源预留后的超时释放问题（如 TCC 中 Try 阶段挂起导致的资源悬挂问题）。

### 实战案例
在电商促销活动中，调用第三方物流接口发货时无法要求对方支持资源冻结，使用 TCC 改造不可能，此时采用 Saga 模式，正向调用发货，失败时执行逆向的“取消发货”或生成“售后工单”最为稳妥。

### 对比表格：Saga vs TCC
| 维度 | TCC 模式 | Saga 模式 |
| :--- | :--- | :--- |
| **一阶段动作** | 资源预留 | 直接提交业务 |
| **代码侵入性** | 高（需拆分三阶段） | 中（需提供补偿接口） |
| **锁竞争/性能** | 锁定资源直到二阶段，性能一般 | 无长期锁，性能较高 |
| **一致性保障** | 较强（准实时） | 最终一致（可能存在中间状态读） |
| **适用场景** | 强一致性要求高、资源敏感型 | 长流程、第三方服务集成 |

## 常见考点
1. **TCC 的空回滚和悬挂是什么？**
   - 空回滚：Try 未执行，Cancel 先执行了（如网络拥堵导致 Cancel 超时重试），需支持空回滚。
   - 悬挂：Cancel 执行完了，Try 才迟到了，导致资源被错误预留。需要通过增加事务记录或状态控制来防止。

2. ** Saga 的长事务如何实现？**
   通常采用编排（Orchestration）模式，由一个协调器负责按顺序调用各服务，或者采用协同（Choreography）模式，通过事件驱动各服务依次执行。

3. ** Saga 的隔离性如何解决？**
   由于缺乏隔离性，通常在业务层通过“悲观锁”（如版本号）或“顺序执行”来避免并发冲突。
