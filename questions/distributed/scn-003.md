# Quarkus框架是什么？它和Spring Boot有什么区别？

🎯 本质：Quarkus是Red Hat开发的云原生Java框架，专为GraalVM Native Image和Kubernetes环境设计，口号是Supersonic Subatomic Java。

📊 Quarkus vs Spring Boot：
| 特性 | Spring Boot (3.x) | Quarkus |
|------|------------------|---------|
| 启动时间(JVM) | 1-3秒 | 0.5-1秒 |
| 启动时间(Native) | 50-100ms | 10-30ms |
| 内存(JVM) | 200-400MB | 80-150MB |
| 内存(Native) | 50-100MB | 20-50MB |
| Native支持 | AOT引擎(后加) | 原生设计(优先) |
| 编程模型 | 注解+反射 | 注解+构建时处理 |
| 热重载 | Spring DevTools | Dev UI (极快) |
| 生态 | 最丰富 | 较新但专注云原生 |
| 异步模型 | WebFlux (Reactor) | Mutiny (Vert.x) |

**实战案例**：某公司需要开发一个高频交易撮合引擎，对延迟极其敏感。团队对比了Spring Boot Native和Quarkus Native。虽然两者都能提供低延迟，但Quarkus内置的Vert.x事件循环模型在处理高并发I/O时，上下文切换开销更低。最终选择Quarkus，实测吞吐量比Spring Boot Native高出约15%，且在开发过程中，Quarkus的Live Reload在秒级完成，显著提升了开发效率，但在集成旧有的Spring Security组件时遇到了适配困难。

**关键代码示例**：
```java
// Quarkus 使用 Mutiny 响应式编程，非阻塞
@GET
@Path("/prices")
public Uni<List<Price>> getPrices() {
    return Price.<Price>listAll()
        .ifNoItem().after(Duration.ofMillis(500)).fail(); // 内置超时控制
}

// Spring Boot WebFlux (对比)
@GetMapping("/prices")
public Flux<Price> getPrices() {
    return Flux.fromIterable(priceRepository.findAll())
        .timeout(Duration.ofMillis(500));
}
```

Quarkus核心设计原则：
1. 构建时处理 - 尽可能多的工作在构建时完成（注解扫描、Bean发现）
2. 减少反射 - 用Jandex索引替代运行时反射扫描
3. GraalVM优先 - 所有扩展都支持Native Image
4. 响应式优先 - 基于Vert.x和Mutiny

Quarkus Live Coding：
修改Java代码后按Ctrl+S，0.1秒内看到效果。不需要重新编译/重启，因为Quarkus在后台替换了字节码。

选择建议：
- 已有Spring生态+团队熟练 → Spring Boot 3.x
- 新项目+Kubernetes原生+追求极致启动/内存 → Quarkus
- Serverless/FaaS → Quarkus（Native Image性能优势明显）
- 微服务规模大+GraalVM → 两者都适合


## 记忆要点

- 定位对比：Spring Boot 是生态丰富的全能框架，而 Quarkus 是专为云原生和极致启动速度设计的框架。
- 核心机制：Quarkus 采用构建时处理和用Jandex索引替代运行时反射，而 Spring Boot 依赖运行时反射和AOT。
- 性能差异：因为 Quarkus 采用 GraalVM优先和Vert.x事件循环，所以其Native启动时间(毫秒级)和内存远超 Spring。
- 开发体验：Quarkus 提供 0.1秒级的 Live Reload 热重载，体验优于传统框架。
