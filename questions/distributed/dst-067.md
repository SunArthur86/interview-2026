# Seata TCC 模式

### Seata TCC 模式

**TCC (Try-Confirm-Cancel)** 是一种应用层柔性事务解决方案，与 Seata AT 模式同为两阶段提交协议，但实现机制完全不同。

#### 核心区别
1.  **侵入性**：TCC 对业务代码有**严重侵入**。
2.  **自定义实现**：事务的每一个阶段都需要业务开发人员自行编码实现，框架无法自动处理数据操作。

#### 三个阶段详解
1.  **Try 阶段**：
    *   **含义**：资源的检查和预留。
    *   **示例**：扣减余额时，Try 阶段不直接扣钱，而是冻结余额或记录“待扣金额”。
2.  **Confirm 阶段**：
    *   **含义**：确认执行业务逻辑。只要 Try 成功，Confirm 必须能成功。
    *   **示例**：正式扣除冻结的余额，将“待扣”状态更新为“已扣”。
3.  **Cancel 阶段**：
    *   **含义**：取消执行业务逻辑，释放 Try 阶段预留的资源。
    *   **示例**：释放冻结的余额，删除“待扣”记录。

#### 适用场景
*   性能要求高，不依赖数据库本地事务锁，适用于高并发场景。
*   逻辑复杂、无法通过简单的 SQL 补偿回滚的业务（如涉及第三方接口调用）。

### 状态流转：TCC 生命周期

```text
    业务开始       Try (检查/预留)        二阶段决策
       │               │                    │
       v               v                    v
   ┌──────┐      ┌──────────┐        ┌──────────┐
   │ 初始 │ ──>  │ 中间状态 │        │          │
   │ 状态 │      │ (资源预留)│        │          │
   └──────┘      └──────────┘        │          │
       │               │             │          │
       │               │ 全部成功    │ 部分失败  │
       │               v             v          v
       │          ┌──────────┐  ┌──────────┐ ┌──────────┐
       └──────────>│ Confirm  │  │ Cancel   │ │  结束    │
                  │ (确认提交)│  │ (取消回滚)│ └──────────┘

                       │             │
                       v             v
                  最终提交状态    最终回滚状态
```

#### 深化：实战与防悬挂

**实战案例**：
在电商秒杀场景中，库存服务使用 TCC 模式。曾遇到因 Try 阶段网络超时导致 TC 判定失败从而触发 Cancel，但随后 Try 请求到达导致库存被重复扣减（即“悬挂”问题）。**解决方案**：在 Try 阶段前插入一事务控制记录，Cancel 时先检查该记录，若 Try 未执行则记录“已 Cancel”状态，Try 执行时若发现状态为“已 Cancel”则直接拒绝执行。

**代码示例**：
```java
// 使用 Seata @LocalTCC 注解定义 TCC 模式
@LocalTCC
public interface StockService {
    
    // Try 阶段：指定两阶段方法名
    @TwoPhaseBusinessAction(name = "prepareDeduct", commitMethod = "commit", rollbackMethod = "rollback")
    boolean prepareDeduct(@BusinessActionContextParameter(paramName = "productId") String productId, int count);

    // Confirm 阶段
    boolean commit(BusinessActionContext context);

    // Cancel 阶段
    boolean rollback(BusinessActionContext context);
}
```

**AT 与 TCC 选型对比**：

| 维度 | AT 模式 | TCC 模式 |
| :--- | :--- | :--- |
| **侵入性** | 低（无侵入） | 高（需改造业务接口） |
| **锁机制** | 全局行锁（DB 锁粒度），并发度较低 | 业务自定义锁（如余额冻结），并发度高 |
| **开发成本** | 低 | 高（需处理幂等、悬挂、空回滚） |
| **性能** | 中等（依赖本地事务和锁竞争） | 高（不依赖 DB 锁，适用于核心链路） |
| **适用** | 80% 的普通业务 | 高并发、核心资产、跨系统非 DB 操作 |

## 记忆要点

- 定义对比：TCC是业务层柔性事务，侵入性高需手写接口，而AT模式低侵入自动生成补偿。
- 三阶段：Try（检查并预留资源）、Confirm（确认执行）、Cancel（取消释放）。
- 适用场景：高并发核心资产、跨系统非DB操作，对性能要求极高。
- 三大痛点：需业务自行解决幂等性、空回滚和资源悬挂问题。
