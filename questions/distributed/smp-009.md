# 微服务如何实现优雅停机？Spring Boot中有哪些最佳实践？

🎯 本质：优雅停机确保应用关闭时，已完成正在处理的请求，拒绝新请求，释放资源，再安全退出，避免服务中断导致的数据不一致或报错。

**优雅停机全流程架构图：**
```text
[K8s/容器环境]          [Spring Boot App]           [外部依赖]
      |                        |                        |
  (1) 发送 SIGTERM ---------> |                        |
      |                        |                        |
      |                 (2) 停止注册新实例            |
      |                        | (从 Nacos/Eureka 下线) |
      |                        |                        |
      |                 (3) 停止接收新流量            |
      |              (Web容器停止监听端口)            |
      |                        |                        |
      |                 (4) 处理存量请求              |
      |              (等待 active threads 结束)      |
      |                        |           <---------->|
      |                        |                        |
      |                 (5) 释放资源                  |
      |         (关闭 DB连接池, MQ消费端, Cache)      |
      |                        |                        |
  (6) 等待/强制Kill --------> |                        |
      |                   (Exit)                      |
```

**详细步骤与最佳实践：**

1. **触发信号**
   - 容器环境：K8s 发送 `SIGTERM`。
   - 虚拟机：`kill -15 <pid>`。

2. **下线服务（关键点）**
   - 调用 `context.close()` 或 `start()` 停止。
   - **Spring Boot Actuator** 触发 `/actuator/shutdown`（需开启）。
   - 必须先从注册中心剔除，防止负载均衡器继续转发流量。

3. **停止接收新请求**
   - Spring Boot 2.3+ 配置：`server.shutdown=graceful`。
   - 容器（Tomcat/Undertow）停止接收新连接，但保持现有连接。

4. **处理存量请求**
   - 配置最大宽限期：`spring.lifecycle.timeout-per-shutdown-phase=30s`。
   - 30秒后仍未完成的请求会被强制中断（取决于Web容器实现）。

5. **资源释放**
   - **数据库**：连接池正常关闭（HikariCP 会等待连接归还）。
   - **消息队列**：停止消费者（`@KafkaListener` 或 `SimpleMessageListenerContainer`），确保 `ack` 已提交。
   - **定时任务**：停止调度器（如 `@Scheduled`, Quartz），防止任务在中途执行。

6. **K8s 配合（重要）**
   - `terminationGracePeriodSeconds`: 设置大于应用停机时间（如 45s）。
   - **preStop Hook**: 必须配置 `sleep 10` (甚至更久)。
     - **原因**：K8s 从 Service/Iptables/IPVS 摘除 Pod 和 Spring Boot 实际停止监听端口之间存在时间差（网络收敛、Endpoint 刷新延迟）。不 sleep 会导致流量在摘除前已发往 Pod，但 Pod 已停机，报错 502。

### 💡 实战深化

**1. 实战案例**
某电商大促期间进行应用滚动发布，由于未配置优雅停机，K8s 发送 SIGTERM 后 Spring Boot 立即退出，导致正在处理的支付请求被强制中断，数据库出现“支付中”但无后续日志的脏数据，且消费者未 Ack 导致消息重复消费。引入优雅停机后，彻底解决了此类数据一致性问题。

**2. 代码示例**
```yaml
# application.yml
server:
  shutdown: graceful # 开启优雅停机
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s # 宽限期30秒

management:
  endpoints:
    web:
      exposure:
        include: shutdown
  endpoint:
    shutdown:
      enabled: true
```

```java
// 确保资源正确销毁（例如 RocketMQ 消费者）
@PreDestroy
public void destroy() {
    log.info("开始关闭 RocketMQ 消费者...");
    if (consumer != null) {
        consumer.shutdown();
    }
}
```

**3. 对比表格：立即停机 vs 优雅停机**

| 维度 | 立即停机 | 优雅停机 |
| :--- | :--- | :--- |
| **处理方式** | 立即杀掉进程 | 等待请求完成、资源释放 |
| **用户体验** | 报错 | 无感（或等待） |
| **数据安全** | 低（易丢失或脏写） | 高（保证事务完整性） |
| **关闭速度** | 快 | 慢（取决于超时配置） |
| **Spring配置** | 默认 | `server.shutdown=graceful` |
