# 在分布式事务中，Saga模式与TCC模式的核心区别是什么？Saga模式下如何处理子事务失败时的补偿逻辑？

TCC是预留资源的强一致性应用层2PC，开发成本高；Saga是将长事务拆为本地短事务，提交即释放锁，失败时按反向顺序补偿。Saga的补偿逻辑要求操作幂等，且需捕获单点异常以防止回滚受阻，适合长流程但存在脏读风险。

## 技术原理

- **TCC 分 Try/Confirm/Cancel 三阶段，强依赖资源预留，锁持有时长；Saga 分步提交，锁在本地事务结束即释放**：TCC 是业务层的两阶段提交——Try 阶段冻结资源（如预扣库存、冻结金额）但不真正变更；全局决策后 Confirm 真正提交，或 Cancel 释放冻结。Try 冻结期间资源被锁定（别人不可用），所以一致性高但持锁久。Saga 没有预留阶段，每个子事务就是普通本地事务，提交即释放锁，靠失败时的补偿事务撤销已提交的影响。
- **TCC 无中间状态可见；Saga 允许中间状态，存在脏读风险，补偿必须幂等**：TCC 的 Try 状态对其他事务不可见（业务自定义隔离），所以无脏读。Saga 的每个子事务一提交就对其他事务可见，订单已创建但支付未完成时外部能看到中间状态，存在脏读和"幻读"风险。补偿操作（如退款、回滚库存）可能因网络重试被执行多次，必须设计成幂等。
- **TCC 适合短事务强一致性；Saga 适合长事务最终一致性，异常处理需防止单点失败阻断回滚**：TCC 锁持有久，事务越长越拖垮吞吐，只适合短事务。Saga 没有持锁适合长流程（如旅行预订涉及多个外部系统），但补偿链任一节点失败会阻断整个回滚，需要重试机制和"补偿失败告警 + 人工兜底"策略。

## 代码示例

TCC 三阶段接口定义（Seata TCC 模式）：

```java
@LocalTCC
public interface OrderTccAction {
    @TwoPhaseBusinessAction(name = "createOrder",
        commitMethod = "confirm", rollbackMethod = "cancel")
    boolean tryCreate(BusinessActionContext ctx, Order order);

    boolean confirm(BusinessActionContext ctx);   // Confirm：订单从"冻结"转"有效"
    boolean cancel(BusinessActionContext ctx);    // Cancel：删除冻结订单
}

// 业务调用
@GlobalTransactional
public void placeOrder(Order order) {
    orderTccAction.tryCreate(null, order);   // Try：预创建，资源被冻结
    // 全局事务结束自动调用 confirm 或 cancel
}
```

Saga 补偿编排（状态机或代码编排）：

```java
// Saga：订机票 → 订酒店 → 租车，任一失败按反向补偿
SagaDSL.define()
    .step("bookFlight",   this::bookFlight,   this::cancelFlight)     // 正向/补偿
    .step("bookHotel",    this::bookHotel,    this::cancelHotel)
    .step("rentCar",      this::rentCar,      this::cancelCar)
    .execute(state);

// 补偿必须幂等（用唯一请求ID + 状态机）
public void cancelFlight(String sagaId, String bookingId) {
    if (cancellationDao.alreadyCancelled(sagaId, bookingId)) return;   // 幂等
    flightApi.cancel(bookingId);
    cancellationDao.markCancelled(sagaId, bookingId);
}
```

## 对比/选型

| 维度 | TCC | Saga |
|------|-----|------|
| 一致性 | 强（资源预留隔离） | 最终（直接提交） |
| 隔离性 | 高（中间态不可见） | 低（中间态可见，脏读风险） |
| 开发成本 | 高（每步写三接口） | 中（每步写补偿） |
| 锁持有 | 长（Try 到 Confirm/Cancel） | 短（本地事务结束即释放） |
| 适合事务长度 | 短事务 | 长流程事务 |
| 典型场景 | 库存扣减、资金转账 | 旅行预订、跨企业流程 |

## 常见坑/注意事项

- **Saga 补偿必须幂等**：网络重试、Saga 重启都可能让补偿多次执行，必须用唯一事务 ID + 状态表（`processed` 标志）做幂等，否则可能多扣一次库存。
- **Saga 补偿不是简单的"反操作"**：有些操作不可逆（如已发送的邮件、已发货的快递），补偿可能是语义上的（发道歉邮件、拦截快递），需要业务层面设计"逻辑补偿"而非"数据回滚"。
- **Saga 的脏读隔离**：业务上要容忍中间状态，或用"前置校验 + 占位"等手段缓解，否则用户看到订单已建却未支付会困惑。
- **Saga 防回滚阻断**：补偿链任一节点失败不能让整个回滚卡死，要有重试 + 死信告警 + 人工兜底，否则系统卡在中间态。
- **TCC 的资源冻结设计难**：Try 阶段怎么"冻结"需要业务配合（预扣字段、冻结金额表），不是所有资源都能干净地预留，设计不当会让 Confirm/Cancel 复杂度爆炸。

1. **TCC分Try/** — TCC分Try/Confirm/Cancel三阶段，强依赖资源预留，锁持有时长；Saga分步提交，锁在本地事务结束即释放
