# Seata 分布式事务的 AT 模式是如何实现“无侵入”的？在一阶段本地事务提交后，如果二阶段全局回滚，它是利用什么机制自动生成回滚 SQL 的？

Seata AT 模式的无侵入性源于对 JDBC 数据源的代理，业务代码使用普通 `@Transactional` 即可。一阶段本地事务提交时，Seata 拦截 SQL，记录修改前后的“前镜像”和“后镜像”生成回滚日志，随业务数据一并提交，保证原子性。二阶段全局回滚时，Seata 利用日志中的“前镜像”和当前数据的“后镜像”对比：若无脏写直接生成反向 SQL；若有脏写则结合全局锁校验，生成补偿 SQL 将数据还原至前镜像状态，实现自动回滚。

## 技术原理

- **数据源代理实现无侵入**：Seata AT 模式通过 `DataSourceProxy` 包装业务的 `DataSource`，所有 JDBC 操作都被代理拦截。业务代码继续用 `@Transactional` 写本地事务，完全感知不到 Seata 的存在——这就是"无侵入"的本质：侵入点在数据源层而非业务代码层。
- **一阶段记录前后镜像构建回滚日志**：执行业务 SQL 前，代理先查询"前镜像"（修改前的数据），执行 SQL 后再查询"后镜像"（修改后的数据），把前后镜像 + SQL 元信息写入 `undo_log` 表，与业务数据在**同一个本地事务**中提交。这样要么业务和回滚日志都成功，都失败，保证原子性。
- **二阶段根据镜像自动生成补偿 SQL**：全局提交时只需异步删除 `undo_log`；全局回滚时取出前镜像，与当前数据比对——若一致（无脏写）直接用前镜像生成反向 UPDATE 还原；若不一致（被其他事务改过）则结合全局锁校验后做补偿。补偿 SQL 由 Seata 自动生成，业务无需手写。

## 代码示例

启用 AT 模式只需三步（无业务代码改动）：

```java
// 1. 业务照常写本地事务，加 @GlobalTransactional 标记全局事务
@GlobalTransactional(timeoutMills = 60000)
public void placeOrder(Order order) {
    orderDao.insert(order);          // 本地事务1：订单库
    storageDao.deduct(order.getProductId(), order.getCount());  // 本地事务2：库存库
    accountDao.debit(order.getUserId(), order.getAmount());    // 本地事务3：账户库
}
// 任一本地事务失败 → TC 协调全局回滚 → 各分支用 undo_log 自动生成反向 SQL 还原
```

每张业务表配套的 `undo_log` 表结构（Seata 自动维护）：

```sql
CREATE TABLE undo_log (
    branch_id     BIGINT      NOT NULL COMMENT '分支事务ID',
    xid           VARCHAR(128) NOT NULL COMMENT '全局事务ID',
    context       VARCHAR(128) NOT NULL,
    rollback_info LONGBLOB    NOT NULL COMMENT '前后镜像JSON',
    log_status    INT         NOT NULL,
    log_created   DATETIME    NOT NULL,
    log_modified  DATETIME    NOT NULL,
    UNIQUE KEY ux_undo_log (xid, branch_id)
);
```

## 对比/选型

| 模式 | 侵入性 | 一致性 | 性能 | 适用场景 |
|------|--------|--------|------|----------|
| AT | 无（数据源代理） | 最终一致（读已提交隔离） | 中（有镜像开销） | 通用、快速接入 |
| TCC | 高（写 Try/Confirm/Cancel） | 强一致 | 高 | 短事务、需强隔离 |
| Saga | 中（写补偿逻辑） | 最终一致 | 高 | 长流程事务 |
| XA | 无 | 强一致 | 低（长锁） | 跨同质 DB |

## 常见坑/注意事项

- **AT 模式隔离级别是读已提交**：一阶段本地事务已提交，其他事务能看到中间状态（脏读风险）。若需强隔离要在 SQL 后加 `FOR UPDATE` 走全局锁。
- **必须每张业务表有主键**：镜像比对和反向 SQL 生成依赖主键定位行，无主键会生成错误的补偿 SQL。
- **undo_log 表不能忘建**：每张业务库都要建 `undo_log` 表，否则回滚时找不到日志导致数据无法还原。
- **脏写场景的兜底**：若回滚时发现数据已被其他全局事务改过，Seata 会抛 `BranchRollbackFailed_Retriable`，需人工介入或靠事务补偿表兜底，不能完全自动恢复。
- **性能开销在镜像查询**：每条 UPDATE 前后各查一次，写密集场景开销明显，高并发热点表慎用。
