# 什么是Docker多阶段构建？如何优化Java应用的Docker镜像？

🎯 本质：多阶段构建允许在一个Dockerfile中使用多个FROM指令，最终镜像只包含最后一个阶段的内容，大幅减小镜像体积并优化构建流程。

**1. 多阶段构建原理**
利用构建中间态，丢弃不需要的工具（如Maven、JDK），只保留运行时必需文件。

```dockerfile
# Stage 1: 构建
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /app
COPY pom.xml .
# 技巧：先利用Docker缓存下载依赖，避免代码变动导致重新下载
RUN mvn dependency:go-offline 
COPY src ./src
RUN mvn package -DskipTests

# Stage 2: 运行
FROM eclipse-temurin:17-jre-alpine
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

**构建架构流程图**：
┌───────────────────────────────────────────────────────┐
│ Stage 1: Builder (Maven + JDK)                        │
│ 1. COPY src/pom                                       │
│ 2. RUN mvn package (Build Fat Jar)                    │
│ Output: /app/target/app.jar (包含所有依赖)            │
└─────────────────────┬─────────────────────────────────┘
                      │ COPY --from=builder
                      ↓ 只提取 Jar 文件
┌───────────────────────────────────────────────────────┐
│ Stage 2: Runtime (JRE ONLY)                           │
│ 1. COPY app.jar                                       │
│ 2. ENTRYPOINT java -jar                               │
│ Final Image Size: ~200MB (丢弃了Maven/JDK ~500MB)     │
└───────────────────────────────────────────────────────┘

**2. 使用Jib（Google）**
# 无需Docker守护进程，直接构建
mvn compile jib:dockerBuild
# 优势：自动分层、自动分析依赖、无需写Dockerfile、更安全。

**3. 安全优化：Distroless基础镜像**
Distroless 镜像只包含应用及其运行时依赖，不包含 Shell、包管理器等。
```dockerfile
FROM gcr.io/distroless/java17-debian12
COPY app.jar /app.jar
ENTRYPOINT ["java", "-jar", "/app.jar"]
```
# 对比 Alpine:
# - Alpine: 有包管理器和Shell，容易导致漏洞，但调试方便。
# - Distroless: 极致安全，无Shell，无法 docker exec -it bash 进入调试。

**4. 极致优化：Native Image（GraalVM）**
将Java代码编译成本地二进制文件，无需JVM。
```dockerfile
FROM oracle/graalvm-native-image:22 AS builder
...编译...
FROM alpine:latest
COPY --from=builder /app/app /app
ENTRYPOINT ["/app"]
# 最终镜像可小至20MB，启动毫秒级。
```

**镜像优化清单**：
1. **基础镜像**：优先使用 `slim` 或 `alpine`，甚至 `distroless`。
2. **层数控制**：合并RUN指令（如 `RUN apk add --no-cache curl && rm -rf /var/cache/apk/*`）。
3. **多阶段**：构建环境与运行环境分离。
4. **缓存利用**：变动频率低的指令（COPY pom.xml）放在前面。

**基础镜像选型对比**：
| 特性 | Debian (OpenJDK) | Alpine (glibc) | Distroless |
| :--- | :--- | :--- | :--- |
| **体积** | 大 (~400MB+) | 小 (~200MB) | 极小 (~180MB) |
| **启动速度** | 一般 | 较快 | 快 |
| **兼容性** | 完美 | 较差(JVM需适配) | 完美(仅运行时) |
| **调试性** | 高(有bash) | 高(有bash) | 极低(无Shell) |
| **安全性** | 一般 | 一般 | 高(攻击面小) |

**实战案例**：在构建使用 `mvn package` 的 Spring Boot 应用时，若网络环境导致依赖下载极慢，可利用“构建缓存阶段”技巧：先 COPY `pom.xml` 并执行 `RUN mvn dependency:resolve`，这样在修改代码但不改依赖时，Docker 会命中缓存直接跳过耗时的下载步骤。

## 记忆要点

- 多阶段本质：构建阶段包含Maven/JDK，而运行阶段只保留JRE和最终JAR，大幅缩减镜像体积。
- 依赖缓存优化：先单独COPY pom.xml下载依赖，再COPY src代码，避免代码变动重新下依赖。
- 镜像选型对比：Alpine体积小但有兼容坑，而Distroless极致安全无Shell攻击面小。
