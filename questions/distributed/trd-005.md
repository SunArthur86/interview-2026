# 在分布式事务场景中，Seata 的 AT 模式与 TCC 模式在实现机制和锁粒度上有何不同？AT 模式是如何做到对业务无侵入的？

Seata AT 模式基于“增强型本地事务”，利用本地数据库的 ACID 特性，通过解析 SQL 拦截前后镜像生成 undo log，写入本地事务并提交，依靠全局锁保证隔离性，最终通过回滚日志反向补偿。它对业务代码完全无侵入。TCC 模式则要求业务接口拆分为 Try、Confirm、Cancel 三个方法，开发者需要手动在 Try 阶段预留资源，所有锁和资源状态由业务逻辑控制，锁粒度完全自定义。TCC 的性能极限通常高于 AT（无全局锁争用），但开发成本极高，且必须处理空回滚、悬挂等异常情况。

### 增强细节：AT 模式的无侵入实现原理
AT 模式的“无侵入”主要依赖于 **DataSource 代理**。
1.  **代理注入**：Seata 在启动阶段，会自动将 Spring 容器中原生的 `DataSource` 替换为 `DataSourceProxy`。
2.  **SQL 解析**：业务代码执行 SQL 时，实际调用的是代理对象。代理器利用 Druid 或其他解析器解析 SQL 语法，识别出涉及的表及主键。
3.  **镜像生成**：在执行业务 SQL 前，查询当前数据记录作为“前置镜像”；执行业务 SQL 后，再次查询作为“后置镜像”。
4.  **Undo Log 与全局锁**：将前后镜像以及 SQL 信息组装成 `Undo Log`，与本条业务 SQL 在同一个本地数据库事务中提交。在提交本地事务前，Seata 会向 TC（事务协调者）申请该记录的**全局锁**。如果拿到锁，则提交本地事务并释放数据库本地锁；如果拿不到（说明别的事务正在修改），则回滚本地事务并重试。

### 增强细节：TCC 模式的锁粒度与状态控制
TCC 模式将锁的控制权完全交给业务层。
*   **锁粒度**：AT 模式锁定的是数据库记录行（行锁），粒度固定；而 TCC 模式锁定的是业务资源，例如账户余额、库存数量，粒度可以是数据库行，也可以是缓存中的 Key，甚至是一段代码逻辑，极其灵活。
*   **状态控制**：AT 模式依赖数据库中间状态；TCC 模式依赖业务字段（如 `freeze_amount` 冻结金额）来标记中间状态。开发者必须保证 `Try` 阶段资源预留成功，`Confirm` 阶段提交资源，`Cancel` 阶段释放预留资源。

### 架构流程对比

```text
+-------------------+         +-------------------+         +-------------------+
|      AT 模式      |         |      TCC 模式     |         |    全局协调 TC    |
+-------------------+         +-------------------+         +-------------------+
| 1. 开启本地事务   |         | 1. Try: 预留资源  |         |                   |
| 2. 查询前置镜像   |         |    (写入业务表)   |         |                   |
| 3. 执行业务 SQL   |         |                   |         |                   |
| 4. 查询后置镜像   |         |                   |         |                   |
| 5. 写入 Undo Log  |         |                   |         |                   |
| 6. 申请全局锁 --> | --------|                   |<--------| 注册/上报分支     |
| 7. 提交本地事务   |         |                   |         |                   |
| 8. 释放 DB 锁     |         |                   |         |                   |
+-------------------+         +-------------------+         +-------------------+

(回滚时)
| 读取 Undo Log     |         | Cancel: 释放预留  |         | 下发回滚命令     |
| 反向生成 SQL      |         |    (更新业务表)   |         |                   |
```

### 实战案例
在高并发库存扣减场景中，AT 模式因需争用数据库行锁和全局锁，吞吐量遭遇瓶颈，且 `Undo Log` 表产生大量行锁竞争。迁移至 TCC 模式后，将资源预校验前置到 Redis 缓存中（Try 阶段），DB 仅在 Confirm 阶段执行最终扣减，显著减少了 DB 持锁时间，QPS 提升了 3 倍。

### 模式对比

| 维度 | Seata AT 模式 | Seata TCC 模式 |
| :--- | :--- | :--- |
| **侵入性** | 低（仅依赖注解和 DataSource 代理） | 高（需编写 Try/Confirm/Cancel 三个接口） |
| **锁粒度** | 行锁（数据库级别） | 业务资源锁（可精细化到缓存 Key） |
| **性能** | 中等（有全局锁开销，回滚需反解析 SQL） | 高（无全局锁，直接操作业务状态） |
| **适用场景** | 基于简单 SQL 的 CRUD 业务，开发效率优先 | 核心高频业务，对性能和一致性要求极高 |
| **数据一致性** | 最终一致性（弱隔离性，可能脏读） | 最终一致性（强隔离性，防悬挂/空回滚需自行处理） |

## 记忆要点

- 侵入性对比：AT基于DataSource代理解析SQL生成undo_log，业务零侵入；TCC需手写Try/Confirm/Cancel三接口
- 锁粒度对比：AT依赖数据库行锁加全局锁（粒度固定）；TCC锁定业务资源如缓存Key（粒度自定义）
- 性能选型：AT有全局锁开销适合普通业务；TCC无全局锁性能极高，适合高并发资产核心系统
