# 详解：三个阶段

### 详解：3PC（三阶段提交）

3PC（Three-Phase Commit）是 2PC 的改进版，将二阶段提交的第一个阶段拆分成了两步，增加了超时机制，分为 CanCommit、PreCommit 和 DoCommit 三个阶段。

#### 阶段一：CanCommit (询问)
1.  **事务询问**：协调者向所有参与者发送包含事务内容的 CanCommit 请求，询问是否可以执行事务（**此时不执行事务，也不锁资源**）；
2.  **反馈询问**：参与者认为可以顺利执行则返回 Yes，否则返回 No。

#### 阶段二：PreCommit (预提交)
协调者根据阶段一的反馈决定操作：

**情形一：执行事务预提交**
1.  **发送预提交请求**：协调者向所有节点发出 PreCommit 请求，进入 Prepared 阶段；
2.  **事务预提交**：参与者执行事务操作，将 Undo 和 Redo 日志写入本机事务日志（**此时锁定资源**）；
3.  **反馈响应**：成功执行后发送 Ack 给协调者，等待最终指令。

**情形二：中断事务**
若任一参与者返回 No 或超时：
1.  **发送中断请求**：协调者发送 Abort 请求；
2.  **中断事务**：参与者收到 Abort 或等待超时后，中断事务。

#### 阶段三：DoCommit (最终提交)
真正进行事务提交的阶段：

**情形一：执行提交**
1.  **发送提交请求**：协调者收到所有 Ack 后，发送 DoCommit 请求；
2.  **事务提交**：参与者正式提交，释放资源；
3.  **反馈结果**：发送 Ack 消息；
4.  **完成事务**：协调者接收所有 Ack 后，完成事务。

**情形二：中断事务**
若协调者收到 No 或超时，发送 Abort 请求，参与者执行回滚并反馈 Ack，最终中断事务。

#### 3PC 状态转换图
```text
       协调者                  参与者
          │                       │
    CanCommit (询问)             │
          │──────────────────────>│
          │<──────────────────────│ Yes/No
          │                       │
   (有No/超时?) ──No──> Abort     │
          │                       │
          ▼ (全Yes)               │
    PreCommit (预提交)            │
          │──────────────────────>│ (执行事务, 写日志, 锁资源)
          │<──────────────────────│ Ack
          │                       │
   (有No/超时?) ──No──> Abort     │
          │                       │
          ▼ (全Ack)                │
    DoCommit (提交)               │
          │──────────────────────>│
          │<──────────────────────│ Ack (完成)
```

#### 关键改进点
*   **超时机制**：在 3PC 中，如果参与者等待 PreCommit 或 DoCommit 阶段超时，它不会无限期阻塞，而是默认执行提交（针对 PreCommit 超时）或中断（视具体实现和阶段而定，通常 PreCommit 超时倾向于提交，因为大概率大家都准备好了）。
*   **减少锁持有时间**：在 CanCommit 阶段不锁资源，只有在 PreCommit 阶段才锁资源，减少了资源无效锁定的时间。

#### 实战案例
在早期的分布式数据库研发中，曾尝试引入 3PC 来解决 2PC 的阻塞问题。但在实测中发现，一旦发生 PreCommit 阶段的网络分区，部分节点因超时自动提交，而协调者实际上决定回滚，导致出现了极难修复的数据分裂，最终不得不回退方案。

#### 代码示例（伪代码逻辑）
```javanclass Participant {
    // 3PC 参与者状态机
    void onPreCommitTimeout() {
        // 关键改进：超时不再阻塞，而是根据状态推断
        if (state == State.PRE_COMMITTED) {
            // 既然已经预提交，大概率其他节点也成功了，自动提交
            doCommit(); 
        } else {
            abort();
        }
    }
}
```

## 常见考点
1.  **3PC 相比 2PC 到底解决了什么？**
    *   主要是解决了单点故障导致的**参与者阻塞**问题。引入超时后，即使协调者挂了，参与者也可以自动决策（提交或回滚），避免永久死锁。但它**不能**保证数据一致性的绝对安全。
2.  **3PC 为什么在实际中很少用？**
    *   虽然减少了阻塞，但引入了额外的网络通信（多一轮 RTT），在高并发下性能反而更差。且在极端网络分区下，依然会出现数据不一致，因此业界更倾向于使用 TCC 或 Saga 等补偿方案。
