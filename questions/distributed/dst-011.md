# CAP 权衡

### 为什么分布式系统中无法同时保证一致性和可用性？

**答**：对于分布式系统而言，分区容错性（P）是基本要求，因此只能在一致性（C）和可用性（A）之间取舍。

- **如果保证 C**：当节点 N1 写数据时，N2 的读写操作必须暂停，直到 N1 同步数据完成。在此期间，N2 无法响应请求，违背了 A。
- **如果保证 A**：N2 必须随时响应请求，即使 N1 正在写数据且未同步。此时读取 N2 可能读到旧数据，违背了 C。

### CAP 权衡

1. **CP 架构**：
   - **选择**：保证 C 和 P，舍弃 A。
   - **场景**：涉及钱财等对一致性要求极高的场景（如银行）。网络故障时宁可停止服务。

2. **AP 架构**：
   - **选择**：保证 A 和 P，舍弃 C（强一致），转而追求最终一致性。
   - **场景**：大多数大型互联网应用。主机众多、故障是常态，为保证高可用（N 个 9），允许短暂的数据不一致，追求最终一致。

3. **CA 架构**：
   - **选择**：保证 C 和 A，舍弃 P。
   - **场景**：实际上放弃了分布式，属于单机系统（如传统单机数据库）。

### 深化内容

#### 实战案例
在设计秒杀系统扣减库存时，我们曾尝试使用 Redis（AP 倾向）做实时扣减。但在极端网络情况下，Redis 主从同步延迟导致“超卖”。后来将库存扣减核心逻辑改为使用 Lua 脚本强制在 Redis 单节点原子执行（牺牲部分可用性换强一致），或者引入 CP 模型的中间件（如 etcd）进行库存预占，确保不会因为网络分区导致数据偏差。

#### 代码示例
以下是一个简单的 CP 模型（模拟两阶段提交）的伪代码，展示一致性是如何通过“阻塞”来实现的：

```java
// CP 模型：必须等待所有节点确认
public boolean writeData(String key, String value) {
    boolean node1Success = node1.write(key, value);
    if (!node1Success) return false; // 任意节点失败，操作失败

    boolean node2Success = node2.write(key, value);
    if (!node2Success) {
        node1.rollback(key); // 回滚 N1
        return false; 
    }
    return true; // 只有全部成功才算成功
}
```

#### 对比表格

| 维度 | CP（一致性与分区容错） | AP（可用性与分区容错） | CA（一致性与可用性） |
| :--- | :--- | :--- | :--- |
| **P (分区)** | **保留** | **保留** | **舍弃**（非分布式） |
| **网络故障时** | 停止服务，拒绝请求 | 继续服务，可能返回旧数据 | 假定无网络故障 |
| **数据状态** | 强一致，实时准确 | 弱一致，最终一致 | 单点数据一致 |
| **性能** | 低（延迟高，需同步） | 高（响应快，异步） | 中（受限于单机性能） |
| **代表技术** | ZooKeeper, HDFS, Consul (CP) | Cassandra, DynamoDB, Eureka | 单机 RDBMS (MySQL/Oracle) |

## 记忆要点

- 为何只能保一个：N1同步数据时若锁N2则违背A，若不锁N2则读到旧数据违背C。
- CP模型：通过同步阻塞实现，任一节点失败则全部回滚，性能低但强一致。
- AP模型：多节点异步复制，高并发高可用，通过版本号/时间戳解决写冲突达到最终一致。
- CA模型仅存在于无网络分区的单机或绝对可靠的局域网（传统RDBMS）。
