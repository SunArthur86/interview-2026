# Sleuth 或 SkyWalking 等全链路追踪系统中，Trace ID 是如何在多线程（如线程池）或异步调用（如 CompletableFuture）中传递的？

在 Java 中，全链路追踪上下文存储在 `ThreadLocal` 中，但线程切换会导致上下文丢失。解决方案是利用阿里巴巴开源的 `TransmittableThreadLocal` (TTL) 或框架提供的装饰器机制。TTL 会在任务提交前捕获主线程上下文，并在子线程执行时进行装饰。对于 `CompletableFuture`，SkyWalking 等插件会代理其回调方法，确保将 Trace ID 注入当前线程。RPC 调用时，Trace ID 通过 HTTP Header 或 RPC Metadata 透传给下游服务，从而实现跨线程、跨进程的链路串联。

## 技术原理

- **ThreadLocal 在异步/线程池中因线程切换会丢失上下文**：`ThreadLocal` 的存储是"以线程为 key"的 Map，子线程是新线程没有继承父线程的 ThreadLocalMap；线程池更严重——工作线程被复用，上一个任务残留的上下文会污染当前任务。所以 Sleuth/SkyWalking 直接用 ThreadLocal 存 Trace ID 在线程池/异步场景必然断链。
- **利用 TransmittableThreadLocal (TTL) 或框架装饰器进行上下文搬运**：阿里 `TransmittableThreadLocal` 配合 `TtlRunnable.get(runnable, ttl)` 装饰器，在任务**提交时**捕获主线程当前的所有 TTL 值快照，在子线程执行前 `beforeExecute` 注入、执行后 `afterExecute` 还原。SkyWalking/Sleuth 的 TraceContext 普遍基于 TTL 或类似机制实现跨线程透传。
- **CompletableFuture 通过代理回调注入上下文**：CompletableFuture 的 `thenApply`/`thenAccept` 等回调可能在任意线程执行，Sleuth 用字节码增强（ByteBuddy）或手动 `wrap` 把每个 lambda 包一层"捕获-注入-还原"逻辑；Django/Spring 也提供 `MDC` 的异步装饰器。
- **RPC 调用时将 Trace ID 放入 Header 或 Metadata 透传**：跨进程时 Trace 上下文无法用 ThreadLocal 共享，必须在出站请求的 HTTP Header（如 `X-B3-TraceId`）或 RPC 元数据（gRPC metadata、Dubbo Attachment）里携带，下游在入站拦截器里解析并重建 ThreadLocal/MDC。

## 代码示例

使用 TTL 跨线程池透传 Trace 上下文：

```java
// 1. 用 TransmittableThreadLocal 替代 ThreadLocal 存 TraceId
TransmittableThreadLocal<String> traceTtl = new TransmittableThreadLocal<>();
traceTtl.set("trace-abc-123");

// 2. 用 TtlExecutors 装饰线程池，所有提交的 Runnable 自动捕获上下文
ExecutorService pool = TtlExecutors.getTtlExecutorService(
    Executors.newFixedThreadPool(8));

pool.submit(() -> {
    // 子线程里能直接读到 trace-abc-123，无需手动传递
    log.info("traceId={}", traceTtl.get());
});
```

CompletableFuture 场景用 Sleuth 的 `LazyTraceExecutor` 装饰：

```java
// Spring Cloud Sleuth 自动包装，CompletableFuture 回调链不断链
CompletableFuture.supplyAsync(() -> {
    log.info("traceId={}", MDC.get("traceId"));   // 正常打印
    return queryOrder();
}, lazyTraceExecutor).thenApply(order -> {
    log.info("traceId={}", MDC.get("traceId"));   // 回调线程也能拿到
    return buildView(order);
});
```

跨进程透传（HTTP Header）：

```java
// 出站：把 TraceId 塞进 Header
Request req = Request.builder()
    .header("X-B3-TraceId", traceTtl.get())
    .header("X-B3-SpanId",  generateSpanId())
    .build();

// 入站（下游拦截器）：从 Header 解析并重建上下文
String traceId = request.header("X-B3-TraceId");
traceTtl.set(traceId);
MDC.put("traceId", traceId);
```

## 对比/选型

| 透传方式 | 适用场景 | 代价 |
|----------|----------|------|
| ThreadLocal | 单线程同步 | 无法跨线程 |
| InheritableThreadLocal | `new Thread()` 创建子线程 | 线程池复用时污染 |
| TransmittableThreadLocal | 线程池/异步任务 | 需装饰 Runnable/Executor |
| Header/Metadata | 跨进程 RPC | 序列化开销 |

## 常见坑/注意事项

- **InheritableThreadLocal 在线程池失效**：它只在 `new Thread()` 时继承一次，线程池线程被复用，继承的是首次创建时主线程的上下文，后续任务都拿到陈旧值。必须用 TTL。
- **@Async 默认不透传**：Spring `@Async` 用默认线程池不透传 MDC，要自定义 `TaskDecorator` 包装。
- **Header 注入别忘了采样标志**：透传时除了 TraceId 还要带 `X-B3-Sampled`，否则下游不采样导致链路断点。
- **Reactor/WebFlux 需 Hooks**：响应式编程没有 ThreadLocal 概念，要用 `Micrometer Tracing` 的 `Hooks.onEachOperator` 把上下文放进 Reactor Context。


## 记忆要点

- 根本痛点：原生ThreadLocal无法跨线程，子线程和线程池中极易丢失上下文。
- 核心方案：使用阿里TransmittableThreadLocal(TTL)，在任务提交前捕获并装饰到子线程。
- 异步处理：对于CompletableFuture，框架会代理其回调方法自动完成上下文注入。
- 跨进程透传：RPC或HTTP调用时，需将TraceId塞入请求Header或Metadata透传给下游。
