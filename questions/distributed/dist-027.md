# 什么是两阶段提交（2PC）协议？有什么缺点？

两阶段提交协议（2PC, Two-Phase Commit）是一种强一致性的分布式事务算法，旨在保证所有分布式节点要么全部提交，要么全部回滚。它是 X/Open 组织提出的 XA 规范的核心实现。

### 执行流程
分为两个阶段：**准备阶段** 和 **提交阶段**。

**角色：**
- **协调者**：负责发起和调度事务。
- **参与者**：负责执行本地事务。

#### 阶段一：准备阶段
1. 协调者向所有参与者发送“准备”请求。
2. 参与者执行事务操作，写入 **Redo Log**（重做日志）和 **Undo Log**（回滚日志），但不提交。
3. 参与者向协调者反馈“同意”或“拒绝”。

#### 阶段二：提交阶段
- **情况 A（全部同意）**：
  1. 协调者向所有参与者发送“提交”请求。
  2. 参与者执行真正的提交操作，释放锁资源，并反馈“完成”。
- **情况 B（任一拒绝或超时）**：
  1. 协调者向所有参与者发送“回滚”请求。
  2. 参与者利用阶段一记录的 **Undo Log** 进行回滚，释放锁资源，并反馈“完成”。

### 实战案例
在某支付系统中使用 MySQL 的 XA 事务处理跨库转账。遇到一次网络抖动，协调者发送 Commit 指令后，网络分区导致部分库未收到指令，从而一直持有行锁不释放。由于当时没有配置锁超时机制，导致相关账户的所有后续转账操作都被阻塞，产生严重雪崩。

### 缺点详解
1. **同步阻塞**：
   - 所有参与者在事务未完成前一直处于阻塞状态，占用数据库连接和锁资源，在高并发下系统吞吐量极低。

2. **单点故障**：
   - 如果协调者在阶段二发送提交/回滚命令前宕机，参与者将一直处于锁定状态，不知道是提交还是回滚，导致**死锁**。
   - 解决方案需要引入超时机制或复杂的恢复逻辑。

3. **数据不一致**：
   - 协调者发送提交请求时，若部分参与者收到提交而部分未收到（网络分区或故障），会导致部分提交、部分未提交的数据不一致。

4. **状态不确定**：
   - 协调者宕机后，新协调者恢复时无法确切知道上一步是否所有节点都同意了，可能需要遍历所有参与者查询状态（成本高）。

### MySQL XA 代码示例
```sql
-- 1. 开启 XA 事务
XA START 'xid_01';
UPDATE user_account SET balance = balance - 100 WHERE user_id = 1;
XA END 'xid_01';

-- 2. 准备阶段（参与者在此时持久化 Redo/Undo 日志，但不释放锁）
XA PREPARE 'xid_01';

-- 3. 提交或回滚（由协调者决定）
-- XA COMMIT 'xid_01';
-- XA ROLLBACK 'xid_01';
```

## 常见考点
1. **三阶段提交（3PC）是如何改进 2PC 的？**：3PC 引入了 **CanCommit**（预询问）和 **PreCommit**（预提交）以及 **Timeout** 机制。主要改进点：一是在 PreCommit 阶段如果协调者挂了，参与者默认会 Abort（避免阻塞）；二是增加了超时机制，让参与者可以主动决定状态。但 3PC 无法解决网络分区导致的不一致问题，且性能更差，实际应用较少。
2. **2PC 在数据库中是如何落地的？**：对应 XA 协议。MySQL 的 XA 事务使用 `XA START`, `XA END`, `XA PREPARE`, `XA COMMIT/ROLLBACK` 命令。通常使用 `ResourceManager`（数据库）和 `TransactionManager`（应用服务器或中间件）来实现。
3. **为什么 Seata 的 AT 模式不需要 2PC 这种强锁？**：Seata AT 模式利用 **Undo Log** 和 **全局锁**，在第一阶段先提交本地事务（释放本地锁），第二阶段通过异步回滚或反向补偿，实现了“最终一致”而非“强一致”，从而避免了

## 记忆要点

- 两阶段指：一阶段准备（写日志锁资源不提交），二阶段提交（全部同意则Commit，否则Rollback）
- 缺点一：因为所有参与者等待协调者指令，所以导致严重同步阻塞，并发极低
- 缺点二：协调者在阶段二宕机会导致参与者死锁，引发单点故障与状态不确定
- 缺点三：网络分区会导致部分节点收到Commit而部分未收到，引发数据不一致
