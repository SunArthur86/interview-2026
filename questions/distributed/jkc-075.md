# 消息广播：Leader服务器将数据发送给其他服务器

消息广播是ZAB协议在集群进入正常运行阶段后的核心流程，其采用**两阶段提交（2PC）**的变种实现高吞吐的一致性广播。

**核心流程与关键细节**：
1. **提案广播**：Leader接收到客户端请求后，为其生成全局单调递增的Zxid（Zookeeper Transaction Id），封装为Proposal提议，通过TCP长连接按顺序发送给所有Follower。
2. **Follower处理**：Follower收到Proposal后，先将其以事务日志形式写入本地磁盘（通过`forceSync`参数控制是否刷盘），然后返回ACK给Leader。ACK中包含当前Follower最新的已接收Zxid。
3. **过半提交**：Leader收到过半Follower的ACK后，认为该事务达成共识。Leader广播Commit消息，自身提交该事务并返回客户端成功响应。
4. **Follower提交**：Follower收到Commit消息后，将之前写入本地日志的事务应用到内存数据树中。

**异常处理边界**：
- Leader崩溃会触发Leader选举，新Leader保证拥有所有已提交提案（通过Zxid最大且Epoch最新原则）。
- 若仅收到部分ACK未过半，Leader会持续重发Proposal，直到过半确认或超时切换状态。

**流水线优化**：为提高吞吐，Leader不需要等待上一个Proposal完全提交，即可发送下一个Proposal。

```text
   客户端       Leader               Follower1              Follower2
     |            |                      |                      |
     | --- 写请求 -->|                      |                      |
     |            |--- Proposal(Zxid) --->|                      |
     |            |                      | [写Log]               |
     |            |                      |--- ACK -------------->|
     |            |--- Proposal(Zxid) --------------------------->|
     |            |                                           [写Log]|
     |            |<--- ACK ----------------------------------|
     | (收到过半ACK)|                      |                      |
     |            |--- Commit ---------->|                      |
     |            |                      | [应用内存]            |
     |            |--- Commit --------------------------------->|
     |<-- 成功 ---|                                           [应用内存]
```

#### 1. 实战案例
**坑点**：Zookeeper 默认事务日志写入磁盘是异步刷盘，若机房断电可能导致已 Commit 但未刷盘的数据丢失。在金融对账场景中，必须将 `forceSync`（`zoo.cfg` 中的 `syncEnabled`）设为 true，牺牲性能换取数据安全。

#### 2. ZAB 协议阶段对比

| 阶段 | 状态 | 核心动作 | 目的 |
| :--- | :--- | :--- | :--- |
| **恢复阶段** | 选举/恢复 | Leader 选举 + 数据同步 | 保证集群数据一致，确立 Leader |
| **广播阶段** | 广播 | 两阶段提交 | 处理客户端写请求，保持数据副本一致 |

#### 3. 关键流程代码示意

```java
// Follower 处理逻辑伪代码
public void processProposal(Proposal proposal) {
    // 1. 写入事务日志
    txnLog.append(proposal.getHeader().getZxid(), proposal.getRequest());
    
    // 2. 发送 ACK 给 Leader (FollowerAck)
    leader.ack(proposal.getZxid());

    // 3. 等待 Leader 的 Commit 指令
    waitForCommit(proposal.getZxid(), () -> {
        // 4. 提交到内存数据库
        memoryDataTree.commit(proposal.getRequest());
    });
}
```

## 常见考点
1. **Zxid的构成**：解释Zxid由高32位的Epoch（纪元，代表Leader变更次数）和低32位的Counter组成，保证了全局有序。
2. **消息队列**：Leader与Follower之间通过消息队列解耦，发送和接收异步处理，避免阻塞。
3. **对比Raft**：ZAB的消息广播类似Raft的Log Replication，但Raft通常要求日志连续性检查（Term/Index匹配），ZAB在广播阶段假设Leader合法，主要依赖Follower的ACK确认。
4. **Follower滞后**：如果Follower落后太多，Leader会将其剔除出广播队列，转入数据恢复模式（快照+截断日志）。


## 记忆要点

- 核心机制：ZAB 广播阶段采用 2PC 变种，Leader 发 Proposal，Follower 落盘后回 ACK。
- 过半提交：Leader 收到过半（>N/2）ACK 后广播 Commit，自身提交并响应客户端。
- 顺序保障：全局单调递增的 Zxid（高32位 Epoch + 低32位 Counter）保证消息顺序一致。
- 性能优化：Leader 采用流水线模式，不必等待上一个事务 Commit 即可发送下一个 Proposal。
