# Seata AT模式的使用前提

Seata AT 模式（Automatic Transaction Mode）是增强型的二阶段提交协议，旨在保持强一致性的语义下提升性能。

### 核心特点
- **无锁设计**：不同于 XA 在一阶段 Prepare 后一直持有数据库锁，AT 模式在一阶段提交本地事务后立即释放数据库锁和连接。
- **自动化补偿**：通过 JDBC 拦截 SQL 自动解析，生成回滚日志，开发者无需手动编写补偿逻辑。

### 使用前提
1. **数据库支持**：
   - 必须基于支持本地 ACID 事务的关系型数据库。
   - **关键限制**：例如 MySQL 必须使用 **InnoDB** 引擎。MyISAM 引擎不支持事务，因此无法使用 Seata AT 模式。
2. **应用环境**：
   - Java 应用（JDK 1.8+）。
   - 必须通过 **JDBC** 访问数据库。因为 Seata 需要对数据源进行代理，通过拦截 JDBC 语句来解析 SQL 并生成前后镜像。如果是 ORM 框架（如 MyBatis/JPA）最终也是基于 JDBC 实现。
3. **表结构支持**：
   - 业务表必须包含主键（用于定位行记录）。
   - 禁止使用外键级联（因为跨表操作在分布式事务中难以原子性回滚）。
   - 业务表中不能存在类型为 Text/Blob 的大字段（可能会导致 Undo Log 存储溢出）。

### 模式流程（两阶段提交演变）
- **一阶段**：
  - 解析 SQL，查询“前镜像”。
  - 执行业务 SQL。
  - 查询“后镜像”。
  - 生成行锁（用于全局写隔离）。
  - 生成 Undo Log（包含前后镜像）。
  - 提交本地事务（业务数据 + Undo Log + 行锁），**释放本地数据库锁**。
- **二阶段**：
  - **提交**：异步删除 Undo Log，快速释放资源。
  - **回滚**：通过一阶段的回滚日志生成反向 SQL（补偿），并校验数据是否被脏写。

### 实战案例
在开发财务对账系统时，曾遇到过将表字段类型从 `VARCHAR` 改为 `TEXT` 导致 AT 模式回滚失败的问题。原因是 `undo_log` 表的序列化存储超限，导致二阶段回滚时反序列化异常，最终被迫回滚字段修改或改用 Seata TCC 模式。

### 常见考点
1. **AT 模式是否真的完全无锁？**：不是。AT 模式引入了“全局锁”的概念。在本地事务提交前，需要向 TC 申请全局锁，如果拿不到全局锁（说明其他全局事务正在修改），会一直重试，直到拿锁或超时回滚。
2. **支持哪些数据库？**：目前官方支持 MySQL、Oracle、PostgreSQL、SQL Server 等，但需要依赖 Undo Log 表的建表脚本支持。
3. **性能瓶颈在哪里？**：虽然一阶段释放了 DB 锁，但解析 SQL、生成镜像、全局锁的获取与网络通信仍会带来性能损耗，不适合高并发且对一致性要求极高的场景（如秒杀）。

## 技术原理

Seata AT 模式的前提约束不是随便定的，每一条都对应着底层实现机制：

- **为什么必须是 InnoDB（ACID）**：AT 模式依赖"业务 SQL + undo_log 写入"在**同一个本地事务**里原子提交。只有支持 ACID 的引擎才能保证"要么业务和 undo_log 都生效、要么都回滚"。MyISAM 没有事务，业务 SQL 成功但 undo_log 写失败时无法回滚，二阶段就无法补偿。
- **为什么必须通过 JDBC**：Seata 通过 `DataSourceProxy` 代理 JDBC 的 `PreparedStatement`，在 SQL 真正执行前拦截并解析（用 Druid 解析器），查询前后镜像。MyBatis/JPA 底层都是 JDBC，所以能被代理；但如果是原生 NoSQL 客户端（MongoDB driver）或非 JDBC 协议，Seata 拦截不到，无法生成 undo_log。
- **为什么必须有主键**：查询前后镜像、生成反向补偿 SQL、加全局行锁，都依赖主键定位行。没有主键时 Seata 无法确定要回滚哪一行，会直接抛异常拒绝执行。
- **为什么禁止外键级联**：外键级联会触发隐式的跨表修改，Seata 解析 SQL 时只能看到显式 SQL 涉及的表，级联修改的表不会记录 undo_log，二阶段回滚时这些隐式修改无法补偿，导致数据不一致。
- **为什么禁止 Text/Blob 大字段**：undo_log 表把前后镜像序列化成 JSON 存到 LONGBLOB。Text/Blob 字段内容可能很大（如几 MB 的文章正文），整行镜像膨胀导致 undo_log 写入慢、存储爆、反序列化超时。曾出现把 VARCHAR 改 TEXT 后回滚反序列化 OOM 的真实故障。

## 注意事项

1. **引擎必须是 InnoDB**：建表时显式 `ENGINE=InnoDB`，迁移老库时排查是否混用 MyISAM，混用会导致 AT 模式失效。
2. **每个业务库都要建 undo_log 表**：Seata 官方提供建表 SQL，字段固定（branch_id/xid/rollback_info 等），漏建会导致一阶段提交失败。
3. **主键必须有且禁外键**：业务表设计阶段就要去掉外键级联（改用应用层保证一致性），所有表加主键，否则 Seata 启动即报错。
4. **大字段单独拆表**：Text/Blob 字段拆到独立的扩展表，主业务表只存指针，避免 undo_log 膨胀。若必须用大字段，考虑改 TCC 模式手写补偿。

## 代码示例

```sql
-- 1. 每个业务库必须建 undo_log 表（AT 模式前提）
CREATE TABLE undo_log (
    branch_id     BIGINT      NOT NULL COMMENT '分支事务ID',
    xid           VARCHAR(128) NOT NULL COMMENT '全局事务ID',
    context       VARCHAR(128) NOT NULL,
    rollback_info LONGBLOB    NOT NULL COMMENT '前后镜像序列化',
    log_status    INT         NOT NULL,
    log_created   DATETIME    NOT NULL,
    log_modified  DATETIME    NOT NULL,
    UNIQUE KEY ux_undo_log (xid, branch_id)
) ENGINE=InnoDB;  -- 注意：undo_log 表本身也要 InnoDB

-- 2. 业务表必须有主键，禁止外键（AT 模式要求）
CREATE TABLE t_order (
    id          BIGINT PRIMARY KEY,    -- 必须有主键
    user_id     BIGINT NOT NULL,
    amount      DECIMAL(10,2),
    status      VARCHAR(20)
    -- 不要写 FOREIGN KEY (user_id) REFERENCES t_user(id) -- 禁止外键级联
) ENGINE=InnoDB;
```

```yaml
# application.yml：AT 模式数据源代理配置
seata:
  enabled: true
  data-source-proxy-mode: AT     # AT 模式，自动代理 DataSource
  tx-service-group: my_tx_group
  service:
    vgroup-mapping:
      my_tx_group: default
  client:
    rm:
      undo:
        data-validation: true    # 回滚前校验后镜像，防脏写
        log-serialization: jackson
```
