# TCC事务案例

在交易场景中，如果交易未完成，不能让客户看到余额减少，此时 TCC 是一种解决方案。TCC 允许在 Try 阶段冻结资源，而不立即影响用户可见余额。

**业务流程示例**：
1. **订单服务**：创建订单，状态为“处理中”。
2. **现金服务**：
   - **Try**：冻结客户现金（资源预留，用户总资产不变，但可用余额减少，冻结金额增加）。
3. **订单服务**：提交订单数据（如物流信息录入）。
4. **现金服务**：
   - **Confirm**：扣除客户冻结的现金（实际扣款，将冻结金额转为支出）。

**异常流程**：
- 若步骤3失败（如订单入库失败），订单服务发起全局回滚。
- 调用 **现金服务**：
   - **Cancel**：解冻/撤销冻结的金额，恢复用户可用余额。

**数据流图**：
```text
  用户
   │
   ▼
┌──────────────┐      Try       ┌──────────────┐
│  订单服务    │───────────────>│  账户服务    │
│ (创建订单)   │<───────────────│ (冻结余额)   │
└──────┬───────┘                 └──────┬───────┘
       │                               │
   Try Success                          │
       │                               │
   [其他业务逻辑]                       │
       │                               │
   全局提交/回滚                        │
       │                               │
       ▼                               ▼
┌──────────────┐  Confirm/Cancel ┌──────────────┐
│  订单服务    │────────────────>│  账户服务    │
│ (更新状态)   │                 │ (扣款/解冻)  │
└──────────────┘                 └──────────────┘
```

## 常见考点
1. **资源预留粒度**：在库存场景中，Try 阶段是扣减真实库存还是预占库存？如何防止超卖？
2. **用户体验**：Try 阶段冻结资源后，前端如何展示用户余额（展示总额 vs 可用额）？
3. **并发冲突**：高并发下，多个 Try 请求同时冻结同一资源时的处理策略。

### 深化补充

**实战案例**：
在电商大促场景中，曾出现 Try 阶段预占库存成功，但下游支付网关超时，导致 Cancel 回滚库存。由于用户端请求重试，造成了同一用户的重复下单，虽然最终库存一致，但产生了大量无效订单流水。需在 Try 阶段增加基于 RequestID 的幂等校验，防止重复扣减。

**代码示例**：
```java
// Try 阶段：冻结资金 (Java)
@Transactional
public boolean tryFreeze(String userId, BigDecimal amount) {
    // 1. 幂等性检查：防止重复 Try
    if (transactionLogDao.exists(userId, amount)) return true;
    
    // 2. 资源检查与预留 (悲观锁防并发)
    int rows = accountDao.freezeBalance(userId, amount, 
        "UPDATE account SET frozen = frozen + ? WHERE user_id = ? AND balance >= ?");
    
    if (rows > 0) {
        // 3. 记录事务日志，用于后续 Confirm/Cancel 判断
        transactionLogDao.insertLog(userId, amount, "TRYING");
        return true;
    }
    return false; // 余额不足
}
```


## 记忆要点

- 业务拆分：用户不可见余额直接扣减，Try仅冻结资金，Confirm才真扣。
- 回滚策略：若业务链路失败，执行Cancel解冻资金，恢复用户可用余额。
- 防超卖：高并发Try需配合乐观锁或唯一索引，防止多次重试导致重复冻结。
- 数据展示：Try冻结后，前端展示总额不变但需同步降低用户的可用余额。
