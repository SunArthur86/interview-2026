# 什么是URL散列？

### URL 散列
**定义**：通过管理客户端请求 URL 信息（或 URI）的散列值，将发送至相同 URL 的请求转发至同一台后端服务器的负载均衡算法。

#### 原理
算法通常基于一致性哈希或简单的取模运算：

1.  **提取特征**：提取请求的 URL（如 `http://api.com/user/1`）。
2.  **计算哈希**：对 URL 字符串进行 Hash 计算（如 MD5, CRC32）。
3.  **映射**：`Hash(URL) % Server_Count` （普通取模） 或映射到哈希环（一致性哈希）。

```text
URL -> Hash Function -> Hash Value -> Mapping Algorithm -> Server Index
```

#### 应用场景
1.  **服务端缓存友好**：
    *   如果应用服务器在本地（堆内或本地磁盘）维护了热点数据的缓存。
    *   URL 散列能保证针对同一个资源（如 `/product/detail?id=100`）的请求总是打到同一台机器。
    *   **效果**：极大地提高了本地缓存的命中率，减少了回源（查库或查 Redis）的次数。
2.  **特定状态维护**：
    *   某些非标准的 Web 应用可能会将状态与 URL 绑定，而非仅与 Session ID 或 Client IP 绑定。
    *   例如，文件上传/下载服务，可能需要确保同一资源的读写落在同一节点以利用本地文件系统的一致性。

#### 对比分析
*   **对比 IP 散列**：
    *   **IP 散列**：保证来自 **同一客户端** 的所有请求打在同一台服务器。适用于 **Session 保持**。缺点是如果某 IP 访问量极大，会导致单机负载过高。
    *   **URL 散列**：保证对 **同一资源** 的请求打在同一台服务器。适用于 **缓存命中**。缺点是如果某个 URL 成为热点（如微博热搜），会导致单机过载（热点倾斜问题）。
*   **解决热点倾斜**：
    *   URL 散列通常结合 **一致性哈希** 使用，并在哈希环上引入 **虚拟节点**。这样当某台服务器负载过高时，可以通过调整虚拟节点权重或动态迁移部分哈希范围来缓解，比简单取模更灵活。

#### 实战案例
在某电商大促活动中，商品详情页使用了 Nginx 的 URL Hash 策略，目的是为了让后端 Java 进程的本地缓存（Guava Cache）命中率达到 100%。然而，当某款商品被推上“秒杀”首页时，该特定 URL 的流量瞬间激增，导致哈希映射对应的后端单机 CPU 飙升至 100%。这暴露了 URL Hash 在应对极端热点时的脆弱性，最终切回轮询并改用分布式 Redis 缓存解决。

#### 关键代码片段 (Nginx配置)
```nginx
upstream backend {
    hash $request_uri consistent; # 一致性哈希，基于 URL
    server 192.168.1.1:8080;
    server 192.168.1.2:8080;
    server 192.168.1.3:8080;
}

server {
    location /api {
        proxy_pass http://backend;
    }
}
```

#### 负载均衡算法对比表

| 算法 | 粒度 | 优点 | 缺点 | 典型应用 |
| :--- | :--- | :--- | :--- | :--- |
| **Round Robin** | 连接 | 分配均匀 | 无状态，无缓存亲和 | 通用 Web 服务 |
| **IP Hash** | 客户端 IP | 会话保持 | 单IP流量过大导致倾斜 | 需要Session粘滞 |
| **URL Hash** | 请求 URL | 缓存亲和，后端命中率极高 | 热点资源导致单机瓶颈 | CDN回源，静态资源服务 |
| **Least Connections** | 连接数 | 动态感知负载 | 需实时维护连接数 | 长连接服务，数据库代理 |

## 常见考点
1.  **URL 散列的主要用途是什么？**（答：提高后端服务器的本地缓存命中率）
2.  **URL 散列和 IP Hash 散列的区别？**（答：前者是基于请求内容分发，后者是基于请求来源分发）
3.  **URL 散列有什么潜在风险？**（答：热点 URL 问题，即大量请求访问同一个 URL 导致单机过载）

## 记忆要点

- 一句话定义：对请求的 URL 进行 Hash 计算并映射，将同一 URL 固定分发至同一 RS
- 核心优势：因为相同资源请求总落同节点，所以能极致提升本地（如 Guava）缓存命中率
- 副作用对比：IP Hash 保证「同一客户端」粘滞，而 URL Hash 保证「同一资源」粘滞
- 致命缺点：若遇突发热点 URL（如秒杀），会导致 Hash 规则失效，流量倾斜至单机引发过载
