# CAP理论中分区容忍性（P）的含义是什么？

CAP
CAP 原则又称 CAP 定理，指的是在一个分布式系统中，Consistency（一致性）、Availability（可用性）、Partition tolerance（分区容错性），三者不可得兼，最多只能同时满足两项。

**一致性**：
在分布式系统中的所有数据备份，在同一时刻是否同样的值。（等同于所有节点访问同一份最新的数据副本）。强一致性要求更新操作成功并返回给客户端后，所有节点在同一时间的数据完全一致。

**可用性**：
在集群中一部分节点故障后，集群整体是否还能响应客户端的读写请求。（对数据更新具备高可用性）。即无论是否出现故障，每个请求都能在“有限时间内”得到“非错误”的响应，但不保证数据是最新的。

**分区容忍性**：
以实际效果而言，分区相当于对通信的时限要求。系统如果不能在时限内达成数据一致性，就意味着发生了分区的情况（P）。在分布式系统中，节点之间通过网络通信，网络是不可靠的，可能会出现丢包、延迟甚至断网（分区）。P 指的是系统在遇到任意网络分区故障时，仍然能够对外提供满足一致性和可用性的服务（或者在其中做权衡）。

- 在分布式系统中，P 是客观存在的物理现象（网络必分区），因此分布式系统通常必须在 CP 和 AP 之间做选择。

### 实战案例
在跨机房部署的分布式缓存中，曾因光缆被挖断导致网络分区。由于系统设计为 CP 模式，为了防止脏写，主节点拒绝写入，导致整个分片不可用，业务中断；后续切换为 AP 模式，允许备节点接收读请求，虽然读取了旧数据，但保障了业务基本可用。

### 架构选型对比
| 模式 | 选择策略 | 代价 | 典型场景 |
| :--- | :--- | :--- | :--- |
| **CP** | 保证 C，牺牲 A | 分区时拒绝服务/超时 | 金融转账、库存扣减、ZooKeeper、HBase |
| **AP** | 保证 A，牺牲 C | 分区时返回脏数据（最终一致） | 社交动态、商品浏览、DNS、Cassandra、Eureka |
| **CA** | 放弃 P | 不支持分布式（仅限单机） | 传统单机数据库 RDBMS |

## 常见考点
1. **CAP 的取舍场景**：
   - **CP**（放弃 A）：当发生分区时，为了保证一致性，系统拒绝请求或阻塞等待，导致暂时不可用。例如：ZooKeeper、分布式数据库（如传统 MySQL 集群强同步模式）、金融系统。
   - **AP**（放弃 C）：当发生分区时，为了保证可用性，系统接受读写请求，但可能返回旧数据（最终一致性）。例如：DNS、Cassandra、Eureka（注册中心优先保证服务可用）。
   - **CA**：在分布式系统中几乎不存在（除非单机），因为必须具备 P 才能称之为分布式系统。
2. **BASE 理论**：为了解决 CAP 中 C 与 A 的矛盾，提出了 BASE（Basically Available 基本可用、Soft state 软状态、Eventually consistent 最终一致性），它是 CAP 中 AP 的延伸，通过牺牲强一致性来获得高可用性，并保证数据“最终”一致。

## 记忆要点

- 概念定义：P 指网络发生分区（丢包/断连）时，系统仍能对外提供容忍与服务。
- 客观前提：因为分布式网络必会故障，所以 P 是必然存在的物理现象。
- 核心取舍：因为 P 必然发生，所以分布式系统只能在 CP（保一致）和 AP（保可用）中二选一。
- BASE 理论：AP 的延伸，通过牺牲强一致性换取基本可用和最终一致性。
