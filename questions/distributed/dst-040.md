# 消息发送一致性

### 消息发送一致性定义
消息发送一致性是指：产生消息的业务动作与消息发送动作必须保持一致。即**业务操作成功，则消息必须发送成功；业务操作失败，则消息不能发送**。这是一种基于可靠消息的最终一致性方案的前提。

### 常规 MQ 流程的缺陷
常规的 MQ 处理流程通常是：
1.  业务操作
2.  发送消息

由于存在网络不确定性，如果步骤 2 失败或超时，业务系统无法判断 MQ 是否已接收消息，从而导致消息丢失或业务与消息状态不一致。

### 解决方案：半消息与本地消息表
为了解决上述问题，主要采用以下两种机制保障一致性：

#### 1. 事务消息（半消息机制）
以 RocketMQ 为例，核心流程如下：
-   **发送半消息**：消息发送到 MQ，但此时对消费者不可见。
-   **执行本地事务**：发送方执行业务逻辑（如数据库操作）。
-   **提交/回滚**：根据本地事务结果，向 MQ 发送 Commit（消息可见）或 Rollback（删除消息）。
-   **回查机制**：若 MQ 未收到确认（如宕机），会主动回查发送方的事务状态。

#### 2. 本地消息表
-   **原子写入**：在同一个本地数据库事务中，同时写入业务数据和消息记录（状态为“待发送”）。
-   **可靠投递**：使用独立的定时任务或业务逻辑末端，扫描“待发送”的消息记录，发送到 MQ。
-   **状态更新**：MQ 确认收到后，更新消息记录状态为“已发送”。

### 实战案例
在电商大促场景下，若使用本地消息表方案，定时任务轮询间隔设置不当（如设置过短为 1秒），会瞬间对 DB 造成巨大的读压力（类似风暴），甚至拖垮主业务流程。我们后来采用“指数退避”策略重试，才将 DB CPU 压力降下来。

### 方案对比

| 维度 | 事务消息 (RocketMQ) | 本地消息表 |
| :--- | :--- | :--- |
| **开发成本** | 中（需实现回查接口） | 低（逻辑简单直观） |
| **性能损耗** | 较低（异步解耦） | 较高（占用 DB 连接和 IO） |
| **依赖限制** | 强依赖特定 MQ | 任意 MQ 及关系型数据库 |
| **时效性** | 高（毫秒级） | 低（取决于轮询间隔） |

### 代码示例（幂等性校验）
```java
public void handleOrderMessage(Message msg) {
    String uniqueId = msg.getKey(); // 唯一业务流水号
    // 1. 利用 Redis setnx 实现幂等
    Boolean success = redisTemplate.opsForValue().setIfAbsent("idempotent:" + uniqueId, "1", 24, TimeUnit.HOURS);
    if (!success) {
        log.info("重复消费，直接返回: {}", uniqueId);
        return;
    }
    
    // 2. 执行业务逻辑
    doBusiness(msg);
}
```

### 消息重复发送问题
即使解决了发送一致性，由于网络波动或 MQ 自身的机制（如 At Least Once 语义），下游消费者可能会收到重复的消息。
-   **原因**：消费者成功处理消息后，在向 MQ 发送确认（ACK）时网络断开，MQ 以为未消费成功，从而再次投递。

### 业务接口幂等性设计
为了应对消息重复，业务接口必须设计为**幂等**，即：**无论同一个请求被调用多少次，其结果都是一样的**。

**常见幂等实现方案**：
1.  **唯一 ID + 数据库唯一索引**：利用业务流水号作为唯一键，插入数据库时利用唯一索引约束，重复请求会报错或忽略。
2.  **Token 机制**：请求前获取 Token，执行时删除 Token，重复请求因 Token 不存在而拒绝。
3.  **乐观锁（版本号）**：更新数据时带上版本号，只有版本号匹配才更新成功。
4.  **去重表**：独立维护一张去重表，记录处理过的消息 ID。

### 常见考点
1.  **为什么要幂等？**：分布式系统中“网络不可靠”是常态，消息重复投递是 MQ 的特性之一，而非 Bug，业务层必须通过幂等来兜底。
2.  **Redis 分布式锁能实现幂等吗？**：可以，通过 `setnx` 设置唯一标识的锁，请求到来先判断锁是否存在。注意锁的过期时间设置和锁的释放逻辑。
3.  **本地消息表和事务消息怎么选？**：对性能要求不高且代码侵入可控可用本地消息表；对高性能、低耦合要求高则用 RocketMQ 事务消息。
4.  **如何保证消息的顺序性？**：这是另一个话题，但常被追问。通常需要通过单一分区、全局有序 ID 或消费者内存缓冲来实现，与幂等性设计配合使用。
