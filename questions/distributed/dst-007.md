# 分布式环境的事务复杂性

### 隔离级别补充

- **读已提交（READ COMMITTED）**：Oracle 默认级别。
- **可重复读（REPEATABLE READ）**：MySQL 默认级别，通过 `for update` 等锁机制可防止幻读。
- **可串行化（SERIALIZABLE）**：最高隔离级别，强制串行执行。

### 分布式环境的事务复杂性

当本地事务扩展到分布式时，复杂性主要来源于以下两点：

#### 1. 存储端的多样性

- **本地事务**：所有数据落在同一个数据库（DB）中，由 DB 统一管理。
- **分布式事务**：数据可能分散在多个不同的 DB 中，或者同时涉及 Redis、MQ 等异构存储系统。不同系统对事务的支持能力不同（例如 Redis 事务较弱），增加了协调难度。

#### 2. 事务链路的延展性

- **本地事务**：通常封装在一个服务的单一方法中，应用层直接控制事务边界。
- **分布式事务**：一个业务请求被拆分为多个微服务，服务间通过网络调用（RPC）形成线状或网状的链路。网络的不稳定性、服务的独立性使得跨服务的事务协调变得非常困难。

### 实战案例
在微服务架构中，常见的“深坑”是部分成功：用户支付成功后，上游支付服务回调通知下游服务更新订单状态，如果此时网络超时，支付服务不知道下游是否收到了请求，若盲目重试可能导致订单被重复更新为“已支付”。这就需要分布式事务框架来处理幂等性和状态机的异常流转。

### 代码示例 (基于 MQ 的最终一致性 - 事务消息伪代码)
```javan// 发送半消息
Message msg = new Message("TopicPay", "TagPay", orderID.toString().getBytes(), body);
SendResult sendResult = transactionMQProducer.sendMessageInTransaction(msg, null);

// 本地事务执行逻辑
@Override
public TransactionListener executeLocalTransaction(Message msg, Object arg) {
    try {
        // 1. 执行本地数据库操作 (如：扣减余额)
        accountService.decrease(userId, amount);
        // 2. 成功则提交消息，通知下游
        return LocalTransactionState.COMMIT_MESSAGE;
    } catch (Exception e) {
        // 失败则回滚消息，下游收不到通知
        return LocalTransactionState.ROLLBACK_MESSAGE;
    }
}
```

### 分布式环境故障难点对比

| 故障类型 | 本地事务环境 | 分布式环境 |
| :--- | :--- | :--- |
| **网络超时** | 极少发生 (本地回环) | 频繁发生 (RPC调用) |
| **部分节点宕机** | 通常是整体不可用 | 局部不可用，系统状态撕裂 |
| **回滚机制** | 数据库自动 Undo Log | 需手动编码补偿 或 依赖框架回滚 |
| **数据一致性** | 强一致，实时可见 | 存在“中间状态”，需处理脏读/脏写 |
| **锁持有时间** | 毫秒级 | 秒级甚至分钟级 (跨网络协调) |

## 记忆要点

- 复杂性根因1：存储多样性。异构存储（MySQL/Redis/MQ）对事务支持不同，极难统一协调。
- 复杂性根因2：链路延展性。微服务RPC调用面临网络超时、节点部分宕机等不可控因素。
- 本地事务DB自动Undo回滚，分布式事务需依赖框架补偿（防部分成功）或处理长锁。
- 实战方案：用MQ事务消息保证最终一致性，先执行本地DB操作再通知Commit/Rollback。
