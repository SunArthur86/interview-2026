# 详解：二个阶段

### 标准 XA 模型（2PC）详解

2PC（Two-Phase Commit）即二阶段提交协议。广泛应用于数据库领域，旨在让基于分布式架构的所有节点在进行事务处理时保持原子性和一致性。绝大部分关系型数据库都基于 2PC 完成分布式事务处理。

2PC 将事务分为两个阶段：**提交事务请求（阶段一）** 和 **执行事务提交（阶段二）**。

#### 阶段一：提交事务请求
1. **事务询问**：协调者向所有参与者发送事务内容，询问是否可以执行提交操作，并开始等待响应；
2. **执行事务**：各参与者节点执行事务操作，将 Undo 和 Redo 操作计入本机事务日志（此时锁住资源，但未提交）；
3. **反馈响应**：各参与者向协调者反馈询问结果。成功执行返回 Yes，否则返回 No。

#### 阶段二：执行事务提交
协调者根据阶段一的反馈决定最终操作，包含两种情形：

**情形一：执行事务提交**
所有参与者均回复 Yes，则执行提交：
1. 发送提交请求：协调者向所有参与者发送 Commit 请求；
2. 事务提交：参与者正式执行提交，并在日志中记录提交标记，释放占用资源；
3. 反馈结果：向协调者发送 Ack 消息；
4. 完成事务：协调者收到所有 Ack 后，完成事务。

**情形二：中断事务**
若任一参与者回复 No 或超时，则中断事务：
1. 发送回滚请求：协调者发送 Rollback 请求；
2. 事务回滚：参与者利用 Undo 日志执行回滚，释放资源；
3. 反馈结果：发送 Ack 消息；
4. 中断事务：协调者收到所有 Ack 后，中断事务。

#### 流程架构图
```text
      协调者 (Coordinator)             参与者 (Participants)
           │                                 │
           │────── 1. CanCommit/Prepare ────>│

           │<──── 2. Yes/No (Vote) ──────────│

           │                                 │
    (所有Yes)                            (任一No/超时)

           │                                 │  │                             │

           │                                 │  │                             │
      [完成事务]                         [中断事务]
```

#### 补充细节
*   **资源锁定**：在阶段一，参与者执行完操作后，必须锁定相关资源，直到收到 Commit 或 Rollback 指令。这是导致同步阻塞的主要原因。
*   **日志写入**：参与者必须先将 Undo 和 Redo 信息持久化到磁盘（强制刷盘），才能反馈 Yes。这是为了在崩溃后能够恢复。
*   **边界条件**：如果阶段一协调者未收到某参与者的响应，协调者会超时并决定中断事务。

#### 实战深化：故障处理

**实战案例**：生产环境曾出现协调者宕机，导致参与者数据库中大量 `XA RECOVER` 能够查到的挂起事务，连接池无法释放。需要人工介入执行 `XA COMMIT` 或 `XA ROLLBACK` 来清理，因此建议配置合理的 `xa.transaction.timeout`。

**代码示例：MySQL命令行操作XA事务**
```sql
-- 1. 开启XA事务
XA START 'xid01';
UPDATE account SET balance = balance - 100 WHERE id = 1;
-- 2. 准备提交（进入阻塞状态，等待协调者指令）
XA END 'xid01';
XA PREPARE 'xid01';

-- 协调者此时发送 COMMIT 或 ROLLBACK
-- XA COMMIT 'xid01';
-- XA ROLLBACK 'xid01';

-- 3. 故障排查：查看挂起的XA事务
XA RECOVER;
```

**对比表格：2PC 阶段一 vs 阶段二**

| 维度 | 阶段一 | 阶段二 |
| :--- | :--- | :--- |
| **核心动作** | 执行SQL，记录日志，锁定资源 | 提交或回滚，释放资源 |
| **持有锁状态** | 持有排他锁，不允许其他事务修改 | 提交后释放锁 / 回滚后释放锁 |
| **耗时影响** | 耗时最长（包含业务逻辑） | 相对较快（仅IO操作） |
| **失败影响** | 任一节点失败则全员回滚 | 协调者失败可能导致长时间阻塞 |

## 常见考点
1.  **2PC 为什么是强一致性但不是高可用？**
    *   因为它保证了 CP（一致性和分区容错性的一部分），但在同步阻塞期间牺牲了可用性（A），且协调者单点故障会导致整个系统不可用。
2.  **阶段一参与者崩溃了怎么办？**
    *   若在写入日志前崩溃，恢复后认为未参与，返回 No；若在写入日志后崩溃，恢复后根据日志状态，保持阻塞等待协调者指令。
