# Spring Boot在Kubernetes中部署有哪些最佳实践？

🎯 本质：K8s部署Spring Boot需要考虑容器化、健康检查、配置管理、优雅停机、资源管理等维度。

**1. Dockerfile最佳实践**
# 多阶段构建
FROM eclipse-temurin:17-jdk AS builder
COPY . /app
WORKDIR /app
RUN ./mvnw package -DskipTests

FROM eclipse-temurin:17-jre
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "/app.jar"]
# 优化：分层JAR、jib插件、Native Image

**2. 健康检查与探针机制**
K8s通过探针感知Pod状态，需与Spring Boot Actuator紧密配合。

# application.yml
management:
  endpoint:
    health:
      probes:
        enabled: true
  health:
    livenessstate:
      enabled: true
    readinessstate:
      enabled: true

**K8s 探针配置流程图**：
┌─────────────┐      HTTP GET      ┌──────────────────────┐
│   Kubelet   │ ──────────────────> │ Spring Boot App     │
│ (Controller)│ /actuator/health/* │ :8080                │
└─────────────┘                     └──────────────────────┘
     │    │                               │
     │    ├─ Startup Probe (初期) ───────>│ 判断应用是否启动   │
     │    │   failureThreshold: 30        │ ( JVM 预热 )       │
     │    │                               │
     │    ├─ Liveness Probe (运行期) ────>│ 判断是否死锁/卡死 │
     │    │   失败 -> 重启 Pod             │ (返回 200 OK)      │
     │    │                               │
     │    └─ Readiness Probe (运行期) ───>│ 判断是否就绪      │
     │          失败 -> 摘除 Service 流量 │ (如: DB连不上)     │

K8s配置示例：
livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 60
  periodSeconds: 10
readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
startupProbe:
  httpGet:
    path: /actuator/health
    port: 8080
  failureThreshold: 30
  periodSeconds: 10

**实战案例**：某次发布新版本后，Service出现502错误。排查发现是因为新版本应用连接外部限流API失败，应用启动但未就绪。由于只配置了`livenessProbe`（只检查进程存活），K8s一直将流量转发给该Pod。添加`readinessProbe`检查外部API连通性后，K8s在应用就绪前自动摘除了流量，实现了零宕机发布。此外，为了防止滚动更新时连接中断，必须配置Tomcat的`connection-timeout`配合K8s的`terminationGracePeriodSeconds`。

**关键代码示例 (Dockerfile JVM优化)**：
```dockerfile
# 建议使用 Distroless 镜像，体积更小且安全性更高
FROM gcr.io/distroless/java17-debian11
COPY --from=builder /app/target/*.jar /app.jar

# 非关键：JVM 参数建议通过环境变量注入
ENTRYPOINT ["java", "-XX:+UseContainerSupport", "-jar", "/app.jar"]
```

**3. 资源管理与JVM调优**
resources:
  requests:
    memory: "256Mi"
    cpu: "250m"
  limits:
    memory: "512Mi"
    cpu: "1000m"

# 关键参数
JAVA_TOOL_OPTIONS: -XX:MaxRAMPercentage=75.0 -XX:+UseContainerSupport
# 原理：JVM自动识别Cgroup内存限制，MaxRAMPercentage控制堆内存占比。
# 留出25%给元空间、线程栈、直接内存和GC开销。

**4. 配置管理**
- 使用ConfigMap挂载application.yml或环境变量。
- 敏感信息（DB密码）使用Secret挂载到环境变量或文件。
- 配置热更新：利用Spring Cloud Kubernetes Config实现配置变更自动刷新（需配合@RefreshScope）。

## 记忆要点

- 探针配合：因为K8s靠探针感知状态，所以必须开启Actuator的liveness(存活)和readiness(就绪)端点。
- 流量治理：Liveness失败导致重启Pod，而Readiness失败仅摘除流量不重启。
- JVM内存：因为JVM需感知Cgroup限制，所以需开启UseContainerSupport并配置MaxRAMPercentage=75%。
- 配置与安全：非敏感配置用ConfigMap，而敏感信息(如密码)必须用Secret挂载。
