# 对于ACID的保证

Saga 模型对 ACID 特性的保证情况：

1. **原子性**：正常流程下保证；异常时通过补偿机制保证“最终”原子性（即要么全部成功，要么全部被补偿）。
2. **一致性**：在事务执行过程中，可能会出现数据暂时不一致的情况（如 A 库已扣，B 库未加），但最终会达到一致状态。
3. **隔离性**：**不满足**。在某个时间点，一个 Saga 可能读到另一个 Saga 部分提交的结果（脏读）。
   - **补充细节**：这会导致几种典型异常：
     - **脏读**：读取到未完成事务的中间状态。
     - **不可重复读**：同一 Saga 内多次读取可能因为其他 Saga 的提交或回滚而结果不同。
     - **幻读**：范围查询可能因为其他 Saga 的补偿操作而产生变化。
   - **解决思路**：通常在业务层采用“**悲观锁**”（如 FOR UPDATE 扣定数据）或“**乐观锁**”（版本号机制）来绕过数据库隔离性的缺失。
4. **持久性**：与本地事务一致，一旦 commit，数据即被持久化。

**总结**：Saga 不提供完整的 ACID 保证，它是一种 Base 理论（基本可用、软状态、最终一致性）的实现。

**实战案例**：在秒杀场景下，两个 Saga 同时读取同一件商品的库存（库存 10）。Saga A 扣减 5 并提交，Saga B 扣减 6 并提交，最终库存变为 -1。由于 Saga 缺乏隔离性，必须引入 Redis 预扣库存或数据库乐观锁（version 字段）来防止超卖。

**代码示例（Java - 乐观锁解决隔离性）**：
```java
// 使用 MyBatis-Plus 乐观锁插件
@Version
private Integer version;

// 扣减操作
public boolean deductStock(Long productId, int count) {
    // UPDATE product SET stock = stock - #{count}, version = version + 1 
    // WHERE id = #{id} AND version = #{oldVersion} AND stock >= #{count}
    int rows = productMapper.deduct(productId, count, this.getVersion());
    return rows > 0; // 更新失败说明版本冲突或库存不足
}
```

**ACID 对比分析**：

| 特性 | 传统事务 (ACID) | Saga 模型 (BASE) |
| :--- | :--- | :--- |
| **原子性** | 强原子性，要么全做要么全不做 | 最终原子性，允许补偿回滚 |
| **一致性** | 实时强一致 | 最终一致，中间存在不一致状态 |
| **隔离性** | 严格隔离（锁/快照） | 无隔离，易脏读/幻读（需应用层控制） |
| **持久性** | 提交即持久 | 同本地事务 |

## 常见考点
1. **Saga 模型下的“丢失更新”问题是如何产生的？**（两个 Saga 同时读取同一旧值并修改，后提交的覆盖前者的补偿效果）
2. **如何解决 Saga 的隔离性问题？**（语义锁、版本号、串行化）
3. **对比 Saga 与 TCC 在一致性保证上的差异。**（TCC 有 Try 阶段预留资源，隔离性略好于 Saga）
