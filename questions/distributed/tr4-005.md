# Redisson 实现分布式锁的 WatchDog（看门狗）机制是如何工作的？

Redisson 的分布式锁基于 Redis 的 Lua 脚本保证原子性。当调用 `lock.lock()` 时，如果未设置 leaseTime（租约时间），Redisson 会启动 WatchDog 机制。WatchDog 本质是一个后台定时任务（默认每 10 秒检查一次）。如果看门狗发现锁还被持有线程占用，它会自动将锁的过期时间（TTL）重新“续命” 30 秒（默认）。这解决了业务执行时间不确定导致锁自动过期的问题。当持有锁的 JVM 宕机或业务完成后，后台任务停止，锁最终会因为超时而被 Redis 自动释放，从而避免死锁。注意，如果显式指定了 leaseTime，WatchDog 不会生效。

## 技术原理

- **触发条件：调用 lock.lock() 且未设置 leaseTime**：Redisson 的 `lock()` 有两个重载——`lock()`（不传租约，启用看门狗）和 `lock(leaseTime, unit)`（显式租约，禁用看门狗）。看门狗只在"未设租约"分支启动，因为显式租约意味着业务自己保证在 leaseTime 内完成，不需要自动续期。
- **工作原理：后台定时任务每 10 秒检查，自动将锁 TTL 重置为 30 秒**：看门狗本质是 Redisson 客户端的一个 `ScheduledExecutorService` 定时任务。它通过 Lua 脚本原子执行：检查锁的 `hash` 字段里持有者是否还是当前线程，若是则 `PEXPIRE` 续期到 30 秒。续期间隔 `lockWatchdogTimeout / 3 = 10s`，TTL 默认 `lockWatchdogTimeout = 30s`，即"每 10 秒续到 30 秒"，留有 20 秒容错（续期失败两次才过期）。
- **失效场景：显式指定 leaseTime 时不生效，或业务完成/宕机后停止**：①`lock(10, SECONDS)` 不启用看门狗，10 秒后锁强制释放；②业务正常 `unlock()` 后定时任务取消；③JVM 进程宕机，定时任务随之消失，锁在剩余 TTL 后自动过期——这正是看门狗的容灾价值：进程挂了不会死锁。

## 代码示例

看门狗自动续期（推荐用法）：

```java
RLock lock = redisson.getLock("order:lock:1001");
try {
    lock.lock();                     // 不传 leaseTime → 启用看门狗，默认 30s TTL，每 10s 续期
    // 业务执行哪怕 5 分钟，看门狗持续续期不会过期
    processOrder();                  // 异常会抛出，finally 释放锁
} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();               // 主动释放，看门狗任务取消
    }
}
```

看门狗失效场景（显式租约）：

```java
lock.lock(10, TimeUnit.SECONDS);     // 启用 10s 租约 → 看门狗不生效
// 若 processOrder 执行超过 10s，锁已被释放，其他线程可能拿到锁导致并发问题
processOrder();
```

调整看门狗参数（构造 Redisson 时）：

```java
Config config = new Config();
config.setLockWatchdogTimeout(60_000);   // 默认 30000ms，调到 60s（续期间隔自动变为 20s）
RedissonClient client = Redisson.create(config);
```

续期 Lua 脚本（Redisson 内部实现简化版）：

```lua
-- KEYS[1]=锁名  ARGV[1]=线程标识  ARGV[2]=新TTL
if redis.call('HEXISTS', KEYS[1], ARGV[1]) == 1 then
    return redis.call('PEXPIRE', KEYS[1], ARGV[2])   -- 还是我持有 → 续期
else
    return 0                                          -- 不再持有 → 不续
end
```

## 对比/选型

| 用法 | leaseTime | 看门狗 | 适用场景 |
|------|-----------|--------|----------|
| `lock()` | 默认 30s | 启用 | 业务耗时不可控（推荐） |
| `lock(lease, unit)` | 自定义 | 禁用 | 明确知道耗时上限 |
| `tryLock(wait, lease, unit)` | 自定义 | 禁用 | 带超时的尝试获取 |

## 常见坑/注意事项

- **看门狗是客户端机制，不是 Redis 服务端的**：续期靠 Redisson 客户端的定时任务。如果客户端进程被 `kill -9` 或 OOM，定时任务立即停止；如果客户端 GC 长时间 STW，续期可能延迟导致锁提前过期——这是看门狗的固有局限。
- **显式 leaseTime 的陷阱**：很多人写 `lock(30, SECONDS)` 以为更安全，实际是禁用了看门狗，业务超时锁就被释放，反而引入并发风险。除非确知耗时上限，否则用无参 `lock()`。
- **lockWatchdogTimeout 别乱调**：调太小（如 3s）续期频率过高增加 Redis 压力，且容错窗口变窄；调太大（如 5min）进程宕机后锁占用资源时间过长。默认 30s 是经验最优。
- **unlock 前要判断 isHeldByCurrentThread**：看门狗续期是异步的，可能锁已被重入计数减到 0 后又被其他线程持有，直接 unlock 会释放别人的锁。
- **重入次数与续期**：看门狗续期的是整个 hash 结构的 TTL，不区分重入次数，只要还持有就续，这点符合预期。
