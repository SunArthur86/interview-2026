# HTTPS的工作原理？(https是怎么建立连接的）

【HTTPS 工作原理与 TLS 握手】

HTTPS = HTTP + SSL/TLS，通过非对称加密交换密钥，通过对称加密传输数据。

【TLS 握手流程图 (RSA 算法)】
```text
Client                                       Server
  |                                            |
  | 1. Client Hello                            |
  | (支持的加密套件、随机数 Random_C) ---------> |
  |                                            |
  | <------- 2. Server Hello + 证书 ----------- |
  | (选定加密套件、随机数 Random_S、证书)      |
  |                                            |
  | 3. 验证证书，提取公钥                       |
  | 生成预主密钥                |
  | (用服务端公钥加密) -----------------------> |
  |                                            |
  | 4. 解密得到预主密钥                       |
  | 生成会话密钥                             |
  | (Session Key = Hash(Random_C+Random_S+PMS))|
  |                                            |
  | <------- 5. Server Finished (加密) -------- |
  | 6. Client Finished (加密) ---------------> |
  |                                            |
  | === 后续使用会话密钥进行对称加密传输 ===== |
```

【HTTPS 建立连接流程 (TLS 握手)】
1. **Client Hello**：客户端发送支持的 SSL/TLS 版本、加密套件列表、压缩算法及一个**随机数 Random_C**。
2. **Server Hello + 证书**：服务端确认协议版本和加密套件，发送数字证书和**随机数 Random_S**。
3. **证书验证**：客户端利用本地内置的 CA 根证书验证服务端证书的合法性（链路是否完整、域名是否匹配、是否过期），并从中提取服务端**公钥**。
4. **预主密钥生成**：客户端生成第三个随机数——**预主密钥**，使用服务端的公钥加密后发送给服务端。
5. **会话密钥计算**：
   - 服务端使用私钥解密得到预主密钥。
   - 双方均使用 Random_C、Random_S 和 Pre-Master，通过约定的伪随机数函数（PRF）生成相同的对称**会话密钥**。后续所有通信都使用此密钥。
6. **完成握手**：双方发送 Finished 消息（包含之前所有握手消息的 Hash，并用会话密钥加密），验证握手过程未被篡改。
7. **加密传输**：握手完成后，应用层数据使用会话密钥进行对称加密传输（性能高）。

## 常见考点
1. **为什么握手用非对称加密，传输用对称加密**？非对称加密计算复杂（CPU消耗大），速度慢；对称加密速度快，适合大数据传输。
2. **中间人攻击 (MITM) 是如何防止的**？通过 CA 机构签发的数字证书。客户端验证证书签名确保证书由可信 CA 颁发，且域名匹配，中间人无法伪造受信任的证书。
3. **TLS 1.2 和 1.3 的区别**？TLS 1.3 将握手次数从 2-RTT 减少到 1-RTT，提升了连接建立速度，并移除了不安全的旧加密算法。
4. **私钥的作用**：服务端私钥仅用于解密 Pre-Master（RSA 握手）或签名（DHE/ECDHE 握手），不用于解密业务数据。

## 记忆要点

- HTTPS核心：非对称加密交换密钥，而对称加密传输数据
- 握手前两步：交换Client/Server随机数和CA数字证书
- 握手后两步：客户端生成预主密钥，服务端公钥加密后安全传回
- 会话密钥：双方用三个随机数生成，后续业务均用此对称加密
- 防中间人：靠可信CA证书签名验证，非对称加解密性能差仅用于握手
