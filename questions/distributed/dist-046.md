# CAP理论中如何选择CP还是AP？

CAP理论：分布式系统不能同时满足一致性（C）、可用性（A）和分区容错性（P）。网络分区P必然存在，只能在C和A之间选一个。

**选择CP（一致性+分区容错）**：
- **场景**：强一致性要求，如金融交易、配置管理、库存扣减。
- **技术选型**：ZooKeeper（ZAB协议）、etcd（Raft）、Consul（CP模式）、HBase。
- **特点**：分区发生时，系统拒绝非Leader节点的写请求（或阻塞读），以此保证数据强一致，牺牲部分可用性。

**选择AP（可用性+分区容错）**：
- **场景**：高可用要求，如电商商品浏览、社交动态、缓存。
- **技术选型**：Eureka、Cassandra、DynamoDB、Nacos（AP模式）、DNS。
- **特点**：分区发生时，各节点继续独立服务（接受读写），允许数据短暂不一致，追求最终一致性。

**实际选择与权衡**：
大多数互联网应用选AP，最终一致性够用。金融场景选CP。且系统可能在不同模块间切换，如订单选CP，商品详情选AP。

**实战案例**：
在配置中心选型中，曾使用Eureka（AP）存储灰度发布开关。因网络抖动导致部分节点读取到旧配置，引发线上事故。后改用Nacos CP模式，确保所有节点配置修改原子性，避免了配置不一致。

**CP vs AP 对比**：
| 维度 | CP (一致优先) | AP (可用优先) |
| :--- | :--- | :--- |
| **核心目标** | 数据强一致 | 服务高可用 |
| **分区表现** | 少数派停止服务 | 所有节点继续服务 |
| **用户体验** | 可能报错/超时 | 可能看到旧数据 |
| **典型中间件** | ZooKeeper, HBase, etcd | Eureka, Cassandra, DynamoDB |
| **业务场景** | 支付、库存、配置 | 商品详情、朋友圈、缓存 |

**## 常见考点**
1. **BASE理论**：AP架构通常结合BASE理论（基本可用、软状态、最终一致性）来解释业务上的妥协。
2. **CP在分区的表现**：CP系统在分区时， minority side（少数派）通常会停止服务或只读，只有 majority side（多数派）能写，这本身就是可用性的损失。
3. **Nacos/Eureka的区别**：Nacos支持AP和CP切换，而Eureka是纯AP设计（自我保护机制），ZooKeeper是CP设计。
4. **配置中心选型**：为什么配置中心通常选CP？（因为配置错误比服务不可用更危险，必须保证所有节点配置一致）。


## 记忆要点

- 因为网络分区P必然发生，所以系统只能在C（一致性）和A（可用性）中二选一
- 选CP：宁可拒绝服务也要数据强一致（如金融、ZK、etcd）
- 选AP：宁可返回旧数据也要服务高可用（如缓存、Eureka）
- 配置中心通常选CP，因为错误配置比短暂不可用更危险
- Nacos支持AP/CP切换，而Eureka是纯AP，ZooKeeper是纯CP
