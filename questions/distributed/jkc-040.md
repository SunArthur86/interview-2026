# RabbitMQ

### RabbitMQ 核心概念与架构

RabbitMQ 是一个由 Erlang 语言开发的 AMQP（高级消息队列协议）的开源实现，广泛应用于分布式系统中存储转发消息。

#### 1. 核心特点
1.  **可靠性**：提供持久化、传输确认、发布确认等机制保证消息不丢失。
2.  **灵活的路由**：在消息进入队列前，通过 Exchange（交换机）路由。支持内置 Exchange 和自定义插件 Exchange。
3.  **高可用性**：支持集群部署和队列镜像，部分节点故障时服务仍可用。
4.  **多协议支持**：支持 AMQP、STOMP、MQTT 等多种协议。
5.  **多语言客户端**：支持 Java、.NET、Ruby 等几乎所有常用语言。
6.  **管理界面**：提供易用的 Web UI 用于监控和管理消息状态。
7.  **跟踪机制**：提供 Firehose 等机制跟踪消息流向，便于排查异常。

#### 2. 基本架构模型
*   **Producer（生产者）**：发送消息的应用程序。
*   **Exchange（交换机）**：接收生产者消息，并根据路由规则将消息分发到绑定在它上面的队列。常见类型有 Direct、Topic、Fanout、Headers。
*   **Queue（队列）**：存储消息，直到消息被消费者消费。由于 Erlang 的特性，RabbitMQ 的性能非常出色。
*   **Consumer（消费者）**：接收消息的应用程序。
*   **Binding Key（绑定键）**：Exchange 和 Queue 之间的虚拟连接规则。
*   **Routing Key（路由键）**：生产者发送消息时指定的键，Exchange 根据它和 Binding Key 的匹配规则来决定路由。

#### 3. 实战案例
**坑点**：生产环境曾出现因未开启 `publisher-confirms` 且网络抖动，导致消息未到达 Exchange 就丢失，引起数据对账不平。核心业务必须强制开启 Confirm 模式并处理回调逻辑。

#### 4. 四种交换机类型对比

| 类型 | 路由规则 | 适用场景 | 复杂度 |
| :--- | :--- | :--- | :--- |
| **Direct** | 完全匹配 Routing Key | 点对点、任务分发 | 低 |
| **Topic** | 通配符匹配 (*, #) | 多级分类广播、通知 | 中 |
| **Fanout** | 忽略 Key，广播到所有绑定队列 | 日志收集、实时更新 | 低 |
| **Headers** | 匹配 Headers 键值对 | 复杂路由规则（极少用） | 高 |

#### 5. 消息可靠性关键代码
```java
// 1. 开启 Confirm 模式 (生产者端)
channel.confirmSelect();

// 2. 发送消息
channel.basicPublish(exchange, routingKey, MessageProperties.PERSISTENT_TEXT_PLAIN, message.getBytes());

// 3. 异步监听确认
channel.addConfirmListener((sequenceNumber, multiple) -> {
    // 发送成功，记录日志或更新 DB
}, (sequenceNumber, multiple) -> {
    // 发送失败，重试或报警
});

// 4. 消费者端手动 ACK
channel.basicConsume(queueName, false, consumer); // 第二个参数 autoAck 设为 false
// 处理业务后
channel.basicAck(envelope.getDeliveryTag(), false);
```

## 常见考点
1.  **消息如何保证不丢失？** 三步走：生产者开启 Confirm 模式确认到达 Exchange；队列开启持久化；消费者开启 Manual Ack 模式（手动确认）。
2.  **Exchange 的四种类型**：Direct（精确匹配）、Topic（通配符匹配）、Fanout（广播）、Headers（几乎不用，根据头部键值对匹配）。
3.  **RabbitMQ 如何避免消息堆积？** 优化消费者消费速度（增加消费者、批量消费）；使用惰性队列；或者增加队列数量分流。
