# seata

Seata 是一款开源的分布式事务解决方案，致力于在微服务架构下提供高性能和简单易用的分布式事务服务。

### 实战案例
在库存服务与订单服务的分布式事务中，利用 Seata AT 模式，开发人员只需在业务方法上添加 `@GlobalTransactional` 注解，Seata 会在底层自动解析 SQL 生成回滚镜像，无需像 TCC 那样手写复杂的三个接口，大大降低了新人的上手门槛。

### 代码示例（Seata AT 模式使用）
```java
@GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
public void createOrder(OrderDTO orderDTO) {
    // 1. 扣减库存（Seata 自动记录前镜像与后镜像）
    stockService.deduct(orderDTO.getProductId(), orderDTO.getCount());
    // 2. 创建订单
    orderService.insert(orderDTO);
    // 3. 若发生异常，Seata TC 自动协调回滚上述操作
}
```

### 核心特性
Seata 为用户提供了 **AT、TCC、SAGA 和 XA** 四种事务模式，打造一站式的分布式解决方案。

## 技术原理

Seata 的架构由三大角色组成，通过 **TC（事务协调器）** 统一管理全局事务的生命周期：

- **三大核心角色**：
  - **TC（Transaction Coordinator）**：独立部署的事务协调器，维护全局事务和分支事务的状态，决定提交或回滚。是 Server 端。
  - **TM（Transaction Manager）**：定义全局事务的开始、提交、回滚边界（即标注 `@GlobalTransactional` 的那一端）。是 Client 端。
  - **RM（Resource Manager）**：管理分支事务上的资源（数据库），向 TC 注册分支，上报状态，执行提交或回滚。每个微服务实例都是一个 RM。
- **AT 模式的自动补偿原理**：
  - **一阶段**：RM 拦截业务 SQL，在本地事务提交前，把"修改前镜像"（before image）和"修改后镜像"（after image）写入 `undo_log` 表，与业务 SQL 在**同一个本地事务**里提交。这样本地锁立即释放，性能高。
  - **二阶段提交**：TC 收到所有分支成功，异步通知 RM 删除 undo_log（无需业务参与）。
  - **二阶段回滚**：TC 发现有分支失败，通知所有 RM 根据 undo_log 的 before image 生成**反向补偿 SQL**（把数据改回原样），自动回滚。
  - 关键：AT 模式靠解析 SQL 自动生成 undo_log，业务零侵入，这是它比 TCC 易用的根本原因。
- **四种模式的定位**：
  - **AT**：无侵入、性能中、最终一致——互联网主流。
  - **TCC**：高侵入、高性能、最终一致——金融核心。
  - **SAGA**：长流程、正向 + 补偿——业务编排。
  - **XA**：强一致、性能差——传统场景。

## 代码示例

Seata AT 模式的完整使用流程：

```java
// 1. 业务方法加 @GlobalTransactional 注解即可（TM 端）
import io.seata.spring.annotation.GlobalTransactional;

@Service
public class OrderService {
    @Autowired private StockService stockService;
    @Autowired private OrderMapper orderMapper;

    @GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
    public void createOrder(OrderDTO dto) {
        // 分支1：扣库存（RM 自动记录 undo_log）
        stockService.deduct(dto.getProductId(), dto.getCount());
        // 分支2：建订单（RM 自动记录 undo_log）
        orderMapper.insert(dto);
        // 若此处抛异常，TC 协调所有分支用 undo_log 反向回滚
    }
}
```

## 注意事项

- **AT 模式要求每个业务库都建 undo_log 表**：Seata 靠这张表存前后镜像，漏建会导致回滚失败。多数据源场景要确保每个库都有。
- **AT 模式有短暂读写不一致**：一阶段本地事务已提交，二阶段全局回滚前的窗口期，其他事务可能读到"即将回滚"的脏数据。配合 `@GlobalLock` 或在读敏感场景使用 SELECT FOR UPDATE。
- **AT 不支持嵌套全局事务**：`@GlobalTransactional` 不能嵌套，嵌套调用会被当作同一全局事务的分支。若需要独立子事务，考虑用本地事务隔离。
- **TC 是单点风险**：TC 宕机会导致全局事务卡住（无法提交/回滚）。生产环境必须部署 TC 集群（3 节点）+ 注册中心（Nacos/Eureka）保证高可用。
- **性能瓶颈在全局锁**：AT 模式下 TC 对热点数据加全局锁，高并发写同一行时会成为瓶颈。热点账户（如平台收款账户）要考虑用 TCC 或消息最终一致替代。
