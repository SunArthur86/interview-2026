# Saga的实现

### Saga 模式的实现细节

Saga 模式主要用于解决长事务问题，其核心实现机制和注意事项如下：

#### 1. 基于状态机引擎的实现
Seata Saga 模式通过状态机引擎驱动业务流转：
1.  **状态定义**：使用 JSON 定义业务流程的状态图。
2.  **节点与补偿**：每个节点调用服务，并定义对应的补偿节点（用于回滚）。
3.  **异常处理**：当流程中某节点抛出异常，引擎自动触发已成功节点的补偿逻辑。

#### 2. 核心功能
支持复杂的编排需求：
*   流程控制：单项选择、并发执行、子流程嵌套。
*   数据处理：参数转换、参数映射。
*   逻辑判断：服务执行状态判断、异常捕获。

#### 3. 缺点与解决方案
*   **不保证隔离性**：
    *   Saga 的各子事务直接提交本地数据，中间状态对其他事务可见。
    *   **应对方案**：在业务层增加“悲观锁”控制（如账户余额字段加版本号），或者利用“排他锁”在资源层面隔离。
*   **业务侵入**：需要为每个正向操作编写补偿逻辑。

#### 深化：实战与并发控制

**实战案例**：
在多节点并发执行的 Saga 流程中，两个订单同时扣减同一库存，导致“超扣”问题（由于各子事务立即提交，无法利用 DB 行锁）。**解决方案**：在数据库扣减 SQL 中显式加入乐观锁条件（`AND version = old_version`），并在状态机捕获 `OptimisticLockingFailureException` 进行重试或回滚。

**代码示例 (Java 状态机定义)**：
```java
// 编排状态机定义
StateMachineBuilder builder = StateMachineBuilderFactory.create();
builder.externalTransition()
    .from("STATE_A")
    .to("STATE_B")
    .on("EVENT_TO_B")
    .when((ctx) -> true) // 条件判断
    .perform(ctx -> {
        // 调用业务服务并处理输入输出映射
        String result = serviceB.doProcess((String) ctx.getVariable("input"));
        ctx.setVariable("bResult", result);
    });
```

**Saga 与 Seata AT 对比**：

| 对比项 | Seata AT 模式 | Seata Saga 模式 |
| :--- | :--- | :--- |
| **实现机制** | JDBC 代理 + Undo Log | 状态机引擎 + 补偿服务 |
| **事务阶段** | 两阶段（Prepare/Commit） | 多阶段（按流程图顺序执行） |
| **回滚粒度** | SQL 级别（自动反向 SQL） | 服务级别（调用补偿接口） |
| **适用场景** | 短链路、DB 强依赖 | 长链路、跨系统、流程编排 |
| **锁竞争** | 全局锁持有至事务结束 | 仅本地事务持有锁，无全局锁 |
