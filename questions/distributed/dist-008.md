# Ribbon内置的负载均衡策略是什么？

### Ribbon 内置的负载均衡策略
Ribbon 是 Netflix 提供的客户端负载均衡器，工作在客户端进程内，其核心接口是 `IRule`。以下是主要的内置策略及其原理：

1.  **RoundRobinRule（轮询）**：
    *   **原理**：按顺序依次轮询服务器列表。这是默认策略。
    *   **适用场景**：服务器性能相近，请求量均匀分布。

2.  **RandomRule（随机）**：
    *   **原理**：使用 `java.util.Random` 在服务列表中随机选择一个。
    *   **适用场景**：请求量巨大时，统计上能达到近似均匀的分布。

3.  **RetryRule（重试）**：
    *   **原理**：内部封装了一个 `IRule`（默认是 `RoundRobinRule`）。先按照选定的策略选取一个服务器，如果选择失败或在指定时间内无响应，则会尝试重试（默认最多重试次数由 `maxRetryMillis` 等参数控制）。
    *   **注意**：重试可能会增加系统的延迟。

4.  **WeightedResponseTimeRule（权重响应时间）**：
    *   **原理**：这是对 RoundRobin 的改进。根据每个服务器的平均响应时间计算权重。
        *   响应时间越短，权重越高，被选中的概率越大。
        *   **实现细节**：后台有一个定时任务，定期统计所有服务器的平均响应时间，然后生成一个权重列表，根据权重区间进行随机选择。
    *   **适用场景**：服务器性能差异较大，希望能将流量导向性能更好的机器。

5.  **BestAvailableRule（最低并发）**：
    *   **原理**：遍历所有服务器，过滤掉处于“熔断跳闸”状态的服务器（依赖 Hystrix），然后选择并发请求数（`ActiveRequestsCount`）最小的服务器。
    *   **适用场景**：对响应延迟敏感，希望避免单个服务器过载。

6.  **AvailabilityFilteringRule（可用性过滤）**：
    *   **原理**：结合了“可用性过滤”和“轮询”。
        *   **过滤逻辑**：
            1.  过滤掉因多次访问故障而处于熔断跳闸状态的服务器。
            2.  过滤掉并发连接数超过阈值（可配置）的服务器。
        *   **选择逻辑**：在剩余的可用服务器中，使用轮询策略。

7.  **ZoneAvoidanceRule（区域感知） - 默认在 Eureka Client 中使用**：
    *   **原理**：复合判断服务器所在 Zone 的运行状况和服务的可用性。
    *   **机制**：
        1.  轮询所有 Zone，过滤出认为“可用”的 Zone（例如断路器打开比例低）。
        2.  在可用 Zone 内部，使用 `AvailabilityFilteringRule` 逻辑选择具体的服务器实例。
    *   **适用场景**：多机房（AWS Zone）部署，优先调用同机房或延迟低的机房。

#### 策略选型对比表

| 策略名称 | 核心逻辑 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **RoundRobinRule** | 轮询 | 简单，分布均匀 | 无法感知性能差异 | 服务器配置同构，无状态服务 |
| **WeightedResponseTimeRule** | 权重(响应时间) | 自动导向快机器 | 统计有延迟，慢机器可能雪崩 | 异构集群，性能差异大 |
| **BestAvailableRule** | 最小并发 | 极致低延迟 | 需维护实时并发计数 | 对RT极度敏感的实时业务 |
| **RetryRule** | 失败重试 | 提高瞬时成功率 | 可能加重下游压力 | 允许短暂重试的业务 |

#### 实战案例
在微服务上线初期，新旧版本共存。旧版本代码处理逻辑慢，容易导致请求堆积。如果使用轮询，流量平均分配会导致旧版本服务响应超时。切换到 `WeightedResponseTimeRule` 后，Ribbon 自动识别出旧版本响应慢并降低其权重，流量自动倾斜到新版本，整体系统吞吐量提升了 30%。

#### 关键代码片段
```java
// 自定义基于权重的策略示例
public class CustomWeightedRule extends AbstractLoadBalancerRule {
    @Override
    public Server choose(Object key) {
        List<Server> servers = getLoadBalancer().getAllServers();
        // 计算权重（伪代码：假设 Server metadata 包含 weight）
        int totalWeight = servers.stream().mapToInt(s -> getWeight(s)).sum();
        int randomWeight = ThreadLocalRandom.current().nextInt(totalWeight);
        // 根据随机权重的落点选择 Server
        for (Server server : servers) {
            randomWeight -= getWeight(server);
            if (randomWeight < 0) return server;
        }
        return servers.get(0);
    }
}
```

## 常见考点
1.  **Ribbon 默认的负载均衡策略是什么？**（答：`RoundRobinRule`，但在结合 Eureka 时若未指定，可能会使用 `ZoneAvoidanceRule`）
2.  **WeightedResponseTimeRule 是如何计算权重的？**（答：响应时间越短，权重越高；后台定时任务计算）
3.  **客户端负载均衡和服务端负载均衡的区别？**（答：客户端负载均衡（如 Ribbon）由消费者自己维护服务列表并选择；服务端（如 Nginx、LVS）由中心节点分配）
4.  **如何自定义 Ribbon 策略？**（答：实现 `IRule` 接口或继承 `AbstractLoadBalancerRule`，配置为 Bean 或在配置文件中指定）
