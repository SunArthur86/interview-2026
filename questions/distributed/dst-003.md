# MySQL的本地事务实现方案

### 并发事务问题

1.  **一类丢失更新**：两个事务读同一数据，一个修改字段1，一个修改字段2，后提交的覆盖了先提交的修改。
2.  **二类丢失更新**：两个事务读同一数据，都修改同一字段，后提交的覆盖了先提交的修改。
3.  **脏读**：读到了未提交的数据，如果该事务回滚，则产生脏读。
4.  **不可重复读**：同一事务内两次读取之间，被另外一个事务修改了数据的内容，导致前后读取内容不一致。
5.  **幻读**：同一事务内两次查询之间，被另外一个事务插入或删除了记录，导致结果集数量不一致。

### ACID 特性

-   **原子性**：事务中的操作要么全部成功，要么全部失败回滚。
-   **一致性**：事务执行前后，数据库的完整性约束没有被破坏。
-   **隔离性**：并发执行的事务之间互不干扰。
-   **持久性**：事务一旦提交，对数据的修改是永久性的，即使系统崩溃也不会丢失。

### MySQL 本地事务实现方案

在单机数据库场景下，MySQL InnoDB 引擎主要通过以下机制保证 ACID：

1.  **原子性**：通过 **Undo Log（撤销日志）** 实现。修改数据前先备份旧版本到 Undo Log，出错或回滚时利用日志回滚数据。
2.  **持久性**：通过 **Redo Log（重做日志）** 实现。事务提交前先将修改写入 Redo Log 并持久化，系统崩溃后可根据 Redo Log 恢复数据（WAL 机制）。
3.  **隔离性**：通过 **数据库锁（共享锁/排他锁）** 和 **MVCC（多版本并发控制）** 实现。
4.  **一致性**：依赖原子性、持久性和隔离性，以及数据库自身的约束（如外键、唯一索引）共同保障。

**实战案例**：在高并发库存扣减场景，如果不加锁或隔离级别设置不当，容易发生“超卖”。通常我们将隔离级别设置为 **Read Committed (RC)** 配合 `SELECT ... FOR UPDATE`（悲观锁）或利用乐观锁版本号机制来解决。MySQL 默认的 RR 级别虽然防幻读，但在高并发下容易产生死锁，需要权衡选择。

**代码示例 (Spring Boot + MySQL 隔离级别配置)**：
```java
// 设置事务隔离级别为 READ_COMMITTED (读已提交)
@Transactional(isolation = Isolation.READ_COMMITTED)
public void deductStock(Long productId, int count) {
    // 悲观锁查询：记录会被锁定，其他事务必须等待
    Product product = productDao.selectByIdForUpdate(productId);
    if (product.getStock() >= count) {
        product.setStock(product.getStock() - count);
        productDao.updateById(product);
    } else {
        throw new RuntimeException("库存不足");
    }
}
```
