# CAP 定理和 BASE 理论的关系是什么？互联网架构为什么选择 AP+BASE？

【CAP 定理】分布式系统在网络分区（P）时，一致性（C）和可用性（A）只能二选一。
- **CP**：保证一致性，分区时拒绝服务（如 ZooKeeper、etcd）。对于少数派节点，直接不可用。
- **AP**：保证可用性，分区时各节点可独立响应（可能不一致）（如 Eureka、Cassandra）。
- **注意**：P（分区容错）是分布式系统的必然属性，网络上一定会出现丢包或延迟，必须在C和A中权衡。

【BASE 理论】是 CAP 中 AP 的工程实践延伸，是对 CAP 中“牺牲一致性”的补充说明：
- **Basically Available（基本可用）**：允许损失部分可用性（如响应时间增加、功能降级、服务限流），但核心功能可用。
- **Soft State（软状态）**：允许系统中的数据存在中间状态，该中间状态不影响系统可用性，即数据在不同节点间同步存在延迟。
- **Eventually Consistent（最终一致性）**：不保证实时强一致，但经过一段时间（ms/s级）后，所有副本的数据最终能达到一致。

【互联网为什么选 AP+BASE？】
1. **用户体验优先**：对于电商、社交场景，用户宁可看到旧的库存数或评论数，也不愿意遇到“服务拒绝访问”或页面打不开。
2. **高并发要求**：CP架构在分区时为了维持一致性通常会阻塞或报错，这在互联网高并发场景下会引发雪崩。AP架构允许继续服务，保护系统吞吐。
3. **最终一致足够**：大部分业务（如点赞数、浏览量、订单状态更新）不需要实时的强一致，允许短暂的数据延迟。
4. **强一致代价高**：传统的2PC/3PC在分布式环境下性能极差，且容易阻塞，无法满足互联网规模。

【什么场景选 CP？】
- **金融转账**：账户金额必须精确，宁可转账失败也不能出现余额不一致。
- **配置中心**：分布式系统配置必须全网一致，错误的配置比没有配置更危险（可能导致全网瘫痪）。
- **分布式锁**：锁的状态必须唯一，出现两个客户端同时持有锁会导致严重并发问题（超卖）。

【如何实现最终一致？】
- **异步消息（MQ）** + 重试 + 幂等性校验。
- **定时补偿/对账**系统（T+1或准实时）。
- **TCC/Saga** 分布式事务模式（通过补偿动作实现最终一致）。

**实战案例**：
电商大促期间，为了保证订单服务的高可用，订单状态更新采用了AP+BASE策略。当用户下单支付后，由于网络抖动导致订单服务和库存服务分区，订单服务先返回“支付成功”（软状态），库存服务稍后通过MQ异步消费消息扣减库存。即使短暂延迟，用户也能立刻看到支付成功页面，极大提升了支付转化率。

**最终一致性实现代码示例（本地消息表）**：
```java
// 1. 在同一个本地事务中，执行业务操作和插入消息记录
@Transactional(rollbackFor = Exception.class)
public void orderPayment(PaymentRequest request) {
    // 扣减余额/更新订单状态
    updateOrderStatus(request.getOrderId(), "PAID");
    // 插入待发送消息到本地消息表
    messageService.insertLocalMessage(new LocalMessage(request.getOrderId(), "INVENTORY_DEDUCT"));
}

// 2. 定时任务扫描并重试发送消息到MQ
@Scheduled(fixedDelay = 5000)
public void processPendingMessages() {
    List<LocalMessage> messages = messageService.fetchPendingMessages(100);
    for (LocalMessage msg : messages) {
        try {
            rocketMQTemplate.syncSend("inventory-topic", msg);
            // 发送成功后更新消息状态为已发送
            messageService.markAsSent(msg.getId());
        } catch (Exception e) {
            // 仅记录日志，等待下次重试，保证最终一致
            log.error("Send message failed, will retry later", e);
        }
    }
}
```

**架构选型对比 (CP vs AP)**：

| 维度 | CP (一致性优先) | AP (可用性优先) |
| :--- | :--- | :--- |
| **核心代表** | ZooKeeper, etcd, HBase | Eureka, Cassandra, DynamoDB, DNS |
| **一致性保障** | 强一致性 (线性一致性) | 最终一致性 |
| **分区表现** | 少数派节点拒绝服务，不脏读 | 节点接受读写，可能返回旧数据，服务不中断 |
| **性能表现** | 写入性能受限于同步复制协议 (Paxos/Raft) | 读写性能高，异步复制延迟低 |
| **适用场景** | 配置管理、分布式锁、元数据存储、银行核心 | 缓存、商品详情、用户画像、物联网数据采集 |
| **复杂度** | 客户端无需处理冲突，服务端逻辑复杂 | 需处理数据冲突、向量时钟、版本收敛 |

**## 常见考点**
1. **ACID vs BASE**：传统数据库（ACID）强调强一致性，而分布式系统（BASE）强调可用性和最终一致性，两者的设计哲学完全不同。
2. **BASE中的软状态**：举个例子，支付系统显示“支付中”就是软状态，最终会变为“支付成功”或“支付失败”，中间状态允许存在。
3. **最终一致性的延迟边界**：虽然说是“最终”，但在工程上通常会定义SLA（如1秒内达到一致），如果超过这个时间就要告警。
4. **刚性事务与柔性事务**：CP对应刚性事务（如2PC），AP对应柔性事务（如TCC、Saga、本地消息表）。


## 记忆要点

- BASE是CAP中AP架构的工程延伸：基本可用、软状态、最终一致性
- 互联网选AP+BASE：因为CP阻塞会引发雪崩，所以宁可牺牲强一致保高并发
- 用户体验优先：用户宁可看到旧数据，也不愿忍受服务报错或超时
- 最终一致实现：本地消息表 + MQ异步重试 + 幂等校验
- 强一致场景选CP：如金融转账、配置中心、分布式锁防超卖
