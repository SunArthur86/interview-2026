# TCP连接如何确保可靠性

【TCP 连接如何确保可靠性】

**1. 校验和**
TCP 头部保持了一个校验和字段。发送方在发送数据前计算校验和，接收方收到后重新计算。如果校验和不匹配，说明数据在传输过程中发生变化（如位翻转），该包会被直接丢弃，等待发送方重传。

**2. 序列号**
- **作用**：对数据字节流进行编号。
- **细节**：接收方根据序列号对接收到的乱序数据包进行重新排序；同时去除重复接收到的数据包（如由于超时重传导致的重复）。

**3. 确认应答 (ACK)**
- **机制**：接收方收到数据后，会回复一个 ACK 包，其中的 ACK 号码等于期望收到的下一个序列号（即：确认号 N = 已收到的最后一个字节序号 + 1）。
- **累积确认**：TCP 的 ACK 通常是累积的，表示“N 之前的所有数据我都收到了”，这可以提高效率。

**4. 超时重传 (RTO)**
- **触发条件**：发送方发出数据后启动定时器，若在 RTO 内未收到 ACK，则认为丢包，进行重传。
- **动态调整**：RTO 不是固定的，由 RTT（往返时间）加权计算得出，以适应网络状况变化。若 RTT 波动大，RTO 会相应增大，避免不必要的重传。

**5. 快速重传**
- **机制**：如果接收方收到了乱序的数据（例如收到了包 1 和 包 3，但没收到包 2），它会重复发送针对包 1 的 ACK。
- **触发**：当发送方连续收到 3 个重复的 ACK（Duplicate ACK），它会意识到包 2 丢失，立即重传包 2，而不必等待 RTO 超时。

**6. 流量控制 (滑动窗口)**
- **目的**：防止发送方发太快，把接收方的缓存撑爆。
- **实现**：接收方在 TCP 头部的“窗口大小”字段中填入自己接收缓冲区的剩余空间。发送方根据此值控制发送速率。
- **零窗口探查**：如果窗口大小为 0，发送方停止发送，并启动定时器探测接收方窗口何时恢复。

**7. 拥塞控制**
- **慢启动**：连接刚建立时，拥塞窗口从 1 开始指数增长，快速试探网络承载能力。
- **拥塞避免**：当 cwnd 达到慢启动阈值后，转为线性增长，避免迅速导致拥塞。
- **拥塞发生**：出现超时或重复 ACK 时，大幅减小拥塞窗口（如减半），进入拥塞避免阶段。
- **快恢复**：配合快速重传，收到 3 个重复 ACK 时，cwnd 减半（而不是降到 1），然后直接进入拥塞避免阶段。

### 机制对比表格
| 机制 | 解决的问题 | 核心手段 | 关键参数/触发条件 |
| :--- | :--- | :--- | :--- |
| **校验和** | 数据位翻转错误 | 数学计算校验 | Checksum 字段 |
| **序列号** | 乱序、重复包 | 字节流编号 | Seq Num |
| **ACK** | 数据确认 | 累积确认机制 | Ack Num |
| **超时重传** | 丢包 (无反馈) | 定时器 | RTO (Retransmission Timeout) |
| **快速重传** | 丢包 (有反馈) | 重复 ACK 触发 | 3 Duplicate ACKs |
| **流量控制** | 接收方处理不过来 | 滑动窗口通告 | 接收窗口 | 
| **拥塞控制** | 网络整体拥堵 | 动态调整发送速率 | 拥塞窗口 | 

```text
   发送方                            接收方
     |                               |
     | ------- [Seq=1, Len=100] ----> | (收到，放入缓冲区)
     | <----- [ACK=101, Win=512] ---- | (回复期望收到的下一个Seq，告知剩余窗口)
     |                               |
     | ------- [Seq=101, Len=100] --> | (丢失)
     |                               |
     | ------- [Seq=201, Len=100] --> | (收到，但乱序，缓存在本地)
     | <----- [ACK=101, Win=412] ---- | (重复ACK，期望101)
     | ------- [Seq=301, Len=100] --> | (收到，乱序)
     | <----- [ACK=101, Win=312] ---- | (重复ACK)
     | ------- [Seq=401, Len=100] --> | (收到，乱序)
     | <----- [ACK=101, Win=212] ---- | (重复ACK x3)
     | (触发快速重传)                |
     | ------- [Seq=101, Len=100] --> | (重发丢失包)
     | <----- [ACK=501, Win=612] ---- | (收到101及后续累积数据，回复新的ACK)
```

### 实战案例
- **TCP_NODELAY**：在开发即时通讯（IM）或游戏时，发现消息延迟很高。原因是开启了 Nagle 算法（合并小包发送），而我们的应用层为了实时性已经频繁发送小包。通过开启 `TCP_NODELAY` 选项禁用 Nagle 算法，虽然增加了包数量，但显著降低了延迟。

### 代码示例 (Linux Socket 设置)
```c
int sock = socket(AF_INET, SOCK_STREAM, 0);

// 禁用 Nagle 算法，降低小包延迟 (适用于 IM/游戏)
int flag = 1;
setsockopt(sock, IPPROTO_TCP, TCP_NODELAY, (char *)&flag, sizeof(int));

// 开启 Keep-Alive，防止死连接
int keepAlive = 1;
setsockopt(sock, SOL_SOCKET, SO_KEEPALIVE, (void *)&keepAlive, sizeof(keepAlive));
```

## 常见考点
1. **序列号的作用**：强调不仅仅是排序，核心是去重和重组。
2. **流量控制与拥塞控制的区别**：前者是端到端的（保护接收方），后者是全局的（保护网络链路）。
3. **快速重传触发条件**：连续收到 3 个重复 ACK，这与超时重传不同。
4. **SYN 包丢失处理**：在连接建立阶段，SYN 包也有重传机制，通常
