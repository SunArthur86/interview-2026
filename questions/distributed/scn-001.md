# 什么是GraalVM Native Image？它和传统JVM运行模式有什么区别？

🎯 本质：GraalVM Native Image是一种AOT(Ahead-Of-Time)编译技术，在构建时将Java字节码直接编译为平台相关的本地机器码可执行文件，不需要JVM运行时。

📊 Native Image vs JVM：
| 特性 | JVM(JIT) | Native Image(AOT) |
|------|----------|-------------------|
| 启动时间 | 秒级(100-2000ms) | 毫秒级(10-50ms) |
| 内存占用 | 100-500MB | 20-80MB |
| 峰值吞吐量 | 高(JIT优化) | 中(无法运行时优化) |
| 预热期 | 有(需要JIT编译) | 无(直接最优) |
| 动态特性 | 完全支持 | 受限(封闭世界假设) |
| 调试难度 | 低(标准工具) | 高(gdb/原生工具) |
| 构建时长 | 秒级 | 分钟级(3-15min) |

**实战案例**：在AWS Lambda上运行Java函数时，使用JVM模式冷启动往往超过2秒，导致经常超时。切换到Native Image后，冷启动降至50ms以内，彻底解决了Lambda的冷启动痛点，但代价是每次CI/CD流水线构建多耗费了5分钟，且遇到Joda-Time等老旧库的反射兼容性问题。

**关键代码示例**：
```java
// 需要显式注册反射，否则运行时会报错
@RegisterForReflection 
public class MyModel {
    private String name;
    // getters/setters
}
// 运行Native Image
// native-image -H:ConfigurationFilesDirectories=src/main/resources/META-INF/native-image/ ...
```

AOT编译的核心原理：
1. 封闭世界假设 - 运行时所有可达的代码在编译时都已知
2. 静态分析 - 从main方法开始做可达性分析，移除不可达代码
3. 提前初始化 - 在构建时执行类初始化器
4. 即时堆快照 - 将初始化后的堆状态序列化到镜像中

Spring Boot 3.x集成：
# pom.xml
<plugin>
    <groupId>org.graalvm.buildtools</groupId>
    <artifactId>native-maven-plugin</artifactId>
</plugin>
# 构建命令
mvn native:compile -Pnative

限制与挑战：
1. 反射/动态代理需要配置（Spring Boot AOT自动生成配置）
2. 不支持运行时类加载
3. 调试信息有限
4. 构建时间长（3-10分钟）
5. 不支持JFR的某些动态特性

适用场景：
- Serverless / FaaS（冷启动敏感）
- 微服务（内存受限的容器环境）
- CLI工具（快速启动）
- Kubernetes中的Scale-to-Zero场景
