# Saga 模式与 TCC 模式在处理长事务时的核心区别是什么？

Saga 和 TCC 都是分布式事务的补偿模式，但粒度不同。TCC (Try-Confirm-Cancel) 要求每个子业务接口都实现三个操作，Try 阶段进行资源预留， Confirm 确认执行，Cancel 取消预留。它对业务代码侵入性强，但一致性控制粒度细，无锁等待时间短，适用于性能要求高的核心系统。Saga 模式则将长事务拆分为一系列本地短事务，每个短事务都有对应的补偿操作。它没有“预留”阶段，直接提交事务，如果后续失败则按顺序执行反向补偿。Saga 的优势是实现简单，无资源锁定，但由于直接提交，可能导致“脏读”，且补偿逻辑比回滚更复杂，适用于非强一致性、流程较长的业务场景。

**实战案例**：
在机票预订行程中，TCC 适合扣款阶段（需严格冻结资金），而 Saga 适合跨航空公司的航班预订衔接（直接预订，失败则退票），避免了因第三方接口不支持预留而导致的开发困难。

**代码示例**：
```java
// Saga 补偿动作接口定义 (Java)
public interface SagaAction {
    // 正向执行：直接提交本地事务
    void execute(TransactionContext context);
    // 补偿执行：必须保证幂等性
    void compensate(TransactionContext context);
}

// 调用链：A.execute -> B.execute -> (B失败) -> B.compensate -> A.compensate
```

**对比表格**：
| 特性 | TCC 模式 | Saga 模式 |
| :--- | :--- | :--- |
| **阶段** | Try (预留), Confirm (确认), Cancel (取消) | 正向提交, 补偿回滚 |
| **资源锁定** | Try 阶段锁定资源直至 Confirm/Cancel | 无锁定，本地事务提交即释放 |
| **隔离性** | 较好，通过资源预留实现隔离 | 较差，存在脏读和并发写冲突风险 |
| **代码侵入性** | 高，需改造原有业务逻辑为三个接口 | 中，需额外编写补偿逻辑 |
| **适用场景** | 一致性要求高、并发量大的核心支付/资产系统 | 长流程、非实时强一致的业务（如供应链、旅行预订） |

## 记忆要点

- 两阶段对比：TCC三步走（Try预留）而Saga两步走（直接提交与补偿）
- 隔离性对比：TCC资源预留隔离好无脏读，而Saga直接提交易脏读
- 侵入性对比：TCC需改写三个接口侵入强，而Saga仅需写补偿逻辑
- 场景选择：核心支付选TCC（需冻结），长流程跨系统选Saga（无锁）

**展开框架：**
1. **TCC含预留阶段** — TCC含预留阶段，锁定资源强一致性；Saga无预留，直接提交。
2. **TCC需改造成三个接口** — TCC需改造成三个接口，侵入性高；Saga只需写补偿逻辑。
3. **TCC适** — TCC适用于高并发核心资产，Saga适用于长流程非强一致场景。
