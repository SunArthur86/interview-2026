# Saga 模式与 TCC 模式在处理长事务时的核心区别是什么？

Saga 和 TCC 都是分布式事务的补偿模式，但粒度不同。TCC (Try-Confirm-Cancel) 要求每个子业务接口都实现三个操作，Try 阶段进行资源预留， Confirm 确认执行，Cancel 取消预留。它对业务代码侵入性强，但一致性控制粒度细，无锁等待时间短，适用于性能要求高的核心系统。Saga 模式则将长事务拆分为一系列本地短事务，每个短事务都有对应的补偿操作。它没有“预留”阶段，直接提交事务，如果后续失败则按顺序执行反向补偿。Saga 的优势是实现简单，无资源锁定，但由于直接提交，可能导致“脏读”，且补偿逻辑比回滚更复杂，适用于非强一致性、流程较长的业务场景。

**实战案例**：
在机票预订行程中，TCC 适合扣款阶段（需严格冻结资金），而 Saga 适合跨航空公司的航班预订衔接（直接预订，失败则退票），避免了因第三方接口不支持预留而导致的开发困难。

**代码示例**：
```java
// Saga 补偿动作接口定义 (Java)
public interface SagaAction {
    // 正向执行：直接提交本地事务
    void execute(TransactionContext context);
    // 补偿执行：必须保证幂等性
    void compensate(TransactionContext context);
}

// 调用链：A.execute -> B.execute -> (B失败) -> B.compensate -> A.compensate
```

**对比表格**：
| 特性 | TCC 模式 | Saga 模式 |
| :--- | :--- | :--- |
| **阶段** | Try (预留), Confirm (确认), Cancel (取消) | 正向提交, 补偿回滚 |
| **资源锁定** | Try 阶段锁定资源直至 Confirm/Cancel | 无锁定，本地事务提交即释放 |
| **隔离性** | 较好，通过资源预留实现隔离 | 较差，存在脏读和并发写冲突风险 |
| **代码侵入性** | 高，需改造原有业务逻辑为三个接口 | 中，需额外编写补偿逻辑 |
| **适用场景** | 一致性要求高、并发量大的核心支付/资产系统 | 长流程、非实时强一致的业务（如供应链、旅行预订） |

## 技术原理

TCC 和 Saga 都是补偿型最终一致事务，但粒度和资源管理策略完全不同。核心区别在于**是否有资源预留阶段**：

- **TCC 的资源预留原理**：Try 阶段做资源预留（如冻结余额、预占库存），资源处于"中间态"——既不可被其他事务使用（隔离性好），也尚未真正消耗。Confirm 才真正扣减，Cancel 解冻释放。因为 Try 就锁定了资源，所以中间状态对外不可见（无脏读），隔离性强。代价是每个业务接口都要改造为三个，侵入大。
- **Saga 的直接提交原理**：没有预留阶段，每个子事务**直接提交本地事务**（如直接扣款、直接下单），资源立即真实变更。如果后续步骤失败，再执行反向补偿（如退款、取消订单）。因为数据已真实落库，中间状态对其他事务可见（有脏读风险），隔离性差。
- **隔离性差异的后果**：
  - TCC：Try 冻结的余额，其他事务看不到也不可用，并发安全。
  - Saga：A 子事务已扣款提交，在整体事务回滚前，其他事务可能读到"已扣款"状态并据此决策（如判断余额不足拒绝贷款），回滚后导致决策基于错误数据。
- **补偿的复杂性**：Saga 的补偿比 TCC 的 Cancel 更难——因为数据可能已被其他事务读过甚至改过，补偿时要处理"部分覆盖"的情况。比如 A 扣了 100，B 又扣了 50（基于 A 后的余额），Saga 回滚 A 时不能简单加回 100（会忽略 B 的改动），要用业务级补偿（如记录操作流水，补偿时反向记账）。

## 代码示例

```java
// TCC 模式：三接口，Try 冻结资源
@LocalTCC
public interface AccountTcc {
    @TwoPhaseBusinessAction(name = "deduct", commitMethod = "confirm", rollbackMethod = "cancel")
    // Try：冻结 amount（available -= amount, frozen += amount），资源被锁定
    boolean tryDeduct(BusinessActionContext ctx,
        @BusinessActionContextParameter(paramName = "userId") Long userId,
        @BusinessActionContextParameter(paramName = "amount") BigDecimal amount);

    boolean confirm(BusinessActionContext ctx);  // Confirm：扣减冻结（frozen -= amount）
    boolean cancel(BusinessActionContext ctx);   // Cancel：解冻（frozen -= amount, available += amount）
}

// Saga 模式：两接口，直接提交 + 补偿
public interface SagaStep {
    // 正向：直接执行并提交本地事务（如真实扣款）
    void execute(SagaContext ctx);
    // 补偿：反向操作（如退款），必须幂等
    void compensate(SagaContext ctx);
}

// Saga 协调器：编排长流程，失败时反向补偿
public class SagaOrchestrator {
    public void executeTravelBooking(TravelOrder order) {
        try {
            bookFlight.execute(order);    // 直接订票提交
            bookHotel.execute(order);     // 直接订房提交
            chargePayment.execute(order); // 直接扣款提交
        } catch (Exception e) {
            // 失败：反向补偿（已执行的步骤逐个 compensate）
            chargePayment.compensate(order);  // 退款
            bookHotel.compensate(order);      // 取消订房
            bookFlight.compensate(order);     // 退票
        }
    }
}
```

```java
// 关键：Saga 补偿必须幂等（可能被重试）
public class PaymentSagaStep implements SagaStep {
    public void compensate(SagaContext ctx) {
        String txId = ctx.getTxId();
        // 用业务流水号防重复退款
        if (refundDao.exists(txId)) return;  // 已退款，跳过
        refundDao.refund(ctx.getUserId(), ctx.getAmount());
        refundDao.record(txId);              // 记录已退款
    }
}
```

## 注意事项

- **隔离性是 Saga 的硬伤**：直接提交导致中间状态可见，对"基于余额/库存做决策"的业务有风险。若业务对隔离性敏感（如风控判断），用 TCC 的资源预留或加业务级锁，不要用裸 Saga。
- **TCC 的空回滚和悬挂**：Try 丢失但 Cancel 先到达（空回滚，没 Try 却 Cancel），或 Cancel 后 Try 延迟到达（悬挂，业务已结束又预留）。必须用事务记录表（记录 xid + 状态）防止。
- **Saga 补偿必须幂等**：网络抖动导致补偿重试，不幂等会重复退款/退票。用唯一业务流水号 + 去重表保证。
- **不能补偿的操作要避免**：如发短信、发邮件这类"不可撤销"的副作用，放事务最后执行（前面都成功才触发），或用"发件箱"模式延迟发送。
- **场景选择**：核心支付、库存扣减等需强隔离和高并发选 TCC；跨系统长流程（旅行预订、订单流转）且第三方不支持预留的选 Saga。
