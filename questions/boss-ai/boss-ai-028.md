# 【巨剧核 AI 陪伴】音视频 AI 生成工程怎么做（TTS/ASR/数字人/视频服务端编排）？

> JD 依据："负责音视频 AI 生成工程、TTS、ASR、数字人、视频生成的服务端编排。"

## 一、典型场景：实时数字人对话

```
用户："小雨今天好累呀"
   ↓
[1] ASR（语音→文本）："今天好累呀"
[2] LLM（想回复）："辛苦了，抱抱～"（情感：关心）
[3] TTS（文本→语音）：合成关心语气的语音
[4] 渲染（数字人）：嘴型+表情同步演
[5] 合成：音频+视频合成最终视频
   ↓
用户看到数字人温柔地说"辛苦了，抱抱～"
```

完整链路 5 阶段，每阶段独立服务。

## 二、五阶段编排（DAG）

```
[ASR] ──→ [LLM] ──┬─→ [TTS] ──┐
                            ├─→ [合成] ──→ 输出
                   └─→ [渲染] ─┘
```

每阶段：
- 独立服务（独立扩缩容）
- 标准接口（输入/输出协议）
- 状态持久化（断了能续）
- 失败重试（按算子级）
- 质量评估（WER/MOS/一致性）

## 三、ASR（语音识别）

```
输入：用户语音
输出：文本 + 时间戳 + 信心分

技术：
  流式 ASR：边听边出文字（实时）
  Whisper / Paraformer / 自研
  多语种 + 方言支持
  VAD（语音活动检测）：分割句末

工程：
  - 噪声环境降准（识别准确率）
  - 时间戳对齐（驱动数字人/字幕）
  - 信心分（低信心触发确认）
  - 流式 + 非流式（实时用流式，精度用非流式）

质量指标：
  WER（词错率）
  实时率（首字延迟）
```

## 四、LLM（思维/文本）

```
输入：ASR 文本 + 上下文 + 情感
输出：回复文本 + 情感标签 + 动作标签

特点（音视频场景）：
  - 输出长度要控（TTS/渲染成本与长度相关）
  - 带情感标签（驱动 TTS/渲染风格）
  - 带动作标签（角色做什么动作：笑/点头/挥手）
  - 流式生成（与 TTS 流式对接）

例：
  LLM 输出：
    text: "辛苦了，抱抱～"
    emotion: "caring"
    action: "hug"
```

## 五、TTS（文本转语音）

```
输入：文本 + 情感标签 + 角色音色
输出：语音 + 音素时间戳

技术：
  - 神经 TTS（VITS/NaturalSpeech/ChatTTS）
  - 声音克隆（少样本学角色音色）
  - 情感合成（happy/sad/angry 不同音色）
  - 流式合成（按 chunk 出，不等全文）

工程：
  - 多角色音色库（每个角色一个 voice_id）
  - 情感→声学参数映射
  - 韵律（停顿/重音）控制
  - 多语种

质量指标：
  MOS（平均意见分，主观）
  实时率
  音色相似度（vs 真人）
```

## 六、数字人渲染

```
输入：TTS 语音 + 情感 + 动作
输出：视频流

技术：
  - 2D：Wav2Lip / SadTalker（唇形同步）
  - 3D：UE/Unity 驱动（高真实感）
  - NeRF/Gaussian Splatting（新兴）

工程：
  - 嘴型同步（viseme 映射，按音素驱动）
  - 表情驱动（按情感/动作）
  - 实时渲染（GPU 加速）
  - 多角度/多场景

质量：
  - 唇形一致性
  - 表情自然度
  - 渲染帧率
```

## 七、合成

```
输入：TTS 音频 + 渲染视频
输出：最终视频（音视频同步）

工程：
  - 时间轴对齐（音画同步）
  - 编码（H.264/H.265）
  - 字幕嵌入（可选）
  - 多分辨率输出
  - CDN 分发
```

## 八、流式与低延迟

```
朴素：五阶段串行 → 端到端 30s
流式优化：
  ASR 流式 → 边识别边传 LLM
  LLM 流式 → 边生成边传 TTS
  TTS 流式 → 边合成边传渲染
  渲染流式 → 边渲染边推前端
  前端流式播放

并行：
  TTS 和渲染可并行（基于同一文本/情感）
  分块处理（第一句合成时，第二句已在 LLM）

极致低延迟（实时对话 < 1s 首 token）：
  - 全链路流式
  - 小模型快速推理
  - 边缘部署（近用户）
  - 预测性生成（预测用户要说什么）
```

## 九、情感贯穿全链路

```
LLM 输出情感标签 → 全链路透传：
  TTS 按情感调整音色（开心 = 高调轻快）
  渲染按情感调整表情（开心 = 微笑）
  动作按情感驱动（开心 = 拍手）

保证一致性：
  情感标签作为元数据贯穿
  各阶段基于同一标签工作
  避免文本说"开心"但语气表情"平淡"
```

## 十、质量评估与控制

```
每阶段独立评估：
  ASR：WER（< 5%）
  LLM：人设一致性 / 共情度
  TTS：MOS（> 4.0）/ 音色相似度
  渲染：唇形一致性 / 自然度
  合成：音画同步 / 编码质量

端到端：
  用户主观评分
  A/B 各环节模型/参数
  异常监控（音画不同步/质量劣化）
```

## 十一、底层本质

音视频 AI 生成工程的本质是**"把多模态 AI 能力编排成可靠可控的服务端流水线"**：

- 五阶段 = 模块化（各专所长）
- DAG = 可编排（依赖+并行）
- 流式 = 低延迟（边生成边推）
- 情感贯穿 = 一致体验
- 质量评估 = 可持续优化

这是把'单一 AI 能力'整合为'多模态产品'的核心工程——是数字人/虚拟主播/AI 视频/短剧生成产品的共同底座。

## 常见考点

1. **实时数字人对话（< 1s）能做到吗？**——能但要极致优化：全链路流式 + 小模型 + 边缘部署 + 预测生成；当前工业界首响应 ~500ms-1s 是顶尖水平。
2. **嘴型同步怎么做？**——TTS 输出音素 + 时间戳 → 映射到 viseme（视觉音素，嘴型分类）→ 渲染按 viseme 驱动嘴型；时间轴严格对齐。
3. **AI 生成的视频怎么审核？**——帧级审核（暴力/色情/合规）+ 音频审核（语音内容）+ 人设一致性（角色在视频里行为符合人设）+ AIGC 标记。

## 延伸：【巨剧核 AI 陪伴】音视频异步任务系统怎么设计？

> 合并自 `boss-ai-005`（相似度 72%）

> JD 依据："负责短剧视频服务端，AI 生成任务；熟悉音视频或 AI 生成异步任务。"

## 一、为什么不能同步等

```
同步请求：用户 → 服务端 → TTS+渲染+合成（30s-5min）→ 返回
后果：HTTP 超时 / 连接占用 / 无法扩容 / 体验崩
```

音视频生成的本质是**长耗时、重资源、多阶段**——与同步 HTTP 模型天然冲突，必须重构为异步任务。

## 二、任务模型

```
Task {
  task_id: 全局唯一
  type: TTS | ASR | DIGITAL_HUMAN | VIDEO_GEN | COMPOSITE
  input: { text / audio / image / params }
  status: CREATED | QUEUED | RUNNING | SUCCESS | FAILED | TIMEOUT | CANCELED
  progress: 0-100
  result_url: ...
  error: ...
  user_id, role_id, priority, idempotency_key
  created_at, updated_at, timeout_at
}
```

## 三、状态机

```
        ┌─────────┐
        │ CREATED │─── 创建并持久化
        └────┬────┘
             ↓
        ┌─────────┐
        │ QUEUED  │─── 入队等资源
        └────┬────┘
             ↓
        ┌─────────┐  ┌──────┐
        │ RUNNING │─→│ 失败 │──可重试──→ QUEUED
        └────┬────┘  └──────┘
             ↓
        ┌─────────┐
        │ SUCCESS │ / FAILED / TIMEOUT / CANCELED
        └─────────┘
```

**关键**：每次状态变更都持久化（DB + 日志），服务重启可从最后状态恢复。

## 四、算子编排（DAG）

复杂任务是多个算子的组合：
```
"用户发语音 → 数字人回复视频"
   ASR（语音转文字）
        ↓
   LLM（生成回复文本）
        ↓
   TTS（文本转语音）─────┐
        ↓                ↓
   渲染（数字人口型）   合成（视频+音频）
        ↓                ↓
        └──────→ 视频产出
```

用 DAG 描述：
- 节点 = 算子（带类型、参数、超时、重试策略）
- 边 = 依赖（前序输出 = 后序输入）
- 可并行的并行（如多语言 TTS 同时跑）

**框架**：Airflow / Temporal / 自研；AI 场景 Temporal（持久化执行）很合适。

## 五、队列与资源调度

```
多级队列：
  - VIP 队列（高优先级，预留 GPU）
  - 普通队列（按提交顺序）
  - 批量队列（低优先级，填空闲资源）

调度策略：
  - 优先级 + 公平调度（防止单用户占满）
  - 按算子类型分队列（TTS 池、渲染池分别调度）
  - 抢占：VIP 可抢占普通任务的 GPU
  - 弹性扩缩容：队列堆积触发扩容 GPU 节点
```

GPU 池化：K8s + device plugin，任务带 GPU 请求量，调度器匹配节点。

## 六、结果回传

```
[1] 回调 webhook：任务完成调用业务方注册的 callback URL
[2] WebSocket 推送：实时进度（10% → 30% → 完成）到前端
[3] 轮询兜底：前端按 task_id 查询（防推送丢失）
[4] 结果存储：对象存储（OSS/S3）+ CDN 加速下载
```

## 七、幂等与去重

```
idempotency_key = hash(input + params + user_id)
提交时检查：
  - 已有相同 key 的 SUCCESS → 直接返回缓存结果
  - 已有 RUNNING → 返回现有 task_id（不重复创建）
  - 无 → 创建新任务
```

防止用户连点、网络重传导致重复扣费、重复生成。

## 八、底层本质

音视频任务系统的本质是**"把同步的请求-响应，重构为异步的任务-状态机-回调"**：

- **解耦**：提交和执行分离（用户不等，服务慢慢算）
- **编排**：复杂流程用 DAG 描述（依赖、并行、失败重试）
- **调度**：稀缺 GPU 资源池化 + 优先级
- **可靠**：状态持久化 + 幂等 + 断点恢复

这是把'一次性函数调用'升级为'可观测、可恢复、可扩展的工业流水线'，是所有 AI 生成产品（Sora/Midjourney/数字人）共同的工程范式。

## 常见考点

1. **任务队列选 Kafka 还是 Redis 还是专门 MQ？**——Kafka 适合高吞吐流式，Redis/Redis Stream 简单灵活，RMQ/RocketMQ 强 ACK；AI 任务推荐专门的任务系统（Temporal）+ DB 状态持久化，纯 MQ 不够。
2. **进度怎么精确算？**——按 DAG 节点完成数加权（如 4 节点，每完成一个 +25%）；或在算子内部上报细粒度进度。
3. **GPU 任务怎么计费？**——按算子类型 + 执行时长 + 资源规格计费；预扣费（创建任务时锁额）+ 实际结算（完成时）+ 失败退还。


## 结构化回答

**30 秒电梯演讲：** 聊到音视频异步任务系统怎么设计，我的理解是——音视频 AI 生成是'用户请求 → 异步任务排队 → 多阶段算子编排（ASR/LLM/TTS/数字人渲染/视频合成）→ 结果回传'的长链路任务系统，核心是状态机和资源调度。打个比方，像一条影视后期流水线——剧本（用户请求）进剪辑室（任务队列），分别过配音（TTS）、字幕（ASR）、特效（数字人）、合成（视频），每一步都有专门的工位（GPU 资源），最后成品通知客户取件（回调）。

**展开框架：**
1. **任务模型** — task_id + 类型（TTS/ASR/数字人/视频）+ 输入 + 状态 + 结果
2. **状态机** — 创建→排队→运行→成功/失败/超时，可重试可取消
3. **算子编排** — DAG（如 ASR→LLM→TTS→渲染）有依赖有并行

**收尾：** 这块我在项目里也踩过坑——想深入的话，可以接着聊：任务超时怎么判定？您更想看哪个方向？
