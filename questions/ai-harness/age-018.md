# K8s 部署 Agent 服务时 HPA 可以按什么指标扩

在 K8s 中部署 Agent 服务时，HPA（Horizontal Pod Autoscaler）的扩容指标选择取决于 Agent 的瓶颈所在。除了基础的 CPU/内存，针对 LLM 应用应重点关注：

1.  **CPU/内存（基础指标）**：
    *   适用：CPU 密集型任务（如本地 Embedding 计算、数据清洗）。
    *   注意：单纯的 LLM 转发代理通常 CPU 消耗不高，以此扩容不够敏感。
2.  **自定义指标 - 请求队列长度**：
    *   原理：监控 Ingress 或 Service Mesh (如 Istio) 的 `pending_requests` 或 `queue_length`。
    *   优势：直接反映服务堆积情况，比 CPU 更能体现「处理不过来了」。
3.  **自定义指标 - P95/P99 延迟**：
    *   原理：通过 Prometheus Adapter 暴露 `http_request_duration_seconds` 分位数。
    *   策略：当 P95 延迟超过阈值（如 2s）时触发扩容，保证用户体验 SLA。
4.  **自定义指标 - 429 比例（上游限流）**：
    *   原理：监控上游 LLM Vendor 返回的 HTTP 429 (Rate Limit) 错误数。
    *   策略：当 499/429 突增时，说明并发请求打爆了上游配额，此时**不应**扩容（反而可能需要限流），除非是为了增加重试队列的吞吐。

**关键注意点**：
*   **冷启动**：模型加载或容器启动慢（Cold Start），可能导致扩容不及时。需配合 `readinessProbe` 和 KEDA (Kubernetes Event-driven Autoscaling) 提前预热。
*   **LLM 长尾延迟**：LLM 生成时间不确定，长请求会占用连接。若基于连接数扩容，需区分「长连接占用」与「高并发积压」。

### 实战案例
在某次大促期间，我们单纯基于 CPU 使用率设置了 HPA。由于 LLM 推理主要消耗网络 I/O 等待时间，CPU 一直维持在 20%，导致请求大量堆积，响应延迟飙升至 10s+。后来我们将指标改为「P99 延迟 > 3s」，系统才能在变慢的第一时间快速扩容，保住了 SLA。

### 代码示例 (HPA YAML - 基于自定义指标)
```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: llm-agent-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: llm-agent
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Pods
    pods:
      metric:
        name: http_request_duration_seconds_p95
      target:
        type: AverageValue
        averageValue: 3000m # 3000ms
```

### 边界情况补充
*   **抖动保护**：为了防止流量毛刺导致的频繁扩缩容（Flapping），需配置 `stabilizationWindowSeconds`（稳定窗口期），即指标持续异常一段时间才触发动作。
*   **扩容上限与资源配额**：必须考虑集群剩余资源，设置合理的 `maxReplicas`，避免业务流量突增时把节点资源打满，导致其他服务甚至 K8s 组件（如 CoreDNS）崩溃。
*   **GPU 资源限制**：如果 Agent 涉及本地推理，HPA 无法直接感知 GPU 利用率。通常需要自定义指标（如 DCGM Exporter 暴露的 GPU Memory Usage）来驱动扩容。

## 易错点
1.  **将 HPA 当作秒级弹性伸缩**：HPA 默认的 loop interval 是 15s，加上扩容 Pod 的启动时间，实际上响应延迟在分钟级，不适合应对秒级流量洪峰（需结合临时的过载保护）。
2.  **忽视 cooldown（冷却时间）**：只设置了扩容阈值，没设置缩容冷却，导致流量高峰一过立刻缩容，而此时缓存冷或连接池未热，引发下一轮性能抖动。

## 面试追问
1.  如果 LLM API 上游限流非常严格（如 50 QPS），下游 Agent 扩容到 10 个 Pod 反而被封禁，如何设计自适应的限流策略？（追问：基于 Client-side 限流与 HPA 联动）
2.  在大模型场景中，请求往往需要长连接（SSE），HPA 基于 QPS 或 Conns 扩容时，如何区分“正在生成”和“挂起”的连接？
3.  KEDA（Kubernetes Event-driven Autoscaling）和标准 HPA 在适用场景上有何本质区别？

## 记忆要点

- 基础指标：CPU/内存仅适用于本地计算密集型任务，LLM 转发代理通常不敏感。
- 核心指标：优先用「P95/P99 延迟」或「请求队列长度」感知服务堆积。
- 特殊指标：监控 429 错误率判断上游限流，避免无效扩容。
- 冷启动应对：配合 KEDA 或 ReadinessProbe 预热，解决扩容滞后问题。

## 结构化回答

**30 秒电梯演讲：** Agent 服务的 HPA 扩容指标要根据排队人数和等待时间加窗口，不只看营业员忙不忙。CPU/内存只适合本地计算密集型任务，LLM 转发代理通常不敏感。核心要用 P95/P99 延迟或请求队列长度感知服务堆积，还要监控 429 错误率判断上游限流避免无效扩容。最后要配合 KEDA 或 ReadinessProbe 预热，解决冷启动滞后。

**展开框架：**
1. **基础指标的局限** — CPU/内存只适用于本地 Embedding 计算、数据清洗等计算密集型任务；单纯的 LLM 转发代理 CPU 消耗不高，用基础指标扩容不够敏感。
2. **核心业务指标** — 优先用 P95/P99 延迟或请求队列长度感知服务堆积，能更真实反映用户感知的压力。
3. **特殊指标与冷启动** — 监控 429 错误率判断是否被上游限流，避免无效扩容；扩容有冷启动滞后，要配合 KEDA 或 ReadinessProbe 预热解决。

**收尾：** 一句话，HPA 要按业务真实压力扩容，不只看资源水位。您想深入聊聊天请求队列长度怎么采集，还是冷启动怎么预热？
