# 金丝雀发布要关注哪些指标

Agent 系统的发布比传统微服务更复杂，因为其表现不仅关乎“稳定性”，还关乎“智商”和“性格”。

1.  **黄金指标详解**：
    *   **延迟**：由于推理是不确定的，关注 **TTFT (Time To First Token)** 首字延迟（用户感知速度）和 **TPOT (Time Per Output Token)** 生成速度。
    *   **业务成功率**：HTTP 200 不代表成功。Agent 需追踪 **Goal Completion Rate**（例如：客服场景中，问题是否真的被解决了，还是只是回复了“不知道”）。
    *   **成本指标**：金丝雀版本若使用了 `gpt-4-turbo` 替代 `gpt-3.5`，必须监控单次调用平均 Token 数，防止成本爆炸。

2.  **Agent 特有指标**：
    *   **循环/幻觉率**：Agent 是否陷入了死循环或不断重复错误的回答。
    *   **工具调用准确率**：模型生成 JSON 参数的成功率（需 parseable）。

3.  **实战案例**：某次升级 Agent 推理逻辑后，发现 P95 延迟未变，但“用户最终放弃率”激增。排查发现新版本虽然对话更流畅，但工具调用的 JSON 格式出错率从 0.1% 上升到了 5%，导致大量流程卡在工具解析阶段。后续将“工具解析失败率”加入金丝雀自动回滚策略。

4.  **关键代码（Prometheus 指标示例）**：
```python
from prometheus_client import Counter, Histogram

# 定义业务指标
tool_success_counter = Counter('agent_tool_success_total', 'Total tool calls succeeded', ['tool_name'])
llm_token_counter = Histogram('agent_llm_tokens', 'LLM token usage per request')
goal_completion_counter = Counter('agent_goal_completion', 'User goal completion', ['status'])

# 在工具调用后记录
def call_tool(tool_name, args):
    try:
        res = execute_tool(tool_name, args)
        tool_success_counter.labels(tool_name=tool_name).inc()
        return res
    except Exception:
        # 失败不增加成功计数，或增加失败计数器
        pass
```

## 常见考点
1.  **自动化程度**：能否基于“任务完成率”自动回滚？（很难，因为该指标通常需要人工标注或延迟反馈，实时性差；通常结合错误率和成本做自动回滚）。
2.  **对比基线**：如何评估新模型的“智商”是否下降？（使用固定的 Golden Dataset（黄金数据集）进行批处理测试，计算答案相似度或准确率）。
3.  **A/B 测试策略**：如何在同一个对话流中测试不同模型？（通常按用户 ID 分流，避免同一用户在不同会话中体验不一致导致困惑）。

## 记忆要点

- 关注 TTFT（首字延迟）和 TPOT（生成速度），而非单纯的总延迟。
- 业务成功率比 HTTP 200 更重要，需追踪 Goal Completion Rate（任务完成率）。
- Agent 特有指标：工具调用准确率（JSON 解析成功率）、循环/幻觉率、单次 Token 成本。
- 金丝雀回滚策略：若工具解析失败率激增或成本超预算，即使延迟正常也应回滚。
