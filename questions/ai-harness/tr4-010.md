# PII（个人身份信息）脱敏在 RAG 系统中有哪些策略？如何在保留语义的情况下实现？

在 RAG 系统中，用户上传的文档可能包含敏感信息，必须在 Embedding 和存储前进行脱敏。策略包括：1. **正则匹配与 NER 模型**：结合规则和预训练模型（如 Microsoft Presidio）识别姓名、邮箱、电话等。2. **基于语境的替换**：将识别到的实体替换为泛化标签（如 `[PERSON]`、`[DATE]`），或者使用匿名化哈希值。**保留语义的挑战**：如果直接用 `[PERSON]` 替换所有名字，可能会丢失“甲和乙是父子”这种关系信息。解决方案是使用**一致的伪名映射**（同一个真实人物替换为同一个假名，如 Alice->Bob），或者仅在检索后的生成阶段进行动态掩码。此外，还需警惕“反推攻击”，即通过唯一属性组合反推身份，因此对组合特征也需模糊处理。

## 易错点
1. **混淆“脱敏”与“加密”**：脱敏是不可逆的过程，目的是保护隐私；加密是可逆的。在 RAG 检索阶段，如果使用加密存储，无法对密文进行语义检索。必须明确是在 Embedding 前明文脱敏，还是采用支持密文检索的特殊加密方案，但后者通常对检索效果影响巨大。
2. **忽视 Embedding 对实体识别的依赖**：如果将实体替换为 `[PERSON]`，虽然保留了部分语义，但可能会丢失该实体在向量空间中的独特特征，导致检索相关性下降。例如，“爱因斯坦的相对论”替换为“[PERSON]的相对论”可能无法精准匹配到具体文档。

## 边界情况
1. **非结构化文档中的表格数据**：简历或财报中的表格往往列名隐含信息（如“2023年薪资”），且表头可能不直接包含 PII 关键词，导致常规规则漏扫。需要结合表格结构解析进行识别。
2. **指代消歧与跨句脱敏**：在长文档中，可能出现“张三...他...”的情况。如果只替换了“张三”为“李四”，而没有处理代词“他”，或者处理不当，会导致上下文指代混乱，影响 LLM 理解或生成错误答案。

## 技术原理

RAG 场景下 PII 脱敏的核心矛盾是：**既要抹掉可识别信息防泄露，又要保留语义特征支持向量检索**。直接加密会破坏语义匹配（密文无法做相似度检索），直接用通用标签（`[PERSON]`）会丢失实体独特性导致召回率下降。正确做法是"一致性伪名映射"。

- **为什么脱敏必须在 Embedding 前**：向量检索依赖文本语义。若先 Embedding 再脱敏，向量里已经编码了原始 PII，脱敏失去意义；若加密存储，密文与 query 的语义相似度无法计算，检索失效。因此必须在明文阶段完成脱敏，再对脱敏后的文本 Embedding。
- **一致性伪名映射的原理**：把同一个真实实体在所有出现处替换为同一个假名（如所有"Alice"→"Bob"），而非全部替换为通用标签 `[PERSON]`。这样保留了"实体的一致性"和"实体间的关系"——"Alice 和 Bob 是父子"脱敏为"Carol 和 Dave 是父子"，父子关系语义仍在，检索"亲属关系"文档时仍能命中。
- **通用标签为什么不行**：把所有名字替换为 `[PERSON]`，"爱因斯坦的相对论"变成"[PERSON]的相对论"，向量空间里所有带人名的文档都高度相似，检索精度急剧下降。实体独特性的丢失让 RAG 召回质量崩塌。
- **脱敏 vs 加密的边界**：脱敏不可逆（用于检索，牺牲可逆性换语义）；加密可逆（可还原原文，但密文破坏语义匹配）。RAG 检索阶段必须用脱敏，若需要原始内容可在召回后于生成阶段动态还原（受限访问）。

## 代码示例

基于 Microsoft Presidio 的 PII 脱敏 + 一致性伪名映射：

```python
from presidio_analyzer import AnalyzerEngine
from presidio_anonymizer import AnonymizerEngine
from presidio_anonymizer.entities import OperatorConfig
import hashlib

class ConsistentPseudoAnonymizer:
    """一致的伪名映射：同一真实实体 -> 同一假名"""
    def __init__(self):
        self.analyzer = AnalyzerEngine()
        self.anonymizer = AnonymizerEngine()
        self.mapping = {}   # 真实值 -> 伪名

    def _get_pseudo(self, real_value: str, entity_type: str) -> str:
        """同一真实值始终映射到同一伪名，保留实体一致性"""
        if real_value not in self.mapping:
            # 用 hash 的稳定前几位生成可读伪名
            h = hashlib.sha256(f"{entity_type}{real_value}".encode()).hexdigest()[:6]
            self.mapping[real_value] = f"{entity_type}_{h}"  # 如 PERSON_a1b2c3
        return self.mapping[real_value]

    def anonymize(self, text: str) -> str:
        # 1. Presidio 识别 PII 实体
        results = self.analyzer.analyze(
            text=text, language="zh",
            entities=["PERSON", "EMAIL_ADDRESS", "PHONE_NUMBER",
                      "ID_CARD", "CREDIT_CARD"],
        )
        # 2. 一致性伪名映射（而非通用 [PERSON]）
        operators = {}
        for r in results:
            real = text[r.start:r.end]
            pseudo = self._get_pseudo(real, r.entity_type)
            operators[r.entity_type] = OperatorConfig(
                "replace", {"new_value": pseudo}
            )
        result = self.anonymizer.anonymize(
            text=text, analyzer_results=results, operators=operators
        )
        return result.text

# RAG 流水线：脱敏必须在 Embedding 前
def rag_pipeline_with_pii(documents: list, embed_fn, query: str):
    anonymizer = ConsistentPseudoAnonymizer()
    # 1. 入库前脱敏
    safe_docs = [anonymizer.anonymize(d) for d in documents]
    # 2. 对脱敏后文本做 Embedding（语义保留）
    vectors = [embed_fn(d) for d in safe_docs]
    vector_store.add(safe_docs, vectors)
    # 3. 查询时同样脱敏
    safe_query = anonymizer.anonymize(query)
    return vector_store.search(embed_fn(safe_query), top_k=5)

original = "Alice 的邮箱 alice@example.com，电话 13800138000，她和 Bob 是同事。"
print(rag_pipeline_with_pii.embed_example := anonymize(original))
# "PERSON_a1b2c3 的邮箱 EMAIL_d4e5f6，电话 PHONE_NUMBER_g7h8i9，
#  她和 PERSON_j0k1l2 是同事。" —— 实体关系（同事）保留，PII 抹掉
```

## 注意事项

- **脱敏 ≠ 加密**：脱敏不可逆用于检索（保留语义），加密可逆但破坏语义匹配。RAG 检索阶段必须脱敏，若需原始内容在召回后受限还原。
- **通用标签会破坏检索**：把所有名字替换为 `[PERSON]` 会丢失实体独特性，导致向量空间里所有带人名的文档高度相似，召回精度崩塌。必须用一致性伪名映射保留实体区分度。
- **脱敏必须在 Embedding 前**：先 Embedding 再脱敏等于没脱敏（向量已编码 PII）；加密存储则无法做语义检索。时序必须是"识别→脱敏→Embedding→存储"。
- **跨句指代要一并处理**：长文档里"张三...他..."只替换"张三"不处理"他"，会导致指代混乱。需做指代消解后统一脱敏，或把代词也按映射替换为对应伪名。
- **防反推攻击**：多个非敏感属性组合（如"某公司 + 某职位 + 某城市"）可能唯一定位身份。对组合特征也要做模糊化或泛化处理，不能只盯单一 PII 字段。

## 记忆要点

- 识别手段：正则匹配 + NER模型(如Microsoft Presidio)精准识别PII。
- 保留语义：用一致伪名映射(Alice→Bob)而非通用标签，保留实体关系。
- 时序关键：脱敏必须在Embedding前执行，否则无法语义检索。
- 脱敏≠加密：脱敏不可逆用于检索，加密可逆但影响语义匹配。
- 易错点：通用标签[PERSON]丢失独特特征，导致检索相关性下降。
