# MCP (Model Context Protocol) 提出的三层架构（传输/会话/能力）分别解决什么问题？它如何促进 AI 工具生态的标准化？

MCP 的三层架构包括：1) **传输层**：定义了数据如何在 Host 和 Server 之间传输（如基于 SSE 或 WebSocket），解决通信协议的兼容性问题；2) **会话层**：管理请求的生命周期、上下文状态和采样，解决多轮对话和状态同步问题；3) **能力层**：定义了 Prompts、Resources（资源）、Tools（工具）的具体 JSON Schema，解决能力发现和调用的互操作性问题。MCP 通过这种解耦，使得任何支持 MCP 的 LLM 应用都能“即插即用”地接入数据源和工具，无需为每个工具编写定制化代码，从而极大地降低了 AI 应用开发的碎片化成本。

**实战案例**：在构建企业级知识库助手时，利用 MCP 将私有 GitLab 仓库作为 Resources 接入，解决了传统 API 集成中因鉴权差异导致的连接超时问题。

**代码示例** (TypeScript - 工具注册):
```typescript
// MCP Server 定义工具能力
server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [{
    name: "fetch_git_log",
    description: "Get latest commit from repo",
    inputSchema: {
      type: "object",
      properties: { repo_id: { type: "string" } },
      required: ["repo_id"]
    }
  }]
}));
```

- **边界情况**：
  - **网络不稳定导致连接中断**：传输层必须支持断线重连和消息补发机制，否则 MCP Server 端的状态可能会与 Host 端不一致，导致“会话漂移”。
  - **资源权限变更**：如果一个 MCP Resource 的 ACL（访问控制列表）在会话期间被修改（如用户被撤销了 GitLab 读取权限），能力层应在下一次请求时返回特定的 Error Code 而非静默失败。
  - **超大资源列表**：当 Server 注册的工具或资源数量达到数千个时，`ListTools` 或 `ListResources` 请求的响应包体可能过大，需要支持分页或流式传输。

## 面试追问
1. **追问 1**：MCP 的传输层支持 SSE 和 WebSocket，在需要 AI Agent 主动向用户推送实时状态（如长时间代码编译进度）的场景下，你会优先选择哪种协议？为什么？（引导：WebSocket 双向通信能力 vs SSE 单向流）
2. **追问 2**：如果多个 MCP Server 提供了同名的 Tool（如两个 Server 都有 `search_files`），Host 端在分发请求时应该如何处理命名冲突？（引导：Namespace 隔离，优先级策略，或者提示用户选择）

## 易错点
1. **误区**：MCP 只是一个简单的 API 包装协议。
   **纠正**：MCP 的核心在于“标准化”和“上下文共享”。它不仅是传输 JSON-RPC，更重要的是 Resources 概念允许 Server 端管理 URI 指向的数据生命周期（如流式读取大文件），避免 Host 端下载全部数据。
2. **误区**：Host 和 Server 必须是一对一的关系。
   **纠正**：MCP 架构允许一个 Host 同时连接多个不同的 MCP Server（如一个连接 GitLab，一个连接 Jira），Host 负责聚合这些能力并统一暴露给 LLM，这是一个多对一的拓扑结构。

## 技术原理

MCP 的三层架构本质是把"AI 应用接入外部能力"这件事标准化，类比 USB 接口的分层设计：

- **传输层 = 物理形状（兼容性）**：定义 Host 与 Server 之间数据怎么传，基于 JSON-RPC 2.0 over SSE 或 WebSocket。SSE 是单向推送（Server → Host），适合简单的资源读取；WebSocket 是双向通信，适合长任务实时状态反馈（如代码编译进度推送）。传输层还负责断线重连和消息补发，保证网络抖动下状态一致。
- **会话层 = 数据握手（生命周期）**：管理请求的生命周期、上下文状态和采样。多轮对话里，会话层维护 session ID、请求-响应配对、超时控制，确保 Host 和 Server 的状态同步。没有会话层，每次请求都是无状态的，无法支持多轮工具调用。
- **能力层 = 设备功能（互操作）**：用 JSON Schema 统一描述 Prompts（提示模板）、Resources（资源，用 URI 指向数据）、Tools（工具，带输入参数 Schema）。Host 通过 `ListTools`/`ListResources` 发现能力，通过 `CallTool` 调用。这让任何支持 MCP 的 LLM 应用都能即插即用，无需为每个工具写定制集成代码。

生态价值在于解耦：工具开发者只需实现 MCP Server，应用开发者只需对接 MCP Host，双方不再需要 N×M 的定制集成，把碎片化成本从 O(N×M) 降到 O(N+M)。

## 注意事项

1. **Host-Server 是多对一拓扑**：一个 Host 可同时连多个 Server（GitLab、Jira、数据库），Host 聚合能力统一暴露给 LLM，别误以为是简单的一对一 API 包装。
2. **同名 Tool 要做 Namespace 隔离**：多个 Server 可能提供同名工具（如都有 `search_files`），Host 需用 Namespace 前缀（`gitlab.search_files`）或优先级策略消歧。
3. **资源权限变更别静默失败**：会话期间 ACL 被修改（如用户被撤销 GitLab 读取权限），能力层应返回明确 Error Code 而非静默失败，否则会难以排查。
4. **超大资源列表要分页**：Server 注册数千个工具/资源时，`ListTools` 响应包体过大，需支持分页或流式传输。

## 对比表

| 层级 | 解决问题 | 类比 | 关键技术 |
|:---|:---|:---|:---|
| **传输层** | 通信协议兼容 | USB 物理形状 | JSON-RPC over SSE/WebSocket |
| **会话层** | 多轮状态同步、生命周期 | USB 数据握手 | session ID、请求-响应配对 |
| **能力层** | 工具/资源互操作 | USB 设备功能 | JSON Schema（Tools/Resources/Prompts） |

| 协议 | 通信方向 | 适用场景 | 实时性 |
|:---|:---|:---|:---|
| **SSE** | 单向（Server→Host） | 简单资源读取、流式输出 | 中 |
| **WebSocket** | 双向 | 长任务进度推送、Agent 主动通知 | 高 |
