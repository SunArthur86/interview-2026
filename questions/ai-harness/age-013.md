# 你会如何设计一个多模型网关的架构

**核心组件设计**：
1.  **统一 Adapter 抽象**：定义标准化接口（如 `chat`, `completion`, `embeddings`），屏蔽不同模型厂商（OpenAI, Anthropic, 通义千问等）的 API 差异（参数格式、流式响应协议）。
2.  **配置中心**：维护模型元数据（能力列表：是否支持 Vision, 函数调用）、路由策略（租户映射到特定模型）、配额限制（RPM/TPM）。
3.  **入口层**：统一鉴权（API Key 校验）、租户路由、请求合法性校验。
4.  **核心控制层**：
    *   **优先级调度**：VIP 租户请求优先插队。
    *   **熔断与重试**：针对特定模型或厂商的错误率熔断；实现指数退避重试机制。
    *   **降级链**：主模型不可用时自动切换至备用模型（如 GPT-4 降级到 GPT-3.5）。
5.  **可观测性**：全链路 Trace ID 透传，统一记录 Token 用量与延迟，支持计费对账。
6.  **安全隔离**：密钥按租户与环境隔离隔离存储，使用 KMS 加密。

7.  **边界情况**：
    *   **非幂等性处理**：流式请求一旦中断，重试机制必须谨慎。如果模型已经开始输出，客户端重试不应导致生成两段不完整的文本。网关需支持“只允许一次正在进行的请求”或基于 Request ID 的去重。
    *   **长上下文截断**：从 GPT-4（128k）降级到 GPT-3.5（16k）时，如果直接透传 Prompt，必报错。网关需具备动态上下文截断与摘要能力。
    *   **计费误差**：不同厂商计费方式不同（有的按字符，有的按 Token），且非标准字符可能导致计费 Token 数暴增（如生僻字）。网关需做预处理估算与实际后计费的校对。

**实战案例**：
在某次大促期间，主力模型（Claude-3 Opus）突遇限流，网关配置的熔断器立即生效，在 50ms 内将流量切换至备用通道，但因备用通道未做输入长度校验，导致部分长文本请求失败。教训是：降级不仅要保证可用性，还要保证**参数兼容性**，需在 Adapter 层增加输入截断与重映射逻辑。

**代码示例**：
```python
# 伪代码：统一适配器接口
class BaseModelAdapter(ABC):
    @abstractmethod
    async def chat(self, messages: List[Dict], **kwargs) -> AsyncIterator[str]:
        pass

class OpenAIAdapter(BaseModelAdapter):
    async def chat(self, messages, **kwargs):
        # 处理 OpenAI 特有的 stream 格式
        async for chunk in self.client.chat.completions.create(...):
            yield chunk.choices[0].delta.content

class QwenAdapter(BaseModelAdapter):
    async def chat(self, messages, **kwargs):
        # 处理通义千问的 DashScope 格式差异
        messages = self._convert_qwen_format(messages)
        async for chunk in self.client.call(...):
            yield chunk.output.text
```

| 特性 | 网关侧控制 | 模型厂商侧控制 |
| :--- | :--- | :--- |
| **限流策略** | 基于租户/业务维度的 RPM/QPS 保护，保护业务预算 | 基于全局 API Key 的 TPM/RPM 保护，保护基础设施 |
| **重试机制** | 智能退避、跨厂商切换，针对网络抖动或 5xx 错误 | 通常无内置重试或仅幂等操作支持 |
| **鉴权维度** | 业务层 API Key、User ID、OAuth | 简单的 Secret Key 鉴权 |
| **计费粒度** | 支持按内部成本中心、部门、项目归档结算 | 仅按单一 API Key 消耗量 |

## 结构化回答

**30 秒电梯演讲：** 多模型网关像万能充电头，适配不同插座，内部统一管理电流和计费。核心组件三件套：统一 Adapter 屏蔽不同厂商 API 差异、配置中心管理路由和元数据、核心控制层做熔断降级限流。降级要保证参数兼容（GPT-4 降到 3.5 得截断长上下文），流式请求中断重试要谨慎避免生成两段不完整文本，计费要做预处理估算加后校对。

**展开框架：**
1. **统一 Adapter 抽象** — 定义标准化接口（chat、completion、embeddings），屏蔽不同厂商的参数格式和流式协议差异，让上游调用方无感知。
2. **配置中心与核心控制层** — 配置中心维护模型元数据（能力、路由策略、配额）；核心控制层做优先级调度、限流、熔断、降级，保障高可用。
3. **降级、幂等与计费** — 降级要保证参数兼容性；流式请求中断重试需谨慎避免不完整文本；不同厂商计费方式不同，网关需做预处理估算与后计费校对。

**收尾：** 一句话，多模型网关用统一抽象加中间件治理异构服务。您想深入聊聊 Adapter 怎么设计，还是降级时的参数兼容怎么处理？
