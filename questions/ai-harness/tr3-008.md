# 针对LLM的模型水印有哪些技术路线？它们对生成文本的质量和推理速度有何影响？

主流技术路线包括基于绿红名单的 Logits 调整和基于后处理的关键词植入。Logits 调整在采样时，利用伪随机数发生器对特定词表的 token 概率进行轻微增减，使其在统计分布上呈现出特定的模式，且不改变语义。关键词植入则是在生成的文本中强制包含特定的隐形字符或词汇。Logits 调整对文本质量影响微乎其微，因为仅微调概率分布，几乎不影响推理速度（仅需计算 Hash 函数）。它确保了生成的每一段文本都能被验证是否源自该模型，是目前防御模型滥用和版权侵犯的重要手段。

## 易错点
1. **混淆水印鲁棒性与不可见性**：往往认为调整 Logits 幅度越大水印越鲁棒，但这会导致生成文本质量急剧下降（如出现重复词或语义不通），需在两者间寻找平衡点。
2. **误认为所有后处理水印都不可见**：关键词植入容易被发现或破坏，实际应用中，基于 Logits 的统计水印才是隐形的主流，而基于修改文本结构或字符的后处理手段通常牺牲了一定的文本可读性。

## 边界情况
1. **短文本与低熵场景**：当生成的文本极短（如“好的”）或本身熵值很低（如代码片段）时，嵌入足够的统计特征非常困难，导致水印检测准确率大幅下降。
2. **对抗性攻击**：攻击者可能对水印文本进行轻微的改写、翻译或添加噪声，这些操作可能在保留原意的情况下破坏水印的统计特性，导致水印失效。

## 面试追问
1. **在分布式生成场景下（如多个 GPU 并行生成），如何保证水印 Key 的一致性和生成分布的统计特性不变？**
2. **如果攻击者通过蒸馏攻击获取了模型的私有副本，这种水印是否还能有效证明版权归属？**
3. **如何量化评估水印对模型生成质量（如 Perplexity）的具体影响？**

## 技术原理

LLM 水印的核心是**在生成时对 token 概率分布做可验证的统计扰动**，让生成的文本携带"来源指纹"，但人类阅读时几乎察觉不到。最主流的方案是 Kirchenbauer 等人提出的绿红名单水印：

- **绿红名单机制**：在每个 token 采样前，用 `(前一个 token 的 hash, 密钥)` 生成一个伪随机种子，把词表随机划分成"绿名单"和"红名单"两半（各约 50%）。采样时给绿名单 token 的 logit 加一个小偏置 δ（如 +2.0），让模型更倾向于选绿名单 token。这个偏置不影响语义（因为绿红名单是伪随机划分的，与 token 语义无关），但会让生成文本里绿名单 token 的比例显著高于 50%。
- **检测原理**：给定一段待测文本和密钥，对每个 token 重新计算它当时属于绿名单还是红名单，统计绿名单 token 的比例 $z$。若 $z$ 显著高于 50%（如达到 65%+），则判定文本含水印；若无水印，$z$ 应在 50% 附近。统计假设检验给出置信度 $p$-value。
- **为什么几乎无延迟**：水印注入只需在采样时多算一个 hash 函数和一次 logit 偏置，复杂度 $O(1)$，对推理速度无可感影响。检测也只需对文本逐 token 算 hash 统计，同样极快。
- **鲁棒性与不可见性的权衡**：偏置 δ 越大，绿名单比例越悬殊、检测越准（鲁棒），但文本质量下降越明显（强行偏离模型自然分布）。经验值 δ≈2.0 能在不可见性和检测率间平衡。

## 代码示例

绿红名单水印的最小实现（采样侧 + 检测侧）：

```python
import hashlib
import torch
import torch.nn.functional as F

class GreenRedListWatermark:
    def __init__(self, vocab_size, key=42, gamma=0.5, delta=2.0):
        self.vocab_size = vocab_size
        self.key = key
        self.gamma = gamma        # 绿名单比例（0.5 = 各一半）
        self.delta = delta        # logit 偏置
        self.green_size = int(vocab_size * gamma)

    def _get_greenlist(self, prev_token_id: int) -> set:
        """基于 (前一个 token, 密钥) 生成伪随机绿名单"""
        seed = int(hashlib.sha256(
            f"{self.key}{prev_token_id}".encode()
        ).hexdigest(), 16) % (2**32)
        gen = torch.Generator().manual_seed(seed)
        perm = torch.randperm(self.vocab_size, generator=gen)
        return set(perm[:self.green_size].tolist())

    def watermark_logits(self, prev_token_id: int, logits: torch.Tensor):
        """在采样前给绿名单 token 加偏置"""
        greenlist = self._get_greenlist(prev_token_id)
        mask = torch.zeros_like(logits)
        mask[list(greenlist)] = self.delta
        return logits + mask

    def detect(self, token_ids: list, z_threshold: float = 4.0) -> dict:
        """检测文本是否含水印（z-score 越高越可能含水印）"""
        green_count = 0
        for i in range(1, len(token_ids)):
            greenlist = self._get_greenlist(token_ids[i-1])
            if token_ids[i] in greenlist:
                green_count += 1
        T = len(token_ids) - 1
        # z-score：标准化绿名单比例偏离期望的程度
        expected = self.gamma
        z = (green_count / T - expected) / (expected * (1 - expected) / T) ** 0.5
        return {"green_ratio": green_count / T, "z_score": z,
                "watermarked": z > z_threshold}

# 采样侧：每个 token 采样前注入水印
wm = GreenRedListWatermark(vocab_size=32000, key=42)
for step in range(max_new_tokens):
    logits = model(input_ids).logits[:, -1, :]
    prev_id = input_ids[0, -1].item()
    logits = wm.watermark_logits(prev_id, logits)   # 加水印偏置
    next_id = torch.multinomial(F.softmax(logits, dim=-1), 1)
    input_ids = torch.cat([input_ids, next_id], dim=-1)

# 检测侧
result = wm.detect(generated_token_ids)
print(result)   # {'green_ratio': 0.68, 'z_score': 8.2, 'watermarked': True}
```

## 注意事项

- **短文本和低熵场景检测率下降**：水印检测依赖大数定律（绿名单比例统计），文本越短统计样本越少、检测越不准。代码、表格这类低熵文本（可选 token 少）水印信号也弱。经验上 T>200 token 才有可靠检测。
- **鲁棒性与不可见性要平衡**：偏置 δ 调太大文本会出现重复、语义不通；调太小检测率掉。δ≈2.0、γ≈0.5 是论文推荐甜点。
- **对抗攻击会破坏水印**：攻击者通过改写、翻译、同义词替换破坏绿名单 token 序列，水印信号衰减。需设计更鲁棒的编码（如基于 n-gram 而非单 token），或容忍一定攻击强度下的检测率下降。
- **密钥管理是关键**：水印检测需要原始密钥，密钥泄露后任何人都能检测甚至伪造水印。生产环境密钥要集中托管、定期轮换、按业务隔离。
