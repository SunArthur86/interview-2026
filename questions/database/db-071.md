# 如何基于Redis实现分布式锁？有哪些注意事项？

### 基于 Redis 实现分布式锁

**1. 基础实现原理**

#### (1) 加锁
使用 `SET key value NX PX timeout` 命令。
- `key`: 锁名（如 `lock:product_101`）。
- `value`: 必须是全局唯一标识（如 UUID + 线程ID），用于标识锁的持有者，防止误删别人的锁。
- `NX`: (Not Exist) 仅当 key 不存在时设置。
- `PX`: 设置过期时间（毫秒），防止死锁。

#### (2) 解锁
解锁必须保证原子性，使用 **Lua 脚本** 实现“先判断 value 是否一致，再删除”。
```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**2. 进阶方案：看门狗机制**
若业务执行时间不确定，超过锁过期时间会导致锁自动释放，引发并发冲突。
- **解决方案**：Redisson 框架实现的“看门狗”。
- **原理**：锁默认没有过期时间（或 30 秒），后台会有一个后台线程（`watchdog`），每隔 1/3 的锁时间（如 10 秒）检查锁是否还被持有，若持有则重置过期时间。

**3. 可重入锁**
基础方案不支持同一线程多次获取锁。
- **实现**：利用 Redis 的 Hash 结构存储。`Key` 为锁名，`Field` 为线程标识，`Value` 为加锁次数（计数器）。
- 加锁时，检查 Field 是否存在，存在则 `hincrby` +1，不存在则 `hset`。

**4. 集群下的 RedLock 算法**
在 Redis Cluster 或 Sentinel 架构下，主从异步复制可能导致锁丢失（如主节点加锁成功但未同步给从节点，主节点挂了，从节点晋升导致锁丢失）。
- **RedLock 算法**：
  1. 获取当前时间戳。
  2. 按顺序依次向 N（通常为 5）个独立的 Redis 节点尝试加锁。
  3. 计算获取锁消耗的时间，只有当在半数以上节点加锁成功 且 耗时小于锁过期时间时，才算加锁成功。
  4. 若失败，向所有节点发起解锁请求。

```text
     Client
       │
       │ SET resource_name unique_value NX PX 30000
       ├──────────────────────────────────────────────>
       │                                              │
       │         Redis Node 1 (Master)                │
       │         ┌─────────────────────┐              │
       │         │ Key Exists? (No)    │              │
       │         │ SET Success (OK)    │              │
       │         └─────────┬───────────┘              │
       │                   │                          │
       │                   │ Async Replication        │
       │                   ▼                          │
       │         ┌─────────────────────┐              │
       │         │ Redis Node 2 (Slave)│              │
       │         └─────────────────────┘              │
       │                                              │
       │ [Crash!] ⚡ (Master down before sync)        │
       │                                              │
       │              Failover (Promote)              │
       │                   │                          │
       │                   ▼                          │
       │         ┌─────────────────────┐              │
       │         │ Redis Node 2 (New   │ <─Lock Lost!  │
       │         │ Master)             │              │
       │         └─────────────────────┘              │
```

## 常见考点
1. **为什么 Set 时要加随机值/UUID？**
   - 为了在解锁时判断这把锁是不是自己加的，避免误删其他客户端持有的锁（如 A 业务执行太慢，锁过期自动释放，B 获得锁，A 执行完直接 Del 删掉了 B 的锁）。
2. **Watchdog 的底层原理？**
   - Redisson 使用 `netty` 的 `HashedWheelTimer` 定时任务，如果锁还持有，就重置 `expireTime`。业务线程只要没结束，看门狗就会一直续期；业务线程结束前，显式调用 unlock，此时会取消看门狗任务。
3. **Redis 宕机了怎么办？**
   - 如果是单机 Redis，宕机会导致锁信息丢失，依赖超时时间自动解锁。如果是 RedLock，需要大多数节点宕机才会导致服务不可用。
4. **Zookeeper 锁和 Redis 锁的区别？**
   - ZK：CP 模型，强一致性，性能较低，通过临时顺序节点实现，易于实现公平锁。
   - Redis：AP 模型，性能高，通过过期时间规避死锁，但极端情况下可能丢失锁。

## 记忆要点

- 原子加锁用SET key uuid NX PX，NX防并发抢锁，PX防死锁
- 解锁必用Lua脚本，因为要保证“判断UUID+删除”的原子性防误删
- 业务超时锁失效？用Redisson看门狗机制定时续期防提前释放
- 可重入锁用Hash结构存，因为要记录线程标识与加锁次数计数器
- 主从集群锁易丢，因为异步复制未完成主机宕机，所以用RedLock向多数节点抢锁
