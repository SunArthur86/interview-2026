# Redis为什么不使用红黑树，而使用跳表是什么？

Redis 的有序集合底层使用了跳表，而不是红黑树，主要原因如下：

### 1. 实现简单与调试难度
- **代码复杂度**：跳表的实现逻辑比红黑树简单得多。红黑树在插入和删除时需要复杂的平衡操作（左旋、右旋、变色），容易出错；而跳表本质上是一个多层链表，插入和删除只需要修改相邻节点的指针。
- **调试友好**：由于结构简单，跳表更容易调试和理解。

### 2. 范围查询性能
- **有序性支持**：Redis 的 ZSet 经常需要执行范围查询（如 ZRANGE）。跳表基于链表结构，遍历区间非常高效，只需找到起点后顺序遍历即可，时间复杂度为 O(log N + M)（M 为元素个数）。
- **红黑树劣势**：红黑树虽然也是有序的，但进行中序遍历实现范围查询相对复杂，且涉及树的递归或迭代遍历，缓存局部性不如跳表的链表结构好。

### 3. 并发扩展性（虽然 Redis 是单线程）
- 在多线程环境下（如 Java 的 ConcurrentSkipListMap），跳表可以通过局部锁实现更高效的并发修改，而红黑树通常需要全树锁。虽然 Redis 是单线程模型，但这体现了数据结构在并发设计上的优势。

### 4. 内存占用对比
- **平均指针数**：跳表每个节点平均包含 1/(1-p) 个指针（Redis p=0.25，即约 1.33 个指针）；而红黑树每个节点固定包含 2 个指针（左右子节点）。
- **额外开销**：红黑树为了保持平衡，每个节点通常需要存储 1 bit 的颜色位（虽然实际占用可能因对齐而更多），而跳表通过概率平衡，不需要存储额外的平衡元数据。

### 实战案例
在实现一个“实时排行榜”功能时，使用 ZSET 的 `ZRANGE` 命令配合 `WITHSCORES` 获取 Top 100 用户。由于底层是跳表，Redis 在处理范围查询时非常快，但在千万级数据下，如果一次性 `ZRANGE 0 -1` 查全量数据，会导致阻塞主线程，生产环境严禁使用。

// Redis Skiplist 插入核心逻辑示意
zskiplistNode *zslInsert(zskiplist *zsl, double score, sds ele) {
    zskiplistNode *update[ZSKIPLIST_MAXLEVEL], *x;
    // 1. 寻找插入位置 (记录每层的前驱节点)
    x = zsl->header;
    for (int i = zsl->level-1; i >= 0; i--) {
        while (x->level[i].forward && 
            (x->level[i].forward->score < score || 
             (x->level[i].forward->score == score && sdscmp(x->level[i].forward->ele,ele) < 0))) {
            x = x->level[i].forward;
        }
        update[i] = x;
    }
    // 2. 随机生成层数并创建节点... (省略)
    // 3. 调整指针插入节点
    for (int i = 0; i < level; i++) {
        x->level[i].forward = update[i]->level[i].forward;
        update[i]->level[i].forward = x;
    }
    return x;
}

### 常见考点
1. **ZSet 的底层组合**：Redis ZSet 同时使用了「压缩列表」和「跳表+字典」，分别在什么情况下使用？（元素数量 < 128 且所有成员长度 < 64 字节时用 ziplist，否则用 skiplist）。
2. **时间复杂度**：跳表查找、插入、删除的平均时间复杂度是多少？（O(log N)），最坏情况是多少？（O(N)，取决于概率，但极低概率）。
3. **为什么层数是 32**：Redis 定义 ZSKIPLIST_MAXLEVEL=32，基于概率计算，层数超过 32 的概率极低（2^32 个元素才可能达到），足以覆盖海量数据。
