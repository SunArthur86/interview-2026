# Redis 主从复制+哨兵（Sentinel）如何实现高可用？故障转移流程是什么？

【主从复制】
- **作用**：读写分离、数据备份、故障恢复。
- **全量同步**：从节点初次连接或主从断开时间过长，主节点执行 `bgsave` 生成 RDB 快照文件发送给从节点，从节点加载 RDB 恢复数据至内存。
- **增量同步**：主节点将后续的写命令记录到 Replication Buffer（复制积压缓冲区）并记录 replid 和 offset。从节点恢复完成后，发送 `PSYNC <replid> <offset>` 请求部分同步。
- **无盘复制**：为了提升主节点性能，主节点可以不落盘生成 RDB，而是直接通过网络发送给从节点。

【哨兵 4 大职责】
1. **监控**：持续监控主从节点是否存活（PING 命令）。
2. **通知**：通过 API 或脚本通知运维人员故障信息。
3. **自动故障转移**：主节点挂掉后，自动将从节点升级为主节点。
4. **配置中心**：客户端连接哨兵获取当前主节点的地址（避免客户端硬编码主节点 IP）。

【故障转移详细流程】
```text
   [Sentinel集群]           [Redis 主从集群]
   ┌─────┐                 ┌──────┐ (旧主)
   │ S1  │────────Ping───▶ │ Master│ (Down)
   └──┬──┘                 └───┬──┘
      │ SDOWN(主观下线)         │ 断开
      │                        │
   ┌──▼───────────────────────┼─────────────┐
   │ Sentinel 间交换信息        │             │
   │ 发现 > Quorum 确认       │             │
   └──┬───────────────────────┼─────────────┘
      │ ODOWN(客观下线)         │
   ┌──▼───────────────────────▼─────────────┐
   │  选举 Leader Sentinel (Raft 协议)     │
   │  Leader 负责执行故障转移               │
   └──┬─────────────────────────────────────┘
      │
   ┌──▼─────────────────────────────────────┐
   │ 1. 选新主: 优先级 > offset > runid    │
   │ 2. 执行切换: 新主执行 SLAVEOF NO ONE  │
   │ 3. 从节点同步: 其他从节点指向新主      │
   │ 4. 通知客户端: 更新主节点地址映射      │
   │ 5. 旧主回归: 若恢复，变成新主的从节点  │
   └─────────────────────────────────────────┘
```
1. **主观下线（SDOWN）**：单个哨兵认为主节点“主观”下线（配置 `down-after-milliseconds` 毫秒无响应）。
2. **客观下线（ODOWN）**：当超过 `quorum`（法定人数）个哨兵确认主节点下线，才升级为“客观”下线，避免误判（如网络分区）。
3. **选举 Leader 哨兵**：确认 ODOWN 后，哨兵集群通过 Raft 协议选举出一个 Leader 哨兵来负责执行具体的切换操作。
4. **选新主**：Leader 哨兵在从节点中筛选新主，优先级如下：
   - 优先级（`slave-priority`/`replica-priority`）越高越好。
   - 复制偏移量（`offset`）越大越好（数据越新）。
   - Run ID 越小越好。
5. **执行切换**：
   - 新主：执行 `SLAVEOF NO ONE`。
   - 其他从：执行 `SLAVEOF new_master_ip`。
   - 通知：通过 Pub/Sub 机制通知客户端更新连接。

【架构要求】
- 至少 3 个哨兵节点（防止哨兵自身单点故障）。
- 建议哨兵部署在不同的物理机器上（防止机房断电）。
- 客户端需要集成 Sentinel 适配器（如 JedisSentinelPool, RedisSentinel）。

## 常见考点
1. **哨兵脑裂导致的数据丢失问题及解决方案？**
   - **现象**：主节点因为网络分区与从节点断连，但自身未挂；从节点被选为新主；客户端还在写旧主；分区恢复后旧主变从，数据被覆盖。
   - **解决**：配置 `min-replicas-to-write` 和 `min-replicas-max-lag`。要求主节点至少有 N 个从节点连接且延迟小于 M 秒，否则拒绝写请求。
2. **为什么哨兵数量推荐奇数？**
   基于 Raft 协议，奇数节点更容易满足“大多数”（Quorum）的条件，且在节点数相同的情况下（如 3 vs 4），容错能力都是 1，奇数更节省资源。
3. **主从切换时，客户端如何感知？**
   客户端订阅 Sentinel 的 `+switch-master` 频道，或者定期轮询 Sentinel 获取当前 master 地址，自动重连。

## 记忆要点

- 故障转移流程：主观下线(SDOWN) -> 客观下线(ODOWN) -> Raft选主 -> 执行切换
- 新主选举规则：依次比较优先级、复制偏移量(offset)和RunID
- 哨兵数量推荐奇数，基于Raft协议更容易满足大多数(Quorum)且节省资源
- 脑裂防范：配置min-replicas-to-write要求主节点至少有N个从节点保持连接
