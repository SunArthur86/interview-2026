# 什么是InnoDB（B+树）？

InnoDB 是 MySQL 的默认存储引擎，其底层使用 B+ 树来组织数据。

### 1. B+ 树结构特点
- **Page（页）**：InnoDB 的基本存储单位是页，默认大小为 16KB。B+ 树的每个节点对应一个页。
- **非叶子节点**：只存储键值和指针，不存储实际数据。这意味着一个页能存更多的索引项，树的高度更低（通常 2-4 层），减少磁盘 I/O。
- **叶子节点**：存储完整的键值和**行数据**（聚簇索引）。所有叶子节点通过双向链表连接，形成有序链表，极大优化了范围查询的性能。

### 2. 聚簇索引（Clustered Index）
- InnoDB 的主键索引就是聚簇索引。**B+ 树的叶子节点直接存储了整行数据**。
- 这意味着“通过主键查询”效率最高：找到主键索引节点，就直接拿到了数据。

### 3. 适用场景
1.  **事务支持**：需要 ACID 特性的业务（如转账、订单）。
2.  **高并发更新**：支持行级锁，处理多重并发更新请求。
3.  **外键约束**：唯一支持外键的引擎。
4.  **崩溃恢复**：具备崩溃恢复能力，保证数据安全。

```text
      │  (Internal Pages)     │      (高度低，扇出大)
      └───────────┬───────────┘
                  │
      ┌───────────┴───────────┐
      │       Leaf Pages      │  <-- 存储键值 + 实际数据行
      │   (Data Pages)        │      (双向链表连接)
      └───────┬───────┬───────┘
              │       │
      ┌───────▼───┐ ┌─▼────────┐
      │ Page 1    │ │ Page 2   │ ...
      │ [Data]    │ │ [Data]   │
      └───────────┘ └───────────┘
      Next_Ptr <===> Prev_Ptr
```

### 5. 补充细节

#### 辅助索引
- **结构**：叶子节点存储的是**索引列的值**和**主键值**，而不是整行数据。
- **回表**：通过辅助索引查询时，先找到主键 ID，再回聚簇索引查找完整数据。如果是覆盖索引（查询的字段都在索引树上），则无需回表。

#### 联合索引与最左前缀
- 对于索引 `(name, age)`，B+ 树是先按 `name` 排序，`name` 相同时再按 `age` 排序。
- **最左前缀原则**：查询条件必须包含最左侧列才能利用索引（如 `where name='xx'` 或 `where name='xx' and age=10`），直接查 `where age=10` 无法走索引。

#### 页分裂与合并
- **页分裂**：当插入数据导致页满时，InnoDB 会将页分裂为两个，降低页填充率（默认 15/16 填充）。如果主键是递增的（如自增 ID），数据总是追加在后面，减少页分裂；如果是随机 UUID，会导致频繁的页分裂和碎片。
- **页合并**：当删除数据导致页填充率低于阈值（默认 1/2）时，InnoDB 会尝试合并相邻页。

#### 实战案例
**踩坑**：在高并发写入场景下，曾使用随机 UUID 作为主键，导致 InnoDB 页分裂严重，磁盘 I/O 飙升，且 Buffer Pool 命中率急剧下降，后改用雪花算法生成的递增 ID 后性能提升显著。

#### 核心数据结构对比
| 特性 | B+ 树 (InnoDB) | B 树 | Hash 索引 |
| :--- | :--- | :--- | :--- |
| **数据存储** | 非叶子节点不存数据，叶子存数据 | 所有节点都存数据 | 仅存储 Hash 值+指针 |
| **查询速度** | 稳定（查询次数由树高决定） | 不稳定（根节点命中快） | O(1) (仅限等值) |
| **范围查询** | 极好（叶子节点双向链表） | 一般（需中序遍历） | 不支持 |
| **I/O 次数** | 少（树矮胖） | 多（树较高） | 少 |

## 常见考点
1.  **为什么使用 B+ 树而不是 B 树或 Hash？**
    - **相比 B 树**：B+ 树非叶子节点不存数据，树更矮胖，磁盘 I/O 更少；叶子节点链表适合范围查询。
    - **相比 Hash**：Hash 仅支持等值查询，不支持范围查询和排序。
2.  **聚簇索引 vs 非聚簇索引**：
    - InnoDB 是聚簇索引（数据文件就是索引文件，一张表只能有一个聚簇索引）。
    - MyISAM 是非聚簇索引（索引文件和数据文件分离，叶子节点存地址，主键索引和辅助索引结构上无本质区别）。
3.  **主键设计建议**：
    - 优先使用**自增 ID**或**单调递增**的 ID，避免页分裂。
    - 避免 UUID（特别是随机无序的），会导致大量的随机 I/O 和页分裂。

## 记忆要点

- 结构：非叶子节点只存索引使树更矮胖，叶子节点存数据且通过双向链表相连
- 聚簇与非聚簇：InnoDB主键即数据(聚簇)，辅助索引存主键需回表(MyISAM纯非聚簇)
- 范围查询：相比B树，B+树只需遍历叶子链表，所以范围查询极快
- 主键设计：推荐自增主键，因为顺序插入避免页分裂，UUID随机写导致碎片多
- 页大小：InnoDB默认页大小为16KB，3层B+树通常可支撑千万级数据
