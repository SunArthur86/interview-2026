# 什么是索引优化的方法？

### 索引区分度

索引区分度指某列不同值的比例（`COUNT(DISTINCT col) / COUNT(*)`）。建立联合索引时，应将区分度高的字段（筛选性好的字段）放在前面，以便快速缩小扫描范围。

### 索引优缺点

**优点**：
- 大大加快检索速度，减少扫描数据量。
- 唯一索引可保证数据唯一性。
- 加速排序和分组操作（利用索引有序性）。

**缺点**：
- 占用额外空间（B+ 树节点）。
- 降低增删改（INSERT/UPDATE/DELETE）效率，需维护索引结构（页分裂、合并）。

### 建立索引注意事项

1. 索引不是越多越好，需平衡读写性能。
2. 数据量小的表无需索引。
3. 频繁更新的列不宜建索引。
4. 区分度低的字段（如性别，只有男/女）不宜建单独索引（除非是联合索引前缀或特定场景覆盖索引）。

### 优化方法

1. **前缀索引**：对长字符串（如邮箱、URL）取前 N 个字符建立索引，减少索引大小。
   - 语法：`INDEX(col_name(10))`
   - 缺点：无法利用索引进行排序（ORDER BY）和覆盖索引扫描。
2. **覆盖索引**：索引包含所有 `SELECT` 查询的字段，避免回表（Select 字段都在索引中）。
   - 是性能优化的首选手段之一。
3. **主键自增**：
   - 减少页分裂，提高插入性能。
   - UUID 等随机主键会导致频繁的页分裂和数据移动，降低写入性能并增加碎片。
4. **最左前缀**：联合索引查询时充分利用最左匹配原则，调整 SQL 中的字段顺序以匹配索引。
5. **避免索引失效**：
   - 避免在索引列上做运算（如 `WHERE age + 1 = 10`）。
   - 避免使用函数（如 `WHERE YEAR(create_time) = 2023`）。
   - 避免隐式类型转换（如字符串列存了数字，查询时传数字）。
   - `LIKE` 查询以 `%` 开头会导致全索引扫描失效。

### B+ 树索引结构简图

```text
          [Root Node]
         /            \
    [Non-Leaf]      [Non-Leaf]
    /  |   \        /   |   \
 [Leaf][Leaf][Leaf] [Leaf]... [Leaf]
  |      |      |    |
[Data] [Data] [Data] [Data] (有序链表，包含主键和索引列)
```

### 常见考点
1. **聚簇索引 vs 非聚簇索引**：InnoDB 中主键是聚簇索引（叶子节点存数据行），二级索引叶子节点存主键值。回表就是拿着二级索引查到的主键去聚簇索引查数据。
2. **索引下推（ICP）**：在联合索引中，对不满足条件的记录在引擎层直接过滤，减少回表（详见 db-029）。
3. **索引失效场景**：重点记忆 `OR` 连接非索引列、`LIKE '%xx'`、类型转换、函数运算这几种常见情况。

### 实战深化

**实战案例**：
某电商订单表使用 UUID 作为主键，在高并发写入场景下出现严重的页分裂和磁盘 I/O 飙升。将主键改为基于 Snowflake 算法的趋势递增数字 ID 后，InnoDB 的插入性能提升了 5 倍以上，且磁盘空间碎片减少了 30%。

**代码示例**：
```sql
-- 优化案例：建立覆盖索引避免回表查询状态
-- 原语句（回表）：SELECT status FROM orders WHERE user_id = 123;
-- 优化后（覆盖索引）：
ALTER TABLE orders ADD INDEX idx_user_status (user_id, status);
-- 此时查询只需扫描索引，无需回表查聚簇索引
```

**方案对比**：

| 策略 | 适用场景 | 优势 | 劣势 |
| :--- | :--- | :--- | :--- |
| **前缀索引** | Blob/Text/长字符串列 | 节省大量存储空间 | 无法覆盖索引、无法排序 |
| **覆盖索引** | 高频查询、避免回表 | 性能极高，无需回表 | 索引字段多，占用空间大 |
| **联合索引** | 多条件组合查询 | 减少索引数量，利用最左前缀 | 需注意字段顺序，维护成本略高 |
| **主键自增** | 高并发写入表 | 减少页分裂，写入性能好 | 数据导入或分库分表时需处理 ID 冲突 |

## 记忆要点

- 黄金法则：联合索引把区分度高（筛效益好）的列放前面，充分利用最左前缀匹配
- 高效手段：建立覆盖索引（SELECT 的字段全在索引中），彻底避免回表返查聚簇索引
- 主键选择：必须用自增 ID，避免 UUID 导致的频繁页分裂与磁盘碎片，提升写入性能
- 索引失效四大天坑：索引列做运算/函数、隐式类型转换、LIKE 以 % 开头、OR 连接非索引列
