# 什么是缓存穿透？如何预防和解决？

### 什么是缓存穿透？

缓存穿透（Cache Penetration）：查询一个**数据库中根本不存在**的数据，导致缓存永远 miss，每次请求都直接打到数据库。如果遭遇恶意攻击（用大量随机构造的不存在的 ID 进行高频查询），极易压垮数据库。

**与击穿/雪崩区别**：
- **穿透**：数据**不存在**（缓存和 DB 都没有）。
- **击穿**：单个**热点 key** 过期（数据存在），瞬间海量请求击穿缓存打向 DB。
- **雪崩**：**大量 key** 同时过期（数据存在），或缓存服务宕机，流量全部打向 DB。

### 缓存穿透架构对比

**正常流程**：
```text
Client -> Cache (Hit) -> Return Data
            |
         (Miss)
            |
            v
         DB -> Write Cache -> Return Data
```

**穿透流程**：
```text
Client -> Cache (Miss) -> DB (Not Found) -> Return Null (No Cache Write)
    ^                         |
    |________(Repeat)_________|
```

### 解决方案

1. **缓存空值**
   - 当 DB 查询为 null 时，也将该 key 缓存起来（Value 设为 null），并设置较短的 TTL（如 30秒-5分钟）。
   - **优点**：实现简单，防御有效。
   - **缺点**：如果恶意构造的 key 非常多，会占用大量缓存内存；且当 DB 真正插入了该 key 后，需要主动清除对应的空 key 缓存。

2. **布隆过滤器**
   - 在请求到达 DB 之前，先经过布隆过滤器。布隆过滤器是一个位数组，能高效判断某个元素**一定不存在**或**可能存在**。
   - **原理**：将所有可能存在的 key 哈希映射到位数组中。查询时，若 key 映射的位有一位为 0，则 key 一定不存在，直接拦截。
   - **优点**：内存占用极省（亿级 key 仅需几十 MB）。
   - **缺点**：有误判率（Hash 冲突导致说存在实际不存在），无法删除元素（除非使用计数布隆过滤器）。

3. **接口限流 + 参数校验**
   - 在业务层进行基础防御：如 ID 必须是正整数、长度限制等，直接拦截明显非法的请求。
   - 对高频访问的 IP 或用户进行限流。

4. **互斥锁**
   - 当缓存 miss 时，只允许一个线程去查 DB，其他线程等待。虽然主要解决击穿，但也能缓解穿透时的并发压力。

### 布隆过滤器工作流图

```text
+----------------+      1. Query Key(k)      +---------------------+
|   Client App   | -----------------------> |   Bloom Filter      |
+----------------+                           +----------+----------+
                                                       |
                                        2. Check Bits in Array
                                                       |
                                     +------------------+------------------+
                                     |                                     |
                               Any Bit = 0?                           All Bits = 1?
                                     |                                     |
                                     v                                     v
                           Return (Not Exist)                 3. Query Redis/DB
                                     |                                     |
                                     +------------------+------------------+
                                                        |
                                                        v
                                                 Return Data
```

### 实战深化

**实战案例**：
曾遇到恶意爬虫攻击，不断请求不存在的商品 ID（如递增的随机数字），导致 MySQL CPU 飙升至 100%。由于无法预先遍历所有商品 ID，我们采用了**Guava 布隆过滤器**，在应用启动时加载现有 ID 集合，有效拦截了 99% 的无效请求，DB 负载瞬间恢复正常。

**代码示例（Java - Guava BloomFilter）**：
```java
// 初始化布隆过滤器（预计插入100万，误判率0.01%）
BloomFilter<Long> filter = BloomFilter.create(Funnels.longFunnel(), 1000000, 0.0001);

// 1. 将所有有效ID预热到过滤器
for (Long id : getAllValidIdsFromDB()) {
    filter.put(id);
}

// 2. 查询拦截
public Product getProduct(Long id) {
    if (!filter.mightContain(id)) {
        // 一定不存在，直接返回，不查DB
        return null; 
    }
    // 可能存在，走缓存/DB逻辑
    return repository.findById(id);
}
```

**方案对比**：

| 方案 | 适用场景 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **缓存空值** | 数据量不大、Key 可穷举或重复率高 | 实现最简单，无需额外组件 | 占用内存，可能被恶意Key撑爆 |
| **布隆过滤器** | 海量数据、Key 随机性强（如ID、URL） | 内存占用极小，拦截效率高 | 存在误判，不支持删除元素，需维护集合 |
| **接口限流** | 恶意攻击、高频异常流量 | 保护系统整体稳定性 | 无法区分业务真伪，可能误伤正常用户 |
| **互斥锁** | 缓存击穿/穿透并发保护 | 防止并发打到 DB | 降低了并发吞吐量，增加了响应时间 |

## 记忆要点

- 概念对比：穿透查根本不存在的数据；击穿查热点单Key过期；雪崩是大面积Key同时过期
- 解决方法一：缓存空值（设置较短 TTL），实现极其简单但可能浪费内存
- 解决方法二：布隆过滤器拦截（前置防守），极其省内存，存在误判率但绝对不会漏报
- 辅助防御：做好接口层的参数合法性校验（如限制非法 ID 格式）直接拦截恶意请求
