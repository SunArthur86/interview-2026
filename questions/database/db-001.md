# MySQL什么使用B+树来作索引，它的优势什么？

MySQL 使用 B+ 树作为索引结构，主要原因在于其对于磁盘 I/O 和范围查询的优化。

**B+ 树的优势**
1.  **磁盘 I/O 少（矮胖）**：B+ 树内部节点只存储索引键，不存储数据，这使得单个节点能容纳更多的索引项（通常由页大小决定，如 16KB），从而降低树的高度。查询数据时，磁盘 I/O 次数极少（通常 3 次以内）。
2.  **范围查询性能强**：B+ 树的所有叶子节点使用双向链表连接。一旦找到范围的起点，只需遍历叶子节点链表即可获取所有数据，非常适合 `> < between` 等操作。
3.  **查询稳定**：所有查询都要走到叶子节点，查询路径长度相同（树的高度），性能稳定。
4.  **全表扫描快**：由于数据都集中在叶子节点，遍历叶子节点链表即可完成全表扫描，极快。

**B+ 树结构示意图**
```
      [根节点 (Page)]
      /      |      
   [P1]     [P2]     [P3]  (内部节点：仅存索引键)
   /  \     /  \     /  \
 [L1][L2]-[L3][L4]-[L5][L6] (叶子节点：存数据/主键+双向链表)
  ^   <->   <->   <->   ^
  |_______________________|
```

**与其他索引对比**
*   **Hash 索引**：仅适合等值查询（=, IN），不支持范围查询，不支持利用索引排序。
*   **B-Tree**：数据分布在节点各处（包括内部节点），范围查询需要中序遍历，涉及随机 I/O，且利用节点指针遍历效率不如 B+ 树的顺序链表。
*   **二叉树**：树高过高，磁盘 I/O 次数多，极度不适合磁盘存储。

**索引分类补充**
*   **聚簇索引**：数据和 B+ 树结构在一起（通常是主键）。叶子节点就是整行数据页。一张表只能有一个聚簇索引。
*   **非聚簇索引（二级索引）**：叶子节点存储索引键值 + 主键值。查询通常需要“回表”（根据主键去聚簇索引查完整数据）。在 MySQL 5.6+ 支持索引下推优化。

**实战案例**
在设计订单表时，若使用随机 UUID 作为主键，会导致 InnoDB 频繁发生页分裂，插入性能下降且磁盘碎片增加。改为自增 ID 或雪花算法生成的有序 ID 后，数据顺序追加，页分裂概率极低，写入 QPS 提升显著。

**代码示例**
```sql
-- 聚簇索引（主键）
CREATE TABLE users (
    id BIGINT AUTO_INCREMENT PRIMARY KEY, -- B+树叶子节点存储完整行数据
    name VARCHAR(255),
    age INT
);

-- 二级索引（辅助索引）
CREATE INDEX idx_name ON users(name); -- 叶子节点存储 name 和 id（主键）

-- 范围查询利用 B+ 树叶子节点链表
SELECT * FROM users WHERE id > 100 AND id < 200; -- 极快，顺序 I/O
```

**对比表格**

| 特性 | B+ 树 | B-Tree | Hash 索引 |
| :--- | :--- | :--- | :--- |
| **存储结构** | 非叶子节点仅存索引，叶子存数据+链表 | 所有节点存数据和索引 | Hash桶 + 链表/数组 |
| **查询类型** | 支持全类型（=, >, <, Like前缀） | 支持全类型 | 仅支持等值 (=, IN) |
| **范围查询** | 极快（遍历叶子链表） | 一般（需要中序回溯） | 不支持 |
| **I/O 次数** | 少（树矮胖） | 较多（树相对较高） | 通常 1 次 |
| **数据稳定性** | 稳定（都必须到叶子节点） | 不稳定（可能在根节点找到） | 不稳定 |

## 常见考点
1.  **聚簇索引的叶子节点存什么？**
    存的是整行数据。而在非聚簇索引（二级索引）中，叶子节点存的是索引列的值和主键值。
2.  **为什么建议使用自增主键？**
    B+ 树插入数据时，如果主键是自增的，新数据总是追加到当前叶子节点的末尾，页分裂概率低，且写入是顺序 I/O。如果是随机主键（如 UUID），可能导致频繁的页分裂和页移动，产生大量磁盘碎片和随机 I/O。
3.  **什么是回表？**
    先通过二级索引查到主键，再通过主键去聚簇索引中查找完整行数据的过程。可以通过覆盖索引来避免回表。
4.  **B+ 树一个节点通常多大？**
    通常设置为数据库页的大小（InnoDB 默认为 16KB），这样一次 I/O 就能读入一个节点。


## 记忆要点

- 非叶子节点不存数据仅存索引，所以单页容纳多索引，树形矮胖致IO极少
- 叶子节点双向链表相连，使得范围查询和排序只需顺序遍历极快
- 聚簇索引存整行数据，二级索引存主键；非主键查询往往需要回表
