# 什么是两阶段提交的过程？

两阶段提交是 MySQL 在开启 binlog 时，为了保证 redo log（物理日志，用于崩溃恢复）和 binlog（逻辑日志，用于主从复制）逻辑一致而采用的机制。过程如下：
1. **Prepare 阶段**：InnoDB 将 redo log 写入，并标记为 Prepare 状态。
2. **Write Binlog 阶段**：MySQL 将 binlog 写入磁盘。
3. **Commit 阶段**：InnoDB 将 redo log 标记为 Commit 状态。
这种机制利用了 XA 事务规范，确保了 redo log 和 binlog 要么都成功，要么都失败，防止了因日志不一致导致的主从数据差异。

**2PC 流程简图：**
```text
┌──────────┐          ┌──────────┐          ┌──────────┐
│ InnoDB   │          │MySQL Server│          │  Binlog  │
│ (Engine) │          │ (Coordinator)│         │   Log    │
└─────┬────┘          └─────┬────┘          └─────┬────┘
      │                    │                    │
      │ 1. Prepare Log     │                    │
      │ (写入 redo 并标记) │                    │
      │───────────────────▶│                    │
      │                    │                    │
      │                    │ 2. Write Binlog    │
      │                    │───────────────────▶│
      │                    │                    │
      │                    │ 3. Commit Log      │
      │ 4. 标记 Commit     │◀───────────────────│
      │◀───────────────────│                    │
      │                    │                    │
```

| 日志类型 | 物理日志/逻辑日志 | 写入方式 | 主要用途 | 循环写 |
| :--- | :--- | :--- | :--- | :--- |
| **Redo Log** | 物理日志（记录页修改） | 循环写（固定大小） | 崩溃恢复 | 是
|
**Binlog** | 逻辑日志（记录 SQL 或行事件） | 追加写（无限追加） | 主从复制、数据恢复 | 否

**实战案例：**
排查主从数据不一致时，发现从库比主库多了一条数据。**原因**：主库写完 Redo Log (Prepare) 后机器宕机，Binlog 未写入，但从库通过某种非正常途径获取了 Binlog（或恢复逻辑错误）。2PC 机制正是为了防止“Redo 提交但 Binlog 未写入”或反之的情况。

**关键代码（查看事务状态）：**
```sql
-- 查看 InnoDB 状态，确认事务处于 PREPARE 还是 COMMIT 状态
SHOW ENGINE INNODB STATUS;
-- 在事务部分可以看到TRANSACTIONS，其中会有 Trx read view和 Trx id
-- 若崩溃恢复后，MySQL会依据 XID 检查 Binlog 是否完整，决定 Rollback 还是 Commit
```

## 常见考点
1. **为什么要先写 redo log 再写 binlog？**
   - 顺序是 redo log (prepare) -> binlog -> redo log (commit)。如果反了，比如写完 binlog 但 redo log 还没 commit 就崩溃了，重启后因为没有 prepare 记录，事务回滚，但从库已经通过 binlog 执行了该事务，导致主从不一致。
2. **Crash-safe 是指什么？**
   - 指 MySQL 崩溃重启后，能够利用 redo log 恢复未提交但已刷盘的数据，或者利用 XID 判断 binlog 是否完整，从而保证事务的原子性和持久性。
3. **redo log 和 binlog 的区别？**
   - **存储内容**：redo log 是物理日志（记录页修改），binlog 是逻辑日志（记录 SQL 语句或行变化）。
   - **写入方式**：redo log 是循环写（固定大小，覆盖），binlog 是追加写（不覆盖，需定期清理）。
   - **用途**：redo log 用于崩溃恢复，binlog 用于主从复制和数据恢复。
