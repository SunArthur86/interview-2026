# 怎么查看一个SQL语句是否使用了索引进行检索是什么？

### 如何查看 SQL 是否使用索引

**1. 使用 EXPLAIN 命令**
在 SQL 语句前加上 `EXPLAIN` 关键字，MySQL 会返回执行计划而不是执行结果。

**2. 关注 `type` 字段**
执行计划中的 `type` 字段指示了访问类型（性能从高到低）：
*   `system` / `const`：单表中最多只有一行匹配，主键或唯一索引查询。
*   `eq_ref`：唯一索引扫描，对于每个索引键，表中只有一条记录与之匹配（常见于 JOIN）。
*   `ref`：非唯一索引扫描，返回匹配某个单独值的所有行。
*   `range`：只检索给定范围的行，使用索引选择范围（如 `BETWEEN`, `>`, `<`）。
*   `index`：遍历索引树（Full Index Scan），通常比 ALL 快，因为索引文件通常比数据文件小。
*   `ALL`：全表扫描，**未使用索引**，效率最低。

**3. 关注 `key` 字段**
`key` 列显示了实际使用的索引名称。如果为 `NULL`，则表示未使用索引。`possible_keys` 列显示可能被使用的索引，对比 `key` 可以判断是否发生了索引失效。

**4. 其他关键指标**
*   `rows`：预估需要扫描的行数。值越小越好。
*   `Extra`：包含额外信息。
    *   `Using index`：表示使用了覆盖索引，不需要回表查询（性能好）。
    *   `Using filesort`：MySQL 需要额外的排序操作（性能差，通常需要在 ORDER BY 字段上加索引）。
    *   `Using temporary`：使用了临时表处理查询（性能极差）。

**索引执行流程对比图**：
```text
全表扫描 (ALL):
[ 数据表聚簇索引 ]
   |
   +----> 逐行扫描所有叶节点 -> 筛选条件 -> 返回结果
   (磁盘 I/O 高，效率低)

索引查询 (ref/range):
[ 辅助索引 B+ Tree ] -> [ 聚簇索引 (回表) ]
   |                           |
   +-> 快速定位到 Key         +-> 根据 PK 定位数据行
   (磁盘 I/O 低，效率高)

覆盖索引:
[ 辅助索引 B+ Tree ]
   |
   +-> 索引叶节点已包含所需所有字段 -> 直接返回
   (无需回表，效率最高)
```

**实战案例**：
在某次慢查询优化中，发现 `WHERE create_time > '2023-01-01'` 很慢，`EXPLAIN` 显示 `type: ALL`。原因是查询字段 `create_time` 存在类型隐式转换（String vs Datetime）导致索引失效。修正类型后，`type` 变为 `range`，查询从 2s 降至 20ms。

**关键代码示例**：
```sql
-- 查看执行计划
EXPLAIN SELECT * FROM orders WHERE user_id = 1001;

-- 重点检查 Extra 列，优化前出现 Using filesort
-- 优化：建立联合索引 (user_id, status) 消除 filesort
ALTER TABLE orders ADD INDEX idx_user_status (user_id, status);
```

## 常见考点
1.  **索引失效场景**：哪些操作会导致索引失效？（如 `LIKE '%abc'`、对字段进行函数运算 `WHERE YEAR(date)=2023`、隐式类型转换、OR 连接部分字段无索引等）。
2.  **最左前缀原则**：联合索引 `(a, b, c)`，为什么查询 `WHERE b=1` 无法使用索引？
3.  **回表**：什么是回表？如何通过覆盖索引优化 SQL？

## 记忆要点

- 首查EXPLAIN：重点看type判断访问类型，最怕ALL全表扫描
- 见NULL必失效：key字段为NULL代表没用索引，possible_keys可辅助排查
- 优劣看Extra：Using index是覆盖索引免回表(极佳)，filesort或temporary需警惕(极差)
