# Redis切片集群的工作原理是什么？

Redis 切片集群通过数据分片将数据分布在多个节点上，解决单机内存和并发性能瓶颈。

### 1. 数据分片（哈希槽）
- **16384 个槽位**：Redis Cluster 预设了 16384 个哈希槽。
- **槽位分配**：每个 master 节点负责处理一部分槽位。例如，节点 A 负责槽 0-5000，节点 B 负责槽 5001-10000。
- **数据映射**：当读写一个 key 时，Redis 使用算法 `CRC16(key) % 16384` 计算出 key 属于哪个槽位，然后路由到对应的节点。

### 2. 节点通信与路由流程（Gossip 协议）
- **去中心化**：Cluster 采用去中心化的 P2P 架构，节点之间通过 Gossip 协议交换信息（节点状态、槽分配等）。
- **元数据更新**：每个节点都会感知到整个集群的槽位映射关系，因此可以正确转发客户端请求。如果节点收到不属于自己槽位的请求，会返回 MOVED 重定向错误，指引客户端去正确的节点访问。

```text
Redis Cluster 请求处理流程:

Client                       Node A (Slot 0-5000)       Node B (Slot 5001-10000)
  |                                |                          |
  | --- GET key1 (CRC16->5001) ---> |                          |
  |                                | 计算: Slot 5001 不属于我  |
  |                                |                          |
  | <---- MOVED 5001 Node.BIP ----- |                          |
  |                                |                          |
  | --- GET key1 (Redirect) ------>|------------------------> |
  |                                |                          | 执行查询
  |                                |                          |
  | <------ Value -----------------|<------------------------ |
```

### 3. 高可用与故障转移
- **主从复制**：每个 master 节点都有若干个 slave 节点。当 master 下线时，集群会从 slave 中选举出新的 master 接管槽位。
- **自动故障检测**：节点间通过心跳检测故障，当超过半数主节点认为某 master 客观下线时，触发故障转移。

### 实战案例
在做“双11大促”扩容时，需要对 Redis Cluster 进行节点伸缩。曾遇到哈希槽迁移期间，客户端未及时更新本地路由缓存，导致大量 `MOVED` 错误，请求时延飙升。解决方案是客户端开启 Smart Client 模式，自动处理重定向并缓存槽位映射关系。

### 对比表格：重定向类型
| 错误类型 | 场景 | 客户端行为 | 
| :--- | :--- | :--- |
| **MOVED** | 槽位已永久迁移到新节点 | 更新本地缓存，后续请求直接发往新节点 |
| **ASK** | 槽位正在迁移中（数据部分迁移） | 仅本次请求发往新节点，不更新缓存（下次可能仍需重定向） |

### 代码示例 (Java Jedis 计算槽位)
```java
// 简单的 CRC16 哈希槽计算逻辑
public static int calculateSlot(String key) {
    if (key == null) return 0;
    // 处理 HashTag {user:1000}
    int start = key.indexOf('{');
    int end = key.indexOf('}');
    if (start > 0 && end > start && end < key.length()) {
        key = key.substring(start + 1, end);
    }
    // CRC16 算法计算
    int crc = CRC16.crc16(key.getBytes());
    return crc & 16383; // 对应 16384 个槽位
}
```

### 常见考点
1. **为什么是 16384 个槽**：
   - 如果槽位太少（如 256），节点管理容易倾斜；
   - 如果槽位太多（如 65536），心跳消息中携带的槽位位图会过大，消耗带宽（65536/8=8KB，而 16384/8=2KB）。
2. **MOVED vs ASK**：MOVED 表示槽位已经永久迁移，客户端需更新本地缓存；ASK 表示槽位正在迁移（数据迁了一半），临时指引客户端去新节点查询，客户端不更新缓存。
3. **哈希标签**：如何保证多个 key 在同一个节点？（使用 `{tag}` 语法，如 `user:1001:profile` 和 `user:1001:orders`，只计算 `user:1001` 的 hash）。
