# 索引有哪些种类

### 索引有哪些种类

**1. 按数据结构分类（底层实现）**
- **B+树索引**：InnoDB 和 MyISAM 的默认索引结构。
  - **特点**：只有叶子节点存储数据/指针，非叶子节点只存索引键值；叶子节点通过双向链表连接。
  - **优势**：树高度低（3-4层即可存千万级数据），减少磁盘 I/O；天然支持范围查询（通过链表遍历）。
- **Hash 索引**：Memory 引擎默认，InnoDB 支持自适应 Hash。
  - **特点**：基于 Key-Value 哈希表。
  - **限制**：仅支持等值查询（`=`，`IN`）；不支持范围查询、排序；不支持部分索引列匹配（如联合索引只查第一列也不行）；哈希冲突多时性能下降。
- **全文索引**：用于搜索长文本（`MATCH AGAINST`）。底层通常使用倒排索引（InnoDB 使用 Full-text Parser）。

**2. 按物理存储分类（数据组织方式）**
- **聚簇索引**：
  - **定义**：索引结构的叶子节点直接存储了整行数据。
  - **InnoDB 规则**：主键索引即为聚簇索引。如果没有主键，选第一个唯一非空索引；还没有，则 InnoDB 自动生成一个隐藏的 `ROW_ID`。
  - **特点**：一张表只能有一个聚簇索引。数据文件本身就是索引文件。
- **非聚簇索引（二级索引）**：
  - **定义**：叶子节点存储的是索引列的值 + 主键值。
  - **回表**：查二级索引拿到主键后，再去聚簇索引查完整行数据的过程。

```text
  InnoDB 索引结构示意图
  ┌───────────────────────┐
  │      聚簇索引 (主键)    │
  │  Leaf Node: [PK|Data]  │  (数据存储在这里)
  └───────────────────────┘
            ▲
            │ 回表
  ┌─────────┴───────────────┐
  │     非聚簇索引 (二级)     │
  │  Leaf Node: [Col|PK]    │  (存储索引值和主键)
  └─────────────────────────┘
```

**3. 按字段特性分类**
- **主键索引**：数据列唯一且不能为空。一张表只能有一个。
- **唯一索引**：数据列唯一，但允许有空值（注意：MySQL 中空值不等于空值，即可以有多个 NULL）。
- **普通索引**：最基本的索引，没有任何限制。
- **联合索引**：多个字段组合创建的索引。遵循「最左前缀原则」。
  - 例如 `(name, age)`，查询 `WHERE name='xx'` 或 `WHERE name='xx' AND age=20` 会命中索引，但 `WHERE age=20` 不会。

**4. 按功能分类**
- **前缀索引**：对长文本（如 VARCHAR(255)）的前 N 个字符建立索引，减少索引体积。缺点是不能使用覆盖索引，也无法使用 ORDER BY/GROUP BY。
- **空间索引**：用于地理空间数据类型（GEOMETRY）。

**5. 实战补充**
- **实战案例**：在电商订单表中，针对 `user_id` 和 `order_status` 建立联合索引 `(user_id, status)`。用户查询“我的待支付订单”时，利用索引避免了全表扫描，但若只查“所有待支付订单”（仅 status 条件），索引将失效。曾遇到过因使用随机 UUID 做主键导致 InnoDB 页分裂严重，写入性能下降 50% 的案例，改为自增 ID 后恢复。
- **代码示例**：
```sql
-- 创建联合索引并利用覆盖索引优化查询
ALTER TABLE orders ADD INDEX idx_user_status (user_id, order_status);

-- 联合索引最左前缀原则与覆盖索引示例
-- Explain 中 Extra: Using Index 表示使用了覆盖索引，无需回表
EXPLAIN SELECT user_id, order_status FROM orders WHERE user_id = 1001;
```
- **对比表格**：聚簇索引 vs 非聚簇索引
| 特性 | 聚簇索引 | 非聚簇索引 (二级索引) |
| :--- | :--- | :--- |
| **叶子节点内容** | 存储整行数据 | 存储索引列值 + 主键值 |
| **数量限制** | 一张表只能有一个 | 一张表可以有多个 |
| **查询效率** | 主键查询极快（一次定位） | 通常需要“回表”查询（两次索引查找） |
| **数据物理顺序** | 数据按照主键顺序物理排列 | 索引项逻辑有序，与数据物理存储无关 |

## 常见考点
1. **聚簇索引和非聚簇索引的区别是什么？**
   聚簇索引叶子节点存数据行，非聚簇索引叶子节点存主键值；聚簇索引主键查询效率高，但二级索引必须回表（除覆盖索引外）。
2. **为什么推荐使用自增主键作为聚簇索引？**
   - **减少页分裂**：自增数据是顺序插入的，记录写入当前页的尾部，页填满才开新页。如果是随机主键（如 UUID），可能导致频繁的页分裂和碎片，降低性能。
   - **占用空间小**：自增主键通常是 BIGINT，比 UUID（36字符）小，能显著减少非聚簇索引叶子节点的存储空间（因为二级索引存主键）。
3. **什么是回表？如何减少回表？**
   - **回表**：先查二级索引拿到主键，再查聚簇索引拿数据。
   - **减少回表**：使用「覆盖索引」，即 SELECT 的字段都在索引中包含，直接从二级索引获取结果，无需回表。

## 记忆要点

- 聚簇索引叶子存整行数据且仅有一个，非聚簇存主键且通常需要回表
- InnoDB主键选用规则：显式主键 > 第一个非空唯一索引 > 隐藏生成ROW_ID
- 自增主键优势：顺序插入减少B+树叶节点分裂，体积小降低二级索引存储开销
- Hash索引不支持范围与排序，B+树叶子双向链表天然支持范围查询
