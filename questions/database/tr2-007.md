# ES (Elasticsearch) 的倒排索引中，FST (Finite State Transducer) 是如何将 Term Dictionary 也就是词典存储在内存中的？相比传统的 B+ 树或 Hash 结构，它有什么优势？

ES 中的 FST 是一种类似 Trie 树但更节省内存的前缀压缩结构。它将 Term Dictionary 中的词汇映射到 Postings List（倒排表）的磁盘地址。FST 的优势在于极高的压缩率和前缀共享能力：它不仅压缩了公共前缀，还压缩了后缀。相比于 B+ 树，FST 在查询时不仅减少磁盘 I/O（常驻内存），而且查找时间复杂度仅与词长相关（通常是 O(k)），通常比 B+ 树的 O(log N) 更快。相比于 Hash 结构，FST 支持前缀模糊查询和范围查询。通过 FST，ES 可以将庞大的词典加载进内存，快速定位到磁盘上的倒排表位置，这是 ES 实现秒级全文检索的关键技术。

## 技术原理

- **存储结构：压缩 Trie 树，共享前缀和后缀**：FST（Finite State Transducer，有限状态转换器）是一种确定性的 acyclic 自动机。它在 Trie 的基础上做了两层压缩：公共前缀共享路径（如 `cat`/`car` 共享 `ca`），公共后缀也合并为同一终点（如 `cat`/`bat` 的 `at` 后缀复用）。这样一个词只需记录"从起点走哪些边"，内存占用比朴素 Trie 小一两个数量级。
- **查询性能：复杂度 O(k) 与词长相关，且常驻内存无磁盘 I/O**：FST 在 ES 里通过 `tip`（term index）文件 mmap 到内存，查找一个 term 只需沿自动机走 k 个字符（k=词长），每步一次内存跳转，与词典总量 N 无关，所以是 O(k) 而非 B+ 树的 O(log N)。FST 的输出是 Posting List 在磁盘（`tim`/`doc` 文件）上的偏移量，定位后才有一次磁盘 I/O。
- **功能优势：相比 Hash 支持前缀和范围查询，相比 B+ 树内存占用更低**：Hash 是 O(1) 精确匹配但完全不支持前缀/范围；B+ 树支持范围但每个节点存 key 字符串，几亿 term 内存爆炸。FST 把词典压缩到常驻内存，既支持 `prefix*`、通配符、正则前缀，又能 O(k) 寻址，是全文检索的最佳折中。

## 对比/选型

| 结构 | 查找复杂度 | 前缀/范围查询 | 内存占用 | 磁盘 I/O |
|------|------------|---------------|----------|----------|
| Hash | O(1) | 不支持 | 中（存原 key） | 取决于实现 |
| B+ 树 | O(log N) | 支持 | 高（存完整 term） | 每层一次 |
| FST（ES） | O(k) | 支持前缀/范围 | 极低（前缀后缀共享） | 仅倒排表一次 |

## 代码示例

用 Lucene 的 `Builder` 构建一个最小 FST 直观感受前缀后缀共享：

```java
import org.apache.lucene.util.fst.Builder;
import org.apache.lucene.util.fst.FST;
import org.apache.lucene.util.BytesRef;

Builder<Long> b = new Builder<>(FST.INPUT_TYPE.BYTE1, Long::longValue);
b.add(new BytesRef("cat"), 1L);   // cat / car 共享前缀 "ca"
b.add(new BytesRef("car"), 2L);   // cat / bat 共享后缀 "at"
b.add(new BytesRef("bat"), 3L);
FST<Long> fst = b.finish();

// 查找：沿自动机走字符，O(k)，k=词长
Long v = Util.get(fst, new BytesRef("car"));   // 返回 2
```

ES 中 FST 对应的 `.tip`（term index，FST 本体，mmap 内存）和 `.tim`（term dictionary，存 term 全文 + posting 指针）文件结构示意：

```
.tip (FST 常驻内存)         .tim (磁盘)              .doc (磁盘)
[ca] --t--> offset_t   -->  "cat"  -> posting_ptr -> [doc1, doc5, ...]
  \--r--> offset_r     -->  "car"  -> posting_ptr -> [doc2, ...]
[ba] --t--> offset_bt  -->  "bat"  -> posting_ptr -> [doc3, ...]
```

## 常见坑/注意事项

- **FST 是只读的**：构建后不可修改，ES 的 segment 是不可变的，新增文档要新建 segment 再 merge，所以 FST 的高压缩是建立在不可变前提下的。
- **词典太大仍可能撑爆内存**：虽然 FST 压缩率高，但 `tip` 默认全量 mmap 进内存，若字段基数极高（如几十亿唯一 term，如 trace_id）仍会导致内存压力，这类高基数字段不建议建全文索引。
- **前缀查询未必命中 FST 优势**：`prefix*` 这类前缀通配会展开大量分支，FST 支持但性能随分支数下降；避免 `*suffix` 这种后缀通配（无法用前缀压缩，需全扫描，可配 `index_options` 或 ngram 缓解）。
- **FST 的 term 是按字典序构建的**：Lucene 要求 `Builder.add` 必须按 BytesRef 字典序递增调用，否则构建报错——这是 FST 实现 acyclic 的前提。


## 记忆要点

- FST是类似Trie树的前缀压缩结构，不仅压缩前缀也复用后缀
- FST将Term Dictionary常驻内存，映射到磁盘倒排表地址
- 对比B+树：FST内存压缩率极高，且查找复杂度O(k)比O(logN)更快
- 对比Hash：Hash不支持前缀查询，而FST支持前缀和范围查询
