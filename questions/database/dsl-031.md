# MySQL的执行引擎有哪些

### MySQL 的存储引擎有哪些

MySQL 插件式存储引擎架构允许根据需求选择，常见引擎如下：

1. **InnoDB**
   - **特性**：支持事务（ACID）、行级锁、外键、崩溃自动恢复。
   - **默认引擎**：MySQL 5.5 及之后默认。
   - **适用场景**：高并发、高可靠性要求的核心业务表。

2. **MyISAM**
   - **特性**：不支持事务、不支持行级锁（仅表锁）、访问速度快。
   - **局限**：崩溃后无法安全恢复。
   - **适用场景**：以读为主、对事务完整性要求不高的场景（现已较少使用）。

3. **Memory（Heap）**
   - **特性**：数据存储在内存中，支持 Hash 索引，读写速度极快。
   - **局限**：服务重启数据丢失，表结构保留。
   - **适用场景**：临时表、缓存查找表（如字典表）。

4. **Archive**
   - **特性**：支持高压缩比的行级存储，仅支持插入和 SELECT，不支持索引。
   - **适用场景**：日志归档、历史数据存储。

**原理细节与补充**：
*   **InnoDB 架构**：所有数据都存储在 `.ibd` 文件中（独立表空间），基于 MVCC（多版本并发控制）实现高并发，通过 Undo Log 实现回滚。
*   **MyISAM 静态压缩**：如果表包含定长字段（如 char, int），MyISAM 会采用静态格式，数据检索极快，但更新时可能出现碎片。
*   **对比总结**：
    - **锁粒度**：InnoDB (行锁) vs MyISAM (表锁)
    - **事务**：InnoDB (支持) vs MyISAM (不支持)
    - **崩溃恢复**：InnoDB (Crash-safe) vs MyISAM (易损坏)

**5. 实战补充**
- **实战案例**：在某高并发秒杀场景初期，使用 MyISAM 做库存计数，因表锁机制导致大量更新请求堵塞，库存扣减严重延迟。后改用 InnoDB 并利用 `UPDATE stock SET count=count-1 WHERE id=1` 的行锁特性，吞吐量提升 10 倍。另外，Memory 引擎常用于存放在线用户的 Session 信息，利用其 Hash 索引实现 O(1) 查找，但需接受重启丢失的风险。
- **代码示例**：
```sql
-- 建表时指定引擎
CREATE TABLE user_logs (
    id INT AUTO_INCREMENT PRIMARY KEY,
    content TEXT,
    created_at DATETIME
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 修改已有表的引擎
ALTER TABLE old_table ENGINE=InnoDB;

-- 查看当前表引擎
SHOW TABLE STATUS WHERE Name='user_logs';
```
- **对比表格**：InnoDB vs MyISAM vs Memory
| 特性 | InnoDB | MyISAM | Memory |
| :--- | :--- | :--- | :--- |
| **事务支持** | 支持 | 不支持 | 不支持 |
| **锁粒度** | 行锁 (支持并发写) | 表锁 (写串行) | 表锁 |
| **存储限制** | 64TB (受限于文件系统) | 256TB | RAM (内存大小) |
| **索引** | B+Tree (聚簇) | B+Tree (非聚簇) | Hash (默认) / B+Tree |
| **崩溃恢复** | 支持 (Crash-safe) | 不易恢复 | 数据丢失 |
| **适用场景** | 核心业务、事务系统 | 只读报表、历史归档 | 临时表、查找表 |

## 记忆要点

- InnoDB支持事务与行锁（默认引擎），MyISAM仅支持表锁且不支持崩溃恢复
- MyISAM查询极快但不适宜高并发写，Memory引擎数据存内存重启即丢
- InnoDB行锁基于索引实现：若查询未命中索引则行锁会退化为表锁
- Archive专用于高压缩比的日志归档，仅支持插入和查询
