# 什么是幻读是如何解决的？

### 隔离级别与并发问题
MySQL InnoDB 默认隔离级别是 **可重复读**。

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
| :--- | :---: | :---: | :---: |
| 读未提交 | ✅ | ✅ | ✅ |
| 读提交 (RC) | ❌ | ✅ | ✅ |
| **可重复读 (RR)** | ❌ | ❌ | **可能发生** |
| 串行化 | ❌ | ❌ | ❌ |

### 什么是幻读
**定义**：指一个事务在前后两次查询同一个范围的数据时，后一次查询看到了前一次查询没看到的新增行（被其他事务插入）。

**核心点**：
1. 侧重于“新增”或“删除”导致的行数变化。
2. 通常是针对“范围查询”而非单行查询。

### 幻读的解决方案
MySQL InnoDB 在 RR 级别下，通过两种机制很大程度上解决了幻读：

#### 1. 快照读 - MVCC (Multi-Version Concurrency Control)
- **适用场景**：普通 `SELECT` 语句。
- **原理**：通过 **Read View**（读视图）和 **Undo Log** 实现。
- **机制**：事务读取时，会生成一个 Read View，后续读取只读取该版本或更早版本的数据，其他事务新插入的数据对当前事务不可见。
- **结论**：MVCC 解决了快照读下的幻读。

#### 2. 当前读 - Next-Key Lock
- **适用场景**：`SELECT ... FOR UPDATE`, `SELECT ... LOCK IN SHARE MODE`, `UPDATE`, `DELETE`。
- **原理**：通过 **Next-Key Lock = Record Lock (行锁) + Gap Lock (间隙锁)** 实现。
- **机制**：不仅锁住存在的记录，还锁住记录之间的“间隙”，阻止其他事务在这个范围内插入新记录。

### Next-Key Lock 原理图示
假设表中有数据：1, 5, 10。

```text
记录:      1           5           10
          │           │           │
间隙:  (-∞, 1]   (1, 5]   (5, 10]   (10, +∞)
          │           │           │
       (Gap+Rec)  (Gap+Rec)  (Gap+Rec)  (Gap)
```
如果执行 `SELECT * FROM t WHERE id = 5 FOR UPDATE`：
- 加锁范围是 `(1, 5]`，即 1 到 5 之间的间隙被锁住，5 这一行被锁住。
- 其他事务无法插入 id=2 或 id=3 的数据，也无法修改 id=5。

## 常见考点
1. **RC 和 RR 隔离级别的本质区别是什么？**（答案：RC 每次查询都生成新的 Read View；RR 只在第一次查询生成 Read View）
2. **Next-Key Lock 在什么情况下会退化为行锁或间隙锁？**（唯一索引等值查询命中记录退化行锁；未命中退化间隙锁）
3. **为什么说 RR 级别只能“很大程度”解决幻读？**（历史版本 bug 或特定构造场景仍可能存在，但通常认为已解决）
