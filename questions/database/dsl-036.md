# 数据库-Redis

### MySQL 与 MongoDB 对比

1.  **数据模型**
    *   **MySQL**：关系型数据库，数据结构化，需要预先定义表结构，遵循 ACID 原则，适合处理实体关系复杂的数据。
    *   **MongoDB**：文档型数据库，无模式，使用 BSON（二进制 JSON）格式存储，支持嵌套文档和数组，字段可动态增减，适合数据结构灵活多变的场景。

2.  **存储引擎**
    *   **MySQL**：插件式引擎架构。
        *   **InnoDB**：默认引擎，支持事务、行级锁、外键、崩溃恢复（Redo/Undo Log）。
        *   **MyISAM**：表级锁，不支持事务，读性能较好，但安全性较低。
    *   **MongoDB**：默认使用 **WiredTiger** 引擎，支持文档级锁和 Checkpoint，提供压缩功能（Snappy/Zlib），数据写入时先写入 Write-Ahead Log (WAL) 保证持久性。

3.  **适用场景**
    *   **MySQL**：适用于事务一致性要求高（如金融转账）、数据结构固定、需要进行复杂关联查询（JOIN、聚合）的业务。
    *   **MongoDB**：适用于数据结构多变（如日志、用户画像）、海量非结构化数据、高并发读写、需要快速水平扩展的业务。

4.  **扩展性**
    *   **MySQL**：通常采用垂直扩展（升级硬件）为主，水平扩展需要实施复杂的分库分表（Sharding-JDBC、MyCAT 等）方案，对业务代码有侵入性。
    *   **MongoDB**：原生支持自动分片，通过 Config Server 管理元数据，Mongos 作为路由层，支持横向平滑扩展，对应用透明。

**实战案例**：在重构某物联网平台时，我们将设备每秒上报的时序数据从 MySQL 迁移到了 MongoDB。MySQL 在海量写入下频繁出现锁等待和索引树分裂导致的性能瓶颈，迁移后利用 MongoDB 的 Schemaless 特性解决了不同设备型号上报字段不一致的问题，写入吞吐量提升了 5 倍。

**对比表格**：

| 维度 | MySQL (关系型) | MongoDB (文档型) |
| :--- | :--- | :--- |
| **数据结构** | 表，行列结构，强 Schema | 集合，BSON 文档，弱 Schema |
| **事务支持** | 完整支持 ACID | 4.0+ 支持多文档事务，但性能略逊 |
| **查询语言** | 标准 SQL | MQL (MongoDB Query Language) |
| **JOIN 操作** | 原生支持，强大 | `$lookup` 支持，但性能较差，建议应用层处理 |
| **扩展方式** | 读写分离，分库分表（复杂） | 原生分片，自动均衡 |

### 数据库慢查询优化

1.  **定义**
    执行时间超过 `long_query_time`（默认 10s，生产环境通常设为 1s 或更短）的 SQL 语句，且未使用索引的查询（取决于 `log_queries_not_using_indexes` 设置）。

2.  **常见原因**
    *   **缺乏索引**：全表扫描导致效率低下，锁表风险增加。
    *   **索引失效**：如对索引列进行了函数计算（`WHERE YEAR(date) = 2023`）、隐式类型转换（字符串与数字比较）、使用了 `LIKE '%xx'`（左模糊）或 `OR` 连接非索引列。
    *   **数据量过大**：单表数据量过亿，即使有索引，回表代价也极高。
    *   **SQL 写法差**：复杂的子查询（优化器可能转为相关子查询）、深层次连接、SELECT * 导致回表消耗大量 I/O。
    *   **硬件/配置瓶颈**：磁盘 I/O 瓶颈（如 Raid 级别不当）、内存不足导致 Buffer Pool 命中率低、频繁换页。

3.  **优化思路与步骤**
    *   **开启慢查询日志**：通过 `slow_query_log = 1` 开启，使用 `mysqldumpslow` 工具分析日志，定位问题 SQL。
    *   **使用 EXPLAIN**：分析执行计划，重点关注 `type`（访问类型，需达到 ref/range/const）、`key`（实际使用的索引）、`rows`（预估扫描行数）、`Extra`（是否出现 Using filesort, Using temporary）。
    *   **优化索引**：
        *   遵循**最左前缀原则**建立联合索引。
        *   利用**覆盖索引**（Covering Index），避免回表查询。
        *   区分度高（基数大）的字段适合建索引。
    *   **优化 SQL 语句**：
        *   避免使用 `SELECT *`，只查询需要的字段。
        *   避免在索引列上做运算。
        *   对于大表的 `LIMIT` 分页，优化为“延迟关联”或记录上次 ID 的游标分页。
    *   **架构优化**：
        *   引入 Redis 做缓存，减轻 DB 压力。
        *   进行读写分离，主库写，从库读。
        *   单表数据量过大时，实施分库分表。

**实战案例**：曾处理一个“深分页”性能问题。SQL 为 `SELECT * FROM orders ORDER BY id LIMIT 1000000, 10`，耗时长达 10 秒。这是因为 MySQL 需要扫描 100 万行数据再抛弃前 100 万行。优化方案改为利用延迟关联，先查主键 `SELECT id FROM orders ORDER BY id LIMIT 1000000, 10`，再根据主键回表查询详情，耗时降至 0.1 秒。

**代码示例（深度分页优化）**：
```sql
-- 优化前：性能极差，越往后翻越慢
SELECT * FROM large_table ORDER BY id LIMIT 1000000, 10;

-- 优化后：延迟关联，利用覆盖索引快速定位 ID
-- 前提：id 是主键或有索引
SELECT a.* FROM large_table a 
INNER JOIN (SELECT id FROM large_table ORDER BY id LIMIT 1000000, 10) b 
ON a.id = b.id;
```
