# 说一说你了解的MVCC机制？

### MVCC (多版本并发控制) 机制详解

**1. MVCC 是什么？**
MVCC (Multi-Version Concurrency Control) 是一种并发控制机制，主要用于在多个事务并发存取数据库时，通过保存数据的**历史版本**，实现**写操作不阻塞读操作**，从而提高数据库系统的并发性能。

**2. InnoDB 实现 MVCC 的核心组件**

#### (1) 隐藏字段
InnoDB 在每行数据中隐式存储以下字段：
- `DB_TRX_ID` (6字节)：**事务ID**。记录最近一次修改该行数据的事务 ID。
- `DB_ROLL_PTR` (7字节)：**回滚指针**。指向该行数据上一个版本的地址（存储在 Undo Log 中）。
- `DB_ROW_ID` (6字节)：**隐藏主键**。如果表没有主键，InnoDB 自动生成该字段。

#### (2) Undo Log (回滚日志)
当事务修改数据时，InnoDB 不会直接覆盖旧数据，而是将旧数据写入 Undo Log。通过 `DB_ROLL_PTR`，旧版本的数据被串联成一个**版本链**。

```text
      Undo Log (版本链)
          ▲
          │ (DB_ROLL_PTR)
          │
    ┌─────┴─────┐    ┌─────────────┐    ┌─────────────┐
    │ Version 3 │ ◀──│ Version 2   │ ◀──│ Version 1   │ (老版本)
    │(Trx: 103) │    │(Trx: 102)   │    │(Trx: 101)   │
    └───────────┘    └─────────────┘    └─────────────┘
                           │
                           │ Current Data in Buffer Pool
                           ▼
                    ┌─────────────┐
                    │ Current Row │
                    │(Trx: 104)   │
                    └─────────────┘
```

#### (3) Read View (读视图)
Read View 是事务进行**快照读**时生成的“一致性视图”，它决定了当前事务能看到版本链中的哪一个版本。Read View 包含以下核心字段：
- `m_ids`：生成 Read View 时，当前系统中**活跃（未提交）**的事务 ID 列表。
- `min_trx_id`：`m_ids` 中最小的事务 ID。
- `max_trx_id`：生成 Read View 时，系统将要分配的下一个事务 ID（即当前最大事务 ID + 1）。
- `creator_trx_id`：生成该 Read View 的事务本身的 ID。

**3. 可见性算法 (核心逻辑)**
当事务访问某行数据时，会拿到该行数据的事务 ID `trx_id`，并与 Read View 进行比较：

1. **如果是自己改的**：若 `trx_id == creator_trx_id`，说明是自己修改的，**可见**。
2. **已提交的老数据**：若 `trx_id < min_trx_id`，说明该版本在 Read View 生成前已提交，**可见**。
3. **未来的数据**：若 `trx_id >= max_trx_id`，说明该版本是在 Read View 生成后才开启的事务修改的，**不可见**。
4. **需判断活跃状态**：若 `min_trx_id <= trx_id < max_trx_id`，需判断 `trx_id` 是否在 `m_ids` 列表中：
   - **在列表中**：说明生成 Read View 时该事务还未提交，**不可见**。
   - **不在列表中**：说明生成 Read View 时该事务已经提交了，**可见**。

**如果不可见**：通过 `DB_ROLL_PTR` 指针去 Undo Log 中找上一个版本，再次递归执行上述逻辑，直到找到可见版本或链表结束。

**4. RC 和 RR 隔离级别的区别**
MVCC 使得 InnoDB 在 **Read Committed (RC)** 和 **Repeatable Read (RR)** 级别下都避免了不可重复读，但生成 Read View 的时机不同：
- **RC (读已提交)**：**每次 SELECT** 都会生成一个新的 Read View。因此，能看到其他事务提交后的新版本。
- **RR (可重复读)**：**只在事务开启后的第一次 SELECT** 生成 Read View，后续所有查询都复用这个 Read View。因此，无论其他事务是否提交新数据，看到的都是事务开始时的样子。

## 常见考点
1. **MVCC 能解决幻读吗？**
   - *答案*：MVCC 解决的是**快照读**下的幻读。对于**当前读**（如 `select ... for update`），InnoDB 使用 **Next-Key Lock (临键锁)** 来解决幻读。
2. **什么是快照读和当前读？**
   - *快照读*：普通的 `SELECT`，读取历史版本，不加锁。
   - *当前读*：读取最新版本并加锁，如 `select ... for update`、`insert`、`update`、`delete`。
3. **为什么 Undo Log 不需要无限保留？**
   - *机制*：由 **Purge Thread**（清理线程）负责。当系统中没有 Read View 需要引用某个 Undo Log 版本时，该版本就可以被回收。
4. **如果主键索引是聚簇索引，辅助索引怎么通过 MVCC 判定可见性？**
   - *机制*：辅助索引记录没有 `DB_TRX_ID`。InnoDB 会在辅助索引页中存储 `MAX_TRX_ID`，或者通过回表到聚簇索引去获取主键记录的 `DB_TRX_ID` 来进行判断。
