# count(*)、count(1)和count(列名)有什么区别？

### 基础概念与执行层面

1.  **`count(*)`**
    *   **含义**：统计行数，**包含 NULL 值**。
    *   **优化**：MySQL 专门对 `count(*)` 进行了优化，它并不会取出所有字段值，而是直接统计表的行数。
    *   **底层机制**：在 InnoDB 引擎中，MySQL 会选择**成本最小的辅助索引**（二级索引）进行遍历统计。如果不存在辅助索引，则使用聚簇索引（主键索引）。

2.  **`count(1)`**
    *   **含义**：统计行数，包含 NULL 值。
    *   **机制**：在解析阶段，MySQL 会将 `count(1)` 中的常量 `1` 视为“某一列”（虚拟列）。实际上，MyISAM 和 InnoDB 在执行层面会对 `count(*)` 和 `count(1)` 进行相同的优化，效率**完全一致**。

3.  **`count(主键)` / `count(列名)`**
    *   **含义**：统计该列**不为 NULL** 的行数。
    *   **机制**：需要读取该列的值并进行非空判断。
    *   **效率差异**：
        *   `count(主键)`：由于主键索引的叶子节点存储完整数据，且非空，虽然是走聚簇索引，但涉及数据读取，略慢于 `count(*)`。
        *   `count(非主键列)`：如果该列允许 NULL，必须逐行判断；如果该列建立了二级索引且不为 NULL，可能比 `count(主键)` 略快（因为二级索引体积更小），但仍慢于 `count(*)`。

---

### 引擎差异与架构图

**InnoDB vs MyISAM**
*   **MyISAM**：内部维护了一个计数器，存储表的总行数。执行 `count(*)` 时直接返回，复杂度 **O(1)**（但如果加了 WHERE 条件，依然需要扫描）。
*   **InnoDB**：由于支持 **MVCC（多版本并发控制）**，同一时刻不同事务看到的数据行数可能不同，因此无法维护一个全局精确的计数器。必须遍历索引树进行统计，复杂度 **O(N)**。

**InnoDB Count(*) 执行流程：**

```text
Server层接收 SQL
   │
   ▼
Optimizer (优化器)
   │
   ├── 寻找成本最小的索引树
   │    (优先选择 Secondary Index，因为 Secondary Index 叶子节点只存索引列+主键，
 │     比 Clustered Index 存完整行数据更小，IO成本更低)
   │
   ▼
遍历索引叶子节点
   │
   ├── 读取每一行
   │
   ▼
Server层统计数量 (不读取具体字段值)
   │
   ▼
返回结果
```

### 性能对比总结

执行效率（针对 InnoDB，全表统计无 WHERE 条件）：
`count(*)` ≈ `count(1)` > `count(二级索引列)` > `count(主键)` > `count(普通未索引列)`

*   **注**：MySQL 8.0.13+ 引入了 `Skip Scan` 等优化，但上述基本规律依然适用。

### 实战案例
在一个亿级用户的社交平台中，运营要求统计“当日活跃人数”。起初使用了 `COUNT(user_id) WHERE login_time > ...`，由于二级索引树庞大，耗时极慢。后改为维护一张独立的 `redis` 计数器或使用 `show table status` 的估算值（对于非精确财务场景），将响应时间从秒级降低到毫秒级。

### 代码示例
```sql
-- 场景：统计非空邮箱用户数（不能使用 count(*)）
SELECT COUNT(email) FROM users; 

-- 场景：统计全表行数（最优）
SELECT COUNT(*) FROM users;
```
