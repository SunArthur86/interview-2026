# AOF的磁盘重写机制是什么？

### AOF 磁盘重写机制

**背景**：
AOF（Append Only File）通过记录写命令来持久化数据。随着运行时间增加，AOF 文件会越来越大，导致恢复数据变慢，占用过多磁盘空间。

**重写机制**：
Redis 提供了 AOF 重写功能，不读取旧的庞大 AOF 文件（避免阻塞），而是直接读取内存中的当前数据，生成对应的最小化写命令，写入一个新的 AOF 文件。

**触发方式**：
1.  **手动触发**：执行 `BGREWRITEAOF` 命令。
2.  **自动触发**：配置 `auto-aof-rewrite-min-size` 和 `auto-aof-rewrite-percentage`。

**流程**：
1.  主进程 fork 出一个子进程。
2.  子进程将内存数据转换为写命令，写入新的 AOF 临时文件。
3.  在此期间，主进程继续处理新命令，并将新命令同时写入 **AOF 重写缓冲区**（不仅写旧 AOF 缓冲区）。
4.  子进程完成后，主进程将重写缓冲区内容追加到新文件。
5.  原子性地用新文件替换旧文件（rename 原子操作）。

**ASCII 流程图**：

```text
+-------------+          +----------------+
|  Redis 主   |          |   磁盘 AOF     |
|   进程      |          |    文件        |
+------+------+          +-------+--------+
       |                          |
       | 1. fork()                |
       |------------------------->|
       |                          |
+------v------+            +------v------+
|  子进程     |            | 旧 AOF 文件  |
| (后台重写)  |            | (停止追加)   |
+------+------+            +-------------+
       |
       | 2. 读取内存数据，生成新命令
       |
       v
+---------------------------+
|  新 AOF 临时文件 (纯净版)  |
+-----------+---------------+
            |
+-----------v---------------+       +------------------+
|  主进程继续处理新写命令   |       |  AOF 重写缓冲区  |
|  (写入旧的 AOF 文件)       |------>| (仅收集重写期间) |
+---------------------------+       +---------+--------+
                                             |
                                             | 3. 子进程结束后
                                             |    追加到新文件末尾
                                             v
                                    +--------+--------+
                                    |  最终新 AOF 文件 |
                                    +--------+--------+
                                             |
                                             | 4. 原子替换 (rename)
                                             v
                                    覆盖旧文件，完成重写
```

**实战案例**：
在双十一大促期间，某 Redis 实例因写入量激增导致 AOF 文件增长过快且频繁触发重写。由于主进程内存占用过大（接近 20GB），`fork` 操作耗时超过 1 秒，导致主进程阻塞，业务侧出现明显的请求毛刺（Latency Spike）。通过将实例拆分为小内存集群并调整 `auto-aof-rewrite-percentage` 避免峰值重写解决。

**代码配置示例**：
```conf
# redis.conf 关键配置
# 触发重写的最小文件尺寸 64MB
auto-aof-rewrite-min-size 64mb
# 文件大小比上次重写后增长 100% 时触发
auto-aof-rewrite-percentage 100
```

## 记忆要点

- 重写原理：不读旧文件，而是直接读取内存当前状态，生成最小命令集。
- 触发方式：手动执行BGREWRITEAOF，或按文件大小及增长比例自动触发。
- 核心流程：fork子进程遍历内存写新文件，期间主进程新命令同时写入重写缓冲区。
