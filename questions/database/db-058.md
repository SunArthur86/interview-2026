# 什么是Redis集群？

**Redis 集群与高可用方案**

**一、主从复制**
1.  **作用**：读写分离，数据备份，故障恢复基础。
2.  **流程**：
    -   **建立连接**：从库发送 `SYNC`/`PSYNC` 命令。
    -   **全量复制**：主库 fork 子进程生成 RDB，发送给从库；期间缓冲的写命令也发送给从库。
    -   **增量复制**：主库将后续写命令持续发送给从库。
3.  **断线重连**：若复制积压缓冲区足够，从库重连后只同步增量，否则进行全量复制。

**二、哨兵模式**
1.  **作用**：监控主从节点状态，自动故障转移。
2.  **核心功能**：
    -   **监控**：定期 ping 检测节点是否下线。
    -   **自动故障转移**：主库下线后，哨兵投票选举新主库，通知其他从库和新主库建立复制关系。
    -   **配置提供**：客户端连接哨兵获取当前主库地址。
3.  **主观下线与客观下线**：
    -   单个哨兵认为主库下线是“主观下线”。
    -   超过Quorum数量的哨兵认为下线是“客观下线”，开始选举。

**三、切片集群**
1.  **作用**：解决单机内存容量限制，实现数据分片存储。
2.  **原理**：引入 **Hash Slot（哈希槽）**，共 16384 个槽。每个节点负责一部分槽。计算公式：`CRC16(key) % 16384`。
3.  **重定向**：客户端访问 Key 时，若槽不在本节点，返回 `MOVED` 指令重定向到正确节点。

**集群架构简图：**
```text

    ┌────────┴────────┐
    │   Redis Cluster │
    │  (16384 Slots)  │

```

| 模式 | 核心功能 | 数据量级 | 高可用能力 | 运维复杂度 |
| :--- | :--- | :--- | :--- | :--- |
| **主从复制** | 读写分离、备份 | 受限于单机内存 | 手动切换 | 低 |
| **哨兵模式** | 自动故障转移 (主从基础) | 受限于单机内存 | 高 (自动切换) | 中 |
| **Cluster 集群** | 数据分片、自动故障转移 | 支持 TB/PB 级 | 高 (去中心化) | 高 |

**实战案例：**
在 Cluster 模式迁移中，因 Hash Slot 迁移未完成且超时时间过短，导致客户端频繁报 `ASK` 错误。**解决**：迁移期间调大 `cluster-node-timeout`，并确保客户端 SDK 支持 `ASK` 重定向逻辑。

**关键代码（配置）：**
```conf
# sentinel.conf
port 26379
sentinel monitor mymaster 127.0.0.1 6379 2  # 2表示quorum，需2个哨兵同意才故障转移
sentinel down-after-milliseconds mymaster 30000
sentinel parallel-syncs mymaster 1
sentinel failover-timeout mymaster 180000

# redis.conf (Cluster)
cluster-enabled yes
cluster-config-file nodes-6379.conf
cluster-node-timeout 15000
cluster-require-full-coverage no  # 部分槽不可用时集群仍可用
```

## 常见考点
1. **主从复制为什么会有全量复制和增量复制？如何优化？**
   - 初次连接或 offset 差距过大触发全量复制。优化：增大 `repl-backlog-size`（复制积压缓冲区），减少断线重连后的全量复制概率。
2. **集群模式下为什么是 16384 个槽？**
   - 这是一个折中值。如果槽太少，节点间数据分布不均；如果槽太多（如 65536），心跳消息中携带的槽位信息会占用过多带宽（8KB vs 16KB），且 Redis 节点通常不会超过 1000 个，16384 足够。
3. **哨兵选举 Leader 的规则是什么？**
   - 首先是优先级（`slave-priority`）高的从库；如果相同，则是复制偏移量最大的从库（数据最最新）；如果还相同，则是 run_id 最小的从库。
