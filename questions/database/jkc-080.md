# 分布式缓存

### 缓存雪崩
**定义**：由于原有缓存失效（或同时失效），新缓存未到期间，所有原本应该访问缓存的请求都去查询数据库，对数据库CPU和内存造成巨大压力，严重的会造成数据库宕机，从而形成一系列连锁反应，造成整个系统崩溃。

**关键原因**：
*   缓存服务器重启。
*   大量 Key 设置了相同的过期时间。

**解决方案**：
1.  **加互斥锁**：在缓存失效时，只允许一个线程去查数据库并回写缓存，其他线程等待或短暂休眠后重试。防止并发打死 DB。
2.  **缓存预热/熔断**：系统启动时加载热点数据；或者在检测到 DB 压力过大时，暂时拒绝请求或返回默认值。
3.  **随机过期时间**：在设置过期时间时，增加一个随机值（如 1-5 分钟），使失效时间分散，避免集体失效。
4.  **高可用架构**：使用 Redis Cluster 或 Sentinel，避免单点故障导致全盘缓存丢失。

---

### 缓存穿透
**定义**：缓存穿透是指查询一个**根本不存在**的数据。缓存没有（因为没写进去），数据库也没有。导致每次请求都绕过缓存直接查数据库。如果有恶意攻击构造大量这种请求，数据库会瞬间崩溃。

**解决方案**：
1.  **布隆过滤器**：
    *   将所有可能存在的数据哈希到一个足够大的 Bitmap 中。
    *   请求来了先问 Bloom Filter：
        *   如果说“不存在”，则数据一定不存在，直接返回，不查 DB。
        *   如果说“存在”，才去查缓存或 DB（可能有极小概率误判）。
2.  **缓存空对象**：
    *   如果查询返回的数据为空（数据不存在），仍然将这个 Key 缓存起来，Value 设为 Null 或特定标识。
    *   **过期时间**：设置较短的过期时间（如 30秒 - 5分钟），防止此时刚好有新数据写入导致不一致。

---

### 缓存击穿
**定义**：某个**极度热点**的 Key（如秒杀商品）过期瞬间，海量并发请求同时击中缓存失效点，直接穿透缓存压向后端数据库。

**区别**：
*   缓存雪崩：很多 Key 同时失效，DB 压力巨大。
*   缓存穿透：查不存在的数据，DB 压力巨大。
*   缓存击穿：某个热点 Key 失效，DB 压力瞬间巨大。

**解决方案**：
1.  **互斥锁**：只让一个线程去查库重建缓存，其他线程等待。实现简单但并发能力受限。
2.  **逻辑过期**：不设置 TTL，而是在 Value 中包含过期时间。由后台异步线程负责更新缓存，前台请求发现过期时，返回旧数据并触发更新（适用于对实时性要求不高的场景）。
3.  **永不过期**：针对极端热点数据，直接不设置过期时间，通过后台任务定期更新。

---

### 缓存预热
**定义**：系统上线后或重启后，将相关的缓存数据直接加载到缓存系统。这样就可以避免在用户请求的时候先查询数据库然后再将数据缓存的问题，用户直接查询事先被预热的缓存数据。

**实现方式**：
*   启动脚本触发全量加载。
*   使用定时任务模拟用户请求进行加载。

---

### 缓存更新
除了缓存服务器自带的缓存失效策略之外（如Redis默认的6种策略），还可以根据具体的业务需求进行自定义的缓存淘汰，常见的策略有两种：
1.  **定时清理**：定时去清理过期的缓存。
2.  **懒加载**：当有用户请求过来时，再判断这个请求所用到的缓存是否过期，过期的话就去底层系统得到新数据并更新缓存。

**Cache Aside Pattern (旁路缓存模式)** - *最常用*
*   **读**：先读缓存 -> 命中返回；未命中读 DB -> 写入缓存 -> 返回。
*   **写**：先更新 DB -> **然后**删除缓存（而不是更新缓存）。

---

### 缓存降级
**定义**：当访问量剧增、服务出现问题（如响应时间慢或不响应）或非核心服务影响到核心流程的性能时，仍然需要保证服务还是可用的，即使是有损服务。系统可以根据一些关键数据进行自动降级，也可以配置开关实现人工降级。

**策略**：
*   返回默认值（如库存为0、推荐空列表）。
*   返回本地静态缓存数据。

**注意**：降级的最终目的是保证核心服务可用，即使是有损的。有些服务是无法降级的（如加入购物车、支付结算）。

#### 实战案例
某电商大促期间，商品详情页缓存设置了相同的 24 小时过期时间，导致每天凌晨大批量 Key 同时失效，数据库 CPU 飙升至 100%。修复方案是：在基础过期时间上增加 `Random.nextInt(300)` 秒的随机偏移量，将失效打平。

#### 代码示例
```java
// 缓存击穿：互斥锁实现
public String getProduct(String key) {
    String value = redis.get(key);
    if (value == null) {
        try {
            // 只有获取锁的线程去查库
            if (redis.setnx(key + ":lock", "1", 10, TimeUnit.SECONDS)) {
                value = database.queryProduct(key);
                redis.setex(key, 3600, value);
            } else {
                // 未获取锁，短暂休眠重试或返回旧值
                Thread.sleep(100);
                return getProduct(key);
            }
        } finally {
            redis.del(key + ":lock");
        }
    }
    return value;
}
```

#### 对比表格

| 现象 | 根本原因 | 典型场景 | 核心解决方案 |
| :--- | :--- | :--- | :--- |
| **缓存雪崩** | 大量 Key 同时失效 | 缓存重启、批量的 Key 设置相同 TTL | 随机 TTL、高可用缓存架构、熔断降级 |
| **缓存穿透** | 查询不存在的数据 | 恶意攻击、业务空查询 | 布隆过滤器、缓存空对象 |
| **缓存击穿** | 热点 Key 瞬时失效 | 秒杀商品、热点新闻过期 | 互斥锁、逻辑过期、永不过期 |

## 常见考点
1.  **为什么是删除缓存而不是更新缓存？**
    *   答：并发写场景下，更新缓存容易导致脏数据（并发写A、B，先更新缓存后更新DB，可能导致不一致）。删除缓存虽然下次读会有一次 miss，但能保证数据最终一致性，且实现更简单
