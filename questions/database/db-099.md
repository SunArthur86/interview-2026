# InnoDB 的 MVCC 机制完整原理是什么？ReadView 如何决定事务能看到哪个版本？

【MVCC（Multi-Version Concurrency Control）】通过为每行数据保存多个版本，让读操作不加锁就能读到一致性快照，大幅提升并发性能。

【三个基础组件】
1.  **隐藏字段**：
    - `DB_TRX_ID`：6字节，最近一次修改该行数据的事务 ID。
    - `DB_ROLL_PTR`：7字节，回滚指针，指向 undo log 中该行的上一个版本。
    - `DB_ROW_ID`：6字节，隐藏主键（如果表没有主键，InnoDB 自动生成）。
2.  **undo log 版本链**：
    - 每次更新（Insert/Update/Delete）都会生成一条 undo log。
    - 通过 `DB_ROLL_PTR` 串联成链表，记录了数据的所有历史版本。
    - 链表头部是最新旧版本，尾部是最早版本。
3.  **ReadView（读视图）**：
    - 事务发起 SELECT（快照读）时生成的逻辑快照。
    - **核心字段**：
        - `m_ids`：生成 ReadView 时，当前系统中活跃（未提交）的事务 ID 列表。
        - `min_trx_id`：`m_ids` 中最小的事务 ID。
        - `max_trx_id`：生成 ReadView 时，系统应分配给下一个事务的 ID（即当前最大事务 ID + 1）。
        - `creator_trx_id`：当前事务自己的 ID。

**ASCII 架构图（版本链示意）**：
```text
Current Row in Data Page
+---------------------------+      +---------------------------+
| Value: "C" (Current)      |      | DB_TRX_ID: 100             |
| DB_ROLL_PTR: ------------>------+ (Modified by Tx 100)       |
+---------------------------+      +---------------------------+
           |
           | Roll Pointer
           v
+---------------------------+      +---------------------------+
| Undo Log 1 (Tx 100)       |      | Value: "B"                 |
| DB_TRX_ID: 100            |      | DB_ROLL_PTR: ------------->|------>
| Type: UPDATE              |      +---------------------------+
+---------------------------+                  |
           |                                         |
           v                                         v
+---------------------------+      +---------------------------+
| Undo Log 2 (Tx 80)        |      | Value: "A"                 |
| DB_TRX_ID: 80             |      | DB_ROLL_PTR: NULL          |
| Type: UPDATE              |      +---------------------------+
+---------------------------+
```

【ReadView 判断可见性规则】
对于某个版本的 `trx_id`：
1.  `trx_id == creator_trx_id` → **可见**（自己修改的，肯定能看到）。
2.  `trx_id < min_trx_id` → **可见**（该版本在 ReadView 生成前已提交）。
3.  `trx_id >= max_trx_id` → **不可见**（ReadView 生成后才开始的事务）。
4.  `min_trx_id <= trx_id < max_trx_id`：
    - 在 `m_ids` 列表中 → **不可见**（生成 ReadView 时还未提交）。
    - 不在 `m_ids` 列表中 → **可见**（生成 ReadView 前已提交）。
5.  如果不可见，则沿 `DB_ROLL_PTR` 找 undo log 中的上一个版本，重复判断。

【RC vs RR 区别】
- **Read Committed (RC)**：**每次** SELECT 都会生成一个新的 ReadView。因此能看到其他事务提交后的最新修改。
- **Repeatable Read (RR)**：只在**第一次** SELECT 生成 ReadView，后续所有查询都复用该 ReadView。因此整个事务看到的是同一快照，实现可重复读。

## 常见考点
1.  **快照读 vs 当前读**：MVCC 只解决了快照读（普通 `SELECT`）的并发问题。当前读（`SELECT ... FOR UPDATE`, `UPDATE`, `DELETE`）读取的是最新版本，必须加锁。
2.  **MVCC 能否完全解决幻读**：MVCC 解决了快照读的幻读。当前读解决幻读主要靠 Next-Key Lock。两者结合使得 InnoDB 的 RR 级别解决了幻读问题。
3.  **purge 机制**：Undo Log 什么时候删除？当系统中没有活跃事务需要访问某个历史版本时，purge 线程会回收这些 undo log。
4.  **二级索引与 MVCC**：二级索引记录中没有 `TRX_ID`，它是如何判断可见性的？（二级索引的叶子节点包含主键值，需要回表到聚簇索引查找 `TRX_ID` 进行判断，这叫回表判断可见性）。

## 记忆要点

- 核心概念：MVCC 是多版本并发控制，让普通快照读不加锁，大幅提升并发读性能

- 可见规则：小于最小活跃 ID 或等于自身可见；大于等于下次分配 ID 或在活跃列表中不可见
