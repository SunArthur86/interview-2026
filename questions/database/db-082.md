# 慢SQL如何排查和优化？

排查步骤：
1. **开启慢查询日志**：配置 `slow_query_log=ON` 和 `long_query_time=1`（超过1秒的SQL会被记录）。建议开启 `log_queries_not_using_indexes=ON` 来记录未使用索引的查询。
2. **分析慢日志**：使用 `mysqldumpslow` 工具分析日志文件，提取出现频率高、扫描行数多的SQL。
3. **使用 EXPLAIN 分析执行计划**：重点关注以下字段：
   - **type**：访问类型，性能从好到坏：`system > const > eq_ref > ref > range > index > ALL`。尽量优化到 `ref` 或 `range` 级别，避免 `ALL`（全表扫描）。
   - **key**：实际使用的索引。如果为 `NULL`，说明未使用索引。
   - **rows**：预计扫描的行数。数值越大越可能需要优化。
   - **Extra**：
     - `Using index`：索引覆盖，好。
     - `Using filesort`：外部排序（需在内存/磁盘中排序），消耗资源。通常是因为ORDER BY字段未走索引。
     - `Using temporary`：使用了临时表（常见于GROUP BY, DISTINCT），消耗大。
     - `Using index condition`：索引下推，比回表好但不如覆盖索引。

优化手段：
1. **合理建索引**：
   - **最左前缀原则**：联合索引 `(a, b, c)`，查询条件必须包含最左边 `a` 才能生效。
   - **区分度高的列放左边**：将区分度（高散列值）高的字段放在联合索引前面。
2. **避免索引失效**：
   - 避免在索引列上做运算（如 `WHERE age + 1 = 10`）、函数操作（如 `WHERE YEAR(date) = '2023'`）。
   - 避免隐式类型转换（如字符串列存了数字查询时不加引号）。
   - 避免使用 `!=`, `<>`, `NOT IN`, `LIKE '%xxx'`（前缀模糊匹配）。
3. **SQL写法优化**：
   - **避免 SELECT ***：只查询需要的列，利用索引覆盖。
   - **大分页优化**：`LIMIT 1000000, 10` 会先扫描1000010行。优化方法：
     - **延迟关联**：`SELECT a.* FROM table a JOIN (SELECT id FROM table LIMIT 1000000, 10) b ON a.id = b.id`。子查询只查主键索引（覆盖索引），速度快，再通过主键回表查询10行数据。
   - **ORDER BY 优化**：确保ORDER BY的字段与索引顺序一致，避免 `Using filesort`。
4. **场景优化**：
   - **OR 优化**：如果OR两边的列都有索引，MySQL可能使用索引合并，但效率一般不高，建议改为 UNION ALL。
   - **EXISTS vs IN**：小表驱动大表原则。外表小用IN，内表小用EXISTS。

### 实战案例
某报表系统上线后CPU飙升，排查发现研发对时间字段 `create_time` 使用了 `WHERE DATE(create_time) = '2023-10-01'`，导致索引失效全表扫描。修改为范围查询 `WHERE create_time >= '2023-10-01 00:00:00' AND create_time <= '2023-10-01 23:59:59'` 后，查询耗时从 5s 降至 10ms。

### 代码示例
```sql
-- Bad: 索引失效，全表扫描
SELECT * FROM orders WHERE YEAR(order_date) = 2023;

-- Good: 走索引，范围扫描
SELECT * FROM orders WHERE order_date >= '2023-01-01 00:00:00' AND order_date < '2024-01-01 00:00:00';
```

### 执行计划对比
| 场景 | type | Extra | 说明 |
| :--- | :--- | :--- | :--- |
| 使用函数 YEAR(col) | ALL | Using where | 索引失效，全表扫描并过滤 |
| 范围查询 BETWEEN | range | Using index condition | 索引范围扫描，性能高 |
| 联合索引跨列 | index | Using where; Using index | 索引全扫描（虽覆盖但效率低） |
| 覆盖索引 | ref | Using index | 性能最优，仅查索引树 |
