# redo log和undo log的区别是什么？

redo log 和 undo log 是 InnoDB 存储引擎中非常重要的两种日志，它们在事务处理中起着互补的作用。

### 1. Undo Log（回滚日志）
- **作用**：
  1. **事务回滚**：当事务执行失败或调用 rollback 时，利用 undo log 将数据回滚到修改前的状态。
  2. **MVCC（多版本并发控制）**： undo log 存储了数据的历史版本链。当其他事务进行快照读时，通过 undo log 找到符合其可见性的历史版本数据。
- **内容**：逻辑日志，记录的是“如何回滚”的操作（例如：把 ID=1 的值从 10 改回 5）。
- **产生时机**：事务执行过程中产生，存放在系统表空间的 undo segment 中。

### 2. Redo Log（重做日志）
- **作用**：
  1. **故障恢复**：事务提交时，只要 redo log 持久化成功，事务就算成功。如果数据库在数据页落盘前崩溃，重启后可以根据 redo log 重放修改，保证数据不丢失。
  2. **提高性能**：写操作不再需要每次都随机写磁盘数据页，而是顺序写 redo log（WAL 技术），大大提高了写入速度。
- **内容**：物理日志，记录的是“在某个数据页的某个偏移量处写了什么值”。
- **存储结构**：通常是固定大小的循环写入文件。
- **产生时机**：事务执行过程中产生，事务提交时刷盘（由 `innodb_flush_log_at_trx_commit` 参数控制）。

### 3. 核心区别与联系
| 特性 | Redo Log | Undo Log |
| :--- | :--- | :--- |
| **存储内容** | 物理修改（页偏移+新值） | 逻辑修改（回滚操作/旧值） |
| **主要目的** | 持久性 (D) | 原子性 (A) & MVCC |
| **生命周期** | 循环利用，Checkpoint 后可覆盖 | 随事务提交及 purge 线程清理 |
| **时间点** | 记录“修改后的值” | 记录“修改前的值” |

```text
InnoDB 事务执行与日志写入流程:

  事务开始
     |
     v
[内存 Buffer Pool 修改]  <---> [写 Redo Log (Buffer)]
     |                                    |
     | (记录 Undo Log)                    | (WAL 机制)
     v                                    v
[磁盘 Undo Log]              [刷盘策略决定 Redo Log 落盘]
```

### 实战案例
曾遇到服务器因断电重启，数据库恢复时间过长。经排查，是由于长时间未执行 Checkpoint，导致 Redo Log 中堆积了大量未落盘的脏页，重启时必须回放所有日志。优化策略是调大 `innodb_log_buffer_size` 并合理配置 `innodb_max_dirty_pages_pct`，加快刷盘频率以缩短恢复窗口。

### 代码示例 (MySQL 配置)
```sql
-- 查看日志相关状态
SHOW VARIABLES LIKE 'innodb_flush_log_at_trx_commit';
-- 推荐生产设为 1 (最安全)，若追求性能可妥协为 2 (写OS缓存)

-- 查看 Undo Log 相关
SHOW VARIABLES LIKE 'innodb_undo%';
```

### 常见考点
1. **WAL (Write-Ahead Logging)**：为什么先写日志再写数据页？（将随机写转为顺序写，保证持久性的同时提高性能）。
2. **刷盘参数**：`innodb_flush_log_at_trx_commit` 设置为 0, 1, 2 的区别？（1: 每次提交落盘；0: 每秒落盘；2: 每次提交写 OS 缓存，每秒落盘）。
3. **日志崩溃恢复顺序**：重启时，先利用 Redo Log 恢复已提交但未落盘的数据，再利用 Undo Log 回滚未提交的事务。
4. **Doublewrite Buffer**：Redo Log 是物理日志，为什么还需要双写缓冲区？（解决「页断裂」问题：如果写 16KB 页时只写了 4KB 就宕机，Redo Log 无法恢复这损坏的页）。

- 作用对比：redo log保持久性(崩溃恢复)，undo log保原子性(回滚)与MVCC

- WAL机制：先写redo log再写磁盘数据页，将随机写转为顺序写提性能
- 刷盘参数：innodb_flush_log_at_trx_commit设1每次落盘最安全，0或2性能好但易丢
- 崩溃恢复顺序：重启时先用redo恢复未落盘脏页，再用undo回滚未提交事务
