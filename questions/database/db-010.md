# 第二范式(2nd NF－每个表只描述一件事情)是什么？

### 第二范式 (2NF)

**定义**：
在满足第一范式（1NF）的基础上，消除非主属性对码的**部分依赖**。即：非主键字段必须完全依赖于主键（针对复合主键的情况）。

**说明**：
如果主键只包含一个字段，则天然满足 2NF。2NF 主要针对复合主键场景，要求非主键列不能只依赖主键的一部分，必须依赖整个主键。

**详细解析**：
假设存在一张订单明细表 `OrderDetail`，包含字段：`(OrderID, ProductID)`, `ProductName`, `Quantity`, `UnitPrice`。

*   **主键**：`(OrderID, ProductID)` (联合主键)
*   **依赖关系**：
    *   `Quantity` 依赖于 `(OrderID, ProductID)` —— **完全依赖**
    *   `ProductName` 只依赖于 `ProductID` —— **部分依赖**
    *   `UnitPrice` 只依赖于 `ProductID` —— **部分依赖**

**违反 2NF 的问题**：
1.  **数据冗余**：同一个 `ProductID` 对应的 `ProductName` 会在多个订单中重复存储。
2.  **更新异常**：修改产品名称时，必须更新所有包含该产品的订单记录，否则数据不一致。
3.  **插入异常**：如果没有订单，新产品无法录入系统（因为主键包含 OrderID，不能为空）。

**规范化处理**：
将表拆分为两张：
1.  `OrderDetail`: `(OrderID, ProductID, Quantity)` —— 满足 2NF
2.  `ProductInfo`: `(ProductID, ProductName, UnitPrice)` —— 满足 2NF

**依赖关系示意图**：
```text
违反 2NF 的设计:
(主键: OrderID + ProductID)
+-----------+------------+-------------+----------+
| OrderID   | ProductID  | ProductName | Quantity |
+-----------+------------+-------------+----------+
|    001    |     A1     |   Mouse     |    2     |  <-- ProductName 只依赖 ProductID
|    002    |     A1     |   Mouse     |    5     |  <-- 数据冗余
+-----------+------------+-------------+----------+

拆分后 (符合 2NF):
[ 订单表 ]                [ 商品表 ]
(OrderID, ProductID)      (ProductID, ProductName)
      |                           |
      +------- OrderID            +------- ProductID
              |                           |
              +----> Quantity             +----> ProductName
```

**实战案例**：
在设计一张“学生选课成绩表”时，主键设为 `(StudentID, CourseID)`。最初我误将 `CourseName` 也放入此表。每当课程改名，需更新全校所有选课该课程的记录。修复方案是将 `CourseName` 移至独立的 `Course` 表，选课表仅保留外键引用。

**对比表格：违反 2NF vs 符合 2NF**

| 维度 | 违反 2NF (未拆分) | 符合 2NF (已拆分) |
| :--- | :--- | :--- |
| **数据冗余** | 高 (商品名称重复存储) | 低 (仅引用 ID) |
| **更新开销** | 高 (需更新所有历史订单) | 低 (只更新商品表一行) |
| **插入限制** | 无法单独录入新品 (需有订单) | 可随意录入新品 |
| **查询复杂度** | 低 (单表查询) | 中 (需 JOIN 关联商品名) |

**关键代码示例**：
```sql
-- 违反 2NF 的设计
CREATE TABLE order_bad (
    order_id INT,
    product_id INT,
    product_name VARCHAR(50), -- 部分依赖，仅依赖 product_id
    quantity INT,
    PRIMARY KEY (order_id, product_id)
);

-- 修正为符合 2NF 的设计
CREATE TABLE products (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(50)
);

CREATE TABLE order_items (
    order_id INT,
    product_id INT,
    quantity INT,
    PRIMARY KEY (order_id, product_id)
);
```

## 常见考点
1.  **函数依赖**：能够准确判断什么是部分函数依赖、完全函数依赖和传递函数依赖。
2.  **实际设计权衡**：在实际的高并发 OLTP 系统中，为了避免过多的 JOIN，有时我们会故意违反 2NF，将部分依赖字段保留在同一张表中（如订单表中冗余商品快照信息），这叫空间换时间。
3.  **候选码与主键**：如果有多个候选码，需确保所有非主属性对每一个候选码都是完全依赖，才算满足 2NF。

## 记忆要点

- 核心定义：在1NF基础上，消除非主属性对码的部分依赖（必须完全依赖）。
- 适用场景：主要针对复合主键，若单字段主键则天然满足2NF。
- 存在风险：因为存在部分依赖，所以会导致数据冗余、更新及插入异常。
- 解决方案：将表拆分，使非主键字段只依附于其完全依赖的主键。
