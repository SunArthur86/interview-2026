# MySQL 的意向锁（IS/IX）是什么？为什么需要它？和 MDL 锁有什么区别？

【意向锁（表级锁）】是 InnoDB 实现的表级锁，用于表明事务打算在表中某些行上加什么类型的锁。
- **IS (意向共享锁)**：事务打算给表中某些行加共享锁（S锁，即读锁）。
- **IX (意向排他锁)**：事务打算给表中某些行加排他锁（X锁，即写锁）。

【为什么需要？】
如果没有意向锁，当某个事务持有了行锁，另一个事务想要对整个表加表锁（如 ALTER TABLE）时，就必须遍历表中的每一行，检查是否有行锁被锁定，这在数据量大时效率极低。有了意向锁，事务在加行锁前会先自动在表级加上对应的意向锁。
- **原理**：表锁的申请者只需检查表上是否有与之冲突的意向锁即可（O(1) 复杂度），无需逐行检查。

【兼容性矩阵】
意向锁之间是兼容的（因为它们代表不同行上的意图，或者只是打算），但与表级的 S/X 锁有冲突：
- IS 兼容 IX（读读兼容，读写意向共存）
- IS 兼容 表级S
- IX 兼容 表级S、表级X
- 表级S 与 IX 冲突
- 表级X 与 IS、IX 都冲突

【MDL 锁（Metadata Lock，元数据锁）】
- **作用位置**：MySQL Server 层，不区分引擎。
- **作用**：保护表结构定义，防止 DML 与 DDL 冲突。
- **加锁规则**：
  - DML（SELECT/INSERT/UPDATE/DELETE）：自动加 MDL 读锁（共享），事务提交后释放。
  - DDL（ALTER/DROP/TRUNCATE）：自动加 MDL 写锁（排他）。
- **线上故障场景**：如果一个长事务正在执行查询（持有 MDL 读锁），后续的对该表的 DDL 语句被阻塞（等待 MDL 写锁）。由于 DDL 队列头阻塞，后续所有对该表的 DML 语句也将排队等待，导致数据库连接数飙升（Thread Full）。

【意向锁 vs MDL 锁】
| 维度 | 意向锁 | MDL 锁 |
| :--- | :--- | :--- |
| **层级** | InnoDB 引擎层 | MySQL Server 层 |
| **目的** | 协调**行锁**与**表锁**的冲突 | 协调**DML**与**DDL**的冲突 |
| **锁粒度** | 表级锁 | 表级锁 |
| **加锁时机** | 在对数据行加锁前自动添加 | 在访问表时自动添加 |

## 常见考点
1. **为什么意向锁是表级锁却设计成「自动加」的？**
   为了减少用户操作复杂度，且意向锁本身就是为了提升表锁检查效率，必须与行锁操作强绑定（DML 语句执行时自动触发），不能由用户手动干预。
2. **如何解决 MDL 锁导致的表被锁死？**
   - 在 DDL 语句中指定 `NOWAIT` 或 `WAIT n`，避免无限等待。
   - 使用 `pt-online-schema-change` 或 `gh-ost` 等工具进行在线 DDL，避免长时间锁表。
   - Kill 掉持有 MDL 读锁的长事务。
3. **意向锁和行锁的加锁顺序是怎样的？**
   先加表级意向锁（IS/IX），成功后再加行级锁（S/X）。这是为了保证加锁的原子性和一致性。

## 记忆要点

- 意向锁（IS/IX）在加行锁前自动添加，用于快速判断表锁冲突，避免逐行检查
- 意向锁之间互相兼容，其核心目的是协调「行锁」与「表锁」的冲突
- MDL锁在Server层，DML加共享读锁，DDL加排他写锁，防止结构变更引发冲突
- 意向锁属于引擎层防锁冲突，MDL属于Server层防DDL/DML冲突
