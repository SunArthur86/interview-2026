# 什么是聚簇索引和非聚簇索引？

聚簇索引：
- **定义**：索引结构的叶子节点直接存储了整行数据。索引文件就是数据文件。
- **特性**：一张表只能有一个聚簇索引（通常为主键）。在InnoDB中，聚簇索引就是主键索引。
- **物理顺序**：数据行的物理存储顺序与索引键的逻辑顺序一致。这也意味着聚簇索引对于主键的范围扫描非常高效。

非聚簇索引（二级索引/辅助索引）：
- **定义**：索引结构的叶子节点存储的是索引列的值和对应的主键值，而不是完整的行数据。
- **回表**：需要通过主键值再去聚簇索引中查找完整的行数据。

区别总结：
- **查询速度**：聚簇索引通常更快（一次B+树查找），非聚簇索引需要“回表”（两次B+树查找：先查二级索引拿到PK，再查聚簇索引拿数据）。
- **数量限制**：聚簇索引一个表只能有一个；非聚簇索引可以有多个。
- **排序**：聚簇索引天然支持按主键排序（ORDER BY）。

**MyISAM 与 InnoDB 的区别**：
- MyISAM 引擎所有索引都是非聚簇索引，索引文件和数据文件是分离的（.MYI 存索引, .MYD 存数据），叶子节点存储的是数据文件的物理地址偏移量。
- InnoDB 引擎主键是聚簇索引，其他索引是非聚簇索引（存储主键值）。

```text
InnoDB 索引结构对比

聚簇索引 (主键)            二级索引 (如 index_name)
┌──────────────┐           ┌──────────────┐
│ Non-Leaf Key │           │ Non-Leaf Key │
│ (仅存索引值)  │           │ (仅存索引值)  │
└──────┬───────┘           └──────┬───────┘
       │                          │
       ▼                          ▼
┌──────────────┐           ┌──────────────┐
│ Leaf Page    │           │ Leaf Page    │
│ [PK|所有列]  │ <─────┐   │ [Name|PK]    │
│ [PK|所有列]  │       │   │ [Name|PK]    │
└──────────────┘       │   └──────────────┘
        ▲             │         │
        │  回表查询   │         │
        └─────────────┘         │
              (通过PK匹配)      │
                               │
                       1. 查二级索引得 PK=5
                       2. 拿着PK=5去左边查聚簇索引
```

## 常见考点
1. **为什么建议使用整型自增主键**：整型占用空间小，自增使得数据顺序写入，减少页分裂。如果使用长的字符串（如UUID）作为主键，会导致非聚簇索引体积变大（因为二级索引叶子节点存了主键），且导致随机IO。
2. **二级索引为什么要存主键而非物理地址**：因为聚簇索引的数据页可能会发生页分裂或合并，导致物理地址变化。如果存物理地址，需要维护所有二级索引的成本太高。存主键只需在聚簇索引变动时维护一次，或者主键本身稳定。
3. **非聚簇索引一定比聚簇索引慢吗**：不一定。如果查询只需要覆盖索引包含的字段，可以直接从二级索引返回，无需回表，这时可能比查聚簇索引（涉及更多列的数据读取）更快（即索引覆盖）。
