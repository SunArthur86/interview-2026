# 什么是内存淘汰策略？

过期删除策略有哪些

定时删除：设置 key 的过期时间时，同时创建一个定时事件，当时间到达时，由事件处理器自动执行 key 的删除操作。这样可以保证过期的 key 会被尽快删除，但删除过期 key 会占用一部分 CPU 时间，对CPU不友好。
惰性删除： 不主动删除过期键，每次从数据库访问 key 时，都检测 key 是否过期，如果过期则删除该 key。这样会使用很少的系统资源，对CPU友好，但是可能会导致过期key长期不被删除，浪费内存空间。
定期删除：每隔一段时间「随机」从数据库中取出一定数量的 key 进行检查，并删除其中的过期key。这种方法减少了删除操作对cpu的影响，也减少了内存空间的浪费，但是效果也没有两者好，且难以确定删除操作执行的时长和频率。

Redis过期删除策略是什么

Redis 选择「惰性删除+定期删除」这两种策略配和使用，以求在合理使用 CPU 时间和避免内存浪费之间取得平衡。
1. 惰性删除依据expireIfNeeded 函数来实现，每次返回或者修改key之前，都会调用该函数检查key 是否过期
如果过期，删除该key, lazyfree_lazy_expire 参数决定是同步删除还是异步删除，然后返回null 给客户端
如果没有过期，则返回正常的键给客户端
2. 定期删除的做法是每隔一段时间「随机」从数据库中取出一定数量的 key 进行检查，并删除其中的过期 key。
默认每隔10s检查一次数据库，配置键为hz: 10
随机抽查的数量是20个key
 定期删除的流程：
从过期字典中随机抽取 20 个 key
检查这 20 个 key 是否过期，并删除已过期的 key；
如果本轮检查的已过期 key 的数量，超过 5 个（20/4），也就是「已过期 key 的数量」占比「随机抽取 key 的数量」大于 25%，则继续重复步骤 1；如果已过期的 key 比例小于 25%，则停止继续删除过期 key，然后等待下一轮再检查。

Redis内存淘汰

Redis通过参数 maxmemory <bytes>  来设定最大运行内存。当系统内存不足时，Redis会根据配置的淘汰策略来删除一些键以释放内存。

内存淘汰策略

1. 不进行数据淘汰
NoEviction： 如果内存不足以执行写操作，Redis将返回错误，默认的淘汰策略。
2. 进行数据淘汰
VolatileTTL：优先淘汰更早过期的键值。
VolatitleLRU:  只对带有过期时间的键使用LRU策略，其他键使用NoEviction策略。
VolatitleRandom： 只对带有过期时间的键使用随机淘汰，其他键使用NoEviction策略。
VolatitleLFU: 只对带有过期时间的键使用LFU策略，其他键使用NoEviction策略。
AllKeysLRU： 根据最近最少使用的原则淘汰最久未使用的键。

### 补充关键细节

**1. 过期策略与内存淘汰的区别**
- **过期删除**：被动+主动清理，目的是清理“无用”数据（Time-To-Live 到了），无论内存是否够用都会执行（惰性+定期）。
- **内存淘汰**：被动防御，只有在内存达到 `maxmemory` 上限时才触发。目的是保活（让 Redis 能继续写入），此时可能会删掉“没过期”但有用的数据（取决于策略）。

**2. 异步删除 (`lazyfree`)**
- 在 Redis 4.0 引入。当删除大 Key（如几 MB 的 List 或 Hash）时，同步删除会阻塞主线程，导致 Redis 响应变慢（延迟抖动）。
- 配合参数如 `lazyfree-lazy-eviction` (淘汰时异步)、`lazyfree-lazy-expire` (过期时异步)，将释放内存的工作放到后台线程（BIO）中执行。

**3. 脏检查与 LRU/LFU 算法关联**
- 在 `allkeys-lru` 等策略中，即使内存未满，Redis 也会在访问 key 时更新其 LRU/LFU 的元数据，以便在需要淘汰时能迅速选出最该被淘汰的 key。

## 常见考点

1. **如果定期删除漏掉了大量过期 key，导致内存满了怎么办？**
   - 内存满时，Redis 会优先进行内存淘汰。即使 key 没过期，如果策略是 `allkeys-*`，也会被删除。
2. **为什么定期删除是随机的，而不是全遍历？**
   - 为了避免阻塞主线程。Redis 是单线程的，全量遍历数据库会导致客户端请求卡顿。
3. **Redis 里的 LRU 算法是完美的吗？**
   - 不是，它是近似 LRU。可以通过调整 `maxmemory-samples` 参数来牺牲 CPU 换取精确度（比如设为 10）。
