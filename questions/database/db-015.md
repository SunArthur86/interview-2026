# 什么是MySQL主从复制？

### MySQL 主从复制原理

MySQL 主从复制将主库的数据变更同步到一个或多个从库，常用于读写分离、数据备份、高可用架构。

**核心流程（基于 binlog 的异步复制）**：

1.  **Master (主库)**：
    *   数据变更发生时，按照事务顺序写入 **Binlog (二进制日志)**。
    *   **Binlog Dump 线程**：当从库连接过来时，负责读取 Binlog 并发送给从库。

2.  **Slave (从库)**：
    *   **I/O 线程**：向主库发起连接，接收 Binlog Dump 线程发送的日志，并写入本地的 **Relay Log (中继日志)**。
    *   **SQL 线程**：读取 Relay Log 中的事件，解析并重放 SQL 语句，将数据变更应用到从库自身的存储引擎。

**ASCII 复制流程图**：

```text
+-------------------+          +-------------------+
|     Master (主库)  |          |     Slave (从库)   |
+---------+---------+          +---------+---------+
          |                              |
| 1. 事务提交 |                           |
| 写入 Binlog|                           |
+-----+-----+                           |
      |                                 |
      | 2. Dump 线程读取 Binlog           |
      +--------------------------------->| (网络传输)
                                        |
+----------------------------------+     |
| 3. IO 线程接收并写入 Relay Log   |<----+
+----------------+-----------------+
                 |
                 | 4. SQL 线程读取 Relay Log
                 v
+----------------------------------+
| 5. 重放 SQL，应用数据变更        |
+----------------------------------+
```

**三种复制模式**：

| 模式 | 机制 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **异步复制** | 主库提交即返回，不等待从库 | 性能最好，延迟最低 | 主库宕机可能丢失数据 | 对数据一致性要求不高的日志、报表 |
| **半同步复制** | 至少等待一个从库接收 Relay Log (ACK) | 数据安全性高，几乎不丢数据 | 主库性能受网络延迟影响 | 金融、支付等核心业务场景 |
| **组复制 (MGR)** | 基于 Paxos/Raft，多数节点确认 | 强一致性，自动故障切换，支持多主 | 架构复杂，网络开销大 | 对高可用和一致性要求极高的架构 |

**复制延迟问题与优化**：
*   **原因**：大事务、主库并发高写入、从库单线程 SQL 线程重放速度跟不上主库写入速度。
*   **优化**：
    *   **并行复制**：MySQL 5.7+ 引入 MTS (Multi-Threaded Slave)。根据“库级别”、“LOGICAL_CLOCK”或“WRITESET”并行回放 Relay Log。
    *   **硬件升级**：从库使用 SSD 加速 IO。
    *   **控制事务大小**：避免大事务，分批提交。

## 常见考点

1.  **主从复制延迟怎么解决？**
    答案核心：业务层面“读后一致性写”（如主库写后强制读主库 1 秒）；架构层面使用并行复制（MTS）；监控 `Seconds_Behind_Master`。

### 深化补充

**实战案例**：
某电商大促期间，主库执行了一个批量更新库存的 `UPDATE` 语句（修改了 50 万行），导致从库应用该 Relay Log 耗时 5 分钟，期间所有读从库的订单查询都查不到旧库存，引发超卖报警。

**关键代码 (SQL)**：
```sql
-- 开启并行复制 (MySQL 5.7+ 优化配置)
-- slave_parallel_workers 设置为 4，表示开启 4 个 SQL 线程并行回放
-- slave_parallel_type 设置为 LOGICAL_CLOCK，基于逻辑时钟并行
SET GLOBAL slave_parallel_workers = 4;
SET GLOBAL slave_parallel_type = LOGICAL_CLOCK;

-- 修复大事务导致的延迟：将大事务拆分为小事务
-- 错误做法：
-- UPDATE order_items SET status = 2 WHERE create_time < '2023-01-01';

-- 正确做法（分批处理）：
UPDATE order_items SET status = 2 WHERE id > 0 AND id < 10000 AND create_time < '2023-01-01';
-- ... 循环执行下一批
```
