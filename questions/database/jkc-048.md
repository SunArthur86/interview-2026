# Standard Column Family（相对应SQL数据库中的table）

### Standard Column Family（标准列族）

**概念**：
**Column Family (CF)** 是 Cassandra 中数据存储的逻辑容器，直接对应于 SQL 数据库中的 **Table（表）**。在 CQL（Cassandra Query Language）出现后，Standard Column Family 即等同于我们常说的“表”。

**结构**：
*   一个 CF 由一系列行组成。
*   每一行由一个唯一的 Key（Row Key/Partition Key）和该 Key 对应的若干个 Column 组成。

**架构示意**：
```text
+---------------------------------------------------+
| Standard Column Family (Table)                    |
+---------------------------------------------------+
| Row Key: "user_100"                               |
| +-----------------------+-----------------------+ |
| | Column: name          | Value: "Alice"        | |
| | Column: age           | Value: 30             | |
| | Column: email         | Value: "a@b.com"     | |
| +-----------------------+-----------------------+ |
+---------------------------------------------------+
| Row Key: "user_101"                               |
| +-----------------------+-----------------------+ |
| | ...                   | ...                   | |
| +-----------------------+-----------------------+ |
+---------------------------------------------------+
```

**特点与细节**：
*   **物理存储**：在物理存储上，同一个 Column Family 的数据通常存储在同一个文件目录下（SSTable）。Cassandra 是按列族存储的，这意味着虽然查询行很方便，但过度设计列族数量会增加 IO 开销。
*   **排序与比较器**：在定义 CF 时，需要指定行键比较器和列名比较器。数据在磁盘上是按 Partition Key 有序存储的，且同一行内的列也是按列名有序存储的（基于特定的排序顺序，如 UTF8 或 Long）。
*   **模式灵活性与固定性**：
    *   **底层**：Cassandra 本质是 Schema-less 的，每一行可以有不同的列。
    *   **CQL 层**：为了兼容 SQL 习惯，Standard Column Family 通常强制要求定义固定的列结构，像传统 SQL 表一样，但在内部实现上仍然支持动态列。
*   **宽行特性**：如果未定义聚类列，Standard CF 可以作为“宽行”存储，即一行包含数百万列，适用于时间序列数据。

**实战案例**：
早期电商系统按“订单状态”（如待支付、已发货）拆分为多个 Table，导致节点上目录碎片过多，Compaction 压力剧增。最佳实践是将同一业务实体聚合在一个 Table（CF）中，利用 `Clustering Column` 进行状态区分。

**代码示例**：
```cql
-- 定义一个利用排序特性的 Standard Column Family (Table)
-- 数据将按 event_time 降序存储在磁盘上，利于范围查询
CREATE TABLE user_events (
  user_id uuid,
  event_time timestamp,
  event_type text,
  payload text,
  PRIMARY KEY (user_id, event_time)
) WITH CLUSTERING ORDER BY (event_time DESC);
```

**Cassandra CF 与 SQL Table 对比**：

| 特性 | Cassandra Table (CF) | SQL Table |
| :--- | :--- | :--- |
| **底层存储顺序** | 总是按 Partition Key 排序 | 无序（堆表）或按索引排序 |
| **写入模式** | 顺序写 | 随机写/日志写 |
| **列灵活性** | 支持动态列（每行列数可不同） | 固定列结构 | 
| **Schema 变更** | 轻量级（元数据修改） | 重量级（可能锁表或重写数据） |
| **文件结构** | SSTable (不可变) | 数据页 (可变/覆盖) |

## 常见考点
1. **Cassandra 中的 Column Family 和 SQL 中的 Table 有什么本质区别？**
   *   答：虽然 CQL 中它们用法相似，但底层 CF 是有序的（Key 有序，列名有序），而 SQL 表通常是无序堆存储（除非有 Clustered Index）。此外，Cassandra 的 CF 原生支持动态列。
2. **什么是宽行？**
   *   答：在 Standard Column Family 中，如果行内列数非常多（例如百万级），称为宽行。设计时需注意避免分区过大导致内存或修复问题。
3. **为什么说 Cassandra 是写密集型优化的？**
   *   答：写入 CF 时是顺序写追加到 MemTable 和 Commit Log，不需要随机 IO，这得益于其列族存储结构。


## 记忆要点

- 基本概念：标准列族直接对应SQL中的Table，是数据存储的逻辑容器。
- 存储特性：底层总是按Partition Key和列名有序排列，SQL堆表通常无序。
- 灵活Schema：同一张表每一行可以拥有数量不等的动态列。
- 查询优化：结合Clustering Column设计宽行，利用排序特性加速范围查询。
