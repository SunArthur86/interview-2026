# Read View在MVCC中是如何工作的是什么？

Read View 是 MVCC（多版本并发控制）中用于判断数据版本可见性的机制。

### 核心作用
实现 **RC（读已提交）** 和 **RR（可重复读）** 隔离级别，使得事务在进行快照读（普通 Select）时，可以看到数据的历史版本，而不需要加锁。

### Read View 内部字段
当事务发起快照读时，会生成一个 Read View，包含以下关键字段：
- **m_ids**：生成 Read View 时，当前系统中活跃（未提交）的事务 ID 列表。
- **min_trx_id**：m_ids 中最小的事务 ID。
- **max_trx_id**：生成 Read View 时，系统应分配给下一个事务的 ID（即当前最大事务 ID + 1）。
- **creator_trx_id**：生成该 Read View 的事务本身的 ID。

### 可见性判断算法与流程
当事务读取某行记录时，通过该行记录隐藏列 `trx_id` 与 Read View 进行比较：
1. **如果是自己改的**：`trx_id == creator_trx_id`，可见。
2. **如果是已提交的旧数据**：`trx_id < min_trx_id`，说明该版本在 Read View 生成前已提交，可见。
3. **如果是未来的新数据**：`trx_id >= max_trx_id`，说明该版本在 Read View 生成后才开启，不可见。
4. **如果是活跃事务**：`min_trx_id <= trx_id < max_trx_id`，需判断 `trx_id` 是否在 `m_ids` 列表中：
   - **在列表中**：说明生成 Read View 时该事务还未提交，不可见。
   - **不在列表中**：说明生成 Read View 时该事务已提交，可见。

如果不可见，则通过 `roll_pointer` 指针顺着 undo log 版本链往下找，直到找到可见的版本为止。

- **实战案例**：在金融对账系统中，需要在一个长事务中多次查询同一账户余额。若使用 RC 级别，每次查询看到其他事务提交的变动会导致对账不平；使用 RR 级别配合 Read View 机制，保证了整个对账过程中看到的数据快照一致。

- **代码示例（概念性伪代码）**：
```java
// 伪代码：Read View 可见性判断逻辑
boolean isVisible(TransactionId trx, ReadView view) {
    if (trx.id == view.creatorId) return true;
    if (trx.id < view.minTrxId) return true; // 已提交
    if (trx.id >= view.maxTrxId) return false; // 未来事务
    return !view.activeIds.contains(trx.id); // 检查是否活跃
}
```

### RC 和 RR 的区别
| 隔离级别 | Read View 生成时机 | 效果 |
| :--- | :--- | :--- |
| **RC (Read Committed)** | 每次执行 SELECT 语句时生成 | 能读到其他事务提交后的最新修改（不可重复读） |
| **RR (Repeatable Read)** | 只在事务启动时生成一次 | 整个事务期间读取的都是同一个快照（可重复读） |
