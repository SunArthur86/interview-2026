# Elasticsearch 的倒排索引中，FST（Finite State Transducer，有限状态传感器）结构是如何优化 Term Dictionary 存储的？它比 B+ Tree 好在哪里？

FST (Finite State Transducer) 是一种 Trie 树的变体，不仅压缩了前缀，还共享了后缀。ES 将 Term Dictionary 从磁盘加载到内存时，使用 FST 结构。相比 B+ Tree，FST 的优势在于极高的内存压缩率（通常能压缩到 1/10 甚至更小）和极快的查询速度。B+ Tree 需要存储每个完整的 Key 和指针，而 FST 将整个词典构建成一个有限状态机，共享公共前缀和后缀的路径，遍历 FST 即可找到 Term 在磁盘上的 Block 偏移量。这使得 ES 能够将庞大的 Term Dictionary 常驻内存，极大减少了磁盘 I/O。

**FST 结构原理细节**：
FST 是一种将键值对映射为图的紧凑结构。它具有两个核心特性：
1.  **最小性**：对于给定的映射关系，它使用的状态数最少。
2.  **有向性**：由一组状态和转换组成，每个转换上有一个输入字符和一个可能的输出值（如 Term 在 Posting List 中的偏移量）。

FST 构建时会将 Terms 排序，从左到右扫描。如果当前 Term 与前一个 Term 有公共前缀，则复用前缀对应的状态路径；只有在分离点（分叉点）才会创建新的状态；如果后缀也相同，则通过共享后缀节点进一步压缩。最终的叶子节点会存储指向磁盘上 Postings List（倒排表）文件指针。

```text
FST 结构示意 (Terms: mop, moth, pop, star, stop)

       ┌──(m)──┐
       │       ▼
       │    (o)──(p) [Offset_1]
       │       │
       │       └──(t)──(h) [Offset_2]
       │
(s) ───┤
       │       ┌──(t)──(a)──(r) [Offset_3]
       │       │
       │       └──(t)──(o)──(p) [Offset_4]
       │
       └──(p)──(o)──(p) [Offset_5]

注：[Offset_x] 代表输出值，即磁盘上的文件指针
公共前缀 'o', 't', 'op' 等均被复用
```

**实战案例**：在一个包含亿级用户标签的ES集群中，Term Dictionary如果用 HashMap 存储需消耗 20GB+ JVM 内存，频繁导致 Full GC。改用 Lucene 的 FST 后，内存占用降至 2GB 左右，查询响应从不稳定的秒级降低到毫秒级，解决了 JVM 内存压力。

**代码示例（Java：构建FST逻辑简化）**：
```java
// 构建 FST 需要排序后的输入
InputsOutputs<Long> pairs = new InputsOutputs<>();
pairs.add(Util.toIntsRef(" mop"), 1L);
pairs.add(Util.toIntsRef(" moth"), 2L);

// 构建器配置
Builder<Long> builder = new Builder<>(
    FST.INPUT_TYPE.BYTE1, NoOutputs.getSingleton());

// 构建最终 FST
FST<Long> fst = builder.build();
```

**对比表格**：

| 特性 | B+ Tree | FST (Finite State Transducer) |
| :--- | :--- | :--- |
| **前缀压缩** | 仅在节点内部有限压缩 | 极致压缩，共享整个前缀和后缀路径 |
| **内存占用** | 高 (存完整Key + 指针) | 极低 (通常为 HashMap 的 1/10) |
| **查询复杂度** | O(log N) | O(Key Length)，通常比 log N 快 |
| **适用场景** | 动态更新频繁的索引 | 读多写少/静态词典，如 ES Term Dictionary |

## 记忆要点

- FST兼具最小性与有向性，将庞大词典构建为共享前后缀的状态机
- FST将Term Dictionary常驻内存，极大减少磁盘IO
- B+树存完整Key耗内存且需O(logN)，而FST极致压缩且查找速度为O(k)
- 适用场景：B+树适合频繁动态更新，而FST适合读多写少静态词典

## 结构化回答


**30 秒电梯演讲：** FST像压缩地铁图，只存站点和分叉（前后缀复用）；B+树像通讯录，每条路都要完整抄写名字，占地又翻得慢。

**展开框架：**
1. **FST是Trie树变体** — 共享前缀与后缀，结构极其紧凑
2. **内存压缩率极高（1/10）** — 使海量Term Dictionary常驻内存
3. **配合磁盘Term** — Index，先内存FST定位，再磁盘Block二分查找
