# 什么是Redis缓存？

Redis 缓存的使用场景与常见问题：

**典型场景：**
1. **缓存热点数据**：商品详情、用户信息，减轻 DB 压力。
2. **会话存储**：分布式 Session。
3. **排行榜/计数器**：ZSET、INCR。
4. **分布式锁**：SETNX/Redlock。
5. **消息队列**：List（简易）、Stream（5.0+）。

**三大经典问题：**
1. **缓存穿透**：查询不存在的 key，每次打到 DB。解决：布隆过滤器、缓存空值（设置较短 TTL）。
2. **缓存击穿**：热点 key 过期瞬间，大量请求打 DB。解决：互斥锁重建、热点 key 永不过期（逻辑过期）。
3. **缓存雪崩**：大量 key 同时过期或 Redis 宕机。解决：过期时间加随机扰动、多级缓存、集群高可用、限流降级。

**缓存一致性：** 先更新 DB 再删缓存（Cache Aside），或用消息队列/binlog 订阅（Canal）异步更新，最终一致。

**缓存更新策略流程：**
```text
┌─────────┐      ┌─────────┐      ┌─────────┐
│ Client  │      │  Cache  │      │   DB    │
└────┬────┘      └────┬────┘      └────┬────┘
     │ Read          │ Miss           │
     │──────────────▶│───────────────▶│
     │               │                │
     │               │                │
     │◀──────────────│◀───────────────│
     │  Return Data  │  Set Data      │
     │               │                │
     │               │                │
     │ Write(Update) │                │
     │───────────────▶│ Delete (Async)│
     │               │───────────────▶│
     │               │                │
```

| 问题 | 现象 | 原因 | 解决方案 |
| :--- | :--- | :--- | :--- |
| **缓存穿透** | DB 压力激增，Redis 无效命中 | 恶意请求不存在的 Key | 1. 布隆过滤器拦截 <br> 2. 缓存空对象 (TTL 短) |
| **缓存击穿** | 热点 Key 失效瞬间 DB 卡死 | 某个极度热点 Key 过期 | 1. 互斥锁防止并发重建 <br> 2. 逻辑过期 (后台异步更新) |
| **缓存雪崩** | 大量请求直接打到 DB，服务不可用 | 大量 Key 集中过期或 Redis 宕机 | 1. TTL 加随机值 <br> 2. 高可用架构 <br> 3. 限流降级 |

**实战案例：**
曾遇到“布隆过滤器误判”导致正常请求被拦截的问题。**排查**：因 Redis 重启导致布隆过滤器数据丢失。**改进**：将布隆过滤器本身持久化到 Redis，并设置 backup 恢复机制，同时保留空值缓存兜底。

**关键代码（互斥锁防击穿）：**
```java
public String getWithMutex(String key) {
    String value = redis.get(key);
    if (StringUtils.isEmpty(value)) {
        // 获取分布式锁
        if (tryLock(key + ":lock")) {
            try {
                // 再次查缓存，防止并发重复查库
                value = redis.get(key);
                if (StringUtils.isEmpty(value)) {
                    value = db.query(); // 查数据库
                    redis.set(key, value, 300);
                }
            } finally {
                unlock(key + ":lock");
            }
        } else {
            // 获取锁失败，休眠重试或返回旧值
            Thread.sleep(100);
            return getWithMutex(key);
        }
    }
    return value;
}
```

## 常见考点
1. **布隆过滤器的原理及缺点？**
   - 原理：利用多个 Hash 函数将 Key 映射到位数组中。优点：空间效率极高。缺点：存在误判，即可能判断 Key 存在但实际不存在；删除困难。
2. **先删缓存还是先更新数据库？为什么？**
   - 推荐先更新 DB，再删缓存。先删缓存可能导致并发请求在读 DB 时缓存未建立，造成数据不一致；先更新 DB 能保证 DB 是最新的源头。
3. **如何保证 Redis 和 DB 的最终一致性？**
   - 延时双删（更新 DB 后删一次，延迟几秒再删一次）或者订阅 MySQL Binlog 通过 Canal 解析后异步更新/删除 Redis。

## 记忆要点

- 穿透对策：布隆过滤器拦截或缓存空值；击穿对策：互斥锁重建或逻辑永不过期。
- 雪崩对策：TTL加随机扰动，多级缓存配合高可用集群，做好限流降级。
- 一致性策略：推荐先更新DB再删缓存，若需强一致可订阅Binlog异步刷新。

## 延伸：缓存穿透、缓存雪崩、缓存击穿是什么？

> 合并自 `db-034`（相似度 69%）

### Redis 单线程模型
1. **纯内存操作**：绝大部分请求在内存中完成，速度快。
2. **IO 多路复用**：单线程处理多个连接请求，减少上下文切换开销（如 epoll）。
3. **避免锁竞争**：单线程模型天然保证了原子性，无需加锁，避免死锁和切换消耗。

*注：指网络请求处理模块单线程，持久化（如 RDB 生成）仍由子进程完成。*

### 缓存异常问题

**1. 缓存穿透**
- **现象**：查询不存在的 key，请求直达 DB。
- **解决**：缓存空值（短 TTL）、布隆过滤器、参数校验。
- **实战案例**：曾经遇到恶意攻击者遍历大量手机号撞库，导致 DB CPU 飙升。后续接入布隆过滤器（Guava 实现）将请求拦截在缓存层。
- **代码示例**：
```java
// 伪代码：缓存空值逻辑
String value = redis.get(key);
if (value == null) {
    User dbUser = db.select(key);
    if (dbUser == null) {
        // 缓存空值，防止频繁穿透
        redis.setex(key, 60, ""); 
    } else {
        redis.set(key, dbUser);
    }
}
```

**2. 缓存击穿**
- **现象**：热点 Key 过期瞬间，大量请求直达 DB。
- **解决**：加互斥锁、设置热点数据永不过期、异步刷新。
- **实战案例**：秒杀活动中商品详情页 Key 过期，数万并发请求打垮 MySQL。改用逻辑过期（不设置 TTL，后台异步更新）成功扛住流量。
- **代码示例**：
```java
// 伪代码：互斥锁防止击穿
public String get(String key) {
    String value = redis.get(key);
    if (value == null) {
        if (redis.setnx(lock_key, 1, 10)) { // 获取分布式锁
            value = db.select(key); // 回源查询
            redis.set(key, value);
            redis.del(lock_key);
        } else {
            Thread.sleep(100); return get(key); // 等待重试
        }
    }
    return value;
}
```

**3. 缓存雪崩**
- **现象**：大量 Key 在同一时间集中过期，或缓存服务器宕机，DB 压力剧增。
- **解决**：过期时间加随机值、高可用架构（集群/哨兵）、限流降级。
- **实战案例**：凌晨全量缓存加载时未加随机偏移量，导致大量 Key 同时在整点失效，触发 DB 告警。在 TTL 基础上增加 `Random(1, 300)` 秒后解决。

### 对比表格：三种缓存异常

| 维度 | 缓存穿透 | 缓存击穿 | 缓存雪崩 |
| :--- | :--- | :--- | :--- |
| **根本原因** | 查询不存在的数据 | 热点 Key 过期 | 大量 Key 同时过期/宕机 |
| **影响范围** | 可能是随机 Key | 单个热点 Key | 大量的 Key |
| **并发压力** | 取决于恶意请求规模 | 极高（热点集中） | 极高（全量请求） |
| **解决方案** | 布隆过滤器、空值缓存 | 互斥锁、永不过期 | 随机 TTL、高可用集群 |

### ## 常见考点
1. **布隆过滤器原理是什么？它有缺点吗？**
   - **原理**：利用位数组和多个哈希函数，判断一个元素是否存在于集合中。存在一定的误判率（可能把不存在说成存在），但不会漏判（存在的一定说存在）。
   - **缺点**：不支持删除元素（删除可能影响其他元素）；存在误判。
2. **如何保证缓存与数据库的一致性？**
   - **策略**：更新数据库后，删除缓存（而非更新缓存）。延时双删策略（更新后立刻删，延时几秒再删一次）。
   - **原因**：并发场景下，更新缓存成本高且容易脏读，删除操作更轻量且符合懒加载思想。
3. **Redis 为什么是单线程还这么快？**
   - **原因**：纯内存操作、避免线程上下文切换和锁竞争、IO 多路复用机制（网络处理高效）。
