# 什么是Redis缓存？

Redis 缓存的使用场景与常见问题：

**典型场景：**
1. **缓存热点数据**：商品详情、用户信息，减轻 DB 压力。
2. **会话存储**：分布式 Session。
3. **排行榜/计数器**：ZSET、INCR。
4. **分布式锁**：SETNX/Redlock。
5. **消息队列**：List（简易）、Stream（5.0+）。

**三大经典问题：**
1. **缓存穿透**：查询不存在的 key，每次打到 DB。解决：布隆过滤器、缓存空值（设置较短 TTL）。
2. **缓存击穿**：热点 key 过期瞬间，大量请求打 DB。解决：互斥锁重建、热点 key 永不过期（逻辑过期）。
3. **缓存雪崩**：大量 key 同时过期或 Redis 宕机。解决：过期时间加随机扰动、多级缓存、集群高可用、限流降级。

**缓存一致性：** 先更新 DB 再删缓存（Cache Aside），或用消息队列/binlog 订阅（Canal）异步更新，最终一致。

**缓存更新策略流程：**
```text
┌─────────┐      ┌─────────┐      ┌─────────┐
│ Client  │      │  Cache  │      │   DB    │
└────┬────┘      └────┬────┘      └────┬────┘
     │ Read          │ Miss           │
     │──────────────▶│───────────────▶│
     │               │                │
     │               │                │
     │◀──────────────│◀───────────────│
     │  Return Data  │  Set Data      │
     │               │                │
     │               │                │
     │ Write(Update) │                │
     │───────────────▶│ Delete (Async)│
     │               │───────────────▶│
     │               │                │
```

| 问题 | 现象 | 原因 | 解决方案 |
| :--- | :--- | :--- | :--- |
| **缓存穿透** | DB 压力激增，Redis 无效命中 | 恶意请求不存在的 Key | 1. 布隆过滤器拦截 <br> 2. 缓存空对象 (TTL 短) |
| **缓存击穿** | 热点 Key 失效瞬间 DB 卡死 | 某个极度热点 Key 过期 | 1. 互斥锁防止并发重建 <br> 2. 逻辑过期 (后台异步更新) |
| **缓存雪崩** | 大量请求直接打到 DB，服务不可用 | 大量 Key 集中过期或 Redis 宕机 | 1. TTL 加随机值 <br> 2. 高可用架构 <br> 3. 限流降级 |

**实战案例：**
曾遇到“布隆过滤器误判”导致正常请求被拦截的问题。**排查**：因 Redis 重启导致布隆过滤器数据丢失。**改进**：将布隆过滤器本身持久化到 Redis，并设置 backup 恢复机制，同时保留空值缓存兜底。

**关键代码（互斥锁防击穿）：**
```java
public String getWithMutex(String key) {
    String value = redis.get(key);
    if (StringUtils.isEmpty(value)) {
        // 获取分布式锁
        if (tryLock(key + ":lock")) {
            try {
                // 再次查缓存，防止并发重复查库
                value = redis.get(key);
                if (StringUtils.isEmpty(value)) {
                    value = db.query(); // 查数据库
                    redis.set(key, value, 300);
                }
            } finally {
                unlock(key + ":lock");
            }
        } else {
            // 获取锁失败，休眠重试或返回旧值
            Thread.sleep(100);
            return getWithMutex(key);
        }
    }
    return value;
}
```

## 常见考点
1. **布隆过滤器的原理及缺点？**
   - 原理：利用多个 Hash 函数将 Key 映射到位数组中。优点：空间效率极高。缺点：存在误判，即可能判断 Key 存在但实际不存在；删除困难。
2. **先删缓存还是先更新数据库？为什么？**
   - 推荐先更新 DB，再删缓存。先删缓存可能导致并发请求在读 DB 时缓存未建立，造成数据不一致；先更新 DB 能保证 DB 是最新的源头。
3. **如何保证 Redis 和 DB 的最终一致性？**
   - 延时双删（更新 DB 后删一次，延迟几秒再删一次）或者订阅 MySQL Binlog 通过 Canal 解析后异步更新/删除 Redis。
