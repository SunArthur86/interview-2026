# 分布式事务有哪些方案？2PC/TCC/Saga/本地消息表/最大努力通知各自的原理和适用场景？

【核心矛盾】
跨数据库/服务的数据修改无法用单机 ACID 事务保证原子性，需引入协调机制保证最终一致性或强一致性。

【五大方案深度解析】

1. **2PC（两阶段提交）**
   - **原理**：
     1. 准备阶段：协调者询问所有参与者“能否提交？”，参与者执行本地事务但不提交，写 Redo/Undo 日志并回复 Yes/No。
     2. 提交阶段：若全 Yes，协调者发 Commit，否则发 Rollback。
   - **关键细节**：
     - **同步阻塞**：所有参与者在准备阶段必须锁定资源，直到协调者下发指令，导致并发度极低。
     - **单点故障**：协调者宕机，参与者一直阻塞。
     - **数据不一致**：协调者发 Commit 时崩溃，部分参与者收到 Commit，部分未收到。
   - **适用场景**：传统单体数据库分库分库（XA 协议），对性能要求不高的后台系统。

2. **TCC（Try-Confirm-Cancel）**
   - **原理**：
     1. **Try**：资源预留，如冻结余额（而非直接扣减）。
     2. **Confirm**：确认执行，使用 Try 预留的资源，实际扣减余额。
     3. **Cancel**：取消执行，释放 Try 预留的资源（解冻）。
   - **关键细节**：
     - **空回滚**：Try 未执行，Cancel 先到达（网络超时导致），需支持 Cancel 能直接返回成功。
     - **悬挂**：Cancel 先执行了，Try 后到达，需拒绝 Try 执行。
     - **幂等性**：Confirm/Cancel 可能重复调用，接口必须幂等。
   - **适用场景**：对性能要求高、一致性要求高的核心业务（如支付、金融账户）。

3. **Saga（长事务）**
   - **原理**：将长事务拆分为多个本地短事务 T1, T2, T3。每个 T 都有对应的补偿动作 C1, C2, C3。
     - 正常流程：T1 -> T2 -> T3。
     - 异常流程（如 T2 失败）：执行 C1（回滚 T1）。
   - **关键细节**：
     - **无隔离性**：可能发生脏写（其他事务读到中间状态），需业务层处理（如加悲观锁、乐观锁）。
     - **补偿复杂**：C1 必须能在 T1 成功后的任何状态下安全回滚。
   - **适用场景**：长流程业务、非强一致性要求（如订票流程：订票->支付->出票）。

4. **本地消息表（基于 MQ 的最终一致性）**
   - **原理**：
     - 发送方将业务操作和消息写入本地同一个数据库（同一个本地事务保证原子性）。
     - 定时任务扫描消息表，将消息发送到 MQ。
     - 消费方消费消息，执行业务，并记录消费日志（ACK）。
   - **架构图**：
     ```text
     ┌──────────────┐      ┌──────────────┐      ┌──────────────┐
     │   服务 A     │      │     MQ       │      │   服务 B     │
     │ (上游业务方)  │      │  (消息队列)   │      │ (下游消费方)  │
     └──────┬───────┘      └──────┬───────┘      └──────┬───────┘
            │                     │                     │
    1.写业务+  │                     │                     │
      消息表   │                     │                     │
            │                     │                     │
    2.定时任务  │                     │                     │
      扫描发送  ├──────────────────>│                     │
            │      3.投递消息       │                     │
            │                     │                     │
            │                     ├──────────────────>  │
            │                     │    4.消费消息         │
            │                     │                     │
            │                     │              5.执行业务 + 记录ACK
            │                     │<──────────────────  │
            │       6.重试(失败时)  │                     │
            ├───────────────────  │                     │
     ```
   - **关键细节**：
     - **消息丢失**：上游写库成功但挂了，没发 MQ -> 定时任务补发。
     - **重复消费**：MQ 可能重复投递，下游必须幂等。
   - **适用场景**：高并发、允许最终一致性的场景（如电商下单后发积分）。

5. **最大努力通知**
   - **原理**：
     - 通知方调用接收方接口，失败则重试（有限次数）。
     - 接收方提供查询接口，供通知方或业务方主动核对状态。
   - **关键细节**：
     - **不可靠**：不保证一定成功，依赖定期对账。
   - **适用场景**：支付回调通知、银行对账接口。

【Seata 框架模式】
- **AT 模式**：改进型 2PC。无需手动编写 TCC 三个接口。解析 SQL 生成“前镜像”和“后镜像”，构建 Undo Log。一阶段提交本地事务并释放锁，二阶段异步回滚（读取 Undo Log 反向生成 SQL）。
- **TCC 模式**：自定义三个接口，性能可控。
- **Saga 模式**：状态机管理。
- **XA 模式**：标准 2PC，占用锁时间长。

## 常见考点
1. **TCC 的空回滚和悬挂是如何产生的？如何解决？**（考察对网络异常边界条件的理解）
2. **本地消息表为什么要用定时任务扫表而不是业务代码直接发 MQ？**（考察事务边界问题，发 MQ 可能失败导致本地事务回滚但消息已发）
3. **Saga 模式如何解决隔离性问题？**（考察业务层加锁策略）

## 记忆要点

- 2PC vs TCC：2PC 强一致重度锁定资源性能差；TCC 分为 Try-Confirm-Cancel 适合支付
- Saga 特点：长事务拆分短事务，失败时需业务层执行补偿动作，存在无隔离性脏写问题
- 最终一致：本地消息表结合 MQ 异步重试，要求下游业务接口必须实现幂等性
- 口诀记忆：2PC 锁阻塞，TCC 资源预留，Saga 靠补偿，MQ 最终一致
