# MySQL 的四种事务隔离级别分别解决了什么问题？为什么 RR 是默认级别？

【四种隔离级别及解决的问题】

1. **READ UNCOMMITTED（读未提交）**
   - **现象**：脏读（读了未提交的数据），不可重复读，幻读。
   - **原理**：不加行锁读。
   - **场景**：几乎不用。

2. **READ COMMITTED（读已提交，RC）**
   - **现象**：解决脏读，存在不可重复读，存在幻读。
   - **原理**：MVCC。每次 SELECT 语句执行时，都会重新生成一个 ReadView。
   - **ReadView 机制**：包含生成时刻所有活跃事务 ID 列表（min_trx_id, max_trx_id, m_ids）。判断可见性时，判断行记录的 trx_id 是否在 m_ids 中。
   - **场景**：Oracle、PostgreSQL 默认；互联网高并发业务。

3. **REPEATABLE READ（可重复读，RR）**
   - **现象**：解决脏读，解决不可重复读，**InnoDB 通过 MVCC + Next-Key Lock 解决了大部分幻读**（注意是“解决幻读问题”而非完全隔离级别定义中的防止）。
   - **原理**：
     - **MVCC**：事务中**第一次** SELECT 生成 ReadView，后续所有 SELECT 复用该 ReadView，看到的是事务开始时的快照。
     - **锁机制**：当前读（SELECT ... FOR UPDATE / UPDATE / DELETE）使用 Next-Key Lock（记录锁 + 间隙锁），防止其他事务插入新记录，从而解决幻读。
   - **场景**：MySQL InnoDB 默认级别。

4. **SERIALIZABLE（串行化）**
   - **现象**：所有问题解决。
   - **原理**：读隐式加共享锁，写加排他锁。
   - **场景**：极低，性能极差。

【为什么 MySQL 默认 RR？】
1. **历史包袱（Binlog 兼容性）**：
   - 早期 MySQL Binlog 只有 Statement 格式（记录 SQL 语句）。在 RC 隔离级别下，由于可能存在更新丢失或主从顺序不一致问题，Statement 模式的 Binlog 无法保证主从复制安全。RR 配合 Statement 是安全的。
   - 虽然 Row 格式（记录行数据变化）解决了 RC 下的复制问题，但 RR 作为默认级别延续了下来。
2. **数据一致性优势**：RR 提供了比 RC 更强的稳定性，避免不可重复读带来的业务逻辑困扰。
3. **InnoDB 的优化**：通过 MVCC，RR 在读场景下不加锁，性能并未比 RC 差太多。

【为什么互联网公司常改用 RC？】
1. **锁开销**：RR 下的当前读会使用 Gap Lock（间隙锁），而 RC 只用 Record Lock（记录锁）。高并发下 Gap Lock 容易导致死锁和锁等待。
2. **死锁概率**：RR 的 Gap Lock 增加了死锁检测的负担。
3. **一致性保障**：配合 Row 格式 Binlog，RC 已能满足主从一致的需求。

## 常见考点
1. **RR 真的完全解决了幻读吗？**（陷阱：MVCC 解决快照读的幻读，Next-Key Lock 解决当前读的幻读）
2. **RC 和 RR 在生成 ReadView 时机上的区别是什么？**（核心：RC 每次 Select 生成，RR 第一次生成后复用）
3. **什么是当前读和快照读？**（Select * 是快照读，Select ... for update / Insert / Update / Delete 是当前读）

## 记忆要点

- 并发问题：读未提交有脏读，RC 解决脏读但不可重复读，RR 解决不可重复读及幻读
- RR 默认原因：早期主从复制用 Statement 格式需 RR 保安全，且数据一致性更稳定
- 改 RC 原因：互联网高并发场景下，RC 无 Gap Lock 仅 Record Lock，大幅减少死锁概率
- 快照对比：MVCC 下 RC 每次重新生成 ReadView，而 RR 一直复用第一次生成的
