# redo log、undo log、binlog 各自的作用是什么？两阶段提交如何保证三者一致？

【三种日志核心作用】

1. **redo log（重做日志）**
   - **位置**：InnoDB 引擎层，存储在 `ib_logfile0` 等文件中。
   - **作用**：**崩溃恢复**。即使事务提交了但数据没刷到磁盘，重启后可重做。
   - **内容**：物理日志，记录“在某个数据页的某个偏移量写入了什么数值”。
   - **WAL 机制**：Write-Ahead Logging。先写日志，再写磁盘。顺序写，性能高。
   - **刷盘策略**：由 `innodb_flush_log_at_trx_commit` 控制（0:交由OS；1:每次刷盘；2:刷到OS缓冲）。

2. **undo log（回滚日志）**
   - **位置**：InnoDB 引擎层，在 undo tablespace 或系统表空间中。
   - **作用**：
     1. **回滚**：事务失败时，利用 undo log 将数据恢复到修改前。
     2. **MVCC**：通过 undo log 的版本链，实现读写不冲突。
   - **内容**：逻辑日志，记录“如何逆向回滚”（如 `INSERT` 对应 `DELETE`，`UPDATE` 记录旧值）。

3. **binlog（归档日志）**
   - **位置**：Server 层，存储在本地磁盘（如 `mysql-bin.000001`）。
   - **作用**：
     1. **主从复制**：Slave 拉取 binlog 重放。
     2. **数据恢复**：全量备份 + 增量 binlog 恢复到任意时间点。
   - **内容**：逻辑日志（Row/Statement/Mixed 格式）。

【两阶段提交（2PC）保证崩溃一致】
**问题背景**：如果不使用两阶段提交，可能出现 redo 写了但 binlog 没写，导致主库有数据从库没数据（主从不一致）；或者 binlog 写了但 redo 没写，主库重启丢数据。

**流程图**：
```text
   InnoDB (引擎层)               MySQL Server (服务层)
        │                              │
        │  1. 执行事务，写数据页        │
        │  2. 写 redo log (prepare)    │
        ├─────────────────────────────>│
        │                              │  3. 写 binlog
        │                              ├───────────────────> (磁盘)
        │                              │
        │  4. 写 redo log (commit)     │
        │<─────────────────────────────┤
        │ (此时事务真正提交)            │
        ▼                              ▼
```

**详细步骤**：
1. **Prepare 阶段**：InnoDB 写 redo log，并将状态标记为 `prepare`。
2. **Write Binlog 阶段**：MySQL Server 写 binlog 到磁盘。
3. **Commit 阶段**：InnoDB 将 redo log 状态标记为 `commit`，事务完成。

【崩溃恢复规则】
重启时，InnoDB 扫描 redo log：
- **情况 A**：redo log 包含完整的 `commit` 标记 → 说明事务已成功，直接提交。
- **情况 B**：redo log 是 `prepare` 状态，但 binlog 存在且完整 → 说明 binlog 写成功了，补全 redo log 的 `commit` 标记，提交事务。
- **情况 C**：redo log 是 `prepare` 状态，但 binlog 不存在/不完整 → 说明事务未成功，回滚事务（利用 undo log）。

## 常见考点
1. **为什么不直接写 binlog 依赖它恢复？**（InnoDB 引擎是插件式的，不能依赖 Server 层的 binlog 做崩溃恢复，必须有自己的 redo log）
2. **双1配置指什么？对性能有何影响？**（`innodb_flush_log_at_trx_commit=1` 和 `sync_binlog=1`，最安全但性能损耗大）
3. **redo log 是循环写，binlog 是追加写，区别是什么？**（考察存储结构和容量规划）


## 记忆要点

- 日志归属：redo 和 undo 属于 InnoDB 引擎层；binlog 属于 Server 层
- 核心作用：redo 保证崩溃恢复（物理日志），undo 保证事务回滚和 MVCC，binlog 用于主从复制和恢复
- 两阶段提：先写 redo prepare，再写 binlog，最后写 redo commit，保证日志逻辑一致
- 崩溃恢复：查 redo 若有 commit 直接提交；若仅 prepare 则校验 binlog 完整性决定提交或回滚
