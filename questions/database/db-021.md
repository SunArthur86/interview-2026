# 什么是分库分表总结？

### 分库分表定义
针对数据量巨大的单张表（比如订单表），按照某种规则（RANGE, HASH取模等），切分到多张表里面去。

**核心区别**：
- **分表**：表分散在同一个库中，主要解决**单表数据量过大**导致的 B+ 树层级过高、查询变慢问题。但瓶颈在于单库的连接数、IO 和 CPU。
- **分库**：表分散在不同的数据库服务器上，主要解决**单机资源瓶颈**（CPU、内存、磁盘 IO、网络带宽）。

### 1. 水平分表
- **原理**：同一个库内，把一个大表的数据按规则拆分到多个结构相同的小表中。
- **优点**：减少单表数据量，索引树变低，查询效率提升；减少 DDL 锁表时间（如添加字段）。
- **缺点**：并未解决单库 IO 瓶颈和连接数限制。

### 2. 水平分库
- **原理**：把同一个表的数据按规则分到不同服务器上的不同数据库中。
- **特点**：每个库的表结构一样，数据无交集，并集是全量数据。
- **优点**：将请求分摊到多台机器，降低单机负载。
- **缺点**：分布式事务、跨库 Join 复杂度增加。

### 切分规则
1. **Range（范围）**：如按 ID 范围（0-1000万, 1000-2000万）。优点是扩容简单；缺点是数据分布不均，可能产生热点。
2. **Hash 取模**：如 `id % db_count`。优点是数据分布均匀；缺点是扩容时需要数据迁移（取模公式变化）。
3. **地理区域/时间**：按省市、日期切分，适合报表类或特定地域业务。

### 3. 垂直分表
- **原理**：将一个大表的字段拆分成多个表。
- **策略**：
  - 将“常用字段”（高频查询）和“不常用字段”拆分。
  - 将“大字段”（text, blob）拆分到附属表，减少 IO 争用和 Buffer Pool 污染。

### 4. 垂直分库
- **原理**：根据业务微服务领域模型，将不同的表分配到不同的数据库。
- **策略**：订单库、用户库、商品库分离。
- **优点**：业务解耦，便于微服务拆分；避免单一库竞争所有硬件资源。

### 技术规划建议
1. **不要过早优化**：单表数据量在 500万-1000万 以内，或 QPS < 1000 时，优先考虑 SQL 优化、索引优化、读写分离、引入缓存。
2. **前瞻性评估**：分库分表是“伤筋动骨”的操作，需提前 1-2 年规划容量、QPS 增长趋势。
3. **数据迁移方案**：做好“双写迁移”或“停机迁移”的预案。
