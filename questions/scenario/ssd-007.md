# 如何设计一个支持千万级并发的IM系统（即时通讯）？

🎯 本质：IM系统核心挑战是海量长连接管理、消息实时投递、消息有序性和一致性。

📊 架构设计：

1. **长连接管理**
- WebSocket/TCP长连接
- Netty作为网络框架（NIO，单机10万+连接）
- 连接层水平扩展：多台Connection Server
- 用户 → 路由：记录用户在哪台Connection Server上

**IM 系统整体架构：**
```text
[Client A] <--> [Conn Server 1] --(Msg)-->
                                       |
                                   [MQ (Kafka)]
                                       |
[Client B] <--> [Conn Server 2] <--(Sync)--
      ^                                   ^
      | (Route Lookup)                   | (Route Lookup)
      +----------- [Route Service] ------+
                      | (User -> Server IP Mapping)
                      v
                [Redis Cluster]
```

2. **消息流转流程**
发送者 → Connection Server → Message Queue → 
Message Processor → 路由查找接收者 → 
接收者的Connection Server → 推送给接收者

3. **消息存储**
在线消息：Redis缓存
历史消息：MySQL/分库分表
消息索引：用户维度的消息列表（按时间排序）

4. **消息投递保证**
- **ACK机制**：服务端发送消息 → 客户端ACK → 服务端标记已送达
- **消息ID（Message ID）设计**：通常采用 `SenderID + Timestamp + Seq` 生成唯一ID，用于去重和ACK确认。
- **离线消息**：用户上线后拉取未读消息
- **消息排序**：单聊按时间戳排序，群聊按服务器接收顺序

5. **群消息优化**
小群（<200人）：写扩散（推到每个成员收件箱）
大群（>200人）：读扩散（成员主动拉取）
超大群（>5000人）：仅推在线成员 + 拉取历史

6. **关键指标**
- 单机连接数：Netty可支撑10-50万TCP连接（需调整内核参数 `ulimit`, `somaxconn`, `tcp_tw_reuse`）
- 消息延迟：<200ms
- 消息吞吐：单机5万条/秒
- 可用性：99.99%

7. **推送优化**
- 安卓推送通道：华为/小米/OPPO/VIVO厂商通道
- iOS：APNs
- 省电：心跳间隔自适应（WiFi短，移动网络长）

8. **安全**
- TLS加密传输
- 消息内容加密（端到端可选）
- 防刷：消息频率限制

## 常见考点
1. **消息可靠性**：如何保证消息“必达”？（发送ACK + 服务端存储未确认消息 + 客户端拉取重试机制）。
2. **海量连接调优**：单机维持百万级长连接需要做哪些OS内核调优？（最大文件句柄数、TCP快速回收、端口范围调优等）。
3. **消息去重**：在弱网环境下，客户端可能会收到重复消息，如何去重？（基于Message ID的去重表）。

## 记忆要点

- 海量长连接：Netty支撑单机十万并发，路由层记录用户与IP映射
- 消息必达机制：服务端下发+客户端ACK+未确认重试拉取防丢
- 弱网去重：基于SenderID+时间戳+Seq生成全局唯一ID去重
- 群聊读写扩散：小群写扩散推收件箱，大群读扩散主动拉取
