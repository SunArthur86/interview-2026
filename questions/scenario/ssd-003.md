# 如何设计一个分布式ID生成方案？

🎯 本质：分布式ID需要全局唯一、趋势递增、高性能、高可用。

📊 主流方案对比：

1. **UUID**
优点：本地生成、无网络开销
缺点：无序（影响B+树索引性能）、太长(36字符)
适用：不需要排序的场景（如TraceID）

2. **数据库自增ID**
优点：简单
缺点：性能瓶颈、单点故障
优化：多主模式（奇偶ID分开），但扩展性差

3. **号段模式（美团Leaf-segment）**
原理：DB分配一批ID给应用缓存
  表结构：biz_tag, max_id, step, version
  应用每次获取step个ID（如1000），用完再申请
优点：DB压力小（每次申请一批）、ID递增
缺点：重启丢失部分ID、可能不连续

4. **Snowflake（雪花算法）**
结构：1bit符号 + 41bit时间戳 + 10bit机器ID + 12bit序列号
  时间戳：毫秒级，可用69年
  机器ID：1024台机器
  序列号：每毫秒4096个ID
优点：本地生成、有序、高性能
缺点：时钟回拨问题

5. **Redis自增**
INCR + INCRBY
优点：简单、高性能
缺点：依赖Redis可用性

6. **Leaf-Snowflake（美团）**
Snowflake改进：用Zookeeper自动分配workerId
解决了手动配置机器ID的问题

时钟回拨解决方案：
1. 回拨时间短（<5ms）：等待回拨时间过去
2. 回拨时间长：用备用workerId或报警
3. 百度UidGenerator：用RingBuffer缓存未来时间段ID

**Snowflake 结构详解：**
```text
0 | 000110010101101000101101010101101010001011101001 (41bit 时间戳)
  |                0001001101 (10bit 机器ID/数据中心ID)
  |                             000101010101 (12bit 毫秒内序列号)
```
* 41bit时间戳：可以使用69年（(1L<<41)/(1000L360024365) ≈ 69年）
* 10bit工作机器ID：5bit数据中心ID + 5bit工作节点ID（支持32*32=1024个节点）
* 12bit序列号：每毫秒可生成4096个ID

**号段模式（双Buffer优化）：**
为了避免在号段用完去DB申请时造成的阻塞，通常采用双Buffer机制。
当前号段用到一定程度（如剩余一半）时，异步去DB申请下一个号段并加载到内存。
```text
Thread --> [ Segment Buffer 1 ] (Current: 1-1000)
             [ Segment Buffer 2 ] (Next:  1001-2000 - Loading Async)
             | (Switch when 1 is 50% used)
```

选择建议：
- 中小规模 → 号段模式（Leaf-segment）
- 大规模+低延迟 → Snowflake（Leaf-snowflake）
- 不需要排序 → UUID

## 常见考点
1. **时钟回拨**：为什么Snowflake对时钟敏感？如果服务器时钟回拨会导致ID重复，通常有哪些解决方案（等待、报错、利用备用位）？
2. **WorkerId分配**：在Docker/K8s环境下，IP可能变动，如何持久化或自动分配WorkerId？（利用Zookeeper持久顺序节点或数据库）。
3. **性能瓶颈**：雪花算法的性能瓶颈通常在哪里？（序列号回环等待下一毫秒的时间消耗，可通过CAS优化）。

## 记忆要点

- 两大主流：中小规模用号段模式重吞吐，大规模低延迟用雪花算法重性能
- 雪花结构：41位时间戳+10位机器ID+12位序列号，本地生成且全局唯一递增
- 时钟回拨：服务器时钟倒退会导致ID重复，短时等待长时切备用workerId
- 号段双Buffer：当前号段消耗过半时即异步加载下一个，避免阻塞应用
