# 如何设计一个短链（Short URL）生成服务？请阐述算法选择以及如何解决哈希冲突和发号分布式唯一性问题。

短链服务核心是将长URL映射为短字符串。主要方案有两种：

1. **哈希算法**：对长URL进行MD5或SHA256计算，取前几位，再Base62编码（a-z, A-Z, 0-9）。**冲突解决**：如果发现生成的短码已存在，在原始字符串后加盐或加扰动因子重新计算，直到不冲突。此方案优点是计算快，不依赖存储，缺点是不可逆（需存DB映射），且存在极小概率冲突需重试。
2. **发号器算法**：利用分布式ID生成器（如Snowflake算法或数据库自增步长）生成唯一数字ID，将其转换为62进制字符串。此方案绝对唯一，无冲突，且生成连续的短链。

**推荐设计**：使用发号器方案以保证唯一性。分布式ID可使用Redis INCR或数据库Flickr算法。存储上，利用Redis缓存热点短链映射，DB持久化。若担心ID连续暴露业务量，可进行乱序处理或在62进制转换时加入混淆逻辑。性能方面，生成端需支持高并发，访问端依赖CDN加速。

**实战案例**：
某营销活动上线后，短时间内生成了千万级短链，由于使用简单的DB自增ID发号，数据库写入成为瓶颈。后优化为Redis Batch INCR（批量获取1000个ID号段到本地内存），应用层直接在内存中转换为62进制短码，数据库写入QPS降低了99%。

**代码示例（ID转62进制算法）**：
```java
public static String idToShortKey(long id) {
    String chars = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";
    StringBuilder shortKey = new StringBuilder();
    while (id > 0) {
        int remainder = (int) (id % 62);
        shortKey.append(chars.charAt(remainder));
        id /= 62;
    }
    // 可能需要补位或反转，视具体约定而定
    return shortKey.reverse().toString();
}
```

**对比表格**：

| 特性 | 哈希算法 (MD5/SHA) | 发号器算法 (ID自增) |
| :--- | :--- | :--- |
| **唯一性保证** | 概率性（存在极低冲突概率） | 绝对唯一 |
| **长度** | 固定较短（可截断） | 随ID增长而变长（但Base62很高效） |
| **性能** | 计算快，但冲突检测可能需查DB | 极快，内存计算即可（预分配号段） |
| **存储依赖** | 必须存储映射关系以解决冲突和跳转 | 必须存储映射关系（因为短链不包含长链信息） |
| **有序性** | 无序，利于负载均衡 | 有序，可能泄露业务量（需混淆） |
| **分布式复杂度** | 低（无状态） | 中（需维护全局唯一ID源） |
