# 如何设计一个短链（Short URL）生成服务？请阐述算法选择以及如何解决哈希冲突和发号分布式唯一性问题。

短链服务核心是将长URL映射为短字符串。主要方案有两种：

1. **哈希算法**：对长URL进行MD5或SHA256计算，取前几位，再Base62编码（a-z, A-Z, 0-9）。**冲突解决**：如果发现生成的短码已存在，在原始字符串后加盐或加扰动因子重新计算，直到不冲突。此方案优点是计算快，不依赖存储，缺点是不可逆（需存DB映射），且存在极小概率冲突需重试。
2. **发号器算法**：利用分布式ID生成器（如Snowflake算法或数据库自增步长）生成唯一数字ID，将其转换为62进制字符串。此方案绝对唯一，无冲突，且生成连续的短链。

**推荐设计**：使用发号器方案以保证唯一性。分布式ID可使用Redis INCR或数据库Flickr算法。存储上，利用Redis缓存热点短链映射，DB持久化。若担心ID连续暴露业务量，可进行乱序处理或在62进制转换时加入混淆逻辑。性能方面，生成端需支持高并发，访问端依赖CDN加速。

**实战案例**：
某营销活动上线后，短时间内生成了千万级短链，由于使用简单的DB自增ID发号，数据库写入成为瓶颈。后优化为Redis Batch INCR（批量获取1000个ID号段到本地内存），应用层直接在内存中转换为62进制短码，数据库写入QPS降低了99%。

**代码示例（ID转62进制算法）**：
```java
public static String idToShortKey(long id) {
    String chars = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";
    StringBuilder shortKey = new StringBuilder();
    while (id > 0) {
        int remainder = (int) (id % 62);
        shortKey.append(chars.charAt(remainder));
        id /= 62;
    }
    // 可能需要补位或反转，视具体约定而定
    return shortKey.reverse().toString();
}
```

**对比表格**：

| 特性 | 哈希算法 (MD5/SHA) | 发号器算法 (ID自增) |
| :--- | :--- | :--- |
| **唯一性保证** | 概率性（存在极低冲突概率） | 绝对唯一 |
| **长度** | 固定较短（可截断） | 随ID增长而变长（但Base62很高效） |
| **性能** | 计算快，但冲突检测可能需查DB | 极快，内存计算即可（预分配号段） |
| **存储依赖** | 必须存储映射关系以解决冲突和跳转 | 必须存储映射关系（因为短链不包含长链信息） |
| **有序性** | 无序，利于负载均衡 | 有序，可能泄露业务量（需混淆） |
| **分布式复杂度** | 低（无状态） | 中（需维护全局唯一ID源） |

## 技术原理

**哈希算法：快但有小概率冲突**
对长 URL 计算 MD5/SHA256，取前几位再 Base62 编码（0-9, a-z, A-Z）。优点是计算快、无状态、不依赖中心化存储；缺点是存在极小概率冲突（不同长链算出相同短码），需要查 DB 校验并加盐重试。此外哈希结果是固定长度但无序，无法从短码反推生成顺序。

**发号器算法：绝对唯一、无冲突**
利用分布式 ID 生成器（Snowflake、DB 号段、Redis INCR）产生全局唯一数字 ID，再转 62 进制字符串。优点是绝对唯一无冲突，短码连续紧凑；缺点是需要维护全局 ID 源，有序性可能泄露业务量（需做混淆或乱序）。

**发号优化：Redis 批量 INCR 解决 DB 自增瓶颈**
单机 DB 自增 ID 在高并发下写入成为瓶颈。优化方案是 Redis 批量 INCR：应用启动时一次性申请一段 ID（如 1000 个）到本地内存，本地原子自增消费，用完再申请。这样 DB/Redis 的 QPS 降低了 1000 倍，同时单点故障时本地仍有号段可用，具备缓冲能力。

## 代码示例

```java
// 1. ID 转 62 进制短码（核心算法）
public static String idToShortKey(long id) {
    String chars = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";
    StringBuilder sb = new StringBuilder();
    while (id > 0) {
        sb.append(chars.charAt((int)(id % 62)));
        id /= 62;
    }
    return sb.reverse().toString();
}
```

```java
// 2. Redis 批量获取号段（解决高并发写入瓶颈）
public class ShortIdGenerator {
    private final AtomicLong localId = new AtomicLong();
    private final AtomicLong localMax = new AtomicLong();
    private static final int STEP = 1000;

    public long nextId() {
        long cur = localId.incrementAndGet();
        if (cur > localMax.get()) {
            synchronized (this) {               // 双重检查
                if (localId.get() > localMax.get()) {
                    long newMax = redis.incrBy("short_id_seq", STEP);
                    localId.set(newMax - STEP + 1);
                    localMax.set(newMax);
                    return localId.getAndIncrement();
                }
            }
        }
        return cur;
    }
}
```

## 注意事项

- 核心方案对比：哈希算法快但有小概率冲突，发号器绝对唯一且无冲突。
- 发号器实现：Snowflake 或 DB 号段，转 62 进制极高效且生成连续短链。
- 发号优化：采用 Redis 批量 INCR 获取号段至本地内存，解决 DB 自增高并发写入瓶颈。
- 发号器有序会泄露业务量，建议在 62 进制转换时加入混淆（如位反转、异或密钥）。
- 短链跳转需缓存热点映射到 Redis，结合 CDN 加速访问，DB 只做兜底持久化。
