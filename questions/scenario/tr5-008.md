# 请设计一个支持全球排名的实时排行榜系统，要求数据更新频率高且能支持分页查询。

实时排行榜设计的核心在于高效维护有序集合。Redis的ZSET（SkipList + Hash）是天然选择，其插入、删除复杂度为O(log N)，且支持根据分数范围查询。针对全球海量数据，单机Redis内存可能不足，需进行分片。

**分片策略**：可按分数范围分片（如0-100分在Shard1，100-200在Shard2），但这涉及数据迁移。更实用的方式是按Key哈希分片（如CRDT），但这使得全局排序变难。

**优化方案**：
1. **本地聚合+全局归并**：请求时并行查询各分片Top N，在应用层归并排序。
2. **懒淘汰机制**：对于分数变化不大的数据，利用定时任务定期固化快照，减少实时计算压力。
3. **读写分离**：利用Redis主从架构，主节点处理高频更新，从节点处理分页查询。
4. **针对分页优化**：对于页数极深的查询（如100万页），建议禁止或采用游标方式（基于Last Score和Last ID），避免SkipList深层遍历。

**实战案例**：
某游戏在跨服战期间，排行榜更新QPS达到50万，使用单一Redis ZSET导致主节点CPU单核跑满，数据更新延迟严重。最终采用“分桶策略”，将玩家按ID哈希分配到100个Redis ZSET中，查询时在内存中归并Top 100，解决了单点瓶颈。

**代码示例（基于游标的分页查询）**：
```python
# 避免使用 ZRANK skip，使用 ZRANGEBYSCORE 实现高性能翻页
# last_score 和 last_member 为上一页最后一条数据的分数和成员

def get_ranking_page(last_score, last_member, page_size):
    # 查询分数小于 last_score，或者分数相同但ID字典序小于 last_member 的数据
    return redis.zrevrangebyscore(
        "global_rank",
        f"({last_score}", # '(' 表示不包含该分数
        "-inf",
        start=0,
        num=page_size,
        withscores=True
    )
```

## 技术原理

排行榜系统的核心是「有序集合 + 分片 + 游标分页」三层设计：

- **ZSET 的数据结构优势**：Redis Sorted Set 底层是「跳表（SkipList）+ 哈希表」的复合结构。哈希表存储 member→score 映射（O(1) 查分数），跳表按 score 有序存储（O(log N) 插入/删除/范围查询）。相比纯平衡树，跳表实现简单且范围查询高效（跳表天然支持区间遍历）。这正是排行榜场景的理想结构——高频更新分数 + 频繁范围查询。
- **分片策略的权衡**：(1) **按分数范围分片**（如 0-1000 分在 Shard1）——全局排序简单（只查最高分片），但热点集中在顶部分片（高手都在一个分片），且分数变化导致数据跨分片迁移；(2) **按 member ID 哈希分片**——数据均匀分布无热点，但全局 Top N 要查所有分片并归并。生产实践多选哈希分片 + 应用层归并，因为归并成本（N 个分片各取 Top N）远低于分数分片的迁移成本。
- **本地聚合 + 全局归并**：查询全局 Top 100 时，并行向 100 个分片各发 `ZREVRANGEBYSCORE 0 100`（取本地 Top 100），应用层用最小堆归并出全局 Top 100。并行查询把延迟控制在单分片级别（如 5ms），归并是 O(N·logK) 内存操作（K=100，可忽略）。
- **深分页的游标优化**：`ZREVRANGE rank_start rank_stop` 用 rank 偏移，深翻页（如第 1000 页）要遍历跳表跳过 20000 个节点，O(log N + offset) 退化严重。游标法用上一页最后一条的 `(score, member)` 做条件：`ZREVRANGEBYSCORE (last_score -inf LIMIT 0 page_size`（分数相同按 member 字典序），每次查询复杂度恒定 O(log N + page_size)，与页深无关。

## 代码示例

```python
# 分桶存储 + 归并查询（生产级实现）
import hashlib

SHARD_COUNT = 100   # 100 个分片

def get_shard_key(member_id: str) -> str:
    """按 member ID 哈希分桶"""
    shard = int(hashlib.md5(member_id.encode()).hexdigest(), 16) % SHARD_COUNT
    return f"rank:shard:{shard}"

def update_score(member_id: str, score: float):
    """更新分数（写入对应分片）"""
    shard_key = get_shard_key(member_id)
    redis.zadd(shard_key, {member_id: score})

def get_global_top_n(n: int = 100) -> list[tuple]:
    """并行查所有分片的 Top N，归并出全局 Top N"""
    import concurrent.futures
    def query_shard(shard: int):
        return redis.zrevrange(f"rank:shard:{shard}", 0, n-1, withscores=True)
    # 并行查询 100 个分片
    with concurrent.futures.ThreadPoolExecutor(max_workers=50) as pool:
        futures = [pool.submit(query_shard, i) for i in range(SHARD_COUNT)]
        all_results = [f.result() for f in futures]
    # 归并：所有分片结果合并后取 Top N
    merged = sorted(
        [item for shard_result in all_results for item in shard_result],
        key=lambda x: x[1], reverse=True
    )[:n]
    return merged
```

```python
# 游标分页（避免深分页性能退化）
def get_ranking_page_cursor(last_score: float, last_member: str, page_size: int = 20):
    """
    基于 (score, member) 游标的分页，复杂度与页深无关
    第一次查询：last_score = +inf, last_member = ''
    """
    if last_score == float('inf'):
        # 第一页：直接取最高分
        return redis.zrevrangebyscore("global_rank", "+inf", "-inf",
                                      start=0, num=page_size, withscores=True)
    # 后续页：取分数 < last_score 的（同分按 member 字典序）
    # Redis 6.2+ 支持 ZRANGEBYSCORE 的 BYSCORE + BYLEX 组合
    members = redis.zrevrangebyscore(
        "global_rank",
        f"({last_score}",          # '(' 排除 last_score 本身
        "-inf",
        start=0, num=page_size, withscores=True
    )
    return members

# 查询用户的排名（分桶场景需聚合）
def get_user_rank(member_id: str) -> int:
    """全局排名 = 该用户在自己分片的排名 + 比它分数高的其他分片人数"""
    shard_key = get_shard_key(member_id)
    local_rank = redis.zrevrank(shard_key, member_id)   # 分片内排名
    score = redis.zscore(shard_key, member_id)
    # 统计其他所有分片中分数 > score 的人数
    higher_count = 0
    for i in range(SHARD_COUNT):
        if f"rank:shard:{i}" == shard_key: continue
        higher_count += redis.zcount(f"rank:shard:{i}", f"({score}", "+inf")
    return local_rank + higher_count + 1
```

## 注意事项

- **分桶数的选择**：分桶太少（如 10 个）单桶仍过大；太多（如 10000 个）查询归并开销大。经验值是「单桶数据量控制在 10 万~100 万」，按总用户数反推分桶数。
- **同分排序的稳定性**：分数相同时 ZSET 按 member 字典序排序，可能与业务期望（如按达到时间先后）不符。解法：score 编码为 `分数.时间戳`（如 `score * 1e10 + (MAX_TS - ts)`），让时间因素融入 score。
- **冷数据下沉**：长期不活跃用户的分数占内存，可把低分用户迁移到 DB，ZSET 只保留活跃或高分用户。查询时若 ZSET 未命中再查 DB。
- **实时更新的热点**：跨服战等场景 QPS 极高，单 ZSET 主节点 CPU 跑满。分桶 + 读写分离（主写从读）+ Pipeline 批量更新可缓解。极端情况用专门的排行榜服务（如腾讯云 TcapusDB）。
- **缓存 Top N**：排行榜前 100 名是热点（80% 查询看第一页），可把 Top 100 缓存到本地或 CDN，TTL 1~5 秒，大幅降低 Redis 压力。更新时失效缓存。
- **分数更新的原子性**：游戏场景「加经验值」用 `ZINCRBY`（原子自增），避免「读-改-写」竞态。批量更新用 Pipeline 减少网络往返。
- **排行榜的刷新策略**：实时排行榜分数随时变，用户看到的排名可能瞬变。业务上可引入「赛季」概念，赛季内排名快照固定，赛季末结算，避免实时刷新的体验问题。
