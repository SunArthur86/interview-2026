# 请设计一个支持全球排名的实时排行榜系统，要求数据更新频率高且能支持分页查询。

实时排行榜设计的核心在于高效维护有序集合。Redis的ZSET（SkipList + Hash）是天然选择，其插入、删除复杂度为O(log N)，且支持根据分数范围查询。针对全球海量数据，单机Redis内存可能不足，需进行分片。

**分片策略**：可按分数范围分片（如0-100分在Shard1，100-200在Shard2），但这涉及数据迁移。更实用的方式是按Key哈希分片（如CRDT），但这使得全局排序变难。

**优化方案**：
1. **本地聚合+全局归并**：请求时并行查询各分片Top N，在应用层归并排序。
2. **懒淘汰机制**：对于分数变化不大的数据，利用定时任务定期固化快照，减少实时计算压力。
3. **读写分离**：利用Redis主从架构，主节点处理高频更新，从节点处理分页查询。
4. **针对分页优化**：对于页数极深的查询（如100万页），建议禁止或采用游标方式（基于Last Score和Last ID），避免SkipList深层遍历。

**实战案例**：
某游戏在跨服战期间，排行榜更新QPS达到50万，使用单一Redis ZSET导致主节点CPU单核跑满，数据更新延迟严重。最终采用“分桶策略”，将玩家按ID哈希分配到100个Redis ZSET中，查询时在内存中归并Top 100，解决了单点瓶颈。

**代码示例（基于游标的分页查询）**：
```python
# 避免使用 ZRANK skip，使用 ZRANGEBYSCORE 实现高性能翻页
# last_score 和 last_member 为上一页最后一条数据的分数和成员

def get_ranking_page(last_score, last_member, page_size):
    # 查询分数小于 last_score，或者分数相同但ID字典序小于 last_member 的数据
    return redis.zrevrangebyscore(
        "global_rank", 
        f"({last_score}", # '(' 表示不包含该分数
        "-inf", 
        start=0, 
        num=page_size, 
        withscores=True
    )
```
