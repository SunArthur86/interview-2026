# 如何设计抽奖系统？保证公平、防刷、高并发。

【场景分析】
抽奖系统核心：概率可控、结果不可预测、防刷防作弊、高并发。

**实战案例**：某年会抽奖活动中，攻击者通过脚本模拟请求“跳过动画直接调用接口”，在1分钟内请求5万次，将剩余的大奖奖品库存瞬间耗尽。修复方案是强制前端增加动态Token + 后端Redis令牌桶限流，并在计算概率前增加了“安全随机数盐值”验证。

【奖品概率算法】
1. 权重随机：
   ```
   奖品A: 权重1（0.1%）
   奖品B: 权重5（0.5%）
   奖品C: 权重994（99.4%）
   总权重1000 → 随机1-1000 → 落在哪个区间
   ```
2. 库存限制：
   - 高价值奖品限量（如iPhone 13台）
   - Redis原子扣减库存
   - 库存为0后概率归零

```java
// 离散别名算法简化版（适合奖品少且固定）
public int drawLottery(List<Prize> prizes) {
    int totalWeight = prizes.stream().mapToInt(Prize::getWeight).sum();
    int randomNum = new SecureRandom().nextInt(totalWeight);
    int cursor = 0;
    for (Prize prize : prizes) {
        cursor += prize.getWeight();
        if (randomNum < cursor) {
            return prize.getId(); // 命中
        }
    }
    return -1; // 保底
}
```

【抽奖流程】
```
用户抽奖请求
  → 防刷检查（频率/设备/风控）
  → 消耗抽奖次数（Redis DECR）
  → 概率计算 → 确定奖品
  → 奖品库存检查（Redis DECR）
  → 库存不足 → 降级到保底奖品
  → 记录中奖结果
  → 异步发放奖品
```

【防作弊设计】
1. 抽奖次数控制：
   - 每日限抽N次
   - 抽奖次数与积分/任务关联
2. 结果不可预测：
   - 服务端计算概率（不是前端）
   - 使用安全随机数（SecureRandom）
   - 前端只展示动画
3. 防刷：
   - 设备指纹去重
   - IP限频
   - 行为验证码
4. 阎王保底机制：
   - 抽N次必中某奖
   - 基于Redis计数

【奖品类型处理对比】
| 类型 | 发放策略 | 一致性要求 | 降级方案 |
|------|----------|------------|----------|
| 虚拟奖品 | 实时自动入账 | 高 | 发放失败补发 |
| 实物奖品 | 异步记录，待填地址 | 中 | 库存不足时发等值积分 |
| 现金红包 | 调用支付接口 | 极高 | 支付失败进入重试队列 |

【高并发优化】
- 奖品概率预计算：Redis存储概率表
- Lua脚本原子操作：扣次数 + 算概率 + 扣库存
- 异步发放：MQ处理实物奖品发放
- 降级策略：高峰期关闭抽奖或降低奖品概率

【奖品发放】
- 虚拟商品（积分/优惠券）：自动发放
- 实物商品：填写地址 → 人工发货
- 大奖（汽车/房子）：公证 + 人工确认

【数据统计】
- 奖品发放统计
- 中奖率监控
- 活动ROI分析
- 异常中奖检测

【## 常见考点】】
1. **概率算法的公平性**：如何防止同一用户在极短时间内连续抽中大奖？（设置用户级冷却时间，或控制高价值奖品的单个用户获取频率）
2. **库存扣减失败后的概率回滚**：若算法算中了奖品 A，但扣减库存时 A 已售罄，如何处理？（降级策略：返回“谢谢参与”或发放一个无库存限制的安慰奖 B，且该过程必须在同一个 Lua 原子操作中完成）
3. **随机数安全性**：`java.util.Random` 为什么不安全？（它是伪随机，可预测；应使用 `java.security.SecureRandom` 或操作系统提供的真随机源）
4. **库存超卖问题**：Redis Lua 脚本中 `decr` 返回值小于 0 如何处理？（Lua 脚本中先判断当前库存，若 >0 则扣减并返回中奖 ID，否则返回保底奖品 ID，避免回滚开销）

## 记忆要点

- 核心流程：风控限流 -> Lua脚本内（扣抽奖次数 + 算概率 + 扣奖品库存）保证原子性
- 安全防作弊：必须服务端算概率，强制用SecureRandom防预测，前端仅做动画展示
- 库存超卖处理：Lua内先判库存，不足则不报错，直接降级返回保底安慰奖
- 大奖防连抽：针对高价值奖品设置用户级冷却时间，严控单用户最高获取频率
- 概率算法：基于权重区间随机落入，库存归零的奖品权重动态置0
