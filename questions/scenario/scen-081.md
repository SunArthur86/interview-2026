# 如何设计一个秒杀系统的库存预热和回滚机制？

【场景分析】
秒杀前需预热库存到缓存，秒杀后需处理超时未支付订单的库存回滚。

**【实战案例】**
某次秒杀活动中，因支付回调延迟，导致大量订单在库存回滚后才支付成功，造成超卖。后来引入了“预占库存”机制，下单时扣减Redis库存并设15分钟TTL，支付成功仅更新DB状态，未支付则等待TTL自动过期回滚，极大简化了逻辑。

【库存预热流程】
```
秒杀前2小时：
1. 从DB加载商品库存到Redis
   SET seckill:stock:{skuId} {stockCount}
2. 加载秒杀活动信息
   SET seckill:info:{activityId} {json}
3. 初始化购买记录Set
   SET seckill:bought:{activityId} (空Set)
4. 检查Redis连接和内存
5. 压测验证预热数据正确性
```

【库存预热保障】
- 双写校验：写入后立即读出对比
- 多机房预热：各机房独立预热
- 数据校验：Redis库存 vs DB库存一致性检查
- 预热失败告警：自动重试 + 人工介入

【库存回滚场景】
1. 订单超时未支付：
   - 创建订单后15分钟未支付
   - 延迟队列触发 → 订单取消 → 库存回滚
2. 用户主动取消：
   - 用户手动取消订单
   - 实时回滚库存
3. 支付失败：
   - 支付渠道返回失败
   - 订单关闭 → 库存回滚
4. 退款：
   - 用户申请退款
   - 退款成功 → 库存回滚（如需）

**【代码示例：Lua脚本 - 原子性扣减库存】**
```lua
-- seckill_stock.lua
local stockKey = KEYS[1]
local userKey = KEYS[2]
local userId = ARGV[1]

-- 1. 校验是否重复购买
if redis.call('sismember', userKey, userId) == 1 then
    return -1 -- 重复购买
end

-- 2. 扣减库存
if redis.call('get', stockKey) > 0 then
    redis.call('decr', stockKey)
    redis.call('sadd', userKey, userId)
    return 1 -- 成功
else
    return 0 -- 库存不足
end
```

【库存回滚流程】
```
触发回滚 →
  Redis: INCR seckill:stock:{skuId} (库存+1)
  Redis: SREM seckill:bought:{activityId} {userId} (移除购买记录)
  DB: UPDATE stock SET num=num+1 WHERE sku_id=?
  MQ: 发送库存回滚事件 → 相关服务处理
```

**【方案对比：库存回滚策略】**
| 策略 | 实现方式 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **主动回滚** | 延时队列/定时任务扫描DB | 精准控制，数据一致性好 | 依赖任务调度系统，实现复杂 |
| **被动/TTL回滚** | Redis Key自动过期 | 无需轮询，性能极高 | 无法区分“取消”还是“过期”，DB同步可能延迟 |
| **消息队列** | 订单超时消息 | 解耦，吞吐量高 | 消息积压风险，需保证可靠消费 |

【回滚一致性】
- Redis和DB需同时回滚
- 先回滚Redis（快），再异步回滚DB
- 定时全量校验Redis vs DB库存
- 异常告警：库存不一致时立即排查

【回滚防重】
- 订单状态机：只有"待支付"状态才能触发回滚
- 回滚操作幂等：检查是否已回滚过

【监控指标】
- 库存预热成功率
- 库存回滚延迟
- Redis vs DB库存差异
- 回滚失败数
- 订单超时率

## 记忆要点

- 核心思想：秒杀前DB数据双写校验预热进Redis，防DB被击穿。
- 防超卖与防重：利用Lua脚本保证原子性，校验Set防重与Decr库存扣减同步执行。
- TTL自动回滚：创建订单时预占Redis库存并设TTL，超时未支付则自动过期回滚。
- 一致性保障：Redis先快速回滚，再异步MQ通知DB回滚，且依赖订单状态机防重复回滚。
