# 如何设计一个朋友圈/社交媒体动态系统？

【场景分析】
朋友圈需求：发布动态（图/文/视频）、好友可见、互动（点赞/评论）、隐私控制。
核心难点：高并发写、读扩散（Feed流）的高吞吐、权限控制的复杂度。

【核心模块】
1. **发布服务**：
   - 流程：内容上传（直传OSS） → 异步转码（FFmpeg集群）→ 存入元数据DB。
   - **扩展性**：秒级URL预签名，减少服务端流量压力。
2. **Feed分发（核心）**：
   - **推模式**：写扩散。发动态时，写入所有粉丝的 Inbox。优点：读极快；缺点：写放大（大V灾难）。
   - **拉模式**：读扩散。读动态时，遍历关注列表，拉取最新动态合并排序。优点：写快；缺点：读慢（聚合计算）。
   - **混合模式**：
     - 普通用户（<5000粉丝）：推模式，写入粉丝 Timeline。
     - 大V（>5000粉丝）：拉模式，粉丝读取时主动拉取。
3. **互动服务**：
   - 点赞：Redis Set/ZSet 存储，异步刷盘 MySQL。
   - 评论：支持楼中楼，使用邻接表或闭包表路径存储。
4. **隐私控制**：
   - **ACL（访问控制列表）**：发布时计算可见用户集。
   - **黑名单**：读时过滤或推时剔除（推时剔除更高效）。

【数据模型】
- **动态表**：分片键为 `user_id`。核心字段 `content_id`, `visibility_rule` (bitmask存储权限)。
- **关系表**：双向好友关系，`followee_id` + `follower_id` 联合唯一索引。

【朋友圈特殊设计】
- **时间线回溯**：点击好友头像，进入该好友的"个人相册"，此时只能看到该好友发布的动态（需剥离权限过滤）。
- **三天可见**：
  - 方案1：查询时过滤 `created_at > NOW() - 3days`（简单但无法利用时间索引）。
  - 方案2：物理删除或标记软删除（定时任务清理旧数据）。

【性能优化】
- **读扩散优化**：拉取好友Feed时，优先查缓存（Redis List/ZSet），未命中回源DB，并异步预热缓存。
- **秒杀式互动**：热门动态的点赞计数使用 `Redis incr`，并配合 Lua 脚本防止超卖（如限制总赞数显示范围）。

## 常见考点
1. **推拉模式的选择阈值**：如何决定多少好友数切换模式？回答：根据系统瓶颈测试，通常 2000-5000 为界。
2. **如何保证Feed不丢？** 回答：发动态成功才返回客户端，MQ 至少一次投递，消费端幂等性设计。
3. **大V粉丝看Feed太慢怎么优化？** 回答：分级拉取，只拉取大V最近 N 条，结合多线程并发拉取，客户端分页加载（无限滚动）。
4. **评论盖楼如何实现？** 回答：一般只支持两级（主评论+回复），回复过多时点击"查看更多"分页加载，避免 N+1 查询。

## 记忆要点

- 核心推拉模式：普通用户用推（写扩散读极快），大V（粉丝>5000）用拉（读扩散防写放大）
- 因为大V粉丝多易写放大，所以用混合模式：推拉结合，阈值通常2000-5000
- 互动设计：点赞用Redis ZSet incr防超卖，评论用邻接表或闭包表支持楼中楼
- 隐私控制对比：发布时算ACL定可见集，黑名单在推时剔除比读时过滤更高效
