# 如何设计微信抢红包系统？红包金额如何拆分？

### 场景分析
抢红包核心需求：发红包（金额拆分存储）、抢红包（并发扣减原子操作）、查记录。

### 红包金额拆分算法
1. **二倍均值法（微信使用）**
   - 每次抢的金额 = 随机(0.01, 剩余金额 / 剩余人数 × 2)
   - 保证公平性，不会出现极端值
   - 最后一个人 = 剩余全部金额
2. **线段切割法**
   - 将总金额看成一条线段，随机切 N-1 刀分成 N 份
   - 更随机但可能出现 0 元

### 存储设计
- **红包基本信息**：Redis Hash 存储（id, 总金额, 剩余个数, 剩余金额）
- **红包明细列表**：Redis List 存储拆分后的金额
- **持久化**：异步写入 MySQL

### 抢红包核心流程
1. **Lua 脚本原子操作**（保证并发安全）
   - 判断红包是否过期 → 判断是否已抢过 → List.pop 金额 → 更新剩余信息
2. 抢到的金额异步写入 DB
3. 24 小时未领取的退回发送者

### 高并发优化
- Redis Cluster + 读写分离
- 抢红包记录用 Bitmap 去重（userId → bit 位）
- 热点红包本地缓存预热
- DB 分库分表：按红包 ID 分片

### 实战案例
某公司春节发红包活动，因未限制用户抢包频率，某脚本用户一秒钟抢到了同一个红包内的多个金额（并发竞态）。修复方案是在 Lua 脚本最开始使用 `hsetnx` 进行原子去重检查。

### 代码示例
```lua
-- 抢红包 Lua 脚本
local redPacketId = KEYS[1]
local userId = ARGV[1]

-- 1. 原子去重：记录用户抢包状态
if redis.call('hsetnx', 'rp_user:'..redPacketId, userId, '1') == 0 then
    return {-1, nil} -- 已抢过
end

-- 2. 出队金额
local amount = redis.call('rpop', 'rp_list:'..redPacketId)
if not amount then
    redis.call('hdel', 'rp_user:'..redPacketId, userId) -- 没钱了回滚状态
    return {0, nil}
end

-- 3. 更新剩余信息
redis.call('hincrby', 'rp_meta:'..redPacketId, 'remain_count', -1)
return {1, amount}
```

### 常见考点
1. **原子性保证**：为什么必须用 Lua 脚本？（Redis 单线程执行 Lua 脚本期间不会被其他命令插队，保证「读-改-写」原子性，解决超抢）。
2. **二倍均值法细节**：为什么要 `× 2`？（保证均值稳定，避免后续抢到的金额忽大忽小，确保每人抢到的金额期望值相同）。
3. **幂等性**：用户疯狂点击按钮，如何防止抢到多个？（Lua 脚本中先用 Set/Bitmap 记录 `hset red_packet_id user_id 1`，若已存在则直接返回错误）。
