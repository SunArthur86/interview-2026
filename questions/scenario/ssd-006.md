# 如何设计一个限流系统？如何在分布式环境中实现？

🎯 本质：限流系统需要在分布式环境中精确控制请求速率，保护后端服务。

📊 架构设计：

1. **多层限流**
- Nginx层：IP限流(limit_req)
- 网关层：API限流（Spring Cloud Gateway RateLimiter）
- 应用层：接口/用户限流（Sentinel）
- DB层：连接池限流

2. **分布式限流方案**

**方案A：Redis + Lua（集中式）**
所有服务共享Redis计数器
优点：精确
缺点：Redis单点/延迟

**方案B：Token Server（Sentinel集群限流）**
独立Token Server下发令牌
优点：高性能
缺点：Token Server单点

**方案C：本地限流 + 全局协调**
每个实例本地限流（如限流1000/QPS），加上全局配额协调
优点：低延迟
缺点：不够精确

3. **动态限流**
- 根据系统负载动态调整阈值（CPU > 80% 降低限流阈值）
- 根据依赖服务状态调整（下游故障时收紧限流）
- 配合自适应限流（不需要预设阈值）

**限流算法对比（漏桶 vs 令牌桶）：**
```text
漏桶:
Requests --> [水桶(固定漏出速率)] --> Processing
效果: 强行平滑流量，突发流量会被丢弃或等待，保护性强。

令牌桶:
[Token Bucket (恒定放入Token)] <-- (Permits)
      ^
      | (取走Token)
Requests --> Processing
效果: 允许一定程度的突发流量（如果有剩余Token），性能友好。
```

Sentinel实现示例：
// 流控规则
FlowRule rule = new FlowRule();
rule.setResource("createOrder");
rule.setGrade(RuleConstant.FLOW_GRADE_QPS);
rule.setCount(100);  // QPS=100
// 热点参数限流
ParamFlowRule hotRule = new ParamFlowRule("queryOrder")
    .setParamIdx(0)
    .setCount(10);  // 每个userId QPS=10

4. **限流后的处理策略**
- 直接拒绝（返回429 Too Many Requests）
- 排队等待（令牌桶等待）
- 降级（返回缓存/默认值）
- 熔断（连续触发后熔断）

## 常见考点
1. **算法选择**：漏桶算法和令牌桶算法的区别是什么？为什么Google Guava RateLimiter使用令牌桶？（令牌桶允许应对突发流量，更适合互联网应用）。
2. **分布式原子性**：Redis Lua脚本实现限流时，如何保证原子性？（Redis单线程执行Lua命令，中间不会插入其他命令）。
3. **滑动窗口 vs 固定窗口**：固定窗口算法存在“临界突发”问题（窗口边界瞬间流量倍增），如何用滑动窗口解决？（记录更细粒度的时间戳，计算当前窗口内累计数量）。
