# 如何设计蓝绿部署和金丝雀发布方案？

【场景分析】
部署策略目标：零停机发布、快速回滚、降低发布风险。

【蓝绿部署】
原理：维护两套完全相同的生产环境（蓝和绿），同一时间只有一套对外服务。

流程：
1. 当前蓝环境提供服务
2. 新版本部署到绿环境
3. 绿环境通过测试验证
4. 路由切换：流量从蓝→绿（秒级切换）
5. 蓝环境保留作为回滚备份
6. 确认稳定后销毁旧环境

优点：
- 零停机切换
- 秒级回滚（切回蓝）
- 完整环境验证

缺点：
- 双倍资源成本
- 数据库迁移复杂
- 适合无状态服务

【金丝雀发布】
原理：逐步将新版本推给一小部分用户，验证后再逐步扩大。

流程：
1. 生产环境运行老版本（10个实例）
2. 新增1个新版本实例（10%流量）
3. 观察1小时，指标正常
4. 新增到3个实例（30%）
5. 继续...直到100%
6. 移除老版本实例

优点：
- 风险最小化
- 真实用户验证
- 渐进式发布

缺点：
- 发布时间长
- 需要流量路由能力
- 版本兼容性要求

【滚动发布】
K8s默认策略：
1. 逐步替换Pod（先启新的，再杀旧的）
2. maxSurge + maxUnavailable控制速度
3. 新旧版本短暂共存

【A/B测试发布】
- 基于用户特征分流（不是流量比例）
- 同时收集新版本的用户行为数据
- 对比指标决定是否全量

【实现方案】
1. K8s Deployment：内置滚动更新
2. Service Mesh（Istio）：精确流量分配
3. API网关：Header/Cookie路由
4. 负载均衡器：权重调整

【数据库兼容】
- 扩展兼容：新增字段有默认值
- 收缩兼容：先改代码再删字段
- 禁止Breaking Change

【选择建议】
关键服务：金丝雀（风险最低）
常规服务：滚动更新（K8s默认）
紧急回滚：蓝绿（秒级回滚）

### 增强补充：技术细节与架构对比

1. **蓝绿部署的流量切换细节**
   - **基础设施层**：通过修改负载均衡（如 Nginx/ALB）的上游服务器配置来实现切换。
   - **Service Mesh 层**：修改 VirtualService 的路由规则。
   - **全链路染色**：若涉及微服务链路，蓝绿环境必须全链路隔离。若 A 服务调用 B 服务，A 是蓝色，B 必须路由到蓝色的 B 实例。通常通过透传 Header（如 `x-env: blue`）实现。

2. **金丝雀发布的流量权重算法**
   - **随机加权**：最简单，Nginx 或 Istio 按百分比随机分发。
   - **一致性哈希**：基于 UserID 或 RequestID 哈希，确保特定用户始终路由到新版本，避免用户体验不一致（如一会新界面一会有界面）。

3. **滚动发布与数据库连接池的坑**
   - K8s 滚动发布时，老 Pod 销毁前，可能仍有数据库连接未释放，导致 DB 连接数瞬间抖动。
   - **解决方案**：配置 `preStop Hook`，在 Pod terminationGracePeriodSeconds 期间，先暂停接收新流量，等待现有请求处理完毕（包括连接池归还）后再销毁。

## 常见考点
1.  **金丝雀发布如何保证数据一致性？**
    *   *提示*：如果新老版本写入不同的 Schema 或数据库，如何回滚？通常要求新版本代码兼容旧数据结构，或者采用双写、并行读写模式。
2.  **蓝绿部署时，数据库怎么办？**
    *   *提示*：通常共享数据库。如果是涉及大表结构变更，需要使用 Online DDL 工具（如 pt-online-schema-change）或先改库（兼容新旧代码）再发布代码。
3.  **K8s 滚动更新中的 `maxSurge` 和 `maxUnavailable` 的含义？**
    *   *提示*：`maxSurge` (最大激增) 允许超过副本数的 Pod 数量；`maxUnavailable` (最大不可用) 允许不可用的 Pod 数量。快速发布通常调大 `maxSurge`；保守发布则调小 `maxUnavailable`。
