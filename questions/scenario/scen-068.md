# 如何设计优惠券系统？支持海量发放、领取、核销。

【场景分析】
优惠券系统核心需求：创建/发放/领取/使用/过期、防超领/防刷、多种券类型。

**实战案例**：在“双11”大促中，曾出现百万用户同时抢“满100减50”神券，导致Redis QPS瞬间达到10万。因Lua脚本未包含“用户资格校验”（如是否为新用户），导致大量无效请求打到DB，引发数据库雪崩。优化后在校验阶段前置布隆过滤器过滤非目标用户。

【券类型】
1. 满减券：满100减20
2. 折扣券：8折
3. 代金券：直接抵扣10元
4. 兑换券：免费兑换商品
5. 运费券：免运费

【数据模型】
```
优惠券模板：
  id, name, type, amount, threshold, total_count, received_count
  start_time, end_time, status

用户优惠券：
  id, user_id, template_id, status(未使用/已使用/已过期)
  receive_time, use_time, order_id
```

【领取流程（高并发）】
```
客户端
  |
  v
[1] 风控检查(限流/黑名单)
  |
  v
[2] 预校验(活动时间/用户资格)
  |
  v
[3] Redis Lua脚本原子操作:
      |-> 检查库存是否 > 0
      |-> 检查用户是否已领 (Set)
      |-> DECR 库存
      |-> SADD 用户领券记录
  |
  +-- 失败 --> 返回错误提示
  |
  v
[4] MQ 异步落库
  |
  v
[5] 返回领取成功
```

```lua
-- 领券原子操作 Lua
local stock_key = KEYS[1]
local user_key = KEYS[2]
local user_id = ARGV[1]
-- 检查库存
if redis.call('get', stock_key) == '0' then
    return -1 -- 库存不足
end
-- 检查是否已领
if redis.call('sismember', user_key, user_id) == 1 then
    return -2 -- 已领取
end
-- 扣库存 + 记录用户
redis.call('decr', stock_key)
redis.call('sadd', user_key, user_id)
return 1
```

【核销流程】
1. 下单时选择优惠券
2. 校验：券是否有效（未过期/未使用/满足条件）
3. 锁定优惠券（状态改为“使用中”）
4. 订单支付成功 → 核销（状态改为“已使用”）
5. 订单取消/退款 → 释放优惠券（状态恢复）

【过期处理】
- 定时任务扫描过期券
- 状态更新为“已过期”
- Redis消息通知用户

【高性能设计】
- 库存预热到Redis
- 用户已领记录用Redis Set快速判断
- 优惠券信息缓存
- DB分库分表（按user_id）

【库存扣减方案对比】
| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| DB 乐观锁 | 数据强一致 | 高并发下大量失败 | 低并发普通券 |
| Redis Lua | 高性能，原子 | 异步写库有延迟风险 | 秒杀/抢券 |
| Redis + MQ | 吞吐量最高 | 最终一致性，需对账 | 极高并发大促 |

【券叠加规则】
- 同一订单最多N张券
- 不同类型券的叠加规则
- 满减和折扣是否可叠加
- 规则引擎管理叠加策略

【## 常见考点】】
1. **库存数据一致性**：Redis 扣减成功但 MQ 消息丢失导致数据库未记录，如何处理？（使用本地消息表或对账任务进行“T+N”数据一致性修复）
2. **大并发下的数据库竞争**：如果不走 Redis 直接扣数据库，乐观锁会导致大量失败，如何优化？（分库分表降低单行锁竞争，或改为 Redis 扣库存后异步写库）
3. **一人多领问题**：如何支持“每人限领 2 张”的数据结构？（Redis 使用 Hash 结构 `HSET user:{userId} {templateId} count`，配合 Lua 脚本检查）
4. **优惠券分布式事务**：下单扣券和扣库存涉及多个微服务，如何保证一致性？（TCC 或 Seata 框架，最终一致性；先锁券，支付成功后再核销）

## 记忆要点

- 核心防超领：Redis Lua脚本保证原子性（判库存 + 查Set去重 + 减库存）
- 高并发抢券：前置风控过滤，扣减Redis后异步发MQ落库DB，实现流量削峰
- 一人多领：改用Hash结构记录单领数量，在Lua内校验防超发
- 数据一致性：Redis扣成功但MQ丢失，需用本地消息表或T+N对账任务修复
- 核销防翻转：下单先锁券（预扣），支付成功才核销，取消订单再异步释放
