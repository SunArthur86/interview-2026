# 如何设计一个微服务的网关限流降级一体化方案？

【场景分析】
网关是流量入口，需要同时实现限流、熔断、降级、缓存等能力，保护后端微服务。

【一体化架构】
```text
Request → [Global Filter Chain]
   │
   ├─ 1. 黑名单/白名单检查
   ├─ 2. 认证鉴权
   ├─ 3. 全局限流（集群总QPS）
   ├─ 4. 接口限流（单接口QPS）
   ├─ 5. 用户限流（User ID 维度）
   ├─ 6. 熔断检查（基于响应时间/异常率）
   ├─ 7. 请求改写/参数校验
   ├─ 8. 本地/分布式缓存检查
   ├─ 9. 路由转发 → 微服务
   ├─ 10. 响应缓存与封装
   └─ 11. 监控指标上报
```

【限流策略矩阵】
| 维度 | 算法 | 阈值 | 存储 | 场景 |
|------|------|------|------|------|
| 全局QPS | 令牌桶 | 100万/s | Redis集群 | 防止整体服务雪崩 |
| 接口QPS | 滑动窗口 | 1万/s | 本地+Redis | 保护单点脆弱接口 |
| 用户QPS | 固定窗口 | 100/min | Redis | 防止刷单/脚本 |
| IP QPS | 漏桶 | 1000/min | Redis | 防止DDoS攻击 |

【熔断策略详解】
- **状态机**：关闭 → 打开 → 半开
- **触发条件**：
  - 慢调用比例：RT > 1000ms 的比例 > 50%
  - 异常比例：抛出异常比例 > 50%
- **恢复机制**：熔断 10s 后进入半开状态，放行 1 个请求，若成功则关闭，失败则重新打开。

【降级策略】
| 服务状态 | 降级动作 |
|---------|---------|
| 熔断中 | 返回默认值/空页面 |
| DB慢 | 关闭非核心写功能，只读 |
| 缓存挂 | 限流直连DB |
| 全过载 | 仅保核心链路（如下单），放弃评论/推荐 |

【Spring Cloud Gateway + Sentinel实现】
```java
// 代码逻辑保持原有，增加细节注释
public class RateLimitFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String userId = getUserId(exchange);
        String resourceId = exchange.getRequest().getPath().value();
        
        // 1. 加载规则 (Sentinel API)
        Entry entry = null;
        try {
            entry = SphU.entry(resourceId, EntryType.IN, 1, userId);
            // 2. 限流检查通过
            return chain.filter(exchange)
                .onErrorResume(e -> handleFallback(exchange, e)); // 异常降级处理
        } catch (BlockException ex) {
            // 3. 触发限流/熔断
            return handleBlock(exchange, ex);
        } finally {
            if (entry != null) {
                entry.exit(1, userId);
            }
        }
    }
}
```

【动态规则推送】
- 规则持久化：Nacos/Apollo
- 推送模式：配置中心 → Sentinel Dashboard → Gateway (长轮询/WS)
- 热更新：无需重启网关，毫秒级生效。

【监控指标】
- 拒绝 QPS、通过 QPS
- 熔断触发次数、恢复时间
- 线程池/信号量使用率（异步网关模式下）

【## 常见考点】
1. **限流算法选择**：为何单机限流用滑动窗口而分布式限流常用令牌桶？（滑动窗口实现更精确，令牌桶处理突发能力更好，适合网络环境）。
2. **网关性能瓶颈**：网关逻辑太重（如加解密、复杂校验）如何优化？（使用 Netty 零拷贝、全异步非阻塞、将 CPU 密集型逻辑下沉到 Sidecar 或独立服务）。
3. **熔断粒度**：是基于服务粒度熔断还是接口粒度？（推荐接口粒度，避免因一个非核心接口异常导致整个服务不可用）。
4. **配置一致性**：多网关实例下，如何保证限流配置同步更新？（依赖配置中心的高可用推送机制，网关本地缓存规则作为兜底）。
