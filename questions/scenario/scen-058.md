# 如何设计一个大规模数据迁移方案？不停机迁移百亿数据。

【场景分析】
数据迁移常见场景：数据库升级、分库分表改造、上云迁移、系统重构。

【迁移挑战】
- 不能停服（7×24服务）
- 数据量大（百亿级）
- 迁移期间有增量写入
- 数据一致性保证
- 可回滚

【双写迁移方案（推荐）】

【Phase 1: 准备阶段】
- 新库建表、分片
- 数据校验工具准备
- 回滚方案设计

【Phase 2: 全量同步】
- 历史数据一次性导入
- 工具：DataX / Spark / 自研
- 按ID范围分批迁移
- 迁移期间记录增量（binlog）

【Phase 3: 增量同步】
- Canal监听老库binlog
- 实时同步增量数据到新库
- 追上全量迁移的gap

【Phase 4: 双写开启】
- 代码改造：同时写新老库
- 老库为主，新库为辅
- 新库写入失败只告警不阻断

【Phase 5: 读流量切换】
- 按比例灰度切读流量到新库
- 1% → 10% → 50% → 100%
- 每个阶段观察和校验数据

【Phase 6: 写流量切换】
- 关闭老库写入
- 新库成为唯一写入源
- 观察稳定性

【Phase 7: 清理】
- 下线双写代码
- 归档老库

【数据迁移全流程图】
```text
  时间轴 →

 Phase 1/2:      [ 老库 ] ──(全量导出)──▶ [ 同步程序/ETL ] ──▶ [ 新库(空) ]
                      │
 Phase 3:           (增量Binlog)
                      ▼
                 [ Canal/MQ ] ───────────────────────────────▶ [ 更新新库 ]
                      │
 Phase 4:  [ 应用 ] ──┼──▶ [ 写老库 ]
           (双写模式)  └──▶ [ 写新库 ]
                      │
 Phase 5:  [ 应用 ] ──┼── 读取(部分) ─────────────────────▶ [ 读新库 ]
                      └── 读取(部分) ──▶ [ 读老库 ]
                      │
 Phase 6:  [ 应用 ] ───────▶ [ 写新库 (唯一源) ]
                      │
 Phase 7:              (关闭双写, 下线老库)
```

【数据校验】
1. 全量校验：
   - 对比新老库每条记录
   - 工具：自研MapReduce/Spark
   - 校验：行数、字段值、CRC32
2. 增量校验：
   - 实时比对binlog和目标库
   - 差异告警
3. 抽样校验：
   - 随机抽样对比
   - 快速但不全面

【回滚方案】
- Phase 5前回滚：关闭双写即可
- Phase 5后回滚：读切回老库
- Phase 6后回滚：新库binlog反向同步到老库

【注意事项】
- 迁移时间窗口：选择低峰期
- 监控：迁移速率、延迟、错误率
- 限速：控制迁移速率避免影响在线服务
- 分批迁移：按业务维度分批，降低风险

## 常见考点
1. **双写阶段如何保证数据一致性（如新老库写入成功时间差）？**
   答：业务代码双写通常无法做到原子性。采用“异步对账”机制，定时任务比对差异并修复。读流量切换前，必须确保数据已经追平且校验通过。
2. **如果新库性能不如老库怎么办？**
   答：在双写阶段只写不读，压力较小。切读流量时要非常小心，从低百分比开始，密切监控新库QPS和RT（响应时间）。如果新库扛不住，立即回滚读流量，并扩容新库或优化索引。
3. **停写切换（Phase 6）瞬间数据丢失风险？**
   答：依赖数据库层面的主从切换机制或配置中心下发开关。确保开关下发是一个“原子操作”或者“Paxos/Raft共识”的结果，避免部分机器写老库、部分机器写新库的分裂脑状态。
4. **Binlog 增量同步延迟过大怎么处理？**
   答：如果同步速度跟不上写入速度，会造成“数据倒挂”（永远追不上）。处理方式：优化同步通道的并发度，对大事务进行拆分，或者在业务低峰期暂停写入进行追赶。


## 记忆要点

- 核心方案：双写法。因业务不停机，故需全量同步结合Binlog增量追平数据。
- 全量阶段：按ID分批迁移老数据，期间通过Canal监听记录增量。
- 双写与切流：先双写（老主新辅），后灰度切读（1%到100%），最后切写。
- 一致性保障：业务双写非原子，必须依赖异步对账修复数据。
- 关键防线：切流前务必确保新老库数据完全追平校验，并设计回滚开关。
