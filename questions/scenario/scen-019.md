# 如何解决缓存雪崩、缓存穿透、缓存击穿问题？

【三大缓存问题及解决方案】

【1. 缓存雪崩】
**场景**：大量 Key 在同一时间大面积过期，或 Redis 宕机，导致海量请求直击 DB。

**解决方案**：
- **过期时间随机化**：设置 TTL 时加入随机值，`TTL = base_time + random(0, 300)`，避免集体失效。
- **Redis 高可用**：搭建 Sentinel 或 Cluster，实现故障秒级切换。
- **多级缓存**：即使 Redis 挂了，本地缓存仍能兜底一部分流量。
- **限流降级**：当 DB QPS 超过阈值，开启熔断，返回默认值或错误页。
- **服务预热**：项目启动或活动前，提前加载热点数据。

【2. 缓存穿透】
**场景**：查询一个**根本不存在**的数据（恶意攻击或业务空缺），导致 Cache 和 DB 都没有，每次请求都穿透到 DB。

**解决方案**：
- **缓存空值**：
  - 当 DB 查询为空时，Redis 缓存一个 `NULL` 或特定值，TTL 设置短一点（如 30秒-5分钟）。
- **布隆过滤器**：
  - **原理**：一个很长的二进制向量和一系列哈希映射。判断 Key 是否一定不存在。
  - **流程**：请求前先问 BF，若 BF 说不存在，则直接拒绝，不走 DB。
  - **优势**：内存占用极小（百万 Key 仅需几 MB）。
- **参数校验**：拦截 ID < 0 等非法请求。

【3. 缓存击穿】
**场景**：某个**极度热点**的 Key（如秒杀商品）突然过期，此时巨量并发请求瞬间击穿缓存，压垮 DB。

**解决方案**：
- **互斥锁**：
  - 只允许一个线程去查 DB 并回写缓存，其他线程等待。
  - ```text
    if cache.get(key) == null:
        if lock.tryLock():  // 获取分布式锁
           val = db.get()
           cache.set(key, val)
           lock.unlock()
        else:
           sleep(10) && retry() // 自旋等待
    ```
- **逻辑过期**：
  - Redis 中不设置 TTL，Value 包含 `data` + `expireTime`。
  - 请求到来时，检查逻辑时间：
    - **未过期**：直接返回。
    - **已过期**：返回旧数据（非阻塞），后台异步线程去更新缓存。
  - *优势*：用户体验好，无需等待。

## 常见考点
1. **布隆过滤器能删除数据吗？为什么？**
   - 不能。因为多个 Key 可能哈希到同一个位置，删除一位会影响其他 Key 的判断。若需删除，需使用 Counting Bloom Filter（计数布隆过滤器），但内存消耗倍增。
2. **互斥锁方案和逻辑过期方案怎么选？**
   - 互斥锁：数据一致性要求高，可容忍短暂阻塞。
   - 逻辑过期：高并发、要求极致响应速度，可容忍短暂的数据不一致（返回旧数据）。
3. **缓存空值方案有什么缺点？**
   - 如果恶意攻击使用大量不同的随机 ID，会导致 Redis 存储大量无用 Key，消耗内存。需结合布隆过滤器使用。

## 记忆要点

- 雪崩（大面积）：因同时过期导致，故TTL加随机值，配合集群高可用与熔断限流
- 穿透（查空值）：因查不存在的Key导致，故缓存空对象或前置布隆过滤器拦截
- 击穿（单热点）：因热点Key突然过期导致，故用互斥锁（强一致）或逻辑过期（高可用）
- 穿透方案缺点：因海量随机ID攻击会耗尽内存，故空值方案需配合布隆过滤器
- 布隆过滤禁忌：因多Key哈希冲突，故不可直接删除，若需删除要用计数布隆
