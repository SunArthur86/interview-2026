# 在秒杀系统设计中，如何利用 Redis 原子操作和异步落库来防止“超卖”现象？如果 Redis 挂了，兜底策略是什么？

防止超卖的核心是将库存扣减操作前置到 Redis 中执行。使用 Lua 脚本保证“读取库存-判断余量-扣减库存”的原子性。脚本中返回剩余库存，若大于等于 0 则扣减成功，随后发送消息到 MQ（如 RocketMQ）异步扣减数据库库存。若 Redis 挂了，兜底策略包括：1. 降级直接访问数据库（通过数据库乐观锁 `UPDATE stock SET num=num-1 WHERE id=? AND num>0` 来保证不超卖，但会极大降低吞吐量）；2. 限流保护，直接拒绝大部分流量；3. 依赖多级缓存或本地缓存 Marker 机制进行快速失败，防止流量击穿数据库。

**1. 实战案例**：
某次生产事故中 Redis 主节点宕机， Sentinel 切换主从期间导致同步丢失（异步复制），系统自动降级到数据库层面扣减。由于兜底 SQL 漏了 `num>0` 条件，导致库存扣成负数。修复后不仅加了条件，还引入了 Redis 暂不可用时的“暂停售卖”静态页兜底。

**2. 代码示例**：
```sql
-- 数据库兜底扣减 SQL (防止 Redis 挂掉时超卖)
-- 利用行锁保证原子性，num>0 是防线，affected_rows > 0 才是成功
UPDATE stock_inventory 
SET num = num - 1, version = version + 1 
WHERE goods_id = 1001 AND num > 0;
```

**3. 对比表格**：
| 维度 | Redis Lua 扣减 | 数据库兜底扣减 |
| :--- | :--- | :--- |
| **性能** | 极高 (10W+ QPS) | 低 (取决于 DB IO 和行锁竞争) |
| **一致性模型** | 最终一致性 (需 MQ 同步) | 强一致性 (事务 ACID) |
| **触发条件** | 正常流量链路 | Redis 故障或预热阶段 |
| **风险点** | Redis 宕机数据丢失 | 高并发下 DB 连接池耗尽 |
| **应对策略** | Sentinel 集群 / RDB+AOF | 严控限流 / 增加机器资源 |
