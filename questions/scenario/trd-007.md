# 在秒杀系统设计中，如何利用 Redis 原子操作和异步落库来防止“超卖”现象？如果 Redis 挂了，兜底策略是什么？

防止超卖的核心是将库存扣减操作前置到 Redis 中执行。使用 Lua 脚本保证“读取库存-判断余量-扣减库存”的原子性。脚本中返回剩余库存，若大于等于 0 则扣减成功，随后发送消息到 MQ（如 RocketMQ）异步扣减数据库库存。若 Redis 挂了，兜底策略包括：1. 降级直接访问数据库（通过数据库乐观锁 `UPDATE stock SET num=num-1 WHERE id=? AND num>0` 来保证不超卖，但会极大降低吞吐量）；2. 限流保护，直接拒绝大部分流量；3. 依赖多级缓存或本地缓存 Marker 机制进行快速失败，防止流量击穿数据库。

**1. 实战案例**：
某次生产事故中 Redis 主节点宕机， Sentinel 切换主从期间导致同步丢失（异步复制），系统自动降级到数据库层面扣减。由于兜底 SQL 漏了 `num>0` 条件，导致库存扣成负数。修复后不仅加了条件，还引入了 Redis 暂不可用时的“暂停售卖”静态页兜底。

**2. 代码示例**：
```sql
-- 数据库兜底扣减 SQL (防止 Redis 挂掉时超卖)
-- 利用行锁保证原子性，num>0 是防线，affected_rows > 0 才是成功
UPDATE stock_inventory 
SET num = num - 1, version = version + 1 
WHERE goods_id = 1001 AND num > 0;
```

**3. 对比表格**：
| 维度 | Redis Lua 扣减 | 数据库兜底扣减 |
| :--- | :--- | :--- |
| **性能** | 极高 (10W+ QPS) | 低 (取决于 DB IO 和行锁竞争) |
| **一致性模型** | 最终一致性 (需 MQ 同步) | 强一致性 (事务 ACID) |
| **触发条件** | 正常流量链路 | Redis 故障或预热阶段 |
| **风险点** | Redis 宕机数据丢失 | 高并发下 DB 连接池耗尽 |
| **应对策略** | Sentinel 集群 / RDB+AOF | 严控限流 / 增加机器资源 |

## 技术原理

防超卖的「Redis 前置 + MQ 异步 + DB 兜底」三层架构，每层解决不同维度的问题：

- **Redis Lua 的原子性原理**：Redis 单线程模型 + Lua 脚本在执行期间不被其他命令打断，天然保证「读库存 → 判断 > 0 → 扣减」三步原子。相比「GET → 客户端判断 → DECR」的多步操作（中间有竞态窗口），Lua 把逻辑放服务端一次执行，消除并发竞态。`DECR` 后若结果 < 0，说明已抢光，`INCR` 回滚——这是「先扣再判」的反悔模式，比「先判再扣」更简洁且无锁。
- **MQ 异步落库的最终一致性**：Redis 扣减成功后，如果同步写 DB，DB 的行锁会成为瓶颈（单商品只能串行扣减）。改为发 MQ 消息异步扣减 DB，DB 侧批量消费、按商品分桶串行处理。代价是引入「Redis 已扣但 DB 未扣」的窗口，需要 MQ 至少一次语义 + 消费幂等保证最终一致。对账机制（定时比对 Redis 余量与 DB 库存）兜底数据偏差。
- **DB 乐观锁兜底**：Redis 宕机时降级到 DB 直接扣减。核心 SQL `UPDATE stock SET num=num-1 WHERE id=? AND num>0` 利用 MySQL 行锁 + WHERE 条件保证原子性——`affected_rows > 0` 才是扣减成功。相比「SELECT num → 应用判断 → UPDATE」的两步（有 TOCTOU 竞态），单条 UPDATE 把判断下推到 DB 引擎层，无锁竞争窗口。
- **本地 Marker 机制的快速失败**：Redis 不可用时，与其让所有请求打到 DB，不如用本地缓存（如 Caffeine）记录「某商品已售罄」标记。请求先查 Marker，售罄则直接返回失败，只有未售罄才走 DB。这把 DB 流量降几个数量级，是保护 DB 的关键防线。

## 代码示例

```lua
-- Redis Lua 库存扣减脚本（含防超卖 + 异步消息触发）
-- KEYS[1] = 库存key, ARGV[1] = 用户ID, ARGV[2] = 订单ID
local stock = redis.call('GET', KEYS[1])
if not stock then return -2 end                          -- key 不存在
local remaining = redis.call('DECR', KEYS[1])
if remaining < 0 then
    redis.call('INCR', KEYS[1])                          -- 回滚：抢光了
    return -1
end
-- 扣减成功，记录已购用户（防重复抢）
redis.call('SADD', KEYS[1] .. ':buyers', ARGV[1])
return remaining                                         -- 返回剩余库存
```

```java
// 秒杀全链路：Redis 前置 + MQ 异步 + DB 兜底
@Service
public class SeckillService {
    @Autowired private RedisTemplate redis;
    @Autowired private RocketMQTemplate mq;
    @Autowired private StockMapper stockMapper;
    @Autowired private CaffeineCache localMarker;   // 本地售罄标记

    public SeckillResult seckill(String skuId, String userId) {
        // 1. 本地 Marker 快速失败（保护 DB）
        if (localMarker.getIfPresent("soldout:" + skuId) != null) {
            return SeckillResult.fail("已售罄");
        }
        // 2. 尝试 Redis 扣减
        try {
            Long remaining = (Long) redis.execute(SECKILL_SCRIPT,
                List.of("stock:" + skuId), userId, genOrderId());
            if (remaining == -1) {
                localMarker.put("soldout:" + skuId, true);   // 标记售罄
                return SeckillResult.fail("手慢了");
            }
            if (remaining == -2) throw new IllegalStateException("库存未初始化");
            // 3. 异步落库（MQ 削峰）
            mq.asyncSend("stock-deduct", new DeductMsg(skuId, userId, orderId));
            return SeckillResult.success(orderId);
        } catch (RedisConnectionFailureException e) {
            // 4. Redis 挂了：降级到 DB 兜底
            return fallbackToDb(skuId, userId);
        }
    }

    @Transactional
    public SeckillResult fallbackToDb(String skuId, String userId) {
        // 限流保护：DB 扛不住时拒绝大部分请求
        if (!rateLimiter.tryAcquire()) return SeckillResult.fail("系统繁忙");
        // DB 乐观锁扣减：行锁 + num>0 保证不超卖
        int affected = stockMapper.deductWithCheck(skuId);
        if (affected == 0) return SeckillResult.fail("已售罄");
        return SeckillResult.success(genOrderId());
    }
}
```

    SET num = num - 1, version = version + 1, update_time = NOW()
    WHERE goods_id = #{skuId} AND num > 0    <!-- num>0 是防超卖的核心条件 -->
</update>
<!-- affected_rows > 0 才算扣减成功 -->

## 注意事项

- **Redis 与 DB 库存的初始化对齐**：活动开始前要把 DB 库存同步到 Redis（`SET stock:sku 1000`）。两者不一致是超卖/少卖的常见原因。建议活动前做一次全量校验，并记录初始值用于对账。
- **Redis 主从切换的数据丢失**：Redis 主从异步复制，主节点宕机时未同步的扣减会丢。Sentinel 切换后新主可能比旧主少扣几笔，导致「Redis 显示剩余 10，实际 DB 已扣完」。应对：切换期间触发 DB 兜底 + 对账补偿。
- **MQ 消费失败的处理**：Redis 扣减成功但 MQ 消费失败（DB 未扣）会导致 Redis 和 DB 不一致。MQ 必须有重试 + 死信队列，死信消息人工兜底。绝不能丢消息（资金/库存场景）。
- **兜底 SQL 的索引**：`UPDATE ... WHERE goods_id=? AND num>0` 的 `goods_id` 必须有索引，否则全表扫描 + 行锁升级为表锁，DB 直接瘫痪。
- **暂停售卖的开关**：极端故障（Redis + DB 都异常）时应能一键暂停售卖（返回静态「活动太火爆」页）。这是最后的保险，避免系统雪崩。开关用配置中心（如 Apollo）动态控制。
- **对账机制的必要性**：每日定时跑对账——Redis 剩余库存 + 已下单数量 应等于 初始库存。任何偏差告警并人工核查。这是发现「隐性超卖/少卖」的唯一手段。
- **热点商品的库存分片**：单一爆款的库存 Key 是热点（单 Redis 实例扛不住）。把库存拆到 N 个子 Key（`stock:sku:0~9` 各存 100），扣减时随机选一个，分摊写压力。注意分片后的对账要汇总所有子 Key。
