# 如何设计一个网盘系统？类似百度网盘，支持大文件上传下载。

【场景分析】
网盘系统需求：多端文件同步、大文件上传、断点续传、秒传、文件分享、权限管理。

【核心功能】
1. 文件上传：分片上传 + 断点续传 + 秒传
2. 文件下载：分片下载 + 断点续传
3. 文件管理：增删改查 + 目录树 + 回收站
4. 文件分享：链接分享 + 密码保护 + 有效期
5. 文件同步：多设备文件同步

【秒传实现】
1. 上传前计算文件MD5
2. 查询服务端是否已存在相同MD5
3. 如存在 → 直接建立引用关系（不实际上传）
4. 如不存在 → 正常上传

【分片上传 + 断点续传】
```
1. 客户端：
   - 大文件分片（每片5MB）
   - 计算每片MD5
   - 上传前询问服务端已上传哪些分片
   - 只上传缺失分片
2. 服务端：
   - 记录分片上传状态
   - 所有分片上传完成后合并
   - 合并后校验整体MD5
```

【实战案例】
在处理10GB大文件上传时，如果客户端将所有分片MD5一次性上传到服务端校验，会导致URL过长或Header溢出。实际开发中应改为“预上传”接口：客户端先上传整体文件Hash，服务端返回缺失的分片序号列表（如 `[1, 3, 5]`），客户端再按需上传。

【代码示例】
```java
// 服务端：合并分片校验 (Java)
public boolean mergeFile(String fileId, String fileHash, int totalChunks) {
    List<File> chunks = getUploadedChunks(fileId);
    if (chunks.size() != totalChunks) return false; // 分片缺失
    
    File mergedFile = mergeChunks(chunks); // 顺序合并
    String actualHash = DigestUtils.md5Hex(new FileInputStream(mergedFile));
    
    if (!actualHash.equals(fileHash)) {
        mergedFile.delete();
        return false; // 文件损坏，校验失败
    }
    saveToOSS(mergedFile); // 上传对象存储
    return true;
}
```

【存储架构】
1. 元数据存储：
   - MySQL：文件信息（名称/大小/路径/权限）
   - Redis：目录树缓存
2. 内容存储：
   - 对象存储（OSS/S3）：文件实际内容
   - 小文件合并：多个小文件合并存储
3. CDN加速：
   - 下载走CDN
   - 热点文件预缓存

【存储方案对比】

| 方案 | 适用场景 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **自建NAS/文件服务器** | 私有云、内网 | 完全自主可控，带宽成本低 | 扩容难，需维护硬盘故障，性能瓶颈明显 |
| **对象存储 (OSS/S3)** | 公有网盘 | 无限扩容，高可靠，自带CDN集成 | 存储成本随量线性增加，流量费较高 |
| **纠删码存储** | 极致成本控制 | 存储利用率高，容错性好 | 读写性能损耗大，恢复数据计算耗时 |

【文件分享】
- 生成分享链接 + 提取码
- 有效期控制（1天/7天/永久）
- 访问次数限制
- 权限控制（查看/下载/编辑）

【存储优化】
- 去重：相同MD5的文件只存一份
- 压缩：自动压缩文本文件
- 冷热分离：30天未访问的文件迁移到低价存储
- 配额管理：用户存储空间限制

【多端同步】
- 增量同步：只同步变化的文件
- 冲突解决：基于版本号 + 时间戳
- 同步队列：变更事件入MQ，异步同步

【安全设计】
- 传输加密：HTTPS
- 存储加密：服务端加密（SSE）
- 访问控制：ACL + Token
- 防盗链：Referer检查 + 签名URL

## 延伸：如何设计一个大文件上传下载系统？支持断点续传和秒传。

> 合并自 `scen-080`（相似度 69%）

【场景分析】
大文件（GB/TB级）上传需解决：网络中断重传、带宽限制、服务器内存压力、秒传（去重）。

**【实战案例】**
某视频网盘系统上线初期，允许全量并发上传分片，导致后端IO被打满。后来将并发控制逻辑下沉到服务端，令客户端必须携带`server_ticket`（服务端签发的通行证）才能上传，成功实现了全局限流。

【分片上传流程】
1. **前端预处理**：
   - 利用WebWorker计算文件MD5（大文件耗时，避免阻塞UI）。
   - 按固定大小（如5MB）切片。
2. **上传初始化**：
   - 请求后端，上传`fileMd5`、`fileName`、`totalChunks`。
3. **秒传检查**：
   - 后端检查`FileRecord`表是否存在该`fileMd5`且状态为`complete`。
   - 存在则直接返回成功（秒传），无需上传任何数据。
4. **分片上传**：
   - 并发上传切片（3-5个并发）。
   - 每个请求携带`uploadId`、`chunkIndex`、`fileMd5`。
5. **合并分片**：
   - 前端上传完毕，请求`/merge`接口。
   - 后端检查是否所有分片都已齐全，然后执行合并（二进制流追加或文件系统rename）。

**【代码示例：Java - 检查秒传与分片状态】**
```java
public UploadCheckResponse checkUploadStatus(String fileMd5) {
    // 1. 检查秒传
    FileInfo fileInfo = fileRepository.findByMd5(fileMd5);
    if (fileInfo != null && Status.COMPLETE.equals(fileInfo.getStatus())) {
        return new UploadCheckResponse(true, Collections.emptyList());
    }
    
    // 2. 检查断点续传：查询Redis中已上传的分片索引
    Set<Integer> uploadedChunks = redisTemplate.opsForSet().members("upload:" + fileMd5);
    return new UploadCheckResponse(false, uploadedChunks);
}
```

【断点续传原理】
```text
   上传中断 (网络波动/关闭浏览器)
             ↓
   用户重新打开页面/选择文件
             ↓
   计算 MD5 → 询问后端 /check?fileMd5=xxx
             ↓
   后端查询 Redis/DB: 已上传 [0, 1, 3, 4]
             ↓
   返回: { "uploaded": [1, 3, 4], "needUpload": [0, 2, 5...] }
             ↓
   前端只上传缺失的分片 2, 5...
```

【服务端存储设计】
- **文件记录表**：`file_record (id, file_name, file_md5, file_path, status, chunk_size)`
- **分片存储路径**：
  - 临时：`/temp/{uploadId}/{chunkIndex}`
  - 正式：`/files/{year}/{month}/{fileMd5}.{ext}`
  - 或对象存储（S3/OSS）：Multipart Upload API。

【秒传与去重实现】
- **Content Addressable Storage (CAS)**：文件物理路径名为`MD5`，而非UUID或自增ID。
- **秒传逻辑**：上传前查MD5，若存在，仅在DB插入一条引用记录，指向同一个物理文件（需处理引用计数，删除时减1，为0才删物理文件）。

**【方案对比：分片上传 vs 直传】**
| 特性 | 分片上传 (断点/秒传) | 普通Form直传 | OSS/云厂商直传 |
| :--- | :--- | :--- | :--- |
| **断点续传** | 支持 | 不支持 | 支持 (需SDK) |
| **秒传** | 支持 (需服务端计算) | 不支持 | 支持 (需服务端记录) |
| **带宽压力** | 在应用服务器 | 在应用服务器 | 在云端 (节省服务器带宽) |
| **实现复杂度** | 高 (需合并/校验) | 低 | 中 (需签名鉴权) |
| **适用场景** | 企业网盘、视频编辑 | 小文件头像上传 | C端用户UGC内容 |

【## 常见考点】
1. **如何保证分片合并后的文件完整性？**
   - 合并完成后，再次计算整个文件的MD5，与前端上报的`fileMd5`比对。不一致则删除重传。
2. **分片大小定为多少合适？**
   - 建议 1MB ~ 10MB。太小HTTP握手开销大；太大单个分片上传失败重传代价高。5MB是常见折中值。
3. **如果用户上传了99%的切片就关掉了浏览器，临时文件怎么处理？**
   - 定时任务（如每天凌晨）扫描`FileRecord`表中状态为`uploading`且`updated_at`超过24小时的数据，清理对应的临时分片和记录。
4. **并发上传时，后端如何处理分片乱序到达？**
   - 分片存储时文件名带上`chunkIndex`（如`temp_123_5.part`），合并时按Index排序再拼接即可，不依赖到达顺序。

## 记忆要点

- 核心原理：前端切片计算MD5，后端比对MD5实现秒传，校验Redis分片索引实现断点续传。
- 性能与并发：分片大小推荐1-10MB（折中5MB），并发上传但乱序到达时按chunkIndex合并。
- 秒传设计：利用文件MD5作为物理路径去重，数据库仅存引用记录。
- 资源清理：为防弃用文件占满磁盘，定时任务扫描超时（如24小时）未完成上传的临时分片并清理。
