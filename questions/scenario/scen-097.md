# 如何设计一个防接口重放攻击的方案？

【场景分析】
重放攻击 是指攻击者截获一个有效的请求，然后再次发送该请求，从而导致系统被重复操作（如重复扣款、重复下单）。

【防重放核心机制】
防重放主要依赖三个要素：时间戳保证时效性，Nonce 保证唯一性，签名 保证完整性。

**架构流程图**：
```
┌─────────────┐              ┌──────────────┐             ┌──────────────┐
│   Client    │              │   API Gateway│             │     Redis    │
└──────┬──────┘              └──────┬───────┘             └──────┬───────┘
       │                           │                           │
       │ 1. 生成 Timestamp, Nonce  │                           │
       │ 2. 计算 Signature          │                           │
       │──────────────────────────>│                           │
       │   Request + Headers       │                           │
       │                           │                           │
       │                           │ 3. 检查时间窗口 (±5min)    │
       │                           │ 4. 检查 Nonce 是否存在     │
       │                           │──────────────────────────>│
       │                           │      SETNX nonce ...      │
       │                           │<──────────────────────────│
       │                           │      (True: 已存在->拒绝)  │
       │                           │                           │
       │                           │ 5. 验证签名                │
       │                           │ 6. 业务处理                │
       │<──────────────────────────│                           │
       │       Response            │                           │
```

【详细实现步骤】

1. **Timestamp（时间戳）**
   - 客户端请求头携带 `X-Timestamp`。
   - 服务端判断 `|CurrentTime - RequestTime| < Threshold`（通常 5 分钟）。
   - **作用**：防止长期的重放攻击，限制 Redis 中 Nonce 的存储时间。

2. **Nonce（Number Used Once）**
   - 客户端生成一个全局唯一的随机字符串（如 UUID）。
   - 服务端收到请求后，去 Redis 查询该 Nonce 是否存在。
   - **操作**：`SET nonce:{value} 1 NX EX 300`（NX 确保原子性，EX 设置过期时间与时间窗口一致）。
   - **作用**：防止时间窗口内的重复攻击。

3. **Signature（签名）**
   - 算法：`HMAC-SHA256(Secret, Method + URL + Timestamp + Nonce + Body)`。
   - Secret：双方约定的密钥，不可在网络上传输。
   - **作用**：防止请求被篡改。如果攻击者修改了参数，签名验证将失败。

【边界条件与特殊情况】

1. **时钟不同步问题**
   - 服务器时间必须准确（使用 NTP 同步）。
   - 客户端若时间不准，可能导致请求被拒。解决方案：客户端先调用获取服务器时间接口，或者适当放宽时间窗口（如 10 分钟），但这会降低安全性。

2. **并发请求冲突**
   - 如果客户端极短时间内（网络抖动）发送两个相同请求，或者恶意并发重放。
   - 利用 Redis `SETNX` 的原子性，保证同一 Nonce 只能被处理一次。第一个请求写入成功，第二个请求发现 Key 已存在，直接拒绝。

3. **网络延迟与重试**
   - 如果客户端因网络超时自动重试，应使用新的 Nonce，而不是复用旧的 Nonce。

【幂等性与防重放的关系】
- **防重放**：侧重于安全，拦截恶意或无效的重复请求，发生在网关层。
- **幂等性**：侧重于业务一致性，确保 legitimate 的重试（如支付回调）不会导致数据错误，发生在业务层。
- **最佳实践**：网关防重放做第一道防线，业务层幂等做兜底。

## 常见考点
1. **为什么 Nonce 需要设置过期时间？**
   - 防止 Redis 内存无限膨胀，配合时间戳机制，旧的非ce自然失效，防止重放攻击的时间窗口仅限于有效期内。
2. **如果攻击者瞬间（1ms内）并发发送 10000 个相同请求怎么办？**
   - Redis 是单线程处理命令，`SETNX` 是原子操作。第一个请求写入成功并处理，后续 9999 个请求都会发现 Key 已存在而被拦截。性能取决于 Redis 的 QPS，通常足以应付。
3. **如何防止请求中的 Body 被篡改？**
   - 必须将 Body 的内容（或 Body 的 Hash）参与签名计算。任何字节的变动都会导致签名不一致。
