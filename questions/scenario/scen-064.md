# 如何设计敏感数据加密和脱敏方案？

【场景分析】
敏感数据包括：手机号、身份证号、银行卡号、密码、地址、姓名等。

【加密分层】

【1. 传输加密】
- HTTPS/TLS：端到端加密
- 双向SSL（mTLS）：服务间通信
- API签名：防篡改（HMAC-SHA256）

【2. 存储加密】
- 密码：BCrypt/SCrypt（不可逆）
  ```java
  BCryptPasswordEncoder encoder = new BCryptPasswordEncoder();
  String hashed = encoder.encode("password123");
  // 存储hashed，永远不存明文
  ```
- 手机号/身份证：AES对称加密
  ```
  存储：AES_encrypt("13812345678", secretKey)
  查询：AES_decrypt(ciphertext, secretKey)
  ```
- 银行卡号：AES加密 + HMAC校验
- 敏感配置：Vault/KMS管理密钥

【3. 展示脱敏】
- 手机号：138****5678
- 身份证：110***********1234
- 银行卡：**** **** **** 1234
- 姓名：张*三
- 邮箱：z***@example.com

实现：MyBatis拦截器 / Jackson序列化脱敏
```java
@Sensitive(type = SensitiveType.PHONE)
private String phone;  // 序列化时自动脱敏
```

【4. 日志脱敏】
- 日志中不能出现敏感数据明文
- 统一脱敏工具：LogUtil.mask(phone)
- 日志框架Filter自动脱敏

【5. 数据库加密】
- 透明数据加密（TDE）：MySQL Enterprise
- 字段级加密：应用层加密
- 全盘加密：磁盘级别

【密钥管理】
- KMS（Key Management Service）：AWS KMS / 阿里云KMS
- Vault：HashiCorp密钥管理
- 密钥轮换：定期更换密钥
- 密钥分级：主密钥 → 数据密钥 → 数据

【数据加密架构图】
```
┌─────────────┐                  ┌───────────────┐
│   Client    │                  │   KMS/Vault   │
│  (App/Web)  │                  │  (密钥管理)    │
└──────┬──────┘                  └───────┬───────┘
       │                                  │
       │ 1. 请求 DataKey (DEK)            │
       │<---------------------------------│
       │ 2. 返回 DEK (明文) & (加密版)     │
       │---------------------------------│
       │                                  │
       │ 3. 内存中使用 DEK 加密数据         │
       │                                  │
       │ 4. 存储: [密文] + [加密版DEK]      │
       ▼                                  │
┌───────────────────────────────────────┐ │
│          Database / Storage            │ │
│  Col_A: Ciphertext (AES-256-GCM)      │ │
│  Col_B: Encrypted_DEK (Wrapped by MK) │ │
└───────────────────────────────────────┘ │
```

- 数据不出境：数据本地化存储

## 常见考点
1. **加密模式选择**：ECB、CBC、GCM 有什么区别？推荐哪种？（答：ECB 不安全（相同明文生成相同密文）；CBC 需要初始向量 IV，但无法防篡改；GCM 模式是 AEAD（认证加密），不仅保密还校验完整性，推荐用于现代应用）
2. **密钥轮换实操**：数据库里存了上亿条加密数据，密钥泄露了怎么轮换？（答：使用信封加密。只需用新的主密钥 MK 重新加密数据密钥 DEK，无需解密重写所有数据，只需更新 DEK 的加密版本）
3. **检索问题**：手机号加密后（如 AES），如何根据手机号查询用户？（答：方案一：确定性加密（如 AES-SIV），相同明文对应相同密文，可建索引但泄露频率信息；方案二：保存哈希值用于检索，密文用于解密）
4. **前端加密意义**：前端 HTTPS 已经加密了，为什么还要做业务层加密？（答：防御内部威胁，防止 DBA 或运维人员直接查库看到明文；满足合规审计要求；即使数据备份泄露也无法还原）

## 记忆要点

- 存储策略：因密码需防破解，故用BCrypt不可逆加密，业务数据用AES对称加密。
- 展示脱敏：通过注解或拦截器，将手机号身份证在序列化时打星号处理。
- 密钥管理：采用信封加密，业务用明文DEK加密数据，仅密文DEK入库由KMS管。
- 安全检索：因AES加密难查询，故额外存哈希索引兼顾安全检索与解密。
- GCM模式：推荐用AES-GCM，因它不仅加密还能校验防篡改。
