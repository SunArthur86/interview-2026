# 如何设计容量评估和扩容方案？应对大促/活动流量。

【场景分析】
容量评估目标：预测系统在目标负载下的资源需求，提前扩容，避免故障。

【容量评估方法】
1. 历史数据推算：
   - 去年双11峰值QPS × 增长率
   - 日均QPS × 峰值系数（通常5-10倍）
2. 压测验证：
   - 全链路压测得出单机QPS
   - 目标QPS / 单机QPS = 所需机器数
3. 业务预估：
   - 大促商品数 × 预期转化率
   - 营销活动覆盖人数

【容量评估清单】
```
目标指标：
- 总QPS: 100万
- 峰值QPS: 150万
- P99延迟: <200ms
- 可用性: 99.99%

各层容量推算：
- 网关层: 100台 × 1.5万QPS/台 = 150万QPS ✓
- 应用层: 200台 × 5000QPS/台 = 100万QPS ✓
- DB层: 主1+从16，读写分离
- Redis: 32节点Cluster
- MQ: 12节点Kafka集群
```

【扩容策略】
1. 预扩容（活动前）：
   - 提前24-48小时扩容
   - 数据预热到缓存
   - 弹性资源池预备
2. 定时扩容：
   - 根据业务曲线定时扩缩
   - 每天10:00-22:00高峰扩容
3. 自动扩容（HPA）：
   - K8s Horizontal Pod Autoscaler
   - CPU>70%自动扩容
   - 基于自定义指标（QPS）
4. 紧急扩容：
   - 手动一键扩容
   - 预热镜像 + 快速部署

【扩容注意事项】
1. 数据库：
   - 主库不能简单扩容（需要分库分表）
   - 从库可快速加（但数据同步需时间）
   - 连接池配置需同步调整
2. 缓存：
   - Redis扩容需数据迁移
   - Codis/Redis Cluster支持在线扩缩
3. 消息队列：
   - 增加分区需Consumer配合
   - 先增Consumer再增分区
4. 负载均衡：
   - 新实例注册后健康检查通过才接流量
   - 避免瞬时全部流量打到新实例

【降级预案】
- 当容量不足以支撑全量流量时
- 按优先级降级：
  P0: 下单/支付（不可降级）
  P1: 商品详情/搜索
  P2: 评论/推荐
  P3: 历史订单/个人中心

### 增强补充：水位规划与限流细节

1. **水位线计算（预留 Buffer）**
   - **原则**：不要以 100% 利用率作为目标。
   - **推荐水位**：
     - CPU < 70% (预留应对突发流量)
     - 内存 < 80% (避免 Swap)
     - 单机 QPS 峰值 = 压测极限值 × 0.8 (安全系数)
   - **公式**：`所需实例数 = 峰值QPS / (单机极限QPS × 0.8)`

2. **依赖服务的瓶颈放大效应**
   - 假设 A 服务依赖 B 服务，A 的扩容会导致 B 的压力线性增加。
   - 如果 B 是 DB，无法横向扩容，那么 A 扩容再多也会把 B 打挂。
   - **解决**：必须在 B 服务（如 DB, Redis, 依赖接口）层独立评估容量，并设置 **自治限流**（B 服务保护自己），而不是仅仅扩容 A。

3. **冷启动对自动扩容的影响**
   - JVM 应用在刚启动时，类加载、JIT 编译未完成，性能极低（可能只有峰值的 20%）。
   - 如果 HPA 监控到 CPU 高立即扩容，新来的 Pod 也是冷状态，不仅分担不了流量，反而消耗资源，可能导致雪崩。
   - **解决**：使用 `startupProbe` 确保就绪后才接流量；或者进行预热；或者在 HPA 中设置 `stabilizationWindowSeconds` 延迟缩容，但对于扩容，通常建议配合 KEDA（基于事件驱动）或提前预热。

## 常见考点
1.  **如何应对缓存击穿带来的容量评估偏差？**
    *   *提示*：压测时缓存命中率可能较高，但大促时 Key 可能失效，导致流量直接打到 DB。容量评估必须包含“缓存穿透/击穿”场景的兜底方案（如互斥锁、布隆过滤器）。
2.  **如果流量超过了预估的峰值，系统自动扩容来不及怎么办？**
    *   *提示*：必须配置**限流**（拒绝请求）、**降级**（返回默认值或旧数据）、**兜底页面**，保证系统存活性（Survivability）优于完整性。
3.  **Kafka 扩容时为何要增加 Partition？**
    *   *提示*：Kafka 的消费并发度由 Partition 数量和 Consumer Group 内的线程数决定（Min(分区数, 线程数)）。如果只扩 Broker 不扩 Partition，吞吐量无法线性提升。

## 记忆要点

- 容量公式：所需实例数 = 目标峰值QPS / (单机极限QPS × 0.8 安全水位)。
- 扩容策略：提前预扩容+定时扩容应对规律，HPA自动扩容应对突发，手动预案兜底。
- 瓶颈放大：因为下层依赖无法按需扩容（如DB），所以必须设置自治限流保护。
- 有状态扩容难点：DB主库需分库分表，Redis需数据迁移，Kafka吞吐受限于Partition数。
