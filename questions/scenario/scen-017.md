# 如何设计一个海量数据的冷热分离方案？热数据百亿，冷数据千亿。

【场景分析】
冷热分离需求：热数据频繁访问（低延迟），冷数据偶尔查询（可接受高延迟），大幅降低数据库存储成本和负载。

【冷热判定标准】
- **时间维度**（最常用）：
  - 热数据：近 7-30 天
  - 温数据：30-180 天
  - 冷数据：>180 天
- **访问频率**：LRCU（最近访问频率），如日访问>10次为热。< 1次为冷。
- **业务维度**：
  - 订单：未完成/退款中为热，已完成>3个月为冷。
  - 用户：活跃用户为热，沉睡用户为冷。

【分层存储架构设计】
1. **热数据层**：
   - **存储**：MySQL (InnoDB SSD) + Redis (L1缓存)。
   - **要求**：高并发事务支持，低延迟（<10ms）。
   - **索引**：保留完整高频查询索引。
2. **温数据层**：
   - **存储**：Elasticsearch / MongoDB。
   - **优势**：ES 支持复杂检索（全文、倒排），适合审计、后台查询。
3. **冷数据层**：
   - **存储**：HBase / 对象存储 (S3/OSS) + Hive。
   - **优势**：成本极低（约为热数据的1/10），适合归档、离线分析。

```text
└────────┬─────────┘        └────────┬─────────┘

         ▼                          ▼
┌───────────────────┐     ┌───────────────────┐
│ 冷数据 (S3/HBase) │     │  归档层           │
│ (压缩/低频)       │     │  (磁带/Glacier)   │
└───────────────────┘     └───────────────────┘
```

【数据迁移机制】
1. **双写/同步策略**：
   - 业务层双写：写 MySQL 同时异步写 ES（需考虑最终一致性）。
   - CDC 监听：Canal/Debezium 监听 MySQL Binlog，解析后同步到 ES 或 HBase。
2. **归档执行**：
   - **定时任务**：凌晨低峰期扫描“已冷却”数据。
   - **流式处理**：利用 Kafka Connect 批量导出到 S3，保证事务性。
3. **数据清理**：
   - 确认冷存储写入成功后，删除热库数据（通常采用逻辑删除 `is_deleted=1`，后续物理清理）。

【查询路由策略（透明路由）】
- **统一网关/ORM中间件**：拦截查询 SQL。
  - 判断条件（如 `create_time > '2023-01-01'`）-> 路由 MySQL。
  - 判断条件（如 `create_time < '2023-01-01'`）-> 路由 ES。
- **二次查询**：若用户明确查询需关联冷热数据，需在内存层聚合。

【冷数据回热】
- 触发：用户查询冷数据。
- 动作：异步将数据加载回 MySQL，并写入 Redis。
- 策略：设置临时 TTL，避免频繁回热污染热库。

## 常见考点
1. **冷热分离时，如何保证数据一致性？**
   - 采用 MQ + 最终一致性；迁移过程中记录“迁移中”状态，防止数据不可读；利用 Binlog 保证增量同步。
2. **如果用户查询的数据跨越了冷热界限（如查询近一年的订单），怎么处理？**
   - 分页查询最复杂。通常路由层将查询拆解：查热库 + 查冷库 -> 内存合并排序 -> 返回。注意深度分页的性能损耗。
3. **如何选择温数据的存储介质？ES 还是 MySQL？**
   - 如果需要模糊搜索、任意字段组合查询，选 ES；如果主要是 ID 查询或简单范围查询，且需要事务支持，可选用廉价 MySQL 实例或 MongoDB。
