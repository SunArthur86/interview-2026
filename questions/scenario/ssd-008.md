# 如何设计一个分布式锁？有哪些实现方案？

🎯 **本质**：分布式锁是在分布式系统中保证同一时刻只有一个客户端执行临界区代码的机制。

📊 **三种主流方案**：

### 1. Redis 分布式锁
**基础实现**：
```bash
SET lock_key unique_value NX PX 30000
// NX：不存在才设置
// PX：过期时间30秒
// unique_value：防止误删别人的锁（需满足全局唯一性，如 UUID+线程ID）
```

**释放锁（Lua 保证原子性）**：
```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

**核心问题：锁过期但业务未完成**
*   **解决**：看门狗 机制自动续期。
*   **原理**：后台线程检测锁持有状态，如果业务未执行完且锁快过期，自动重置过期时间。
*   **Redisson 实现**：
    ```java
    RLock lock = redisson.getLock("myLock");
    lock.lock();  // 默认30秒过期，每10秒自动续期（锁续期默认间隔为过期时间的1/3）
    lock.tryLock(10, 30, TimeUnit.SECONDS);  // 尝试获取
    ```

**RedLock 算法（Redis 作者提出）**：
*   **原理**：在 N (N 通常为奇数，如 5) 个独立 Redis 实例上获取锁。计算获取锁消耗时间，只有当在大多数节点（N/2 + 1）上成功获取锁，且消耗时间小于锁有效期时，才算获取成功。
*   **目的**：解决单点 Redis 主从切换（异步复制导致数据丢失）引发的锁失效问题（A 节点拿锁，未同步给 B，A 挂了，B 变主，别人拿锁）。

### 2. ZooKeeper 分布式锁
**原理**：利用临时有序节点。客户端创建 `/lock/seq-` 临时有序节点，获取所有子节点，判断自己是否序号最小。若最小则获得锁；否则监听（Watch）前一个节点的删除事件。

**优势**：
*   **公平锁**（FIFO）：按节点序号排队。
*   **锁释放自动通知**：Watch 机制避免了无效的自旋轮询，降低 CPU 消耗。
*   **会话断开自动释放**：基于 Session 生命周期，解决死锁问题。

**劣势**：性能低于 Redis（写操作需要经过 Leader 投票，存在网络开销）。

### 3. 数据库分布式锁
*   **方案 A：唯一索引**
    `INSERT INTO lock_table(resource, thread_id) VALUES('order_123', 'uuid');`
    *   唯一索引冲突则获取失败。
    *   缺点：不支持重入，锁释放需删除记录。
*   **方案 B：乐观锁**
    `UPDATE table SET version=version+1 WHERE id=? AND version=?`
    *   适用于读多写少。
*   **方案 C：悲观锁**
    `SELECT * FROM table WHERE id=? FOR UPDATE;`
    *   利用数据库行锁，必须使用在事务中，且索引命中才生效。

**架构图示（Redis 锁流程）**：
```
Client A                   Client B                   Redis Cluster
   |                           |                           |
   |--- SET NX PX 30000 ------>|                           |
   |<-- OK --------------------|                           |
   | (执行业务逻辑...)          |                           |
   |                           |--- SET NX PX 30000 ------>|
   |                           |<-- (nil) -----------------|
   |                           | (自旋等待/休眠)           |
   |--- (Lua) GET + DEL ------>|                           |
   |<-- 1 ---------------------|                           |
   |                           |--- SET NX PX 30000 ------>|
   |                           |<-- OK --------------------|
   |                           | (获得锁)                   |
```

**选择建议**：
| 方案 | 性能 | 可靠性 | 复杂度 | 适用场景 |
|------|------|--------|--------|---------|
| Redis | 高 | 中（主从切换有风险） | 低 | 大多数场景、高并发 |
| ZK | 中 | 高 (CP 模型) | 中 | 强一致要求、公平锁需求 |
| MySQL | 低 | 高 | 低 | 低并发、已有数据库资源 |

**注意事项**：
1. 锁要有过期时间（防死锁）。
2. 锁要可重入（同一线程可多次获取，通常存入 Hash 结构记录计数）。
3. 释放锁要验证持有者（防误删）。

## 常见考点
1. **Redis 主从架构锁丢失问题**：既然有主从复制延迟，为什么不用 RedLock？或者 RedLock 有什么缺点？（成本高、性能差、严重依赖时钟）。
2. **ZK 锁的实现细节**：为什么要监听前一个节点而不是监听所有子节点？（羊群效应，防止连锁唤醒导致 ZK 压力骤增）。
3. **Redisson 的看门狗续期原理**：如果主线程宕机，看门狗线程如何停止？（基于 Netty 的 HashedWheelTimer，通过引用计数或线程生命周期绑定）。
4. **数据库锁死锁与回滚**：使用 `FOR UPDATE` 时如果事务时间过长会占用连接池，如何优化？

## 记忆要点

- 核心方案对比：Redis性能高但主从切换有丢锁风险，ZK强一致可靠但性能低
- Redis锁基石：SET NX PX保证互斥与防死锁，Lua删锁防误删
- 锁续期原理：后台看门狗线程定期检测，若业务未完则重置过期时间
- RedLock算法：向N个独立节点获取锁，半数以上成功才算获取成功
