# 如何设计一个短链系统？数据量约百亿级别，日均访问量十亿次。

【场景分析】
短链系统核心需求：长URL转短URL（发号器），短URL跳转长URL（重定向），QPS极高。

【整体架构】
1. 发号器服务：生成短码（6位Base62编码，可表达568亿组合）
   - 方案A：自增ID + Base62编码（简单但中心化）
   - 方案B：号段模式（Leaf-Segment，DB发号+内存分配，性能高）
   - 方案C：Snowflake算法（分布式ID，含时间戳+机器号）
2. 存储层：
   - 短码→长URL映射：MySQL分库分表（按短码hash分128库×128表）
   - 百亿数据：每个表约600万条，MySQL完全可控
3. 缓存层：Redis集群缓存热点短链（LRU淘汰）
4. 读路径：CDN → Nginx → Redis（命中95%+）→ MySQL

【核心流程】
- 生成短链：长URL → Hash/MurmurHash查重 → 发号器拿ID → Base62编码 → 存MySQL + Redis
- 访问短链：短码 → Redis查询 → 302临时重定向（利于统计）或301永久重定向（利于缓存）

【性能优化】
- 读写分离：写走Master，读走Slave
- 布隆过滤器：快速判断短码是否存在，避免缓存穿透
- 批量预写：发号器本地缓存号段，减少DB压力
- 监控：短链生成QPS、重定向延迟、缓存命中率

【实战案例】
在双11大促期间，曾出现因生成短链的机器时钟回拨导致 Snowflake ID 重复，进而引发覆盖写入的严重事故。后切换为 Leaf 号段模式，并增加 `biz_tag` 业务维度隔离，彻底解决了时钟回拨和 ID 冲突问题。

【Leaf 号段模式关键代码（伪代码）】
```java
// Double Buffer 机制预加载号段，避免数据库成为瓶颈
public synchronized Long getId() {
    if (currentBuffer.maxId - currentBuffer.currentId > threshold) {
        return currentBuffer.nextId();
    }
    // 当前号段剩余不足，切换到另一号段
    updateBuffer();
    return currentBuffer.nextId();
}
```

【跳转实现（Nginx/OpenResty）】
```lua
-- 使用 Lua 脚本在 Nginx 层直接访问 Redis，不走 Java 应用层，性能最高
local short_key = ngx.var.arg_key 
local long_url = redis.call("GET", "short:" .. short_key)
if long_url then
    return ngx.redirect(long_url, 302)
else
    return ngx.exit(404)
end
```

【扩展功能】
- 访问统计（PV/UV）→ 异步写入ClickHouse
- 过期机制：TTL + 定时清理
- 防恶意：频率限制 + 黑名单
- 自定义短码：预留前缀检测

### 常见考点
1. **发号器选型**：为什么 Snowflake 在时钟回拨时会有问题？号段模式步长设置为多少合适？（如 1000，需平衡 DB 压力与重启 ID 浪费）。
2. **跳转状态码**：301 和 302 的区别及对浏览器缓存、统计数据准确性的影响（302 浏览器不缓存，统计更准）。
3. **哈希冲突**：如果使用 MurmurHash 直接生成短码，发生冲突如何处理？（加随机后缀重试，或强行递增 ID 覆盖，通常不推荐直接哈希作为最终短码）。


## 记忆要点

- 发号器三方案：自增Base62、号段模式Leaf、雪花算法（需防时钟回拨）
- 因为百亿数据需扩展，所以MySQL按短码Hash分128库x128表
- 性能扛十亿读：读路径CDN->布隆过滤->Redis->MySQL，重定向不走Java
- 跳转核心选型：302临时重定向利于业务统计，301永久重定向省带宽缓存
