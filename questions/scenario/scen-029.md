# 如何设计一个支付系统？支持千万级日订单，金额零误差。

【场景分析】
支付系统核心要求：
- **高可用**：服务不能挂，支持降级
- **高一致**：资金绝对准确（账平）
- **安全**：防篡改、防重放、防伪造
- **可追溯**：每一笔资金流向都有记录

【支付系统核心架构】
1. **收银台**：聚合支付方式，统一API
2. **支付网关**：
   - 统一接入层：适配微信/支付宝/银联等不同渠道的API差异
   - 路由策略：根据费率、渠道限额、成功率选择最优渠道
   - 限流与风控：防止恶意刷单、盗刷
3. **核心账务**：
   - 记账逻辑：复式记账（借贷必相等）
   - 账户体系：记账主体、账户类型（余额、冻结、营销）
   - 分库分表：按user_id分片，单库单表数据量可控
4. **渠道交互**：处理请求发起、状态同步、异步回调

【支付核心流程（状态机视角）】
```
[Init] ──(用户支付)──> [Paying] ──(渠道成功)──> [Success]
  │                        │                 │
  │                        │(渠道失败/取消)   │(回调通知业务)
  │                        ↓                 ↓
  │                     [Fail/Closed]      [Done]
  │
  └──>(超时/重试)──> [Unknown] ──(对账发现)──> [Success/Fail]
```

【核心设计难点】
1. **幂等控制**：
   - **原因**：网络抖动导致用户重复点击、渠道重复回调
   - **方案**：
     - 请求层：前端防抖（按钮置灰）
     - 接口层：Redis SETNX `pay:token:{orderId}` 
     - 数据层：支付表对`out_trade_no`（商户订单号）建立唯一索引
2. **分布式事务与一致性**：
   - 扣减库存（订单服务） + 创建支付单（支付服务） + 扣减余额（账户服务）
   - **方案一：TCC (Try-Confirm-Cancel)**：强一致性，代码侵入性强，需实现三个阶段
   - **方案二：本地消息表（最终一致性）**：
     - 订单服务落单，同时记录“待支付”消息到本地表
     - 定时任务轮询发送消息到MQ
     - 支付服务消费MQ，处理成功后ACK
3. **异步回调与对账**：
   - 支付结果由渠道异步通知（Notify URL）
   - **处理逻辑**：验签（RSA2） → 幂等校验 → 更新支付单状态 → 发送MQ通知下游业务
   - **对账系统（T+1）**：
     - 下载渠道账单
     - 比对我方流水与渠道流水
     - 差异处理：
       - **长款**：渠道有我方无（可能回调丢失），主动补单
       - **短款**：我方有渠道无（可能支付失败但记录错误），人工核实冲正

【安全设计】
- **传输安全**：强制HTTPS，公私钥对加密敏感字段
- **签名机制**：MD5/RSA签名，防止参数篡改（所有参数按ASCII排序+Salt）
- **防重放**：请求携带`timestamp`和`nonce`，服务端缓存`nonce`，5分钟内拒绝重复
- **数据存储**：银行卡号、实名信息AES加密存储，日志脱敏

【高可用设计】
- **多活架构**：同城双活，异地灾备（数据冷备）
- **容错**：渠道超时自动切换备用渠道（路由降级）
- **熔断**：服务调用失败率超阈值触发熔断，防止雪崩
- **限流**：单用户支付频率限制，防止暴力攻击

## 常见考点
1. **分布式事务选择**：支付系统中为什么通常不用Seata AT模式，而偏爱TCC或消息队列？（AT有锁开销，性能弱；TCC/MQ在金融场景更可控）
2. **幂等实现**：如果支付回调先到了，更新了数据库，但MQ发送失败，怎么办？（本地消息表补偿或重试机制）
3. **对账细节**：如果渠道一直在扣款中（Pending），何时算超时？（通常设置30min或24h，超时后主动查询渠道接口确认）
4. **金额计算**：为什么不能用Double/Float计算金额？（精度丢失，如0.1+0.2!=0.3，必须用BigDecimal或Long存分）

## 记忆要点

- 核心设计：网关层统一适配渠道并路由，核心账务采用复式记账法（借贷必相等）
- 防重保障：接口用Redis Token防抖，数据层对out_trade_no建唯一索引做终极兜底
- 状态机与异步：订单流转必走状态机校验，支付结果依赖异步回调+防重放验签
- 资金最终一致：核心用本地消息表或TCC，T+1定时对账平账（长款补单、短款冲正）

## 结构化回答



**30 秒电梯演讲：** 像银行转账：记账要双人复核（幂等），每天晚上必须核对账目（对账），一分钱不能差。

**展开框架：**
1. **核心流程** — 创建→支付→回调→状态更新
2. **幂等设计** — 唯一ID防重复扣款
3. **事务控制** — TCC或本地消息表保一致

**收尾：** 支付系统如何保证幂等？
