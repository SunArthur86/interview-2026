# 如何设计直播弹幕系统？百万级用户同时在线，每秒十万条弹幕。

### 场景分析
弹幕系统特点：写入极高（10w+/s）、需实时推送、消息可丢失（容忍）、按直播间隔离。

### 实战案例
某大型游戏直播平台曾出现“弹幕爆炸”事故：由于未对热度直播间做流量隔离，某大主播翻车瞬间弹幕洪量级涌入同一 Kafka 分区，导致网卡打满、分区拥堵，进而引发普通直播间大面积推辞延迟。**教训**：必须对热点直播间进行物理隔离或独立的 Topic 命名空间（如 `live_hot_{room_id}`）。

### 架构设计
1. **连接层**
   - WebSocket 长连接（Netty/Go）
   - 每台服务器维持 5-10 万连接
   - 连接路由表：userId→serverId（Redis）
2. **消息分发**
   - 用户发弹幕 → API 网关 → 消息服务 → 按直播间分发
   - Redis Pub/Sub 或 Kafka 按直播间 Topic 分发
3. **弹幕推送**
   - 从 Kafka 消费 → 写入直播间消息队列 → WebSocket 推送给在线用户

```text
┌──────────┐  HTTP/WSS  ┌──────────┐         ┌──────────┐
│  用户    │───────────▶│  网关    │────────▶│ 逻辑服务 │
└──────────┘           └──────────┘         └────┬─────┘
                                               │  (鉴权/去重)
                                               ▼
┌──────────┐    分发    ┌──────────────────────────────┐
│  消息队列 │◀──────────│        Kafka / Redis Cluster │
│ (Kafka)  │           │  (Topic: live_room_{room_id}) │
└────┬─────┘           └──────────────────────────────┘
     │ 消费                        │ 生产
     ▼                            ▼
┌──────────┐    推送    ┌──────────┐    长连接    ┌──────────┐
│ 推送服务 │───────────▶│ 连接节点 │─────────────▶│   用户   │
│ (Consumer)│           │(Netty/Go)│              │ (Client) │
└──────────┘           └──────────┘              └──────────┘
```

### 性能优化
- 弹幕聚合：100ms 内的弹幕合并推送，减少 WebSocket 帧数
- 分级推送：VIP 用户实时推，普通用户聚合推
- 弹幕限流：每用户每秒 N 条，超出丢弃
- 热点直播间：独立服务器集群 + 本地消息队列

### 存储设计
- 实时弹幕：Redis List/Timeline（保留最近 1000 条）
- 历史弹幕：异步写入 MongoDB/ES（支持检索）
- 弹幕计数：Redis HyperLogLog 统计 UV

### 消息可靠性
- 弹幕容忍丢失：用 UDP 或 WebSocket 不保证可靠
- 关键弹幕（打赏）：走单独可靠通道
- 弹幕审核：AI 实时过滤 + 人工审核敏感词

### 弹性扩容
- K8s 根据连接数自动扩容 WebSocket 服务器
- 大主播开播预热专属资源

### 补充细节：消息有序性与保序
- **消息保序**：Kafka 分区内有序，需确保同一主播的弹幕路由到同一 Partition，或者使用 Redis Stream 保证单房间内的严格时序。
- **TCP 粘包处理**：推送层需处理 WebSocket 的帧边界，防止大弹幕或高频小弹幕导致的粘包/半包问题。

### 关键代码示例 (Go：批量弹幕推送)
```go
// 推送服务中的批量聚合逻辑
func (s *PushService) BatchPush(roomID string, msgs []Message) {
    // 针对普通用户开启聚合窗口
    buffer := s.GetBuffer(roomID)
    buffer.Add(msgs)
    if buffer.Full() || buffer.Timeout() {
        batch := buffer.Flush()
        payload, _ := json.Marshal(batch)
        // 利用 Netty 的零拷贝特性或 Go 的 syscall 写入
        s.connHub.WriteToRoom(roomID, payload)
    }
}
```

### 常见考点
1. **海量连接下的内存管理**：10万连接，每个连接占用多少内存？如何优化？（答：使用 Netty 的堆外内存，减少 GC 压力；调优 OS TCP 参数）。
2. **弱网环境**：用户网络波动时，如何保证弹幕不卡顿？（答：客户端心跳检测，自动重连；服务端维护消息队列，重连后断点续传或放弃历史消息）。
3. **广播风暴**：百万人的直播间，一条弹幕要推百万次，CPU 直接打满怎么办？（答：使用组播协议优化内网传输；或者按用户等级分层，对普通用户进行采样/丢弃）。
